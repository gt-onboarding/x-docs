<div id="oauth-10a">
  ## OAuth 1.0a
</div>

Molti endpoint della piattaforma per sviluppatori di X utilizzano il metodo OAuth 1.0a per agire, o effettuare richieste API, per conto di un account X. Ad esempio, se disponi di un’app per sviluppatori di X, puoi effettuare richieste API per conto di qualsiasi account X, a condizione che l’utente autorizzi la tua app.

Nota: se non conosci concetti come HMAC-SHA1 e percent encoding, ti consigliamo di consultare la sezione “strumenti utili” qui sotto, che elenca alcuni client API che semplificano notevolmente il processo di autenticazione.

<div id="key-concepts">
  ### Concetti chiave
</div>

<div id="signing-a-request-with-keys-and-tokens">
  #### Firmare una richiesta con chiavi e token
</div>

Devi firmare ogni richiesta all’API passando diverse chiavi e token generati nell’intestazione Authorization. Per iniziare, puoi generare diverse chiavi e token nella pagina dei dettagli della tua [app per sviluppatori X](/it/resources/fundamentals/developer-apps), tra cui:

|     |     |
| :--- | :--- |
| API key e secret:<br /><br />`oauth_consumer_key`<br /><br />`oauth_consumer_secret` | Considerali come il nome utente e la password che rappresentano la tua app per sviluppatori X quando effettua richieste all’API. |
| Access token e secret:<br /><br />`oauth_token`<br /><br />`oauth_token_secret` | Un access token e il relativo secret sono credenziali specifiche dell’utente utilizzate per autenticare le richieste API OAuth 1.0a. Indicano l’account X per conto del quale viene effettuata la richiesta.<br /><br />Puoi generare il tuo access token e il relativo secret se desideri che la tua app effettui richieste per conto dello stesso account X associato al tuo account sviluppatore nella pagina dei dettagli della tua app per sviluppatori X.<br /><br />Se desideri generare access token per un utente diverso, vedi “Effettuare richieste per conto degli utenti” di seguito. |

<div id="making-requests-on-behalf-of-users">
  #### Effettuare richieste per conto degli utenti
</div>

Quando crei una firma, ti serve un set di token di accesso che rappresenti l’utente per conto del quale effettuerai la richiesta.

Puoi generare un set di token di accesso che rappresenta l’account X proprietario dell’app per sviluppatori X dalla [pagina dei dettagli dell’app,](https://developer.x.com/content/developer-twitter/en/apps) ma se vuoi effettuare una richiesta per conto di un altro account X, il proprietario di quell’account deve concederti l’accesso accedendo al proprio account come parte del [flusso OAuth a 3 fasi](/it/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow). L’output di questo processo è un set di token di accesso (oauth\_token e oauth\_token\_secret) che può essere utilizzato per effettuare una richiesta OAuth 1.0a.

Una volta che disponi di queste chiavi e token, puoi [creare una firma](/it/resources/fundamentals/authentication#creating-a-signature) da zero. Non lo raccomandiamo a meno che tu non sappia esattamente cosa stai facendo, oppure se stai usando uno degli strumenti menzionati di seguito per effettuare una richiesta a un endpoint che richiede OAuth 1.0a.

A titolo di riferimento, ecco un esempio di richiesta cURL firmata con tutti i token generati inclusi nell’intestazione Authorization:

```bash
curl --request POST \
  --url 'https://api.x.com/1.1/statuses/update.json?status=Hello%20world' \
  --header 'authorization: OAuth oauth_consumer_key="CONSUMER_API_KEY", oauth_nonce="OAUTH_NONCE", oauth_signature="OAUTH_SIGNATURE", oauth_signature_method="HMAC-SHA1", oauth_timestamp="OAUTH_TIMESTAMP", oauth_token="ACCESS_TOKEN", oauth_version="1.0"' \
```

<Note>
  **Nota:** I token di accesso utente sono sensibili e devono essere protetti con molta attenzione. Quando vengono generati i token di accesso, l’utente che rappresentano affida alla tua applicazione il compito di mantenerli sicuri. Se la sicurezza delle chiavi API e dei token di accesso utente viene compromessa, la tua applicazione potrebbe esporre informazioni private e funzionalità dell’account. Ti invitiamo ad approfondire come [proteggere chiavi e token di accesso](/it/resources/fundamentals/developer-apps).
</Note>

<div id="useful-tools">
  ### Strumenti utili
</div>

Il processo di [creazione della firma di una richiesta](/it/resources/fundamentals/authentication#creating-a-signature) è complesso. Ti consigliamo di utilizzare una libreria client per API che generi automaticamente gran parte dei dati di autenticazione:

|     |     |
| :--- | :--- |
| [Postman](https://www.getpostman.com/) | Un client API che consente di creare e inviare richieste REST. Leggi il nostro tutorial “[Per iniziare con Postman](/it/tutorials/postman-getting-started)” per saperne di più su questo strumento. |
| [Insomnia](https://insomnia.rest/) | Insomnia è un client per REST API con gestione dei cookie, variabili d’ambiente, generazione del codice e autenticazione per Mac, Windows e Linux. |
