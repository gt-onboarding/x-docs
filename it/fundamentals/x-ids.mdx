---
title: ID di X
icon: id-card-clip
---

Ogni oggetto all'interno di X — un Tweet, un Messaggio Diretto, un Utente, una Lista e così via — ha un ID univoco.

Agli albori della piattaforma, questi ID erano numeri abbastanza piccoli da poter essere generati in sequenza. Nel tempo, per supportare la crescita, gli ID sono passati da 32 bit a 64 bit. Oggi gli ID di X sono interi senza segno a 64 bit univoci, basati sul tempo invece che sequenziali. L’ID completo è composto da un timestamp, un identificativo del worker e un numero di sequenza. X ha sviluppato un servizio interno noto come "Snowflake" per generare in modo coerente questi ID ([maggiori informazioni sul blog di X](https://blog.x.com/engineering/en_us/a/2010/announcing-snowflake.html)).

Numeri grandi come quelli a 64 bit possono creare problemi con i linguaggi di programmazione che rappresentano gli interi con meno di 64 bit. Un esempio è JavaScript, dove gli interi sono limitati a 53 bit. Per ovviare a questo, nei progetti originali delle API di X (v1/1.1), i valori degli ID venivano restituiti in due formati: sia come interi sia come stringhe.

```json
{"id": 10765432100123456789, "id_str": "10765432100123456789"}
```

Se esegui il comando `(10765432100123456789).toString()` nella console JavaScript del browser, il risultato sarà `"10765432100123458000"`: l’intero a 64 bit perde precisione a causa della conversione (talvolta chiamata "munging", ovvero una modifica distruttiva dei dati).

Nelle API di X fino alla versione 1.1, dovresti sempre usare la rappresentazione in stringa del numero per evitare perdite di precisione.

Nelle versioni più recenti dell’API, tutti i valori interi di grandi dimensioni sono rappresentati come stringhe per impostazione predefinita.
