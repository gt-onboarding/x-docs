<div id="overview">
  ## Panoramica
</div>

Se disponi di App esistenti, puoi visualizzarle, modificarle o eliminarle tramite la [pagina App del portale sviluppatori](https://developer.x.com/en/portal/projects-and-apps).

Per accedere all’API X e all’API X Ads è necessario un set di credenziali di [autenticazione](/it/resources/fundamentals/authentication), note anche come chiavi e token, da inviare con ogni richiesta. Queste credenziali possono assumere forme diverse a seconda del tipo di autenticazione richiesto dallo specifico endpoint che stai utilizzando.

Di seguito sono riportate le diverse credenziali che puoi generare nella tua App e come utilizzarle:

- **[Chiave API e Secret](/it/resources/fundamentals/authentication#api-key-and-secret):** In pratica, il nome utente e la password della tua App. Le userai per autenticare le richieste che richiedono [OAuth 1.0a User Context](/it/resources/fundamentals/authentication#oauth-1-0a-2) o per generare altri token, come i token di accesso utente o il token di accesso dell’App.

- **[Access Token e Secret](/it/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow):** In generale, gli Access Token rappresentano l’utente per conto del quale stai effettuando la richiesta. Quelli che puoi generare tramite il portale sviluppatori rappresentano l’utente proprietario dell’App. Li userai per autenticare le richieste che richiedono [OAuth 1.0a User Context](/it/resources/fundamentals/authentication#oauth-1-0a-2). Se desideri effettuare richieste per conto di un altro utente, dovrai utilizzare il flusso OAuth a 3 passaggi affinché ti autorizzi.

- **[Client ID e Client Secret](/it/resources/fundamentals/authentication#oauth-2-0):** Queste credenziali vengono utilizzate per ottenere un token di accesso utente con autenticazione OAuth 2.0. In modo simile a OAuth 1.0a, i token di accesso utente vengono utilizzati per autenticare richieste che forniscono informazioni private dell’account utente o eseguono azioni per conto di un altro account, ma con ambiti granulari per un controllo più fine su quali permessi l’applicazione client ha sull’utente.

- **[App only Access Token](/it/resources/fundamentals/authentication#bearer-token-also-known-as-app-only):** Userai questo token quando effettui richieste a endpoint che rispondono con informazioni pubblicamente disponibili su X.

Oltre a generare le chiavi e i token necessari per effettuare richieste all’API X, potrai anche impostare le [autorizzazioni](/it/resources/fundamentals/developer-apps#app-permissions) di accesso, documentare il caso d’uso o lo scopo dell’App, definire un [URL di callback](/it/resources/fundamentals/developer-apps#callback-urls) e modificare altre impostazioni relative all’ambiente di sviluppo della tua App all’interno della [dashboard di gestione](https://developer.x.com/en/portal/projects-and-apps).

<div id="apps-and-projects">
  ### App e Progetti
</div>

Puoi utilizzare le App e i [Progetti](/it/resources/fundamentals/projects) per organizzare il tuo lavoro con la Piattaforma per sviluppatori X in base ai casi d’uso. Ogni Progetto può includere un’App con il piano Free dell’API X, fino a due App con il piano Basic e fino a tre App con il piano Pro.

Se desideri accedere ai nuovi endpoint dell’[API X v2](/it/x-api/introduction), dovrai utilizzare chiavi e token di un’App associata a un Progetto.

Se hai App create prima del lancio dei Progetti, saranno visibili nella sezione denominata “Standalone Apps”. Le Standalone Apps sono App al di fuori della struttura dei Progetti. Se associ una Standalone App a un Progetto, potrà quindi effettuare richieste agli endpoint v2.

<div id="developer-portal-dashboard">
  ### Dashboard del portale sviluppatori
</div>

Puoi [accedere alla dashboard](https://developer.x.com/content/developer-twitter/en/apps) per gestire le App associate al tuo account. Per saperne di più, visita la nostra pagina di documentazione sul [portale sviluppatori](/it/resources/fundamentals/developer-portal). La dashboard consente agli sviluppatori di eseguire rapidamente le seguenti attività:

- Visualizzare le App standalone esistenti e il relativo App ID.
- Creare un nuovo Progetto, un’App o un’App standalone.
- Eliminare un Progetto o un’App non utilizzati.
- Rivedere o aggiornare le impostazioni di una specifica App, inclusi nome, descrizione, sito web, URL di callback e [autorizzazioni](/it/resources/fundamentals/developer-apps#app-permissions).
- Rigenerare le credenziali specifiche dell’App, come API Key & Secret, App Access Token e gli Access Token utente del proprietario dell’App.

<div id="signing-up-for-access">
  ### Registrazione per l’accesso
</div>

Se disponi di App X esistenti, puoi visualizzare e modificare le tue App tramite la [dashboard delle App](https://developer.x.com/content/developer-twitter/en/portal/projects-and-apps) di X, se hai effettuato l’accesso al tuo account X su developer.x.com. Tieni presente che **non** è necessario creare un nuovo account per gestire tutte le App precedentemente create su developer.x.com.

Se devi creare una nuova App X, devi prima [registrarti per un account sviluppatore](https://developer.x.com/en/portal/petition/essential/basic-info). Se fai parte di un [account del team](/it/resources/fundamentals/developer-portal#team-management), devi avere il ruolo di amministratore per creare una nuova App.

<div id="automated-account-labeling-for-bots">
  ### Etichetta “Account automatizzato” per i bot
</div>

Puoi aggiungere l’etichetta “Account automatizzato” ai tuoi account bot per far sapere agli utenti su X che il tuo bot è un account automatizzato. Questi bot eseguono azioni programmate tramite l’API X. Quando aggiungi l’etichetta “Account automatizzato” al tuo bot, rafforzi la fiducia del tuo pubblico, legittimi il tuo account e ti distingui dai bot di spam. Questo aiuta le persone su X a comprendere meglio lo scopo del tuo account quando interagiscono con il tuo bot.

Per applicare un’etichetta al tuo account bot, segui questi passaggi:

1. Vai alle impostazioni del tuo account
2. Seleziona “Il tuo account”
3. Seleziona “Automazione”
4. Seleziona “Gestione account”
5. Quindi seleziona l’account X che esegue il tuo account bot
6. Inserisci la tua password per accedere
7. Infine, dovresti vedere la conferma che l’etichetta è stata applicata al tuo account.

  ## Gestione delle App
</div>

<div id="introduction">
  ### Introduzione
</div>

La [dashboard X App](https://developer.x.com/en/portal/projects-and-apps) consente agli sviluppatori di svolgere in modo rapido e semplice le seguenti attività:

- Visualizzare le App e i [Progetti](/it/resources/fundamentals/projects) esistenti e il relativo App ID.
- Creare un nuovo Progetto o un'App standalone.
- Eliminare un Progetto, un'App o un'App standalone.
- Aprire le impostazioni di una specifica App facendo clic sull'App. All'interno delle impostazioni è possibile visualizzare i dettagli dell'App, le chiavi e i token e le autorizzazioni.
- Aggiornare le impostazioni di autenticazione utente della propria App per utilizzare OAuth 1.0a o OAuth 2.0.

<Note>
  **Nota:**

  Le chiavi e i token dell'App non sono più visibili nel Developer Portal e devono essere salvati in modo sicuro al momento della generazione. Si consiglia di utilizzare un gestore di password per archiviare le chiavi e i token.

  È possibile rivelare un suggerimento sulla chiave API per aiutare ad associare la credenziale all'App corrispondente.
</Note>

<div id="app-settings">
  ### Impostazioni app
</div>

<div id="app-details">
  #### Dettagli dell'App
</div>

Qui puoi modificare l’icona dell’App, il nome dell’App, la descrizione dell’App, l’URL del tuo sito web, [callback URLs/redirect URIs](/it/resources/fundamentals/developer-apps#callback-urls), l’URL dei termini di servizio, l’URL dell’informativa sulla privacy, il nome dell’organizzazione, l’URL dell’organizzazione e lo scopo o caso d’uso dell’App.

OAuth 2.0 e OAuth 1.0a sono metodi di autenticazione che consentono agli utenti di accedere alla tua App con X. Consentono inoltre alla tua App di effettuare richieste specifiche per conto degli utenti autenticati. Puoi attivare uno o entrambi i metodi per la tua App.

È importante mantenere aggiornate queste informazioni. Il nome della tua App e l’URL del sito web verranno mostrati come origine nei metadati di qualsiasi Tweet creato in modo programmatico dalla tua applicazione. Se modifichi il caso d’uso di un’App X, assicurati di aggiornarlo in queste impostazioni per garantire la conformità ai [Developer Terms](https://developer.x.com/content/developer-twitter/en/developer-terms/agreement-and-policy).

Se la tua applicazione presenta un tag “suspended”, significa che la tua app viola uno o più [developer terms](https://developer.x.com/en/developer-terms.html) di X, come le nostre [automation rules](https://help.x.com/en/rules-and-policies/twitter-automation). Il team di policy della piattaforma per sviluppatori comunicherà con gli sviluppatori tramite l’indirizzo email associato all’account X del proprietario dell’App; per verificarlo, consulta le [impostazioni del tuo account X](https://x.com/settings/your_twitter_data/account). Le email di notifica relative alle sospensioni verranno inviate a questo indirizzo con un oggetto simile a “Application suspension notice” e conterranno indicazioni specifiche su cosa fare. Per collaborare con il team di X nella gestione delle sospensioni, controlla la tua email per istruzioni dettagliate oppure utilizza il nostro [platform help form](https://help.x.com/forms/platform).

<div id="keys-and-tokens">
  #### Chiavi e token
</div>

All'interno di un'App in un Progetto o tramite un'App autonoma puoi visualizzare, rigenerare o revocare i seguenti token:

- [API Key (Consumer Key) e API Secret (Consumer Secret)](/it/resources/fundamentals/authentication#api-key-and-secret)
- [App Access Token](/it/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)
- [User Access Token e Access Token Secret](/it/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) -
  L'Access Token e l'Access Token Secret disponibili nel portale sviluppatori si riferiscono all'utente proprietario dell'App.

**Nota bene** - Se desideri effettuare richieste per conto di un utente diverso (cioè non l'utente proprietario dell'App), dovrai utilizzare il [flusso OAuth 1.0a a 3 leg](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens) oppure il flusso OAuth 2.0 [Authorization Code con PKCE](/it/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) per generare un set di User Access Token. Utilizzerai quindi questi token specifici dell’utente nella tua richiesta all’API.

<div id="user-authentication-settings">
  #### Impostazioni di autenticazione utente
</div>

Puoi configurare l’autenticazione della tua App come OAuth 1.0a oppure OAuth 2.0. OAuth 2.0 può essere utilizzato solo con l’API X v2. OAuth 2.0 consente di scegliere ambiti di autorizzazione specifici e granulari, che forniscono permessi puntuali per conto dell’utente. OAuth 1.0a può essere utilizzato con l’API X v1.1 e v2 e adotta un’autorizzazione più ampia con ambiti meno dettagliati.

<div id="oauth-10a-application-user-permissions">
  #### Autorizzazioni OAuth 1.0a applicazione-utente
</div>

Se sei il proprietario dell’App, puoi modificare le autorizzazioni dell’App (sola lettura; lettura e scrittura; oppure lettura, scrittura e messaggi diretti). In questo modo controlli a quali risorse ed eventi hai accesso tramite le API X (nota: alcune risorse richiedono autorizzazioni aggiuntive direttamente da X).

Puoi anche abilitare o disabilitare su questa pagina la possibilità per le tue App di richiedere gli indirizzi email degli utenti (sono necessari un URL dei Termini di servizio e un URL dell’Informativa sulla privacy nella pagina “Dettagli dell’App”).

<div id="oauth-20-type-of-app">
  #### Tipo di App OAuth 2.0
</div>

Se selezioni OAuth 2.0 come metodo di autenticazione utente, dovrai scegliere il tipo di App che stai creando. Le opzioni sono Native App, Single Page App, Web App, Automated App o bot. Le Native App e le Single Page App sono client pubblici, mentre le Web App e le Automated App o i bot sono client confidenziali.

I client confidenziali si autenticano in modo sicuro con il server di autorizzazione e proteggono il tuo client secret. I client pubblici sono applicazioni che di solito vengono eseguite in un browser o su un dispositivo mobile e non possono utilizzare i tuoi client secret. Se selezioni un tipo di App che è un client confidenziale, ti verrà fornito un client secret.

<div id="app-permissions">
  ## Autorizzazioni dell’app
</div>

<div id="oauth-10a-app-permissions">
  ### Autorizzazioni dell’app OAuth 1.0a
</div>

Le autorizzazioni dell’app descrivono il livello di accesso per l’autenticazione applicazione-utente OAuth 1.0a. Le autorizzazioni dell’app si configurano per singola applicazione nelle impostazioni della tua [X App](/it/resources/fundamentals/developer-apps).

Sono disponibili tre livelli di autorizzazione:

1. Sola lettura
2. Lettura e scrittura
3. Lettura, scrittura e accesso ai Messaggi diretti

Esiste un’autorizzazione aggiuntiva per richiedere la visibilità dell’indirizzo email di un utente: può essere combinata con uno qualsiasi dei tre livelli elencati sopra.

Se un livello di autorizzazione viene modificato, tutti i token utente già emessi per quella X App devono essere revocati e gli utenti devono ri-autorizzare l’app affinché il token erediti le autorizzazioni aggiornate.

È buona prassi richiedere solo il livello minimo di accesso ai dati dell’account utente necessario all’applicazione o al servizio.

<div id="read-only">
  ### Sola lettura
</div>

Questo livello di autorizzazione consente l’accesso in lettura alle risorse di X, inclusi (ad esempio) i post di un utente, la home timeline e le informazioni del profilo. Non consente di leggere i messaggi diretti di un utente e non consente di aggiornare alcun elemento o oggetto.

<div id="read-and-write">
  ### Lettura e scrittura
</div>

Questo livello di autorizzazione consente l’accesso in lettura e scrittura alle risorse di X. Oltre alla lettura, permette di pubblicare post, seguire utenti e aggiornare elementi delle informazioni del profilo di un utente. Consente inoltre di nascondere le risposte per conto dell’utente autenticato. Questo livello di autorizzazione non consente alcun accesso ai Messaggi Diretti (lettura, scrittura o eliminazione incluse).

<div id="read-write-and-access-direct-messages">
  ### Lettura, scrittura e accesso ai Messaggi Diretti
</div>

Questo livello di autorizzazioni include l’accesso a tutto quanto sopra e aggiunge la possibilità di leggere, scrivere ed eliminare i Messaggi Diretti per conto di un utente.

- [POST /2/dm\_conversations/:dm\_conversation\_id/messages](/it/x-api/direct-messages/send-a-new-message-to-a-dm-conversation)
- [POST /2/dm\_conversations/](/it/x-api/direct-messages/create-a-new-dm-conversation)
- [POST /2/dm\_conversations/with/:participant\_id/messages](/it/x-api/direct-messages/send-a-new-message-to-a-user)
- [GET /2/dm\_conversations/with/:user\_id/dm\_events](/it/x-api/direct-messages/get-dm-events-for-a-dm-conversation)
- [GET /2/dm\_conversations/:dm\_conversation\_id/dm\_events](/it/x-api/direct-messages/get-dm-events-for-a-dm-conversation-1)
- [GET /2/dm\_events](/it/x-api/direct-messages/get-recent-dm-events)

<div id="determining-permissions">
  ### Determinare le autorizzazioni
</div>

Tutte le richieste API autenticate restituiscono l’intestazione `x-access-level` nella risposta HTTP. Il valore dell’intestazione indica il livello di autorizzazione attualmente in uso. I valori possibili sono read, read-write e read-write-directmessages.

<div id="callback-urls">
  ## URL di callback
</div>

I metodi di autenticazione OAuth 1.0a User Context e OAuth 2.0 Authorization Code con PKCE consentono agli sviluppatori di effettuare richieste per conto di diversi utenti di X che hanno completato uno specifico flusso di accesso. Attualmente, ci sono due flussi che puoi utilizzare per consentire agli utenti di autorizzare la tua applicazione:

- [Flusso di authorization code OAuth 2.0 con PKCE](/it/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2)
- [Flusso OAuth 1.0a a 3 step](/it/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) (e, separatamente, il [flusso Accedi con X](/it/resources/fundamentals/authentication#log-in-with-x))

Quando gli utenti seguono questi flussi, hanno bisogno di una pagina web o di una destinazione a cui essere reindirizzati dopo aver effettuato correttamente l’accesso e aver fornito l’autorizzazione all’App dello sviluppatore. Questa pagina web o destinazione successiva è chiamata URL di callback.

Quando si configurano questi flussi per i potenziali utenti, gli sviluppatori devono includere un URL di callback nelle richieste agli endpoint di autenticazione che compongono i flussi menzionati in precedenza. Ad esempio, gli sviluppatori che utilizzano OAuth 1.0a User Context devono passare il parametro `callback_url` quando effettuano una richiesta all’endpoint [GET oauth/request\_token](/it/resources/fundamentals/authentication#post-oauth-request-token). Allo stesso modo, gli sviluppatori che utilizzano OAuth 2.0 Authorization Code con PKCE devono passare il parametro `redirect_uri` nella richiesta all’endpoint GET oauth2/authorize.

Oltre a utilizzare questi parametri, lo sviluppatore deve anche assicurarsi che l’URL di callback sia stato aggiunto alla allowlist degli URL di callback della propria [App](/it/resources/fundamentals/developer-apps), che si trova nella pagina delle impostazioni dell’App del [portale sviluppatori](/it/resources/fundamentals/developer-portal).

Se configurato correttamente, nell’ambito di questi flussi gli sviluppatori verranno reindirizzati all’URL di callback dopo aver eseguito correttamente l’accesso a X.

<div id="things-to-keep-in-mind">
  ### Aspetti da considerare
</div>

Quando configuri i tuoi URL di callback, ci sono alcuni aspetti da considerare:

**Codifica HTTP dei parametri di query**

Poiché stai passando l’URL di callback come parametro di query tramite `callback_url` o `redirect_uri`, assicurati di applicare la codifica HTTP all’URL.

**Limiti degli URL di callback**

Esiste un limite fisso di 10 URL di callback nella dashboard di X Apps. L’URL di callback deve sempre corrispondere esattamente tra l’URL autorizzato che aggiungi alla dashboard delle App e il parametro che inserisci nel flusso di autorizzazione.

Se desideri includere dati specifici della richiesta nell’URL di callback, puoi utilizzare il parametro `state` per memorizzare i dati che verranno inclusi dopo il reindirizzamento dell’utente. Puoi codificare i dati direttamente nel parametro `state` oppure usare il parametro come ID di sessione per mantenere lo stato sul server.

**Non usare localhost come URL di callback**
Invece di usare localhost, utilizza un host personalizzato in locale oppure http(s)://127.0.0.1

**URL con protocollo personalizzato**
Se desideri sfruttare il deep linking su mobile, puoi utilizzare URL con protocollo personalizzato con una parte di path e di dominio, come twitter://callback/path. Tuttavia, esiste un elenco di protocolli non consentiti che devi evitare. Puoi consultare l’elenco dei protocolli non consentiti qui sotto.

<div id="error-example">
  ### Esempio di errore
</div>

Se utilizzi un URL di callback che non è stato correttamente aggiunto alle impostazioni della tua App nel portale per sviluppatori, riceverai il seguente messaggio di errore:

**OAuth 1.0a**

```json HTTP 403 - Forbidden

{
  "errors": [
    {
      "code":415,"message":"Callback URL not approved for this client application. Approved callback URLs can be adjusted in your application settings."
    }
  ]
}
```

OR

```xml
<?xml version="1.0" encoding="UTF-8"?>
<hash>
<error>Callback URL not approved for this client application. Approved callback URLs can be adjusted in your application settings</error>
<request>/oauth/request_token</request>
</hash>
```

**OAuth 2.0**

```json HTTP 400

{
  "error": "invalid_request",
  "error_description": "Value passed for the redirect uri did not match the uri of the authorization code."
}
```

**Risoluzione dei problemi**

Se ricevi un errore, assicurati che l'URL di callback che stai utilizzando nelle richieste di autenticazione sia codificato in URL (URL-encoded) e che corrisponda a un URL di callback aggiunto alla allowlist per l'App le cui chiavi e token stai utilizzando nella richiesta.
