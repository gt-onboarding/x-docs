<div id="typical-response-structure">
  ## Struttura tipica della risposta
</div>

Le risposte con esito positivo sono indicate da un codice HTTP della serie 200 e da un payload JSON contenente gli oggetti richiesti, creati, modificati o eliminati, insieme alla rappresentazione dell’interpretazione che il server dà alla tua richiesta.

Se invii una richiesta andata a buon fine, nella risposta riceverai un nodo `request` che riproduce la tua richiesta.

_Esempio:_ [GET accounts/abcdefg/campaigns?with\_deleted=true](/it/x-ads-api/campaign-management#campaigns)

```json
{
  /* the data of your response... */,
  "request": {
    "params": {
      "account_id": "abcdefg",
      "with_deleted": "true"
    }
  }
}
```

Il campo `data` nelle risposte JSON contiene gli oggetti specifici associati alla risorsa utilizzata. Il nodo `data` sarà un array JSON quando la risposta può includere uno o più risultati. Sarà invece un oggetto JSON quando è possibile un solo risultato. In alcuni rari casi, potresti vedere una risposta che normalmente includerebbe una collezione restituita come mappa (hashmap). In tal caso, considera la singola mappa come un oggetto dello stesso tipo specificato nel campo `type`.

<div id="error-response-structure">
  ## Struttura della risposta di errore
</div>

Le risposte di errore vengono restituite con un codice HTTP non appartenente alla serie 200. Di solito è presente una risposta JSON, ma alcuni errori restituiscono body di tipo diverso. In queste circostanze, in cui la struttura della risposta non può essere analizzata, considera prioritario il significato del codice HTTP. Ad esempio, potresti occasionalmente vedere un HTTP 404 insieme a una risposta HTML. In questo caso, è ragionevole presumere che il contenuto non sia stato trovato (HTTP 404 significa “Not Found”).

Le risposte di errore tipiche seguono una struttura simile a quella delle risposte con esito positivo. La natura dell’errore è comunicata in un nodo `errors` della risposta. Il nodo `errors/code` indica un codice di errore costante in CAPS\_CASE che puoi usare a livello programmatico per prendere decisioni di risoluzione. Il nodo `errors/message` indica una descrizione (di solito) leggibile da un essere umano dell’errore, in inglese. Possono essere presenti campi aggiuntivi per fornire dettagli più specifici sull’errore.

```json
{
  "errors": [
    {
      "parameter": "start_time",
      "details": "invalid date",
      "code": "INVALID_PARAMETER",
      "value": "",
      "message": "Expected time, got \"\" for start_time"
    }
  ],
  "request": {
    "params": {
      "account_id": "hkk5"
    }
  }
}
```

Nell’esempio sopra, è stata effettuata una richiesta a un endpoint di analytics con un valore non valido per il parametro `start_time`. Il valore di `errors/code` per le richieste con parametri non validi è `INVALID_PARAMETER`.
