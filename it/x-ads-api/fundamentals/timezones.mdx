<div id="timezones-accounts-and-billing">
  ## Fusi orari, account e fatturazione
</div>

I valori datetime sono sempre restituiti in UTC (come indicato dalla Z alla fine del valore). I datetime possono essere specificati in qualsiasi fuso orario in un comando POST o PUT utilizzando il formato standard ISO 8601 per il fuso orario. L’ora è rappresentata utilizzando un sottoinsieme di [ISO 8601](http://en.wikipedia.org/wiki/ISO_8601). Più precisamente, la stringa strptime per il nostro formato data è `%Y-%m-%dT%l:%M:%S%z`. Il fuso orario dell’account dell’inserzionista determina l’ora effettiva in cui i numeri di fatturazione ufficiali vengono congelati.

Quando si interroga l’API a livello di account ([GET accounts](/it/x-ads-api/campaign-management#accounts)), si ottengono informazioni sul fuso orario simili a queste:

```json
    {
      "request": {
        "params": {
          "account_id": "18ce54d4x5t"
        }
      },
      "data": {
        "name": "API McTestface",
        "business_name": null,
        "timezone": "America/Los_Angeles",
        "timezone_switch_at": "2016-07-21T07:00:00Z",
        "id": "18ce54d4x5t",
        "created_at": "2016-07-21T22:42:09Z",
        "salt": "54cb7b5a34183f77d82dd6d0f4329691",
        "updated_at": "2017-09-09T06:42:14Z",
        "business_id": null,
        "approval_status": "ACCEPTED",
        "deleted": false
      }
    }
```

Sono inclusi due valori relativi al fuso orario: `timezone` (consulta [i fusi orari su Wikipedia](https://en.wikipedia.org/wiki/Tz_database#Names_of_time_zones) per una panoramica) e `timezone_switch_at`. Nota che `timezone_switch_at` è fornito nel fuso orario UTC (+00:00), ma rappresenterà sempre la mezzanotte nel fuso orario indicato.

Il fuso orario dell’inserzionista non è modificabile tramite l’API. Questo attributo è definito a livello contrattuale/fatturazione dal responsabile dell’account dell’inserzionista presso X.

Tieni presente il valore `timezone_switch_at` quando crei report e interroghi i nostri endpoint di analisi, poiché si verificherà un intervallo nel giorno in cui l’account passa dal fuso orario America/Los\_Angeles al nuovo valore locale.

<div id="specifying-datetime-values-with-timezone">
  ### Specificare valori datetime con fuso orario
</div>

I valori datetime sono sempre restituiti in UTC (come indicato dalla `Z` alla fine del valore). Le date/ore possono essere specificate in qualsiasi fuso orario in un comando POST o PUT utilizzando il formato dello [standard ISO 8601](http://en.wikipedia.org/wiki/ISO_8601) per il fuso orario. Ad esempio, `2017-07-10T08:00:00-0800` è un valore di input valido e verrà automaticamente convertito nel valore UTC `2017-07-10T16:00:00Z`.

Quando si utilizzano gli endpoint di analytics con `granularity` pari a `DAY` o `TOTAL`, il valore `start_time` deve essere impostato a mezzanotte del giorno desiderato nel fuso orario locale del titolare dell’account. L’offset del fuso orario da utilizzare è quello del giorno corrente, non quello del giorno in oggetto. Ad esempio, per un account pubblicitario in America/Los\_Angeles durante l’ora legale del Pacifico, l’offset UTC è -0700. Pertanto, in una richiesta di analytics, l’orario dovrebbe essere specificato come: `start_time=2017-05-21T07:00:00Z` oppure `start_time=2017-05-21T00:00:00-0700`. Se l’account pubblicitario fosse in Asia/Tokyo, dove l’offset è sempre +09:00, i valori andrebbero specificati come: `start_time=2017-05-20T15:00:00Z` oppure `start_time=2017-05-21T00:00:00+0900`.

  ### Formati di offset UTC accettati
</div>

Supportati: `Z`, `-HHMM`, `+HHMM`, `-HH:MM`, `+HH:MM`, `-HH`, `+HH`
