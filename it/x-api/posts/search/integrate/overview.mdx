<div id="authentication">
  #### Autenticazione
</div>

Tutti gli endpoint dell’API X v2 richiedono di autenticare le richieste con un set di credenziali, note anche come chiavi e token. Puoi utilizzare OAuth 1.0a User Context, OAuth 2.0 App-Only o OAuth 2.0 Authorization Code con PKCE per autenticare le richieste all’endpoint di ricerca recente. Devi utilizzare OAuth 2.0 App-Only quando usi l’endpoint di ricerca dell’archivio completo.

[OAuth 2.0 App-Only](/it/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3) richiede semplicemente di includere un [OAuth 2.0 App Access Token](/it/resources/fundamentals/authentication#bearer-token-also-known-as-app-only) nella richiesta. Puoi generare un App Access Token direttamente all’interno di un’App per sviluppatori oppure generarne uno utilizzando l’endpoint [POST oauth2/token](/it/resources/fundamentals/authentication#post-oauth2-token).

[OAuth 1.0a User Context](/it/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) richiede di utilizzare le tue API Keys, gli Access Tokens dell’utente e alcuni altri parametri per [creare un’intestazione di autorizzazione](https://developer-staging.x.com/resources/fundamentals/authentication/authorizing-a-request), che poi includerai nella richiesta. Gli Access Tokens devono essere associati all’utente per conto del quale stai effettuando la richiesta. Se desideri generare un set di Access Tokens per un altro utente, quest’ultimo deve autorizzare la tua App utilizzando il [flusso OAuth a 3 passaggi](https://developer-staging.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens).

Tieni presente che OAuth 1.0a può essere complesso da utilizzare. Se non hai familiarità con questo metodo di autenticazione, ti consigliamo di usare una [libreria](/it/x-api/tools-and-libraries/overview), uno strumento come Postman o OAuth 2.0 per autenticare le richieste. Se desideri recuperare un Post o metriche private da questi endpoint, dovrai utilizzare OAuth 1.0a User Context oppure OAuth 2.0 Authorization Code con PKCE, che garantiscono le autorizzazioni appropriate dall’utente proprietario di quel contenuto.

[OAuth 2.0 Authorization Code con PKCE](/it/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) consente un maggiore controllo sull’ambito di un’applicazione e flussi di autorizzazione tra più dispositivi. OAuth 2.0 permette di selezionare ambiti specifici e granulari che concedono permessi mirati per conto di un utente.

Per abilitare OAuth 2.0 nella tua App, devi attivarlo nelle impostazioni di autenticazione della tua App, disponibili nella sezione Impostazioni dell’App del portale per sviluppatori.

<Note>
  **Nota bene**

  Se richiedi i seguenti [campi](/it/x-api/fundamentals/fields), è richiesto OAuth 1.0a User Context oppure OAuth 2.0 Authorization Code:

  - tweet.fields.non\_public\_metrics
  - tweet.fields.promoted\_metrics
  - tweet.fields.organic\_metrics
  - media.fields.non\_public\_metrics
  - media.fields.promoted\_metrics
  - media.fields.organic\_metrics
</Note>

####

Portale sviluppatori, Progetti e App per sviluppatori

Per utilizzare qualsiasi endpoint dell’API X v2, devi esserti [registrato per un account sviluppatore](https://developer.x.com/en/portal/petition/essential/basic-info), aver configurato un Progetto all’interno di tale account e creato un’App per sviluppatori all’interno di quel Progetto. Le chiavi e i token di quell’App per sviluppatori funzioneranno con questi endpoint di ricerca.

Puoi utilizzare le chiavi e i token di un Progetto con qualsiasi [livello di accesso](/it/x-api/getting-started/about-x-api) per inviare richieste all’endpoint di ricerca recente. Tuttavia, dovrai utilizzare un Progetto con livello di accesso Pro o Enterprise per inviare richieste all’endpoint di ricerca dell’archivio completo. Se disponi dell’[accesso Enterprise](/it/x-api/getting-started/getting-access), avrai accesso a funzionalità aggiuntive, tra cui operatori aggiuntivi e query più lunghe.
 

<div id="rate-limits">
  #### Limiti di velocità
</div>

Ogni giorno, molte migliaia di sviluppatori inviano richieste all’API X. Per gestire il volume, su ciascun endpoint vengono applicati [limiti di velocità](/it/x-api/fundamentals/rate-limits) che restringono il numero di richieste che ogni sviluppatore può effettuare per conto di un’app o di un utente autenticato.

A seconda del metodo di autenticazione utilizzato si applicano limiti di velocità diversi. I limiti a livello di app si applicano a un’app che effettua richieste utilizzando OAuth 2.0 App-Only, mentre i limiti a livello di utente si applicano alle richieste effettuate per conto dello specifico utente autorizzante utilizzando OAuth 1.0a User Context o OAuth 2.0 Authorization Code con PKCE. Entrambi questi limiti si basano sulla frequenza delle richieste all’interno di una finestra di 15 minuti.

Ad esempio, un’app che utilizza l’autenticazione OAuth 2.0 App-Only per effettuare richieste all’endpoint di ricerca recente può effettuare 450 richieste (incluse quelle di paginazione) in un intervallo di 15 minuti. La stessa app, nello stesso intervallo di 15 minuti e con due diversi utenti autenticati (utilizzando OAuth 1.0a User Context o OAuth 2.0 Authorization Code con PKCE), può effettuare fino a 180 richieste (incluse quelle di paginazione) all’endpoint di ricerca recente per ciascun utente autenticato.
 

<div id="fields-and-expansions">
  #### Campi ed espansioni
</div>

L’API X v2 consente di selezionare con precisione quali dati ottenere dall’API utilizzando [campi](/it/x-api/fundamentals/fields) ed [espansioni](/it/x-api/fundamentals/expansions). Il parametro expansions consente di includere (espandere) gli oggetti referenziati nel payload. Ad esempio, questo endpoint permette di richiedere oggetti come poll, place, media e altri utilizzando il parametro expansions.

I parametri fields consentono di scegliere esattamente quali campi, all’interno dei diversi oggetti dati, si desidera ricevere. Per impostazione predefinita, l’oggetto Post principale restituito da questi endpoint include i campi id e text (oltre a edit\_history\_tweet\_ids per i Post creati dopo l’introduzione di quella funzionalità). Per ricevere campi aggiuntivi come author\_id o public\_metrics, è necessario richiederli esplicitamente tramite i parametri fields. Alcuni campi importanti da considerare per l’integrazione sono i dati dei sondaggi, le [metriche](/it/x-api/fundamentals/metrics), le [annotazioni dei Post](/it/x-api/fundamentals/post-annotations) e il campo [conversation ID](/it/x-api/fundamentals/conversation-id).

Abbiamo aggiunto una guida su come [utilizzare campi ed espansioni insieme](/it/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) nel nostro [data dictionary dell’API X v2](/it/x-api/fundamentals/data-dictionary).
 

<div id="metrics">
  #### Metriche
</div>

Gli endpoint dell’API X v2 permettono di richiedere metriche direttamente dagli oggetti restituiti, a condizione che nella richiesta si includano i [campi](/it/x-api/fundamentals/fields) appropriati.

Esistono alcune limitazioni relative alle metriche dei Post di cui è opportuno essere consapevoli, in particolare per quanto riguarda la privacy degli utenti e i seguenti campi di risposta:

- tweet.fields.non\_public\_metrics
- tweet.fields.promoted\_metrics
- tweet.fields.organic\_metrics
- media.fields.non\_public\_metrics
- media.fields.promoted\_metrics
- media.fields.organic\_metrics

I campi indicati includono metriche private; ciò significa che, quando si utilizza l’endpoint di ricerca recente, è necessario essere autorizzati dal publisher del Post per recuperarli per suo conto, utilizzando [OAuth 1.0a User Context](/it/resources/fundamentals/authentication). Poiché questo metodo di autenticazione è utilizzabile solo con la ricerca recente, non sarà possibile recuperare tali metriche tramite l’endpoint di ricerca dell’archivio completo.

Ad esempio, per ricevere le `non_public_metrics` dei Post dell’utente con ID 1234, è necessario includere nella richiesta i token di accesso associati a tale utente. È possibile far autorizzare la propria app agli utenti e ottenere un set di token di accesso associati a loro utilizzando il [flusso OAuth a 3 step](/it/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow).

Tutte le `non_public_metrics`, le `organic_metrics` e le `promoted_metrics` sono disponibili solo per i Post creati negli ultimi 30 giorni. Ciò significa che, quando si richiedono i campi indicati, i risultati si adegueranno automaticamente per includere solo i Post degli ultimi 30 giorni.

Se si richiedono tali campi, verranno restituiti solo i Post creati dall’utente autenticato; per tutti gli altri Post verrà restituito un messaggio di errore.
 

<div id="building-search-queries">
  #### Creazione di query di ricerca
</div>

La funzionalità principale di questi endpoint è l’uso di un’unica query di ricerca per filtrare i post che ti vengono recapitati. Queste query sono composte da operatori che fanno match su attributi del post e dell’utente, come parole chiave del messaggio, hashtag e URL. Gli operatori possono essere combinati in query con logica booleana e parentesi per affinare il comportamento di corrispondenza della query.

Puoi consultare la nostra guida su [come creare una query](/it/x-api/posts/search/integrate/build-a-query) per saperne di più.

<div id="pagination">
  #### Paginazione
</div>

Questi endpoint utilizzano la paginazione per restituire rapidamente le risposte. Quando i risultati superano il numero inviabile in un’unica risposta (fino a 100 post per la ricerca recente e 500 per la ricerca dell’archivio completo), è necessario paginare. Usa il parametro max\_results per definire quanti risultati restituire per pagina e il parametro pagination\_token per ottenere la pagina successiva di risultati. Per maggiori dettagli, consulta la nostra [guida alla paginazione](/it/x-api/fundamentals/pagination).

<div id="post-caps">
  #### Limiti di post
</div>

Gli endpoint di Search Posts sono soggetti a un limite sul numero di post che possono restituire in un determinato mese, indipendentemente dalla paginazione.

Indipendentemente dall’endpoint di ricerca utilizzato, i post restituiti verranno conteggiati ai fini dei [limiti di post](/it/x-api/fundamentals/post-cap) a livello di Progetto. L’utilizzo è visibile nel portale per sviluppatori e il “mese” inizia nel giorno di rinnovo dell’abbonamento indicato nella [dashboard del portale per sviluppatori](https://developer.x.com/en/portal/dashboard).

**Modifiche ai post**

I post idonei alle modifiche possono essere modificati fino a cinque volte nei 30 minuti successivi alla pubblicazione del post originale. Gli endpoint di ricerca forniranno sempre la versione più recente del post. Se richiedi solo post pubblicati 30 o più minuti fa, riceverai sempre la versione finale del post. Tuttavia, se hai un caso d’uso quasi in tempo reale e stai interrogando post pubblicati negli ultimi trenta minuti, tali post potrebbero essere stati modificati dopo che li hai ricevuti. Questi post possono essere “reidratati” con la ricerca o con l’endpoint Post Lookup per confermarne lo stato finale. Per saperne di più su come funzionano le modifiche ai post, consulta la pagina [Fondamenti delle modifiche ai post](/it/x-api/fundamentals/edit-posts).
