<div id="likes-lookup-standard-v11-compared-to-x-api-v2">
  ### Ricerca dei Mi piace: standard v1.1 rispetto all’API X v2
</div>

Se hai lavorato con l’endpoint standard v1.1 [GET favorites/list](https://developer.x.com/en/docs/x-api/v1/tweets/post-and-engage/api-reference/get-favorites-list), l’obiettivo di questa guida è aiutarti a comprendere somiglianze e differenze tra gli endpoint di ricerca dei Mi piace dello standard v1.1 e dell’API X v2.

Con la v2 abbiamo anche introdotto un nuovo endpoint per gli utenti che hanno messo Mi piace, che consente di ottenere informazioni sugli utenti a cui è piaciuto un post.

- **Somiglianze**
  - Autenticazione
  - Limiti di frequenza
- **Differenze**
  - URL degli endpoint

  - Limitazioni delle richieste

  - Requisiti per App e Progetti

  - Parametri della richiesta

  - Nuovo formato JSON

<div id="similarities">
  #### Somiglianze
</div>

**Autenticazione**

Sia gli endpoint per la ricerca dei Like dello standard v1.1 sia quelli dell’API X v2 utilizzano [OAuth 1.0a User Context](/it/resources/fundamentals/authentication#oauth-1-0a-2) o [OAuth 2.0 Bearer Token](/it/resources/fundamentals/authentication#bearer-token-also-known-as-app-only). Pertanto, se in precedenza utilizzavi gli endpoint standard v1.1 [GET favorites/list](https://developer.x.com/en/docs/x-api/v1/tweets/post-and-engage/api-reference/get-favorites-list), puoi continuare a utilizzare lo stesso metodo di autenticazione anche dopo la migrazione alla versione API X v2, se lo desideri.

A seconda della libreria/pacchetto di autenticazione che preferisci, l’autenticazione con Bearer Token è probabilmente il modo più semplice per iniziare e può essere impostata tramite una semplice intestazione di richiesta. Per sapere come generare un Bearer Token, consulta [questa guida su OAuth 2.0 Bearer Token](/it/resources/fundamentals/authentication#bearer-token-also-known-as-app-only).

**Limiti di utilizzo**

L’endpoint standard v1.1 GET favorites/list ha un limite di 75 richieste ogni 15 minuti per utente. L’endpoint corrispondente per i Post con Mi piace in v2 ha lo stesso limite. Tuttavia, questo endpoint v2 prevede anche un ulteriore limite di 75 richieste ogni 15 minuti per App.

<div id="differences">
  #### Differenze
</div>

**URL degli endpoint**

- Endpoint standard v1.1:
  - GET https://api.x.com/1.1/favorites/list.json
    (elenco di Post dell’utente specificato)
  - Non esiste un equivalente v1.1 all’endpoint v2 che restituisce gli utenti che hanno messo Mi piace
- Endpoint API X v2:
  - GET https://api.x.com/2/users/:id/liked\_tweets
    (elenco di Post a cui ha messo Mi piace l’utente con l’ID specificato)
  - GET https://api.x.com/2/tweets/:id/liking\_users
    (elenco di utenti che hanno messo Mi piace al Post con l’ID specificato)

**Limitazioni delle richieste**

L’endpoint v2 per i Post piaciuti consente di richiedere da 5 a 100 Post per richiesta, ma puoi recuperare tutti i Mi piace di un Post usando i token di paginazione. Anche l’endpoint v1.1 GET favorites/list consente di recuperare tutti i Mi piace dei Post, ma puoi richiedere da 20 a 200 Post per richiesta.

Per l’endpoint v2 degli utenti che hanno messo Mi piace, il limite è di 100 utenti per Post. 
 

**Requisiti per App e Progetti**

Gli endpoint dell’API X v2 richiedono l’uso di credenziali di una [developer App](/it/resources/fundamentals/developer-apps) associata a un [Project](/it/resources/fundamentals/developer-apps) quando autentichi le tue richieste. Tutti gli endpoint dell’API X v1.1 possono usare credenziali di App autonome o di App associate a un progetto.

**Parametri della richiesta**

I seguenti endpoint standard v1.1 accettavano due parametri di query (user\_id o screen\_name). L’API X v2 accetta solo l’ID utente numerico, che deve essere passato come parte del path dell’endpoint.

Una delle differenze più significative tra la versione standard v1.1 e l’API X v2 riguarda la selezione dei campi restituiti nel payload. Per gli endpoint standard, erano disponibili diversi parametri per specificare quali campi o insiemi di campi restituire nel payload, mentre l’API X v2 semplifica questi parametri in [fields](/it/x-api/fundamentals/fields) ed [expansions](/it/x-api/fundamentals/expansions). 
 

**Nuovo formato JSON**

L’API X v2 introduce nuovi design JSON per gli oggetti restituiti dalle API, inclusi gli oggetti [Post](/it/x-api/fundamentals/data-dictionary#tweet) e [user](/it/x-api/fundamentals/data-dictionary#user).

- A livello radice del JSON, gli endpoint standard restituiscono oggetti utente in un array statuses, mentre l’API X v2 restituisce un array data. 
- Invece di fare riferimento a “statuses” Retweeted e Quoted, il JSON dell’API X v2 fa riferimento a Retweeted e Quoted Tweets. Molti campi legacy e deprecati, come contributors e user.translator\_type, vengono rimossi. 
- Invece di usare sia favorites (nell’oggetto Post) sia favourites (nell’oggetto user), l’API X v2 usa il termine like. 
- X adotta la convenzione per cui i valori JSON senza valore (ad esempio null) non vengono inclusi nel payload. Gli attributi di Post e user sono inclusi solo se hanno valori non null.

Oltre alle modifiche apportate al nuovo formato JSON, abbiamo introdotto anche un nuovo set di campi per l’oggetto Post, tra cui:

- [conversation\_id](/it/x-api/fundamentals/conversation-id)
- Due nuovi campi di [Post annotations](/it/x-api/fundamentals/post-annotations), inclusi context ed entities
- Diversi nuovi campi di [metrics](/it/x-api/fundamentals/metrics)
