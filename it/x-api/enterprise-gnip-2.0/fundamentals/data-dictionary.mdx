<div id="introduction">
  ### Introduzione
</div>

`Enterprise`

I Post sono il blocco costitutivo fondamentale di tutto ciò che riguarda X. Tutte le API di X che restituiscono Post forniscono i dati codificati in JavaScript Object Notation (JSON). JSON si basa su coppie chiave-valore, con attributi denominati e valori associati. Gli oggetti Post recuperati dall’API includono l’“aggiornamento di stato” di un utente X, ma anche i Retweet, le risposte e i Tweet con citazione sono oggetti Post. Se un Post è correlato a un altro Post, come un Retweet, una risposta o un Tweet con citazione, ciascuno sarà identificato o incorporato nell’oggetto Post. Anche il Post più semplice, nel formato dati nativo di X, presenta oggetti JSON annidati per rappresentare altri attributi del Post, come l’autore, gli utenti menzionati, il luogo taggato, gli hashtag, i simboli cashtag, i contenuti multimediali o i link URL. Quando si lavora con i dati di X, questo è un concetto importante da comprendere. Il formato dei dati del Post che riceverai dall’API X dipende dal tipo di Post ricevuto, dall’API X che stai utilizzando e dalle impostazioni di formattazione.

Gli endpoint Enterprise che restituiscono oggetti Post sono stati aggiornati per fornire i metadati necessari a comprendere la cronologia delle modifiche del Post. Scopri di più su questi metadati nella pagina ["Modifica dei post" — Fondamenti](/it/x-api/fundamentals/edit-posts).

```JSON
{
	"created_at": "Fri Feb 14 19:00:55 +0000 2020",
	"id_str": "1228393702244134912",
	"text": "Che cosa ha scritto lo sviluppatore nel suo biglietto di San Valentino?\n  \nwhile(true) {\n    I = Love(You);  \n}",
	"entities": {
		"hashtags": [],
		"symbols": [],
		"user_mentions": [],
		"urls": []
	},
	"user": {
		"entities": {
			"url": {}
		}
	},
	"place": {}
}
```

<div id="available-data-formats">
  ### Formati di dati disponibili
</div>

> Nota: si raccomanda vivamente di utilizzare il formato [Enriched Native](/it/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) per le API di dati enterprise.
>
> - Il formato Enriched Native include tutti i nuovi metadati introdotti dal 2017, come i [metadati dei sondaggi](/it/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata), e metriche aggiuntive come reply\_count e quote\_count.
>
> - Il formato Activity Streams non è stato aggiornato con nuovi metadati o arricchimenti dalla [modifica dei caratteri](https://blog.x.com/official/en_us/topics/product/2017/Giving-you-more-characters-to-express-yourself.html) del 2017.

Le API di dati enterprise forniscono dati in due formati. Il formato enterprise più vicino al formato nativo standard v1.1 è Enriched Native. Il formato legacy dei dati enterprise è Activity Streams, originariamente implementato e utilizzato da Gnip come formato normalizzato per X e altri provider di dati social dell’epoca. Sebbene questo formato sia ancora disponibile, dal 2017 X ha investito in nuove funzionalità e sviluppi solo sul formato Enriched Native.

Il formato Enriched Native è esattamente ciò che suggerisce il nome: include gli oggetti nativi di X insieme ad arricchimenti aggiuntivi disponibili per i prodotti di dati enterprise, come i metadati di “unwinding” degli URL, i dati di geolocalizzazione del profilo, i metadati dei sondaggi e metriche di engagement aggiuntive.

- [Arricchimento delle URL espanse e migliorate](/it/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls)
- [Arricchimento delle regole di corrispondenza](/it/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#matching-rules)
- [Arricchimento dei metadati dei sondaggi](/it/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata)
- [Arricchimento della geolocalizzazione del profilo](/it/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo)

<div id="parsing-best-practices">
  ### Best practice di parsing
</div>

- Il JSON di X è codificato in UTF-8.
- I parser dovrebbero tollerare con facilità variazioni nell’ordine dei campi. Si deve presumere che il JSON dei post sia fornito come una mappa di dati non ordinata.
- I parser dovrebbero tollerare l’aggiunta di nuovi campi.
- I parser JSON devono tollerare i campi mancanti, poiché non tutti i campi compaiono in tutti i contesti.
- In generale è sicuro considerare un campo null, un insieme vuoto e l’assenza di un campo come equivalenti.

<div id="post-object">
  #### Oggetto Post
</div>

Quando si utilizzano prodotti di dati enterprise, noterai che gran parte del dizionario dei dati è simile al formato nativo dei dati dei Post, con alcuni metadati aggiuntivi. Il livello base del formato nativo arricchito utilizza molti degli stessi nomi di oggetti del [formato dati dell’API X v1.1](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview.html). L'oggetto Post ha un lungo elenco di attributi di “livello radice”, inclusi attributi fondamentali come `id`, `created_at` e `text`. Gli oggetti Post includono anche oggetti annidati come `user`, `entities` ed `extended_entities`. Gli oggetti Post includono inoltre altri [oggetti Post annidati](/it/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#nested-post-objects) come retweeted\_status, quoted\_status ed extended\_tweet. Il formato nativo arricchito include anche un oggetto matching\_rules.

<div id="nested-post-objects">
  #### Oggetti Post annidati
</div>

In diversi casi, un oggetto Post include altri oggetti annidati. Se lavori con oggetti annidati, quel payload JSON conterrà più oggetti Post e ciascun oggetto Post potrà contenere i propri oggetti. L’oggetto al livello radice conterrà informazioni sul tipo di azione eseguita, cioè se si tratta di un Retweet o di un Quote Tweet, e può anche contenere un oggetto che descrive il Post “originale” condiviso. I Post estesi includono un oggetto esteso annidato che supera i 140 caratteri, utilizzato per evitare breaking change quando è stato effettuato l’aggiornamento nel 2017. Ogni dizionario di oggetti annidati è descritto di seguito.

**Retweet**

I Retweet contengono sempre due oggetti Post. Il Post “originale” oggetto del Retweet è fornito in un oggetto "retweeted\_status". L’oggetto al livello radice incapsula il Retweet stesso, includendo un oggetto User per l’account che esegue l’azione di Retweet e l’orario del Retweet. Il Retweet è un’azione per condividere un Post con i propri follower e non è possibile aggiungere nuovi contenuti. Inoltre, non è possibile fornire una (nuova) posizione con un Retweet. Sebbene il Post “originale” possa essere stato geotaggato, gli oggetti "geo" e "place" del Retweet saranno sempre null.

Anche prima dell’introduzione dei Post estesi, l’oggetto "entities" al livello radice era in alcuni casi troncato e incompleto a causa della stringa "RT @username " aggiunta al messaggio del Post oggetto del Retweet. Nota che, se un Retweet viene a sua volta retwittato, "retweet\_status" punterà comunque al Post originale, il che significa che il Retweet intermedio non è incluso. Un comportamento simile si osserva quando si usa x.com per “visualizzare” un Retweet. Se copi l’ID del Post univoco assegnato all’azione di Retweet, viene visualizzato il Post originale.

Di seguito è riportata una struttura di esempio per un Retweet. Ancora una volta, quando si analizzano i Retweet, è fondamentale analizzare l’oggetto "retweeted\_status" per ottenere il messaggio del Post completo (originale) e i metadati delle entità.

```json
{
	"tweet": {
		"text": "RT @author original message",
		"user": {
			"screen_name": "Retweeter"
		},
		"retweeted_status": {
			"text": "original message",
			"user": {
				"screen_name": "OriginalTweeter"
			},
			"place": {},
			"entities": {},
			"extended_entities": {}
		}
	},
	"entities": {},
	"extended_entities": {}
}
```

<div id="quote-tweets">
  ##### Post con citazione
</div>

I Post con citazione sono simili ai Retweet, ma includono un nuovo messaggio. Questi nuovi messaggi possono contenere il proprio set di hashtag, link e altri metadati relativi alle “entities”. I Post con citazione possono anche includere informazioni sulla posizione condivise dall’utente che pubblica il Post con citazione, insieme a contenuti multimediali come GIF, video e foto.

I Post con citazione contengono almeno due oggetti Post e, in alcuni casi, tre. Il Post citato, che a sua volta può essere un Tweet citato, è fornito in un oggetto "quoted\_status". L’oggetto a livello radice incapsula il Post con citazione stesso, includendo un oggetto User per l’account che effettua l’azione di condivisione e l’ora del Post con citazione.

Nota: ora i Post con citazione possono includere foto, GIF o video, aggiunti tramite l’interfaccia utente di Post. Quando nel messaggio del Post con citazione sono presenti link a contenuti multimediali ospitati esternamente, l’"entities.urls" a livello radice li descriverà. I contenuti multimediali allegati ai Post con citazione appariranno nei metadati "extended\_entities" a livello radice.

Quando i Post con citazione sono stati introdotti per la prima volta, un link abbreviato (URL t.co) veniva aggiunto al messaggio del Post “originale” e fornito nel campo "text" a livello radice. Inoltre, i metadati per quell’URL t.co erano inclusi nell’array "entities.urls" a livello radice. A maggio 2018 abbiamo modificato questo comportamento: l’URL t.co abbreviato del Tweet citato non è più incluso nel campo "text" a livello radice. Inoltre, i metadati del Tweet citato non sono più inclusi nei metadati "entities.urls". Invece, i metadati URL del Tweet citato sono in un nuovo oggetto "quoted\_status\_permalink" a livello radice (o di primo livello), quindi allo stesso livello dell’oggetto "quoted\_status".

<div id="extended-posts">
  ##### Post estesi
</div>

Il JSON che descrive i _Post estesi_ è stato introdotto con il lancio dei Post da 280 caratteri a novembre 2017. Il JSON del Post è stato esteso per includere questi messaggi più lunghi, senza interrompere le migliaia di app che analizzano questi oggetti fondamentali di X. Per garantire la piena compatibilità retroattiva, sono stati mantenuti il campo originale "text" da 140 caratteri e gli oggetti delle entità derivati da esso. Nel caso di Post più lunghi di 140 caratteri, questo campo "text" a livello radice risulterebbe troncato e quindi incompleto. Poiché gli oggetti "entities" a livello radice contengono array di metadati chiave estratti dal messaggio "text", come hashtag e link inclusi, tali raccolte risulterebbero incomplete. Ad esempio, se un messaggio del Post fosse lungo 200 caratteri, con un hashtag alla fine, l'array legacy "entities.hashtags" a livello radice non lo includerebbe.

È stato introdotto un nuovo campo "extended\_tweet" per contenere i messaggi di Post più lunghi e i metadati completi delle entità. L'oggetto "extended\_tweet" fornisce il campo "full\_text", che contiene il messaggio del Post completo e non troncato quando supera i 140 caratteri. L'oggetto "extended\_tweet" contiene anche un oggetto "entities" con array completi di hashtag, link, menzioni, ecc.

I Post estesi sono identificati da un booleano "truncated" a livello radice. Quando true ("truncated": true), è necessario analizzare i campi "extended\_tweet" invece dei campi a livello radice.

Si noti nell'esempio JSON seguente che il campo "text" a livello radice è troncato e l'array "entities.hashtags" a livello radice è vuoto, sebbene il messaggio del Post includa tre hashtag. Poiché si tratta di un Post esteso, il campo "truncated" è impostato su true e l'oggetto "extended\_tweet" fornisce metadati completi del Post per "full\_text" ed "entities".

<div id="native-enriched-geo-objects">
  ### Oggetti geo nativi arricchiti
</div>

I post possono essere associati a una posizione, generando un post “geotaggato”. Le posizioni dei post possono essere assegnate tramite l’interfaccia utente di X o durante la pubblicazione via API. Le posizioni dei post possono essere un punto esatto oppure un X Place con un “bounding box” che descrive un’area più ampia, da una sede fino a un’intera regione.

Esistono tre oggetti JSON a livello “root” utilizzati per descrivere la posizione associata a un post: [place](/it/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#place-data-dictionary), [geo](/it/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-geo-objects) e [coordinates](/it/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#coordinates-object-data-dictionary). 

Inoltre, il formato nativo arricchito include, all’interno dell’oggetto utente, il [luogo derivato](/it/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#derived-locations) dell’[arricchimento del profilo geo](/it/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo).

L’oggetto `place` è sempre presente quando un post è geotaggato con un Place. I Place sono posizioni specifiche e denominate, con le corrispondenti coordinate geografiche. Quando gli utenti decidono di associare una posizione al loro post, viene presentato loro un elenco di X Place candidati. Quando si utilizza l’API per pubblicare, è possibile allegare un X Place specificando un place\_id al momento della pubblicazione. I post associati ai Place non provengono necessariamente da quella posizione, ma potrebbero anche essere _riguardanti_ quella posizione.

Gli oggetti `geo` e `coordinates` sono presenti (non null) solo quando al post è assegnata una posizione esatta. Se viene fornita una posizione esatta, l’oggetto `coordinates` restituirà un array \[long, lat] con le coordinate geografiche e verrà assegnato un X Place corrispondente a quella posizione.

<div id="introduction">
  ## Introduzione
</div>

Le entità forniscono metadati e informazioni contestuali aggiuntive sui contenuti pubblicati su X. La sezione `entities` contiene array di elementi comuni inclusi nei Post: hashtag, menzioni di utenti, link, ticker azionari (simboli), sondaggi di X e media allegati. Questi array sono utili per gli sviluppatori durante l’ingestione dei Post, poiché X ha essenzialmente pre-elaborato (o pre-parsato) il corpo del testo. Invece di dover cercare ed estrarre esplicitamente queste entità nel corpo del Post, il parser può accedere direttamente a questa sezione JSON e trovarle lì.

Oltre a semplificare il parsing, la sezione `entities` include anche utili metadati di valore aggiunto. Ad esempio, se utilizzi l’[arricchimento Enhanced URLs](/it/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls), i metadati URL includono gli URL completamente espansi, oltre ai titoli e alle descrizioni dei siti web associati. Un altro esempio: quando sono presenti menzioni di utenti, i metadati delle entità includono l’ID utente numerico, utile quando si effettuano richieste a molte API di X.

Ogni payload JSON del Post include una sezione `entities`, con il set minimo di attributi `hashtags`, `urls`, `user_mentions` e `symbols`, anche se nessuna di queste entità è presente nel messaggio del Post. Ad esempio, se esamini il JSON di un Post con corpo “Hello World!” e senza media allegati, il JSON del Post includerà il seguente contenuto con array di entità contenenti zero elementi:

```
"entities": {
        "hashtags": [
        ],
        "urls": [
        ],
        "user_mentions": [
        ],
        "symbols": [
        ]
      }
```

Note:

- le entità media e polls compaiono solo quando quel tipo di contenuto è presente nel Post.
- se lavori con media nativi (foto, video o GIF), l’[oggetto Extended Entities](/it/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#x-extended-entities) è la soluzione consigliata.

### Oggetto hashtag  

La sezione `entities` conterrà un array `hashtags` con un oggetto per ogni hashtag presente nel corpo del Post e un array vuoto se non sono presenti hashtag.

L’operatore PowerTrack `#` viene utilizzato per eseguire il match sull’attributo `text`. L’operatore `has:hashtags` effettua il match se è presente almeno un elemento nell’array.

|     |     |     |
| :--- | :--- | :--- |
| Campo | Tipo | Descrizione |
| indices | Array di Int | Un array di interi che indica gli offset all’interno del testo del Post in cui l’hashtag ha inizio e fine. Il primo intero rappresenta la posizione del carattere # nella stringa di testo del Post. Il secondo intero rappresenta la posizione del primo carattere dopo l’hashtag. Pertanto, la differenza tra i due numeri corrisponde alla lunghezza del nome dell’hashtag più uno (per il carattere ‘#’). Esempio:<br /><br />"indices":\[32,38] |
| text | String | Nome dell’hashtag, escluso il carattere iniziale ‘#’. Esempio:<br /><br />"text":"nodejs" |

### Oggetto media  

La sezione `entities` conterrà un array `media` con un singolo oggetto media se è stato “allegato” un oggetto media al Post. Se non è stato allegato alcun media nativo, nella sezione `entities` non sarà presente alcun array `media`. Per i seguenti motivi, per elaborare i media nativi del Post è consigliabile utilizzare la sezione `extended_entities`:
\+ Il campo `type` dei media indicherà sempre “photo”, anche nei casi in cui al Post siano allegati un video o una GIF.
\+ Anche se è possibile allegare fino a quattro foto, nella sezione `entities` verrà elencata solo la prima.

<div id="photo-media-url-formatting">
  ### Formattazione degli URL dei media foto
</div>

I media foto su X possono essere caricati in dimensioni diverse. È consigliabile caricare l'immagine più piccola che sia comunque sufficientemente grande da adattarsi a uno specifico viewport dell'immagine. Per caricare dimensioni diverse, è necessario combinare il [Size Object](#size) e [media\_url](#media) (o [media\_url\_https](#media)) in un formato specifico. Utilizzeremo l'oggetto di esempio della [media entity](#entitiesobject) già fornito per costruire un URL dei media foto.

`media_url` o `media_url_https` possono essere caricati anche singolarmente, il che comporta il caricamento, per impostazione predefinita, della variante medium. È preferibile, tuttavia, fornire un URL dei media foto completamente formattato quando possibile.

Ci sono tre parti di un URL dei media foto:

|     |     |
| :--- | :--- |
| Base URL | Il Base URL è l'URL del media senza l'estensione del file.<br /><br />Per esempio:  <br /><br />"media\_url\_https": "https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg",<br /><br />Il Base URL è quindi:  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq_ |
| Format | Il formato è il tipo di foto in cui l'immagine è fornita. I formati possibili sono _jpg_ o _png_, indicati come estensione dell'URL del media.<br /><br />Per esempio:  <br /><br />"media\_url\_https": "https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg",<br /><br />Il formato è quindi: _jpg_ |
| Name | Il name è il nome del campo della dimensione da caricare.<br /><br />Per esempio:<br /><br />  <br /> "sizes":   <br />   "thumb":   <br />     "h": 150,  <br />     "resize": "crop",  <br />     "w": 150  <br />   ,  <br />   "large": <br />     "h": 1366,  <br />     "resize": "fit",  <br />     "w": 2048  <br />   },  <br />   "medium": <br />     "h": 800,  <br />     "resize": "fit",  <br />     "w": 1200  <br />   },  <br />   "small": <br />     "h": 454,  <br />     "resize": "fit",  <br />     "w": 680  <br />   }  <br /> }  <br />}<br /><br />Il name quando si carica la foto in dimensione large sarebbe: _large_ |

Prendiamo queste tre parti (Base URL, Format e Name) e le combiniamo nell'URL dei media foto da caricare. Esistono 2 formati per caricare le immagini in questo modo, _legacy_ e _modern_. Tutti i caricamenti di immagini devono smettere di usare il formato _legacy_ e utilizzare il formato _modern_. L'uso del formato _modern_ comporterà un hit rate della CDN migliore per il chiamante, migliorando così le latenze di caricamento poiché è meno probabile dover generare e caricare i media dal data center.

|     |     |
| :--- | :--- |
| Legacy format | Il formato legacy è deprecato. Tutti i caricamenti di foto devono migrare al formato moderno.<br /><br />_\<base\_url>.\<format>:\<name>_  <br /><br />Per esempio:  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg:large  <br />_ |
| Modern format | Il formato moderno per il caricamento delle foto è stato introdotto su X nel 2015 ed è di fatto lo standard dal 2017. Tutti i caricamenti di foto devono utilizzare questo formato.<br /><br />_\<base\_url>?format=\<format>\&name=\<name>_  <br /><br />Per esempio:  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq?format=jpg\&name=large  <br />_<br /><br />Nota: i parametri della query string dell’URL dei contenuti foto sono in ordine alfabetico. Se il caricamento dei media dovesse aggiungere ulteriori parametri di query, l’ordinamento alfabetico dovrebbe continuare a essere rispettato. Per esempio, se ci fosse un nuovo parametro di query ipotetico chiamato _preferred\_format_, andrebbe dopo _format_ e _name_ nella query string. |

### Oggetto URL

Se utilizzi l’[Enhanced URL enrichment](http://support.gnip.com/enrichments/enhanced_urls.html), gli operatori `url_title:` e `url_description:` vengono utilizzati per effettuare il match sugli attributi `unwound.title` e `unwound.description`.

Esempio:<br /><br />"indices":\[30,53] |

<div id="retweet-and-quote-tweet-details">
  ## Dettagli su Retweet e Quote Tweet
</div>

Dal punto di vista dell’API X, Retweet e Quote Tweet sono tipi speciali di Post che contengono il Post originale come oggetto incorporato. Pertanto, gli oggetti Retweet e Quote Tweet sono genitori di un Post figlio “originale” (e quindi raddoppiano le dimensioni). I Retweet hanno un oggetto di livello superiore "retweeted\_status" e i Quote Tweet hanno un oggetto "quoted\_status". Per coerenza, questi oggetti Retweet e Quote Tweet di livello superiore hanno anche una proprietà text e le relative entities. Tuttavia, le entities a livello superiore possono differire dalle entities fornite dalle entities “originali” incorporate. Nel caso dei Retweet, il nuovo testo viene anteposto al corpo del Post originale. Per i Post citati, il nuovo testo viene aggiunto in coda al corpo del Post.

In generale, la best practice è recuperare il testo, le entities, l’autore originale e la data dal Post originale in retweeted\_status quando presente. Un’eccezione riguarda le entities di X che fanno parte della citazione aggiuntiva. Vedi sotto per ulteriori dettagli e suggerimenti.

<div id="retweets">
  ### Retweet
</div>

Un dettaglio importante dei Retweet è che non è possibile aggiungere ulteriori _entità_ X al Post. Gli utenti non possono aggiungere hashtag, URL o altri elementi quando ritwittano. Tuttavia, l’attributo di testo del Retweet (di livello superiore) è composto dal testo originale del Post con “RT @username: ” anteposto.  

In alcuni casi, soprattutto con account dal nome utente lungo, la combinazione di questi nuovi caratteri e del corpo del Post originale può facilmente superare il limite di 140 caratteri del testo del Post originale. Per mantenere la compatibilità con la visualizzazione e l’archiviazione basate su 140 caratteri, il corpo di livello superiore tronca la parte finale del corpo del Post e aggiunge un’ellissi (“…”). Di conseguenza, alcune entità di livello superiore posizionate alla fine del Post originale potrebbero risultare errate o mancanti, ad esempio nel caso di un hashtag o un URL troncati.

<div id="quote-tweets">
  ### Quote Tweet
</div>

I Quote Tweet sono stati introdotti nel 2016 e si differenziano dai Retweet perché quando “citi” un Post stai aggiungendo nuovo contenuto “sopra” un Post condiviso. Questo nuovo contenuto può includere quasi tutto ciò che può avere un Post originale, inclusi nuovo testo, hashtag, menzioni e URL.

I Quote Tweet possono contenere media nativi (foto, video e GIF) e compaiono all’interno dell’oggetto entities.

Poiché le entità di X possono essere aggiunte, le entità del Quote è probabile che differiscano da quelle originali.

In questo esempio, un nuovo URL e un hashtag sono stati inseriti alla fine del Quote Tweet.

<div id="introduction">
  #### Introduzione
</div>

Se un Post contiene contenuti multimediali nativi (condivisi tramite l’interfaccia utente del Post anziché tramite un link esterno), sarà presente anche una sezione extended\_entities. Per qualsiasi contenuto multimediale nativo (foto, video o GIF), extended\_entities è la fonte di metadati preferita per diversi motivi. Attualmente è possibile allegare fino a quattro foto a un Post. I metadati entities conterranno solo la prima foto (fino al 2014 era possibile includerne una sola), mentre la sezione extended\_entities includerà tutte le foto allegate. Con i contenuti nativi, un’ulteriore limitazione dei metadati entities.media è che il tipo di media indicherà sempre “photo”, anche nei casi in cui il contenuto allegato sia un video o una GIF animata. Il tipo effettivo è specificato nell’attributo extended\_entities.media\[].type ed è impostato su _photo_, _video_ o _animated\_gif_. Per questi motivi, se lavorate con contenuti multimediali nativi, i metadati extended\_entities sono la scelta consigliata.

Tutti i Post con foto, video e GIF animate allegate includono un oggetto JSON `extended_entities`. L’oggetto `extended_entities` contiene un singolo array `media` di oggetti `media` (vedere la sezione `entities` per il relativo dizionario dati). Nessun altro tipo di entità, come hashtag e link, è incluso nella sezione `extended_entities`. L’oggetto `media` nella sezione `extended_entities` è identico, per struttura, a quello incluso nella sezione `entities`.

I Post possono avere un solo tipo di contenuto multimediale allegato. Per le foto è possibile allegarne fino a quattro. Per i video e le GIF se ne può allegare uno. Poiché i metadati `type` dei contenuti nella sezione `extended_entities` indicano correttamente il tipo (“photo”, “video” o “animated\_gif”) e supportano fino a 4 foto, questa è la fonte di metadati preferita per i contenuti nativi.

```
{
      "extended_entities": {
        "media": [

        ]
      }
    }
```

</Note>

<Note>
  Nota: è fortemente consigliato usare il formato [Enriched Native](/it/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) per le API dati enterprise. 

  - Il formato Enriched Native include tutti i metadati introdotti dal 2017, come i [metadati dei sondaggi](/it/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata), e metriche aggiuntive come reply\_count e quote\_count.

  - Il formato Activity Streams non è stato aggiornato con nuovi metadati o arricchimenti dall’[aggiornamento sul conteggio dei caratteri](https://blog.x.com/official/en_us/topics/product/2017/Giving-you-more-characters-to-express-yourself.html) del 2017.

<div id="activity-object">
  ### Oggetto Activity
</div>

Activity Streams è una trasposizione dello schema a oggetti del [formato dati originale](https://developer.x.comhttps://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) di X, creata da Gnip per “normalizzare il formato” dei dati dei Post e di altri dati dei social media, utilizzando lo [schema di base Activity di terze parti descritto qui](https://activitystrea.ms/head/activity-schema.html). I Post vengono normalizzati nello schema di Activity Streams, includendo i tipi di oggetto note, person, place e service come oggetti annidati. I Post possono contenere altri oggetti di attività annidati relativi al Post per i Retweet o altri, tra cui twitter\_quoted\_status e long\_object.

Il tipo di oggetto di livello base “activity” è simile all’[oggetto Post di livello base](/it/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) del formato nativo arricchito. Esempi di payload nel formato Activity Streams sono disponibili [qui](/it/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#activity-streams-payload-examples).

<div id="nested-post-activity-obejcts">
  #### Oggetti di attività dei Post annidati
</div>

In diversi casi, un oggetto Post includerà altri Post annidati. L’oggetto di livello radice conterrà informazioni sul tipo di azione eseguita, ad esempio se si tratta di un Retweet o di un Quote Tweet, e può anche includere un oggetto che descrive il Post “originale” condiviso. I Post estesi includono un oggetto esteso annidato che supera i 140 caratteri, utilizzato per evitare modifiche incompatibili quando l’aggiornamento è stato rilasciato nel 2017. Ogni dizionario di oggetti annidati è descritto di seguito.

<div id="location-object">
  ### Oggetto Location
</div>

Gli oggetti Location possono esistere all’interno dell’oggetto actor impostato a livello di account X oppure all’interno dell’oggetto profileLocations dell’[oggetto gnip](/it/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#gnip-object). Gli oggetti Location hanno il tipo di oggetto place e possono includere un nome, un indirizzo o coordinate geografiche. Gli oggetti Location sono simili a [Geo](/it/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-geo-objects) nel formato nativo arricchito.

<div id="location-data-dictionary">

</div>

| Campo | Tipo | Descrizione |

| objectType | string | Vedi [qui](http://activitystrea.ms/head/activity-schema.html#place) per maggiori dettagli. Esempio:<br /><br />**"objectType":** "place" |
| displayName | string | Il nome completo della località.  <br />  <br />**"displayName":** "United States" |
| name | string | Nome della località secondo il formato JSON dei place di X. |
| link | string | Un collegamento alla rappresentazione JSON completa del place su X.  <br />  <br />**"link":** "https://api.x.com/1.1/geo/id/27c45d804c777999.json" |
| geo | object | L’oggetto delle coordinate geografiche di X, che può essere un poligono o un punto.<br /><br />Vedi [geo](https://developer.x.com/en/docs/x-api/v1/data-dictionary/object-model/geo#coordinates) |
| countryCode | String | Codice paese abbreviato che rappresenta il paese in cui si trova questa località. Esempio:<br /><br />**"countryCode": "US"** |
| country | String | Nome del paese in cui si trova questa località. Esempio:<br /><br />**"country": "United States"** |

### Introduzione\*\*

Alla base, X è una rete di comunicazione pubblica, in tempo reale e globale. Dal 2006, l’evoluzione di X è stata guidata sia dai modelli d’uso e dalle convenzioni degli utenti sia da nuove funzionalità e miglioramenti del prodotto. Se utilizzi i dati di X per ricerche storiche, comprendere la cronologia di questa evoluzione è importante per far emergere i Post d’interesse dall’archivio dei dati.

X è stato lanciato come una semplice app mobile via SMS ed è cresciuto fino a diventare una piattaforma di comunicazione completa. Una piattaforma con un set completo di API. Le API sono sempre state un pilastro della rete X. La [prima API è arrivata poco dopo il lancio di X](https://blog.x.com/2006/introducing-the-twitter-api). Quando nel 2009 è stata introdotta la geolocalizzazione dei Post, è stata resa disponibile tramite una [Geo API](https://blog.x.com/2009/think-globally-tweet-locally) (e successivamente la possibilità di “geotaggare” un Post è stata integrata nell’interfaccia utente di X.com). Oggi, le API di X alimentano la rete di comunicazione bidirezionale che è diventata la fonte di notizie dell’ultima ora e di condivisione delle informazioni. Le opportunità per costruire su questo canale di comunicazione globale e in tempo reale sono infinite.

X mette a disposizione due API storiche che forniscono accesso a ogni Post disponibile pubblicamente: Historical PowerTrack e la [Full-Archive Search API](https://developer.x.com/en/docs/x-api/v1/tweets/search/overview). Entrambe le API forniscono un set di _operatori_ utilizzati per interrogare e raccogliere Post d’interesse. Questi operatori eseguono il match su una varietà di attributi associati a ogni Post, centinaia di attributi come il contenuto testuale del Post, il nome dell’account dell’autore e i link condivisi nel Post. I Post e i loro attributi sono codificati in JSON, un comune formato testuale di interscambio dati. Pertanto, con l’introduzione di nuove funzionalità sono comparsi nuovi attributi JSON e, in genere, sono stati introdotti nuovi operatori API per effettuare il match su tali attributi. Se il tuo caso d’uso prevede la necessità di _ascoltare_ ciò che il mondo ha detto su X, più comprenderai quando gli operatori hanno iniziato ad avere metadati JSON su cui eseguire il match, più efficaci potranno essere i tuoi filtri storici PowerTrack.

Successivamente, introdurremo alcuni concetti chiave che preparano il terreno per comprendere come gli aggiornamenti dei metadati dei Post influenzino l’individuazione del segnale di dati d’interesse.

### Concetti chiave\*\*

#### Dalle convenzioni degli utenti agli _oggetti di prima classe_ di X

Gli utenti di X hanno introdotto in modo organico nuovi modelli di comunicazione, oggi fondamentali per la rete di X. Un esempio emblematico è l’hashtag, ormai quasi universalmente utilizzato su tutti i social network. Gli hashtag sono nati come un modo per organizzare conversazioni e argomenti. Su una rete con centinaia di milioni di messaggi al giorno, gli strumenti per trovare Post di interesse sono cruciali, e gli hashtag sono diventati un metodo essenziale. Poco dopo la diffusione degli hashtag, questi hanno ricevuto riconoscimento e supporto ufficiali da X. Quando gli hashtag sono diventati un _“oggetto di prima classe”_, ciò ha comportato diverse conseguenze: gli hashtag sono diventati cliccabili/ricercabili nell’interfaccia utente di X.com e sono entrati a far parte della famiglia delle _entities_ di X, insieme alle @mentions, ai media allegati, ai simboli azionari e ai link condivisi. Queste entità sono comodamente codificate in un array JSON già analizzato, semplificando per gli sviluppatori l’elaborazione, la scansione e l’archiviazione.

I Retweet sono un altro esempio di convenzioni guidate dagli utenti che diventano oggetti ufficiali. Il retweeting è emerso come un modo per “inoltrare” contenuti ad altri. È iniziato come un processo manuale di copia/incolla di un Post e di anteposizione del pattern “RT @”. Questo processo è stato poi automatizzato tramite un nuovo pulsante Retweet, completo di nuovi metadati JSON. È nato il Retweet “ufficiale”. Altri esempi includono le “menzioni”, la condivisione di media e link web, e la condivisione della posizione insieme al proprio Post. Ciascuno di questi modelli d’uso ha portato a nuove funzionalità dell’interfaccia utente di [x.com](https://x.com/), a nuovo JSON di supporto e quindi a nuovi modi di effettuare il match sui Post. Tutti questi attributi fondamentali del Post hanno dato origine a PowerTrack Operators utilizzati per effettuare il match su di essi.

<div id="post-metadata-mutability-updates-and-currency">
  #### Metadati dei Post, mutabilità, aggiornamenti e attualità
</div>

Sebbene i messaggi dei Post possano avere una lunghezza massima predefinita, la descrizione JSON di un Post è composta da oltre 100 attributi. Attributi come chi ha pubblicato, a che ora, se si tratta di un Post originale o di un Retweet, e un array di oggetti di prima classe come hashtag, menzioni e link condivisi. Per l’account che ha pubblicato, è presente un oggetto User (o Actor) con una varietà di attributi che forniscono il profilo dell’utente e altri metadati dell’account. I profili includono una breve descrizione biografica, una località di riferimento (testo libero), la lingua preferita e un link facoltativo a un sito web.

Alcuni metadati dell’account non cambiano mai (ad es. l’ID utente numerico e la data di creazione), altri cambiano lentamente nel tempo, mentre altri ancora variano più frequentemente. Le persone cambiano lavoro e si trasferiscono. Le aziende aggiornano le proprie informazioni. Quando si raccolgono Post storici, è importante capire che alcuni metadati sono _come erano al momento della pubblicazione_, mentre altri sono _come sono al momento dell’invio della query_.

Con tutte le API storiche, la descrizione del profilo, il nome visualizzato e gli attributi “home” del profilo dell’utente vengono aggiornati ai valori presenti al momento della query.

<div id="native-media">
  #### Media “nativi”
</div>

X.com e le app mobile di X consentono di aggiungere foto e video ai Post cliccando un pulsante e sfogliando le proprie gallerie. Poiché queste azioni sono ora integrate nativamente, i video e le foto condivisi in questo modo vengono definiti media “nativi”.

Molti operatori di query funzionano con queste risorse “native”, tra cui `has:videos`, `has:images` e `has:media`. Questi operatori corrispondono solo a contenuti multimediali condivisi tramite le funzionalità di X. Per individuare altri media ospitati al di fuori della piattaforma X, è consigliabile usare operatori che si basano sui metadati degli URL.

Prima di addentrarci nei dettagli dei prodotti Historical PowerTrack e Ricerca dell’archivio completo, facciamo una panoramica dell’evoluzione di X, come prodotto e come piattaforma, nel tempo.

**Cronologia di X**

Di seguito troverai una _cronologia_ selezionata di X. La maggior parte di questi aggiornamenti ha in qualche modo inciso in modo sostanziale sul comportamento degli utenti, sui contenuti JSON dei Post, sugli operatori di query o su tutti e tre. Considerando X come piattaforma API, i seguenti eventi hanno influito sui payload JSON usati per codificare i Post. A loro volta, questi dettagli JSON influenzano il modo in cui le API storiche di X effettuano il matching.

**Suggerimenti per il filtraggio**

Conoscere la cronologia di X su quando e come sono state aggiunte nuove funzionalità può aiutarti a creare query più efficaci. Qui, una query indica un _filtro_ o una _regola_ applicata dalle API storiche di X all’archivio dei Post, usando gli Operatori PowerTrack per eseguire il match sul JSON del Post. Un esempio è l’Operatore `lang:`, utilizzato per trovare i Post in una lingua specificata. X fornisce un servizio di classificazione della lingua (supporta oltre 50 lingue) e le API di X includono questi metadati nel JSON generato per ogni Post. Quindi, se un Post è scritto in spagnolo, l’attributo JSON “lang” è impostato su “es”. Pertanto, se crei un filtro con la clausola `lang:es`, corrisponderà solo ai messaggi dei Post classificati come spagnoli.

La cronologia può anche aiutare a interpretare meglio i dati dei Post ricevuti. Poniamo che tu stia analizzando la condivisione di contenuti sulle Olimpiadi estive del 2008 e del 2012. Se applicassi solo l’Operatore `is:retweet` per trovare i Retweet, nel 2008 non verrebbe trovato alcun dato. Tuttavia, per il 2012 probabilmente ci sarebbero milioni di Retweet. Da ciò potresti erroneamente concludere che nel 2008 i Retweet non fossero una consuetudine tra gli utenti, o che semplicemente nessuno avesse fatto Retweet su quelle Olimpiadi. Poiché i Retweet sono diventati un oggetto di prima classe nel 2009, è necessario aggiungere una clausola di regola `”RT @”` per aiutarne l’identificazione nel 2008.

Sia i Retweet sia la classificazione della lingua dei Post sono esempi di attributi dei Post con una lunga storia e molti dettagli di prodotto. Di seguito approfondiremo questi e altri gruppi di attributi importanti per effettuare il match e comprendere i Dati X.

<div id="recognizing-false-negatives">
  #### Riconoscere i falsi negativi
</div>

Quando si scrivono filtri, è importante ricordare che tutti gli operatori sui metadati hanno una “data di nascita”. Se crei un filtro con un operatore che agisce su metadati introdotti dopo la pubblicazione del Post, otterrai un falso negativo. Ad esempio, supponiamo che tu sia interessato a tutti i Post che menzionano “snow” e condividono un video. Se crei una regola con l’operatore `has:videos`, che corrisponde ai Post con video nativi, quella clausola non corrisponderà a nessun Post precedente al 2015.

Tuttavia, la condivisione di video era comune su X molto prima del 2015. In precedenza gli utenti condividevano link a video ospitati altrove, ma nel 2015 X ha integrato nuove funzionalità di condivisione video direttamente nella piattaforma. Per trovare questi Post precedenti di interesse, includeresti una clausola di regola come `url:"youtube.com"`.

Nota: con le API di Ricerca, ci sono stati alcuni esempi di metadati “retrocompilati” quando l’indice è stato ricostruito. Un buon esempio sono i $cashtag, diventati di uso comune per discutere i simboli azionari nel 2009. Dopo l’introduzione dell’operatore $cashtag nel 2015, l’indice di Ricerca è stato ricostruito e, nel processo, l’entità “simbolo” è stata estratta da tutti i corpi dei Post, inclusi i primi del 2006, quando il carattere `$` era usato principalmente come gergo; “I hope it $now$ $oon!”.

<div id="identifying-and-filtering-on-post-attributes-important-to-your-use-case">
  #### Identificare e filtrare gli attributi dei Post importanti per il tuo caso d’uso
</div>

Alcuni metadati, come gli ID numerici degli account X, esistono fin dall’inizio (e sono un esempio di metadati dell’account che non cambiano mai). Altri metadati sono stati introdotti solo molto tempo dopo l’avvio di X nel 2006. Esempi di nuovi metadati introdotti includono i metadati dei Retweet, le localizzazioni dei Post, i titoli e le descrizioni degli URL e i contenuti multimediali “nativi”. Di seguito sono riportati alcuni dei tipi più comuni di attributi dei Post che sono stati sostanzialmente influenzati da questi aggiornamenti della piattaforma X.

Il comportamento di filtraggio/corrispondenza di questi attributi dipende, nella maggior parte dei casi, da quale API storica dei Post viene utilizzata. Per aiutarti a determinare quale prodotto sia più adatto alla tua ricerca e al tuo caso d’uso, i dettagli degli attributi riportati di seguito includono informazioni di alto livello sul prodotto.

<div id="x-profiles">
  #### Profili X
</div>

Poiché X, nella sua essenza, è un canale di comunicazione globale in tempo reale, la ricerca sui dati dei Post pone spesso l’accento su chi sta comunicando. Spesso è utile sapere dove un utente X “ha casa”. Sapere che la bio di un account menziona interessi e hobby può indirizzare verso Post pertinenti. È molto comune voler monitorare i Post degli account di interesse. Gli attributi del profilo sono fondamentali per tutti questi casi d’uso.

Ogni account su X ha un profilo che include metadati come l’@handle X, il nome visualizzato, una breve bio, la località (testo libero inserito dall’utente), il numero di follower e molti altri. Alcuni attributi non cambiano mai, come l’ID utente numerico e la data di creazione dell’account. Altri in genere cambiano di giorno in giorno, di settimana in settimana o di mese in mese, come il numero di Post pubblicati e il numero di account seguiti e di follower. Altri attributi dell’account possono cambiare in qualsiasi momento, ma tendono a farlo meno frequentemente: nome visualizzato, località e bio.

Il payload JSON di ogni Post include i metadati del _profilo dell’account_ dell’autore del Post. Se si tratta di un Retweet, include anche i metadati del profilo dell’account che ha pubblicato il Post originale.

La mutabilità dei metadati del profilo di un Post dipende interamente dal prodotto storico utilizzato. Le API di ricerca restituiscono Post storici con le impostazioni del profilo _così come sono al momento del recupero_. Per Historical PowerTrack, il profilo è _così com’era al momento della pubblicazione del Post_, fatta eccezione per i dati precedenti al 2011. Per i Post antecedenti al 2011, i metadati del profilo riflettono il profilo com’era nel settembre 2011.

<div id="original-post-and-retweets">
  #### Post originali e Retweet
</div>

I Retweet sono un altro esempio di convenzioni guidate dagli utenti diventate oggetti ufficiali. Il retweet è nato come un modo per “inoltrare” contenuti ad altri. Inizialmente era un processo manuale di copia/incolla di un Post con l’aggiunta in testa del pattern “RT @”. Questo processo è stato poi automatizzato con un nuovo pulsante Retweet, completo di nuovi metadati JSON. È nato il Retweet “ufficiale” e l’azione di retwittare è diventata un evento di Post di prima classe. Insieme al nuovo pulsante Retweet, sono stati introdotti nuovi metadati, come il payload completo del Post originale.

Il fatto che un Post sia originale o condiviso è un comune “interruttore” di filtraggio. In alcuni casi serve solo contenuto originale. In altri casi, l’engagement sul Post è prioritario, quindi i Retweet sono fondamentali. L’operatore PowerTrack `is:retweet` consente di includere o escludere i Retweet. Se si estraggono dati antecedenti ad agosto 2009, è necessario adottare due strategie per individuare i Retweet (o escluderli). Prima di agosto 2009 occorre verificare il testo del Post stesso, utilizzando la corrispondenza di frase esatta, per trovare corrispondenze con il pattern “RT @ ”. Per i periodi successivi ad agosto 2009 è disponibile l’operatore `is:retweet`.

<div id="post-language-classifications">
  #### Classificazioni della lingua dei Post
</div>

La lingua in cui è scritto un Post è di interesse comune. La lingua del Post può aiutare a dedurre la posizione di un Post e spesso per l’analisi o la visualizzazione è necessaria solo una lingua specifica. (I profili X dispongono anche di un’impostazione della lingua preferita.)

Per il filtraggio in base alla classificazione della lingua di un Post, i prodotti storici di X ([Search API](https://developer.x.com/en/docs/x-api/v1/tweets/search/overview/enterprise) e Historical PowerTrack) differiscono notevolmente. Quando è stato creato l’archivio di Search, tutti i post sono stati integrati retrospettivamente con la classificazione linguistica di X. Pertanto, l’operatore `lang:` è disponibile per l’intero archivio dei post. Con Historical PowerTrack, i metadati della classificazione linguistica di X sono disponibili nell’archivio a partire dal 26 marzo 2013. 

<div id="geo-referencing-posts">
  #### Geo-referenziazione dei Post
</div>

Essere in grado di indicare dove è stato pubblicato un Post (cioè geo-referenziarlo) è importante per molti casi d’uso. Esistono tre metodi principali per geo-referenziare i Post:

- Riferimenti geografici nel testo di un Post
- Post geotaggati dall’utente.
- Località “home” del profilo dell’account impostata dall’utente

  ##### Riferimenti geografici in un Post
</div>

Eseguire il matching su riferimenti geografici all’interno del Post, pur essendo spesso il metodo più complesso perché dipende da conoscenze locali, è un’opzione applicabile all’intero archivio di Post. Ecco un esempio di corrispondenza georeferenziata del 2006 per l’area di San Francisco basata sul filtro “golden gate”:

https://x.com/biz/statuses/28311

<div id="posts-geo-tagged-by-the-user">
  ##### Post con geotag aggiunti dall'utente
</div>

Nel novembre 2009 X ha introdotto la Post Geotagging API, che ha consentito di associare ai post un geotag con una posizione esatta. Nel giugno 2010 X ha introdotto X Places, che rappresentano un'area geografica a livello di luogo, quartiere o città. Circa l'1-2% dei post è geotaggato utilizzando uno dei due metodi.

La cronologia di geotagging disponibile dipende dalla Historical API in uso. Con le Search API la possibilità di iniziare a eseguire il matching sui post con alcuni Geo Operators è partita a marzo 2010, e con altri a febbraio 2015. Se si utilizza Historical PowerTrack, la georeferenziazione inizia il 1º settembre 2011. Quando è stato creato l'archivio di Historical PowerTrack, tutto il geotagging precedente a questa data non è stato incluso.

<div id="account-profile-home-location-set-by-a-user">
  ##### Posizione “home” del profilo dell’account impostata dall’utente
</div>

Tutti gli utenti di X possono impostare la Posizione del profilo, indicando la loro località di residenza. Milioni di utenti X forniscono queste informazioni, aumentando significativamente la quantità di geodati nel Firehose di X. Questi metadati di posizione sono stringhe non normalizzate, generate dagli utenti e in formato libero. Circa il 30% degli account ha metadati Profile Geo risolvibili a livello di paese.

Come per la geolocalizzazione dei Post, i metodi di corrispondenza e i periodi temporali disponibili dipendono dall’API storica utilizzata. Historical PowerTrack consente agli utenti di eseguire corrispondenze personalizzate su queste stringhe in formato libero. Per semplificare tale processo, X fornisce anche un [arricchimento Profile Geo](/it/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo) che, ove possibile, esegue la geocodifica, fornendo metadati normalizzati e i relativi operatori. Gli operatori Profile Geo sono disponibili sia in Historical PowerTrack sia nelle API di ricerca. Con Historical PowerTrack, questi metadati Profile Geo sono disponibili a partire da giugno 2014. Con le API di ricerca, questi metadati sono disponibili a partire da febbraio 2015.

<div id="shared-links-and-media">
  #### Link condivisi e media
</div>

La condivisione di link a pagine web, foto e video è sempre stata un caso d’uso fondamentale di X. Agli inizi, tutte queste azioni prevedevano l’inclusione di un URL nel messaggio del Post stesso. Nel 2011 X ha integrato la condivisione diretta delle foto nella propria interfaccia utente. Nel 2016 sono stati aggiunti i video nativi.

Alla luce di questa evoluzione, esiste una varietà di Operatori di filtraggio utilizzati per eseguire il matching su questi contenuti. C’è un insieme di Operatori che verificano se i Post includono link, foto e video condivisi. Inoltre, poiché la maggior parte degli URL condivisi su X viene accorciata per consumare meno caratteri del Post (ad es. generati da servizi come bitly o tinyurl), X fornisce arricchimenti dei dati che producono un URL completo ed espanso su cui è possibile effettuare il matching. Ad esempio, se volessi eseguire il matching di Post che includono link relativi a X e ai sistemi di allerta precoce, un filtro che fa riferimento a “comunicazione di condizioni meteo estreme” corrisponderebbe a un Post contenente questo URL [http://bit.ly/1XV1tG4](http://bit.ly/1XV1tG4).

Nel marzo 2012 è stato introdotto l’[arricchimento degli URL espansi](/it/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls). Prima di allora, i payload dei Post includevano solo l’URL fornito dall’utente. Quindi, se l’utente includeva un URL accorciato, poteva risultare difficile eseguire il matching sugli URL (espansi) di interesse. Sia con Historical PowerTrack sia con le API di Ricerca, questi metadati sono disponibili a partire da marzo 2012.

Nel luglio 2016 è stato introdotto l’arricchimento degli URL avanzati. Questa versione avanzata fornisce nel payload del Post il titolo e la descrizione HTML del sito web, insieme agli Operatori per eseguire il matching su tali elementi. Con Historical PowerTrack, questi metadati diventano disponibili a luglio 2016. Con le API di Ricerca, questi metadati iniziano a comparire a dicembre 2014.

Nel settembre 2016 X ha introdotto gli “allegati nativi”, per cui un link condiviso in coda non viene conteggiato ai fini del limite di 140 caratteri del Post. Entrambi gli arricchimenti degli URL si applicano ancora a questi link condivisi.

Per ulteriori dettagli specifici di prodotto sul filtraggio degli URL, consulta gli articoli corrispondenti.
