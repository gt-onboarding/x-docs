L’API X fornisce i seguenti codici di risposta ed errore per aiutare a comprendere e diagnosticare i problemi nell’immediato. Utilizza la guida al debug e l’indice degli errori riportati di seguito per ulteriori dettagli.

Le risposte con esito positivo sono indicate da un codice HTTP della serie 200 e da un payload JSON contenente gli oggetti richiesti, creati, modificati o eliminati, insieme a una descrizione di come il server ha interpretato la tua richiesta.

Le risposte di errore vengono restituite con un codice HTTP non appartenente alla serie 200. Codici di errore differenti indicano cause diverse. L’API X cerca di restituire [codici di stato HTTP](https://en.wikipedia.org/wiki/List_of_HTTP_status_codes) appropriati per ogni richiesta.

<div id="x-api-v2-http-status-codes">
  ## Codici di stato HTTP dell'API X v2
</div>

| Code | Text | Description | Troubleshooting tips |
| :--- | :--- | :---------- | :------------------- |
| 200 | OK | La richiesta è stata eseguita correttamente. | |
| 400 | Bad Request	| La richiesta non è valida o non può essere elaborata. Un messaggio di errore allegato fornirà ulteriori dettagli. Le richieste senza autenticazione o con parametri di query non validi sono considerate non valide e restituiranno questa risposta. | Ricontrolla il formato della query JSON. Ad esempio, se la regola contiene virgolette doppie associate a un operatore di corrispondenza esatta o ad altri operatori, potresti doverle eseguire l’escape con una barra rovesciata per distinguerle dalla struttura del formato JSON. |
| 401 | Unauthorized | Si è verificato un problema nell’autenticare la richiesta. Potrebbe dipendere da credenziali di autenticazione mancanti o errate. Questo codice può anche essere restituito in altre circostanze non definite. | Verifica di utilizzare il [metodo di autenticazione](https://docs.x.com/it/resources/fundamentals/authentication/overview) corretto e che le credenziali siano corrette. |
| 403 | Forbidden | La richiesta è compresa, ma è stata rifiutata o l’accesso non è consentito. Un messaggio di errore allegato spiegherà il motivo. | Verifica che il tuo account sviluppatore disponga dell’accesso all’endpoint che stai tentando di usare. Potrebbe anche essere necessario inserire la tua App in allowlist (ad es. Engagement API o Ads API) o richiedere l’accesso. |
| 404 | Not Found | L’URI richiesto non è valido o la risorsa richiesta, come un utente, non esiste. | Verifica di usare parametri validi e l’URI corretto per l’endpoint che stai utilizzando. |
| 409 | Connection Exception | Restituito quando si tenta di connettersi a un flusso filtrato privo di regole. | Verifica di aver creato almeno una regola per il flusso a cui ti stai connettendo. Il flusso filtrato restituisce solo Post che corrispondono a una regola attiva. Se non ci sono regole, il flusso non restituirà alcun Post. |
| 429 | Too Many Requests | Restituito quando una richiesta non può essere servita perché è stato esaurito il [rate limit](/it/resources/fundamentals/rate-limits) dell’App o il [limite di post](/it/x-api/fundamentals/post-cap). Vedi [Rate limiting](/it/resources/fundamentals/rate-limits). | Verifica il numero di richieste per intervallo di tempo consentito con l’endpoint che stai usando. Attendi il reset dell’intervallo. Distribuisci le richieste per evitare di raggiungere i limiti oppure esegui l’upgrade al piano dati successivo disponibile. |
| 500 | Internal Server Error | Qualcosa non funziona. Di solito è un errore temporaneo, ad esempio in situazioni di carico elevato o se un endpoint ha problemi temporanei. | Controlla la [pagina di stato](/it/status) dell’API X o il [forum della community degli sviluppatori](https://devcommunity.x.com/) per vedere se altri riscontrano problemi simili, oppure attendi e riprova più tardi. |
| 501 | Unimplemented | L’API X non supporta questo endpoint e non può soddisfare la richiesta. |  |
| 503 | Service Unavailable | I server X sono operativi, ma sovraccarichi di richieste. Riprova più tardi. | Controlla la [pagina di stato](/it/status) dell’API X o il [forum della community degli sviluppatori](https://devcommunity.x.com/) per vedere se altri riscontrano problemi simili, oppure attendi e riprova più tardi. |

Quando si verifica un errore durante una richiesta, nel corpo della risposta vengono restituite informazioni dettagliate per facilitare la diagnosi. Un campo type, che è un URI, indica la natura del problema, mentre campi aggiuntivi forniscono ulteriori dettagli. I campi type, title e detail vengono sempre restituiti in questi corpi (vedi tabella sotto). Eventuali campi aggiuntivi, come nell’esempio seguente, variano a seconda del tipo di errore.

```json
{
  "client_id": "101010101",
  "required_enrollment": "Standard Basic",
  "registration_url": "https://developer.twitter.com/en/account",
  "title": "Client Forbidden",
  "detail": "Questa richiesta deve essere effettuata utilizzando un account sviluppatore approvato e iscritto all’endpoint richiesto. Scopri di più consultando la nostra documentazione.",
  "reason": "client-not-enrolled",
  "type": "https://api.x.com/2/problems/client-forbidden"
}
```

<div id="partial-errors">
  ## Errori parziali
</div>

In alcuni casi potresti vedere gli errori descritti sopra in una risposta che ha restituito un codice di stato 200. In queste situazioni, l'endpoint è progettato per restituire i dati disponibili fornendo al contempo errori dettagliati su ciò che non è stato possibile restituire.

Ad esempio, l'[endpoint di ricerca dei Post](/it/x-api/posts/lookup/introduction) consente a una [App developer](/it/resources/fundamentals/developer-apps#overview) di X di richiedere più di un ID. Se alcuni di quei Post sono disponibili, ma uno è stato eliminato, i Post disponibili verranno restituiti nel campo data della risposta. Un ulteriore campo errors verrà incluso nel payload per indicare quali Post richiesti non è stato possibile restituire. Viene utilizzato lo stesso formato degli errori dell'intera richiesta per semplificare la diagnosi dei problemi.

<div id="error-information">
  ### Informazioni sugli errori
</div>

Ogni tipo di problema indica la natura dell’errore riscontrato. Un elenco completo dei possibili problemi è disponibile anche nella nostra [specifica API](https://api.x.com/2/openapi.json).

L’API X tenta di restituire [codici di stato HTTP](https://en.wikipedia.org/wiki/List_of_HTTP_status_codes) appropriati per ogni richiesta.

| Titolo | Tipo | Descrizione |
| :--- | :--- | :--- |
| Problema generico | `about:blank` | Un problema generico senza informazioni aggiuntive oltre a quelle fornite dal codice di stato HTTP. |
| Problema di richiesta non valida | `https://api.X.com/2/problems/invalid-request` | Un problema che indica che questa richiesta non è valida. Se la tua richiesta include un body POST, assicurati che il contenuto sia JSON valido e rispetti la [specifica OpenAPI](https://api.x.com/2/openapi.json). |
| Problema: risorsa non trovata | `https://api.X.com/2/problems/resource-not-found` | Un problema che indica che un determinato Post, Utente, ecc. non esiste. |
| Problema: non autorizzato per la risorsa | `https://api.X.com/2/problems/not-authorized-for-resource` | Un problema che indica che non sei autorizzato a visualizzare un determinato Post, Utente, ecc. |
| Problema: client non autorizzato | `https://api.X.com/2/problems/client-forbidden` | Un problema che indica che al tuo client non è consentito effettuare questa richiesta. |
| Problema: risorsa non consentita | `https://api.X.com/2/problems/disallowed-resource` | Un problema che indica che la risorsa richiesta viola i principi di questa API. |
| Problema: autenticazione non supportata | `https://api.X.com/2/problems/unsupported-authentication` | Un problema che indica che l’autenticazione utilizzata non è supportata. |
| Problema: limite di utilizzo superato | `https://api.X.com/2/problems/usage-capped` | Un problema che indica che è stato superato un limite di utilizzo. |
| Problema di connessione | `https://api.X.com/2/problems/streaming-connection` | Un problema che indica che c’è qualcosa che non va con la connessione. |
| Problema: client disconnesso | `https://api.X.com/2/problems/client-disconnected` | Il tuo client si è disconnesso. |
| Problema: disconnessione operativa | `https://api.X.com/2/problems/operational-disconnect` | Sei stato disconnesso per motivi operativi. |
| Problema: limite regole | `https://api.X.com/2/problems/rule-cap` | Hai superato il numero massimo di regole. |
| Problema: lunghezza regole | `https://api.X.com/2/problems/rule-length` | Hai superato il numero massimo di caratteri consentiti per la tua query o regola in base al tuo livello di accesso. Vedi [livelli di accesso](/it/x-api/getting-started/about-x-api#x-api-access-levels-and-versions). |
| Problema: regole non valide | `https://api.X.com/2/problems/invalid-rules` | La regola che hai inviato non è valida. |

<div id="troubleshooting-tips">
  ## Suggerimenti per la risoluzione dei problemi
</div>

<Accordion title="Debug del codice">
  Durante lo sviluppo di un’applicazione è normale imbattersi, di tanto in tanto, in errori o anomalie. Di seguito trovi alcuni suggerimenti su come eseguire il debug del codice:

  Inizia scomponendo il problema in parti più piccole per individuare dove si verifica. Ad esempio, se stai integrando un endpoint REST o di streaming, la causa potrebbe essere una delle seguenti:

  - L’endpoint richiede la [corretta autenticazione](/it/resources/fundamentals/authentication/overview).
  - L’endpoint richiede parametri e intestazioni validi. Le eventuali regole di filtro devono usare gli operatori corretti e la [sintassi](/it/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#building-rules-and-queries) appropriata.
  - L’URI dell’endpoint deve essere corretto e, nel caso degli endpoint REST, va usato il metodo HTTP appropriato.
  - I dati o la risorsa a cui stai cercando di accedere potrebbero non essere disponibili per il tuo account (ad esempio, i dati privati sono accessibili solo agli utenti autenticati).
  - Il tuo attuale pacchetto dati fornisce accesso solo a determinati endpoint e a specifici limiti di rate. Consulta il [portale sviluppatori](https://developer.x.com/en/portal/products) per maggiori dettagli.
  - Il tuo codice utilizza una libreria di terze parti per integrare l’endpoint.
  - Il tuo codice deve analizzare correttamente la risposta dell’endpoint.

  Leggi il messaggio di errore associato: dovrebbe fornire un buon indizio sulla causa. Consulta le tabelle nella [sezione dei codici di errore e di risposta](#x-api-http-status-codes) per suggerimenti di troubleshooting specifici per ciascun codice.

  Per gli endpoint REST, puoi usare un client REST come [Postman](https://www.postman.com/) o [Insomnia](https://insomnia.rest/) per verificare i passaggi da (1) a (5) sopra (vedi la guida ["Primi passi con Postman"](/it/tutorials/postman-getting-started)). Se la richiesta con il client REST restituisce un codice di stato 200 (successo), è probabile che il problema riguardi il tuo codice o la libreria in uso, non la richiesta all’endpoint.

  Per gli endpoint di streaming, puoi usare cURL (uno strumento da riga di comando per inviare o ricevere richieste utilizzando la sintassi degli URL) per verificare se il problema riguarda la richiesta all’endpoint (passaggi da (1) a (5) sopra) o il tuo codice (passaggi da (6) a (7) sopra).
</Accordion>

<Accordion title="Problemi di autenticazione">
  - Verifica di utilizzare il [metodo di autenticazione](/it/resources/fundamentals/authentication/overview) richiesto dall’endpoint. Puoi identificarlo nella pagina di riferimento API dell’endpoint.
  - Verifica che le tue credenziali di autenticazione siano corrette. Puoi controllare o rigenerare le chiavi e i token dell’App nella [sezione Apps](https://developer.x.com/en/portal/projects-and-apps) della dashboard sviluppatore (alla voce “Details”).
  - Verifica di aver [autorizzato correttamente la richiesta OAuth 1.0a](/it/resources/fundamentals/authentication/oauth-1-0a/authorizing-a-request) con `oauth_nonce`, `oauth_signature` e `oauth_timestamp`.
  - Se il problema persiste, valuta l’uso di una libreria OAuth, di un client REST come [Postman](https://www.postman.com/) o [Insomnia](https://insomnia.rest/), oppure di [xurl](https://github.com/xdevplatform/xurl).

  Consulta la [guida all’autenticazione](/it/resources/fundamentals/authentication/overview) per ulteriori informazioni su tutti i punti sopra.
</Accordion>

<Accordion title="Problemi di limitazione della frequenza e limiti di post">
  Un errore 429 può verificarsi se raggiungi un limite di frequenza per un determinato endpoint o se raggiungi un Limite di post.

  Se viene restituito lo specifico errore “Too many requests”, hai raggiunto il limite di frequenza dell'endpoint. In altre parole, hai superato il numero massimo di richieste consentite per un endpoint in un determinato periodo di tempo.

  Nota che i limiti di frequenza sono impostati a livello di App e a livello di utente.

  - **Livello X App** indica il numero di richieste consentite quando si utilizza OAuth 2.0 App-Only, dove i limiti di frequenza sono determinati globalmente per l'intera App. Ad esempio, se un metodo consente 15 richieste per finestra di limitazione, potrai effettuare 15 richieste per finestra per conto della tua X App. Questo limite è considerato completamente separato dal limite a livello utente. Per saperne di più, consulta la nostra guida su [OAuth 2.0 App-Only](/it/resources/fundamentals/authentication/oauth-2-0/application-only).
  - **Livello contesto utente** indica il numero di richieste che possono essere effettuate per token di accesso utente quando si utilizza OAuth 1.0a User Context o OAuth 2.0 Authorization Code con PKCE. Ad esempio, se un metodo consente 15 richieste per finestra di limitazione, sono consentite 15 richieste per finestra e per token di accesso utente. Scopri come ottenere i token di accesso di un utente con [OAuth 1.0a](/it/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens) e [OAuth 2.0](/it/resources/fundamentals/authentication/oauth-2-0/user-access-token).

  Inizia controllando i limiti di frequenza per l'endpoint che stai utilizzando. Puoi trovare queste informazioni nella pagina di riferimento API dell'endpoint e nel nuovo dashboard del portale sviluppatori.

  Consulta la nostra documentazione per ulteriori informazioni sui limiti di frequenza, incluse le modalità per utilizzare le intestazioni HTTP per monitorare lo stato della tua App rispetto a un determinato limite, come riprendersi da un errore 429 dovuto a limitazione e suggerimenti per evitare di incorrere nella limitazione:

  - [Limiti di frequenza per gli endpoint v2](/it/x-api/fundamentals/rate-limits)

  Se hai ricevuto lo specifico errore “Usage cap exceeded: Monthly product cap”, significa che hai raggiunto i Limiti di post per il tuo livello di accesso. Trovi tutti i dettagli su cosa siano questi [Limiti di post](/it/x-api/fundamentals/post-cap) nella nostra documentazione.
</Accordion>

<Accordion title="Problemi di post mancanti">
  Segui i passaggi seguenti se ti aspettavi che venisse restituito un post, ma l'endpoint non lo ha consegnato.

  - Controlla la tua regola per assicurarti di utilizzare gli [operatori](/it/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#enterprise-operators) e la [sintassi](/it/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#building-rules-and-queries) corretti. Suddividi la regola in clausole più piccole per verificare di utilizzare la sintassi corretta.
  - Se l'account da cui è stato inviato il post era [protetto](https://help.twitter.com/en/safety-and-security/public-and-protected-tweets) al momento della creazione del post, il post non verrà restituito, anche se l'account è pubblico al momento della richiesta all'endpoint. In genere puoi verificarlo utilizzando la [Ricerca avanzata di X](https://x.com/search-advanced): se un post non viene mostrato tramite la funzione di Ricerca avanzata di X, dovresti presumere che non verrà restituito dall'endpoint.

  I passaggi seguenti si applicano solo agli endpoint di streaming:

  - Eri connesso al flusso quando è stato inviato il post? Ricorda che il timestamp fornito nell'oggetto Post indica l'ora in UTC. Se si è verificata una disconnessione quando è stato inviato il post, consulta le [funzionalità di ripristino e ridondanza](/it/x-api/posts/filtered-stream/integrate/recovery-and-redundancy-features#recovery-and-redundancy) disponibili per recuperare eventuali dati mancanti.
  - La tua regola era in vigore quando è stato inviato il post? Ricorda che il timestamp fornito nell'oggetto Post indica l'ora in UTC.
</Accordion>

<Accordion title="Problemi di disconnessione per buffer pieno">
  Potresti ricevere uno dei seguenti errori quando il tuo stream non riesce a tenere il passo con la velocità alla quale stiamo distribuendo i dati e la tua App non consuma i dati dallo stream abbastanza rapidamente:

  ```
  This stream has been disconnected because your client was unable to keep up with us.
  ```

  ```
  This stream has been disconnected for operational reasons.
  ```

  Consentiamo che la consegna resti indietro per un certo periodo e disponiamo di un buffer di staging temporaneo per ciascun stream dalla nostra parte; ma se non recuperi, avviamo una disconnessione per permetterti di riconnetterti al punto corrente nel tempo. Tieni presente che questo può comportare una perdita di dati (per i dati presenti nel buffer al momento della disconnessione per buffer pieno).

  Questi eventi possono verificarsi in corrispondenza di grandi picchi di dati. In generale, consigliamo di utilizzare un processo di buffering per consumare rapidamente i dati, separato dal processo di elaborazione.

  Se sei un cliente enterprise che utilizza gli endpoint v1.1, puoi trovare ulteriori informazioni su come ottimizzare la tua App per prevenire disconnessioni di questo tipo nei nostri articoli sulla [connessione](/it/x-api/enterprise-gnip-2.0/powertrack-api#disconnections-explained) e sul consumo di dati in streaming [qui](https://developer.x.com/en/docs/tutorials/consuming-streaming-data) e [qui](/it/x-api/enterprise-gnip-2.0/powertrack-api#planning-for-high-volume-social-data-events).

  È disponibile una serie di strumenti per recuperare i Post persi a causa di una disconnessione, inclusi quelli elencati di seguito. Nota che i seguenti strumenti sono disponibili solo con gli endpoint v1.1 a livello di accesso enterprise.

  - [**Connessioni ridondanti**](/it/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - Con più connessioni, consuma lo stream da più server per evitare la perdita di dati quando una connessione viene interrotta.
  - [**Replay**](/it/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - Recupera i dati degli ultimi 5 giorni utilizzando uno stream separato.
  - [**Backfill**](/it/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - Riconnettiti entro 5 minuti e riparti da dove eri rimasto.
  - **Historical PowerTrack** - Recupera i dati dall’intero archivio di X.
</Accordion>

<div id="before-you-post-a-question">
  ### Prima di pubblicare una domanda
</div>

- Cerca nella documentazione per sviluppatori di X informazioni relative al tuo problema
- Cerca nel forum della community se altri sviluppatori hanno posto domande simili
- Leggi le linee guida del forum della community

<div id="when-you-post-a-question-make-sure-to-include-the-following-information">
  ### Quando pubblichi una domanda, assicurati di includere le seguenti informazioni
</div>

- Una descrizione del problema
- La chiamata API effettuata (includi le intestazioni, se possibile)
- La risposta restituita da X (includi eventuali messaggi di errore)
- Cosa ti saresti aspettato di ricevere
- Elenco dei passaggi eseguiti per la risoluzione del problema
- Elenco dei passaggi necessari per riprodurre il problema
- Se pertinente, l’intervallo temporale in cui si è verificato il problema
- Se pertinente, l’ID dell’App, l’ID del Post, ecc.
- Eventuali snippet di codice o screenshot rilevanti

Includi un solo argomento/domanda per post.

Se hai richieste di funzionalità o feedback, inviali tramite il [Modulo di feedback della Piattaforma Sviluppatori X](https://t.co/devfeedback).

Per problemi relativi alle policy, come la sospensione dell’App, contatta l’[assistenza policy](https://help.twitter.com/forms/platform).

Per problemi relativi a X, come l’accesso o il supporto dell’account, utilizza il [Centro assistenza X](https://help.twitter.com/en/contact-us).
