<div id="getting-started-with-the-mutes-lookup-endpoint">
  ## Guida introduttiva all'endpoint di lookup dei silenziati
</div>

Questa guida rapida ti aiuterà a effettuare la tua prima richiesta all'endpoint di lookup dei silenziati utilizzando Postman.

Visita il nostro repository GitHub [API X v2 sample code](https://github.com/xdevplatform/Twitter-API-v2-sample-code) se desideri vedere esempi di codice in diverse lingue.

<Note>
  **Prerequisiti**

  Per completare questa guida, avrai bisogno di un set di [chiavi e token](/it/resources/fundamentals/authentication) per autenticare la tua richiesta. Puoi generare queste chiavi e questi token seguendo questi passaggi:

  - [Registrati per un account sviluppatore](https://developer.x.com/en/apply-for-access) e attendi l'approvazione.
  - Crea un [Project](/it/resources/fundamentals/projects) e una [developer App](/it/resources/fundamentals/developer-apps) associata nel portale per sviluppatori.
  - Vai alla pagina “Keys and tokens” della tua App per generare le credenziali richieste. Assicurati di salvare tutte le credenziali in un luogo sicuro.
</Note>

<div id="steps-to-build-a-mutes-lookup-request">
  ### Passaggi per creare una richiesta di lookup dei silenzi
</div>

<div id="step-one-start-with-a-tool-or-library">
  #### Primo passaggio: iniziare con uno strumento o una libreria
</div>

Sono disponibili diversi strumenti, esempi di codice e librerie che puoi utilizzare per inviare una richiesta a questo endpoint, ma qui useremo Postman per semplificare il processo.

Per importare la raccolta dell’API X v2 in Postman, fai clic sul seguente pulsante:

<Button href="https://app.getpostman.com/run-collection/9956214-784efcda-ed4c-4491-a4c0-a26470a67400">
  Aggiungi API X v2 a Postman
</Button>

Una volta importata la raccolta API X v2 in Postman, vai alla cartella “Mutes” e seleziona “Mutes lookup”.
 

<div id="step-two-authenticate-your-request">
  #### Passaggio due: autenticare la richiesta
</div>

Per inviare correttamente una richiesta all’API X, devi verificare di avere le autorizzazioni necessarie. Per farlo con questo endpoint, devi autenticare la richiesta usando [OAuth 1.0a User Context](/it/resources/fundamentals/authentication) oppure [OAuth 2.0 Authorization Code con PKCE](it/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2).

In questo esempio utilizzeremo OAuth 1.0a User Context.

Devi aggiungere le tue chiavi e i tuoi token — in particolare API Key, API Secret Key, OAuth 1.0a User Access Token e OAuth 1.0a User Access Token Secret — in Postman. Puoi farlo selezionando l’ambiente denominato “API X v2” nell’angolo in alto a destra di Postman e aggiungendo chiavi e token nei campi “initial value” e “current value” (facendo clic sull’icona a forma di occhio accanto al menu a discesa dell’ambiente).

Se l’operazione è stata eseguita correttamente, queste variabili verranno importate automaticamente nella scheda Authorization della richiesta.
 

<div id="step-three-specify-a-user">
  #### Passaggio tre: Specificare un utente
</div>

Con questo endpoint devi specificare il tuo ID utente o l’ID di un utente autenticato per vedere chi è stato silenziato da te o dall’utente autenticato.

In Postman, vai alla scheda "Params" e inserisci l’ID dell’utente autenticato nella colonna "Value" del parametro id sotto “Path Variables” (in fondo alla sezione), assicurandoti di non includere spazi prima o dopo l’ID.

Sopra la sezione “Path Variables” noterai che sono disponibili “Query Params” opzionali. Per questo esempio, selezioneremo la variabile max\_results e imposteremo il valore su 5.

|     |     |     |
| :--- | :--- | :--- |
| **Key** | **Value** | **Parameter Type** |
| `id` | (il tuo ID utente) | Path |
| max\_results | 5   | Query |

<div id="step-four-identify-and-specify-which-fields-you-would-like-to-retrieve">
  #### Passaggio quattro: identifica e specifica quali campi desideri recuperare
</div>

Se fai clic sul pulsante "Send" dopo il passaggio tre, nella risposta riceverai i campi predefiniti dell’[oggetto utente](/it/x-api/fundamentals/data-dictionary#user): id, name e username.

Se desideri ricevere campi aggiuntivi oltre a id, name e username, devi specificarli nella richiesta usando i parametri [fields](/it/x-api/fundamentals/fields) e/o [expansions](/it/x-api/fundamentals/expansions).

Per questo esercizio, richiederemo tre insiemi aggiuntivi di campi da oggetti diversi:

1. Il campo aggiuntivo user.created\_at negli oggetti utente primari.
2. I campi predefiniti dell’oggetto dei Post fissati associati per gli utenti restituiti: id e text.
3. Il campo aggiuntivo tweet.created\_at negli oggetti Post associati.

In Postman, vai alla scheda "Params" e aggiungi la seguente coppia chiave:valore alla tabella "Query Params":

|     |     |     |
| :--- | :--- | :--- |
| **Key** | **Value** | **Returned fields** |
| user.fields | created\_at | user.created\_at |
| expansions | pinned\_tweet\_id | tweet.id, tweet.text |
| tweet.fields | created\_at | includes.tweets.created\_at |

Ora dovresti vedere un URL simile con il tuo ID utente al posto dell’ID di esempio, accanto al pulsante "Send":

`https://api.x.com/2/users/1324848235714736129/muting?user.fields=created_at&expansions=pinned_tweet_id&tweet.fields=created_at&max_results=5`

<div id="step-five-make-your-request-and-review-your-response">
  #### Passaggio cinque: invia la richiesta e verifica la risposta
</div>

Una volta configurato tutto, fai clic sul pulsante "Send" e riceverai una risposta simile al seguente esempio:

```
{
  "data": [
    {
      "username": "TwitterDev",
      "created_at": "2013-12-14T04:35:55.000Z",
      "id": "2244994945",
      "name": "Twitter Dev",
      "pinned_tweet_id": "1430984356139470849"
    }
  ],
  "includes": {
    "tweets": [
      {
        "created_at": "2021-08-26T20:03:51.000Z",
        "id": "1430984356139470849",
        "text": "Help us build a better Twitter Developer Platform!\n \nTake the annual developer survey &gt;&gt;&gt; https://t.co/9yTbEKlJHH https://t.co/fYIwKPzqua"
      }
    ]
  },
  "meta": {
    "result_count": 1
  }
}
```

<div id="step-six-paginate-through-your-results">
  #### Passaggio sei: eseguire la paginazione dei risultati
</div>

Potresti notare un oggetto meta alla fine della risposta. Se hai ricevuto un next\_token, significa che è disponibile un’altra pagina di risultati da recuperare. Per ottenere la pagina successiva, prendi il valore del campo next\_token e aggiungilo alla richiesta come valore di un ulteriore parametro di query pagination\_token.
 

|     |     |
| :--- | :--- |
| **Chiave** | **Valore** |
| pagination\_token | 1710819323648428707 |

Se invii la richiesta dopo aver aggiunto questo parametro, i cinque risultati successivi verranno restituiti con il payload seguente, poiché nel passaggio tre abbiamo specificato max\_results a 5. Puoi continuare a ripetere questo processo finché non vengono restituiti tutti i risultati, ma puoi anche usare il parametro max\_results per richiedere fino a 1000 utenti per richiesta, così da ridurre la necessità di paginare.
