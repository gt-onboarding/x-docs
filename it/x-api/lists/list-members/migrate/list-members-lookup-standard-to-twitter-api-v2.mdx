<div id="list-members-lookup-standard-v11-compared-to-x-api-v2">
  ### Ricerca dei membri delle liste: Standard v1.1 rispetto all’API X v2
</div>

Se hai lavorato con gli endpoint standard v1.1 [GET lists/members](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-members) e [GET lists/memberships](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-memberships), l’obiettivo di questa guida è aiutarti a comprendere somiglianze e differenze tra gli endpoint dei membri delle liste dello standard v1.1 e dell’API X v2.

- **Somiglianze**
  - Metodi di autenticazione
- **Differenze**
  - URL degli endpoint
  - Limiti di velocità
  - Requisiti di App e Progetto
  - Limiti sul numero di oggetti dati per richiesta
  - Formati dei dati di risposta
  - Parametri della richiesta

<div id="similarities">
  #### Somiglianze
</div>

**Autenticazione**

Entrambe le versioni dell’endpoint supportano sia [OAuth 1.0a User Context](/it/resources/fundamentals/authentication#oauth-1-0a-2) sia l’autenticazione [App only](/it/resources/fundamentals/authentication#oauth-2-0). Pertanto, se in precedenza utilizzavi uno degli endpoint standard v1.1 per i membri delle Liste, puoi continuare a usare lo stesso metodo di autenticazione passando alla versione dell’API X v2.

A seconda della libreria o del pacchetto di autenticazione scelto, l’autenticazione App only è probabilmente il modo più semplice per iniziare e può essere configurata con una semplice intestazione della richiesta. Per sapere come generare un token di accesso App only, consulta [questa guida su App only](/it/resources/fundamentals/authentication#bearer-token-also-known-as-app-only).

<div id="differences">
  #### Differenze
</div>

**URL degli endpoint**

- Endpoint standard v1.1:
  - GET https://api.x.com/1.1/lists/members.json
    (Recupera i membri di una Lista specifica)
  - GET https://api.x.com/1.1/lists/memberships.json
    (Recupera le Liste di cui un utente è membro)
- Endpoint API X v2:
  - GET https://api.x.com/2/lists/:id/members
    (Recupera i membri di una Lista specifica)

  - GET https://api.x.com/2/users/:id/list\_memberships
    (Recupera le Liste di cui un utente è membro)

**Limiti di rate**

|     |     |
| :--- | :--- |
| **Standard v1.1** | **API X v2** |
| /1.1/lists/members.json<br /><br />900 richieste per finestra di 15 minuti con OAuth 1.0a User Context<br /><br />15 richieste per finestra di 15 minuti con sola App | /2/lists/:id/members<br /><br />900 richieste per finestra di 15 minuti con OAuth 1.0a User Context<br /><br />900 richieste per finestra di 15 minuti con OAuth 2.0 Authorization Code con PKCE<br /><br />900 richieste per finestra di 15 minuti con sola App |
| /1.1/lists/memberships.json<br /><br />15 richieste per finestra di 15 minuti con OAuth 1.0a User Context<br /><br />15 richieste per finestra di 15 minuti con sola App | /2/users/:id/list\_memberships<br /><br />15 richieste per finestra di 15 minuti con OAuth 1.0a User Context<br /><br />15 richieste per finestra di 15 minuti con OAuth 2.0 Authorization Code con PKCE<br /><br />15 richieste per finestra di 15 minuti con sola App |

**Requisiti di App e Progetto**

Gli endpoint dell'API X v2 richiedono l'uso di credenziali di una [developer App](/it/resources/fundamentals/developer-apps) associata a un [Project](/it/resources/fundamentals/projects) quando si autenticano le richieste. Tutti gli endpoint dell'API X v1.1 possono usare credenziali di App autonome o di App associate a un progetto.

**Limiti di oggetti dati per richiesta**

L'endpoint standard v1.1 /1.1/lists/members consente di restituire fino a 5000 utenti per richiesta. I nuovi endpoint v2 consentono di restituire fino a 100 utenti per richiesta. Per impostazione predefinita vengono restituiti 100 oggetti utente; per modificare il numero di risultati è necessario passare il parametro di query max\_results= con un numero tra 1 e 100; è quindi possibile passare il next\_token restituito nel payload di risposta al parametro di query pagination\_token nella richiesta successiva.

Inoltre, l'endpoint /1.1/lists/memberships consente di restituire fino a 1000 Liste per richiesta. Con la sostituzione v2, l'endpoint consente fino a 100 Liste per richiesta. Per impostazione predefinita vengono restituiti 100 oggetti Lista; utilizzare i parametri di query max\_results= e pagination\_token allo stesso modo di /1.1/lists/members per modificare il numero di risultati.

**Formato dei dati di risposta**

Una delle differenze maggiori tra le versioni degli endpoint standard v1.1 e API X v2 è il modo in cui si selezionano i campi restituiti nel payload.

Per gli endpoint standard, molti campi della risposta sono inclusi per impostazione predefinita, con la possibilità di usare parametri per indicare quali campi aggiuntivi o insiemi di campi includere nel payload.

La versione API X v2 /users/:id/list\_memberships fornisce per impostazione predefinita i campi id e name della Lista. Per richiedere campi o oggetti aggiuntivi, è necessario usare i parametri [fields](/it/x-api/fundamentals/fields) ed [expansions](/it/x-api/fundamentals/expansions). Qualsiasi campo della Lista richiesto da questo endpoint verrà restituito nell'oggetto Lista principale. Qualsiasi oggetto e campo espanso verrà restituito in un oggetto includes all'interno della risposta. È quindi possibile associare gli oggetti espansi all'oggetto Lista principale confrontando gli ID presenti sia nell'oggetto principale sia in quello espanso.

Ecco alcuni esempi di possibili campi ed espansioni della Lista:

- created\_at

- follower\_count

- member\_count

- owner\_id

- description

- private

|     |     |
| :--- | :--- |
| **Endpoint** | **Espansione** |
| /2/lists/:id/members | pinned\_tweet\_id |
| /2/users/:id/list\_memberships | owner\_id |

Ti invitiamo a leggere ulteriori dettagli su questi nuovi parametri nelle rispettive guide, oppure consultando la nostra guida su [come usare fields e expansions](/it/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions). 

Abbiamo anche preparato una [guida alla migrazione del formato dei dati](/it/x-api/migrate/data-format-migration) che può aiutarti a mappare i campi standard v1.1 ai più recenti campi v2. Questa guida ti fornirà anche la specifica expansion e il parametro fields che dovrai includere nella tua richiesta v2 per ottenere campi specifici. 

Oltre alle modifiche relative al modo in cui richiedi determinati campi, l’API X v2 introduce anche nuovi design JSON per gli oggetti restituiti dalle API, inclusi gli oggetti [Post](/it/x-api/fundamentals/data-dictionary#tweet) e [user](/it/x-api/fundamentals/data-dictionary#user).

- Al livello root del JSON, gli endpoint standard restituiscono oggetti Post in un array statuses, mentre l’API X v2 restituisce un array data. 

- Invece di fare riferimento a “statuses” Retweeted e Quoted, il JSON dell’API X v2 fa riferimento a Retweeted e Quoted Tweets. Molti campi legacy e deprecati, come contributors e user.translator\_type, vengono rimossi. 

- Invece di usare sia favorites (nell’oggetto Post) sia favourites (nell’oggetto user), l’API X v2 usa il termine like. 

- X adotta la convenzione secondo cui i valori JSON privi di valore (ad esempio null) non vengono inclusi nel payload. Gli attributi di post e user sono inclusi solo se hanno valori non null.
