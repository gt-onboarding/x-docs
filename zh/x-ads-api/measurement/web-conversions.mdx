### 前提条件 

<div id="ads-api-access-new-applications">
  #### **Ads API 访问——新申请**
</div>

- 使用 Conversion API 的首要前提是拥有[开发者账号](https://developer.x.com/en/portal/dashboard)和[Ads API 访问权限](https://developer.x.com/en/docs/twitter-ads-api/getting-started)。完整流程见[Ads API 入门](https://developer.x.com/en/docs/twitter-ads-api/getting-started)指南。请注意以下事项：

**步骤 1：开发者账号**

- 申请[开发者账号](https://developer.x.com/en/portal/dashboard)时，选择订阅我们的某个[方案](https://developer.x.com/en/portal/products/)，可即时获批。
- **注意**：作为最佳实践，我们强烈建议使用贵公司的官方 X 账号来创建开发者账号并申请 Ads API 访问权限。如果开发者账号关联的是个人开发者账号，后续将无法转移相关凭据。为便于持续管理，最好归属公司账号，并按需使用[多用户登录](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html)。否则，至少应将账号从默认设置更新为自定义（页头图片、头像、个人简介与简介 URL），并启用双重身份验证。

**步骤 2：Ads API 申请**

- 请确保为 Ads API 申请准备好正确的**应用** **ID**。应用 ID 可在[开发者平台](https://developer.x.com/en/portal/dashboard)的 项目与应用 下找到。示例：16489123
- 通过联系您的 X 客户代表申请 Ads API 访问权限。

<div id="ads-api-access-existing-applications">
  #### **广告 API 访问 — 现有应用**
</div>

- 如果你已有一个正在使用中的广告 API 应用，则该应用及现有访问令牌均可用于 Conversion API。

<div id="access-tokens">
  #### **访问令牌**
</div>

- 属于 Ads API 应用所有者的用户账号（handle）的用户访问令牌可直接在开发者门户生成并获取。由于该令牌用于你自己的 X 账号，因此称为“个人访问令牌”。有关身份验证和开发者门户的总体信息，请参见[此处](/zh/resources/fundamentals/authentication/overview)。
- 针对除该 Ads API 应用所有者之外的其他用户账号，其用户访问令牌必须通过三方（3-legged）OAuth 流程生成。使用三方 OAuth 生成访问令牌的选项包括：
  - [使用 twurl 工具进行基于 Web 授权的命令行方式](https://github.com/twitter/twurl#getting-started)
  - [基于 PIN 授权的命令行方式](https://developer.x.com/resources/fundamentals/authentication/pin-based-oauth)
  - [实现三方 OAuth 模式的自定义 Web 流程](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)
- 与 Conversion API 搭配使用的任何用户令牌，必须对应具有 AD\_MANAGER 或 ACCOUNT\_ADMIN 访问级别的用户，可通过 [authenticated\_user\_access](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/authenticated-user-access) 端点进行检查。
- **注意：** 令牌本身（按上述方式创建后）可以共享给不具有 AD\_MANAGER 或 ACCOUNT\_ADMIN 访问级别的用户以供使用。

<div id="creating-the-conversion-api-event">
  ### 创建 Conversion API 事件
</div>

要使用 Conversion API，您需要在 Ads Manager 中创建一个新的转化事件，或复用已与 X Pixel 一起创建并使用的现有事件。若希望在 Pixel 与 Conversion API 事件之间进行去重，必须使用您为 Pixel 创建的现有事件。 

<div id="option-1-using-an-existing-conversion-event-in-ads-manager">
  ##### 选项 1：在 Ads Manager 中使用现有的转化事件
</div>

如果你想沿用已与 X 像素配合使用的现有事件，可以这样做，你需要从该事件中获取 Event ID。若你对同一事件同时使用像素和 Conversion API，请确保在 Pixel 代码片段与 Conversion API 请求中都设置去重键（在请求中作为 `conversion_id`），以在同一事件的 Pixel 与 Conversion API 之间进行去重。更多信息请参见 d. 测试事件与去重 章节。 

<div id="option-2-creating-a-new-conversion-event-in-ads-manager">
  ##### 选项 2：在 Ads Manager 中创建新的转化事件：
</div>

在创建事件之前，务必先在 Events Manager 中创建一个 Event Source。要验证你的账户是否已添加 Event Source（X Pixel），请前往 Events Manager，查看左侧菜单中是否有 X Pixel。

如果你尚未添加 Event Source，请按以下步骤创建 Event Source。

1. 访问 ads.x.com
2. 导航到左上角的 **Tools**，点击 **Events Manager**
3. 在右上角选择 Add event source 以 **Add an event source**，如果你左侧侧边栏尚无 X Pixel 事件来源
   1. X Pixel 事件来源的 ID 即为你的 **Pixel ID**

现在你已有一个 Event Source 和 Pixel ID。你需要在该 Event Source 中为希望跟踪的转化创建一个事件：

1. 在 X Pixel 事件来源内，选择右侧的 **Add events**
2. 选择使用 **Conversion API** 安装
3. 你将看到此事件在 API 中会使用到的 **Pixel ID** 和 **Event ID**
   1. 该事件的 ID 即为你的 Event ID
4. 点击 Save，你的转化事件就已创建并可使用

### 为转化事件准备标识符 

目前我们需要至少传递一个标识符，例如点击 ID（`twclid`）、电子邮件地址或电话号码。若使用 IP 地址或用户代理，则必须再提供第二个标识符，才能进行正确的转化匹配。

传递更多标识符将提升转化匹配率。

|     |     |     |
| :--- | :--- | :--- |
| 客户匹配字段 | 格式 | 是否需要哈希处理？ |
| X Click ID | 由 X 生成（[了解更多](https://business.x.com/en/help/campaign-measurement-and-analytics/conversion-tracking-for-websites.html#faq)） | 否  |
| 电子邮件地址 | 去除首尾空格 | 需要（SHA256） |
| 电话号码 | 按照 [E.164 标准](https://en.wikipedia.org/wiki/E.164) | 需要（SHA256） |

|     |     |     |
| :--- | :--- | :--- |
| IP 地址 | 去除首尾空格 | 否  |
| 用户代理 | 去除首尾空格 | 否  |

#### 1. 准备 X Click ID 标识符 

建议在转化请求中始终包含 Click ID

在用户进入目标网站后，如可用，应从查询字符串参数 twclid 中解析出 Click ID。 

基础 JavaScript 代码示例：

```js
var queryString = document.location.search;
if (queryString.has('twclid') {
  twitterClickID = getParam(queryString, 'twclid');
  // 推荐的后续步骤：记录日志，写入本地存储
}
```

建议如下： 

1. 当 URL 查询参数中存在 twclid 值时，务必进行解析。

2. 将该数据与相关的表单字段或转化事件信息一并存储。

将 Click ID 关联到转化事件和工作流信息，可支持以下场景：批处理、用于分析并基于多条网站导航路径创建转化事件的算法，以及批量上传。

Event Source URL 应进行[URL 编码](https://en.wikipedia.org/wiki/Percent-encoding)，用于表示触发该事件的网页。

#### 2. 准备电子邮件标识符 

支持的客户匹配字段可以发送，但应先规范化，并在需要时进行哈希处理以保护隐私。

信息必须使用 SHA256 进行哈希，且不得使用盐值。 

例如，对于电子邮件地址 test@x.com，应以其哈希值发送给我们：d360d510a224510f373931ce2d6215a799f5a9c1cef221b0149b6b6b50cced62。

#### 3. 准备电话号码标识符 

电话号码应按 [E164 标准](https://en.wikipedia.org/wiki/E.164) 传入，且必须使用 SHA256 进行无盐哈希。 

例如，对于美国电话号码 +11234567890，应以哈希值的形式发送给我们：1fa6b8d986d9b9cd01bf36951815158bbde9f520c0567c835dfe34783d0a4231。

<div id="4-prepare-ip-address-identifier">
  #### 4. 准备 IP 地址标识符
</div>

IP 地址必须与另一个标识符（twclid、电子邮件地址、电话号码或用户代理）一并传递。该标识符无需进行哈希处理。

该值采用点分十进制表示法，由四个用点分隔的数字组成。例如，某个用户的 IP 地址可能是 8.25.197.25。

<div id="5-prepare-user-agent-identifier">
  #### 5. 准备 User Agent 标识符
</div>

User Agent 需要与另一种标识符（twclid、电子邮件地址、电话号码或 IP 地址）一同传递。该标识符无需进行哈希处理。

此标识符使服务器能够识别发出请求的用户代理对应的应用、操作系统、供应商和/或版本。例如，该值可传为：Mozilla/5.0 (Macintosh; Intel Mac OS X 10\_15\_7) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/127.0.0.0 Safari/537.36。

<div id="constructing-the-conversion-event-request">
  ### 构造转化事件请求
</div>

`POST: version/measurement/conversions/:pixel_id`

为指定广告账号发送转化事件。请检查响应状态码以确认成功（HTTP 200 OK）。建议在返回错误码时启用重试机制并记录基础日志。

有关该端点的 URL 和 POST 请求体参数的详细信息，请参阅 [API 参考](https://developer.x.com/en/docs/twitter-ads-api/measurement/api-reference/conversions) 部分。

<div id="rate-limit">
  ### 速率限制
</div>

速率限制为每个账户、每 15 分钟内 60,000 个事件。

请注意，您的服务器端代码可能需要在此次请求调用之外实现逻辑，包括：

1. 对用户行为进行埋点（日志记录），以便按事件发送正确的转化数据
2. 实现必要的逻辑，以过滤掉已行使相关隐私选择的用户的转化事件——例如，他们在广告主网站上选择退出跟踪或拒绝出售其个人信息
3. 与事件触发器和页面进行集成，以捕获事件并发送转化

</div>

当你的事件成功接收到转化后，12 至 24 小时内，Ads Manager 的“转化跟踪”页面上“Single event web tag”的状态应显示为 TRACKING。通过 Conversion API 发送转化不会影响正在投放中的广告系列。

<div id="duplication-between-pixel-and-conversion-api">
  ### Pixel 与 Conversion API 之间的去重
</div>

如果你希望在 Pixel 与 Conversion API 请求之间进行去重，我们提供 conversion\_id 作为去重键。去重仅在事件级别生效。换言之，要在 Pixel 与 CAPI 请求之间实现去重，广告主必须在两者请求中使用同一事件，并使用相同的 conversion\_id。去重仅适用于在 48 小时内接收到的事件。

<div id="conversion-tracking-overview">
  ## 转化跟踪（概览）
</div>

<div id="summary">
  ### 摘要
</div>

转化跟踪可衡量 X 用户在查看并与您在 X 上的广告互动后完成目标行为的人数。它让你能够评估哪些广告系列驱动了网站访问、注册和购买等行为。借此，广告主可获得站外效果衡量能力，更好地理解其直接响应型广告的表现，从而以更具成本效益的方式获客。

通过转化标签，广告主可以跟踪用户转化，并将其归因到 X 上的相关广告系列。这使他们能够具备优化可见性，以便将广告系列优化至满足其每次获客成本（CPA）目标。

广告主可通过转化跟踪衡量多种网站行为，并可根据其广告系列所要推动的目标选择一种或多种：

- 站点访问：用户访问广告主网站的落地页
- 购买：用户在广告主网站完成某产品或服务的购买
- 下载：用户从广告主网站下载文件，例如白皮书或软件包
- 注册：用户注册广告主的服务、新闻通讯或电子邮件订阅
- 自定义：兜底类别，适用于不属于上述类别的自定义行为

X 的转化跟踪为广告主提供完整的转化归因视图。与客户可能曾用于替代 X 自身转化跟踪能力的第三方测量系统（例如与第三方跟踪标签配对的唯一点击 URL）相比，X 的转化标签还可跟踪归因于中上层漏斗互动（如推文展开、转推、收藏、回复、关注）以及展示所带来的转化。

<div id="faq">
  ### 常见问题
</div>

<AccordionGroup>
  <Accordion title="转化跟踪标签如何工作？">
    首先，广告主会创建一个转化标签（由 X 提供的一段代码），并将其放置在自己的网站上。这样，当用户完成指定操作时，标签即可用来衡量转化。

    随后，用户在 X 客户端看到广告主的广告，被引导至广告主的网站并执行已被打标签的操作。如果用户在广告主于标签设置时指定的归因窗口内完成该操作，标签会识别该用户此前与 X 广告发生过互动。然后标签会“触发”，即向 X 的服务器发送通知，以便将此次转化归因到带来该转化的广告。
  </Accordion>

  <Accordion title="在广告系列设置流程中，是否可以选择本次广告系列使用哪些跟踪像素？">
    不可以，我们的产品不支持将特定转化标签绑定到特定广告系列。相反，一旦设置好标签，系统会自动记录是哪则广告为某个标签带来了转化。
  </Accordion>

  <Accordion title="转化标签的默认归因窗口设置是什么？">
    默认的展示后归因窗口：1 天

    默认的互动后归因窗口：14 天

    这些默认设置可在转化标签设置期间或标签创建后的任何时间进行更改。互动后归因窗口可选 1、7、14、30、60、90 天；展示后归因窗口可选无、1、7、14、30、60、90 天。
  </Accordion>

  <Accordion title="有哪些有效的 DR 创意与策略能更好地带动转化？">
    尽管各客户的目标、情况和策略不同，以下做法在参与转化跟踪 alpha 或 beta 的客户中被证明有效：

    **创意：**

    - **优惠**：将折扣、促销或免运费等优惠与推广推文（Promoted Tweet）搭配，提升目标行动的吸引力
    - **抽奖与竞赛**：尤其对知名品牌，抽奖和竞赛能带动转化
    - **推文文案实验**：测试全大写与小写（如 FREE 与 free、NOW 与 now）
    - **截止日期**：设置截止时间以促使用户立即行动（优惠至 12 月 12 日！）
    - **加入更具吸引力的图片**：值得测试在推文创意中使用强视觉冲击的图片是否有助于提升转化；效果可能因客户的具体产品/服务而异

    **定向：**

    - @handle 定向与兴趣分类定向：将推文文案与 @handle 与预期受众高度匹配可促进转化
    - 使用小众但高量级的关键词：在演出垂直中，使用与艺人/音乐人相关的关键词（如其姓名）效果显著
    - 定制受众：同时使用 TA Web 与转化跟踪的客户，其 CPA 低于使用其他定向方式的对照组

    你的广告系列分层越细，转化报告的可操作性就越强。比如，优化包含 4 个关键词的广告系列，远比优化包含 50 个关键词的广告系列更容易。
  </Accordion>
</AccordionGroup>

<div id="error-handling-and-explanation">
  ### 错误处理与说明
</div>

只有当单个请求中所包含的所有转化均无任何错误时，请求才会成功。若任一转化出现错误，端点将返回所有适用错误的列表。

<div id="x-ads-api-error-codes-overview">
  ### X 广告 API 错误代码概览
</div>

以下是 X 广告 API 中错误代码的完整列表：

[https://developer.x.com/en/docs/twitter-ads-api/response-codes](https://developer.x.com/en/docs/twitter-ads-api/response-codes)

成功的 Conversion API 响应会返回 200 段的 HTTP 状态码，并附带包含所请求对象的 JSON 负载。

<div id="when-there-is-a-500-series-http-code-its-related-to-a-server-issue-instead-of-your-request-or-account-set-up-please-check-the-x-api-status-page-or-the-developer-community-forum-in-case-others-are-having-similar-issues">
  #### **当出现 500 系列 HTTP 状态码时**，问题与服务器有关，而非你的请求或账户设置。请查看 [X API 状态页](https://api.twitterstat.us/)或[开发者社区论坛](https://devcommunity.x.com/)，以了解是否有他人遇到类似问题。
</div>

<div id="when-there-is-a-400-series-http-code-the-common-cases-are">
  #### 当出现 400 系列 HTTP 状态码时，常见情况包括
</div>

- 400 Bad Request（请求不符合规范）

- 403 Forbidden（与该开发者账号的 API 访问权限相关的问题）

- 404 Not Found（该端点的 URL 或参数可能不正确）

<div id="conversion-api-error-codes">
  ### Conversion API 错误码
</div>

<div id="400-bad-request-scenarios">

</div>

<div id="401-unauthorized">
  ### 401 未授权
</div>

**原因**：缺少或不正确的身份验证凭据

**解决方案**：请按照[设置文档](https://developer.x.com/en/docs/twitter-ads-api/measurement/guides/conversion-api)中的身份验证步骤操作，选择以下三种身份验证方式之一：

对于不属于 Ads API 应用拥有者的账号所使用的用户访问令牌（User Access Tokens），必须通过三方 OAuth 流程生成。使用三方 OAuth 生成访问令牌（Access Token）的方式包括：

- [使用 twurl 工具进行基于 Web 的授权（命令行）](https://github.com/twitter/twurl#getting-started)

- [使用基于 PIN 的授权（命令行）](https://developer.x.com/resources/fundamentals/authentication/pin-based-oauth)

- [实现三方 OAuth 模式的自定义网页流程](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)

用于 Conversion API 的任何用户令牌必须对应具有 AD\_MANAGER 或 ACCOUNT\_ADMIN 访问级别的用户，可通过 [authenticated\_user\_access](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/authenticated-user-access) 端点进行检查。

### 403 访问被拒绝 

| 原因 | 类型 | 错误消息 |

| 事件 ID 不属于与该像素 ID（UWT ID）关联的所提供账户 | 404 未找到 | event\_id \<event\_id> 不属于所提供的账户 |

`POST version/measurement/conversions/:pixel_id`

为单个 Event Tag ID 发送网站转化事件。

应检查响应状态码以确认成功（HTTP 200 OK）。建议在返回错误码时实现重试机制并进行基本日志记录。
