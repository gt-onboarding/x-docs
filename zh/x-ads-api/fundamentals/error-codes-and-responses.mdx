<div id="typical-response-structure">
  ## 典型响应结构
</div>

成功的响应以 200 系列 HTTP 状态码表示，并包含基于 JSON 的负载，其中包括被请求、创建、修改或删除的对象，以及服务器对你的请求的解析。

如果你发送的请求成功，响应中会包含一个用于回显请求的 `request` 节点。

_示例：_ [GET accounts/abcdefg/campaigns?with\_deleted=true](/zh/x-ads-api/campaign-management#campaigns)

```json
{
  /* the data of your response... */,
  "request": {
    "params": {
      "account_id": "abcdefg",
      "with_deleted": "true"
    }
  }
}
```

JSON 响应中的 `data` 字段包含与所用资源相关的具体对象。当响应可能包含一个或多个结果时，`data` 节点为 JSON 数组；当响应中至多只有一个结果时，它将返回为 JSON 对象。在极少数情况下，你可能会看到通常应包含集合的响应却返回了一个哈希映射。在这种情况下，可将该单个哈希映射视为与 `type` 字段所指定类型相同的对象。

<div id="error-response-structure">
  ## 错误响应结构
</div>

错误响应会返回非 200 段的 HTTP 状态码。通常会附带 JSON 响应，但某些错误可能返回不同类型的响应体。在无法解析响应结构时，应以 HTTP 状态码的核心含义为准。例如，你可能偶尔会看到带有 HTML 响应体的 HTTP 404。在这种情况下，可以认为内容不存在（HTTP 404 表示“未找到”）。

典型的错误响应与成功响应具有类似的结构。错误的性质会在响应的 `errors` 节点中传达。`errors/code` 节点会给出一个 CAPS\_CASE 的常量错误码，你可以以编程方式使用该错误码来做出处理决策。`errors/message` 节点通常会给出该错误的英文可读描述。还可能附加其他字段，以提供更细粒度的错误信息。

```json
{
  "errors": [
    {
      "parameter": "start_time",
      "details": "invalid date",
      "code": "INVALID_PARAMETER",
      "value": "",
      "message": "Expected time, got \"\" for start_time"
    }
  ],
  "request": {
    "params": {
      "account_id": "hkk5"
    }
  }
}
```

在上述示例中，向某个分析端点发起的请求为 `start_time` 参数提供了无效值。对于包含无效参数的请求，其 `errors/code` 为 `INVALID_PARAMETER`。
