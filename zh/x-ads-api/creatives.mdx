<div id="overview">
  ## 概览
</div>

创意指在广告系列中可被推广的任何实体。帖子可以包含文本、图片、GIF、视频或卡片。卡片可以包含图片或视频。

图片、GIF 或视频类创意可通过 [POST media/upload](/zh/x-api/media/upload-media#post-media-upload)（仅支持图片的简易上传端点）或 POST media/upload（分片）端点进行上传。上传后可将其添加到

卡片：

- POST accounts/:account\_id/cards
  推文：
- POST accounts/:account\_id/tweets - 若要将卡片添加到推文，请使用 card\_uri 参数。
  定时推文：
- POST accounts/:account\_id/scheduled\_tweets

有关卡片的更多信息，请参见 Cards 页面。Promoted Video 页面提供了将视频与卡片或推文关联的详细说明。

<div id="cards">
  ### 卡片
</div>

X 广告 API 支持多种可用于推文的卡片类型，随后可在广告活动中推广。**注意**：一旦发布为推文，卡片详情将向公众公开，可能包含有关卡片所有者的信息。

<div id="image">
  #### 图片
</div>

以下图片规范适用于 [Cards](/zh/x-ads-api/creatives#cards-2) 中使用的素材。图片大小必须不超过 3MB，且宽度至少为 800px。此外，我们支持以下宽高比：

- 网站：1:1 和 1.91:1
- 图片应用下载：1:1 和 1.91:1
- 投票：1.91:1
- 图片会话：1.91:1
- 图片私信：1.91:1

我们支持以下图片格式：.bmp、.jpeg、.png。

<div id="video">
  #### 视频
</div>

以下视频规范适用于用于[卡片](/zh/x-ads-api/creatives#cards-2)的素材。我们支持以下宽高比：

- 视频网站：16:9 和 1:1
- 视频应用下载：16:9 和 1:1
- 投票：16:9
- 视频会话：16:9
- 视频私信：16:9

<div id="promoted-video">
  ### 推广视频
</div>

本文档简要概述如何通过 Ads API 上传并推广视频。

Ads API 支持在 [Tweets](/zh/x-ads-api/creatives#tweets-2) 中使用推广视频，以及以下卡片类型：

- [视频网站](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [视频应用下载](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [视频对话](/zh/x-ads-api/creatives#video-conversation-cards)

首先，使用 [POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html) 端点上传视频。然后使用 `media_id`，通过 POST accounts/:account\_id/videos 端点将该视频关联到广告账号。视频的 `id`（有时称为 `media_key`）将在后续请求中使用。该字符串以整数开头，后跟下划线，并以一个长整型值结尾。例如：`13_875943225764098048`。

<div id="promoted-video-in-tweets">
  #### 推广视频（在推文中）
</div>

要创建推文，请使用 [POST accounts/:account\_id/tweet](/zh/x-ads-api/creatives#get-accounts-account-id-tweets) 端点，并提供该视频的 `id`。在此步骤中，你还可以填写视频标题、描述和行动号召（CTA）。这些内容将对用户可见。

<div id="promoted-video-in-cards">
  #### 卡片中的推广视频
</div>

“视频应用下载”卡片和“视频对话”卡片支持添加海报图片。使用 [POST media/upload](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload.html) 端点上传一张用于这些卡片的图片。

使用以下端点之一创建卡片：

- [POST accounts/:account\_id/cards/video\_website](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [POST accounts/:account\_id/cards/video\_app\_download](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [POST accounts/:account\_id/cards/video\_conversation](https://devcommunity.x.com/t/ads-api-version-11/168814)

使用视频的 `id`，以及可选的图片 `media_id`（用于海报）。

最后，使用 POST accounts/:account\_id/tweet 端点创建推文。通过 `card_uri` 参数将卡片附加到推文。

<div id="general-information">
  #### 常规信息
</div>

有关通过 API 上传视频的详细指南，请参阅[视频上传指南](https://developer.x.com/en/docs/media/upload-media/uploading-media.html)。

视频也可以作为贴片广告素材进行推广。请参阅[视频观看贴片目标指南](/zh/x-ads-api/campaign-management#video-views-preroll-objective)以获取详细说明。

- （截至 2015-10-22）在上传用于推广内容的视频时，所有发送至 [POST media/upload（分片）](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html) 端点的 `INIT` 命令请求都必须将 `media_category` 参数设置为 `amplify_video`。使用此新参数可确保视频以异步方式预处理并为推广内容做好准备。上传视频后，可使用 `STATUS` 命令检查异步处理是否完成。
- 当前允许的推广视频时长上限为 10 分钟，文件大小不超过 500MB。
- 上传的视频应为 mp4 或 mov 格式。
- 上传的视频通常处理较快，但处理时间可能因视频时长和文件大小而异。
- 上传的海报图片应为 png 或 jpg 格式。对纵横比或尺寸没有硬性要求，但海报图片会调整以适配视频播放器。

<div id="scheduled-tweets">
  ### 预定发布的推文
</div>

<div id="introduction">
  #### 简介
</div>

“定时推文”允许广告主或用户创建一条可安排在未来某个时间发布的推文。除了能够创建和管理这些推文外，API 还支持将这些推文关联到某个 line item，以便在推文发布后进行推广。这样，广告主可以提前创建原生推文，并在任何关键节点前规划其广告素材。例如，安排一条推文创意在新产品发布公告发布的同时即时上线。

Scheduled Tweets API 端点所提供的全部功能如下：

- 创建、修改和查看新定时的推文
- 将某条定时推文关联到某个 line item
- 查询和管理现有的定时推文
- 当某条定时推文上线后，检索上线推文的 `id`

<div id="api-endpoints">
  #### API 端点
</div>

与上述功能相关的所有端点如下：

  #### 预定推文管理
</div>

- [POST accounts/:account\_id/scheduled\_tweets](/zh/x-ads-api/creatives#post-accounts-account-id-scheduled-tweets)（创建新的预定推文）
- [PUT accounts/:account\_id/scheduled\_tweets/:scheduled\_tweet\_id](/zh/x-ads-api/creatives#example-request-39)（修改现有预定推文）
- [DELETE accounts/:account\_id/scheduled\_tweets/:scheduled\_tweet\_id](/zh/x-ads-api/creatives#example-request-40)（使用其 `id` 删除预定推文）
- [GET accounts/:account\_id/scheduled\_tweets/preview/:scheduled\_tweet\_id](/zh/x-ads-api/creatives#scheduled-tweets-2)（预览现有预定推文）

<div id="scheduled-promoted-tweets">
  #### 计划投放的推广推文
</div>

- [GET accounts/:account\_id/scheduled\_promoted\_tweets](/zh/x-ads-api/campaign-management#get-accounts-account-id-scheduled-promoted-tweets)（获取所有计划投放的推广推文列表）
- [GET accounts/:account\_id/scheduled\_promoted\_tweets/:scheduled\_promoted\_tweet\_id](/zh/x-ads-api/campaign-management#get-accounts-account-id-scheduled-promoted-tweets-scheduled-promoted-tweet-id)（根据其 `id` 查询指定的计划投放推广推文）
- [POST accounts/:account\_id/scheduled\_promoted\_tweets](/zh/x-ads-api/campaign-management#post-accounts-account-id-scheduled-promoted-tweets)（创建新的计划投放推广推文）
- [DELETE accounts/:account\_id/scheduled\_promoted\_tweets/:scheduled\_promoted\_tweet\_id](/zh/x-ads-api/campaign-management#delete-accounts-account-id-scheduled-promoted-tweets-scheduled-promoted-tweet-id)（根据其 `id` 删除现有的计划投放推广推文）

<div id="scheduled-tweet-view">
  #### 预定推文视图
</div>

- [GET accounts/:account\_id/scheduled\_tweets/preview/:scheduled\_tweet\_id](/zh/x-ads-api/creatives#scheduled-tweets-2)（查看现有的预定推文）

由于预定推文与“实时”推文属于相互独立的实体，对这类推文进行创建或编辑时会运行两套不同的验证规则。第一套验证规则在创建预定推文时运行，具体包括：

<div id="scheduled-tweet-create">
  #### 预定推文创建：
</div>

- 验证已认证用户是否有权为指定的 @handle 创建有机推文。若要创建“仅用于推广”的推文，已认证用户必须是该账户的用户，并且拥有[Tweet composer permissions](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access)。
- 验证在距 scheduled\_at 时间前后 15 分钟的时间窗口内，待创建的预定推文不超过 30 条。出现 SCHEDULED\_TWEET\_LIMIT\_EXCEEDED 错误，表示在同一未来的 15 分钟时间段内安排了过多的预定推文。广告主需要删除现有的预定推文，或将 scheduled\_at 时间适当前后调整。

<div id="scheduled-tweet-goes-live">
  #### 预定推文进入“上线”状态：
</div>

- 这些验证规则会在 scheduled\_at 时间运行，与通过 API 创建常规推文时应用的规则相同。例如，如果预定推文同时包含图片和 GIF，则该预定推文不会上线，且 scheduled\_status 将被设置为 FAILED

<div id="workflow">
  #### 工作流程
</div>

**创建新的定时推文**

可以使用 [POST accounts/:account\_id/scheduled\_tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets) 端点创建新的定时推文。该端点要求提供以下参数：`scheduled_at` 时间；如果推文不包含任何媒体实体，还需提供推文 `text`。此外，该端点还提供一些可选项，允许你通过 `as_user_id` 参数代表另一个 @handle 创建定时推文，并可添加卡片（`card_uri`）和媒体（`media_ids`）。注意，推文只能包含同一类型的实体，即视频、GIF 或图片之一。`nullcast` 参数用于控制该推文是否为“仅推广”推文。所有新创建的定时推文默认均为“仅推广”（`nullcast=true`）。如果将 `nullcast=false`，则会创建一条自然定时推文。

成功创建定时推文后，响应会包含一个 `id` 字段，即该定时推文本身的唯一标识符。除了该字段外，还会返回一个名为 `tweet_id` 的字段。该字段初始为 `null`，当推文发布后，会填充为“已发布”推文的 ID。

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_tweets" -d 'scheduled_at=2017-12-24T23:59:00Z&text=Happy Holidays!'
```

这将创建如下定时推文：

```json
{
  "request": {
    "params": {
      "text": "Happy Holidays!",

    }
  },
  "data": {
    "completed_at": null,
    "id_str": "917507899668099072",
    "text": "Happy Holidays!",
    "user_id": "3271358660",
```

**将定时推文关联到一个线项**

虽然定时推文可用于创建“自然推文”，我们也允许合作伙伴创建“仅推广”（`nullcast=true`）推文，这两者中的任意一种都可以关联到一个线项。为此，我们还提供了 [POST accounts/:account\_id/scheduled\_promoted\_tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets) 端点。该端点在一次 API 调用中仅允许将一条“推广型定时推文”关联到一个线项。若要将多条定时推文关联到同一线项，需要进行多次 API 调用。

请注意，无法修改现有的“推广型定时推文”。

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_promoted_tweets" -d 'line_item_id=a44qc&scheduled_tweet_id=917507899668099072'
```

```json
{
  "data": {
    "line_item_id": "a44qc",
    "id": "26576",
    "created_at": "2017-10-09T22:24:16Z",
    "updated_at": "2017-10-09T22:24:16Z",
    "scheduled_tweet_id": "917507899668099072",
    "tweet_id": null,
    "deleted": false
  },
  "request": {
    "params": {
      "line_item_id": "a44qc",
      "scheduled_tweet_id": 917507899668099100,
      "account_id": "aaaaa"
    }
  }
}
```

该端点仅创建指定定时推文与线项之间的关联。一旦广告系列/线项的投放日期进入当前，线项将自动开始投放相应的“上线”推文。在此步骤中，我们会验证定时推文是否处于 `SCHEDULED` 状态，以及该定时推文是否符合给定目标，但不会执行其他校验。任何适用于线项与定时推文的其余校验规则将在推文“上线”时执行。

为确保广告系列投放不出现问题，建议将定时推文的 `scheduled_at` 时间设在广告系列/线项的投放日期之前。

例如，若定时推文设置在广告系列开始日期之后才上线（且仅有一条推文关联到一个线项），则广告系列将处于 `ACTIVE` 状态；然而由于定时推文尚未上线，将没有可用于投放的创意。

**定时推文管理**

其余一组端点允许 API 使用方管理其所有定时推文和推广型定时推文。这些 API 可用于返回所有定时推文的列表（可选按给定状态过滤），以及通过 `id` 查询指定的定时推文。

<div id="what-happens-when-a-scheduled-tweet-goes-live">
  #### 计划推文上线时会发生什么？
</div>

当某条计划推文即将上线（即在 `scheduled_at` 时间）时，将进行以下更新：

- 会创建“已上线”的推文，但可能存在最长 1 秒的延迟
- 会将 `tweet_id` 添加到以下实体：
- 计划推文
- 推广的计划推文
- 会创建一个新的推广推文实体

<div id="best-practices">
  #### 最佳实践
</div>

在创建或推广定时推文时，建议遵循以下最佳实践：

- 在创建定时推文时确保该推文有效（例如，一条推文只能包含图片、视频或 GIF 中的一种，不能混合使用）
- 确保广告活动的投放日期（即 `start_time` 和 `end_time`）与该定时推文的 `scheduled_at` 时间一致
- 定时推文的计划时间不应超过未来一年（365 天）
- 目前不支持定时推文的预览功能（即在创建前预览定时推文）

<div id="media-library">
  ### 媒体库
</div>

<div id="introduction">
  #### 简介
</div>

Media Library 端点用于管理 X 广告账户中的图片、GIF 和视频。库中的媒体资产可用于帖子（Tweets）以及创建卡片（cards）。这些资产也可在多个[创意](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/overview)中重复使用，无需反复上传相同资产。

<div id="api-endpoints">
  #### API 端点
</div>

- [POST media/upload](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload#post-media-upload) 或 [POST media/upload（分片）](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload-init#post-media-upload-init)（上传媒体）
- [POST accounts/:account\_id/media\_library](https://developer.x.com/x-ads-api/creatives#get-accounts-account-id-media-library#post-accounts-account-id-media-library)（将媒体添加到媒体库）

<div id="adding-to-the-library">
  #### 添加到库
</div>

将媒体添加到库需要两步。首先，使用 [POST media/upload](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload#post-media-upload) 端点或 [POST media/upload（分块）](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload-init#post-media-upload-init) 端点集上传素材。（有关多部分上传流程的详情，请参阅[分块媒体上传](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/uploading-media/chunked-media-upload)指南。）

```
twurl -X POST -H upload.x.com "/1.1/media/upload.json?additional_owners=756201191646691328" --file latte.jpeg --file-field "media"
```

```json
{
   "media_id":966947208837742592,
   "media_id_string":"966947208837742592",
   "size":74194,
   "expires_after_secs":86400,
   "image":{
      "image_type":"image/jpeg",
      "w":800,
      "h":418
   }
}
```

接着，使用该 media ID，通过 [POST accounts/:account\_id/media\_library](https://developer.x.com/x-ads-api/creatives#get-accounts-account-id-media-library#post-accounts-account-id-media-library) 端点将媒体添加到广告账户的库中。

```
twurl -X POST -H ads-api.x.com "/9/accounts/18ce54d4x5t/media_library?file_name=latte.jpeg&media_category=TWEET_IMAGE&media_key=966947208837742592&name=Latte"
```

```json
{
   "request":{
      "params":{
         "name":"Latte",
         "file_name":"latte.jpeg",
         "media_category":"TWEET_IMAGE",
         "account_id":"18ce54d4x5t",
         "media_key":966947208837742592
      }
   },
   "data":{
      "tweeted":false,
      "name":"Latte",
      "file_name":"latte.jpeg",
      "media_url":"https://pbs.twimg.com/media/DWtJXQNUQAAdPZj.jpg",
```

**注意：** 上传后若直接发布包含图片、GIF 或视频的推文，也会将该媒体添加到库中。

<div id="request-parameters">
  #### 请求参数
</div>

所有 Media Library 的 POST 请求都需要提供媒体标识符。该值会在上传步骤中返回。使用 media\_id（如上例所示）时，还必须同时指定 media\_category。类别共有四种取值：AMPLIFY\_VIDEO、TWEET\_GIF、TWEET\_IMAGE 和 TWEET\_VIDEO。

可选地，可以为 Media Library 中的对象设置 name 和 file\_name。这些属性有助于用户在库中区分不同的媒体变体。

对于视频，还可以设置标题和描述。其值应作为 video\_title 和 video\_description 请求参数，随 [POST accounts/:account\_id/tweet](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet) 端点一并传递。在推文中，这些文本会显示在视频下方。

<div id="attributes">
  #### 属性
</div>

Media Library 正式引入了 media\_key 概念。它是库中对象的唯一标识符。media\_key 是如下格式的字符串值：13\_875943225764098048。我们的所有卡片端点均已完全支持它。

此外，Media Library 的响应还包含以字符串表示的 media\_id。对于当前尚不接受 media\_key 的资源，会包含该字段：[Tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet)\*、[Tweet preview](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#get-accounts-account-id-tweet-preview)\* 和 [Scheduled Tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets)。我们正在推进在各处对 media\_key 的支持。

aspect\_ratio 属性会针对 GIF 和视频返回。可据此筛选仅接受特定纵横比的卡片所使用的媒体。

\*这些端点支持 video\_id 参数，该参数即为一个 media\_key。

<div id="identifying-cards">
  ### 识别 Cards
</div>

<div id="introduction">
  #### 介绍
</div>

Card 是可自定义的广告格式，使用媒体，并可与网站、应用或号召性用语关联，以促进特定的用户互动，例如发起 Direct Message。它们可以附加到 Tweets、Scheduled Tweets 或 Draft Tweets。

可以通过两种方式在 Tweet 对象中引用 Card：使用该卡片的 card\_uri 或其 preview\_url。下面给出了每种方式的示例值。

| card\_uri | preview\_url |
| :--- | :--- |
| card://1043282691834048513 | https://cards.x.com/cards/18ce54d4x5t/68w3s |

**注意**：从 Ads API 版本 3 起，对于新创建的 Card，在 cards 响应中仅生成并返回 card\_uri。

**注意**：从 Ads API 版本 5 起，cards 响应中不再返回 preview\_url。

Tweet 对象响应中包含的引用类型取决于 Tweet 的创建方式。换言之，如果使用 card\_uri 请求参数创建 Tweet，则响应中会显示该 card URI 值。相反，如果在 Tweet 文本中包含了 preview\_url，则响应中会显示该预览 URL。

<div id="identifying-media">
  ### 识别媒资
</div>

<div id="introduction">
  #### 介绍
</div>

媒体（图片、GIF 和视频）可添加到推文和卡片中。此外，视频可用作贴片前素材，图片可在 [X Audience Platform](https://developer.x.com/en/docs/ads/measurement/overview/twitter-audience-platform) 上投放推广。本节说明如何在这些实体中定位媒体引用。

媒体标识符有两种类型：ID 和 key。下面给出每种的示例值。

| **媒体 ID** | **媒体 key** |
| :--- | :--- |
| 1029825579531807971 | 13\_1029825579531807971 |

媒体 key 由 ID 加上一个数字前缀和一个下划线构成。

<div id="images">
  #### 图像
</div>

下表展示了当前各与图像相关的资源响应中可用的标识符类型以及对应的属性名称。

| **资源** | **标识符** | **属性** |
| :--- | :--- | :--- |
| Image cards | 无 |  |
| Tweet | 二者都有 | `entities["media"]["id_str"]`  `entities["media"]["media_key"]` |
| Scheduled Tweet | 二者都有 | `media_ids` 和 `media_keys` |
| Draft Tweet | 二者都有 | `media_ids` 和 `media_keys` |
| Account Media | 无 |  |
| Media Library | 二者都有 | `media_id` 和 `media_key` |

Image cards 和 Account Media 的图像不包含对任何媒体标识符的引用。Tweets 仅包含媒体 ID。Scheduled 和 Draft Tweets 同时包含媒体 ID 和媒体 key。Media Library 也会返回两者。

对于 Tweets，entities\["media"] 数组中对象的 id 和 id\_str 字段对应媒体 ID。当一条 Tweet 包含多张图片时，对每个媒体实体的引用只能在 extended\_entities\["media"] 中找到。

除了标识符引用外，通常还需要获取图像的 URL。

| **资源** | **属性** | **格式** |
| :--- | :--- | :--- |
| Image cards | `image` | .jpg |
| Tweet\* | `entities["media"][0]["media_url"]` 或 `extended_entities["media"][i]["media_url"]` | .jpg |
| Scheduled Tweet | 无 |  |
| Draft Tweet | 无 |  |
| Account Media | `media_url` | .jpg |
| Media Library | `media_url` | .jpg |

\* 该 URL 的位置取决于该 Tweet 是否包含单张或多张图片。

所有 image cards 都包含一个名为 image 的响应属性，其中包含 X 的图像 URL。（对于 image app download cards，该属性名为 wide\_app\_image。）

对于 Tweets，media URL 的位置取决于媒体类型以及所使用的端点。对于仅包含单张图片的 Tweets，URL 可在 entities\["media"]\[0]\["media\_url"] 中找到，这在 Ads API 和 Standard API 中均适用。而当 Tweets 包含多张图片时，这些 URL 只能在 extended\_entities\["media"]\[i]\["media\_url"] 中找到，该字段仅在 Standard API 中提供。

<div id="video">
  #### 视频
</div>

下表展示了当前各类与视频相关的资源在响应中可用的标识符类型及其对应的属性名称。

| **资源** | **标识符** | **属性** |
| :--- | :--- | :--- |
| 视频卡片 | 可能为任一类型 | `video_content_id` |
| 视频投票卡片 | 无 |  |
| 推文 | 两者皆有 | `entities["media"]["id_str"]`  `entities["media"]["media_key"]` |
| 已安排发布的推文 | 两者皆有 | `media_ids` 和 `media_keys` |
| 草稿推文 | 两者皆有 | `media_ids` 和 `media_keys` |
| 账户媒体 | 媒体 key | `video_id` |
| 媒体库 | 两者皆有 | `media_id` 和 `media_key` |

尽管视频卡片（带视频的投票卡片除外）包含一个 video\_content\_id 响应属性，但其返回值类型并不一致：有时为媒体 ID，其他情况下为媒体 key。

以下是关于如何获取视频 URL 的说明。

| **资源** | **属性** | **格式** |
| :--- | :--- | :--- |
| 视频卡片 | `video_url` 和 `video_hls_url` | .vmap  .m3u8 |
| 带视频的推文 | `extended_entities["media"][i]["video_info"]["variants"][j]["url"]` | .mp4 |
| 已安排发布的推文 | 无 |  |
| 草稿推文 | 无 |  |
| 账户媒体 | 无 |  |
| 媒体库 | `media_url` | .mp4 |

视频卡片包含 `video_url` 和 `video_hls_url` 响应属性，分别为 .vmap 和 .m3u8 的 URL。

<div id="media-library">
  #### 媒体库
</div>

有时需要获取媒体资产的更多信息。以视频卡片为例，可能需要获取 mp4 URL，而不是 vmap。相关信息可在媒体库中查看。有关可用信息的详细说明，请参阅我们的[媒体库指南](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/guides/media-library)。大多数属于广告账户的 FULL promotable user 的资产都可以在库中找到，但也有一些例外。

**获取媒体**

如上所述，图片卡片不包含媒体 ID 或媒体键的引用。因此，无法通过媒体库获取其资产。[Account Media](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/account-media#account-media) 的图片同样如此。

视频卡片要求在创建之前，视频资产必须已添加至媒体库（或其前身 Videos 资源）。因此，这些资产始终可以在媒体库中检索。Account Media 的 PREROLL 资产同样如此。

最后，推文中的媒体始终保证存在于媒体库中。

下表汇总了哪些资产可在媒体库中检索，并说明资源响应是否包含可用于查找的标识符。

| **资源** | **在媒体库中** |
| :--- | :--- |
| 图片卡片 | 否 |
| 视频卡片 | 是\* |
| 推文（任意媒体）\*\* | 是 |
| 预定发布的推文 | 是 |
| 草稿推文 | 是 |
| Account media 图片 | 否 |
| Account media 视频（`PREROLL`） | 是 |

\* 适用于 `video_content_id` 为媒体键的卡片。当该值为媒体 ID 时，资产仍存在于媒体库中，但检索它需要在其前加数字前缀和下划线。
\*\* 推文仅返回媒体 ID。虽然资产保证存在于媒体库中，但获取它需要在其前加数字前缀和下划线。

**与 Account Media 的交互**

有两种情况会将添加到媒体库中的媒体资产自动添加到 Account Media 资源中。

- 当 AMPLIFY\_VIDEO 资产被添加到媒体库时，会自动作为 PREROLL 创意类型添加为 Account Media 资产。
- 当具有特定尺寸的图片（参见我们[枚举页面](https://developer.x.com/content/developer-twitter/en/docs/ads/general/overview/enums)中的“Creative Types”）被添加到媒体库时，会自动添加为 Account Media 资产。创意类型（例如，INTERSTITIAL）取决于图片尺寸。

<div id="tweets">
  ### 推文
</div>

<div id="introduction">
  #### 简介
</div>

X 广告 API 支持三种类型的推文：已发布、已排程和草稿。

<div id="nullcasted-tweets">
  #### Nullcasted 推文
</div>

推文可以是 nullcasted（亦称“仅推广”）或自然推文。已发布的 nullcasted 推文虽为公开内容，但不会出现在用户的公开时间线中。相比之下，自然推文会投放给用户的关注者，并会显示在用户的公开时间线中。

**创建推文**

三个用于创建推文的端点均支持一个布尔类型的 nullcast 参数，使 API 使用者可以选择创建 nullcasted 或自然推文。nullcasted 推文可以由用户本人创建，也可以由任何获授权可代表该用户创建推文的人创建。自然推文只能由[完全可推广用户](https://developer.x.com/content/developer-twitter/en/docs/tutorials/ads-api-hierarchy-terminology#promotable-users)创建。

**更新推文**

可更新已计划发送与草稿推文的 nullcast 属性。对于已计划发送的推文，可在推文的 scheduled\_at 时间之前进行编辑。草稿推文可无限期编辑。但一旦发布，就无法在 nullcasted 与自然之间相互转换。

<div id="promoting-tweets">
  #### 推广推文
</div>

只有已发布和已安排的推文可以推广。它们可以是 nullcast 或自然内容，均不受限制。广告主在获得许可的前提下，可以推广自己的推文或其他用户的推文。（参见：[Promoting another user's Tweets](https://developer.x.com/content/developer-twitter/en/docs/tutorials/promoting-another-users-content) 了解更多信息。）

单个广告系列中可以推广多条推文。同样，单条推文也可以在一个或多个广告系列中推广。

要推广已发布的推文，请使用 [POST accounts/:account\_id/promoted\_tweets 端点](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/promoted-tweets#post-accounts-account-id-promoted-tweets)。这会将已发布的推文与一个 line item 关联。要推广已安排的推文，请使用 [POST accounts/:account\_id/scheduled\_promoted\_tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets) 端点。

<div id="tweet-ids">
  #### 推文 ID
</div>

已发布、已排期和草稿的推文 ID 均为[数值](https://developer.x.com/content/developer-twitter/en/docs/basics/twitter-ids)——它们是 64 位无符号整数。例如，下面这条已发布推文的 ID 为 1166476031668015104。

<div id="carousels">
  ### 轮播图
</div>

<div id="introduction">
  #### 介绍
</div>

X 广告 API 支持创建和检索视频轮播和图片轮播。轮播是一种卡片类型，可包含 2 至 6 个媒体素材。轮播卡片可以将用户引导至网站，或引导他们安装移动应用。关于轮播的概述、优势、最佳实践和常见问题，请参见我们的 [Carousel Ads on X](https://business.x.com/en/advertising/carousels.html) 页面。

与其他卡片类型一样，轮播可用于推文，这些推文随后可以被推广。工作流程与您已熟悉的相同：

1. 上传媒体
2. 创建卡片
3. 创建推文
4. 推广推文

唯一的区别在于卡片的创建方式。其他卡片创建请求接受查询参数，而轮播卡片创建请求仅接受 JSON POST 请求体。

<div id="endpoints">
  #### 端点
</div>

X 广告 API 支持创建和获取轮播。

要创建任意类型的轮播，请使用 [POST accounts/:account\_id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#post-accounts-account-id-cards) 端点。要获取轮播，请使用 [GET accounts/:account\_id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#get-accounts-account-id-cards) 端点。

<div id="json-post-body">
  #### JSON POST 请求体
</div>

轮播由两个组件构成。第一个组件用于指定要使用的媒体资产；第二个组件用于指定网站或应用的信息。

具体而言，轮播卡片按以下顺序由这些组件创建：

- 一个 `SWIPEABLE_MEDIA` 组件，接受一个媒体键数组
- 下列组件中的一个：
- 一个 `DETAILS` 组件，用于指定网站信息
- 一个 `BUTTON` 组件，用于指定应用信息

`SWIPEABLE_MEDIA` 组件必须包含一个 `media_keys` 数组，可指定 2 到 6 张图片或视频。传入媒体键的顺序将决定它们的渲染顺序。

```
{
  "type": "SWIPEABLE_MEDIA",
  "media_keys": [
    "13_1089771253848666112",
    "13_1191948012077092867"
  ]
}
```

请注意，你可以通过请求 [GET accounts/:account\_id/media\_library](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/media-library#get-accounts-account-id-media-library) 端点来获取媒体键。

第二个组件对象的构成取决于你是希望将用户引导至网站，还是鼓励他们安装应用。下表汇总了这两种选项。（注意：表中列出的所有键均为必填。）

|  | **网站** | **应用** |
| :--- | :--- | :--- |
| 指定组件类型 | `"type": "DETAILS"` | `"type": "BUTTON"` |
| 标题/标签 | `"title": "X"` | `"label": { "type": "ENUM", "value": "INSTALL" }` |
| 目标地址 | `"destination": { "type": "WEBSITE", "url": "https://www.x.com" }` | `"destination": { "type": "APP", ... }` |

将以上内容组合起来，下面是一个网站轮播的 JSON POST 请求体示例。

```
{
  "name": "website carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "DETAILS",
      "title": "X",
      "destination": {
        "type": "WEBSITE",
        "url": "https://www.x.com"
      }
    }
  ]
}
```

`BUTTON` 组件中的应用目标对象需要提供国家代码以及至少一个应用标识符，并可选填深度链接。有关这些字段的说明，请参阅[参考文档](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#app-destination)。

将以上内容组合起来，下面是一个应用轮播的 JSON POST 请求体示例。

```
{
  "name": "app carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        "type": "APP",
        "country_code": "US",
        "googleplay_app_id": "com.twitter.android",
        "iphone_app_id": "333903271"
      }
    }
  ]
}
```

<div id="parameters">

</div>

| 名称 | 描述 |
| :--- | :--- |

#### POST accounts/:account\_id/cards

为指定账户创建一张新卡片。

卡片创建请求仅接受 JSON 格式的 POST 请求体。必须将 `Content-Type` 设为 `application/json`。

有关详细用法示例，请参阅我们的[轮播指南](https://developer.x.com/en/docs/twitter-ads-api/creatives/guides/carousels)。

  #### 资源 URL
</div>

`https://ads-api.x.com/12/accounts/:account_id/cards`

<div id="parameters">
  #### 参数
</div>

JSON POST 正文必须包含卡片的 `name` 和一个 `components` 数组。组件以对象形式表示，用于描述卡片面向广告主的属性。

以下示例展示了负载的一般结构（但包含不可用的信息）。

```
{
  "name": "some name",
  "components": [
    {
      "type": "TYPE_ENUM",

    }
  ]
}
```

下面提供关于组件的更多信息。

| 名称 | 说明 |
| :--- | :--- |
| account\_id   _必填_ | 被使用的账号标识符。出现在资源路径中，通常是除 [GET accounts](https://developer.x.com/en/docs/ads/campaign-management/api-reference/accounts#get-accounts) 之外所有 Advertiser API 请求所需的参数。指定的账号必须与已认证用户关联。 类型：string 示例：`18ce54d4x5t` |
| name   _必填_ | 卡片名称。最大长度：80 个字符。 类型：string 示例：`component based card` |
| components   _有时必填_ | 用于创建卡片的组件。更多信息见下文。不可与 `slides` 同时指定。注意：组件的顺序很重要。 类型：对象数组 |
| slides   _有时必填_ | 使用此“数组的数组”来创建多目标轮播。将每个卡片描述为组件的分组。每个幻灯片都应完整表示一个卡片。不可与 `components` 同时指定。注意：每个幻灯片的顺序很重要。 类型：数组的数组 |

<div id="components">
  #### 组件
</div>

每个组件必须包含一个 `type` 字段，用于确定对象的模式（schema）。X 广告 API 支持以下组件类型，分为基于媒体和基于描述的组件。

- 媒体：
- `MEDIA`：单个视频或图片
- `SWIPEABLE_MEDIA`：2–6 个视频或图片
- 描述：
- `DETAILS`
- `BUTTON`

每个组件都有一组必填字段（除 `type` 键之外）。这些字段列在下表中。

| 组件 `type` | 字段 | 值类型 |
| :--- | :--- | :--- |
| `MEDIA` | `media_key` | string |
| `SWIPEABLE_MEDIA` | `media_keys` | array of strings |
| `DETAILS` | `title`   `destination` | string   object |
| `BUTTON` | `label`   `destination` | object   object |

下面是在 `components` 数组中 `BUTTON` 组件的示例（有意省略 `name` 键）。（省略号表示需要补充更多信息的地方。）

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        ...
      },
      "destination": {
        ...
      }
    }
  ]
}
```

组件对象的声明顺序决定了它们从上到下的渲染顺序。卡片必须使用一个基于媒体的组件，并搭配一个 `DETAILS` 或 `BUTTON` 组件创建。基于描述的组件渲染在媒体下方，并具有关联的目标位置，可以是 URL 或移动应用。

**Label**

Label 定义显示在按钮上的文本，因此仅适用于 `BUTTON` 组件。Label 对象有两个必填键：`type` 和 `value`。`type` 必须设置为 `ENUM`，`value` 可以是以下之一：`BOOK`、`CONNECT`、`INSTALL`、`OPEN`、`ORDER`、`PLAY` 或 `SHOP`。

在前述示例基础上，下面展示了 `BUTTON` 组件中的 `label` 对象。

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        ...
      }
    }
  ]
}
```

**Destination**

Destination 表示广告主希望将用户导向的位置。在 `DETAILS` 或 `BUTTON` 组件中始终为必填。有两种目标类型：`WEBSITE` 或 `APP`。

**注意**：网站目标只能与 `DETAILS` 组件搭配使用，应用目标只能与 `BUTTON` 组件搭配使用。

**网站目标**

| 名称 | 描述 |
| :--- | :--- |
| type   _必填_ | 目标类型，用于决定其模式。类型：enum  可能的取值：`WEBSITE` |
| url   _必填_ | 将用户重定向到的网站 URL。类型：string  示例：`https://devcommunity.x.com/c/advertiser-api` |

**应用目标**

| 名称 | 描述 |
| :--- | :--- |
| type   _必填_ | 目标类型，用于决定其模式。类型：enum  可能的取值：`APP` |
| country\_code   _必填_ | 应用销售所在国家的 [ISO 3166-1 alpha-2](https://en.wikipedia.org/wiki/ISO_3166-1_alpha-2) 两字母代码。类型：string  示例：`US` |
| googleplay\_app\_id   _有时必填_ | Google Play 应用包名。**注意**：以下至少一项为必填：`ios_app_store_identifier` 或 `googleplay_app_id`。类型：string  示例：`com.twitter.android` |
| ios\_app\_store\_identifier   _有时必填_ | iOS 应用商店标识符。**注意**：以下至少一项为必填：`ios_app_store_identifier` 或 `googleplay_app_id`。类型：string  示例：`333903271` |
| googleplay\_deep\_link   _可选_ | 指向所推广 Android 应用的深度链接。**注意**：仅在已提供 `googleplay_app_id` 时可用。类型：string |
| ios\_deep\_link   _可选_ | 指向所推广 iOS 应用的深度链接。**注意**：仅在已提供 `ios_app_store_identifier` 时可用。类型：string |

#### POST accounts/:account\_id/tweet

为该账户的完整可投放用户（默认）或在 `as_user_id` 参数中指定的用户创建一条推文。支持创建静默投放（默认）和自然推文。静默投放推文不会出现在公共时间线中，也不会触达关注者。两种类型均可用于广告系列。

如果经过身份验证的用户不是此账户的 `FULL` 可投放用户，请调用 [GET accounts/:account\_id/authenticated\_user\_access](https://developer.x.com/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access) 端点以确定其是否有权代表该用户发推。权限 `TWEET_COMPOSER` 表示该用户可使用此端点，代表 `FULL` [可投放用户](https://developer.x.com/en/docs/ads/campaign-management/api-reference/promotable-users#get-accounts-account-id-promotable-users) 创建静默投放推文。

当为媒体使用 [upload.x.com 端点](https://developer.x.com/rest/reference/post/media/upload) 时，请将与传递给此端点的 `as_user_id` 相同的值，作为 `additional_owners` 参数中的 `user_id` 传入。
