---
title: 速率限制
---

每天都有成千上万的开发者向 X API 发起请求。为应对庞大的请求量，我们对可发起的请求数量设定了限制。这些限制有助于为我们的开发者社区提供稳定且可扩展的 API。

允许的最大请求数基于时间间隔，通常以指定的时间窗口计算。最常见的间隔是 15 分钟。例如，某个端点若限制为每 15 分钟 900 次请求，则在任意 15 分钟窗口内最多可发起 900 次请求。

速率限制取决于所用的身份验证方式。比如，若使用 [OAuth 1.0a User Context](/zh/resources/fundamentals/authentication)，每组用户的 [Access Tokens](/zh/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) 在每个周期都有各自的速率限制。或者，若使用 [OAuth 2.0 Bearer Token](/zh/resources/fundamentals/authentication#oauth-2-0)，你的应用在每个时间周期也会有单独的限制。超出这些限制时，会返回错误。

<div id="rate-limits-and-authentication-method">
  ## 速率限制与认证方式
</div>

速率限制同时适用于开发者应用和用户访问令牌两个层级：

- **[OAuth 2.0 Bearer Token](/zh/resources/fundamentals/authentication/oauth-2-0/application-only)：应用级速率限制**
  该方式允许你以开发者应用的名义发起一定数量的请求。使用此认证方式时，限制取决于使用 Bearer Token 发起的请求数。
  - 例：若限制为每 15 分钟 450 次请求，则你可在该时间窗口内以应用名义发起 450 次请求。

- **[OAuth 1.0a User Context](/zh/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens)：用户级速率限制**
  该方式允许以由用户访问令牌标识的 X 用户名义发起请求。例如，若需获取帖子的私有指标，请使用通过[三方 OAuth 流程](/zh/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)为该用户生成的用户访问令牌进行认证。
  - 例：若限制为每位用户每 15 分钟 900 次请求，则在该时间窗口内你最多可对每位用户发起 900 次请求。

<div id="http-headers-and-response-codes">
  ## HTTP 标头与响应代码
</div>

使用 HTTP 标头，根据最近一次请求来判断你的应用在指定速率限制内的当前状态。

- `x-rate-limit-limit`：该端点的速率限制上限
- `x-rate-limit-remaining`：15 分钟窗口内剩余的请求次数
- `x-rate-limit-reset`：距离速率限制重置的剩余时间（UTC 纪元秒）

<div id="error-responses">
  ### 错误响应
</div>

如果应用在某个端点超出速率限制，API 将返回 [HTTP 429 “请求过多”](http://tools.ietf.org/html/rfc6585) 响应，并在响应正文中包含以下错误信息：

```json
{ "errors": [ { "code": 88, "message": "Rate limit exceeded" } ] }
```

<div id="recovering-from-a-rate-limit">
  ## 从速率限制中恢复
</div>

当超出这些速率限制时，端点会返回 429“请求过多”错误。如下所述，出现速率限制错误时，最佳实践是检查指示限制重置时间的 HTTP 标头，并在那之前暂停请求。<br />当出现“请求过多”或速率限制错误时，需要降低请求频率。触发速率限制错误后，可检查 `x-rate-limit-reset:` HTTP 标头以了解速率限制的重置时间<br /><br />。另一种常见做法是使用指数退避：起初请求间隔较短（例如几秒），此后每次重试前将间隔加倍，直到请求成功，或达到一个合理的最大间隔（例如几分钟）。<br /><br /> 理想情况下，客户端应感知现有速率限制，并在当前超限的时间窗口到期前暂停请求。如果你超出了 15 分钟的限制，那么在重试前等待一两分钟是合理的。<br /><br /> 请注意，除了这些关于请求次数的限制外，Standard Basic 访问级别还通过最近搜索和过滤流端点每月最多提供 500,000 条帖子。如果你已超出每月的帖子配额，更合理的做法是让应用发出通知，记录其每月的计费/入驻日，并在该日期之前暂停请求。

<div id="tips-to-avoid-being-rate-limited">
  ### 避免触发速率限制的提示
</div>

以下提示旨在帮助您进行防御性编码，降低触发速率限制的可能性。受速率限制影响，您可能想提供的某些应用功能（尤其在结果新鲜度方面）可能无法实现。如果您的应用以实时信息为目标，请参阅过滤流与采样流端点。

<div id="caching">
  #### 缓存
</div>

如果预计会频繁使用，请存储 API 响应。不要在每次页面加载时都调用 API，而应在本地缓存响应。

<div id="prioritize-active-users">
  #### 优先处理活跃用户
</div>

如果你的站点需要跟踪大量 X 用户（例如获取他们的当前状态或 X 使用情况的统计数据），请考虑仅为最近登录过你站点的用户请求数据。

<div id="adapt-to-the-search-results">
  #### 适配搜索结果
</div>

如果你的应用监控了大量搜索词，对于没有结果的搜索，应当比有结果的搜索降低查询频率。通过使用退避策略，你可以及时跟进热门查询，同时避免在几乎不变的查询上浪费请求资源。或者，考虑使用过滤流端点，并用你的搜索查询进行过滤。

<div id="denylist">
  #### 拒用名单
</div>

如果某个应用程序滥用速率限制，它将被拒用。被拒用的应用无法从 X API 获得响应。如果你或你的应用已被拒用并认为这是一个误判，你可以使用我们的[平台支持表单](https://support.x.com/forms/platform)请求协助。请包含以下信息：

1. 说明你认为你的应用为何被拒用。
2. 如果你已不再触发速率限制，请详细说明你是如何解决该问题的。
