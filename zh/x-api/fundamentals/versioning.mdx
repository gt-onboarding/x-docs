title: "版本管理"
---

<p>随着 X API v2 的发布，我们推出了一项新的版本管理策略，帮助开发者更清楚地了解何时可能对 X 公共 API 进行变更，以及何时需要迁移到新版本。</p>
<p>我们将通过<a href="https://developer.x.com/en/updates/stay-informed">沟通渠道</a>通知开发者有关 X API 的弃用、下线、变更和新增事项，便于他们在开发路线图中做好相应规划。所有对 API 的变更都会记录在更新日志中。</p>
<p>目前 X API 提供三个不同的版本。除非你的用例所需功能尚未在 v2 提供，否则我们强烈建议在规划集成时采用 X API v2。</p>
<p>如需了解各版本的详细信息，请访问以下页面：</p>

<ul>
  <li><a href="/zh/x-api/introduction">X API v2</a></li>
  <li>Gnip 2.0（<a href="/zh/x-api/enterprise-gnip-2.0/enterprise-gnip">企业版</a>）</li>
</ul>

<div id="versioning-strategy">
  ## 版本控制策略
</div>

<p>X API 的版本将通过在端点路由路径中声明的版本号来表示：</p>
<p>`https://api.x.com/2/tweets`</p>
<p>我们会在需要时发布 X API 的主版本，但频率不高于每 12 个月一次。当在 API 中引入不兼容变更时，将发布主版本。在发布新的主版本时，我们会提供迁移指南，帮助开发者迁移到新版本。 </p>
<p>不兼容变更要求开发者修改代码，以在其应用中维持现有功能。非破坏性变更将以增量方式添加，并在准备就绪时发布到最新版本；除非你希望利用新功能，否则开发者无需进行任何改动。</p>
<p>如果必须在版本周期中途推出不兼容变更（出于安全或隐私原因），该变更将应用于最新版本。</p>

<div id="breaking-changes">
  ### 重大变更
</div>

<p>这些变更要求开发者修改代码，以保持其应用程序的现有功能。</p>

<ul>
  <li>新增必需参数</li>
  <li>移除现有端点</li>
  <li>移除响应中的任何字段（无论必需或可选）</li>
  <li>移除查询参数</li>
  <li>重构输入或输出格式（例如，将顶层字段改为子字段，或将错误位置改为内联）</li>
  <li>更改现有输入参数或输出值的名称或数据类型</li>
  <li>更改字段名称</li>
  <li>更改资源名称</li>
  <li>更改响应代码</li>
  <li>更改错误类型</li>
  <li>更改现有授权范围</li>
</ul>

<div id="non-breaking-changes">
  ### 非破坏性更改
</div>

<ul>
  <li>新增一个端点</li>
  <li>新增一个可选参数</li>
  <li>新增一个响应字段</li>
  <li>修改错误消息中的文本</li>
  <li>提供新的权限范围（scope）</li>
  <li>将字段“置空”（出于隐私/安全原因，将某个值改为 null，作为不完全移除该字段的替代方案）</li>
</ul>

<div id="deprecation-and-retirement">
  ### 弃用与退役
</div>

<p>首先，以下是我们在 X API 中对“弃用”和“退役”的定义：</p>

<ul>
  <li><b>弃用：</b>该功能不再由团队维护。围绕该功能不会再发布任何新特性；若产品出现缺陷或问题，我们着手修复的可能性极低。 </li>
  <li><b>退役：</b>该功能将不再可用。</li>
</ul>

<p>在大多数情况下，一旦新版本发布，先前版本会被标记为弃用。版本会在弃用状态保留一段时间，之后将被退役。 </p>
<p>请<a href="https://developer.x.com/en/updates/stay-informed">保持关注</a>，以了解未来的弃用与退役更新。</p>
