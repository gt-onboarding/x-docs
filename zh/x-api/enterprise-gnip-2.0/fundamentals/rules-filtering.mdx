<div id="getting-started-with-enterprise-rules-and-queries">
  ## 企业级规则与查询入门
</div>

使用企业级运算符的产品会根据你设置的筛选规则向你提供社交数据。规则由一个或多个“子句”组成，子句可以是关键词、精确短语，或众多企业级运算符之一。在开始使用企业级运算符构建规则之前，请务必先查看下文所述语法，浏览可用运算符列表，并了解构建规则时的限制。你还应理解规则在逻辑评估方面的细微差别，详见“[运算顺序](#orderofoperations)”一节。

多个子句可以同时通过 “and” 与 “or” 逻辑进行组合。

**请注意：** “And” 逻辑通过在子句之间留空格来表示，而 “or” 逻辑通过使用全部大写的 `OR` 来表示。

每条规则最长可包含 2,048 个字符，对正向子句（你希望匹配或筛选的内容）和负向子句（你希望排除且不匹配的内容）的数量不设上限。
 

### 构建规则和查询 

**关键词匹配**

关键词匹配与搜索界面中的查询类似。例如，下面这个企业级运算符规则会匹配正文中包含“social”一词的活动。

`social`

**用空格进行 AND 连接**

添加另一个关键词等同于增加一个匹配条件。例如，此规则只会匹配文本中同时包含“social”和“media”的活动，顺序不限——_在词与词之间使用空格等同于布尔 AND 逻辑_。如果在规则中显式写出 AND，将会被规则端点拒绝。

`social media`

**使用大写 OR 进行 OR 连接**

很多情况下需要布尔 OR 逻辑，也很容易实现。_注意，OR 运算符必须为大写；小写 “or” 会被当作普通关键词处理。_

`social OR data`

**取反（排除）术语**

有时需要排除包含某些关键词的结果（布尔 NOT 逻辑）。例如，包含“happy”但排除文本中包含“birthday”的活动。

`social -personality`

**使用括号分组**

可以用括号对上述逻辑进行分组组合，扩展为更复杂的查询。

`(social OR data) (academic OR research) -personality -information -university`

这只是开始——虽然上述示例仅依赖于对关键词的分词匹配，企业产品还提供运算符以在文本上执行不同类型的匹配。

**精确匹配**

`"social media research"`

**子串匹配**

`contains:info`

**近邻匹配**

`"social media research"~3`

此外，其他运算符还支持按文本之外的社交数据独有维度进行筛选。 

**发布帖子的用户**

`from:XDevelopers`

**以美国科罗拉多州博尔德市 Pearl St. 为中心 10 英里范围内的地理标记推文**

`point_radius:[-105.27346517 40.01924738 10.0mi]`

**将所有内容组合在一起**

这些可以与文本筛选器结合使用，逻辑方式与上述一致。

`(social OR data) (academic OR research OR "social media research") point_radius:[-105.27346517 40.01924738 10.0mi] lang:en -personality -information -university`

<div id="boolean-syntax">
  ### 布尔语法
</div>

上一节的示例使用了多种布尔逻辑与分组方式。请参阅下表，了解各自的语法与要求。

|     |     |     |
| :--- | :--- | :--- |
| **逻辑类型** | **运算符语法** | 说明 |
| AND | social data | 在两个运算符之间留空白即表示 AND 逻辑。  <br />  <br />匹配同时包含两个关键词（“social”“data”）的活动。  <br />  <br />**不要在规则中显式写 AND。只使用空白。显式的 AND 会被当作普通关键词处理。** |
| OR  | social OR data | 要将两个运算符进行 OR 连接，请在它们之间插入以空白分隔的全大写 OR。  <br />  <br />匹配包含任一关键词（“social” 或 “data”）的活动。  <br />  <br />注意，如果在同一条规则中同时使用 OR 和 AND，请先理解“[运算顺序](#OrderOfOperations)”一节所述的优先级，并按下文所述用括号将未取反的运算符分组，以确保规则按预期运行。  <br />  <br />规则中必须使用大写 “OR”。小写 “or” 会被视为普通关键词。 |
| NOT | social data  <br />-apple -android -phone | 在运算符或一组运算符前紧跟一个 `-` 符号即可取反。  <br />  <br />示例规则匹配包含关键词 “social” 的活动，但排除包含关键词 “data” 的活动。  <br />  <br />不允许使用会请求“除否定项外抓取整条 firehose”的取反 OR。例如，`apple OR -ipad` 是无效的，因为它会匹配除提及 “ipad” 之外的所有活动。 |
| 分组 | (social OR data) -XDevelopers -api | 用括号将多个运算符括起可创建一个功能性“组”。<br /><br />组可以像单个子句一样，通过空白（AND）或 OR 与其他子句连接。但最佳实践是不要通过对整个组加负号来统一取反；相反，应分别对每个运算符取反，并用空白（AND）串联。<br /><br />例如，不要使用 -(iphone OR imac OR macbook)，应改为：-iphone -imac -macbook。  <br />  <br />当单条规则同时包含 AND 和 OR 时，分组尤为重要，因为规则评估遵循特定的运算顺序。详见下文。 |

**请注意：** 运算符可以是正向或负向。

**正向运算符** 定义你希望在结果中**包含**的内容。例如，`has:hashtags` 表示“我想要包含话题标签的活动”。

**负向运算符** 定义你希望从结果中**排除**的内容，通过使用上述布尔 NOT 逻辑来实现。例如，`-has:hashtags` 表示“排除任何包含话题标签的活动，即使它们在其他方面符合我的规则”。

高级运算符产品对正负子句的数量不设限制，但整体最大长度为 2,048 个字符。
 

<div id="order-of-operations">
  #### 运算顺序
</div>

在单个规则中同时使用 AND 与 OR 时，规则的评估遵循以下运算顺序：

1. 先合并由 AND 逻辑连接的运算符
2. 再应用由 OR 逻辑连接的运算符

**示例：**

- `apple OR iphone ipad` 将被评估为 `apple OR (iphone ipad)`
- `ipad iphone OR android` 将被评估为 `(iphone ipad) OR android`

为消除歧义并确保按预期评估规则，请在合适的地方使用括号对术语进行分组。例如：

- `(apple OR iphone) ipad`
- `iphone (ipad OR android)`

<div id="punctuation-diacritics-and-case-sensitivity">
  #### 标点、变音符号与大小写敏感性
</div>

如果你为企业级运算符指定带有重音或变音符号的关键词或话题标签规则，它将仅匹配帖子文本中同样带有相应变音符号的内容（话题标签或关键词）。包含关键词 `Diacr**í**tica` 或话题标签 `#cumplea**ñ**os` 的规则将匹配“Diacr**í**tica”或“#cumplea**ñ**os”，但不会匹配缺少重音符或波浪号的“Diacritica”或“#cumpleanos”，即缺少 **í** 或字母 **ñ** 的情况。

带有重音或变音符号的字符与普通字符一样处理，不会被视为词边界。例如，规则 cumplea**ñ**os 只会匹配包含单词 cumpleaños 的活动，而不会匹配包含 cumplea、cumplean 或 os 的活动。

所有运算符的评估均不区分大小写。例如，规则 `Cat` 将匹配以下所有内容：“cat”“CAT”“Cat”。

<div id="powertrack-rule-tags">
  #### PowerTrack 规则标签
</div>

如我们的“[匹配规则](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#matching-rules)”页面所述，可在创建每条规则时为其指定一个标签。标签不会影响过滤，但可用于在您的应用中对规则进行逻辑分组。每条规则只能有一个标签，最大长度为 255 个字符。通过 API 创建时，应在 JSON 格式的规则中包含该标签，具体说明见“匹配规则”页面。

<div id="putting-rules-in-json-format">
  #### 以 JSON 格式编写规则
</div>

要通过 API 在流中添加或删除规则，必须使用 JSON 格式。基本上，需要将每条规则写成以下结构：

`{"value":"insert_rule_here"}`

**包含双引号的规则**

如果规则中包含用于精确匹配或其他运算符的双引号字符（`"`），必须使用反斜杠进行转义，以将其与 JSON 结构区分开来。

`"social data" @XDevelopers`

对应的 JSON 格式为：

`{"value":"\"social data\" @XDevelopers"}`

**包含双引号字符串字面量的规则**

如果要在精确匹配中将双引号作为字符串字面量包含在内，则必须进行双重转义。比如，对于匹配精确短语 "Toys “R” Us" 的规则（包括围绕 "R" 的双引号），其纯文本表示如下：

`"Toys \"R\" Us"`

转换为 JSON 格式时，应使用以下结构：

`{"value":"\"Toys \\\"R\\\" Us\""}`

**带有标签的规则**

如上所述，如需为规则添加可选标签，只需在规则中增加一个 `tag` 字段：

`{"value":"\"social data\" @XDevelopers","tag":"RULE-TAG-01"}`

**API 请求的格式**

通过 API 在流中添加或删除规则时，多个 JSON 格式的规则应以逗号分隔，并放入名为 “rules” 的 JSON 数组中，如下所示：

`{"rules":[{"value":"from:XDevelopers"},{"value":"\"social data\" @XDevelopers","tag":"RULE-TAG-01"}]}`

<div id="operators-that-match-quote-tweets">
  #### 匹配引用推文的运算符
</div>

在使用 [PowerTrack API](/zh/x-api/enterprise-gnip-2.0/powertrack-api) 和 Historical PowerTrack API 时，以下运算符会同时匹配被引用的原始帖子和新的引用推文中的内容。

但如果您使用的是 [Search API](/zh/x-api/enterprise-gnip-2.0/fundamentals/search-api)，这些运算符将只匹配引用推文本身的内容，不会匹配被引用的原始帖子的任何内容。

- `Keywords`
- `Phrases`
- `Proximity`
- `#hashtags`
- `@mentions`
- `$cashtags`
- `url:`
- `url_contains:`
- `has:links`
- `has:mentions`
- `has:hashtags`
- `has:media`
- `has:symbols`
- `is:quote`
- `is:reply`
