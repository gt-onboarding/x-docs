<div id="introduction">
  ### 介绍
</div>

`Enterprise`

帖子是 X 中一切内容的基本原子单位。所有返回帖子的 X API 都会以 JavaScript 对象表示法（JSON）编码提供数据。JSON 由键值对组成，包含具名属性及其对应的值。从 API 检索到的帖子对象包括 X 用户的“状态更新”，转发、回复和引用推文同样都是帖子对象。若某条帖子与另一条帖子相关（如转发、回复或引用推文），相关关系都会在该帖子对象中被标识或内嵌。即便是最简单的、采用原生 X 数据格式的帖子，也会包含嵌套的 JSON 对象，用于表示帖子的其他属性，例如作者、被提及的用户、标注的地点、话题标签、股票代码标签、媒体或 URL 链接。在处理 X 数据时，理解这一点非常重要。你从 X API 接收的帖子数据格式取决于收到的帖子类型、你所使用的 X API，以及格式设置。

返回帖子对象的 Enterprise 端点已更新，以提供理解帖子编辑历史所需的元数据。请在“编辑帖子”基础知识页面了解更多相关元数据：["Edit Posts" fundamentals](/zh/x-api/fundamentals/edit-posts)。

在原生 X 格式中，JSON 负载将包含“根级”属性，以及嵌套的 JSON 对象（此处以 `{}` 表示）：

```JSON
{
	"created_at": "Fri Feb 14 19:00:55 +0000 2020",
	"id_str": "1228393702244134912",
	"text": "开发者在情人节贺卡上写了什么？\n  \nwhile(true) {\n    I = Love(You);  \n}",
	"entities": {
		"hashtags": [],
		"symbols": [],
		"user_mentions": [],
		"urls": []
	},
	"user": {
		"entities": {
			"url": {}
		}
	},
	"place": {}
}
```

<div id="available-data-formats">
  ### 可用的数据格式
</div>

> 请注意：强烈建议在企业数据 API 中使用 [Enriched Native](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) 格式。
>
> - Enriched Native 格式包含自 2017 年以来新增的所有元数据，例如 [投票元数据](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata)，以及 reply\_count、quote\_count 等附加指标。
>
> - 自 2017 年 [字符更新](https://blog.x.com/official/en_us/topics/product/2017/Giving-you-more-characters-to-express-yourself.html) 起，Activity Streams 格式未再新增元数据或增强内容。

企业数据 API 提供两种数据格式。与标准 v1.1 原生格式最接近的企业格式是 Enriched Native。较早的企业数据格式为 Activity Streams，最初由 Gnip 实现，并作为当时在 X 与其他社交媒体数据提供商之间的通用规范化格式使用。尽管该格式仍可用，但自 2017 年以来，X 仅在 Enriched Native 格式上投入新功能与改进。

Enriched Native 格式名副其实，既包含原生的 X 对象，也包含企业数据产品可用的附加增强内容，例如 URL 解链元数据、个人资料地理信息、投票元数据和额外交互指标。

- [扩展与增强的 URL 增强](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls)
- [匹配规则增强](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#matching-rules)
- [投票元数据增强](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata)
- [个人资料地理信息增强](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo)

### 解析最佳实践

- X JSON 使用 UTF-8 编码。
- 解析器应能从容应对字段顺序的变化。应假定 Post JSON 以无序的哈希映射形式提供。
- 解析器应能容忍新增字段。
- 由于并非所有字段都会在所有上下文中出现，JSON 解析器必须能容忍缺失字段。
- 通常可以将被置空的字段、空集合以及字段缺失视为等同

#### 帖子对象

在使用企业级数据产品时，您会注意到数据字典的大部分内容与帖子的原生数据格式相近，并包含一些额外的丰富元数据。原生丰富格式的基础层级使用了与 [X API v1.1 数据格式](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview.html) 大致相同的对象名称。帖子对象具有一长串“根级”属性，包括 `id`、`created_at` 和 `text` 等基础属性。帖子对象还包含嵌套对象，如 `user`、`entities` 和 `extended_entities`。帖子对象还会包含其他[嵌套帖子对象](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#nested-post-objects)，例如 retweeted\_status、quoted\_status 和 extended\_tweet。原生丰富格式还会包含一个 matching\_rules 对象。

#### 嵌套的 Post 对象

在多种情况下，Post 对象会包含其他嵌套对象。如果你在处理嵌套对象，那么该 JSON 负载将包含多个 Post 对象，并且每个 Post 对象都可能包含其自身的对象。根级对象将包含所执行操作的类型信息，即是否为转推（Retweet）或引用推文（Quote Tweet），也可能包含一个描述所分享“原始”Post 的对象。扩展的 Posts 将包含一个嵌套的扩展对象，用于承载超过 140 个字符的内容——这一机制在 2017 年的更新中用于避免破坏性变更。下文逐一说明每个嵌套对象字典。

**Retweets**

Retweet 始终包含两个 Post 对象。被转推的“原始”Post 位于 "retweeted\_status" 对象中。根级对象封装了 Retweet 本身，包括执行转推操作的账户的 User 对象以及转推时间。Retweet 是将 Post 分享给你的关注者的一种操作，且不能添加任何新内容。此外，Retweet 不能提供新的位置信息。即使“原始”Post 可能带有地理标签，Retweet 的 "geo" 和 "place" 对象也将始终为 null。

即使在引入扩展 Posts 之前，根级 "entities" 对象在某些情况下也会被截断且不完整，这是因为会在被转推的 Post 文本前附加字符串 "RT @username "。注意，如果一个 Retweet 再次被 Retweet，"retweet\_status" 依然会指向原始 Post，这意味着中间的 Retweet 不会被包含。使用 x.com“展示”Retweet 时也会出现类似行为。如果你复制分配给该 Retweet“动作”的唯一 Post ID，将会展示原始 Post。

下面是一个 Retweet 的结构示例。同样地，在解析 Retweet 时，关键是解析 "retweeted\_status" 对象，以获取完整的（原始）Post 文本和实体元数据。

```json
{
	"tweet": {
		"text": "RT @author original message",
		"user": {
			"screen_name": "Retweeter"
		},
		"retweeted_status": {
			"text": "original message",
			"user": {
				"screen_name": "OriginalTweeter"
			},
			"place": {},
			"entities": {},
			"extended_entities": {}
		}
	},
	"entities": {},
	"extended_entities": {}
}
```

##### 引用推文

引用推文与转推类似，但它们包含一条新的帖子。新的帖子可以包含其自有的一组话题标签、链接，以及其他“实体”元数据。引用推文还可以包含由发布该引用推文的用户共享的位置信息，以及 GIF、视频和照片等媒体。

引用推文至少包含两个帖子对象，在某些情况下包含三个。被引用的帖子（其本身也可能是一条引用推文）位于“quoted\_status”对象中。根级对象封装了引用推文本身，其中包括执行分享操作的账号的用户对象，以及引用推文的时间。

请注意，现在可以通过“Post”用户界面为引用推文添加照片、GIF 或视频。当引用推文的内容中包含指向外部托管媒体的链接时，根级的“entities.urls”会描述这些链接。附加到引用推文的媒体将出现在根级的“extended\_entities”元数据中。

在引用推文最初推出时，会在“原始”帖子末尾附加一个缩短链接（t.co URL），并呈现在根级的“text”字段中。此外，该 t.co URL 的元数据会包含在根级的“entities.urls”数组中。2018 年 5 月，我们进行了修改，使得指向被引用推文的缩短 t.co URL 将不再包含在根级的“text”字段中。其次，被引用推文的相关元数据将不再包含在“entities.urls”元数据中。取而代之的是，被引用推文的 URL 元数据将位于根级（或顶级）的一个新的“quoted\_status\_permalink”对象中，与“quoted\_status”对象处于同一层级。

<div id="quote-tweets">
  ##### 扩展帖子
</div>

在 2017 年 11 月推出 280 字符的帖子时，引入了用于描述\_扩展帖子\_的 JSON。为在不影响成千上万解析这些基础 X 对象的应用的前提下支持更长的消息，对帖子 JSON 进行了扩展。为实现完全的向后兼容，保留了原始的 140 字符 'text' 字段以及从中解析出的实体对象。对于长度超过 140 字符的帖子，根级 'text' 字段会被截断，因而是不完整的。由于根级 'entities' 对象包含从 'text' 消息中解析出的关键元数据数组（例如包含的标签和链接），这些集合也会不完整。举例来说，如果一条帖子为 200 个字符，且在末尾包含一个标签，那么传统的根级 'entities.hashtags' 数组将不会包含该标签。

为此，引入了新的 'extended\_tweet' 字段，用于保存较长的帖子消息及完整的实体元数据。当长度超过 140 字符时，"extended\_tweet" 对象会提供包含完整、未截断帖子消息的 "full\_text" 字段。"extended\_tweet" 对象还包含一个 "entities" 对象，其中提供完整的标签、链接、提及等数组。

扩展帖子通过根级的 "truncated" 布尔值标识。当其为 true（"truncated": true）时，应解析 "extended\_tweet" 字段而非根级字段。

请注意下面的 JSON 示例：尽管帖子消息包含三个标签，根级 "text" 字段已被截断，且根级 "entities.hashtags" 数组为空。由于这是一个扩展帖子，"truncated" 字段被设为 true，"extended\_tweet" 对象提供了完整的 "full\_text" 和 "entities" 帖子元数据。

```json
{
	"created_at": "Thu May 10 17:41:57 +0000 2018",
	"id_str": "994633657141813248",
	"text": "Just another Extended Tweet with more than 140 characters, generated as a documentation example, showing that [\"tru… https://t.co/U7Se4NM7Eu",
	"display_text_range": [0, 140],
	"truncated": true,
	"user": {
		"id_str": "944480690",
		"screen_name": "FloodSocial"
	},
	"extended_tweet": {
		"full_text": "Just another Extended Tweet with more than 140 characters, generated as a documentation example, showing that [\"truncated\": true] and the presence of an \"extended_tweet\" object with complete text and \"entities\" #documentation #parsingJSON #GeoTagged https://t.co/e9yhQTJSIA",
		"display_text_range": [0, 249],

				"indices": [211, 225]
			}, {
				"text": "parsingJSON",
				"indices": [226, 238]

				"indices": [239, 249]
			}]
		}

		"hashtags": []
	}
}
```

### 原生增强地理对象

帖子可以关联一个位置，从而变为“地理标记”帖子。可以通过 X 的用户界面为帖子分配位置，或在使用 API 发帖时进行分配。帖子位置可以是精确的“点”坐标，或是一个带有“边界框”的 X Place，用于描述更大的区域，范围可从某个场馆到整个地区。

有三个“根级” JSON 对象用于描述与帖子关联的位置：[place](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#place-data-dictionary)、[geo](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-geo-objects) 和 [coordinates](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#coordinates-object-data-dictionary)。

此外，原生增强格式还会在用户对象内包含 [profile geo enrichment 的](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo) [派生位置](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#derived-locations)。

当帖子使用某个 place 进行地理标记时，`place` 对象始终存在。Place 是具有相应地理坐标的特定命名位置。当用户决定为其帖子分配位置时，会呈现一组候选的 X Place 列表。使用 API 发帖时，可以通过在发帖时指定 place\_id 来附加一个 X Place。与 Place 关联的帖子不一定是在该位置发布的，也可能只是与该位置\_相关\_。

`geo` 和 `coordinates` 对象仅在帖子被分配了\_精确位置\_时才会出现（非空）。如果提供了精确位置，`coordinates` 对象将返回一个包含地理坐标的 \[long, lat] 数组，并会分配与该位置相对应的一个 X Place。

<div id="x-entities">
  ## 简介
</div>

实体为在 X 上发布的内容提供元数据和额外的上下文信息。`entities` 部分包含帖子中常见元素的数组：话题标签、用户提及、链接、股票代码（符号）、X 投票以及所附媒体。由于 X 已对文本正文完成预处理/预解析，开发者在摄取帖子时可直接复用这些数组。你的解析器无需在帖子正文里显式检索这些实体，可直接读取该 JSON 区段即可获取。

除了便于解析，`entities` 部分还提供了有用的“增值”元数据。比如，当你使用[增强型 URL 丰富](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls)时，URL 元数据会包含完全展开的 URL，以及关联的网站标题和描述。再比如，当存在用户提及时，实体元数据会包含数值型用户 ID，这在调用许多 X API 时非常有用。

每个帖子的 JSON 负载都包含一个 `entities` 部分，其中至少包括 `hashtags`、`urls`、`user_mentions` 和 `symbols` 属性，即使这些实体并未出现在帖子内容中也是如此。例如，如果你查看一条正文为 “Hello World!” 且未附带媒体的帖子的 JSON，其中将包含如下内容，且各实体数组的项数为 0：

```
"entities": {
        "hashtags": [
        ],
        "urls": [
        ],
        "user_mentions": [
        ],
        "symbols": [
        ]
      }
```

注意：

- 仅当帖子中包含相应类型的内容时，media 和 polls 实体才会出现。
- 如果你使用原生媒体（照片、视频或 GIF），请使用[扩展实体对象](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#x-extended-entities)。

### 实体对象

`entities` 和 `extended_entities` 两个部分均由实体对象的数组组成。下面将介绍这些实体对象，并提供数据字典，用于说明对象的属性名称、类型以及简短描述。我们还会标注哪些 PowerTrack 运算符可匹配这些属性，并附上一些示例 JSON 载荷。

这是帖子中常见实体的集合，包括话题标签、链接和用户提及。`entities` 对象确实包含一个 `media` 属性，但该属性在 `entities` 部分的实现只有在帖子包含单张照片时才完全准确。对于包含多张照片、视频或动画 GIF 的帖子，请参阅 `extended_entities` 部分。

### 话题标签对象

PowerTrack 的 `#` 运算符用于匹配 `text` 属性。当数组中至少有一项时，`has:hashtags` 运算符将匹配。

|     |     |     |
| :--- | :--- | :--- |
| 字段 | 类型 | 描述 |

### 媒体对象  

如果帖子“附加”了任何媒体对象，`entities` 部分会包含一个 `media` 数组，其中只含一个媒体对象。如果未附加原生媒体，则 `entities` 中不会出现 `media` 数组。基于以下原因，处理帖子的原生媒体应使用 `extended_entities` 部分：
\+ 媒体的 `type` 将始终显示为 “photo”，即使帖子中附加的是视频或 GIF。
\+ 即便最多可附加四张照片，`entities` 部分中也只会列出第一张。

### 照片媒体 URL 格式

X 上的照片媒体支持按不同尺寸加载。建议加载能够覆盖目标视口、且尽可能小的图像尺寸。要加载不同尺寸，需要按特定格式将 [Size Object](#size) 与 [media\_url](#media)（或 [media\_url\_https](#media)）组合。我们将使用前文提供的 [media entity](#entitiesobject) 示例对象来构造照片媒体 URL。

单独加载 `media_url` 或 `media_url_https` 也可使用，默认会加载中等大小的变体。但在条件允许时，更推荐提供完整格式化的照片媒体 URL。

照片媒体 URL 由三部分组成：

|     |     |
| :--- | :--- |
| Base URL | Base URL 指去掉文件扩展名的媒体 URL。<br /><br />例如：  <br /><br />"media\_url\_https": "https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg",<br /><br />则 Base URL 为：  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq_ |
| Format | Format 指图像的文件格式。可选格式为 _jpg_ 或 _png_，由媒体 URL 的扩展名决定。<br /><br />例如：  <br /><br />"media\_url\_https": "https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg",<br /><br />则格式为：_jpg_ |
| Name | Name 指要加载的尺寸字段名。<br /><br />例如：<br /><br />  <br /> "sizes":   <br />   "thumb":   <br />     "h": 150,  <br />     "resize": "crop",  <br />     "w": 150  <br />   ,  <br />   "large": <br />     "h": 1366,  <br />     "resize": "fit",  <br />     "w": 2048  <br />   },  <br />   "medium": <br />     "h": 800,  <br />     "resize": "fit",  <br />     "w": 1200  <br />   },  <br />   "small": <br />     "h": 454,  <br />     "resize": "fit",  <br />     "w": 680  <br />   }  <br /> }  <br />}<br /><br />当加载大尺寸照片时，Name 为：_large_ |

将这三部分（Base URL、Format 和 Name）组合即可得到要加载的照片媒体 URL。以这种方式加载图像有两种格式：_legacy_ 和 _modern_。所有图像加载应停止使用 _legacy_ 格式，改用 _modern_ 格式。使用 _modern_ 格式可提高调用方的 CDN 命中率，从而降低从数据中心生成并加载媒体的概率，改善加载延迟。

|     |     |
| :--- | :--- |
| 旧格式 | 旧格式已弃用。所有照片媒体加载都应迁移到现代格式。<br /><br />_\<base\_url>.\<format>:\<name>_  <br /><br />例如：  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg:large  <br />_ |
| 现代格式 | X 于 2015 年确立了用于加载照片的现代格式，并自 2017 年起成为事实标准。所有照片媒体加载都应迁移到此格式。<br /><br />_\<base\_url>?format=\<format>\&name=\<name>_  <br /><br />例如：  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq?format=jpg\&name=large  <br />_<br /><br />注意：照片媒体 URL 的查询字符串参数按字母顺序排列。若将来添加新的查询参数，仍需保持字母顺序。例如，若有一个名为 _preferred\_format_ 的假设新参数，它应在查询字符串中排在 _format_ 和 _name_ 之后。 |

### URL 对象 

`entities` 部分包含一个 `urls` 数组，其中每个对象对应帖子正文中的一个链接；如果没有链接，则该数组为空。

当数组中至少有一项时，`has:links` 运算符会匹配。`url:` 运算符用于匹配 `expanded_url` 属性。如果你使用 [Expanded URL enrichment](http://support.gnip.com/enrichments/expanded_urls.html)，则 `url:` 运算符用于匹配 `unwound.url`（完全展开的 URL）属性。如果你使用 [Enhanced URL enrichment](http://support.gnip.com/enrichments/enhanced_urls.html)，则 `url_title:` 和 `url_description:` 运算符用于匹配 `unwound.title` 和 `unwound.description` 属性。

|     |     |     |
| :--- | :--- | :--- |
| 字段 | 类型 | 说明 |
| display\_url | String | 粘贴/输入到帖子中的 URL。示例：<br /><br />"display\_url":"bit.ly/2so49n2" |
| expanded\_url | String | `display_url` 的展开版本。示例：<br /><br />"expanded\_url":"http://bit.ly/2so49n2" |
| indices | Array of Int | 一个整数数组，表示 URL 在帖子文本中开始和结束的偏移位置。第一个整数表示 URL 在帖子文本中首个字符的位置；第二个整数表示 URL 结束后首个非 URL 字符的位置。示例：<br /><br />"indices":\[30,53] |
| url | String | 包装后的 URL，对应直接嵌入原始帖子文本的值，以及 indices 参数的值。示例：<br /><br />"url":"https://t.co/yzocNFvJuL" |

如果你使用了 Expanded 和/或 Enhanced URL enrichment，以下元数据可在 `unwound` 属性下获得：

|     |     |     |
| :--- | :--- | :--- |
| 字段 | 类型 | 说明 |
| url | String | 帖子中包含的链接的完全展开版本。示例：<br /><br />"url":"https://blog.x.com/en\_us/topics/insights/2016/using-twitter-as-a-go-to-communication-channel-during-severe-weather-events.html" |
| status | Int | 展开过程的最终 HTTP 状态码，200 表示成功。示例：<br /><br />200 |
| title | String | 链接的 HTML 标题。示例：<br /><br />"title":"Using X as a ‘go-to’ communication channel during severe weather" |
| description | String | 链接的 HTML 描述。示例：<br /><br />"description":"Using X as a ‘go-to’ communication channel during severe weather" |

## 转发与引用推文详情

从 X API 的角度来看，转发与引用推文是将原始帖子作为嵌入对象包含在内的特殊类型帖子。因此，转发与引用推文对象是包含一个“原始”子帖子的父对象（因此体积加倍）。转发对象包含顶层的 "retweeted\_status" 对象，引用推文对象包含 "quoted\_status" 对象。为保持一致性，这些顶层的转发与引用推文对象也具有 text 属性及其关联的 entities。不过，顶层的 entities 可能与嵌入的“原始”帖子中的 entities 不同。对于转发，会在原始帖子正文前添加新文本；对于引用推文，会在帖子正文后追加新文本。

通常的最佳实践是（如存在）从 retweeted\_status 中的原始帖子获取文本、entities、原作者和日期。例外情况是获取属于附加引用部分的 X 实体。更多详情与提示见下文。

### 转推

关于转推的一个重要细节是，无法在该帖子中添加额外的 X 实体。用户在转推时不能添加话题标签、URL 或其他内容。不过，转推（顶层）的 text 属性由原帖文本组成，并在前面加上“RT @username: ”。

在某些情况下，尤其是用户名较长的账号，新增的这些字符与原帖正文相加，很容易超过原帖 140 个字符的文本长度限制。为保持对基于 140 字符的显示与存储的兼容，顶层正文会截断帖文末尾并添加省略号（“…”）。因此，原帖末尾位置的某些顶层实体可能不正确或缺失，例如被截断的话题标签或 URL 条目。

此帖子 https://x.com/FloodSocial/status/907974220298125312 的帖子文本如下：

Just another test Post that needs to be exactly 140 characters with trailing URL and hashtag ![](https://abs.twimg.com/emoji/v2/72x72/1f4a7.png "Droplet")![](https://abs.twimg.com/emoji/v2/72x72/1f4a7.png "Droplet")![](https://abs.twimg.com/emoji/v2/72x72/1f4a7.png "Droplet")![](https://abs.twimg.com/emoji/v2/72x72/1f4a7.png "Droplet")![](https://abs.twimg.com/emoji/v2/72x72/1f4a7.png "Droplet")![](https://abs.twimg.com/emoji/v2/72x72/1f4a7.png "Droplet")![](https://abs.twimg.com/emoji/v2/72x72/1f4a7.png "Droplet")![](https://abs.twimg.com/emoji/v2/72x72/1f4a7.png "Droplet")![](https://abs.twimg.com/emoji/v2/72x72/1f4a7.png "Droplet")![](https://abs.twimg.com/emoji/v2/72x72/1f4a7.png "Droplet")![](https://abs.twimg.com/emoji/v2/72x72/1f4a7.png "Droplet")![](https://abs.twimg.com/emoji/v2/72x72/1f4a7.png "Droplet")![](https://abs.twimg.com/emoji/v2/72x72/1f4a7.png "Droplet")  [http://wapo.st/2w8iwPQ](https://t.co/R5iMzxWelp "http://wapo.st/2w8iwPQ")  [#Testing](https://x.com/hashtag/Testing?src=hash)

在上述示例中，URL 和话题标签都受到了影响。由于话题标签被完全截断而 URL 被部分截断，它们在顶层实体中缺失。你还会注意到，由于 text 字段前缀中的“RT @floodsocial: ”，出现了额外的 user\_mentions 顶层实体。

不过，retweeted\_status 中的帖子文本和实体完全反映了原帖，没有截断或错误的实体，因此我们建议在处理转推时依赖嵌套的 retweeted\_status 对象。

<div id="retweets">
  ### 引用推文
</div>

引用推文于 2016 年推出，与转推不同之处在于，当你“引用”一条帖子时，会在被分享的帖子“之上”添加新的内容。该新内容几乎可以包含原帖所具备的一切，包括新文本、话题标签、提及和 URL。

引用推文可以包含原生媒体（照片、视频和 GIF），并会出现在 entities 对象中。

由于可以新增 X 实体，引用中的实体很可能与原帖的实体不同。

在此示例中，一个新的 URL 和话题标签被放在了引用推文的末尾。

这条帖子 https://x.com/FloodSocial/status/907983973225160704 的正文如下：

                  当岛屿被淹时既奇怪又同样悲伤…… 引用推文的跨大西洋测试 | [@thisuser](https://x.com/andypiper) [@thatuser](https://x.com/johnd)[http://bit.ly/2vMMDuu](https://t.co/fzFgMhWlPO "http://bit.ly/2vMMDuu")  [#testing](https://x.com/hashtag/testing?src=hash)

在这种情况下，顶层 entities 并未反映引用的细节。 

不过，extended\_tweet 中的正文和 entities 能完整、准确地反映引用推文，没有截断或错误的实体，因此我们建议在处理引用推文时依赖嵌套的 \_extended\_tweet 对象。

<div id="quote-tweets">

#### 介绍

如果帖子包含原生媒体（通过帖子界面直接附加，而非通过外部链接），则还会有一个 extended\_entities 部分。对于任何原生媒体（照片、视频或 GIF），extended\_entities 出于多种原因是首选的元数据来源。目前，一条帖子最多可以附加四张照片。entities 元数据只会包含第一张照片（2014 年之前只能包含一张），而 extended\_entities 部分会包含所有附加的照片。对于原生媒体，entities.media 元数据的另一个不足在于，其媒体类型始终标示为“photo”，即使附加的是视频或动画 GIF。实际的媒体类型在 extended\_entities.media\[].type 属性中指定，取值为 photo、video 或 animated\_gif。基于以上原因，处理原生媒体时应优先使用 extended\_entities 元数据。

所有附有照片、视频或动画 GIF 的帖子都会包含一个 `extended_entities` JSON 对象。`extended_entities` 对象包含一个 `media` 数组，由 `media` 对象构成（其数据字典见 `entities` 部分）。`extended_entities` 不包含其他实体类型，如话题标签和链接。`extended_entities` 中的 `media` 对象在结构上与 `entities` 中的完全一致。

每条帖子一次只能附加一种类型的媒体。照片最多可附加四张；视频和 GIF 最多可附加一个。由于 `extended_entities` 中媒体的 `type` 元数据能正确指示媒体类型（“photo”、“video” 或 “animated\_gif”），且支持最多 4 张照片，因此它是原生媒体的首选元数据来源。

```
{
      "extended_entities": {
        "media": [

        ]
      }
    }
```

### 活动对象

Activity Streams 是由 Gnip 创建的一种对象架构映射，用于将 X 的[原始数据格式](https://developer.x.comhttps://developer.x.com/en/docs/x-api/v1/data-dictionary/overview)“规范化”为帖子数据和其他社交媒体数据，采用第三方的[Activity Base Schema（见此处）](https://activitystrea.ms/head/activity-schema.html)。帖子会被规范化为 Activity Streams 架构，其中包含 note、person、place 和 service 等对象类型作为嵌套对象。帖子还可以包含其他嵌套的帖子活动对象（用于转发），或包含 twitter\_quoted\_status、long\_object 等。

基础级别的对象类型“activity”与原生富化格式中的[帖子基础级对象](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object)类似。可在[此处](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#activity-streams-payload-examples)查看 Activity Streams 格式的示例载荷。

### 位置对象

位置对象可以存在于 X 账户级别设置的 actor 对象中，或存在于 [gnip 对象](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#gnip-object) 的 profileLocations 对象中。位置对象的类型为 place，并且可以包含名称、地址或地理坐标。位置对象与原生增强格式中的 [Geo](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-geo-objects) 对象类似。

#### 位置数据字典

| 字段 | 类型 | 描述 |
| :--- | :--- | :--- |
| objectType | string | 有关更详细的信息请参见[此处](http://activitystrea.ms/head/activity-schema.html#place)。示例：<br /><br />**"objectType":** "place" |
| displayName | string | 位置的全称。<br /><br />**"displayName":** "United States" |
| name | string | 来自 X 的 place JSON 格式的地点名称。 |
| link | string | 指向该地点完整 X JSON 表示的链接。<br /><br />**"link":** "https://api.x.com/1.1/geo/id/27c45d804c777999.json" |
| geo | object | 来自 X 的地理坐标对象，可为多边形或点。<br /><br />参见 [geo](https://developer.x.com/en/docs/x-api/v1/data-dictionary/object-model/geo#coordinates) |
| countryCode | String | 表示包含该地点的国家/地区的二位国家代码。示例：<br /><br />**"countryCode": "US"** |
| country | String | 包含该地点的国家/地区名称。示例：<br /><br />**"country": "United States"** |

### 引言\*\*

从本质上讲，X 是一个公开、实时、全球性的通信网络。自 2006 年以来，X 的演进既受到用户使用模式与社区约定的推动，也得益于新产品功能与增强。如果你将 X 数据用于历史研究，理解这一演进的时间线对于从数据档案中发现感兴趣的帖子至关重要。

X 最初作为一个简单的短信移动应用推出，后来发展为一个综合性的通信平台——一个拥有完整 API 集的平台。API 一直是 X 网络的支柱。[第一版 API 在 X 推出后不久就发布](https://blog.x.com/2006/introducing-the-twitter-api)。2009 年首次引入帖子地理标记时，该功能通过 [Geo API](https://blog.x.com/2009/think-globally-tweet-locally) 提供（随后“地理标记”帖子的能力被集成到 X.com 的用户界面中）。如今，X 的 API 驱动着这一双向通信网络，它已成为突发新闻与信息共享的来源。基于这一全球、实时通信渠道进行构建的机会无穷无尽。

X 提供了两个历史 API，可访问每一条公开可用的帖子：Historical PowerTrack 和 [Full-Archive Search API](https://developer.x.com/en/docs/x-api/v1/tweets/search/overview)。这两个 API 都提供一组用于查询与收集目标帖子的“运算符”。这些运算符可匹配与每条帖子相关的多种属性——多达数百种，例如帖子的文本内容、作者的账户名，以及帖子中分享的链接。帖子及其属性采用 JSON 编码，这是一种常见的基于文本的数据交换格式。因此，随着新功能的引入，新的 JSON 属性会随之出现，通常也会引入新的 API 运算符以匹配这些属性。如果你的用例需要“聆听”世界在 X 上说过的话，那么越能理解各运算符从何时开始拥有可供匹配的 JSON 元数据，你的历史 PowerTrack 过滤器就会越有效。

接下来，我们将介绍一些关键概念，为理解帖子元数据的更新如何影响你定位所需的数据信号奠定基础。

### 关键概念\*\*

#### 从用户约定到 X 的一等对象

X 用户自发引入了新的、如今已成为基础的交流模式。一个开创性的例子是话题标签（hashtag），如今几乎在所有社交网络中普遍使用。话题标签最初被用于组织对话与主题。在一个每天产生数亿条消息的网络中，帮助查找感兴趣帖子的工具至关重要，而话题标签已成为一种基础方法。随着话题标签的使用增长不久后，它们获得了 X 的官方认可与支持。当话题标签成为“_一等_”对象时，这意味着多方面的变化：它们在 X.com 的用户界面中变得可点击、可搜索；它们也与 @提及、附加媒体、股票代码和分享链接一道，成为 X 的 _entities_ 家族成员。这些实体被方便地编码在一个预解析的 JSON 数组中，便于开发者处理、扫描和存储。

转发（Retweet）是另一个由用户约定演变为官方对象的例子。转发最初作为将内容“转发”给他人的方式出现，起初是手动复制粘贴一条帖子，并在前面加上“RT @”模式。这个流程最终通过新的转发按钮实现了自动化，并配套新的 JSON 元数据。“官方”转发由此诞生。其他例子包括“提及”、分享媒体和网页链接，以及在帖子中分享位置。上述每一种用法都催生了新的 [x.com](https://x.com/) 用户界面功能、相应的 JSON 支持，从而带来了匹配帖子的全新方法。所有这些基础的帖子属性最终都发展为用于匹配它们的 PowerTrack 运算符。

<div id="from-user-conventions-to-x-first-class-objects">
  #### 帖子元数据、可变性、更新与时效性
</div>

虽然帖子正文有固定的最大字符数限制，但帖子的 JSON 描述包含 100 多个属性。这些属性包括由谁在何时发布、是原创帖子还是转发，以及一个由一等对象组成的数组，例如话题标签、提及和分享的链接。对于发布该帖子的账号，会有一个用户（或参与者）对象，包含多种属性，提供该用户的个人资料以及其他账号元数据。个人资料包括简短的个人简介、主页位置（自由文本）、首选语言，以及可选的网站链接。

有些账号元数据永不更改（例如数值型用户 ID 和创建日期），有些会随时间缓慢变化，而另一些属性则更为频繁地变化。人们会更换工作、搬家；公司会更新其信息。当你收集历史帖子时，理解哪些元数据是“发布当时的状态”，而哪些元数据是“提交查询时的状态”非常重要。

对于所有历史类 API，用户的个人资料简介、显示名称以及个人资料中的“主页位置”属性都会更新为查询时的取值。

<div id="post-metadata-mutability-updates-and-currency">
  #### “原生”媒体
</div>

X.com 和 X 移动应用支持通过点击按钮并从相册中选择，将照片和视频添加到帖子中。由于这已被集成为一等操作，以这种方式分享的视频和照片称为“原生”媒体。

许多查询运算符可用于这些“原生”资源，包括 `has:videos`、`has:images` 和 `has:media`。它们只会匹配通过 X 功能分享的媒体内容。若要匹配托管在 X 平台之外的媒体，请使用基于 URL 元数据的运算符。

因此，在深入介绍 Historical PowerTrack 和 全档案搜索 的产品细节之前，让我们先回顾一下 X 作为产品与平台随时间的演进。

**过滤提示**

熟悉 X 的时间线以及新功能何时、如何添加，有助于你创建更有效的查询。这里所说的查询指应用于帖子档案的“过滤器”或“规则”，由 X 历史 API 使用 PowerTrack Operators 在 Post JSON 上进行匹配。比如 `lang:` Operator，用于匹配指定语言的帖子。X 提供语言分类服务（支持 50 多种语言），X API 会在为每条帖子生成的 JSON 中提供这些元数据。因此，如果一条帖子是用西班牙语撰写的，则“lang” JSON 属性会被设置为“es”。因此，如果你构建带有 `lang:es` 子句的过滤器，它只会匹配被分类为西班牙语的帖子消息。

时间线信息也有助于更好地解读接收到的帖子数据。假设你在研究 2008 年和 2012 年夏季奥运会相关内容的分享。如果你只应用 `is:retweet` Operator 来匹配转发，2008 年将不会有任何匹配数据；而在 2012 年则可能会有数百万条转发。你可能会据此错误地得出结论：在 2008 年，转发并不是一种用户惯例，或者根本没有人转发那届奥运会的内容。由于转发在 2009 年才成为一等对象，你需要添加 `“RT @”` 规则子句来帮助识别 2008 年的相关内容。

转发和帖子语言分类都是具有悠久历史且包含诸多产品细节的帖子属性示例。下文将讨论这些以及其他对匹配和理解 X 数据至关重要的属性类别的更多细节。

<div id="2017">
  #### 识别假阴性
</div>

在编写过滤器时，需要牢记的一点是：所有可供运算符匹配的元数据都有“生效”日期。如果你使用的运算符依赖于在该帖子发布之后才引入的元数据，就会产生假阴性。比如，你想查找所有提到“snow”且分享视频的帖子。如果你使用匹配含有“原生”视频帖子的 `has:videos` 运算符来构建规则，那么该子句将无法匹配到2015年之前的任何帖子。

不过，早在2015年之前，在 X 上分享视频就很普遍。那时用户通常分享托管在其他网站的视频链接；到了2015年，X 才把“分享视频”功能直接内置到平台中。要找到这些更早的相关帖子，你应该加入类似 `url:"youtube.com"` 的规则子句。

请注意，在 Search API 中，索引重建时会对部分元数据进行“回填”。一个典型例子是 $cashtags：2009 年起它被广泛用于讨论股票代码。2015 年引入 $cashtag 运算符后，Search 索引被重建，在此过程中会从所有帖子正文中提取符号实体，甚至包括 2006 年早期当 `$` 主要用于俚语时的内容：“I hope it $now$ $oon!”。

<div id="recognizing-false-negatives">
  #### 识别并基于与你的用例相关的重要帖子属性进行筛选
</div>

某些元数据（例如 X 账户的数值 ID）自平台上线之初即已存在（也是从不变化的账户元数据的一个示例）。其他元数据直到 X 于 2006 年启动后很久才引入。新引入的元数据示例包括转发相关元数据、帖子地理位置、URL 标题与描述，以及“原生”媒体。下方列出了在这些 X 平台更新中受到根本性影响的一些最常见帖子属性类型。

这些属性的筛选/匹配行为在多数情况下取决于所使用的历史帖子 API。为帮助你确定哪款产品最适合你的研究与用例，下面提供的属性详情包含高层级的产品信息。

<div id="identifying-and-filtering-on-post-attributes-important-to-your-use-case">
  #### X 个人资料
</div>

鉴于 X 的本质是一个全球性的实时通信渠道，基于 Post 数据的研究往往更关注“是谁在发声”。了解某位 X 用户的常驻地通常很有帮助。知道账号简介中提及的兴趣与爱好，常常能引导你发现相关的 Post。监听来自目标账号的 Posts 也十分常见。以上这些用例的核心都在于个人资料属性。

X 上的每个账号都有一个个人资料，其中包含诸如 X @handle、显示名称、简短简介、常驻地（用户输入的自由文本）、粉丝数量等元数据。某些属性永不改变，例如数值型用户 ID 和账号创建时间。其他属性通常会按日、按周或按月变化，例如发布的 Posts 数量、关注数和粉丝数。还有一些属性虽可随时变更，但总体变更频率较低：显示名称、常驻地和简介。

每条 Post 的 JSON 负载都包含该 Post 作者的账号个人资料元数据。若为转发（Retweet），还会包含发布原始 Post 的账号的个人资料元数据。

Post 的个人资料元数据的可变性完全取决于所使用的历史产品。Search APIs 提供的历史 Posts，其个人资料设置为检索时的状态。对于 Historical PowerTrack，个人资料为 Post 发布当时的状态，但不包含 2011 年之前的数据。对于早于 2011 年的 Posts，其个人资料元数据反映的是 2011 年 9 月时的资料状态。

<div id="x-profiles">
  #### 原始帖子与转发
</div>

转发是用户自发约定演变为官方对象的又一例证。转发最初作为把内容“转发”给他人的方式出现，起初是手动流程：复制/粘贴一条帖子，并在前面加上“RT @”模式。随后通过新增的转发按钮实现了自动化，并配套了新的 JSON 元数据。“官方”转发由此诞生，转发这一操作成为一等帖子事件。随同新的转发按钮，还引入了新的元数据，例如原始帖子的完整载荷。

判断帖子是原创还是分享，是一个常见的过滤“开关”。在某些情况下，只需要原创内容；在另一些情况下，帖子互动是首要考量，因此转发至关重要。PowerTrack `is:retweet` 运算符可用于包含或排除转发。如果拉取 2009 年 8 月之前的数据，用户需要采用两种不同的转发匹配（或不匹配）策略。对于 2009 年 8 月之前，需要检查帖子正文本身，使用精确短语匹配查找“RT @”模式。对于 2009 年 8 月之后的时期，可使用 `is:retweet` 运算符。

<div id="original-post-and-retweets">
  #### 帖子语言分类
</div>

帖子所使用的语言是一个普遍关注点。帖子语言有助于推断帖子的地理位置，且在分析或展示中通常只需要特定语言。（X 个人资料也有首选语言设置。）

在按帖子语言分类进行筛选时，X 的历史产品（[Search API](https://developer.x.com/en/docs/x-api/v1/tweets/search/overview/enterprise) 和 Historical PowerTrack）有所不同。构建搜索存档时，所有帖子都被回填了 X 的语言分类，因此可在整个帖子存档中使用 `lang:` 运算符。对于 Historical PowerTrack，X 的语言分类元数据在存档中自 2013 年 3 月 26 日起可用。 

<div id="post-language-classifications">
  #### 帖子的地理定位
</div>

能够确定帖子发布的位置（即对其进行地理定位）对许多用例都很重要。对帖子进行地理定位主要有三种方法：

- 帖子内容中的地理信息引用
- 由用户地理标注的帖子
- 用户在账号资料中设置的“所在地”位置

##### 帖子内容中的地理引用

在帖子内容中匹配地理引用，尽管这种方法通常最具挑战性（因其依赖本地知识），但可应用于整个帖子档案。以下是一个基于“golden gate”筛选条件、针对旧金山地区的 2006 年地理引用匹配示例：

https://x.com/biz/statuses/28311

<div id="geographical-references-in-a-post-message">
  ##### 由用户地理标记的帖子
</div>

2009 年 11 月，X 推出 Post Geotagging API，使帖子可以附加精确位置的地理标记。2010 年 6 月，X 推出 X Places，用于表示场馆、街区或城镇尺度的地理区域。约有 1–2% 的帖子通过上述任一方式进行了地理标记。

可用的地理标记历史取决于你使用的历史 API。使用 Search API 时，部分地理运算符（Geo Operators）自 2010 年 3 月起即可用于匹配帖子，其他运算符自 2015 年 2 月起可用。若你使用 Historical PowerTrack，地理引用自 2011 年 9 月 1 日开始。构建 Historical PowerTrack 归档时，该日期之前的所有地理标记均未被纳入。

<div id="posts-geo-tagged-by-the-user">
  ##### 由用户设置的账户资料“居住地”位置
</div>

所有 X 用户都可以设置其资料位置，用于标示其居住地。数以百万计的 X 用户提供了该信息，显著增加了 X Firehose 中的地理数据量。此位置元数据为非规范化、由用户生成的自由格式字符串。约有 30% 的账户具备可解析到国家级别的资料地理元数据。

与帖子地理信息类似，匹配方法和可用的时间范围取决于所使用的历史 API。Historical PowerTrack 允许用户对这些自由格式字符串进行自定义匹配。为简化该过程，X 还提供了[资料地理增强](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo)，在可行时执行地理编码，并提供规范化的元数据及相应的运算符。资料地理运算符在 Historical PowerTrack 和 Search API 中均可用。在 Historical PowerTrack 中，这些资料地理元数据自 2014 年 6 月起可用；在 Search API 中，该元数据自 2015 年 2 月起可用。

#### 共享链接与媒体

在 X 上共享网页链接、照片和视频，一直是核心用例。历史上，这些操作最初都需要在帖子正文中直接包含一个 URL 链接。2011 年，X 在其用户界面中原生支持了照片分享。2016 年，又加入了原生视频。

基于这段历史，出现了多种用于匹配此类内容的筛选运算符。有一组运算符可用于判断帖子是否包含共享的链接、照片或视频。此外，由于 X 上的大多数 URL 会被缩短以减少帖子字符占用（例如由 bitly 或 tinyurl 等服务生成），X 提供了数据增强，可生成完整的展开 URL 供匹配使用。比如，如果你想匹配包含讨论 X 与预警系统的链接的帖子，引用“severe weather communication”的过滤器就会匹配到包含该 [http://bit.ly/1XV1tG4](http://bit.ly/1XV1tG4) URL 的帖子。

2012 年 3 月，推出了[展开 URL 增强](/zh/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls)。在此之前，帖子载荷中仅包含用户提供的 URL。因此，如果用户使用了短链接，就很难匹配到你感兴趣的（展开后的）URL。在 Historical PowerTrack 和 Search API 中，这些元数据自 2012 年 3 月起可用。

2016 年 7 月，又推出了增强 URL 增强。该增强版会在帖子载荷中提供网站的 HTML 标题与描述，并提供相应的匹配运算符。对于 Historical PowerTrack，这些元数据自 2016 年 7 月起可用；对于 Search API，这些元数据自 2014 年 12 月开始陆续出现。

2016 年 9 月，X 引入了“原生附件”，即末尾共享链接不计入 140 个字符的帖子长度限制。上述两种 URL 增强仍适用于这些共享链接。

有关 URL 筛选的其他、产品相关的详细信息，请参阅相应文章。
