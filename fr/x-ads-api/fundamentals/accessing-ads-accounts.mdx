<div id="user-accounts-vs-ad-accounts">
  # Comptes utilisateurs vs comptes publicitaires
</div>

Il existe deux types de comptes distincts impliqués dans l’utilisation de l’API publicitaire X : les comptes publicitaires et les comptes utilisateurs X. Dans toute la documentation de l’API publicitaire X, le terme « compte » fait généralement référence au compte publicitaire.

- Les comptes publicitaires sont enregistrés sur **business.x.com** et identifiés dans l’API par account\_id. Ils sont directement liés aux sources de financement et utilisent le contenu d’un ou plusieurs comptes utilisateurs X en tant qu’« utilisateurs promouvables ». Chaque compte publicitaire peut accorder des autorisations à un ou plusieurs comptes utilisateurs X. Le compte publicitaire, ou « compte actuel », est présent dans presque toutes les URL exécutées sous la forme d’un paramètre en ligne :account\_id.
- Les comptes utilisateurs X (par exemple @AdsAPI) sont identifiés par user\_id dans l’API publicitaire X. Un ou plusieurs de ces comptes peuvent être associés à un compte publicitaire. Le compte utilisateur X authentifié qui effectue des requêtes sur l’API est appelé « utilisateur actuel ». La liste des comptes publicitaires auxquels l’utilisateur actuel a accès peut être obtenue via [GET accounts](/fr/x-ads-api/campaign-management#accounts). Les « utilisateurs promouvables » sont des identifiants X (handles) pouvant être promus par un compte publicitaire spécifique. Pour plus de détails, voir [Obtaining Ads Account Access](/fr/x-ads-api/introduction).

<div id="methods-for-ad-account-access">
  ## Méthodes d’accès au compte publicitaire
</div>

Il existe deux méthodes pour effectuer des requêtes vers l’API publicitaire pour le compte d’un annonceur :

1. Effectuer des requêtes au nom d’un annonceur (**recommandé**)
2. Effectuer des requêtes en utilisant votre compte auquel l’accès au compte d’un annonceur a été accordé, par exemple une [agence](https://business.x.com/en/blog/agencies-manage-multiple-Twitter-Ads-accounts.html) gérant plusieurs comptes.

Ce document propose un bref aperçu des différences entre ces options et doit être utilisé conjointement avec nos autres ressources, comme la [FAQ sur la connexion multi-utilisateur](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html).

Comme décrit dans [Autoriser une requête](/fr/resources/fundamentals/authentication#authorizing-a-request#authorizing-a-request), toutes les requêtes vers l’API publicitaire nécessitent des en-têtes Authorization utilisant [OAuth 1.0a](/fr/resources/fundamentals/authentication) avec un jeton d’accès obtenu via un flux [OAuth](/fr/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) « à 3 acteurs » ([3-legged](/fr/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)). Votre application devra implémenter un flux OAuth web pour [obtenir des jetons d’accès](/fr/resources/fundamentals/authentication#oauth-1-0a-2). Les développeurs de l’API publicitaire X ne doivent jamais demander à nos annonceurs X de partager leurs identifiants de connexion.

Par défaut, chaque [application développeur X](/fr/resources/fundamentals/developer-apps) contient un **jeton d’accès** statique pouvant être utilisé pour effectuer des requêtes vers l’API publicitaire pour le compte propriétaire de l’application. Ces identifiants sont idéaux pour des cas d’usage mono-compte, sans nécessiter un flux OAuth « à 3 acteurs » ni [basé sur un code PIN](/fr/resources/fundamentals/authentication#pin-based-authorization). Si vous n’accédez pas à un autre compte publicitaire X, utilisez ces identifiants [monoutilisateur](/fr/resources/fundamentals/authentication#oauth-1-0a-2) plutôt que les étapes suivantes.

<div id="levels-of-access">
  ## Niveaux d’accès
</div>

<div id="app-level-permissions">
  ### Autorisations au niveau de l’application
</div>

Chaque utilisateur disposera d’un niveau d’accès, défini lors de la [demande d’accès à l’API publicitaire](/fr/x-ads-api/getting-started/step-by-step-guide) :

- **Conversions uniquement** : l’utilisateur a accès aux endpoints [Mobile](/fr/x-ads-api/measurement/mobile-conversions) et [Web Conversion](/fr/x-ads-api/measurement/web-conversions) avec des droits [lecture et écriture](/fr/resources/fundamentals/developer-apps).
- **Accès standard** : l’utilisateur a accès aux endpoints [Analytics](/fr/x-ads-api/analytics), [Campaign Management](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/overview), [Creatives](https://developer.x.com/en/docs/twitter-ads-api/creatives/overview), [Custom Audiences](https://developer.x.com/en/docs/twitter-ads-api/audiences/overview) et [Conversion](/fr/x-ads-api/measurement/web-conversions) avec des droits [lecture et écriture](/fr/resources/fundamentals/developer-apps).

**Remarque :** les développeurs de l’API publicitaire qui ont demandé l’accès avant juillet 2023 peuvent avoir des niveaux d’accès et des autorisations différents, et peuvent être limités à cinq jetons OAuth. Consultez notre guide sur [l’augmentation de l’accès](/fr/x-ads-api/getting-started/increasing-access) pour obtenir l’accès à des endpoints supplémentaires ou lever les limites de jetons pour les applications existantes.

<div id="ad-account-level-permissions">
  ### Autorisations au niveau du compte publicitaire
</div>

Chaque utilisateur ayant accès à un compte publicitaire dispose d’une autorisation spécifique au niveau du compte : **Administrateur de compte**, **Gestionnaire de publicités**, **Analyste de campagne**, **Analyste organique** et **Gestionnaire de créations**. Consultez **business.x.com** pour la documentation la plus récente sur les autorisations au niveau du compte. Les applications doivent récupérer les autorisations de l’utilisateur actuellement authentifié via le point de terminaison de l’API [Authenticated User Access](/fr/x-ads-api/campaign-management#authenticated-user-access) afin de déterminer à quels points de terminaison de l’API et à quelles fonctionnalités publicitaires elles peuvent accéder.

**Remarque :** Tous les jetons utilisateur utilisés avec la [Conversion API](/fr/x-ads-api/measurement/web-conversions) doivent appartenir à des utilisateurs disposant des autorisations au niveau du compte **Administrateur de compte** ou **Gestionnaire de publicités**.                                                                                                                                              

<div id="methods-of-obtaining-access-tokens">
  ## Méthodes pour obtenir des jetons d’accès
</div>

<div id="1-obtain-an-advertisers-user-access-token">
  ### 1. Obtenir le jeton d’accès d’un annonceur (Utilisateur)
</div>

Il existe deux méthodes pour obtenir un **jeton d’accès** d’annonceur. La plus courante consiste à utiliser un [flux OAuth à 3 volets](/fr/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) directement depuis votre interface Web. Les applications qui n’ont pas d’interface publique accessible aux annonceurs peuvent implémenter un processus OAuth [par code PIN](/fr/resources/fundamentals/authentication#pin-based-authorization). Une fois que l’utilisateur a terminé le flux à 3 volets, votre application disposera des identifiants nécessaires pour effectuer des requêtes sur son compte publicitaire via l’API.

L’obtention d’identifiants utilisateur via un flux OAuth est la méthode que nous **recommandons vivement** à la majorité des développeurs de l’API publicitaire X pour accéder à un compte annonceur. Elle vous permet d’appeler l’API au nom d’un utilisateur et d’agir en son nom. Ces jetons n’expirent pas, mais peuvent être révoqués par l’utilisateur à tout moment.

<div id="2-obtain-your-developer-access-token">
  ### 2. Obtenez votre jeton d’accès (Développeur)
</div>

Cette option nécessite que l’annonceur [accorde à votre @username](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html) (ou à vos @usernames) l’accès à son compte API publicitaire X via l’interface X sur business.x.com. Les **jetons d’accès** obtenus via le flux OAuth à 3 étapes pour votre compte permettront d’accéder au compte API publicitaire X de l’annonceur.

Cela vous permet d’appeler l’API en utilisant les jetons OAuth de votre propre @username plutôt que ceux de l’annonceur. La principale distinction pour cette option est que vous ne pouvez créer des Posts « Promoted-Only » que si l’autorisation de délégation/de composition de Post a été accordée à votre @username.

Pour obtenir l’accès à la création de Posts « Promoted-Only » au nom de l’utilisateur promouvable `FULL` sur le compte, vous devez également avoir l’autorisation de créer des Posts dans ce flux. Cela activera l’accès via l’autorisation `TWEET_COMPOSER` sur le point de terminaison [GET accounts/:account\_id/authenticated\_user\_access](/fr/x-ads-api/campaign-management#accounts).

<div id="differences-between-these-methods">
  ## Différences entre ces méthodes
</div>

|     |     |     |
| :--- | :--- | :--- |
|     | Jeton OAuth d’annonceur (utilisateur) | Jeton OAuth de développeur  <br /> (@username ajouté pour distinguer le compte) |
| Accéder au compte publicitaire | ✔   | ✔   |
| Créer des posts au nom de l’utilisateur | ✔   | ✔\*  |
| Gérer les campagnes | ✔   | ✔   |
| Accéder aux analyses | ✔   | ✔   |
| Créer des Cards au nom de l’utilisateur | ✔   | ✔   |
| Capacité du développeur à accéder  <br />via l’interface X Ads |     | ✔   |
| [Limites de débit](/fr/x-api/enterprise-gnip-2.0/fundamentals/rate-limits) | Distinctes par annonceur | Distinctes par compte annonceur |
