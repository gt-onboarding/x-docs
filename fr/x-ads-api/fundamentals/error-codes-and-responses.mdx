<div id="typical-response-structure">
  ## Structure de réponse typique
</div>

Les réponses réussies sont indiquées par un code HTTP de la série 200 et une charge utile JSON contenant le ou les objets demandés, créés, modifiés ou supprimés, ainsi qu’une indication de la manière dont le serveur a interprété votre requête.

Si vous envoyez une requête qui réussit, votre réponse comprendra un nœud `request` reprenant votre requête.

_Exemple:_ [GET accounts/abcdefg/campaigns?with\_deleted=true](/fr/x-ads-api/campaign-management#campaigns)

```json
{
  /* the data of your response... */,
  "request": {
    "params": {
      "account_id": "abcdefg",
      "with_deleted": "true"
    }
  }
}
```

Le champ `data` dans les réponses JSON contient les objets spécifiques associés à la ressource utilisée. Le nœud `data` est un tableau JSON lorsque la réponse peut comporter un ou plusieurs résultats. Il est renvoyé sous forme d’objet JSON lorsqu’un seul résultat est possible. Dans de rares cas, vous pouvez voir une réponse qui inclurait normalement une collection remplacée par un objet (hashmap). Dans ce cas, considérez que l’objet unique est du même type que celui indiqué dans le champ `type`.

<div id="error-response-structure">
  ## Structure de la réponse d’erreur
</div>

Les réponses d’erreur sont renvoyées avec un code HTTP hors de la série 200. En général, une réponse JSON est jointe, mais certaines erreurs renvoient un corps d’un autre type. Dans les situations où la structure de la réponse ne peut pas être analysée, considérez que la signification fondamentale du code HTTP prévaut. Par exemple, vous pouvez occasionnellement voir un HTTP 404 accompagné d’une réponse HTML. Dans ce cas, il est raisonnable de supposer que le contenu est introuvable (HTTP 404 signifie « Not Found »).

Les réponses d’erreur typiques suivent une structure similaire à celle des réponses réussies. La nature de l’erreur est communiquée dans un nœud `errors` de la réponse. Le nœud `errors/code` indiquera un code d’erreur constant en MAJUSCULES que vous pouvez traiter de manière programmatique pour prendre des décisions de résolution. Le nœud `errors/message` indiquera une description (généralement) lisible par un humain de l’erreur en anglais. Des champs supplémentaires peuvent être ajoutés pour fournir des détails plus précis sur l’erreur.

```json
{
  "errors": [
    {
      "parameter": "start_time",
      "details": "invalid date",
      "code": "INVALID_PARAMETER",
      "value": "",
      "message": "Expected time, got \"\" for start_time"
    }
  ],
  "request": {
    "params": {
      "account_id": "hkk5"
    }
  }
}
```

Dans l’exemple ci-dessus, une requête vers un endpoint d’analytique a été effectuée avec une valeur non valide pour le paramètre `start_time`. Le `errors/code` pour les requêtes comportant des paramètres non valides est `INVALID_PARAMETER`.
