<div id="bearer-token-also-known-as-app-only">
  ### Jeton Bearer (également appelé app-only)
</div>

Le jeton Bearer OAuth 2.0 authentifie les requêtes au nom de votre [application développeur](/fr/resources/fundamentals/developer-apps). Comme cette méthode est spécifique à l’application, elle n’implique aucun utilisateur. Elle est généralement destinée aux développeurs qui ont besoin d’un accès en lecture seule aux informations publiques.

Cette méthode d’authentification requiert d’inclure un jeton Bearer dans votre requête, que vous pouvez générer dans la section « Keys and tokens » de vos applications développeur. Voici un exemple de requête avec un jeton Bearer factice :

```json
  curl "https://api.x.com/2/tweets?ids=1261326399320715264,1278347468690915330" \
  -H "Authorization: Bearer AAAAAAAAAAAAAAAAAAAAAFnz2wAAAAAAxTmQbp%2BIHDtAhTBbyNJon%2BA72K4%3DeIaigY0QBrv6Rp8KZQQLOTpo9ubw5Jt?WRE8avbi"
```

Les appels à l’API utilisant l’authentification app-only sont soumis à des [limites de débit](/fr/resources/fundamentals/rate-limits) par endpoint, au niveau de l’application.

Pour utiliser cette méthode, vous aurez besoin d’un jeton Bearer, que vous pouvez générer en transmettant votre clé API et votre secret via l’endpoint [POST oauth2/token](/fr/resources/fundamentals/authentication/api-reference#post-oauth2-token), ou en le générant dans la section « keys and tokens » des paramètres de votre application dans le [portail développeur](/fr/resources/fundamentals/developer-portal).

Si vous souhaitez révoquer un jeton Bearer, vous pouvez utiliser l’endpoint [POST oauth2/invalidate\_token](/fr/resources/fundamentals/authentication/api-reference#post-oauth2-invalidate-token), ou cliquer sur « revoke » à côté du jeton Bearer dans la section « keys and tokens » des paramètres de votre application.

<div id="oauth-20-authorization-code-flow-with-pkce">
  ### Flux de code d’autorisation OAuth 2.0 avec PKCE
</div>

Le flux de code d’autorisation OAuth 2.0 avec PKCE vous permet d’authentifier au nom d’un autre utilisateur tout en offrant un meilleur contrôle sur les portées (scopes) d’une application et en améliorant les parcours d’autorisation sur plusieurs appareils. En d’autres termes, les développeurs qui créent des applications pour les utilisateurs de X auront davantage de contrôle sur les informations que leur application demande à ses utilisateurs, de sorte que vous ne sollicitiez de vos utilisateurs finaux que les données et informations nécessaires.

Ce protocole d’autorisation moderne vous permet de proposer à vos utilisateurs finaux un parcours de consentement plus fluide pour autoriser votre application, n’affichant que les portées (scopes) spécifiques que vous avez demandées. Non seulement cela réduit votre charge de données, mais cela peut également accroître la confiance des utilisateurs finaux.
