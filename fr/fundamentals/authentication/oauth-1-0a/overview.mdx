<div id="oauth-10a">
  ## OAuth 1.0a
</div>

De nombreux endpoints sur la plateforme développeur X utilisent la méthode OAuth 1.0a pour agir, ou effectuer des requêtes API, au nom d’un compte X. Par exemple, si vous disposez d’une application développeur X, vous pouvez effectuer des requêtes API au nom de n’importe quel compte X, à condition que cet utilisateur ait authentifié votre application.

Remarque : si vous n’êtes pas familiarisé avec des notions telles que HMAC-SHA1 et l’encodage pourcentuel, nous vous recommandons de consulter la section « outils utiles » ci-dessous, qui répertorie des clients API simplifiant grandement le processus d’authentification.

<div id="key-concepts">
  ### Concepts clés
</div>

<div id="signing-a-request-with-keys-and-tokens">
  #### Signature d’une requête avec des clés et des jetons
</div>

Vous devez signer chaque requête à l’API en transmettant plusieurs clés et jetons générés dans un en-tête d’autorisation. Pour commencer, vous pouvez générer plusieurs clés et jetons sur la page de détails de votre [application développeur X](/fr/resources/fundamentals/developer-apps), notamment les éléments suivants :

|     |     |
| :--- | :--- |
| Clé d’API et secret :<br /><br />`oauth_consumer_key`<br /><br />`oauth_consumer_secret` | Considérez-les comme le nom d’utilisateur et le mot de passe qui représentent votre application développeur X lors de l’envoi de requêtes à l’API. |
| Jeton d’accès et secret :<br /><br />`oauth_token`<br /><br />`oauth_token_secret` | Un jeton d’accès et un secret de jeton d’accès sont des identifiants propres à l’utilisateur, utilisés pour authentifier les requêtes d’API OAuth 1.0a. Ils indiquent le compte X pour le compte duquel la requête est effectuée.<br /><br />Vous pouvez générer votre propre jeton d’accès et son secret si vous souhaitez que votre application effectue des requêtes au nom du même compte X associé à votre compte développeur sur la page de détails de l’application développeur X.<br /><br />Si vous souhaitez générer des jetons d’accès pour un autre utilisateur, consultez « Effectuer des requêtes au nom des utilisateurs » ci-dessous. |

<div id="making-requests-on-behalf-of-users">
  #### Effectuer des requêtes au nom d’utilisateurs
</div>

Lors de la création d’une signature, vous avez besoin d’un ensemble de jetons d’accès représentant l’utilisateur au nom duquel vous allez effectuer une requête.

Vous pouvez générer un ensemble de jetons d’accès représentant le compte X propriétaire de l’application développeur X depuis la [page des détails de l’app](https://developer.x.com/content/developer-twitter/en/apps). En revanche, si vous souhaitez effectuer une requête au nom d’un autre compte X, le propriétaire de ce compte doit vous accorder l’accès en se connectant à son compte dans le cadre du [flux OAuth à 3 étapes](/fr/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow). Le résultat de ce processus est un ensemble de jetons d’accès (oauth\_token et oauth\_token\_secret) qui peuvent être utilisés pour effectuer une requête OAuth 1.0a.

Une fois que vous disposez de ces clés et jetons, vous pouvez soit [créer une signature](/fr/resources/fundamentals/authentication#creating-a-signature) à partir de zéro — ce que nous ne recommandons pas, sauf si vous savez exactement ce que vous faites — soit utiliser l’un des outils mentionnés ci-dessous pour envoyer une requête à un endpoint qui nécessite OAuth 1.0a.

Pour référence, voici un exemple de requête cURL signée avec tous les jetons générés, transmis dans un en-tête d’autorisation :

```bash
curl --request POST \
  --url 'https://api.x.com/1.1/statuses/update.json?status=Hello%20world' \
  --header 'authorization: OAuth oauth_consumer_key="CONSUMER_API_KEY", oauth_nonce="OAUTH_NONCE", oauth_signature="OAUTH_SIGNATURE", oauth_signature_method="HMAC-SHA1", oauth_timestamp="OAUTH_TIMESTAMP", oauth_token="ACCESS_TOKEN", oauth_version="1.0"' \
```

<Note>
  **Remarque :** Les jetons d’accès utilisateur sont sensibles et doivent être protégés avec le plus grand soin. Lorsque des jetons d’accès sont générés, l’utilisateur qu’ils représentent fait confiance à votre application pour les garder en sécurité. Si la sécurité des clés API et des jetons d’accès utilisateur est compromise, votre application pourrait potentiellement exposer des informations privées et des fonctionnalités de compte. Nous vous encourageons à en savoir plus sur la [sécurisation des clés et des jetons d’accès](/fr/resources/fundamentals/developer-apps).
</Note>

<div id="useful-tools">
  ### Outils utiles
</div>

Le processus de [signature d’une requête](/fr/resources/fundamentals/authentication#creating-a-signature) est complexe. Nous recommandons d’utiliser une bibliothèque cliente pour API qui génère automatiquement une grande partie du jeton d’authentification :

|     |     |
| :--- | :--- |
| [Postman](https://www.getpostman.com/) | Un client API qui vous permet de créer et d’envoyer des requêtes REST. Lisez notre tutoriel « [Prise en main de Postman](/fr/tutorials/postman-getting-started) » pour en savoir plus sur cet outil. |
| [Insomnia](https://insomnia.rest/) | Insomnia est un client API REST avec gestion des cookies, variables d’environnement, génération de code et authentification, disponible pour Mac, Windows et Linux. |
