title: "Gestion des versions"
---

<p>Avec l’API X v2, nous avons lancé une nouvelle stratégie de gestion des versions qui permet aux développeurs de mieux anticiper les évolutions des API publiques de X et de savoir quand ils devront migrer vers de nouvelles versions. </p>
<p>Les développeurs seront informés des dépréciations, retraits, modifications et ajouts à l’API X via nos <a href="https://developer.x.com/en/updates/stay-informed">canaux de communication</a>, afin qu’ils puissent planifier en conséquence l’intégration de ces changements dans leur feuille de route de développement. Toutes les modifications de l’API seront consignées dans le journal des changements.</p>
<p>L’API X comporte actuellement trois versions. Nous recommandons vivement d’utiliser l’API X v2 lors de la planification de votre intégration, sauf si une fonctionnalité nécessaire à votre cas d’usage n’a pas encore été publiée en v2. </p>
<p>Pour en savoir plus sur chaque version, veuillez consulter les pages suivantes :</p>

<ul>
  <li><a href="/fr/x-api/introduction">API X v2</a></li>
  <li>Gnip 2.0 (<a href="/fr/x-api/enterprise-gnip-2.0/enterprise-gnip">entreprise</a>)</li>
</ul>

<div id="versioning-strategy">
  ## Stratégie de versionnement
</div>

<p>Le versionnement de l’API X est représenté par des numéros de version déclarés dans le chemin de nos routes d’endpoints :</p>
<p>`https://api.x.com/2/tweets`</p>
<p>Nous prévoyons de publier des versions majeures de l’API X selon les besoins, mais pas plus d’une fois tous les 12 mois. Une version majeure sera publiée lorsque des changements incompatibles seront introduits dans l’API. Nous publierons des guides de migration lors du lancement d’une nouvelle version majeure pour aider les développeurs à passer à la nouvelle version. </p>
<p>Un changement incompatible oblige les développeurs à modifier leur code afin de conserver les fonctionnalités existantes de leur application. Les changements non incompatibles seront additifs et déployés sur la version la plus récente lorsqu’ils seront prêts, sans nécessiter de travail de la part du développeur sauf si vous souhaitez tirer parti de la nouvelle fonctionnalité.</p>
<p>Si un changement incompatible doit être déployé en milieu de cycle (pour des raisons de sécurité ou de confidentialité), il sera appliqué à la version la plus récente.</p>

<div id="breaking-changes">
  ### Modifications incompatibles
</div>

<p>Ces changements obligent les développeurs à modifier leur code pour préserver les fonctionnalités existantes de leur application.</p>

<ul>
  <li>Ajout d’un nouveau paramètre requis</li>
  <li>Suppression d’un endpoint existant</li>
  <li>Suppression de tout champ dans la réponse (qu’il soit requis ou facultatif)</li>
  <li>Suppression d’un paramètre de requête</li>
  <li>Restructuration du format d’entrée ou de sortie (par exemple, transformer un champ de niveau supérieur en sous-champ, ou rendre les erreurs affichées en ligne)</li>
  <li>Modification du nom ou du type de données d’un paramètre d’entrée existant ou d’une valeur de sortie</li>
  <li>Modification du nom d’un champ</li>
  <li>Modification du nom de la ressource</li>
  <li>Modification d’un code de réponse</li>
  <li>Modification des types d’erreur</li>
  <li>Modifications des portées d’autorisation existantes</li>
</ul>

<div id="non-breaking-changes">
  ### Changements non disruptifs
</div>

<ul>
  <li>Ajout d’un nouvel endpoint</li>
  <li>Ajout d’un nouveau paramètre optionnel</li>
  <li>Ajout d’un nouveau champ dans la réponse</li>
  <li>Modification du texte des messages d’erreur</li>
  <li>Disponibilité de nouvelles étendues (scopes)</li>
  <li>« Mise à null » de champs (remplacement de la valeur d’un champ par null pour des raisons de confidentialité/sécurité, en alternative à sa suppression complète)</li>
</ul>

<div id="deprecation-and-retirement">
  ### Obsolescence et retrait
</div>

<p>Tout d’abord, voici notre définition de ce que « obsolescence » et « retrait » signifient pour l’API X :</p>

<ul>
  <li><b>Obsolescence :</b> La fonctionnalité n’est plus prise en charge par l’équipe. Aucune nouvelle évolution ne sera apportée à cette fonctionnalité et, en cas d’anomalies ou de problèmes avec le produit, la probabilité que nous proposions un correctif est extrêmement faible. </li>
  <li><b>Retrait :</b> La fonctionnalité ne sera plus accessible.</li>
</ul>

<p>Dans la plupart des cas, dès qu’une nouvelle version est publiée, la version précédente est marquée comme obsolète. Les versions restent dans cet état pendant un certain temps, après quoi elles sont retirées. </p>
<p>Veuillez <a href="https://developer.x.com/en/updates/stay-informed">rester informé</a> pour en savoir plus sur les futures obsolescences et retraits.</p>
