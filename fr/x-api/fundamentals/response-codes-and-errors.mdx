L’API X fournit les codes de réponse et d’erreur suivants pour faciliter la compréhension et le dépannage en temps réel. Utilisez le guide de débogage et l’index des erreurs ci-dessous pour obtenir des informations supplémentaires.

Les réponses réussies sont signalées par un code HTTP de la série 200 et un payload au format JSON contenant le ou les objets demandés, créés, modifiés ou supprimés, ainsi qu’une indication de la manière dont le serveur a interprété votre requête.

Les réponses d’erreur sont renvoyées avec un code HTTP hors de la série 200. Des codes d’erreur différents correspondent à des causes d’erreur différentes. L’API X s’efforce de renvoyer des [codes d’état HTTP](https://en.wikipedia.org/wiki/List_of_HTTP_status_codes) appropriés pour chaque requête.

<div id="x-api-v2-http-status-codes">
  ## Codes d’état HTTP de l’API X v2
</div>

| Code | Texte | Description | Conseils de dépannage |
| :--- | :--- | :---------- | :------------------- |
| 200 | OK | La requête a abouti. | |
| 400 | Requête incorrecte | La requête est invalide ou ne peut pas être traitée. Un message d’erreur accompagnant apportera des précisions. Les requêtes sans authentification ou avec des paramètres de requête invalides sont considérées comme invalides et renverront cette réponse. | Vérifiez le format de votre requête JSON. Par exemple, si votre règle contient des guillemets doubles utilisés avec un opérateur de correspondance exacte ou autre, vous devrez peut‑être les échapper avec une barre oblique inversée pour les distinguer de la structure du JSON. |
| 401 | Non autorisé | Un problème est survenu lors de l’authentification de votre requête, dû à des informations d’identification manquantes ou incorrectes. Cela peut aussi être renvoyé dans d’autres circonstances non définies. | Vérifiez que vous utilisez la bonne [méthode d’authentification](https://docs.x.com/resources/fundamentals/authentication/overview) et que vos identifiants sont corrects. |
| 403 | Interdit | La requête est comprise, mais elle a été refusée ou l’accès n’est pas autorisé. Un message d’erreur accompagnant expliquera pourquoi. | Vérifiez que votre compte développeur inclut l’accès au point de terminaison que vous essayez d’utiliser. Il peut être nécessaire de faire placer votre App sur liste d’autorisation (p. ex., Engagement API ou Ads API) ou de demander l’accès. |
| 404 | Introuvable | L’URI demandée est invalide ou la ressource demandée (p. ex., un utilisateur) n’existe pas. | Vérifiez que vous utilisez des paramètres valides et l’URI correcte pour le point de terminaison concerné. |
| 409 | Exception de connexion | Renvoyé lors d’une tentative de connexion à un flux filtré sans aucune règle. | Vérifiez que vous avez créé au moins une règle sur le flux auquel vous vous connectez. Un flux filtré ne renvoie que les Posts qui correspondent à une règle active. S’il n’y a aucune règle, le flux ne renverra aucun Post. |
| 429 | Trop de requêtes | Renvoyé lorsqu’une requête ne peut pas être servie parce que le [plafond de débit](/fr/resources/fundamentals/rate-limits) de l’App ou le [plafond de posts](/fr/x-api/fundamentals/post-cap) a été atteint. Voir [Limitation de débit](/fr/resources/fundamentals/rate-limits). | Vérifiez le nombre de requêtes autorisées par fenêtre temporelle pour le point de terminaison utilisé. Attendez la réinitialisation de la fenêtre. Espacez vos requêtes pour éviter d’atteindre les limites de débit ou passez au forfait de données supérieur. |
| 500 | Erreur interne du serveur | Un dysfonctionnement s’est produit. Il s’agit généralement d’une erreur temporaire, par exemple en cas de forte charge ou si un point de terminaison rencontre temporairement des problèmes. | Consultez la [page d’état](/fr/status) de l’API X ou le [forum de la communauté des développeurs](https://devcommunity.x.com/) pour voir si d’autres rencontrent des problèmes similaires, ou patientez et réessayez plus tard. |
| 501 | Non implémenté | L’API X ne prend pas en charge ce point de terminaison et ne peut pas satisfaire la requête. |  |
| 503 | Service indisponible | Les serveurs X sont opérationnels, mais surchargés de requêtes. Réessayez plus tard. | Consultez la [page d’état](/fr/status) de l’API X ou le [forum de la communauté des développeurs](https://devcommunity.x.com/) pour voir si d’autres rencontrent des problèmes similaires, ou patientez et réessayez plus tard. |

Lorsqu’une erreur survient pendant une requête, des informations détaillées sur l’erreur sont renvoyées dans le corps de la réponse pour aider à diagnostiquer le problème. Un champ type, qui est un URI, indique la nature du problème, tandis que des champs supplémentaires apportent des détails. Les champs type, title et detail sont toujours renvoyés dans ces corps (voir le tableau ci‑dessous). Tout champ supplémentaire, comme dans l’exemple ci‑dessous, variera selon le type d’erreur.

```json
{
  "client_id": "101010101",
  "required_enrollment": "Standard Basic",
  "registration_url": "https://developer.twitter.com/en/account",
  "title": "Client Forbidden",
  "detail": "This request must be made using an approved developer account that is enrolled in the requested endpoint. Learn more by visiting our documentation.",
  "reason": "client-not-enrolled",
  "type": "https://api.x.com/2/problems/client-forbidden"
}
```

<div id="partial-errors">
  ## Erreurs partielles
</div>

Dans certains cas, vous pouvez voir les erreurs décrites ci-dessus dans une réponse ayant renvoyé un code d’état 200. Dans ces situations, l’endpoint est conçu pour renvoyer les données disponibles, tout en fournissant des erreurs détaillées concernant ce qui n’a pas pu être renvoyé.

Par exemple, l’[endpoint de recherche de posts](/fr/x-api/posts/lookup/introduction) permet à une [application développeur](/fr/resources/fundamentals/developer-apps#overview) X de demander plusieurs identifiants. Si certains de ces posts sont disponibles mais que l’un d’eux a été supprimé, les posts disponibles seront renvoyés dans le champ data de la réponse. Un champ errors supplémentaire sera inclus dans la charge utile, indiquant quel(s) post(s) demandé(s) n’ont pas pu être renvoyé(s). Le même format que pour les erreurs de requête complètes est utilisé afin de faciliter le diagnostic des problèmes.

<div id="error-information">
  ### Informations sur les erreurs
</div>

Chaque type de problème indique la nature de l’erreur rencontrée. Une liste complète des problèmes susceptibles de survenir figure également dans notre [spécification de l’API](https://api.x.com/2/openapi.json).

L’API X s’efforce de renvoyer des [codes d’état HTTP](https://en.wikipedia.org/wiki/List_of_HTTP_status_codes) appropriés pour chaque requête.

| Titre | Type | Description |
| :--- | :--- | :--- |
| Problème générique | `about:blank` | Problème générique sans information supplémentaire au-delà de celle fournie par le code d’état HTTP. |
| Problème de requête invalide | `https://api.X.com/2/problems/invalid-request` | Problème indiquant que cette requête est invalide. Si votre requête inclut un corps POST, assurez-vous que le contenu est un JSON valide et qu’il correspond à la [spécification OpenAPI](https://api.x.com/2/openapi.json). |
| Problème de ressource introuvable | `https://api.X.com/2/problems/resource-not-found` | Problème indiquant qu’un Post, un utilisateur, etc. donné n’existe pas. |
| Problème d’autorisation sur la ressource | `https://api.X.com/2/problems/not-authorized-for-resource` | Problème indiquant que vous n’êtes pas autorisé à consulter un Post, un utilisateur, etc. particulier. |
| Problème client interdit | `https://api.X.com/2/problems/client-forbidden` | Problème indiquant que votre client n’est pas autorisé à effectuer cette requête. |
| Problème de ressource non autorisée | `https://api.X.com/2/problems/disallowed-resource` | Problème indiquant que la ressource demandée enfreint les principes de cette API. |
| Problème d’authentification non prise en charge | `https://api.X.com/2/problems/unsupported-authentication` | Problème indiquant que l’authentification utilisée n’est pas prise en charge. |
| Problème de dépassement de plafond d’utilisation | `https://api.X.com/2/problems/usage-capped` | Problème indiquant qu’un plafond d’utilisation a été dépassé. |
| Problème de connexion | `https://api.X.com/2/problems/streaming-connection` | Problème indiquant un dysfonctionnement de la connexion. |
| Problème de client déconnecté | `https://api.X.com/2/problems/client-disconnected` | Votre client s’est déconnecté. |
| Problème de déconnexion opérationnelle | `https://api.X.com/2/problems/operational-disconnect` | Vous avez été déconnecté pour des raisons opérationnelles. |
| Problème de plafond de règles | `https://api.X.com/2/problems/rule-cap` | Vous avez dépassé le nombre maximal de règles. |
| Problème de longueur de règle | `https://api.X.com/2/problems/rule-length` | Vous avez dépassé le nombre maximal de caractères autorisés pour votre requête ou règle, selon votre niveau d’accès. Voir les [niveaux d’accès](/fr/x-api/getting-started/about-x-api#x-api-access-levels-and-versions). |
| Problème de règle invalide | `https://api.X.com/2/problems/invalid-rules` | La règle que vous avez soumise est invalide. |
| Problème de règle en double | `https://api.X.com/2/problems/duplicate-rules` | La règle que vous avez soumise est un doublon. |

<div id="troubleshooting-tips">
  ## Conseils de dépannage
</div>

<Accordion title="Déboguer votre code">
  Lors de la création d’une application, il est normal de rencontrer des erreurs ou des problèmes inattendus. Vous trouverez ci-dessous quelques conseils pour déboguer votre code :

  Commencez par décomposer le problème en parties plus petites afin d’identifier où il se situe. Par exemple, si vous intégrez un endpoint REST ou de streaming, le problème peut provenir de l’un des éléments suivants :

  - Le endpoint nécessite une [authentification](/fr/resources/fundamentals/authentication/overview) appropriée.
  - Le endpoint nécessite des paramètres et des en-têtes valides. Toutes les règles de filtrage sont construites avec les bons opérateurs et la [syntaxe](/fr/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#building-rules-and-queries) adéquate.
  - L’URI du endpoint doit être correcte et, dans le cas des endpoints REST, la bonne méthode HTTP doit être utilisée.
  - Les données ou la ressource auxquelles vous tentez d’accéder ne vous sont pas accessibles (par exemple, les données privées ne sont disponibles que pour les utilisateurs authentifiés).
  - Votre offre de données actuelle vous donne accès uniquement à certains endpoints et à des limites de débit spécifiques. Consultez votre [portail développeur](https://developer.x.com/en/portal/products) pour plus de détails.
  - Votre code utilise une bibliothèque tierce pour intégrer le endpoint dans votre code.
  - Votre code doit analyser correctement la réponse du endpoint.

  Lisez le message d’erreur associé. Il devrait vous donner une bonne indication de la nature du problème. Utilisez les tableaux de la [section des codes d’erreur et de réponse](#x-api-http-status-codes) pour des conseils de dépannage spécifiques à chaque code.

  Pour les endpoints REST, vous pouvez utiliser un client REST comme [Postman](https://www.postman.com/) ou [Insomnia](https://insomnia.rest/) pour valider les étapes (1) à (5) ci-dessus (consultez notre guide « [Prise en main avec Postman](/fr/tutorials/postman-getting-started) »). Si la requête avec le client REST renvoie un code d’état 200 (succès), vous pouvez supposer que le problème provient de votre code ou de la bibliothèque que vous utilisez, et non de la requête au endpoint elle-même.

  Pour les endpoints de streaming, vous pouvez utiliser cURL (un outil en ligne de commande pour envoyer ou récupérer des requêtes en utilisant la syntaxe d’URL) afin de vérifier si le problème provient de la requête vers le endpoint (étapes (1) à (5) ci-dessus) ou de votre code lui-même (étapes (6) à (7) ci-dessus).
</Accordion>

<Accordion title="Problèmes d’authentification">
  - Vérifiez que vous utilisez la [méthode d’authentification](/fr/resources/fundamentals/authentication/overview) requise pour le endpoint. Cela peut être identifié via la page de référence de l’API du endpoint.
  - Vérifiez que vos identifiants d’authentification sont corrects. Vous pouvez vérifier ou régénérer les clés et jetons de votre App dans la [section Apps](https://developer.x.com/en/portal/projects-and-apps) de votre tableau de bord développeur (sous « Détails »).
  - Vérifiez que vous avez correctement [autorisé votre requête OAuth 1.0a](/fr/resources/fundamentals/authentication/oauth-1-0a/authorizing-a-request) avec `oauth_nonce`, `oauth_signature` et `oauth_timestamp` pour votre requête.
  - Si le problème persiste, envisagez d’utiliser une bibliothèque OAuth, un client REST comme [Postman](https://www.postman.com/) ou [Insomnia](https://insomnia.rest/), ou [xurl](https://github.com/xdevplatform/xurl).

  Consultez notre [guide sur l’authentification](/fr/resources/fundamentals/authentication/overview) pour plus d’informations sur l’ensemble des points ci-dessus.
</Accordion>

<Accordion title="Problèmes de limitation de débit et de plafonds de posts">
  Une erreur 429 peut se produire si vous atteignez une limite de débit pour un endpoint donné ou si vous atteignez un plafond de posts.

  Si l’erreur spécifique « Too many requests » est renvoyée, vous avez atteint la limite de débit de l’endpoint. En d’autres termes, vous avez dépassé le nombre maximal de requêtes autorisées pour un endpoint sur une période donnée.

  Notez que les limites de débit sont définies au niveau de l’application et au niveau de l’utilisateur.

  - **Niveau X App** indique le nombre de requêtes autorisées lors de l’utilisation d’OAuth 2.0 App-Only, où les limites de débit sont déterminées globalement pour l’ensemble de l’application. Par exemple, si une méthode autorise 15 requêtes par fenêtre de limitation, elle vous permet d’effectuer 15 requêtes par fenêtre pour le compte de votre X App. Cette limite est entièrement indépendante de la limite au niveau utilisateur. En savoir plus dans notre guide sur [OAuth 2.0 App-Only](/fr/resources/fundamentals/authentication/oauth-2-0/application-only).
  - **Niveau contexte utilisateur** indique le nombre de requêtes pouvant être effectuées par Jeton d’accès utilisateur lors de l’utilisation d’OAuth 1.0a User Context ou d’OAuth 2.0 Authorization Code avec PKCE. Par exemple, si une méthode autorise 15 requêtes par fenêtre de limitation, elle permet 15 requêtes par fenêtre et par Jeton d’accès utilisateur. Lisez notre guide sur la façon d’obtenir les Jetons d’accès d’un utilisateur avec [OAuth 1.0a](/fr/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens) et [OAuth 2.0](/fr/resources/fundamentals/authentication/oauth-2-0/user-access-token).

  Commencez par vérifier les limites de débit de l’endpoint que vous utilisez. Vous trouverez ces informations sur la page de référence de l’API de l’endpoint et dans le nouveau tableau de bord du portail développeur.

  Consultez notre documentation pour davantage d’informations sur les limites de débit, notamment comment utiliser les en-têtes HTTP pour suivre la position de votre application par rapport à une limite donnée, comment se remettre d’un code d’erreur 429 lié à la limitation de débit et des conseils pour éviter d’être limité dès le départ :

  - [Limites de débit pour les endpoints v2](/fr/x-api/fundamentals/rate-limits)

  Si vous avez reçu l’erreur spécifique « Usage cap exceeded: Monthly product cap », cela signifie que vous avez atteint le plafond de posts pour votre niveau d’accès. Nous fournissons de nombreux détails sur ces [plafonds de posts](/fr/x-api/fundamentals/post-cap) dans notre page de documentation.
</Accordion>

<Accordion title="Problèmes de posts manquants">
  Suivez les étapes ci-dessous si vous vous attendiez à ce qu’un post soit renvoyé, mais qu’il n’a pas été délivré par l’endpoint.

  - Vérifiez votre règle pour vous assurer que vous utilisez les bons [opérateurs](/fr/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#enterprise-operators) et la bonne [syntaxe](/fr/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#building-rules-and-queries). Scindez la règle en clauses plus petites pour vérifier que la syntaxe est correcte.
  - Si le compte depuis lequel le post a été envoyé était [protégé](https://help.twitter.com/en/safety-and-security/public-and-protected-tweets) au moment de la création du post, le post ne sera pas renvoyé — même si le compte est public au moment de la requête à l’endpoint. Vous pouvez généralement vérifier cela via la [Recherche avancée X](https://x.com/search-advanced) : si un post n’apparaît pas via la fonctionnalité de Recherche avancée X, vous devez supposer qu’il ne sera pas renvoyé par l’endpoint.

  Les étapes suivantes s’appliquent uniquement aux endpoints de streaming :

  - Étiez-vous connecté au flux lorsque le post a été envoyé ? Rappelez-vous que l’horodatage fourni dans l’objet Post indique l’heure en UTC. Si vous avez subi une déconnexion lorsque le post a été envoyé, consultez les [fonctionnalités de récupération et de redondance](/fr/x-api/posts/filtered-stream/integrate/recovery-and-redundancy-features#recovery-and-redundancy) disponibles pour reconstituer toute donnée manquée.
  - Votre règle était-elle en place lorsque le post a été envoyé ? Rappelez-vous que l’horodatage fourni dans l’objet Post indique l’heure en UTC.
</Accordion>

<Accordion title="Problèmes de déconnexion dus à un tampon plein">
  Vous pouvez rencontrer l’une des erreurs suivantes lorsque votre flux ne suit pas la cadence à laquelle nous livrons les données et que votre application ne consomme pas les données assez rapidement :

  ```
  This stream has been disconnected because your client was unable to keep up with us.
  ```

  ```
  This stream has been disconnected for operational reasons.
  ```

  Nous tolérons un retard de livraison pendant un certain temps et disposons côté serveur d’un tampon intermédiaire temporaire pour chaque flux ; mais si vous ne rattrapez pas, nous procédons à une déconnexion afin de vous permettre de vous reconnecter au point courant. Veuillez noter que cela peut entraîner une perte de données (pour les données présentes dans le tampon au moment de la déconnexion pour tampon plein).

  Ces situations peuvent se produire lors de fortes pointes de volume. En général, nous recommandons d’utiliser un mécanisme de tampon pour consommer rapidement les données, distinct du processus de traitement.

  Si vous êtes un client Entreprise utilisant les endpoints v1.1, vous pouvez en savoir plus sur l’optimisation de votre application pour éviter ce type de déconnexion dans nos articles sur la [connexion](/fr/x-api/enterprise-gnip-2.0/powertrack-api#disconnections-explained) et sur la consommation de données en streaming [ici](https://developer.x.com/en/docs/tutorials/consuming-streaming-data) et [ici](/fr/x-api/enterprise-gnip-2.0/powertrack-api#planning-for-high-volume-social-data-events).

  Il existe une gamme d’outils pour récupérer les posts manqués en raison d’une déconnexion, notamment ceux listés ci-dessous. Notez que les outils suivants ne sont disponibles qu’avec les endpoints v1.1 au niveau d’accès Entreprise.

  - [**Redundant Connections**](/fr/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - Avec plusieurs connexions, consommez le flux depuis plusieurs serveurs pour éviter les données manquées lorsqu’une connexion est interrompue.
  - [**Replay**](/fr/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - Récupérez les données des 5 derniers jours via un flux distinct.
  - [**Backfill**](/fr/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - Reconnectez-vous dans les 5 minutes et reprenez là où vous vous êtes arrêté.
  - **Historical PowerTrack** - Récupérez des données depuis l’ensemble de l’archive X.
</Accordion>

</div>

<div id="when-you-post-a-question-make-sure-to-include-the-following-information">
  ### Lorsque vous publiez une question, veillez à inclure les informations suivantes
</div>

- Une description du problème
- L’appel d’API effectué (incluez les en-têtes, si possible)
- La réponse renvoyée par X (incluez tout message d’erreur)
- Ce que vous vous attendiez à recevoir à la place
- La liste des étapes effectuées pour dépanner le problème
- La liste des étapes nécessaires pour reproduire le problème
- Le cas échéant, la période pendant laquelle le problème est survenu
- Le cas échéant, l’ID d’app, l’ID de post, etc.
- Tout extrait de code ou toute capture d’écran pertinent(e)

Veuillez n’inclure qu’un seul sujet/une seule question par post.

Si vous avez des demandes de fonctionnalités ou des commentaires, veuillez les soumettre via le [formulaire de retour d’expérience de la plateforme développeurs X](https://t.co/devfeedback).

Pour les problèmes liés aux règles, comme la suspension d’une application, veuillez contacter le [support des règles](https://help.twitter.com/forms/platform).

Pour les problèmes liés à X, comme la connexion et l’assistance au compte, veuillez utiliser le [Centre d’assistance X](https://help.twitter.com/en/contact-us).
