<div id="tweet">
  ### Tweet
</div>

Les Tweets sont l’élément de base de tout l’écosystème Twitter. L’objet Tweet comporte une longue liste de champs « racine », tels que `id`, `text` et `created_at`. Les objets Tweet sont également l’objet « parent » de plusieurs objets enfants, notamment `user`, `media`, `poll` et `place`. Utilisez le paramètre de champ `tweet.fields` pour demander ces champs racine sur l’objet Tweet.

L’objet Tweet peut être consulté et étendu dans la ressource utilisateur. Des Tweets supplémentaires liés au Tweet demandé peuvent également être consultés et étendus dans la ressource Tweet. L’objet est disponible via l’extension `?expansions=pinned_tweet_id` dans la ressource utilisateur ou `?expansions=referenced_tweets.id` dans la ressource Tweet pour obtenir l’objet avec les seuls champs par défaut. Utilisez l’extension avec le paramètre de champ `tweet.fields` lorsque vous demandez des champs supplémentaires pour compléter l’objet.

| Field Value           | Type   | Description                                                                                                                           | How it Can Be Used                                                  |
|:-----------------------|:--------|:---------------------------------------------------------------------------------------------------------------------------------------|:----------------------------------------------------------------------|
| **id (default)**      | string | L’identifiant unique du Tweet demandé.                                                                                                | Utilisez ceci pour récupérer de manière programmatique un Tweet spécifique. |
| **text (default)**    | string | Le texte UTF-8 du Tweet. Voir [twitter-text](https://github.com/twitter/twitter-text/) pour le détail des caractères valides.        | Extraction de mots-clés et analyse/classification des sentiments.    |
| **edit\_history\_tweet\_ids (default)** | object | Identifiants uniques indiquant toutes les versions d’un Tweet. Pour les Tweets sans modification, il y aura un seul ID. Pour les Tweets avec un historique de modification, il y aura plusieurs ID. | Utilisez ces informations pour retrouver l’historique des modifications d’un Tweet. |
| **attachments**       | object | Spécifie le type de pièces jointes (le cas échéant) présentes dans ce Tweet.                                                         | Comprendre les objets renvoyés pour les expansions demandées.        |
| **author\_id**         | string | L’identifiant unique de l’utilisateur qui a publié ce Tweet.                                                                          | Hydrater l’objet User, partager un jeu de données pour relecture par les pairs. |
| **context\_annotations** | array | Contient des annotations de contexte pour le Tweet.                                                                                  | Reconnaissance/extraction d’entités, analyse thématique.             |
| **conversation\_id**   | string | L’ID du Tweet d’origine de la conversation (qui inclut les réponses directes et leurs sous-réponses).                                | Utilisez ceci pour reconstruire la conversation à partir d’un Tweet. |
| **created\_at**        | date (ISO 8601) | Date et heure de création du Tweet.                                                                                                | Utile pour l’analyse de séries temporelles et pour comprendre quand un Tweet a été créé. |
| **edit\_controls**     | object | Indique pendant combien de temps le Tweet peut encore être modifié et le nombre de modifications restantes.                          | Utilisez ceci pour déterminer si un Tweet est éligible à la modification. |
| **entities**          | object | Entités extraites du texte du Tweet. Voir les entités dans Twitter Objects.                                                          | Fournit des informations supplémentaires sur les hashtags, URL, mentions, etc. |
| **in\_reply\_to\_user\_id** | string | Si le Tweet représenté est une réponse, ce champ contient l’ID de l’auteur du Tweet original.                                       | Déterminer si un Tweet répondait à un autre Tweet.                   |
| **lang**              | string | Langue du Tweet, si détectée par Twitter.                                                                                            | Classer les Tweets par langue.                                       |
| **non\_public\_metrics** | object | Mesures d’engagement non publiques pour le Tweet au moment de la requête. Nécessite une authentification en contexte utilisateur.   | Déterminer le nombre total d’impressions générées pour le Tweet.     |
| **organic\_metrics**   | object | Mesures d’engagement, suivies dans un contexte organique, pour le Tweet au moment de la requête. Nécessite une authentification en contexte utilisateur. | Mesurer l’engagement organique du Tweet.                             |
| **possibly\_sensitive** | boolean | Indique si le contenu peut être considéré comme sensible.                                                                            | Étudier la circulation de certains types de contenu.                 |
| **promoted\_metrics**  | object | Mesures d’engagement, suivies dans un contexte sponsorisé, pour le Tweet au moment de la requête. Nécessite une authentification en contexte utilisateur. | Mesurer l’engagement du Tweet lorsqu’il était sponsorisé.            |
| **public\_metrics**    | object | Mesures d’engagement publiques pour le Tweet au moment de la requête.                                                               | Mesurer l’engagement du Tweet.                                       |
| **referenced\_tweets** | array  | Liste des Tweets auxquels ce Tweet fait référence, tels que des Retweets, des Tweets cités ou des réponses.                         | Comprendre les aspects conversationnels des retweets, etc.           |
| **reply\_settings**    | string | Indique qui peut répondre à un Tweet donné. Les options sont « everyone », « mentioned\_users » et « followers ».                    | Déterminer les paramètres de réponse pour le Tweet.                  |
| **withheld**          | object | Contient des détails relatifs aux restrictions pour le [contenu retenu](https://help.x.com/en/rules-and-policies/tweet-withheld-by-country). |                                                                     |
| **media\_metadata**          | array | Contient les métadonnées des médias joints au Tweet.                                          | Obtenir des métadonnées supplémentaires comme le `alt_text` du média joint à un Tweet. |

**Récupération d’un objet Tweet**

**Exemple de requête**

Dans la requête suivante, nous demandons des champs pour le Tweet sur l’endpoint [Tweets lookup](/fr/resources/fundamentals/rate-limits). Assurez-vous de remplacer « $BEARER\_TOKEN » par votre propre [Bearer Token](/fr/resources/fundamentals/authentication) généré.

```bash
curl --request GET 'https://api.x.com/2/tweets?ids=1212092628029698048&tweet.fields=attachments,author_id,context_annotations,created_at,entities,geo,id,in_reply_to_user_id,lang,possibly_sensitive,public_metrics,referenced_tweets,text,withheld&expansions=referenced_tweets.id' --header 'Authorization: Bearer $BEARER_TOKEN'
```

<div id="user">
  ### Utilisateur
</div>

L’objet utilisateur contient les métadonnées du compte d’un utilisateur X décrivant l’utilisateur référencé. Il s’agit de l’objet principal renvoyé par l’endpoint [users lookup](/fr/x-api/users/lookup/introduction). Pour demander des champs utilisateur supplémentaires sur cet endpoint, utilisez simplement le paramètre `user.fields`.

L’objet utilisateur peut également apparaître comme objet enfant et être inclus dans l’objet Tweet. Il est disponible via les expansions `?expansions=author_id` ou `?expansions=in_reply_to_user_id` pour obtenir une version condensée avec uniquement les champs par défaut. Utilisez ces expansions avec le paramètre de champs `user.fields` pour demander des champs supplémentaires et compléter l’objet.

| :--- | :--- | :--- | :--- |
| id (par défaut) | chaîne | L’identifiant unique de cet utilisateur.<br /><br />`"id": "2244994945"` | À utiliser pour récupérer par programmation des informations sur un utilisateur X spécifique. |
| name (par défaut) | chaîne | Le nom de l’utilisateur, tel qu’il l’a défini sur son profil. Pas nécessairement un nom de personne. En général limité à 50 caractères, mais susceptible d’évoluer.<br /><br />`"name": "Twitter Dev"` |     |
| username (par défaut) | chaîne | Le nom d’affichage, le handle ou l’alias X avec lequel cet utilisateur s’identifie. Les noms d’utilisateur sont uniques mais peuvent changer. **Récupération d’un objet utilisateur**

**Exemple de requête**

Dans la requête suivante, nous demandons des champs pour l’utilisateur sur le point de terminaison [users lookup](/fr/x-api/users/lookup/introduction). Veillez à remplacer `$BEARER_TOKEN` par votre propre [jeton Bearer](/fr/resources/fundamentals/authentication#using-and-generating-an-app-only-bearer-token) généré.
 

```{
  curl --request GET 'https://api.x.com/2/users?
  ids=2244994945&user.fields=created_at,description,entities,id,location,name,pinned_tweet_id,profile_image_url,protected,url,username,verified,withheld&expansions=pinned_tweet_id'
  --header 'Authorization: Bearer $BEARER_TOKEN'
  }
```

<div id="space">
  ### Espace
</div>

Les Espaces permettent de s’exprimer et d’interagir via des conversations audio en direct. Le dictionnaire de données d’un Espace contient les métadonnées pertinentes le concernant ; tous les détails sont mis à jour en temps réel.

Les objets utilisateur peuvent être trouvés et étendus dans la ressource utilisateur. Ils peuvent être inclus via des expansions en ajoutant au moins l’un de `host_ids`, `creator_id`, `speaker_ids` ou `mentioned_user_ids` au paramètre de requête `expansions`.

Contrairement aux posts, les Spaces sont éphémères et deviennent indisponibles une fois qu’ils se terminent ou lorsqu’ils sont annulés par leur créateur. Lorsque votre application traite des données de Spaces, vous êtes responsable de renvoyer les informations les plus à jour et devez supprimer les données qui ne sont plus disponibles sur la plateforme. Les [endpoints de recherche des Spaces](/fr/x-api/spaces/lookup/introduction) peuvent vous aider à respecter les attentes et l’intention des utilisateurs.

<div id="direct-message-events">
  ### Événements de messages privés
</div>

Les conversations de messages privés (DM) sont composées d’événements. L’API X v2 prend actuellement en charge trois types d’événements : MessageCreate, ParticipantsJoin et ParticipantsLeave.

Les objets d’événements de DM sont renvoyés par les points de terminaison [Recherche de messages privés](/fr/x-api/direct-messages/lookup/introduction), et un événement MessageCreate est créé lorsque des messages privés sont créés avec succès via les points de terminaison [Gérer les messages privés](/fr/x-api/direct-messages/lookup/introduction).

Lorsque vous demandez des événements de DM, trois attributs (ou champs) d’objet d’événement sont inclus par défaut : `id`, `event_type` et `text`. Pour recevoir des champs d’événement supplémentaires, utilisez le paramètre [fields](/fr/x-api/fundamentals/fields) `dm_event.fields` pour en sélectionner d’autres. Les autres champs d’événement disponibles incluent : `dm_conversation_id`, `created_at`, `sender_id`, `attachments`, `participant_ids` et `referenced_tweets`.

Plusieurs de ces champs fournissent les identifiants d’autres objets X liés à l’événement de message privé :

- `sender_id` - L’identifiant du compte qui a envoyé le message, ou qui a invité un participant à une conversation de groupe
- `participant_ids` - Un tableau d’identifiants de comptes. Pour les événements ParticipantsJoin et ParticipantsLeave, ce tableau contiendra un seul identifiant correspondant au compte qui a créé l’événement
- `attachments` - Fournit les identifiants de médias pour le contenu téléversé sur X par l’expéditeur
- `referenced_tweets` - Si une URL de post est trouvée dans le champ `text`, l’identifiant de ce post est inclus dans la réponse

Les [expansions](/fr/x-api/fundamentals/expansions) `sender_id`, `participant_ids`, `referenced_tweets.id` et `attachments.media_keys` sont disponibles pour développer ces identifiants d’objets X.

**Récupérer un objet d’événement de Message Privé**

**Exemple de requête**

Pour cet exemple, nous allons créer une requête qui récupère les événements associés à une conversation en tête-à-tête. Cette requête renverra les champs fondamentaux des événements de Message Privé, ainsi que des champs supplémentaires pour les Tweets référencés et leurs auteurs. Construisons une requête qui demande :

- Des attributs d’événement fondamentaux, tels que sa date de création et la conversation dont il fait partie (dm\_conversation).

- Le texte de tout Tweet référencé et sa date de publication.

<div id="how-to-use-fields-and-expansions">
  ## Comment utiliser les champs et les expansions
</div>

Par défaut, les objets de données de l’API X v2 incluent un petit nombre de champs lorsque vous effectuez une requête sans utiliser les paramètres [fields](/fr/x-api/fundamentals/fields) ou [expansions](/fr/x-api/fundamentals/expansions). Ce guide vous montrera comment utiliser les paramètres de requête `fields` et `expansions` afin de recevoir des objets et des champs supplémentaires dans votre réponse.

Dans ce guide, nous allons demander plusieurs champs illustrés dans la capture d’écran du Post suivante.
 

![Cette image montre une capture d’écran d’un Post publié par @X. Vous pouvez voir le texte du Post, le nom d’utilisateur, la date et l’heure de publication, la source et les métriques publiques. Elle inclut également une vidéo.](https://cdn.cms-twdigitalassets.com/content/dam/developer-twitter/ScreenShotDataDictionaryGuide.png.twimg.1920.png)

Comme vous pouvez le voir dans la capture d’écran, plusieurs informations visibles sont liées au Post, notamment l’auteur du Post, les métriques du Post, l’horodatage de création, la vidéo et le nombre de vues de la vidéo. D’autres données ne sont pas visibles dans la capture d’écran, mais restent néanmoins disponibles sur demande. 

Lors de l’envoi d’une requête à l’API, la réponse par défaut est simple et ne contient que les champs par défaut du Post (id et text). Vous recevrez également uniquement l’objet principal renvoyé par l’endpoint que vous utilisez, et non les objets de données associés pouvant être liés à l’objet principal.

Cette simplicité, combinée aux paramètres fields et expansions, vous permet de demander uniquement les champs dont vous avez besoin, selon votre cas d’usage.

```bash
curl --request GET --url 'https://api.x.com/2/tweets?ids=1260294888811347969' \
```
