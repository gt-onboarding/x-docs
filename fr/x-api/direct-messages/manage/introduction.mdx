Les Messages privés permettent des conversations privées sur X. Il s’agit de l’une des fonctionnalités les plus populaires de X, avec une grande variété de cas d’usage, allant des discussions de groupe entre amis à l’assistance client pour des marques du monde entier. De nouvelles versions v2 des endpoints des Messages privés seront introduites progressivement. Cette première étape inclut des endpoints essentiels pour créer des Messages privés et lister les événements des conversations. Pour la première fois, l’API X v2 prend en charge les conversations de groupe.

Cette version initiale de Manage Direct Messages comprend trois méthodes POST pour créer des Messages privés :

- **POST /2/dm\_conversations/with/:participant\_id/messages** - Crée un Message privé en tête-à-tête. Cette méthode crée soit une nouvelle conversation 1-1, soit récupère la conversation en cours et y ajoute le Message privé. Le paramètre de chemin :participant\_id est l’ID utilisateur du compte recevant le message.
- **POST /2/dm\_conversations** - Crée une nouvelle conversation de groupe et y ajoute un Message privé. Ces requêtes nécessitent une liste des participants à la conversation. Notez que vous pouvez créer plusieurs conversations avec la même liste de participants. Ces requêtes renvoient toujours un nouvel ID de conversation.
- **POST /2/dm\_conversations/:dm\_conversation\_id/messages** - Crée un Message privé et l’ajoute à une conversation existante. Le paramètre de chemin :dm\_conversation\_id est l’ID de la conversation à laquelle le message sera ajouté.

Notez que les IDs d’événements de Messages privés sont communs entre v1.1 et v2 (ainsi que l’app X), donc les méthodes v1.1 pour masquer/supprimer des Messages privés peuvent être utilisées avec ce nouvel endpoint v2. Notez également que les Enterprise et Premium Account Activity APIs prennent en charge les messages 1-1 v2, mais ne prennent pas encore en charge les conversations de groupe.

Il existe une limite de débit utilisateur de 200 requêtes par 15 minutes pour la méthode POST. Il existe également une limite de 1 000 requêtes par 24 heures et par utilisateur. De plus, il existe une limite de 15 000 requêtes par 24 heures. Notez que ces limites de débit sont partagées entre ces endpoints POST.

Comme vous effectuez des requêtes au nom d’un utilisateur avec les endpoints manage Posts, vous devez vous authentifier soit avec [OAuth 1.0a User Context](https://developer-staging.x.com/resources/fundamentals/authentication), soit avec [OAuth 2.0 Authorization Code with PKCE](fr/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2), et utiliser un jeton d’accès utilisateur associé à un utilisateur ayant autorisé votre App. Pour générer ce jeton d’accès utilisateur avec OAuth 1.0a, vous pouvez utiliser le [flux OAuth à 3 étapes](/fr/resources/fundamentals/authentication#using-and-generating-an-app-only-bearer-token). Pour générer un jeton d’accès utilisateur avec OAuth 2.0, vous pouvez utiliser le [flux d’autorisation Authorization Code with PKCE](/fr/resources/fundamentals/authentication#how-to-connect-to-endpoints-using-oauth-2-0-authorization-code-flow-with-pkce)).

<Note>
  **Configuration du compte**

  Pour accéder à ces endpoints, vous aurez besoin :

  - D’un [compte développeur](https://developer.x.com/en/portal/petition/essential/basic-info) approuvé.
  - De vous authentifier en utilisant les clés et jetons d’une [App développeur](/fr/resources/fundamentals/developer-apps) qui se trouve dans un [Project](/fr/resources/fundamentals/projects).

  En savoir plus sur l’accès aux endpoints de l’API X v2 dans notre [guide de démarrage](/fr/x-api/getting-started/getting-access).
</Note>
