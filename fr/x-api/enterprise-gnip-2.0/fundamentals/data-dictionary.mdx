<div id="introduction">
  ### Introduction
</div>

`Enterprise`

Les posts sont l’unité atomique de base de tout ce qui concerne X. Toutes les API X qui renvoient des posts fournissent ces données encodées en JSON (JavaScript Object Notation). Le JSON repose sur des paires clé-valeur, avec des attributs nommés et des valeurs associées. Les objets Post récupérés via l’API incluent la « mise à jour de statut » d’un utilisateur X, mais les Retweets, réponses et Tweets cités sont également des objets Post. Si un post est lié à un autre (Retweet, réponse ou Tweet cité), chacun sera identifié ou intégré dans l’objet Post. Même le post le plus simple, au format de données natif de X, contient des objets JSON imbriqués représentant d’autres attributs d’un post, tels que l’auteur, les utilisateurs mentionnés, le lieu tagué, les hashtags, les symboles de cashtag, les médias ou les liens URL. Lorsqu’on travaille avec les données X, c’est un concept important à maîtriser. Le format des données de post que vous recevrez de l’API X dépend du type de post, de l’API X utilisée et des paramètres de format.

Les endpoints Enterprise qui renvoient des objets Post ont été mis à jour pour fournir les métadonnées nécessaires à la compréhension de l’historique des modifications du post. Pour en savoir plus, consultez la page ["Modifier des posts"](/fr/x-api/fundamentals/edit-posts).

<div id="available-data-formats">
  ### Formats de données disponibles
</div>

> Veuillez noter : il est fortement recommandé d’utiliser le format [Native enrichi](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) pour les API de données Enterprise.
>
> - Le format Native enrichi inclut toutes les nouvelles métadonnées depuis 2017, telles que les [métadonnées de sondage](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata), ainsi que des métriques supplémentaires comme reply\_count et quote\_count.
>
> - Le format Activity Streams n’a pas été mis à jour avec de nouvelles métadonnées ou enrichissements depuis la [mise à jour des caractères](https://blog.x.com/official/en_us/topics/product/2017/Giving-you-more-characters-to-express-yourself.html) de 2017.

Les API de données Enterprise délivrent des données sous deux formats différents. Le format Enterprise le plus proche du format natif standard v1.1 est le Native enrichi. L’ancien format de données Enterprise est Activity Streams, initialement implémenté et utilisé par Gnip comme format normalisé à travers X et d’autres fournisseurs de données sociales à l’époque. Bien que ce format soit toujours disponible, X n’a investi dans de nouvelles fonctionnalités et évolutions que sur le format native enrichi depuis 2017.

Le format native enrichi est conforme à son intitulé : il inclut les objets natifs X ainsi que des enrichissements supplémentaires disponibles pour les produits de données Enterprise, tels que les métadonnées de déroulage d’URL, la géolocalisation de profil, les métadonnées de sondage et des métriques d’engagement supplémentaires.

- [Enrichissement des URL développées et améliorées](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls)
- [Enrichissement des règles de correspondance](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#matching-rules)
- [Enrichissement des métadonnées de sondage](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata)
- [Enrichissement de la géolocalisation du profil](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo)

### Bonnes pratiques d’analyse

- Le JSON de X est encodé en UTF-8.
- Les parseurs doivent facilement tolérer des variations dans l’ordre des champs. Il faut supposer que le JSON de post est fourni comme une table de hachage non ordonnée.
- Les parseurs doivent tolérer l’ajout de « nouveaux » champs.
- Les parseurs JSON doivent tolérer les champs « manquants », car tous les champs n’apparaissent pas dans tous les contextes.
- Il est généralement sûr de considérer qu’un champ nul, un ensemble vide et l’absence d’un champ sont équivalents.

#### Objet Post

Lorsque vous utilisez des produits de données d’entreprise, vous constaterez qu’une grande partie du dictionnaire de données est similaire au format natif des données de Post, avec des métadonnées enrichies supplémentaires. Le niveau de base du format natif enrichi reprend nombre des mêmes noms d’objets que le [format de données de l’API X v1.1](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview.html). L’objet Post comporte une longue liste d’attributs de « niveau racine », dont des attributs fondamentaux tels que `id`, `created_at` et `text`. Les objets Post comportent également des objets imbriqués comme `user`, `entities` et `extended_entities`. Ils peuvent aussi inclure d’autres [objets Post imbriqués](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#nested-post-objects) tels que retweeted\_status, quoted\_status et extended\_tweet. Le format natif enrichi inclut en outre un objet matching\_rules.

<div id="x-data-dictionary">
  ##### Attributs supplémentaires du Post
</div>

Les API X qui fournissent des Posts (p. ex. l’endpoint GET statuses/lookup) peuvent inclure ces attributs supplémentaires du Post :

| Attribute | Type | Description |

#### Objets Post imbriqués

Dans plusieurs cas, un objet Post inclut d’autres objets imbriqués. Si vous travaillez avec des objets imbriqués, la charge utile JSON contiendra alors plusieurs objets Post, et chaque objet Post peut contenir ses propres objets. L’objet de niveau racine inclut des informations sur le type d’action effectuée, c’est-à-dire s’il s’agit d’un Retweet ou d’un Quote Tweet, et peut également contenir un objet décrivant le Post « original » partagé. Les Posts étendus incluent un objet étendu imbriqué qui dépasse 140 caractères, utilisé pour éviter toute rupture de compatibilité lors de la mise à jour de 2017. Chaque dictionnaire d’objet imbriqué est décrit ci-dessous.

**Retweets**

Les Retweets contiennent toujours deux objets Post. Le Post « original » faisant l’objet du Retweet est fourni dans un objet "retweeted\_status". L’objet de niveau racine encapsule le Retweet lui-même, y compris un objet User pour le compte effectuant l’action de Retweet et l’heure du Retweet. Le Retweet est une action consistant à partager un Post avec vos abonnés, sans possibilité d’ajouter de nouveau contenu. De plus, un (nouvel) emplacement ne peut pas être fourni avec un Retweet. Bien que le Post « original » puisse être géolocalisé, les objets "geo" et "place" du Retweet seront toujours null.

Même avant l’introduction des Posts étendus, l’objet "entities" de niveau racine était, dans certains cas, tronqué et incomplet en raison de la chaîne "RT @username " ajoutée au message du Post faisant l’objet du Retweet. Notez que si un Retweet est lui-même retweeté, "retweet\_status" pointe toujours vers le Post original, ce qui signifie que le Retweet intermédiaire n’est pas inclus. Un comportement similaire est observé lors de l’utilisation de x.com pour « afficher » un Retweet. Si vous copiez l’ID de Post unique attribué à l’« action » de Retweet, le Post original est affiché.

Ci-dessous se trouve un exemple de structure pour un Retweet. À nouveau, lors de l’analyse des Retweets, il est essentiel d’examiner l’objet "retweeted\_status" pour obtenir le message du Post (original) complet et les métadonnées d’entités.

```json
{
	"tweet": {
		"text": "RT @author original message",
		"user": {
			"screen_name": "Retweeter"
		},
		"retweeted_status": {
			"text": "original message",
			"user": {
				"screen_name": "OriginalTweeter"
			},
			"place": {},
			"entities": {},
			"extended_entities": {}
		}
	},
	"entities": {},
	"extended_entities": {}
}
```

##### Tweets cités

Les Tweets cités sont similaires aux Retweets, à la différence qu’ils incluent un nouveau message de Post. Ces nouveaux messages peuvent contenir leur propre lot de hashtags, de liens et d’autres métadonnées d’« entités ». Les Tweets cités peuvent également inclure des informations de localisation partagées par l’utilisateur qui publie le Tweet cité, ainsi que des médias tels que des GIF, des vidéos et des photos.

Les Tweets cités contiennent au minimum deux objets Post, et dans certains cas trois. Le Post cité, qui peut lui‑même être un Tweet cité, est fourni dans un objet « quoted\_status ». L’objet de niveau racine encapsule le Tweet cité lui‑même, y compris un objet User pour le compte qui effectue l’action de partage et l’horodatage du Tweet cité.

Notez que les Tweets cités peuvent désormais inclure des photos, des GIF ou des vidéos, ajoutés via l’interface utilisateur « Post ». Lorsque des liens vers des médias hébergés en externe sont inclus dans le message du Tweet cité, le « entities.urls » de niveau racine les décrit. Les médias joints aux Tweets cités apparaissent dans les métadonnées « extended\_entities » de niveau racine.

Lorsque les Tweets cités ont été lancés pour la première fois, un lien raccourci (URL t.co) était ajouté au message du Post « original » et fourni dans le champ « text » de niveau racine. De plus, les métadonnées de cette URL t.co étaient incluses dans le tableau « entities.urls » de niveau racine. En mai 2018, nous avons modifié ce comportement afin que l’URL t.co raccourcie pointant vers le Tweet cité ne soit plus incluse dans le champ « text » de niveau racine. De même, les métadonnées du Tweet cité ne sont plus incluses dans les métadonnées « entities.urls ». À la place, les métadonnées d’URL du Tweet cité se trouvent dans un nouvel objet « quoted\_status\_permalink » au niveau racine (ou niveau supérieur), donc au même niveau que l’objet « quoted\_status ».

<div id="quote-tweets">
  ##### Posts étendus
</div>

Le JSON décrivant les _Posts étendus_ a été introduit lors du lancement des Posts de 280 caractères en novembre 2017. Le JSON des Posts a été étendu pour encapsuler ces messages plus longs, sans casser les milliers d’applications qui analysent ces objets X fondamentaux. Pour assurer une rétrocompatibilité complète, le champ « text » d’origine de 140 caractères, ainsi que les objets d’entités qui en sont extraits, ont été conservés. Pour les Posts de plus de 140 caractères, ce champ « text » au niveau racine est tronqué et donc incomplet. Étant donné que les objets « entities » au niveau racine contiennent des tableaux de métadonnées clés extraites du message « text » (comme les hashtags et les liens inclus), ces collections deviennent incomplètes. Par exemple, si un message de Post comporte 200 caractères avec un hashtag à la fin, le tableau hérité « entities.hashtags » au niveau racine ne l’inclura pas.

Un nouveau champ « extended\_tweet » a été introduit pour contenir les messages de Post plus longs et les métadonnées d’entités complètes. L’objet « extended\_tweet » fournit le champ « full\_text » qui contient le message de Post complet, non tronqué, lorsqu’il dépasse 140 caractères. L’objet « extended\_tweet » contient également un objet « entities » avec des tableaux complets de hashtags, liens, mentions, etc.

Les Posts étendus sont identifiés par un booléen « truncated » au niveau racine. Lorsqu’il est à true (« truncated »: true), les champs « extended\_tweet » doivent être analysés au lieu des champs au niveau racine.

Notez dans l’exemple JSON ci-dessous que le champ « text » au niveau racine est tronqué et que le tableau « entities.hashtags » au niveau racine est vide, même si le message du Post inclut trois hashtags. Comme il s’agit d’un Post étendu, le champ « truncated » est défini sur true, et l’objet « extended\_tweet » fournit des métadonnées complètes du Post pour « full\_text » et « entities ».

### Objets géo natifs enrichis

Les posts peuvent être associés à un lieu, produisant un post « géolocalisé ». Les lieux peuvent être définis via l’interface utilisateur de X ou lors de la publication d’un post à l’aide de l’API. Le lieu peut être un « point » exact ou un X Place avec une « boîte englobante » décrivant une zone plus large, d’un site à une région entière.

Il existe trois objets JSON « au niveau racine » utilisés pour décrire le lieu associé à un post : [place](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#place-data-dictionary), [geo](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-geo-objects) et [coordinates](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#coordinates-object-data-dictionary).

De plus, le format natif enrichi inclut le [lieu dérivé](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#derived-locations) de l’[enrichissement de géolocalisation de profil](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo) au sein de l’objet utilisateur.

L’objet `place` est toujours présent lorsqu’un post est géolocalisé avec un lieu. Les Places sont des lieux spécifiques et nommés, avec des coordonnées géographiques associées. Lorsque les utilisateurs choisissent d’associer un lieu à leur post, une liste de X Places candidates leur est proposée. Lors de l’utilisation de l’API pour publier, un X Place peut être associé en spécifiant un place\_id lors de la publication. Les posts associés à des Places ne proviennent pas nécessairement de ce lieu, mais peuvent aussi être _à propos_ de ce lieu.

Les objets geo et `coordinates` ne sont présents (non nuls) que lorsque le post est associé à un _lieu exact_. Si un lieu exact est fourni, l’objet `coordinates` renverra un tableau \[long, lat] avec les coordonnées géographiques et un X Place correspondant à ce lieu sera associé.

<div id="x-entities">
  ## Introduction
</div>

Les entités fournissent des métadonnées et des informations contextuelles supplémentaires sur le contenu publié sur X. La section `entities` contient des tableaux d’éléments courants présents dans les Posts : hashtags, mentions d’utilisateur, liens, tickers boursiers (symboles), sondages X et médias associés. Ces tableaux sont pratiques pour les développeurs lors de l’ingestion de Posts, puisque X a en quelque sorte prétraité, ou pré-analysé, le corps du texte. Plutôt que de devoir rechercher et identifier explicitement ces entités dans le corps du Post, votre parseur peut accéder directement à cette section JSON, où elles se trouvent.

Au-delà de ces facilités de parsing, la section `entities` fournit également des métadonnées utiles à valeur ajoutée. Par exemple, si vous utilisez l’[enrichissement Enhanced URLs](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls), les métadonnées d’URL incluent les URL entièrement développées, ainsi que les titres et descriptions de sites web associés. Autre exemple : lorsqu’il y a des mentions d’utilisateur, les métadonnées des entités incluent l’identifiant utilisateur numérique, utile pour effectuer des requêtes vers de nombreuses API X.

Chaque payload JSON de Post inclut une section `entities`, contenant au minimum les attributs `hashtags`, `urls`, `user_mentions` et `symbols`, même si aucune de ces entités ne figure dans le message du Post. Par exemple, si vous examinez le JSON d’un Post dont le corps est « Hello World! » et sans média associé, le JSON du Post inclura le contenu suivant avec des tableaux d’entités contenant zéro élément :

```
"entities": {
        "hashtags": [
        ],
        "urls": [
        ],
        "user_mentions": [
        ],
        "symbols": [
        ]
      }
```

Notes :

- les entités media et polls n’apparaissent que lorsque ce type de contenu fait partie du Post.
- si vous travaillez avec des médias natifs (photos, vidéos ou GIF), l’[objet Extended Entities](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#x-extended-entities) est la référence à utiliser.

### Objet Entities

Les sections `entities` et `extended_entities` sont toutes deux constituées de tableaux d’_objets_ d’entité. Ci-dessous, vous trouverez des descriptions de chacun de ces objets, ainsi que des dictionnaires de données décrivant les noms et types d’attributs de l’objet, ainsi qu’une brève description. Nous indiquerons également quels opérateurs PowerTrack correspondent à ces attributs et inclurons quelques exemples de charges utiles JSON.

Une collection d’entités courantes présentes dans les Posts, notamment des hashtags, des liens et des mentions d’utilisateurs. L’objet `entities` comprend un attribut `media`, mais sa mise en œuvre dans la section `entities` n’est pleinement exacte que pour les Posts avec une seule photo. Pour tous les Posts comportant plus d’une photo, une vidéo ou un GIF animé, reportez-vous à la section `extended_entities`.

### Objet Hashtag  

La section `entities` contient un tableau `hashtags` avec un objet pour chaque hashtag présent dans le corps du Post, et un tableau vide s’il n’y a aucun hashtag.

L’opérateur PowerTrack `#` est utilisé pour faire correspondre l’attribut `text`. L’opérateur `has:hashtags` correspond s’il y a au moins un élément dans le tableau.

|     |     |     |
| :--- | :--- | :--- |
| Champ | Type | Description |
| indices | Tableau d’entiers | Tableau d’entiers indiquant les positions (offsets) dans le texte du Post où le hashtag commence et se termine. Le premier entier représente l’emplacement du caractère # dans la chaîne de texte du Post. Le second entier représente l’emplacement du premier caractère après le hashtag. Ainsi, la différence entre les deux nombres correspond à la longueur du nom du hashtag, plus un (pour le caractère « # »). Exemple :<br /><br />"indices":\[32,38] |
| text | Chaîne | Nom du hashtag, sans le caractère « # » initial. Exemple :<br /><br />"text":"nodejs" |

### Objet média  

La section `entities` contiendra un tableau `media` avec un seul objet média si un média a été « attaché » au Post. \+ Le `type` du média indiquera toujours « photo », même lorsqu’une vidéo ou un GIF est attaché au Post.
\+ Bien que jusqu’à quatre photos puissent être attachées, seule la première sera listée dans la section `entities`.

### Formatage des URL de médias photo

Les médias photo sur X peuvent être chargés dans différentes tailles. Il est préférable de charger l’image la plus petite qui soit suffisamment grande pour s’adapter à une zone d’affichage donnée. Pour charger différentes tailles, l’[objet Size](#size) et [media\_url](#media) (ou [media\_url\_https](#media)) doivent être combinés selon un format particulier. Nous utiliserons l’exemple d’[entité media](#entitiesobject) déjà fourni pour construire une URL de média photo.

Les `media_url` ou `media_url_https` peuvent être utilisées telles quelles, ce qui charge par défaut la variante medium. Il est toutefois préférable de fournir une URL de média photo entièrement formatée lorsque c’est possible.

Il y a trois parties dans une URL de média photo :

|     |     |
| :--- | :--- |
| Base URL | La Base URL est l’URL du média sans l’extension de fichier.<br /><br />Par exemple :  <br /><br />"media\_url\_https": "https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg",<br /><br />La Base URL est donc :  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq_ |
| Format | Le format correspond au type de photo. Les formats possibles sont _jpg_ ou _png_, fournis comme extension de l’URL du média.<br /><br />Par exemple :  <br /><br />"media\_url\_https": "https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg",<br /><br />Le format est donc : _jpg_ |
| Name | Le Name est le nom du champ indiquant la taille à charger.<br /><br />Par exemple :<br /><br />  <br /> "sizes":   <br />   "thumb":   <br />     "h": 150,  <br />     "resize": "crop",  <br />     "w": 150  <br />   ,  <br />   "large": <br />     "h": 1366,  <br />     "resize": "fit",  <br />     "w": 2048  <br />   },  <br />   "medium": <br />     "h": 800,  <br />     "resize": "fit",  <br />     "w": 1200  <br />   },  <br />   "small": <br />     "h": 454,  <br />     "resize": "fit",  <br />     "w": 680  <br />   }  <br /> }  <br />}<br /><br />Le Name lors du chargement de la photo en grande taille serait : _large_ |

Nous prenons ces trois parties (Base URL, format et Name) et les combinons pour former l’URL du média photo à charger. Il existe 2 formats pour charger des images de cette manière, _legacy_ et _modern_. Tous les chargements d’images doivent cesser d’utiliser le format _legacy_ et adopter le format _modern_. L’utilisation du format _modern_ améliorera le taux de réussite de cache sur le CDN pour l’appelant, réduisant ainsi les latences de chargement en diminuant la probabilité de devoir générer et charger le média depuis le centre de données.

|     |     |
| :--- | :--- |
| Legacy format | Le format hérité est obsolète. Tous les chargements de médias photo doivent passer au format moderne.<br /><br />_\<base\_url>.\<format>:\<name>_  <br /><br />Par exemple :  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg:large  <br />_ |
| Modern format | Le format moderne de chargement des photos a été introduit chez X en 2015 et fait référence de facto depuis 2017. Tous les chargements de médias photo doivent passer à ce format.<br /><br />_\<base\_url>?format=\<format>\&name=\<name>_  <br /><br />Par exemple :  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq?format=jpg\&name=large  <br />_<br /><br />Remarque : les paramètres de la chaîne de requête de l’URL du média photo sont classés par ordre alphabétique. Si le chargement des médias amenait l’ajout de paramètres supplémentaires, l’ordre alphabétique devrait continuer d’être respecté. Par exemple, si un nouveau paramètre hypothétique nommé _preferred\_format_ était ajouté, il viendrait après _format_ et _name_ dans la chaîne de requête. |

## Détails sur les Retweets et les Tweets cités

Du point de vue de l’API X, les Retweets et les Tweets cités sont des types particuliers de Posts qui contiennent le Post d’origine en tant qu’objet intégré. Ainsi, les Retweets et les objets de Tweets cités sont les parents d’un Post « original » enfant (et doublent donc la taille). Les Retweets possèdent un objet de niveau supérieur "retweeted\_status", et les Tweets cités possèdent un objet "quoted\_status". Par cohérence, ces objets de niveau supérieur pour les Retweets et les Tweets cités disposent également d’une propriété text et d’entities associées. Cependant, les entities au niveau supérieur peuvent différer des entities fournies par les entities « originales » intégrées. Dans le cas des Retweets, un nouveau texte est préfixé au corps du Post d’origine. Pour les Posts cités, un nouveau texte est suffixé au corps du Post.

De manière générale, la meilleure pratique consiste à récupérer le texte, les entities, l’auteur d’origine et la date à partir du Post d’origine dans retweeted\_status lorsque celui-ci existe. Une exception concerne la récupération des entities X qui font partie de l’ajout dans la citation. Voir ci-dessous pour plus de détails et des conseils.

### Retweets

Un détail important concernant les Retweets est qu’aucune autre _entité_ X ne peut être ajoutée au Post. Les utilisateurs ne peuvent pas ajouter de hashtags, d’URL ni d’autres détails lorsqu’ils effectuent un Retweet. Toutefois, l’attribut text (au niveau supérieur) du Retweet est composé du texte du Post original précédé de « RT @username: ».

Dans certains cas, en particulier pour les comptes ayant des noms d’utilisateur longs, la combinaison de ces nouveaux caractères et du corps du Post original peut facilement dépasser la limite de 140 caractères du texte du Post original. Afin de préserver la compatibilité avec l’affichage et le stockage basés sur 140 caractères, le corps de niveau supérieur tronque la fin du corps du Post et ajoute une ellipse (« … »). Par conséquent, certaines entités de niveau supérieur positionnées à la fin du Post original peuvent être incorrectes ou manquantes, par exemple dans le cas d’un hashtag ou d’une URL tronqués.

Ce Post, https://x.com/FloodSocial/status/907974220298125312, contient le texte de Post suivant :

  ### Tweets cités
</div>

Les Tweets cités ont été introduits en 2016 et se distinguent des Retweets en ce que lorsque vous « citez » un Post, vous ajoutez du nouveau contenu « par-dessus » un Post partagé. Les Tweets cités peuvent contenir des médias natifs (photos, vidéos et GIF) et apparaissent sous l’objet entities.

Étant donné que des entités X peuvent être ajoutées, les entités de la citation sont probablement différentes des entités originales.

#### Introduction

Si un Post contient des médias natifs (partagés via l’interface utilisateur du Post plutôt que via un lien externe), une section extended\_entities sera également présente. Pour tout média natif (photo, vidéo ou GIF), extended\_entities est la source de métadonnées recommandée pour plusieurs raisons. Actuellement, jusqu’à quatre photos peuvent être jointes à un Post. Les métadonnées entities ne contiendront que la première photo (jusqu’en 2014, une seule photo pouvait être incluse), tandis que la section extended\_entities inclura toutes les photos jointes. Avec les médias natifs, une autre limite des métadonnées entities.media est que le type de média indiquera toujours « photo », même lorsque le média joint est une vidéo ou un GIF animé. Le type réel du média est spécifié dans l’attribut extended\_entities.media\[].type et prend la valeur _photo_, _video_ ou _animated\_gif_. Pour ces raisons, si vous travaillez avec des médias natifs, les métadonnées extended\_entities sont à privilégier.

Tous les Posts avec des photos, vidéos et GIF animés joints incluent un objet JSON `extended_entities`. L’objet `extended_entities` contient un unique tableau `media` d’objets `media` (voir la section `entities` pour son dictionnaire de données). Aucun autre type d’entité, comme les hashtags et les liens, n’est inclus dans la section `extended_entities`. L’objet `media` de la section `extended_entities` est identique, en structure, à celui inclus dans la section `entities`.

Les Posts ne peuvent avoir qu’un seul type de média joint. Pour les photos, jusqu’à quatre peuvent être jointes. Pour les vidéos et les GIF, un seul peut être joint. Étant donné que les métadonnées `type` de la section `extended_entities` indiquent correctement le type de média (« photo », « video » ou « animated\_gif ») et prennent en charge jusqu’à 4 photos, il s’agit de la source de métadonnées privilégiée pour les médias natifs.

```
{
      "extended_entities": {
        "media": [

        ]
      }
    }
```

Lorsqu’un annonceur choisit de restreindre la lecture vidéo aux seules plateformes détenues et exploitées par X, l’objet `video_info` est remplacé par un objet `additional_media_info`.

L’objet `additional_media_info` contient des informations média supplémentaires fournies par l’éditeur, telles que `title`, `description` et le drapeau `embeddable`. Le contenu vidéo n’est accessible qu’aux clients officiels de X lorsque `embeddable=false`. Dans ce cas, toutes les URL vidéo fournies dans la charge utile seront hébergées par X, afin que l’utilisateur puisse ouvrir la vidéo sur une propriété appartenant à X en cliquant sur le lien.

## Objets de données Enterprise Activity Streams

<Note>
  Vous souhaitez en savoir plus sur la façon dont le format de données Activity Streams correspond au format de l’API X v2 ?
</Note>

Consultez notre guide de comparaison : [Activity Streams comparé à l’API X v2](/fr/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2)

<Note>
  Veuillez noter : il est fortement recommandé d’utiliser le format [Enriched Native](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) pour les API de données Enterprise.

  - Le format Enriched Native inclut toutes les nouvelles métadonnées depuis 2017, telles que les [métadonnées de sondage](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata), ainsi que des métriques supplémentaires comme reply\_count et quote\_count.

  - Le format Activity Streams n’a pas été mis à jour avec de nouvelles métadonnées ou enrichissements depuis la [mise à jour du nombre de caractères](https://blog.x.com/official/en_us/topics/product/2017/Giving-you-more-characters-to-express-yourself.html) en 2017.
</Note>

### Objet Activity

Activity Streams est une représentation de schéma d’objets du [format de données d’origine](https://developer.x.comhttps://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) de X, créée par Gnip pour « normaliser le format » des données de posts et d’autres données de médias sociaux, en s’appuyant sur le [schéma Activity Base décrit ici](https://activitystrea.ms/head/activity-schema.html) fourni par un tiers. Les posts sont normalisés selon le schéma Activity Streams, notamment via les types d’objets note, person, place et service, utilisés comme objets imbriqués. Les posts peuvent également inclure d’autres objets d’activité de post imbriqués pour les Retweets ou autres, notamment twitter\_quoted\_status et long\_object.

Le type d’objet de niveau de base « activity » est similaire à l’[objet Post de niveau de base](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) du format natif enrichi. Des exemples de payloads au format Activity Streams sont disponibles [ici](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#activity-streams-payload-examples).

<div id="deprecated-attributes">
  #### Objets d’activité de Post imbriqués
</div>

Dans plusieurs cas, un objet Post inclura d’autres Posts imbriqués. Si vous travaillez avec des objets imbriqués, la charge utile JSON contiendra alors plusieurs objets, et chaque objet Post peut contenir ses propres objets. L’objet au niveau racine contiendra des informations sur le type d’action effectuée, c’est‑à‑dire s’il s’agit d’un Retweet ou d’un Tweet cité, et peut également contenir un objet décrivant le Post « original » partagé. Les Posts étendus incluront un objet étendu imbriqué qui dépasse 140 caractères, utilisé pour éviter des changements rétro‑incompatibles lors de la mise à jour de 2017. Chaque dictionnaire d’objet imbriqué est décrit ci‑dessous.

### Introduction\*\*

Au fond, X est un réseau de communication public, en temps réel et mondial. Depuis 2006, l’évolution de X a été portée à la fois par les usages et conventions des utilisateurs et par les nouvelles fonctionnalités et améliorations produit. Si vous exploitez des données X pour des recherches historiques, comprendre la chronologie de cette évolution est essentiel pour faire ressortir des Posts pertinents depuis l’archive de données.

X a été lancé comme une simple application SMS sur mobile et est devenu une plateforme de communication complète. Une plateforme dotée d’un ensemble exhaustif d’API. Les API ont toujours été un pilier du réseau X. La [première API est arrivée peu après le lancement de X](https://blog.x.com/2006/introducing-the-twitter-api). Lorsque l’annotation géographique des Posts a été introduite en 2009, elle a été proposée via une [API Geo](https://blog.x.com/2009/think-globally-tweet-locally) (et, plus tard, la possibilité de « géomarquer » un Post a été intégrée à l’interface utilisateur de X.com). Aujourd’hui, les API de X alimentent le réseau de communication bidirectionnel devenu une source de dernières nouvelles et de partage d’informations. Les opportunités de construire sur ce canal de communication mondial et en temps réel sont infinies.

X met à disposition deux API historiques qui donnent accès à chaque Post public : Historical PowerTrack et l’[API de recherche sur l’archive complète](https://developer.x.com/en/docs/x-api/v1/tweets/search/overview). Les deux API fournissent un ensemble d’_opérateurs_ utilisés pour interroger et collecter des Posts d’intérêt. Ces opérateurs correspondent à une variété d’attributs associés à chaque Post — des centaines d’attributs, tels que le contenu textuel du Post, le nom de compte de l’auteur et les liens partagés dans le Post. Les Posts et leurs attributs sont encodés en JSON, un format d’échange de données textuel couramment utilisé. Ainsi, à mesure que de nouvelles fonctionnalités étaient introduites, de nouveaux attributs JSON apparaissaient et, généralement, de nouveaux opérateurs d’API étaient ajoutés pour cibler ces attributs. Si votre cas d’usage implique d’_écouter_ ce que le monde a dit sur X, plus vous savez depuis quand les opérateurs disposent de métadonnées JSON sur lesquelles faire correspondre, plus vos filtres PowerTrack historiques seront efficaces.

Ensuite, nous présenterons quelques concepts clés qui posent les bases pour comprendre comment les mises à jour des métadonnées des Posts influent sur la découverte de votre signal de données d’intérêt.

### Concepts clés\*\*

#### Des conventions d’utilisateurs aux _objets de première classe_ de X

Les utilisateurs de X ont, de manière organique, introduit de nouveaux modes de communication qui sont désormais fondamentaux au sein du réseau X. Un exemple fondateur est le hashtag, aujourd’hui presque universel sur tous les réseaux sociaux. Les hashtags ont été introduits pour organiser les conversations et les sujets. Sur un réseau comptant des centaines de millions de messages par jour, les outils permettant de trouver des Posts pertinents sont essentiels, et les hashtags en sont devenus un moyen clé. Peu après la généralisation des hashtags, ils ont reçu un statut officiel et le support de X. Lorsque les hashtags sont devenus un _« objet de première classe »_, cela a impliqué plusieurs choses. D’une part, les hashtags sont devenus cliquables et recherchables dans l’interface de X.com. D’autre part, ils ont rejoint la famille des _entités_ de X, aux côtés des @mentions, des médias joints, des symboles boursiers et des liens partagés. Ces entités sont encodées de façon pratique dans un tableau JSON pré-parsé, ce qui facilite leur traitement, leur analyse et leur stockage par les développeurs.

Les retweets constituent un autre exemple de conventions initiées par les utilisateurs devenant des objets officiels. Le retweet est apparu comme un moyen de « transférer » du contenu à d’autres. Il a commencé comme un processus manuel consistant à copier-coller un Post et à le faire précéder du motif « RT @ ». Ce processus a finalement été automatisé via un nouveau bouton Retweet, accompagné de nouvelles métadonnées JSON. Le retweet « officiel » était né. D’autres exemples incluent les « mentions », le partage de médias et de liens web, ainsi que l’ajout d’une localisation à votre Post. Chacun de ces schémas d’usage a donné lieu à de nouvelles fonctionnalités dans l’interface de [x.com](https://x.com/), à de nouvelles structures JSON de support, et donc à de nouveaux moyens de faire correspondre des Posts. L’ensemble de ces attributs fondamentaux des Posts a abouti à des PowerTrack Operators permettant de les faire correspondre.

<div id="from-user-conventions-to-x-first-class-objects">
  #### Métadonnées de Post, mutabilité, mises à jour et actualité
</div>

Bien que les Posts puissent comporter jusqu’à un nombre fixe de caractères, la description JSON d’un Post comprend plus de 100 attributs. Parmi eux : qui a publié, à quel moment, s’il s’agit d’un Post original ou d’un Retweet, ainsi qu’un tableau d’objets de premier niveau tels que hashtags, mentions et liens partagés. Pour le compte qui a publié, il existe un objet User (ou Actor) avec divers attributs fournissant le profil de l’utilisateur et d’autres métadonnées du compte. Les profils incluent une courte biographie, un lieu d’origine (texte libre), une langue préférée et un lien de site web facultatif.

Certaines métadonnées de compte ne changent jamais (p. ex. l’ID utilisateur numérique et la date de création), d’autres évoluent lentement, tandis que certains attributs changent plus fréquemment. Les personnes changent d’emploi et déménagent. Les entreprises mettent à jour leurs informations. Lorsque vous collectez des Posts historiques, il est important de comprendre que certaines métadonnées sont telles qu’elles étaient au moment de la publication, tandis que d’autres sont telles qu’elles sont au moment où la requête est soumise.

Avec toutes les API historiques, la description du profil utilisateur, le nom d’affichage et les attributs « home » du profil sont actualisés avec les valeurs en vigueur au moment de la requête.

<div id="post-metadata-mutability-updates-and-currency">
  #### Médias « natifs »
</div>

X.com et les applications mobiles X permettent d’ajouter des photos et des vidéos à un Post en cliquant sur un bouton et en parcourant vos galeries. Étant désormais intégrées comme actions de premier ordre, les vidéos et photos partagées de cette manière sont qualifiées de médias « natifs ».

De nombreux opérateurs de requête fonctionnent avec ces ressources « natives », notamment `has:videos`, `has:images` et `has:media`. Ils ne correspondent qu’au contenu média partagé via les fonctionnalités de X. Pour faire correspondre d’autres médias hébergés en dehors de la plateforme X, vous devrez utiliser des opérateurs qui s’appuient sur les métadonnées d’URL.

Ainsi, avant d’examiner les détails des produits Historical PowerTrack et Recherche sur l’archive complète, faisons un tour de l’évolution de X, en tant que produit et plateforme, au fil du temps.

**Chronologie de X**

Vous trouverez ci-dessous une _chronologie_ sélective de X. La plupart de ces mises à jour de X ont, d’une manière ou d’une autre, profondément affecté le comportement des utilisateurs, le contenu JSON des Posts, les opérateurs de requête, ou les trois. En considérant X comme une plateforme d’API, les événements suivants ont affecté d’une manière ou d’une autre les charges utiles JSON utilisées pour encoder les Posts. En retour, ces détails JSON influencent la manière dont les API historiques de X les font correspondre.

Notez que cette liste chronologique est généralement précise mais non exhaustive.

<div id="native-media">
  #### 2006
</div>

- Octobre
  - @replies devient une convention.
  - Les $cashtags apparaissent pour la première fois, mais leur utilisation pour mentionner des symboles boursiers ne devient courante qu’au début de 2009. Les $cashtags deviennent des liens cliquables/recherchables en juin 2012.
- Novembre - Introduction des favoris.

<div id="2006">
  #### 2007
</div>

- Janvier - les @replies deviennent un objet de première classe avec un bouton Répondre dans l’UI et des métadonnées `in_reply_to`.
- Avril - les Retweets deviennent une convention.
- Août - les #hashtags s’imposent comme principal outil de recherche et d’organisation des Posts.

<div id="2007">
  #### 2009
</div>

- Février - Les $cashtags deviennent une convention courante pour discuter des symboles boursiers.
- Mai - La version « bêta » du Retweet est introduite avec « Via @ » ajouté en préfixe au corps du Post.
- Juin - Les comptes vérifiés sont introduits.

**Conseils de filtrage**

Connaître la chronologie de X indiquant quand et comment de nouvelles fonctionnalités ont été ajoutées peut vous aider à créer des requêtes plus efficaces. Ici, une requête désigne un _filtre_ ou une _règle_ appliqué par les API historiques de X à l’archive des Posts, à l’aide des opérateurs PowerTrack pour faire correspondre le JSON des Posts. Par exemple, l’opérateur `lang:` est utilisé pour faire correspondre des Posts dans une langue donnée. X propose un service de classification des langues (plus de 50 langues prises en charge) et les API X exposent ces métadonnées dans le JSON généré pour chaque Post. Ainsi, si un Post est rédigé en espagnol, l’attribut JSON « lang » est défini sur « es ». Donc, si vous construisez un filtre avec la clause `lang:es`, il ne fera correspondre que des messages classés comme espagnols.

Les informations de la chronologie peuvent également aider à mieux interpréter les données de Post reçues. Supposons que vous étudiiez le partage de contenu concernant les Jeux olympiques d’été de 2008 et 2012. Si vous appliquiez uniquement l’opérateur `is:retweet` pour faire correspondre les Retweets, aucune donnée ne correspondrait en 2008. En revanche, pour 2012, il y aurait probablement des millions de Retweets. De cela, vous pourriez conclure à tort qu’en 2008 les Retweets n’étaient pas une pratique des utilisateurs, ou que simplement personne n’a retweeté au sujet de ces Jeux olympiques. Puisque les Retweets sont devenus un objet de première classe en 2009, vous devez ajouter une clause de règle `”RT @”` pour aider à les identifier en 2008.

Les Retweets et la classification de la langue des Posts sont deux exemples d’attributs de Post avec une longue histoire et de nombreux détails produit. Ci-dessous, nous aborderons plus en détail ces classes d’attributs et d’autres importantes pour faire correspondre et comprendre les données X.

<div id="2017">
  #### Reconnaître les faux négatifs
</div>

Lorsqu’il s’agit d’écrire des filtres, un point important à retenir est que tous les opérateurs portant sur des métadonnées ont des dates d’« apparition ». Si vous créez un filtre avec un opérateur qui agit sur des métadonnées introduites après la publication du Post, vous obtiendrez un faux négatif. Par exemple, supposons que vous vous intéressiez à tous les Posts qui mentionnent « snow » et partagent une vidéo. Si vous créez une règle avec l’opérateur `has:videos`, qui correspond aux Posts contenant des vidéos natives, cette clause ne fera correspondre aucun Post antérieur à 2015.

Cependant, le partage de vidéos était courant sur X bien avant 2015. Avant cette date, les utilisateurs partageaient des liens vers des vidéos hébergées ailleurs, mais en 2015, X a intégré de nouvelles fonctionnalités de partage de vidéos directement dans la plateforme. Pour retrouver ces anciens Posts d’intérêt, vous incluriez une clause de règle telle que `url:"youtube.com"`.

Notez qu’avec les API de recherche, il existe des cas où des métadonnées ont été « rétro-complétées » lors de la reconstruction de l’index. Un bon exemple concerne les $cashtags, devenus largement utilisés pour discuter des symboles boursiers en 2009. Après l’introduction de l’opérateur $cashtag en 2015, l’index de recherche a été reconstruit et, au passage, l’entité symbole a été extraite de tous les corps de Post, y compris au début de 2006 lorsque le caractère « $ » était principalement utilisé comme argot : « I hope it $now$ $oon! ».

<div id="recognizing-false-negatives">
  #### Identifier et filtrer les attributs de Post importants pour votre cas d’usage
</div>

Certaines métadonnées, comme les identifiants numériques des comptes X, existent depuis le premier jour (et constituent un exemple de métadonnées de compte qui ne changent jamais). D’autres métadonnées n’ont été introduites que bien après le lancement de X en 2006. Parmi les nouveaux types de métadonnées introduits figurent les métadonnées liées aux Retweets, les emplacements des Posts, les titres et descriptions d’URL, ainsi que les médias « natifs ». Vous trouverez ci-dessous certains des types d’attributs de Post les plus courants qui ont été profondément affectés par ces mises à jour de la plateforme X.

Le comportement de filtrage/de correspondance pour ceux-ci dépend, dans la plupart des cas, de l’API de posts historique utilisée. Pour vous aider à déterminer quel produit convient le mieux à vos recherches et à votre cas d’usage, les détails d’attributs fournis ci-dessous incluent des informations produit de haut niveau.

<div id="identifying-and-filtering-on-post-attributes-important-to-your-use-case">
  #### Profils X
</div>

Puisqu’au cœur de X se trouve un canal de communication mondial en temps réel, les recherches fondées sur des données de Posts mettent souvent l’accent sur qui communique. Il est souvent utile de savoir où un utilisateur X se dit « chez lui ». Savoir qu’une bio de compte mentionne des centres d’intérêt et des loisirs peut souvent vous orienter vers des Posts pertinents. Il est très courant de vouloir capter les Posts provenant de comptes d’intérêt. Les attributs de profil sont essentiels pour tous ces cas d’usage.

Chaque compte sur X possède un profil qui inclut des métadonnées telles que le @handle X, le nom d’affichage, une courte bio, le lieu d’origine (texte libre saisi par l’utilisateur), le nombre d’abonnés et bien d’autres. Certains attributs ne changent jamais, comme l’identifiant utilisateur numérique et la date de création du compte. D’autres varient généralement au jour le jour, de semaine en semaine ou de mois en mois, comme le nombre de Posts publiés et le nombre de comptes suivis et d’abonnés. D’autres attributs du compte peuvent également changer à tout moment, mais ont tendance à évoluer moins fréquemment : nom d’affichage, lieu d’origine et bio.

La charge utile JSON de chaque Post inclut des métadonnées de _profil de compte_ pour l’auteur du Post. S’il s’agit d’un Retweet, elle inclut également les métadonnées de profil du compte qui a publié le Post original.

La mutabilité des métadonnées de profil d’un Post dépend entièrement du produit historique utilisé. Les API de recherche fournissent des Posts historiques avec les paramètres de profil _tels qu’ils sont au moment de la récupération_. Pour Historical PowerTrack, le profil est _tel qu’il était au moment où le Post a été publié_, sauf pour les données antérieures à 2011. Pour les Posts antérieurs à 2011, les métadonnées de profil reflètent le profil tel qu’il était en septembre 2011.

<div id="x-profiles">
  #### Post original et Retweets
</div>

Les Retweets sont un autre exemple de conventions initiées par les utilisateurs devenues des objets officiels. Le retweet est apparu comme un moyen de « transférer » du contenu à d’autres. Il a commencé comme un processus manuel consistant à copier-coller un Post et à le faire précéder du motif « RT @ ». Ce processus a ensuite été automatisé via un nouveau bouton Retweet, avec de nouvelles métadonnées JSON. Le Retweet « officiel » est né et l’action de retweeter est devenue un événement de Post de premier ordre. Avec le nouveau bouton Retweet, de nouvelles métadonnées ont été introduites, telles que la charge utile complète du Post d’origine.

Le fait qu’un Post soit original ou partagé est un critère de filtrage courant. Dans certains cas, seul le contenu original est nécessaire. Dans d’autres, l’engagement sur le Post est primordial, et les Retweets sont alors essentiels. L’opérateur PowerTrack `is:retweet` permet d’inclure ou d’exclure les Retweets. Si vous récupérez des données antérieures à août 2009, vous devez prévoir deux stratégies pour la détection (ou non) des Retweets. Avant août 2009, il faut vérifier le texte du Post lui-même, en utilisant une correspondance de phrase exacte, pour identifier le motif « RT @ ». Pour les périodes postérieures à août 2009, l’opérateur `is:retweet` est disponible.

<div id="original-post-and-retweets">
  #### Classifications de langue des posts
</div>

La langue dans laquelle un post est rédigé suscite un intérêt fréquent. La langue d’un post peut aider à inférer sa localisation et, souvent, une seule langue spécifique suffit pour l’analyse ou l’affichage. (Les profils X disposent également d’un paramètre de langue préférée.)

Pour filtrer selon la classification de langue d’un post, les produits historiques de X ([Search API](https://developer.x.com/en/docs/x-api/v1/tweets/search/overview/enterprise) et Historical PowerTrack) diffèrent sensiblement. Lors de la création de l’archive Search, tous les posts ont été rétro-remplis avec la classification de langue de X. Par conséquent, l’opérateur `lang:` est disponible pour l’ensemble de l’archive des posts. Avec Historical PowerTrack, les métadonnées de classification de langue de X sont disponibles dans l’archive à partir du 26 mars 2013. 

<div id="post-language-classifications">
  #### Géoréférencement des posts
</div>

Pouvoir indiquer où un post a été publié (c’est-à-dire le géoréférencer) est important pour de nombreux cas d’usage. Il existe trois méthodes principales pour géoréférencer des posts :

- Références géographiques dans le contenu d’un post
- Posts géolocalisés par l’utilisateur
- Emplacement « domicile » du profil du compte défini par l’utilisateur

##### Références géographiques dans un message de Post

L’appariement sur des références géographiques dans le message de Post, bien que souvent la méthode la plus difficile car dépendante des connaissances locales, est possible sur l’ensemble de l’archive de Posts. Voici un exemple de correspondance géoréférencée de 2006 pour la région de San Francisco basé sur un filtre « golden gate » :

https://x.com/biz/statuses/28311

<div id="geographical-references-in-a-post-message">
  ##### Posts géolocalisés par l’utilisateur
</div>

En novembre 2009, X a introduit son API de géolocalisation des Posts, qui permettait d’associer aux Posts une position exacte. En juin 2010, X a lancé X Places, qui représente une zone géographique à l’échelle d’un lieu, d’un quartier ou d’une ville. Environ 1 à 2 % des Posts sont géolocalisés selon l’une ou l’autre méthode.

L’historique de géolocalisation disponible dépend de l’API historique que vous utilisez. Avec les API de recherche, la possibilité de commencer à faire correspondre des Posts à l’aide de certains opérateurs Geo a débuté en mars 2010, et avec d’autres en février 2015. Si vous utilisez Historical PowerTrack, la géoréférence commence le 1er septembre 2011. Lors de la constitution de l’archive Historical PowerTrack, toutes les géolocalisations antérieures à cette date n’ont pas été incluses.

<div id="posts-geo-tagged-by-the-user">
  ##### Lieu « home » du profil de compte défini par un utilisateur
</div>

Tous les utilisateurs de X peuvent définir le Lieu de leur profil, indiquant leur lieu de résidence. Des millions d’utilisateurs fournissent cette information, ce qui augmente considérablement la quantité de géodonnées dans le X Firehose. Ces métadonnées de localisation sont une chaîne libre, non normalisée et générée par l’utilisateur. Environ 30 % des comptes disposent de métadonnées Profile Geo pouvant être résolues au niveau du pays.

Comme pour la géolocalisation des posts, les méthodes d’appariement et les périodes disponibles dépendent de l’API historique utilisée. Historical PowerTrack permet aux utilisateurs d’effectuer leur propre appariement personnalisé sur ces chaînes libres. Pour faciliter ce processus, X propose également un [Profile Geo Enrichment](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo) qui effectue le géocodage lorsque c’est possible, en fournissant des métadonnées normalisées et des opérateurs correspondants. Les opérateurs Profile Geo sont disponibles à la fois dans Historical PowerTrack et dans les API de recherche. Avec Historical PowerTrack, ces métadonnées Profile Geo sont disponibles à partir de juin 2014. Avec les API de recherche, ces métadonnées sont disponibles à partir de février 2015.

#### Liens partagés et médias

Le partage de liens vers des pages web, de photos et de vidéos a toujours été un cas d’usage fondamental sur X. Au début, toutes ces actions consistaient à inclure une URL directement dans le message du Post. En 2011, X a intégré le partage de photos directement dans son interface utilisateur. En 2016, les vidéos natives ont été ajoutées.

Compte tenu de cet historique, il existe une variété d’opérateurs de filtrage permettant de cibler ce type de contenu. Certains opérateurs détectent si des Posts contiennent des liens partagés, des photos ou des vidéos. De plus, comme la plupart des URL partagées sur X sont raccourcies pour consommer moins de caractères dans un Post (p. ex. via des services comme bitly ou tinyurl), X fournit des enrichissements de données qui génèrent une URL complète et développée sur laquelle on peut faire correspondre. Par exemple, si vous souhaitez cibler des Posts contenant des liens traitant de X et des systèmes d’alerte précoce, un filtre faisant référence à « severe weather communication » correspondrait à un Post contenant cette URL : [http://bit.ly/1XV1tG4](http://bit.ly/1XV1tG4).

En mars 2012, l’[enrichissement d’URL développée](/fr/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls) a été introduit. Avant cette date, les charges utiles des Posts incluaient uniquement l’URL telle que fournie par l’utilisateur. Ainsi, si l’utilisateur incluait une URL raccourcie, il pouvait être difficile de faire correspondre les URL (développées) d’intérêt. Avec Historical PowerTrack et les API de recherche, ces métadonnées sont disponibles à partir de mars 2012.

En juillet 2016, l’enrichissement d’URL améliorée a été introduit. Cette version fournit le titre et la description HTML d’un site web dans la charge utile du Post, ainsi que des opérateurs pour faire des correspondances sur ces champs. Avec Historical PowerTrack, ces métadonnées sont disponibles à partir de juillet 2016. Avec les API de recherche, ces métadonnées commencent à apparaître en décembre 2014.

En septembre 2016, X a introduit les « pièces jointes natives », où un lien partagé en fin de message n’est pas comptabilisé dans la limite de 140 caractères d’un Post. Les deux enrichissements d’URL s’appliquent toujours à ces liens partagés.

Pour d’autres détails spécifiques au produit concernant le filtrage des URL, consultez les articles correspondants.
