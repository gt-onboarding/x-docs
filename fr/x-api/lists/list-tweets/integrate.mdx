<div id="authentication">
  #### Authentification
</div>

Tous les endpoints de l’API X v2 exigent d’authentifier vos requêtes avec un ensemble d’identifiants, également appelés clés et jetons. Vous pouvez utiliser le contexte utilisateur OAuth 1.0a, App only, ou le code d’autorisation OAuth 2.0 avec PKCE pour authentifier vos requêtes vers cet endpoint.

[OAuth 1.0a en contexte utilisateur](/fr/resources/fundamentals/authentication#oauth-1-0a-2), ce qui signifie que vous devez utiliser un ensemble de clés d’API et de jetons d’accès utilisateur pour effectuer une requête avec succès. Les jetons d’accès doivent être associés à l’utilisateur pour le compte duquel vous effectuez la requête. Si vous souhaitez générer un ensemble de jetons d’accès pour un autre utilisateur, celui-ci doit autoriser votre App via le [flux OAuth à 3 acteurs](/fr/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow).

Veuillez noter qu’OAuth 1.0a peut être difficile à utiliser. Si vous n’êtes pas familier avec cette méthode d’authentification, nous vous recommandons d’utiliser une [bibliothèque](/fr/x-api/tools-and-libraries/overview), un outil comme Postman, ou d’utiliser OAuth 2.0 ou App only pour authentifier vos requêtes.

[Code d’autorisation OAuth 2.0 avec PKCE](/fr/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) permet un meilleur contrôle de la portée d’une application et des flux d’autorisation sur plusieurs appareils. OAuth 2.0 vous permet de choisir des portées fines qui vous confèrent des autorisations précises au nom d’un utilisateur.

Pour activer OAuth 2.0 dans votre App, vous devez l’activer dans les paramètres d’authentification de votre App, accessibles dans la section Paramètres de l’App du portail développeur.

[App only](/fr/resources/fundamentals/authentication#app-only-authentication-and-oauth-2-0-bearer-token) exige simplement de transmettre un [jeton d’accès App only](/fr/resources/fundamentals/authentication#app-only-authentication-and-oauth-2-0-bearer-token) avec votre requête. Vous pouvez soit générer un jeton d’accès App only directement au sein d’une App développeur, soit en générer un à l’aide de l’endpoint [POST oauth2/token](/fr/resources/fundamentals/authentication#post-oauth2-token).

<div id="developer-portal-projects-and-developer-apps">
  #### Portail développeur, Projets et applications développeur
</div>

Pour obtenir un ensemble d’informations d’authentification compatibles avec les endpoints de l’API X v2, vous devez [créer un compte développeur](https://developer.x.com/en/portal/petition/essential/basic-info), configurer un [Projet](/fr/resources/fundamentals/projects) dans ce compte, puis créer une [application développeur](/fr/resources/fundamentals/developer-apps) au sein de ce Projet. Vous pourrez ensuite retrouver vos clés et jetons dans votre application développeur.
 

<div id="rate-limits">
  #### Limites de débit
</div>

Chaque jour, des dizaines de milliers de développeurs envoient des requêtes à l’API X. Pour aider à gérer le volume de ces requêtes, des [limites de débit](https://developer.x.com/x-api/fundamentals/rate-limits) sont appliquées à chaque endpoint, ce qui limite le nombre de requêtes que vous pouvez effectuer au nom de votre application ou d’un utilisateur authentifié.

Cet endpoint est soumis à des limites de débit à la fois au niveau de l’application et au niveau de l’utilisateur. La limite au niveau de l’application signifie que vous, développeur, ne pouvez effectuer qu’un certain nombre de requêtes vers cet endpoint sur une période donnée depuis n’importe quelle application (en utilisant soit la clé API et la clé secrète API, soit le jeton Bearer). La limite au niveau de l’utilisateur signifie que l’utilisateur authentifié pour lequel vous effectuez la requête ne peut effectuer une recherche de posts de liste qu’un certain nombre de fois, et ce, depuis n’importe quelle application développeur.

Le tableau ci-dessous indique les limites de débit pour chaque endpoint.

|     |     |     |
| :--- | :--- | :--- |
| **Endpoint** | **Méthode HTTP** | **Limite de débit** |
| /2/lists/:id/tweets | GET | 900 requêtes par 15 minutes |

Champs et expansions

L’endpoint GET de l’API X v2 permet aux utilisateurs de sélectionner précisément les données à retourner par l’API à l’aide d’outils appelés `fields` et `expansions`. Le paramètre `expansions` vous permet de développer les objets référencés dans la charge utile. Par exemple, la recherche de posts de liste vous permet de récupérer les [expansions](/fr/x-api/fundamentals/expansions) suivantes :

- `author_id`

Le paramètre `fields` vous permet de sélectionner exactement quels [champs](/fr/x-api/fundamentals/fields), au sein des différents objets de données, vous souhaitez recevoir. Cet endpoint renvoie principalement des objets Post. Par défaut, l’objet Post renvoie les champs `id` et `text`. Pour recevoir des champs supplémentaires tels que `tweet.created_at` ou `tweet.lang`, vous devez les demander explicitement à l’aide d’un paramètre fields.

Nous avons ajouté à notre [dictionnaire de données de l’API X v2](/fr/x-api/fundamentals/data-dictionary) un guide sur l’utilisation conjointe des [fields et expansions](/fr/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions).

Le tableau ci-dessous présente les champs et expansions disponibles pour l’endpoint de recherche :

|     |     |     |
| :--- | :--- | :--- |
| **Endpoint** | **Champs** | **Expansions** |
| /2/lists/:id/tweets | `tweet.fields`<br /><br />`user.fields` | `author_id` |

Pagination

La recherche de posts de liste peut renvoyer un volume important de données. Afin de garantir des résultats cohérents et performants à tout moment, nous utilisons la pagination. La pagination est une fonctionnalité des endpoints de l’API X v2 qui renvoient plus de résultats que ce qu’une seule réponse ne peut contenir. Lorsque cela se produit, les données sont renvoyées sous la forme d’une série de « pages ». En savoir plus sur la [pagination des résultats](/fr/x-api/fundamentals/pagination).
