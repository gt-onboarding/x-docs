<div id="manage-mutes-standard-v11-compared-to-x-api-v2">
  ### Gérer les sourdines : Standard v1.1 comparé à l’API X v2
</div>

Si vous avez travaillé avec les endpoints standard v1.1 [POST mutes/users/create](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/mute-block-report-users/api-reference/post-mutes-users-create) et [POST mutes/users/destroy](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/mute-block-report-users/api-reference/post-mutes-users-destroy), ce guide a pour but de vous aider à comprendre les similitudes et les différences entre les endpoints de gestion des sourdines du standard v1.1 et de l’API X v2.

- **Similarités**
  - Contexte utilisateur OAuth 1.0a
- **Différences**
  - URL des endpoints
  - Exigences pour les applications et les projets
  - Méthodes HTTP
  - Paramètres de requête
     

<div id="similarities">
  #### Similarités
</div>

**Méthode d’authentification OAuth 1.0a en contexte utilisateur**

Les deux versions de l’endpoint prennent en charge [OAuth 1.0a en contexte utilisateur](https://developer.x.com/content/developer-twitter/resources/fundamentals/authentication). Par conséquent, si vous utilisiez auparavant l’un des endpoints standard v1.1 pour gérer les mises en sourdine, vous pouvez continuer à utiliser la même méthode d’authentification en migrant vers la version de l’API X v2.

<div id="differences">
  #### Différences
</div>

**URLs d’endpoint**

- Endpoints Standard v1.1 :
  - POST https://api.x.com/1.1/mutes/users/create.json
    (mettre un utilisateur en sourdine)
  - POST https://api.x.com/1.1/mutes/users/destroy.json
    (retirer un utilisateur de la sourdine)
- Endpoint API X v2 :
  - POST https://api.x.com/2/users/:id/muting
    (mettre un utilisateur en sourdine)
  - DELETE https://api.x.com/2/users/:source\_user\_id/muting/:target\_user\_id
    (retirer un utilisateur de la sourdine) 

**Exigences relatives aux applications et aux projets**

Les endpoints de l’API X v2 exigent que vous utilisiez des identifiants issus d’une [application développeur](/fr/resources/fundamentals/developer-apps) associée à un [projet](/fr/resources/fundamentals/projects) lors de l’authentification de vos requêtes. Tous les endpoints de l’API X v1.1 peuvent utiliser des identifiants provenant d’applications autonomes ou d’applications associées à un projet.

**Paramètres de requête**

Les paramètres de requête Standard v1.1 suivants ont des équivalents dans l’API X v2 :

| Standard v1.1 | API X v2 |
| :--- | :--- |
| user\_id | target\_user\_id |
| screen\_name | Aucun équivalent |

Veuillez noter que les paramètres Standard v1.1 sont passés en tant que paramètres de requête, tandis que les paramètres de l’API X v2 sont passés en tant que paramètres de corps (pour l’endpoint POST) ou de chemin (pour l’endpoint DELETE).

Par ailleurs, l’id de l’utilisateur qui met en sourdine un utilisateur cible n’est pas requis avec les endpoints Standard v1.1, puisque les jetons d’accès transmis avec [OAuth 1.0a User Context](/fr/resources/fundamentals/authentication) permettaient d’identifier quel utilisateur initiait l’action de mise en sourdine/suppression de la sourdine.
