<div id="overview">
  ## Visão geral
</div>

Se você já tem Apps, pode visualizá-los, editá-los ou excluí-los na [página de App do portal do desenvolvedor](https://developer.x.com/en/portal/projects-and-apps).

O acesso à API do X e à API de Anúncios do X requer um conjunto de credenciais de [autenticação](/pt-BR/resources/fundamentals/authentication), também conhecidas como chaves e tokens, que devem ser enviadas com cada requisição. Essas credenciais podem assumir diferentes formas, dependendo do tipo de autenticação exigido pelo endpoint específico que você está usando.

A seguir estão as diferentes credenciais que você pode gerar no seu App e como usá-las:

- **[API Key and Secret](/pt-BR/resources/fundamentals/authentication#api-key-and-secret):** Essencialmente, o nome de usuário e a senha do seu App. Você usará essas credenciais para autenticar requisições que exigem [OAuth 1.0a User Context](/pt-BR/resources/fundamentals/authentication#oauth-1-0a-2) ou para gerar outros tokens, como tokens de acesso de usuário ou App Access Token.

- **[Access Token and Secret](/pt-BR/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow):** Em geral, os Access Tokens representam o usuário em nome de quem você está fazendo a requisição. Os que você pode gerar pelo portal do desenvolvedor representam o usuário proprietário do App. Você usará esses tokens para autenticar requisições que exigem [OAuth 1.0a User Context](/pt-BR/resources/fundamentals/authentication#oauth-1-0a-2). Se quiser fazer requisições em nome de outro usuário, será necessário usar o fluxo OAuth de 3 etapas para que ele autorize você.

- **[Client ID and Client Secret](/pt-BR/resources/fundamentals/authentication#oauth-2-0):** Essas credenciais são usadas para obter um Access Token de usuário com autenticação OAuth 2.0. Assim como no OAuth 1.0a, os user Access Tokens são usados para autenticar requisições que fornecem informações privadas da conta do usuário ou executam ações em nome de outra conta, porém com escopos granulares para maior controle sobre quais acessos o aplicativo cliente tem ao usuário.

- **[App only Access Token](/pt-BR/resources/fundamentals/authentication#bearer-token-also-known-as-app-only):** Você usará esse token ao fazer requisições a endpoints que respondem com informações publicamente disponíveis no X.

Além de gerar as chaves e tokens necessários para fazer requisições à API do X, você também poderá definir [permissões](/pt-BR/resources/fundamentals/developer-apps#app-permissions) de acesso, documentar o caso de uso ou propósito do App, definir uma [URL de callback](/pt-BR/resources/fundamentals/developer-apps#callback-urls) e modificar outras configurações relacionadas ao seu ambiente de desenvolvimento do App no [painel de gerenciamento](https://developer.x.com/en/portal/projects-and-apps).

<div id="apps-and-projects">
  ### Apps e Projetos
</div>

Você pode usar Apps e [Projetos](/pt-BR/resources/fundamentals/projects) para organizar seu trabalho com a X Developer Platform por caso de uso. Cada Projeto pode incluir um App com o plano Free da API do X, até dois Apps com o plano Basic e até três Apps com o plano Pro.

Se você quiser acessar os novos endpoints da [API do X v2](/pt-BR/x-api/introduction), será necessário usar chaves e tokens de um App associado a um Projeto.

Se você tiver Apps criados antes do lançamento de Projetos, eles ficarão visíveis na seção intitulada "Standalone Apps". Standalone Apps são Apps fora da estrutura de Projetos. Se você anexar um Standalone App a um Projeto, ele passará a poder fazer solicitações aos endpoints v2.

<div id="developer-portal-dashboard">
  ### Painel do portal do desenvolvedor
</div>

Você pode [visitar o painel](https://developer.x.com/content/developer-twitter/en/apps) para gerenciar os Apps associados à sua conta. Para saber mais, consulte nossa página de documentação sobre o [portal do desenvolvedor](/pt-BR/resources/fundamentals/developer-portal). O painel permite que desenvolvedores executem, de forma rápida e fácil, as seguintes tarefas:

- Visualizar seus Apps independentes existentes e seus respectivos IDs de App.
- Criar um novo Projeto, App ou App independente.
- Excluir um Projeto ou App não utilizado.
- Revisar ou atualizar as configurações de um App específico, incluindo nome, descrição, site, URL de callback e [permissões](/pt-BR/resources/fundamentals/developer-apps#app-permissions).
- Regenerar credenciais específicas do App, como API Key e Secret, App Access Token e os Access Tokens de usuário do proprietário do App.

<div id="signing-up-for-access">
  ### Inscrevendo-se para obter acesso
</div>

Se você já tem Apps do X, pode visualizar e editar suas Apps pelo [painel de Apps](https://developer.x.com/content/developer-twitter/en/portal/projects-and-apps) do X, desde que esteja conectado à sua conta do X em developer.x.com. Observe que você **não** precisará criar uma conta para gerenciar quaisquer Apps que foram criadas anteriormente em developer.x.com.

Se você precisar criar uma nova App do X, será necessário [se inscrever para uma conta de desenvolvedor](https://developer.x.com/en/portal/petition/essential/basic-info). Se você for membro de uma [conta de equipe](/pt-BR/resources/fundamentals/developer-portal#team-management), deverá ter a função de administrador para criar uma nova App.

<div id="automated-account-labeling-for-bots">
  ### Rotulagem de Conta Automatizada para bots
</div>

Você pode adicionar um rótulo de Conta Automatizada às suas contas de bot para informar aos usuários no X que seu bot é uma conta automatizada. Esses bots executam ações programadas por meio da API do X. Ao adicionar um rótulo de Conta Automatizada ao seu bot, você gera confiança com seu público, legitima sua conta e se diferencia de bots de spam. Isso ajuda as pessoas no X a entenderem melhor o propósito da sua conta ao interagir com seu bot.

Para adicionar um rótulo à sua conta de bot, siga estas etapas:

1. Vá para as configurações da sua conta
2. Selecione "Sua conta"
3. Selecione "Automação"
4. Selecione "Gerenciamento de conta"
5. Em seguida, selecione a conta do X que executa seu bot
6. Digite sua senha para entrar
7. Por fim, você deverá ver a confirmação de que o rótulo foi aplicado à sua conta.

  ## Gerenciamento de apps
</div>

<div id="introduction">
  ### Introdução
</div>

O [dashboard do X App](https://developer.x.com/en/portal/projects-and-apps) permite que desenvolvedores executem, de forma rápida e simples, as seguintes tarefas:

- Visualizar seus Apps e [Projetos](/pt-BR/resources/fundamentals/projects) existentes e seus respectivos IDs de App.
- Criar um novo Projeto ou um App independente.
- Excluir um Projeto, um App ou um App independente.
- Abrir as configurações de um App específico clicando nele. Nas configurações, é possível ver os detalhes do App, chaves e tokens, e permissões.
- Atualizar as configurações de autenticação de usuário do seu App para usar OAuth 1.0a ou OAuth 2.0.

<Note>
  **Observação:**

  Todas as chaves e tokens de App não estão mais visíveis no Portal do Desenvolvedor e devem ser salvos com segurança assim que gerados. Recomendamos usar um gerenciador de senhas para armazenar suas chaves e tokens.

  Você pode revelar uma dica da chave da API para ajudá-lo a corresponder sua credencial ao App correspondente.
</Note>

<div id="app-settings">
  ### Configurações do app
</div>

<div id="app-details">
  #### Detalhes do App
</div>

Aqui você pode editar o ícone do App, o nome do App, a descrição do App, a URL do seu site, [URLs de callback/URIs de redirecionamento](/pt-BR/resources/fundamentals/developer-apps#callback-urls), a URL dos termos de serviço, a URL da política de privacidade, o nome da organização, a URL da organização e o propósito ou caso de uso do App.

OAuth 2.0 e OAuth 1.0a são métodos de autenticação que permitem que os usuários entrem no seu App com o X. Eles também permitem que seu App faça solicitações específicas em nome de usuários autenticados. Você pode ativar um ou ambos os métodos para o seu App.

É importante manter essas informações atualizadas. O nome do seu App e a URL do site serão exibidos como a fonte nos metadados de quaisquer posts criados programaticamente pelo seu aplicativo. Se você alterar o caso de uso de um App do X, certifique-se de atualizar o caso de uso nessas configurações para garantir a conformidade com os [Termos do Desenvolvedor](https://developer.x.com/content/developer-twitter/en/developer-terms/agreement-and-policy).

Se o seu aplicativo tiver uma etiqueta indicando “suspended”, isso ocorre porque ele está violando um ou mais dos [termos do desenvolvedor](https://developer.x.com/en/developer-terms.html) do X, como nossas [regras de automação](https://help.x.com/en/rules-and-policies/twitter-automation). A equipe de políticas da plataforma para desenvolvedores se comunicará com os desenvolvedores pelo endereço de e-mail configurado na conta do X do proprietário do App; para revisar esse endereço de e-mail, confira suas [configurações da conta do X](https://x.com/settings/your_twitter_data/account). E-mails de notificação sobre suspensões serão enviados para esse endereço com um título semelhante a “Application suspension notice” e conterão informações específicas sobre o que fazer a seguir. Para trabalhar com a equipe do X na resolução de suspensões, verifique seu e-mail para instruções específicas ou use nosso [formulário de ajuda da plataforma](https://help.x.com/forms/platform).

<div id="keys-and-tokens">
  #### Chaves e tokens
</div>

Dentro de um App em um Projeto ou por meio de um App independente, você pode visualizar, regenerar ou revogar os seguintes tokens:

- [API Key (Consumer Key) e API Secret (Consumer Secret)](/pt-BR/resources/fundamentals/authentication#api-key-and-secret)
- [App Access Token](/pt-BR/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)
- [User Access Token e Access Token Secret](/pt-BR/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) -
  O Access Token e o Secret disponíveis no portal do desenvolvedor referem-se ao usuário proprietário do App.

**Atenção** - Se você quiser fazer solicitações em nome de um usuário diferente (ou seja, que não seja o proprietário do App), será necessário usar o [fluxo OAuth 1.0a com 3 etapas](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens) ou o fluxo OAuth 2.0 [Authorization Code com PKCE](/pt-BR/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) para gerar um conjunto de Access Tokens de usuário. Em seguida, use esses tokens específicos do usuário na sua solicitação à API.

<div id="user-authentication-settings">
  #### Configurações de autenticação do usuário
</div>

Você pode definir as configurações de autenticação do seu App como OAuth 1.0a ou OAuth 2.0. OAuth 2.0 pode ser usado apenas com a API do X v2. OAuth 2.0 permite escolher escopos específicos e granulares que concedem permissões detalhadas em nome do usuário. OAuth 1.0a pode ser usado com a API do X v1.1 e v2 e utiliza uma autorização mais ampla com escopos gerais.

<div id="oauth-10a-application-user-permissions">
  #### Permissões de aplicativo-usuário do OAuth 1.0a
</div>

Se você é o proprietário do App, pode ajustar as permissões do App (somente leitura; leitura e gravação; ou leitura, gravação e mensagens diretas). Isso controla a quais recursos e eventos você terá acesso por meio das APIs do X (observação: alguns recursos exigem permissão adicional diretamente do X).

Você também pode ativar ou desativar, nesta página, a capacidade dos seus Apps de solicitar endereços de e-mail dos usuários (isso requer que uma URL dos Termos de Serviço e uma URL da Política de Privacidade estejam informadas na página "Detalhes do App").

<div id="oauth-20-type-of-app">
  #### Tipo de App OAuth 2.0
</div>

Se você selecionar OAuth 2.0 como método de autenticação de usuário, será necessário escolher o tipo de App que está criando. As opções são Native App, Single Page App, Web App, Automated App ou bot. Native App e Single Page App são clientes públicos, e Web App e Automated App ou bot são clientes confidenciais.

Clientes confidenciais se autenticam com segurança no servidor de autorização e mantêm seu client secret protegido. Clientes públicos são aplicativos que geralmente rodam em um navegador ou em um dispositivo móvel e não conseguem usar client secrets. Se você selecionar um tipo de App que seja um cliente confidencial, um client secret será fornecido.

<div id="app-permissions">
  ## Permissões do aplicativo
</div>

<div id="oauth-10a-app-permissions">
  ### Permissões do app OAuth 1.0a
</div>

As permissões do app descrevem o nível de acesso para a autenticação aplicativo-usuário do OAuth 1.0a. As permissões do app são configuradas por aplicativo nas configurações do seu [X App](/pt-BR/resources/fundamentals/developer-apps).

Há três níveis de permissão disponíveis:

1. Somente leitura
2. Leitura e escrita
3. Leitura, escrita e acesso a Mensagens Diretas

Existe uma permissão adicional para solicitar a visibilidade do endereço de e-mail de um usuário — ela pode ser combinada com qualquer um dos três níveis listados acima.

Se um nível de permissão for alterado, quaisquer tokens de usuário já emitidos para esse X App devem ser descartados e os usuários devem reautorizar o app para que o token herde as permissões atualizadas.

Uma boa prática é solicitar apenas o nível mínimo de acesso aos dados da conta de um usuário necessário para o funcionamento do aplicativo ou serviço.

<div id="read-only">
  ### Somente leitura
</div>

Esse nível de permissão concede acesso de leitura aos recursos do X, incluindo (por exemplo) os Posts de um usuário, a linha do tempo inicial e as informações de perfil. Ele não permite acesso à leitura das Mensagens Diretas de um usuário e não permite atualizar nenhum elemento ou objeto.

<div id="read-and-write">
  ### Leitura e escrita
</div>

Este nível de permissão permite acesso de leitura e escrita aos recursos do X. Além do acesso de leitura, ele também permite publicar posts, seguir usuários e atualizar elementos das informações de perfil de um usuário. Também permite ocultar respostas em nome do usuário autenticado. Este nível de permissão não concede acesso algum a Mensagens Diretas (incluindo leitura, escrita ou exclusão).

<div id="read-write-and-access-direct-messages">
  ### Ler, escrever e acessar Mensagens Diretas
</div>

Este nível de permissão inclui tudo o que foi mencionado acima e acrescenta a capacidade de ler, escrever e excluir Mensagens Diretas em nome de um usuário.

- [POST /2/dm\_conversations/:dm\_conversation\_id/messages](/pt-BR/x-api/direct-messages/send-a-new-message-to-a-dm-conversation)
- [POST /2/dm\_conversations/](/pt-BR/x-api/direct-messages/create-a-new-dm-conversation)
- [POST /2/dm\_conversations/with/:participant\_id/messages](/pt-BR/x-api/direct-messages/send-a-new-message-to-a-user)
- [GET /2/dm\_conversations/with/:user\_id/dm\_events](/pt-BR/x-api/direct-messages/get-dm-events-for-a-dm-conversation)
- [GET /2/dm\_conversations/:dm\_conversation\_id/dm\_events](/pt-BR/x-api/direct-messages/get-dm-events-for-a-dm-conversation-1)
- [GET /2/dm\_events](/pt-BR/x-api/direct-messages/get-recent-dm-events)

<div id="determining-permissions">
  ### Determinando permissões
</div>

Todas as solicitações autenticadas à API retornam o cabeçalho `x-access-level` na resposta HTTP. O valor desse cabeçalho indica o nível de permissão atualmente em uso. Os valores possíveis são read, read-write e read-write-directmessages.

<div id="callback-urls">
  ## URLs de callback
</div>

Os métodos de autenticação OAuth 1.0a User Context e OAuth 2.0 Authorization Code com PKCE permitem que desenvolvedores façam solicitações em nome de diferentes usuários do X que passaram por um fluxo específico de autenticação. Atualmente, há dois fluxos que você pode usar para permitir que os usuários autorizem seu aplicativo:

- [Fluxo de código de autorização do OAuth 2.0 com PKCE](/pt-BR/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2)
- [Fluxo OAuth 1.0a de 3 etapas](/pt-BR/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) (e, separadamente, o [fluxo Entrar com o X](/pt-BR/resources/fundamentals/authentication#log-in-with-x))

Conforme os usuários passam por esses fluxos, eles precisam de uma página da web ou de um local para o qual serão redirecionados após concluírem o login com sucesso e concederem autorização ao App do desenvolvedor. Essa página ou local de destino é chamada de URL de callback.

Ao configurar esses fluxos para que seus usuários em potencial os utilizem, os desenvolvedores devem incluir uma URL de callback em suas solicitações aos endpoints de autenticação que compõem os fluxos mencionados anteriormente. Por exemplo, desenvolvedores que usam OAuth 1.0a User Context devem enviar o parâmetro `callback_url` ao fazer uma solicitação ao endpoint [GET oauth/request\_token](/pt-BR/resources/fundamentals/authentication#post-oauth-request-token). Da mesma forma, desenvolvedores que usam OAuth 2.0 Authorization Code com PKCE devem enviar o parâmetro `redirect_uri` com sua solicitação ao endpoint GET oauth2/authorize.

Além de usar esses parâmetros, o desenvolvedor também deve garantir que a URL de callback tenha sido adicionada à allowlist de URLs de callback do [App](/pt-BR/resources/fundamentals/developer-apps), que pode ser encontrada na página de configurações do App no [portal do desenvolvedor](/pt-BR/resources/fundamentals/developer-portal).

Se tudo estiver configurado corretamente, os usuários serão redirecionados para a URL de callback após entrarem no X com sucesso como parte desses fluxos.

<div id="things-to-keep-in-mind">
  ### Coisas a considerar
</div>

Ao configurar suas URLs de callback, há alguns pontos que você deve levar em conta:

**Codifique em HTTP seus parâmetros de consulta**

Como você está passando a URL de callback como um parâmetro de consulta por meio dos parâmetros `callback_url` ou `redirect_uri`, certifique-se de codificar a URL em HTTP.

**Limites de URL de callback**

Há um limite fixo de 10 URLs de callback no painel do X Apps. Sua URL de callback deve sempre corresponder exatamente entre a URL de callback na lista de permissões que você adiciona ao painel de Apps e o parâmetro que você inclui no fluxo de autorização.

Se você quiser incluir dados específicos da requisição na URL de callback, use o parâmetro `state` para armazenar dados que serão incluídos após o redirecionamento do usuário. Você pode codificar os dados no próprio parâmetro `state` ou usar o parâmetro como um ID de sessão para armazenar o estado no servidor.

**Não use localhost como URL de callback**
Em vez de usar localhost, use um host personalizado localmente ou http(s)://127.0.0.1

**URLs de protocolo personalizadas**
Se você quiser aproveitar o deep linking em dispositivos móveis, utilize URLs de protocolo personalizadas com partes de caminho e domínio, como twitter://callback/path. No entanto, temos uma lista de protocolos não permitidos que você deve evitar. Você pode revisar a lista de protocolos não permitidos abaixo.

<div id="error-example">
  ### Exemplo de erro
</div>

Se você usar uma URL de callback que não foi adicionada corretamente às configurações do seu App no portal do desenvolvedor, receberá a seguinte mensagem de erro:

**OAuth 1.0a**

```json HTTP 403 - Forbidden

{
  "errors": [
    {
      "code":415,"message":"Callback URL not approved for this client application. Approved callback URLs can be adjusted in your application settings."
    }
  ]
}
```

OU

```xml
<?xml version="1.0" encoding="UTF-8"?>
<hash>
<error>Callback URL not approved for this client application. Approved callback URLs can be adjusted in your application settings</error>
<request>/oauth/request_token</request>
</hash>
```

**OAuth 2.0**

```json HTTP 400

{
  "error": "invalid_request",
  "error_description": "Value passed for the redirect uri did not match the uri of the authorization code."
}
```

**Solução de problemas**

Se você receber um erro, verifique se a URL de callback usada nas suas solicitações de autenticação está codificada em URL (percent-encoding) e se corresponde exatamente a uma URL de callback que foi adicionada à allowlist do App cujas chaves e tokens você está usando na solicitação.
