<div id="list-lookup-standard-v11-compared-to-x-api-v2">
  ### Consulta de listas: padrão v1.1 em comparação com a API do X v2
</div>

Se você tem trabalhado com os endpoints do padrão v1.1 [GET lists/show](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-show) e [GET lists/ownerships](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-ownerships), o objetivo deste guia é ajudar você a entender as semelhanças e diferenças entre os endpoints de consulta de listas do padrão v1.1 e da API do X v2.

- **Semelhanças**
  - Métodos de autenticação
  - Limites de taxa
- **Diferenças**
  - URLs de endpoint
  - Requisitos de App e Projeto
  - Limites de objetos de dados por requisição
  - Formatos de dados de resposta
  - Parâmetros de requisição

<div id="similarities">
  #### Semelhanças
</div>

**Autenticação**

Ambas as versões do endpoint oferecem suporte a [OAuth 1.0a User Context](/pt-BR/resources/fundamentals/authentication#oauth-1-0a-2) e a [App only](/pt-BR/resources/fundamentals/authentication#app-only-authentication-and-oauth-2-0-bearer-token). Portanto, se você usava anteriormente um dos endpoints padrão v1.1 de consulta de listas, pode continuar usando o mesmo método de autenticação ao migrar para a versão da API do X v2.

Dependendo da biblioteca/pacote de autenticação escolhido, a autenticação App only provavelmente é a forma mais simples de começar e pode ser definida com um cabeçalho de requisição. Para saber como gerar um token de acesso App only, consulte [este guia de App only](/pt-BR/resources/fundamentals/authentication#using-and-generating-an-app-only-bearer-token).

**Limites de taxa**

|     |     |
| :--- | :--- |
| **Padrão v1.1** | **API do X v2** |
| /1.1/lists/show.json<br /><br />75 solicitações por janela de 15 minutos com OAuth 1.0a User Context<br /><br />75 solicitações por janela de 15 minutos com App only | /2/lists/:id<br /><br />75 solicitações por janela de 15 minutos com OAuth 1.0a User Context<br /><br />75 solicitações por janela de 15 minutos com OAuth 2.0 Authorization Code with PKCE |
| /1.1/lists/ownerships.json<br /><br />15 solicitações por janela de 15 minutos com OAuth 1.0a User Context<br /><br />15 solicitações por janela de 15 minutos com App only | /2/users/:id/owned\_lists<br /><br />15 solicitações por janela de 15 minutos com OAuth 1.0a User Context<br /><br />15 solicitações por janela de 15 minutos com OAuth 2.0 Authorization Code with PKCE<br /><br />15 solicitações por janela de 15 minutos com App only |

<div id="differences">
  #### Diferenças
</div>

**URLs de endpoints**

- Endpoints padrão v1.1:
  - GET https://api.x.com/1.1/lists/show.json
    (Consultar uma Lista específica)
  - GET https://api.x.com/1.1/lists/ownerships.json
    (Consultar Listas pertencentes a um usuário específico)
- Endpoint da API do X v2:
  - GET https://api.x.com/2/lists/:id
    (Consultar uma Lista específica)

  - GET https://api.x.com/2/users/:id/owned\_lists
    (Consultar Listas pertencentes a um usuário específico)

**Requisitos de App e Projeto**

Os endpoints da API do X v2 exigem que você use credenciais de um [App de desenvolvedor](/pt-BR/resources/fundamentals/developer-apps) associado a um [Projeto](/pt-BR/resources/fundamentals/projects) ao autenticar suas solicitações. Todos os endpoints da API do X v1.1 podem usar credenciais de Apps independentes ou Apps associados a um Projeto.

**Limites de objetos de dados por solicitação**

O endpoint padrão v1.1 /lists/ownerships permite retornar até 1000 Listas por solicitação. Os novos endpoints v2 permitem retornar até 100 Listas por solicitação. Por padrão, 100 objetos de usuário serão retornados; para alterar o número de resultados, você precisará passar o parâmetro de consulta max\_results= com um número entre 1 e 100; em seguida, você pode passar o next\_token retornado no payload da resposta para o parâmetro de consulta pagination\_token na sua próxima solicitação.

**Formato de dados da resposta**

Uma das maiores diferenças entre as versões de endpoints padrão v1.1 e API do X v2 é como você seleciona quais campos retornam no seu payload.

Para os endpoints padrão, muitos campos da resposta são retornados por padrão e você tem a opção de usar parâmetros para identificar quais campos adicionais ou conjuntos de campos devem retornar no payload.

A versão da API do X v2 retorna apenas os campos id e name da Lista por padrão. Para solicitar quaisquer campos ou objetos adicionais, você precisará usar os parâmetros [fields](/pt-BR/x-api/fundamentals/fields) e [expansions](/pt-BR/x-api/fundamentals/expansions). Quaisquer campos de Lista solicitados desse endpoint retornarão no objeto principal da Lista. Quaisquer objetos e campos de Post ou de usuário expandidos retornarão em um objeto includes dentro da sua resposta. Você poderá então correlacionar quaisquer objetos expandidos ao objeto da Lista correspondendo os IDs presentes tanto no objeto de usuário quanto no objeto de Post expandido.

Aqui estão exemplos de possíveis campos e expansões de Lista:

- created\_at

- follower\_count

- member\_count

- owner\_id

- description

- private

|     |     |
| :--- | :--- |
| **Endpoint** | **Expansão** |
| /2/lists/:id | owner\_id |
| /2/users/:id/owned\_lists | owner\_id |

Recomendamos que você leia mais sobre esses novos parâmetros em seus respectivos guias ou consultando nosso guia sobre [como usar fields e expansions](/pt-BR/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions).

Também preparamos um [guia de migração de formato de dados](/pt-BR/x-api/migrate/data-format-migration) que pode ajudá-lo a mapear campos padrão v1.1 para os campos mais novos da v2. Esse guia também fornecerá o parâmetro específico de expansão e de campo que você precisará passar com sua solicitação v2 para retornar campos específicos.

Além das mudanças em como você solicita determinados campos, a API do X v2 também está introduzindo novos designs JSON para os objetos retornados pelas APIs, incluindo objetos de [Post](/pt-BR/x-api/fundamentals/data-dictionary#tweet) e de [usuário](/pt-BR/x-api/fundamentals/data-dictionary#user).

- No nível raiz do JSON, os endpoints padrão retornam objetos de Post em um array statuses, enquanto a API do X v2 retorna um array data.

- Em vez de se referir a “statuses” Retweeted e Quoted, o JSON da API do X v2 se refere a Retweeted e Quoted Tweets. Muitos campos legados e obsoletos, como contributors e user.translator\_type, estão sendo removidos.

- Em vez de usar tanto favorites (no objeto de Post) quanto favourites (no objeto de usuário), a API do X v2 usa o termo like.

- O X está adotando a convenção de que valores JSON sem valor (por exemplo, null) não são escritos no payload. Atributos de Post e de usuário só são incluídos se tiverem valores não nulos.

**Parâmetros de solicitação**

Os seguintes parâmetros de solicitação padrão v1.1 têm equivalentes na API do X v2:

**Pesquisa de Lista por ID**

|     |     |
| :--- | :--- |
| **Padrão v1.1** | **API do X v2** |
| list\_id | id  |
| slug | Sem equivalente |
| owner\_screen\_name | Sem equivalente |
| owner\_id | Solicitado com o parâmetro expansions/fields |

**Pesquisa de Lista pertencente ao usuário**

|     |     |
| :--- | :--- |
| **Standard v1.1** | **API do X v2** |
| user\_id | id  |
| screen\_name | Sem equivalente |
| count | max\_results |
| cursor | pagination\_token |
