<div id="list-members-lookup-standard-v11-compared-to-x-api-v2">
  ### Consulta de membros de listas: comparação entre o v1.1 Padrão e a API do X v2
</div>

Se você trabalha com os endpoints do v1.1 padrão [GET lists/members](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-members) e [GET lists/memberships](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-memberships), este guia tem como objetivo ajudar você a entender as semelhanças e diferenças entre os endpoints de membros de listas do v1.1 padrão e da API do X v2.

- **Semelhanças**
  - Métodos de autenticação
- **Diferenças**
  - URLs dos endpoints
  - Limites de taxa
  - Requisitos de App e Projeto
  - Limites de objetos de dados por solicitação
  - Formatos de dados de resposta
  - Parâmetros de solicitação

<div id="similarities">
  #### Semelhanças
</div>

**Autenticação**

Ambas as versões do endpoint oferecem suporte a [OAuth 1.0a User Context](/pt-BR/resources/fundamentals/authentication#oauth-1-0a-2) e [App only](/pt-BR/resources/fundamentals/authentication#oauth-2-0). Portanto, se você usava anteriormente um dos endpoints padrão de membros de lista v1.1, pode continuar usando o mesmo método de autenticação ao migrar para a API do X v2.

Dependendo da biblioteca/pacote de autenticação que você usar, a autenticação App only provavelmente é a maneira mais fácil de começar e pode ser configurada com um cabeçalho de requisição simples. Para saber como gerar um token de acesso do tipo App only, consulte [este guia de App only](/pt-BR/resources/fundamentals/authentication#bearer-token-also-known-as-app-only).

<div id="differences">
  #### Diferenças
</div>

**URLs de endpoint**

- Endpoints padrão v1.1:
  - GET https://api.x.com/1.1/lists/members.json
    (Consultar membros de uma Lista especificada)
  - GET https://api.x.com/1.1/lists/memberships.json
    (Consultar as Listas das quais um usuário é membro)
- Endpoint da API do X v2:
  - GET https://api.x.com/2/lists/:id/members
    (Consultar membros de uma Lista especificada)

  - GET https://api.x.com/2/users/:id/list\_memberships
    (Consultar as Listas das quais um usuário é membro)

**Limites de taxa**

|     |     |
| :--- | :--- |
| **Padrão v1.1** | **API do X v2** |
| /1.1/lists/members.json<br /><br />900 solicitações por janela de 15 minutos com OAuth 1.0a (User Context)<br /><br />15 solicitações por janela de 15 minutos com App only | /2/lists/:id/members<br /><br />900 solicitações por janela de 15 minutos com OAuth 1.0a (User Context)<br /><br />900 solicitações por janela de 15 minutos com OAuth 2.0 Authorization Code com PKCE<br /><br />900 solicitações por janela de 15 minutos com App only |
| /1.1/lists/memberships.json<br /><br />15 solicitações por janela de 15 minutos com OAuth 1.0a (User Context)<br /><br />15 solicitações por janela de 15 minutos com App only | /2/users/:id/list\_memberships<br /><br />15 solicitações por janela de 15 minutos com OAuth 1.0a (User Context)<br /><br />15 solicitações por janela de 15 minutos com OAuth 2.0 Authorization Code com PKCE<br /><br />15 solicitações por janela de 15 minutos com App only |

**Requisitos de App e Projeto**

Os endpoints da API do X v2 exigem que você use credenciais de um [developer App](/pt-BR/resources/fundamentals/developer-apps) associado a um [Project](/pt-BR/resources/fundamentals/projects) ao autenticar suas solicitações. Todos os endpoints da API do X v1.1 podem usar credenciais de Apps independentes ou Apps associados a um projeto.

**Limites de objetos de dados por solicitação**

O endpoint padrão v1.1 /1.1/lists/members permite retornar até 5.000 usuários por solicitação. Os novos endpoints v2 permitem retornar até 100 usuários por solicitação. Por padrão, 100 objetos de usuário são retornados; para alterar o número de resultados, você deve passar o parâmetro de consulta max\_results= com um número entre 1 e 100; em seguida, você pode passar o next\_token retornado no payload da resposta para o parâmetro de consulta pagination\_token na sua próxima solicitação.

Além disso, o endpoint /1.1/lists/memberships permite retornar até 1.000 Listas por solicitação. Com a substituição na v2, o endpoint permite até 100 Listas por solicitação. Por padrão, 100 objetos de Lista são retornados; use os parâmetros de consulta max\_results= e pagination\_token da mesma forma que em /1.1/lists/members para alterar o número de resultados.

**Formato de dados da resposta**

Uma das maiores diferenças entre as versões de endpoint padrão v1.1 e API do X v2 é como você seleciona quais campos retornam no payload.

Nos endpoints padrão, muitos campos de resposta são retornados por padrão e, depois, você tem a opção de usar parâmetros para identificar quais campos adicionais ou conjuntos de campos devem retornar no payload.

A versão da API do X v2 /users/:id/list\_memberships retorna os campos id e name da Lista por padrão. Para solicitar campos ou objetos adicionais, você deve usar os parâmetros [fields](/pt-BR/x-api/fundamentals/fields) e [expansions](/pt-BR/x-api/fundamentals/expansions). Quaisquer campos de Lista solicitados nesse endpoint retornarão no objeto de Lista primário. Quaisquer objetos e campos expandidos retornarão em um objeto includes dentro da sua resposta. Você pode então correlacionar os objetos expandidos ao objeto de Lista primário correspondendo os IDs presentes tanto no objeto primário quanto no expandido.

Aqui estão exemplos de possíveis campos e expansões de Lista:

- created\_at

- follower\_count

- member\_count

- owner\_id

- description

- private

|     |     |
| :--- | :--- |
| **Endpoint** | **Expansão** |
| /2/lists/:id/members | pinned\_tweet\_id |
| /2/users/:id/list\_memberships | owner\_id |

Recomendamos ler mais sobre esses novos parâmetros em seus respectivos guias ou consultar nosso guia sobre [como usar fields e expansions](/pt-BR/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions). 

Também preparamos um [guia de migração de formato de dados](/pt-BR/x-api/migrate/data-format-migration) que pode ajudar você a mapear campos padrão da v1.1 para os campos mais recentes da v2. Esse guia também indica a expansão e o parâmetro de campo específicos que você precisará incluir na sua solicitação da v2 para retornar campos específicos. 

Além das mudanças na forma de solicitar determinados campos, a API do X v2 também está introduzindo novos designs de JSON para os objetos retornados pelas APIs, incluindo os objetos de [Post](/pt-BR/x-api/fundamentals/data-dictionary#tweet) e de [usuário](/pt-BR/x-api/fundamentals/data-dictionary#user).

- No nível raiz do JSON, os endpoints padrão retornam objetos de Post em um array statuses, enquanto a API do X v2 retorna um array data. 

- Em vez de se referir a “statuses” Retweeted e Quoted, o JSON da API do X v2 se refere a Tweets Retweeted e Quoted. Muitos campos legados e obsoletos, como contributors e user.translator\_type, estão sendo removidos. 

- Em vez de usar tanto favorites (no objeto Post) quanto favourites (no objeto user), a API do X v2 utiliza o termo like. 

- O X está adotando a convenção de que valores JSON sem valor (por exemplo, null) não são escritos no payload. Atributos de Post e de usuário só são incluídos se tiverem valores não nulos.

**Parâmetros de solicitação**

Os seguintes parâmetros de solicitação padrão da v1.1 têm equivalentes na API do X v2:

**Consulta de membros da Lista**

|     |     |
| :--- | :--- |
| **Padrão v1.1** | **API do X v2** |
| list\_id | id  |
| slug | Sem equivalente |
| owner\_screen\_name | Sem equivalente |
| owner\_id | Sem equivalente |
| count | max\_results |
| cursor | pagination\_token |
| include\_entities | Sem equivalente |
| skip\_status | Sem equivalente |

**Consulta de associação à Lista**

|     |     |
| :--- | :--- |
| **Padrão v1.1** | **API do X v2** |
| user\_id | id  |
| screen\_name | Sem equivalente |
| count | max\_results |
| cursor | pagination\_token |
