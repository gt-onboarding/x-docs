<div id="introduction">
  ## Introdução
</div>

Com o lançamento da versão v2 da API do X, adotamos um novo formato de resposta de dados e um novo método para solicitar diferentes objetos e campos, que estamos chamando simplesmente de formato da API do X v2.

Na seção de diferenças gerais, você pode conhecer algumas mudanças relevantes para usuários padrão e corporativos. Também reunimos um guia específico para o [formato Nativo do v1.1 padrão](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview), o [formato Nativo Enriquecido](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) corporativo e o [formato Activity Streams](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#activity-object) corporativo, que ajuda a mapear campos e explica quais campos e expansões você deve usar para solicitar os novos campos da v2.

- [Formato Nativo para API do X v2 (v1.1 padrão)](/pt-BR/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2)
- [Nativo Enriquecido para API do X v2 (corporativo)](/pt-BR/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2)
- [Activity Streams para API do X v2 (corporativo)](/pt-BR/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2)

Você também pode se interessar pela nossa [ferramenta visual de migração de formato de dados](/pt-BR/x-api/migrate/data-format-migration#visual-data-format-migration-tool), que ajuda a ver rapidamente as diferenças entre o [formato de dados da API do X v1.1](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) e o [formato da API do X v2](/pt-BR/x-api/introduction).

<div id="general-differences">
  ### Diferenças gerais
</div>

<div id="requesting-objects-and-fields">
  #### Solicitando objetos e campos
</div>

Uma das maiores mudanças entre os endpoints anteriores ao v2 e o v2 é que a versão mais recente retorna apenas alguns campos por padrão, enquanto os endpoints Standard, Premium e Enterprise retornam a maioria dos campos por padrão. A nova versão utiliza parâmetros chamados [fields](/pt-BR/x-api/fundamentals/fields) e [expansions](/pt-BR/x-api/fundamentals/expansions) para solicitar especificamente dados adicionais além dos padrões, o que significa que você pode solicitar apenas os dados de que precisa, sem ter que ingerir campos que não são relevantes para você.

Quaisquer campos que você solicitar e que se relacionem ao objeto de dados primário serão retornados nesse objeto primário junto com os valores padrão. No entanto, se você solicitar quaisquer objetos expandidos usando o parâmetro expansions, os objetos secundários serão retornados em um novo objeto includes. Você pode relacionar os objetos expandidos no objeto includes ao objeto primário usando o campo id, que será retornado em ambos.

Por exemplo, se você estiver usando o endpoint v2 [Post lookup](/pt-BR/x-api/posts/lookup/introduction) e incluir o parâmetro expansions=author\_id em sua solicitação, você receberá o campo author\_id dentro do objeto Post primário, bem como um objeto user por Post no objeto includes, cada um dos quais incluirá o campo padrão id que pode ser usado para relacionar o objeto user ao objeto Post. Aqui está um exemplo de como isso fica:

```
{
  "data": [
    {
      "author_id": "2244994945",
      "id": "1397568983931392004",
      "text": "The Twitter Developer Platform. Ooh la la! https://t.co/iGTdPXBfOv https://t.co/Ze8z8EODdg"
    }
  ],
  "includes": {
    "users": [
      {
        "id": "2244994945",
        "name": "Twitter Dev",
        "username": "TwitterDev"
      }
    ]
  }
}
```

<div id="updated-json-design">
  #### Design de JSON atualizado
</div>

Além das mudanças em como você solicita determinados campos, a API do X v2 também está introduzindo novos designs de JSON para os objetos retornados pelas APIs, incluindo objetos de [Post](/pt-BR/x-api/fundamentals/data-dictionary#tweet) e de [user](/pt-BR/x-api/fundamentals/data-dictionary#user).

- No nível raiz do JSON, os endpoints padrão retornam objetos de Post em um array **statuses**, enquanto a API do X v2 retorna um array **data**.
- Em vez de se referir a “statuses” Retweeted e Quoted, o JSON da API do X v2 se refere a Tweets retuitados e citados. Muitos campos legados e descontinuados, como **contributors** e **user.translator\_type**, estão sendo removidos.
- Em vez de usar tanto **favorites** (no objeto Post) quanto **favourites** (no objeto user), a API do X v2 usa o termo like.
- O X está adotando a convenção de que valores JSON sem valor (por exemplo, null) não são escritos no payload. Atributos de Post e user só são incluídos se tiverem valores não nulos. 
   

<div id="new-v2-fields">
  #### Novos campos da v2
</div>

Também introduzimos um novo conjunto de campos no [objeto Post](/pt-BR/x-api/fundamentals/data-dictionary#tweet), incluindo:

- Um campo [conversation\_id](/pt-BR/x-api/fundamentals/conversation-id)
- Dois novos campos de [annotations](/pt-BR/x-api/fundamentals/post-annotations), incluindo context e entities
- Vários novos campos de [metrics](/pt-BR/x-api/fundamentals/metrics)
- Um novo campo reply\_setting, que indica quem pode responder a um determinado Post

<div id="migrating-from-standard-v11s-data-format-to-v2">
  ### Migrando do formato de dados do v1.1 padrão para o v2
</div>

Se ainda não o fez, recomendamos começar pela introdução de [migração de formatos de dados](/pt-BR/x-api/migrate/data-format-migration). Você também pode se interessar pela nossa [ferramenta visual de migração de formato de dados](/pt-BR/x-api/migrate/data-format-migration#visual-data-format-migration-tool), que ajuda a ver rapidamente as diferenças entre o [formato de dados da X API v1.1](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) e o [formato da API do X v2](/pt-BR/x-api/fundamentals/data-dictionary).

O formato de dados do v1.1 padrão, também conhecido como formato nativo, é o principal formato entregue pelos endpoints do [v1.1 padrão](https://developer.x.com/en/docs/twitter-api/v1).

Se você estiver usando o produto premium, consulte o [guia de dados nativos enriquecidos](/pt-BR/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2). Clientes Enterprise podem estar usando dados nativos enriquecidos ou activity streams, dependendo da configuração no console do Gnip.

<div id="migrating-from-native-enriched-data-format-to-v2">
  ### Migrando do formato de dados Native Enriched para a v2
</div>

O formato de dados Native Enriched é usado por nossos produtos [enterprise](/pt-BR/x-api/enterprise-gnip-2.0/enterprise-gnip).

O formato de dados Native Enriched foi atualizado para fornecer metadados de Tweets editados. Para saber mais sobre os metadados de edição de Tweets, confira a página [Fundamentos de edição de Tweets](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/edit-tweets).

Se você estiver usando os endpoints padrão v1.1, consulte o [guia do padrão v1.1 para v2](/pt-BR/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2). Se você estiver usando os produtos enterprise com Activity Streams, também temos um guia de [Activity Streams para v2](/pt-BR/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2).

A API do X v2 introduz novos designs JSON para os objetos de [Tweet](/pt-BR/x-api/fundamentals/data-dictionary#tweet) e de [usuário](/pt-BR/x-api/fundamentals/data-dictionary#user).

- No nível raiz do JSON, o formato Native Enriched retorna objetos de Tweet em um array results, enquanto a API do X v2 retorna um array data.
- Em vez de usar tanto favorites (no objeto de Tweet) quanto favourites (no objeto de usuário), a API do X v2 usa o termo like.
- O X está adotando a convenção de que valores JSON sem valor (por exemplo, null) não são incluídos no payload. Atributos de Tweet e de usuário só são incluídos se tiverem valores não nulos.
- Todos os campos id na v2 estarão no formato string

Além das mudanças no novo formato JSON, também introduzimos um novo conjunto de campos no objeto de Tweet, incluindo:

- conversation\_id
- reply\_settings
- alt\_text em mídia
- Dois novos campos de [annotations](/pt-BR/x-api/fundamentals/post-annotations), incluindo context e entities
- Vários novos campos de [metrics](/pt-BR/x-api/fundamentals/metrics)
- Vários novos campos de [polls](/pt-BR/x-api/fundamentals/data-dictionary#poll)

Muitos campos legados e descontinuados estão sendo removidos:

- contributors
- Alguns campos de entities.media e extended\_entities.media
- filter\_level
- timestamp\_ms
- truncated

<div id="migrating-from-activity-streams-data-format-to-v2">
  ### Migrando do formato de dados Activity Streams para a v2
</div>

O formato de dados Activity Streams está disponível em nossos produtos [enterprise](/pt-BR/x-api/enterprise-gnip-2.0/enterprise-gnip).

O formato de dados Activity Streams foi atualizado para fornecer metadados de Tweets editados. Para saber mais sobre os metadados de edição de Tweet, consulte a página [Fundamentos de edição de Tweets](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/edit-tweets).

Se você estiver usando os endpoints padrão v1.1, consulte o [guia de migração do padrão v1.1 para a v2](/pt-BR/x-api/migrate/overview). Se você estiver usando os endpoints premium ou o formato Native Enriched para enterprise, consulte o [guia de migração de Native Enriched para a v2](/pt-BR/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2).

A API do X v2 apresenta novos formatos JSON para objetos de [post](/pt-BR/x-api/fundamentals/data-dictionary#tweet) e de [usuário](/pt-BR/x-api/fundamentals/data-dictionary#user).

- No nível raiz do JSON, o formato Activity Streams retorna objetos de Tweet em um array results, enquanto a API do X v2 retorna um array data.
- Em vez de se referir a “atividades” de Retweet e de Citação, o JSON da API do X v2 se refere a Tweets retuitados e citados.
- Em vez de usar tanto favorites (no objeto Tweet) quanto favourites (no objeto user), a API do X v2 usa o termo like.
- O X adota a convenção de que valores JSON sem valor (por exemplo, null) não são escritos no payload. Atributos de Tweet e user só são incluídos se tiverem valores não nulos.
- Todos os campos id na v2 estarão em formato de string.

Além das mudanças no novo formato JSON, também introduzimos um novo conjunto de campos no objeto Tweet, incluindo:

- conversation\_id
- reply\_settings
- alt\_text em mídia
- Dois novos campos de [annotations](/pt-BR/x-api/fundamentals/post-annotations), incluindo context e entities
- Vários novos campos de [metrics](/pt-BR/x-api/fundamentals/metrics)
- Vários novos campos de [polls](/pt-BR/x-api/fundamentals/data-dictionary#poll)

Muitos campos legados e preteridos estão sendo removidos ou substituídos:

- display\_text\_range
- generator
- gnip
- link
- objectType
- provider
- twitter\_entities.symbols substituído por data.entities.cashtags
- Certos campos de twitter\_extended\_entities.media e twitter\_entities.media
- twitter\_filter\_level
- twitterTimeZone
- verb

<div id="visual-data-format-migration-tool">
  ### Ferramenta visual de migração de formato de dados
</div>

A ferramenta visual de migração de formato de dados é um aplicativo web que exibe os campos correspondentes do [formato de dados da X API v1.1](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) para o [formato da API do X v2](/pt-BR/x-api/fundamentals/data-dictionary) para um determinado objeto de post (Tweet) ou de usuário. Você pode fornecer um ID de post (Tweet) ou um ID de usuário ao aplicativo para visualizar esse mapeamento.

Observe que será necessário fazer login com sua conta do Twitter para usar o app.
