---
title: Limites de taxa
---

Todos os dias, muitos milhares de desenvolvedores fazem solicitações à API do X. Para ajudar a gerenciar o grande volume dessas solicitações, definimos limites para a quantidade de solicitações que podem ser feitas. Esses limites ajudam a oferecer a API confiável e escalável da qual nossa comunidade de desenvolvedores depende.

O número máximo de solicitações permitido é baseado em um intervalo de tempo, normalmente ao longo de um período ou janela especificada. O intervalo mais comum é de quinze minutos. Por exemplo, um endpoint com limite de 900 solicitações por 15 minutos permite até 900 solicitações em qualquer intervalo de 15 minutos.

Os limites de taxa dependem do método de autenticação. Por exemplo, se estiver usando [OAuth 1.0a User Context](/pt-BR/resources/fundamentals/authentication), cada conjunto de [Access Tokens](/pt-BR/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) dos usuários tem seu próprio limite de taxa por período. Já se estiver usando [OAuth 2.0 Bearer Token](/pt-BR/resources/fundamentals/authentication#oauth-2-0), seu app terá um limite próprio e separado por período. Quando esses limites são excedidos, um erro é retornado.

<div id="rate-limits-and-authentication-method">
  ## Limites de taxa e método de autenticação
</div>

Os limites de taxa são definidos tanto no nível do App de desenvolvedor quanto no nível do token de acesso do usuário:

- **[OAuth 2.0 Bearer Token](/pt-BR/resources/fundamentals/authentication/oauth-2-0/application-only): Limite de taxa do App**
  Esse método permite fazer um determinado número de solicitações em nome do seu App de desenvolvedor. Ao usar esse método de autenticação, os limites são determinados pelas solicitações feitas com um Bearer Token.
  - Exemplo: Com um limite de 450 solicitações por intervalo de 15 minutos, você pode fazer 450 solicitações em nome do seu App dentro desse intervalo.

- **[OAuth 1.0a User Context](/pt-BR/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens): Limite de taxa do usuário**
  Esse método permite fazer solicitações em nome de um usuário do X identificado pelo Access Token do usuário. Por exemplo, ao recuperar métricas privadas de Posts, autentique com Access Tokens de usuário para esse usuário, gerados usando o [fluxo OAuth com 3 partes](/pt-BR/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow).
  - Exemplo: Com um limite de 900 solicitações por usuário a cada 15 minutos, você pode fazer até 900 solicitações por usuário nesse período.

<div id="http-headers-and-response-codes">
  ## Cabeçalhos HTTP e códigos de resposta
</div>

Use os cabeçalhos HTTP para entender em que ponto sua aplicação está em relação a um determinado limite de taxa, com base na solicitação mais recente feita.

- `x-rate-limit-limit`: teto do limite de taxa para o endpoint
- `x-rate-limit-remaining`: solicitações restantes na janela de 15 minutos
- `x-rate-limit-reset`: tempo restante antes de o limite de taxa ser redefinido (em segundos desde a época UTC)

<div id="error-responses">
  ### Respostas de erro
</div>

Se um aplicativo ultrapassar o limite de taxa de um endpoint, a API retornará uma resposta [HTTP 429 “Too Many Requests”](http://tools.ietf.org/html/rfc6585) com a seguinte mensagem de erro no corpo da resposta:

```json
{ "errors": [ { "code": 88, "message": "Rate limit exceeded" } ] }
```

<div id="recovering-from-a-rate-limit">
  ## Recuperando de um limite de taxa
</div>

Quando esses limites de taxa são excedidos, o endpoint retorna um erro 429 “Too Many Requests”. Conforme discutido abaixo, quando ocorrem erros de limite de taxa, uma prática recomendada é verificar os cabeçalhos HTTP que indicam quando o limite será reiniciado e pausar as solicitações até lá.<br />Quando ocorre um erro de “Too Many Requests” ou de limitação de taxa, a frequência de solicitações precisa ser reduzida. Quando um erro de limite de taxa ocorrer, o cabeçalho HTTP `x-rate-limit-reset:` pode ser consultado para saber quando a limitação será reiniciada<br /><br />. Outro padrão comum é o backoff exponencial, em que o tempo entre solicitações começa pequeno (por exemplo, alguns segundos) e depois é dobrado antes de cada nova tentativa. Isso continua até que uma solicitação seja bem-sucedida ou até que seja atingido um tempo máximo razoável entre solicitações (por exemplo, alguns minutos).<br /><br /> Idealmente, o cliente tem ciência dos limites de taxa existentes e pode pausar as solicitações até que a janela atualmente excedida expire. Se você exceder um limite de 15 minutos, faz sentido esperar um ou dois minutos antes de tentar novamente.<br /><br /> Observe que, além desses limites no número de solicitações, o nível de acesso Standard Basic fornece até 500.000 posts por mês nos endpoints de busca recente e fluxo filtrado. Se você excedeu o limite mensal de posts, faz mais sentido que seu app emita uma notificação, saiba o dia de renovação no mês e suspenda as solicitações até esse dia.

<div id="tips-to-avoid-being-rate-limited">
  ### Dicas para evitar atingir limites de taxa
</div>

As dicas abaixo ajudam você a programar de forma defensiva e reduzir a chance de atingir limites de taxa. Alguns recursos do aplicativo que você talvez queira oferecer são simplesmente inviáveis devido a esses limites, especialmente no que diz respeito à atualidade dos resultados. Se informações em tempo real forem um objetivo do seu aplicativo, considere usar os endpoints de fluxo filtrado e de fluxo amostrado.

<div id="caching">
  #### Cache
</div>

Armazene as respostas da API se prever uso frequente. Em vez de chamar a API a cada carregamento de página, armazene a resposta em cache localmente.

<div id="prioritize-active-users">
  #### Priorize usuários ativos
</div>

Se o seu site acompanha muitos usuários do X (por exemplo, obtendo o status atual deles ou estatísticas sobre o uso do X), considere solicitar dados apenas dos usuários que tenham feito login recentemente no seu site.

<div id="adapt-to-the-search-results">
  #### Adapte-se aos resultados de busca
</div>

Se seu aplicativo monitora um grande volume de termos de busca, faça consultas com menos frequência para as buscas que não retornam resultados do que para as que retornam. Ao aplicar um backoff, você pode manter-se atualizado sobre consultas populares sem desperdiçar recursos solicitando consultas que raramente mudam. Como alternativa, considere usar o endpoint Fluxo filtrado e aplicar filtros com suas consultas de busca.

<div id="denylist">
  #### Lista de bloqueio
