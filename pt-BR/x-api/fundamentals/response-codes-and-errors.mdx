A API do X fornece os seguintes códigos de resposta e erro para ajudar a entender e depurar problemas em tempo real. Use o guia de depuração e o índice de erros abaixo para obter contexto adicional.

Respostas bem-sucedidas são indicadas por um código HTTP da série 200 e um payload em JSON contendo os objetos solicitados, criados, modificados ou excluídos, juntamente com a interpretação do servidor sobre a sua solicitação.

Respostas de erro são retornadas com um código HTTP fora da série 200. Códigos de erro diferentes indicam motivos diferentes para o erro. A API do X procura retornar [códigos de status HTTP](https://en.wikipedia.org/wiki/List_of_HTTP_status_codes) apropriados para cada solicitação.

<div id="x-api-v2-http-status-codes">
  ## Códigos de status HTTP da API do X v2
</div>

| Code | Text | Description | Troubleshooting tips |
| :--- | :--- | :---------- | :------------------- |
| 200 | OK | A solicitação foi bem-sucedida! | |
| 400 | Bad Request	| A solicitação é inválida ou não pode ser atendida. Uma mensagem de erro complementar trará mais detalhes. Solicitações sem autenticação ou com parâmetros de consulta inválidos são consideradas inválidas e retornarão esta resposta. | Verifique o formato da sua consulta JSON. Por exemplo, se sua regra contiver aspas duplas associadas a uma correspondência exata ou a outro operador, talvez seja necessário escapá-las usando uma barra invertida para distingui-las da estrutura do JSON. |
| 401 | Unauthorized | Ocorreu um problema ao autenticar sua solicitação. Isso pode ser devido a credenciais ausentes ou incorretas. Também pode ser retornado em outras circunstâncias não definidas. | Verifique se você está usando o [método de autenticação](https://docs.x.com/resources/fundamentals/authentication/overview) correto e se suas credenciais estão corretas. |
| 403 | Forbidden | A solicitação foi compreendida, mas foi recusada ou o acesso não é permitido. Uma mensagem de erro explicará o motivo. | Verifique se sua conta de desenvolvedor tem acesso ao endpoint que você está tentando usar. Talvez seja necessário colocar seu App na allowlist (por exemplo, Engagement API ou Ads API) ou inscrever-se para obter acesso. |
| 404 | Not Found | A URI solicitada é inválida ou o recurso solicitado, como um usuário, não existe. | Verifique se você está usando parâmetros válidos e a URI correta para o endpoint em uso. |
| 409 | Connection Exception | Retornado ao tentar se conectar a um fluxo filtrado que não possui regras. | Verifique se você criou ao menos uma regra no fluxo ao qual está se conectando. O fluxo filtrado retorna apenas Posts que correspondam a uma regra ativa. Se não houver regras, o fluxo não retornará nenhum Post. |
| 429 | Too Many Requests | Retornado quando uma solicitação não pode ser atendida porque o [limite de taxa](/pt-BR/resources/fundamentals/rate-limits) do App ou o [Limite de posts](/pt-BR/x-api/fundamentals/post-cap) foi atingido. Consulte [Limitação de taxa](/pt-BR/resources/fundamentals/rate-limits). | Verifique o número de solicitações por janela de tempo permitido no endpoint em uso. Aguarde a redefinição da janela. Distribua suas solicitações para não atingir os limites de taxa ou faça upgrade para o próximo plano de dados disponível. |
| 500 | Internal Server Error | Algo está com problemas. Geralmente é um erro temporário, por exemplo, em situação de alta carga ou quando um endpoint está com instabilidade temporária. | Verifique a [página de status](/pt-BR/status) da API do X ou o [fórum da comunidade de desenvolvedores](https://devcommunity.x.com/) caso outras pessoas estejam enfrentando problemas semelhantes, ou simplesmente aguarde e tente novamente mais tarde. |
| 501 | Unimplemented | A API do X não oferece suporte a este endpoint e não pode atender à solicitação. |  |
| 503 | Service Unavailable | Os servidores do X estão ativos, mas sobrecarregados com solicitações. Tente novamente mais tarde. | Verifique a [página de status](/pt-BR/status) da API do X ou o [fórum da comunidade de desenvolvedores](https://devcommunity.x.com/) caso outras pessoas estejam enfrentando problemas semelhantes, ou simplesmente aguarde e tente novamente mais tarde. |

Quando ocorre um erro durante uma solicitação, informações detalhadas sobre o erro são retornadas no corpo da resposta para ajudar a diagnosticar o problema. Um campo type, que é uma URI, indica a natureza do problema, enquanto campos adicionais fornecem mais detalhes. Os campos type, title e detail sempre serão retornados nesses corpos (veja a tabela abaixo). Quaisquer campos adicionais, como no exemplo abaixo, variam de acordo com o tipo de erro.

```json
{
  "client_id": "101010101",
  "required_enrollment": "Standard Basic",
  "registration_url": "https://developer.twitter.com/en/account",
  "title": "Client Forbidden",
  "detail": "This request must be made using an approved developer account that is enrolled in the requested endpoint. Learn more by visiting our documentation.",
  "reason": "client-not-enrolled",
  "type": "https://api.x.com/2/problems/client-forbidden"
}
```

<div id="partial-errors">
  ## Erros parciais
</div>

Em alguns casos, você pode ver os erros detalhados acima em uma resposta que retornou o código de status 200. Nesses casos, o endpoint foi projetado para retornar os dados que conseguir, fornecendo ao mesmo tempo erros detalhados sobre o que não pôde retornar.

Por exemplo, o [endpoint de busca de Posts](/pt-BR/x-api/posts/lookup/introduction) permite que um [App de desenvolvedor](/pt-BR/resources/fundamentals/developer-apps#overview) do X solicite mais de um ID. Se alguns desses Posts estiverem disponíveis, mas um deles tiver sido excluído, os Posts disponíveis serão retornados no campo data da resposta. Um campo adicional errors será incluído no payload, indicando quais Post(s) solicitados não puderam ser retornados. O mesmo formato é usado que em erros de solicitação completa para facilitar o diagnóstico de problemas.

<div id="error-information">
  ### Informações sobre erros
</div>

Cada tipo de problema indica a natureza do erro encontrado. Você também pode consultar a lista completa de problemas em nossa [especificação da API](https://api.x.com/2/openapi.json).

A API do X tenta retornar [códigos de status HTTP](https://en.wikipedia.org/wiki/List_of_HTTP_status_codes) apropriados para cada solicitação.

| Título | Tipo | Descrição |
| :--- | :--- | :--- |
| Problema genérico | `about:blank` | Um problema genérico sem informações adicionais além das fornecidas pelo código de status HTTP. |
| Problema de solicitação inválida | `https://api.X.com/2/problems/invalid-request` | Indica que esta solicitação é inválida. Se sua solicitação incluir um corpo POST, verifique se o conteúdo é JSON válido e está em conformidade com a [especificação OpenAPI](https://api.x.com/2/openapi.json). |
| Problema de recurso não encontrado | `https://api.X.com/2/problems/resource-not-found` | Indica que um Post, Usuário etc. específico não existe. |
| Problema de recurso não autorizado | `https://api.X.com/2/problems/not-authorized-for-resource` | Indica que você não tem permissão para ver um Post, Usuário etc. específico. |
| Problema de cliente proibido | `https://api.X.com/2/problems/client-forbidden` | Indica que seu cliente não tem permissão para fazer esta solicitação. |

<div id="troubleshooting-tips">
  ## Dicas de solução de problemas
</div>

<Accordion title="Depuração do seu código">
  Ao criar um aplicativo, é normal encontrar erros ou problemas inesperados de tempos em tempos. A seguir estão algumas dicas de como você pode depurar seu código:

  Comece dividindo o problema em partes menores para identificar onde está a falha. Por exemplo, se você estiver integrando um endpoint REST ou de streaming, o problema pode estar em qualquer um dos itens a seguir:

  - O endpoint requer [autenticação](/pt-BR/resources/fundamentals/authentication/overview) adequada.
  - O endpoint requer parâmetros e cabeçalhos válidos. Quaisquer regras de filtragem devem ser construídas com os operadores corretos e a [sintaxe](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#building-rules-and-queries) apropriada.
  - O URI do endpoint deve estar correto e, no caso de endpoints REST, é preciso usar o método HTTP correto.
  - Os dados ou o recurso que você está tentando acessar podem não estar acessíveis a você (por exemplo, dados privados só estão disponíveis para usuários autenticados).
  - Seu pacote de dados atual dá acesso apenas a determinados endpoints e a limites de taxa específicos. Consulte seu [portal do desenvolvedor](https://developer.x.com/en/portal/products) para mais detalhes.
  - Seu código usa uma biblioteca de terceiros para integrar o endpoint ao seu código.
  - Seu código precisa analisar corretamente a resposta do endpoint.

  Leia a mensagem de erro correspondente. Ela deve dar uma boa indicação do problema. Use as tabelas na [seção de códigos de erro e resposta](#x-api-http-status-codes) para dicas de solução de problemas específicas de cada código.

  Para endpoints REST, você pode usar um cliente REST como o [Postman](https://www.postman.com/) ou o [Insomnia](https://insomnia.rest/) para validar as etapas (1) a (5) acima (veja nosso guia ["Introdução ao Postman"](/pt-BR/tutorials/postman-getting-started)). Se a solicitação com o cliente REST retornar um código de status 200 (sucesso), você pode supor que o problema está no seu código ou na biblioteca que você está usando, e não na solicitação ao endpoint em si.

  Para endpoints de streaming, você pode usar cURL (uma ferramenta de linha de comando para enviar ou receber solicitações usando a sintaxe de URL) para verificar se o problema está na solicitação ao endpoint (etapas (1) a (5) acima) ou no seu próprio código (etapas (6) a (7) acima).
</Accordion>

<Accordion title="Problemas de autenticação">
  - Verifique se você está usando o [método de autenticação](/pt-BR/resources/fundamentals/authentication/overview) adequado exigido para o endpoint. Isso pode ser identificado na página de referência da API desse endpoint.
  - Verifique se suas credenciais de autenticação estão corretas. Você pode conferir ou regenerar as chaves e os tokens do seu App na [seção Apps](https://developer.x.com/en/portal/projects-and-apps) do seu painel de desenvolvedor (em “Details”).
  - Verifique se você [autorizou corretamente sua solicitação OAuth 1.0a](/pt-BR/resources/fundamentals/authentication/oauth-1-0a/authorizing-a-request) com `oauth_nonce`, `oauth_signature` e `oauth_timestamp`.
  - Se o problema persistir, considere usar uma biblioteca OAuth, um cliente REST como o [Postman](https://www.postman.com/) ou o [Insomnia](https://insomnia.rest/), ou o [xurl](https://github.com/xdevplatform/xurl).

  Consulte nosso [guia sobre autenticação](/pt-BR/resources/fundamentals/authentication/overview) para obter mais informações sobre todos os itens acima.
</Accordion>

<Accordion title="Problemas de rate limit e limites de posts">
  Um erro 429 pode ocorrer se você atingir o rate limit de um endpoint específico ou se atingir o limite de posts.

  Se o erro específico “Too many requests” for retornado, você atingiu o rate limit do endpoint. Em outras palavras, você excedeu o número máximo de solicitações permitidas para um endpoint em um período de tempo definido.

  Observe que os rate limits são definidos nos níveis por App e por usuário.

  - **Nível de X App** indica o número de solicitações permitidas ao usar OAuth 2.0 App-Only, onde os rate limits são determinados globalmente para todo o App. Por exemplo, se um método permite 15 solicitações por janela de rate limit, então ele permite 15 solicitações por janela em nome do seu X App. Esse limite é considerado completamente separado do limite em nível de usuário. Leia mais em nosso guia sobre [OAuth 2.0 App-Only](/pt-BR/resources/fundamentals/authentication/oauth-2-0/application-only).
  - **Nível de contexto do usuário** indica o número de solicitações que podem ser feitas por Access Token de usuário ao usar OAuth 1.0a User Context ou OAuth 2.0 Authorization Code com PKCE. Por exemplo, se um método permite 15 solicitações por janela de rate limit, então ele permite 15 solicitações por janela e por Access Token de usuário. Leia mais em nosso guia sobre como obter os Access Tokens de um usuário com [OAuth 1.0a](/pt-BR/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens) e [OAuth 2.0](/pt-BR/resources/fundamentals/authentication/oauth-2-0/user-access-token).

  Comece verificando os rate limits do endpoint que você está usando. Você pode encontrar essas informações na página de referência da API do endpoint e no novo painel do portal do desenvolvedor.

  Consulte nossa documentação para obter informações adicionais sobre rate limits, incluindo como usar cabeçalhos HTTP para acompanhar a posição do seu App em relação a um determinado rate limit, como se recuperar de um erro 429 e dicas para evitar ser limitado:

  - [Limites de taxa para endpoints v2](/pt-BR/x-api/fundamentals/rate-limits)

  Se você recebeu o erro específico “Usage cap exceeded: Monthly product cap”, isso significa que você atingiu o limite de posts do seu nível de acesso. Temos muitos detalhes sobre esses [limites de posts](/pt-BR/x-api/fundamentals/post-cap) em nossa página de documentação.
</Accordion>

<Accordion title="Problemas de posts ausentes">
  Siga as etapas abaixo se você esperava que um Post fosse retornado, mas ele não foi entregue pelo endpoint.

  - Verifique sua regra para garantir que você está usando os [operadores](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#enterprise-operators) e a [sintaxe](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#building-rules-and-queries) corretos. Divida a regra em cláusulas menores para garantir que você está usando a sintaxe correta.
  - Se a conta da qual o Post foi enviado estava [protegida](https://help.twitter.com/en/safety-and-security/public-and-protected-tweets) no momento em que o Post foi criado, o Post não será retornado — mesmo que a conta esteja pública no momento da solicitação ao endpoint. Normalmente, você pode verificar isso usando a [X Advanced Search](https://x.com/search-advanced): se um Post não aparecer usando a funcionalidade X Advanced Search, você deve presumir que ele não será retornado pelo endpoint.

  As etapas a seguir se aplicam apenas a endpoints de streaming:

  - Você estava conectado ao stream quando o Post foi enviado? Lembre-se de que o timestamp entregue no objeto Post indica o horário em UTC. Se você teve uma desconexão quando o Post foi enviado, revise os [recursos de recuperação e redundância](/pt-BR/x-api/posts/filtered-stream/integrate/recovery-and-redundancy-features#recovery-and-redundancy) disponíveis para preencher quaisquer dados perdidos.
  - Sua regra estava ativa quando o Post foi enviado? Lembre-se de que o timestamp entregue no objeto Post indica o horário em UTC.
</Accordion>

<Accordion title="Problemas de desconexão por buffer cheio">
  Você pode receber um dos seguintes erros quando seu fluxo não acompanha a velocidade com que entregamos os dados e seu App não consome os dados do fluxo rápido o suficiente:

  ```
  This stream has been disconnected because your client was unable to keep up with us.
  ```

  ```
  This stream has been disconnected for operational reasons.
  ```

  Permitimos que a entrega fique atrasada por um período e mantemos um buffer temporário de preparação para cada fluxo do nosso lado; porém, se você não recuperar o atraso, iniciamos uma desconexão para que você possa se reconectar no ponto atual. Observe que isso pode levar à perda de dados (para os dados que estiverem no buffer no momento da desconexão por buffer cheio).

  Isso pode ocorrer em grandes picos de dados. Em geral, recomendamos usar um processo de enfileiramento para consumir dados rapidamente, separado do processo de processamento.

  Se você é um cliente enterprise usando endpoints v1.1, saiba mais sobre como otimizar seu App para evitar desconexões como esta em nossos artigos sobre [conexão](/pt-BR/x-api/enterprise-gnip-2.0/powertrack-api#disconnections-explained) e sobre consumo de dados de streaming [aqui](https://developer.x.com/en/docs/tutorials/consuming-streaming-data) e [aqui](/pt-BR/x-api/enterprise-gnip-2.0/powertrack-api#planning-for-high-volume-social-data-events).

  Há uma variedade de ferramentas disponíveis para recuperar posts perdidos devido a uma desconexão, incluindo as listadas abaixo. Observe que as seguintes ferramentas estão disponíveis apenas com endpoints v1.1 no nível de acesso enterprise.

  - [**Conexões redundantes**](/pt-BR/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - Com várias conexões, consuma o fluxo a partir de vários servidores para evitar perda de dados quando uma delas for desconectada.
  - [**Replay**](/pt-BR/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - Recupere dados dos últimos 5 dias usando um fluxo separado.
  - [**Backfill**](/pt-BR/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - Reconecte em até 5 minutos e retome de onde parou.
  - **Historical PowerTrack** - Recupere dados de todo o arquivo do X.
</Accordion>

<div id="before-you-post-a-question">
  ### Antes de publicar uma pergunta
</div>

- Pesquise na documentação para desenvolvedores do X por informações relacionadas ao seu problema
- Procure no fórum da comunidade por perguntas semelhantes de outros desenvolvedores
- Leia as diretrizes do fórum da comunidade

<div id="when-you-post-a-question-make-sure-to-include-the-following-information">
  ### Ao publicar uma pergunta, certifique-se de incluir as seguintes informações
</div>

- Uma descrição do problema
- A chamada de API feita (inclua os cabeçalhos, se possível)
- A resposta retornada pelo X (inclua quaisquer mensagens de erro)
- O que você esperava receber em vez disso
- Lista das etapas realizadas para solucionar o problema
- Lista das etapas necessárias para reproduzir o problema
- Se relevante, o período em que o problema ocorreu
- Se relevante, o ID do App, ID do Post, etc.
- Qualquer amostra de código ou captura de tela relevante

Inclua apenas um tópico/pergunta por post.

Se você tiver solicitações de recursos ou feedback, envie-os pelo [Formulário de feedback da Plataforma de Desenvolvedores do X](https://t.co/devfeedback).

Para questões relacionadas a políticas, como suspensão de Apps, entre em contato com o [suporte de políticas](https://help.twitter.com/forms/platform).

Para questões relacionadas ao X, como login e suporte à conta, use o [X Help Desk](https://help.twitter.com/en/contact-us).
