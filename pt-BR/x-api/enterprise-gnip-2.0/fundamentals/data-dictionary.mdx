<div id="introduction">
  ### Introdução
</div>

`Enterprise`

Posts são o bloco atômico básico de tudo no X. Todas as APIs do X que retornam Posts fornecem esses dados codificados em JavaScript Object Notation (JSON). O JSON é baseado em pares chave-valor, com atributos nomeados e valores associados. Objetos de Post recuperados da API incluem a “atualização de status” de um usuário do X, e Retweets, respostas e Tweets com citação também são objetos de Post. Se um Post estiver relacionado a outro, como um Retweet, uma resposta ou um Tweet com citação, cada um será identificado ou incorporado ao objeto de Post. Mesmo o Post mais simples, no formato de dados nativo do X, terá objetos JSON aninhados para representar outros atributos de um Post, como o autor, usuários mencionados, local marcado, hashtags, símbolos de cashtag, mídia ou links de URL. Ao trabalhar com dados do X, é importante entender esse conceito. O formato dos dados de Post que você receberá da API do X depende do tipo de Post, da API do X que está usando e das configurações de formato.

Endpoints Enterprise que retornam objetos de Post foram atualizados para fornecer os metadados necessários para entender o histórico de edições do Post. Saiba mais sobre esses metadados na página de fundamentos ["Editar Posts"](/pt-BR/x-api/fundamentals/edit-posts).

<div id="available-data-formats">
  ### Formatos de dados disponíveis
</div>

> Observação: é altamente recomendável usar o formato [Native Enriched](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) para as APIs de dados empresariais.
>
> - O formato Native Enriched inclui todos os novos metadados desde 2017, como [metadados de enquetes](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata), e métricas adicionais como reply\_count e quote\_count.
>
> - O formato Activity Streams não recebe novos metadados ou enriquecimentos desde a [atualização de caracteres](https://blog.x.com/official/en_us/topics/product/2017/Giving-you-more-characters-to-express-yourself.html) em 2017.

As APIs de dados empresariais entregam dados em dois formatos. O formato empresarial mais próximo do formato nativo padrão v1.1 é o Native Enriched. O formato empresarial legado é o Activity Streams, originalmente implementado e usado pela Gnip como um formato normalizado no X e em outros provedores de dados de mídias sociais à época. Embora esse formato ainda esteja disponível, o X tem investido apenas no desenvolvimento de novos recursos para o formato Native Enriched desde 2017.

O formato Native Enriched é exatamente isso: inclui objetos nativos do X, bem como enriquecimentos adicionais disponíveis para produtos de dados empresariais, como metadados de expansão de URL, geolocalização de perfil, metadados de enquetes e métricas adicionais de engajamento.

- [Enriquecimento de URLs expandidas e aprimoradas](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls)
- [Enriquecimento de regras correspondentes](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#matching-rules)
- [Enriquecimento de metadados de enquetes](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata)
- [Enriquecimento de geolocalização de perfil](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo)

### Melhores práticas de análise de JSON

- O JSON do X é codificado em UTF-8.
- Os analisadores devem lidar facilmente com variações na ordenação dos campos. Deve-se presumir que o JSON de posts é servido como um mapa/objeto não ordenado.
- Os analisadores devem tolerar a adição de novos campos.
- Os analisadores de JSON devem tolerar campos ausentes, já que nem todos os campos aparecem em todos os contextos.
- Em geral, é seguro considerar um campo nulo, um conjunto vazio e a ausência de um campo como equivalentes.

#### Objeto Post

Ao usar produtos de dados empresariais, você notará que grande parte do dicionário de dados é semelhante ao formato nativo dos dados de Post, com alguns metadados adicionais enriquecidos. O nível básico do formato nativo enriquecido utiliza muitos dos mesmos nomes de objetos do [formato de dados da X API v1.1](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview.html). O objeto Post tem uma longa lista de atributos no “nível raiz”, incluindo atributos fundamentais como `id`, `created_at` e `text`. Objetos Post também terão objetos aninhados para incluir `user`, `entities` e `extended_entities`. Objetos Post também terão outros [objetos Post aninhados](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#nested-post-objects), como `retweeted_status`, `quoted_status` e `extended_tweet`. O formato nativo enriquecido também incluirá um objeto `matching_rules`.

| Attribute | Type | Description |
| :--- | :--- | :--- |
| current\_user\_retweet | Object | _Perspectival_ Aparece apenas em métodos que suportam o parâmetro include\_my\_retweet, quando definido como true. #### Objetos de Post aninhados

Em vários casos, um objeto Post incluirá outros objetos aninhados. Se você estiver trabalhando com objetos aninhados, esse payload JSON conterá vários objetos Post, e cada objeto Post poderá conter seus próprios objetos. O objeto no nível raiz trará informações sobre o tipo de ação realizada, isto é, se é um Retweet ou um Quote Tweet, e também poderá conter um objeto que descreve o Post “original” que está sendo compartilhado. Posts estendidos incluirão um objeto estendido aninhado que ultrapassa 140 caracteres, o que foi usado para evitar alterações incompatíveis quando a atualização foi feita em 2017. Cada dicionário de objeto aninhado é descrito abaixo.

**Retweets**

Retweets sempre contêm dois objetos Post. O Post “original” sendo retweetado é fornecido em um objeto "retweeted\_status". O objeto no nível raiz encapsula o próprio Retweet, incluindo um objeto User para a conta que realiza a ação de Retweet e o horário do Retweet. Retweetar é uma ação para compartilhar um Post com seus seguidores, e nenhum conteúdo novo pode ser adicionado. Além disso, uma nova localização não pode ser fornecida em um Retweet. Embora o Post “original” possa ter geolocalização, os objetos "geo" e "place" do Retweet sempre serão nulos.

Mesmo antes da introdução de Posts estendidos, o objeto "entities" no nível raiz era, em alguns casos, truncado e incompleto devido à string "RT @username " anexada à mensagem do Post sendo retweetado. Observe que, se um Retweet for retweetado, o "retweet\_status" ainda apontará para o Post original, o que significa que o Retweet intermediário não é incluído. Comportamento semelhante é observado ao usar x.com para “exibir” um Retweet. Se você copiar o ID de Post exclusivo atribuído à ação de Retweet, o Post original é exibido.

A seguir, um exemplo de estrutura para um Retweet. Novamente, ao analisar Retweets, é fundamental analisar o objeto "retweeted\_status" para obter a mensagem completa do Post (original) e os metadados de entidades.

```json
{
	"tweet": {
		"text": "RT @author original message",
		"user": {
			"screen_name": "Retweeter"
		},
		"retweeted_status": {
			"text": "original message",
			"user": {
				"screen_name": "OriginalTweeter"
			},
			"place": {},
			"entities": {},
			"extended_entities": {}
		}
	},
	"entities": {},
	"extended_entities": {}
}
```

##### Tweets com citação

Tweets com citação são muito semelhantes a Retweets, exceto por incluírem uma nova mensagem de Post. Essas novas mensagens podem conter seu próprio conjunto de hashtags, links e outros metadados de “entities”. Tweets com citação também podem incluir informações de localização compartilhadas pelo usuário que publica o Tweet com citação, juntamente com mídia como GIFs, vídeos e fotos.

Tweets com citação contêm pelo menos dois objetos Post e, em alguns casos, três. O Post que está sendo citado, que por sua vez pode ser um Tweet com citação, é fornecido em um objeto "quoted\_status". O objeto no nível raiz encapsula o próprio Tweet com citação, incluindo um objeto User para a conta que realiza a ação de compartilhamento e o horário do Tweet com citação.

Observe que agora Tweets com citação podem ter fotos, GIFs ou vídeos adicionados por meio da interface do usuário “Post”. Quando links para mídia hospedada externamente são incluídos na mensagem do Tweet com citação, o "entities.urls" do nível raiz os descreverá. A mídia anexada a Tweets com citação aparecerá nos metadados "extended\_entities" do nível raiz.

Quando os Tweets com citação foram lançados pela primeira vez, um link encurtado (URL t.co) era anexado à mensagem do Post “original” e fornecido no campo "text" do nível raiz. Além disso, os metadados desse URL t.co eram incluídos no array "entities.urls" do nível raiz. Em maio de 2018, alteramos isso para que a URL t.co encurtada do Tweet citado não seja incluída no campo "text" do nível raiz. Em segundo lugar, os metadados do Tweet citado não serão incluídos nos metadados "entities.urls". Em vez disso, os metadados de URL do Tweet citado estarão em um novo objeto "quoted\_status\_permalink" no nível raiz (ou nível superior), no mesmo nível do objeto "quoted\_status".

<div id="quote-tweets">
  ##### Posts estendidos
</div>

O JSON que descreve Posts estendidos foi introduzido quando os Posts de 280 caracteres foram lançados em novembro de 2017. O JSON de Post foi ampliado para encapsular essas mensagens mais longas sem interromper os milhares de apps que processam esses objetos fundamentais do X. Para fornecer total compatibilidade retroativa, o campo original de 140 caracteres "text" e os objetos de entidades derivados dele foram mantidos. No caso de Posts com mais de 140 caracteres, esse campo "text" no nível raiz fica truncado e, portanto, incompleto. Como os objetos "entities" no nível raiz contêm arrays de metadados-chave extraídos da mensagem "text", como hashtags e links incluídos, essas coleções ficam incompletas. Por exemplo, se uma mensagem de Post tiver 200 caracteres, com uma hashtag no final, o array legado "entities.hashtags" no nível raiz não a incluirá.

Foi introduzido um novo campo "extended\_tweet" para armazenar as mensagens de Post mais longas e os metadados completos de entidades. O objeto "extended\_tweet" fornece o campo "full\_text", que contém a mensagem completa do Post, sem truncamento, quando tiver mais de 140 caracteres. O objeto "extended\_tweet" também contém um objeto "entities" com arrays completos de hashtags, links, menções etc.

Posts estendidos são identificados por um booleano "truncated" no nível raiz. Quando true ("truncated": true), os campos de "extended\_tweet" devem ser processados em vez dos campos do nível raiz.

Observe, no exemplo de JSON abaixo, que o campo "text" no nível raiz está truncado e o array "entities.hashtags" no nível raiz está vazio, embora a mensagem do Post inclua três hashtags. Como este é um Post estendido, o campo "truncated" está definido como true e o objeto "extended\_tweet" fornece metadados completos do Post em "full\_text" e "entities".

### Objetos nativos de geolocalização enriquecida

Posts podem ser associados a um local, gerando um Post “com geotag”. Os locais podem ser definidos pela interface do usuário do X ou ao publicar um Post pela API. O local do Post pode ser um “ponto” exato ou um X Place com uma “caixa delimitadora” que descreve uma área maior, variando de um estabelecimento a uma região inteira.

Há três objetos JSON de “nível raiz” usados para descrever o local associado a um Post: [place](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#place-data-dictionary), [geo](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-geo-objects) e [coordinates](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#coordinates-object-data-dictionary).

Além disso, o formato nativo enriquecido inclui, dentro do objeto de usuário, o [local derivado](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#derived-locations) do [enriquecimento de geolocalização de perfil](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo).

O objeto `place` está sempre presente quando um Post é geotagueado com um lugar. Places são locais específicos, nomeados, com coordenadas geográficas correspondentes. Quando os usuários decidem atribuir um local ao seu Post, é apresentada uma lista de X Places candidatos. Ao usar a API para publicar, um X Place pode ser anexado especificando um place\_id no momento da postagem. Posts associados a Places não são necessariamente publicados a partir desse local; eles também podem ser _sobre_ esse local.

Os objetos `geo` e `coordinates` só estão presentes (não nulos) quando o Post é atribuído a um _local exato_. Se um local exato for fornecido, o objeto `coordinates` trará um array \[long, lat] com as coordenadas geográficas, e um X Place correspondente a esse local será atribuído.

#### Dicionário de dados de Place

| Field | Type | Description |

<div id="x-entities">
  ## Introdução
</div>

As entidades fornecem metadados e informações contextuais adicionais sobre o conteúdo publicado no X. A seção `entities` traz arrays de elementos comuns incluídos em Posts: hashtags, menções a usuários, links, tickers de ações (símbolos), enquetes do X e mídia anexada. Esses arrays são convenientes para desenvolvedores ao ingerirem Posts, já que o X essencialmente pré-processou (ou pré-parseou) o corpo do texto. Em vez de precisar procurar e identificar explicitamente essas entidades no corpo do Post, seu parser pode ir direto a esta seção JSON, onde elas já estão disponíveis.

Além de facilitar a análise, a seção `entities` também fornece metadados úteis que agregam valor. Por exemplo, se você estiver usando o [enriquecimento Enhanced URLs](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls), os metadados de URL incluem URLs totalmente expandidas, bem como títulos e descrições de sites associados. Outro exemplo: quando há menções a usuários, os metadados de entidades incluem o ID numérico do usuário, o que é útil ao fazer solicitações para muitas APIs do X.

Todo payload JSON de um Post inclui uma seção `entities`, com o conjunto mínimo de atributos `hashtags`, `urls`, `user_mentions` e `symbols`, mesmo que nenhuma dessas entidades faça parte da mensagem do Post. Por exemplo, se você examinar o JSON de um Post com o corpo “Hello World!” e sem mídia anexada, o JSON do Post incluirá o conteúdo a seguir, com arrays de entidades contendo zero itens:

```
"entities": {
        "hashtags": [
        ],
        "urls": [
        ],
        "user_mentions": [
        ],
        "symbols": [
        ]
      }
```

Observações:

- as entidades de media e polls só aparecerão quando esse tipo de conteúdo fizer parte do Post.
- se você estiver trabalhando com mídia nativa (fotos, vídeos ou GIFs), o [objeto Extended Entities](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#x-extended-entities) é o mais indicado.

### Objeto Entities

As seções `entities` e `extended_entities` são compostas por arrays de _objetos_ de entidade. Abaixo, você encontrará descrições de cada um desses objetos de entidade, incluindo dicionários de dados que descrevem os nomes dos atributos do objeto, seus tipos e uma breve descrição. Também indicaremos quais operadores do PowerTrack correspondem a esses atributos e incluiremos alguns payloads JSON de exemplo.

Uma coleção de entidades comuns encontradas em Posts, incluindo hashtags, links e menções a usuários. O objeto `entities` inclui um atributo `media`, mas sua implementação na seção `entities` é totalmente precisa apenas para Posts com uma única foto. Para todos os Posts com mais de uma foto, um vídeo ou um GIF animado, consulte a seção `extended_entities`.

### Objeto de hashtag  

A seção `entities` conterá um array `hashtags` com um objeto para cada hashtag incluída no corpo do Post, e incluirá um array vazio se não houver hashtags.

O operador PowerTrack `#` é usado para fazer correspondência com o atributo `text`. O operador `has:hashtags` fará correspondência se houver pelo menos um item no array.

|     |     |     |
| :--- | :--- | :--- |
| Campo | Tipo | Descrição |
| indices | Array de Int | Um array de inteiros indicando os deslocamentos dentro do texto do Post onde a hashtag começa e termina. O primeiro inteiro representa a posição do caractere # na string de texto do Post. O segundo inteiro representa a posição do primeiro caractere após a hashtag. Portanto, a diferença entre os dois números será o comprimento do nome da hashtag mais um (pelo caractere ‘#’). Exemplo:<br /><br />"indices":\[32,38] |
| text | String | Nome da hashtag, sem o caractere ‘#’ inicial. Exemplo:<br /><br />"text":"nodejs" |

### Objeto de mídia  

A seção `entities` conterá um array `media` com um único objeto de mídia se algum objeto de mídia tiver sido “anexado” ao Post. Se nenhuma mídia nativa tiver sido anexada, não haverá um array `media` em `entities`. Pelos seguintes motivos, a seção `extended_entities` deve ser usada para processar a mídia nativa do Post:
\+ O campo `type` de mídia sempre indicará “photo”, mesmo quando um vídeo ou um GIF estiver anexado ao Post.
\+ Embora seja possível anexar até quatro fotos, apenas a primeira será listada na seção `entities`.

Um valor de _crop_ indica que a mídia foi cortada para se ajustar a uma resolução específica. Exemplo:<br /><br />"resize":"crop" |

### Formatação de URL de mídia de foto

A mídia de foto no X pode ser carregada em diferentes tamanhos. É recomendável carregar a menor imagem cujo tamanho ainda seja suficiente para se ajustar a uma determinada área de visualização. Para carregar diferentes tamanhos, o [Objeto Size](#size) e [media\_url](#media) (ou [media\_url\_https](#media)) precisam ser combinados em um formato específico. Usaremos o objeto de exemplo de [entidade de mídia](#entitiesobject) já fornecido para construir uma URL de mídia de foto.

As `media_url` ou `media_url_https`, por si só, podem ser carregadas, o que fará com que a variante média seja carregada por padrão. No entanto, é preferível fornecer uma URL de mídia de foto totalmente formatada sempre que possível.

Há três partes em uma URL de mídia de foto:

|     |     |
| :--- | :--- |
| Base URL | A URL base é a URL da mídia sem a extensão do arquivo.<br /><br />Por exemplo:  <br /><br />"media\_url\_https": "https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg",<br /><br />A URL base é:  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq_ |
| Format | O formato é o tipo de foto no qual a imagem está codificada. Os formatos possíveis são _jpg_ ou _png_, fornecidos como a extensão da URL da mídia.<br /><br />Por exemplo:  <br /><br />"media\_url\_https": "https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg",<br /><br />O formato é: _jpg_ |
| Name | O nome é o campo do tamanho a ser carregado.<br /><br />Por exemplo:<br /><br />  <br /> "sizes":   <br />   "thumb":   <br />     "h": 150,  <br />     "resize": "crop",  <br />     "w": 150  <br />   ,  <br />   "large": <br />     "h": 1366,  <br />     "resize": "fit",  <br />     "w": 2048  <br />   },  <br />   "medium": <br />     "h": 800,  <br />     "resize": "fit",  <br />     "w": 1200  <br />   },  <br />   "small": <br />     "h": 454,  <br />     "resize": "fit",  <br />     "w": 680  <br />   }  <br /> }  <br />}<br /><br />O nome ao carregar a foto no tamanho grande seria: _large_ |

Pegamos essas três partes (URL base, formato e nome) e as combinamos na URL de mídia de foto a ser carregada. Existem 2 formatos para carregar imagens dessa maneira, _legacy_ e _modern_. Todos os carregamentos de imagem devem deixar de usar o formato _legacy_ e passar a usar o formato _modern_. O uso do formato _modern_ resultará em melhor taxa de acerto da CDN para quem faz a chamada, reduzindo a latência de carregamento por ser menos provável precisar gerar e carregar a mídia a partir do data center.

|     |     |
| :--- | :--- |
| Formato legado | O formato legado está obsoleto. Todos os carregamentos de mídia de foto devem migrar para o formato moderno.<br /><br />_\<base\_url>.\<format>:\<name>_  <br /><br />Por exemplo:  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg:large  <br />_ |
| Formato moderno | O formato moderno para carregamento de fotos foi estabelecido no X em 2015 e é o padrão de fato desde 2017. Todos os carregamentos de mídia de foto devem migrar para esse formato.<br /><br />_\<base\_url>?format=\<format>\&name=\<name>_  <br /><br />Por exemplo:  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq?format=jpg\&name=large  <br />_<br /><br />Observação: os itens na string de consulta do URL de mídia de foto estão em ordem alfabética. Se o carregamento de mídia vier a incluir itens de consulta adicionais, a ordenação alfabética continuará sendo necessária. Por exemplo, se houvesse um novo item de consulta hipotético chamado _preferred\_format_, ele viria após _format_ e _name_ na string de consulta. |

## Detalhes sobre Retweets e Tweets com citação

Do ponto de vista da API do X, retweets e tweets com citação são tipos especiais de posts que contêm o post original como um objeto incorporado. Assim, objetos de retweet e de tweet com citação são “pais” de um post “original” filho (e, portanto, têm o dobro do tamanho). Retweets possuem um objeto de nível superior "retweeted\_status", e tweets com citação possuem um objeto "quoted\_status". Para manter a consistência, esses objetos de nível superior de retweet e de tweet com citação também têm uma propriedade text e entidades associadas. No entanto, as entidades no nível superior podem diferir das entidades fornecidas pelo objeto “original” incorporado. No caso de retweets, um novo texto é adicionado antes do corpo do post original. Para posts com citação, um novo texto é adicionado após o corpo do post.

Em geral, a melhor prática é recuperar o texto, as entidades, o autor original e a data a partir do post original em retweeted\_status sempre que isso existir. A exceção é ao obter entidades do X que fazem parte do conteúdo adicional da citação. Veja abaixo mais detalhes e dicas.

### Retweets

Um detalhe importante sobre Retweets é que nenhuma _entidade_ adicional do X pode ser adicionada ao Post. Os usuários não podem adicionar hashtags, URLs ou outros detalhes quando fazem Retweet. No entanto, o atributo de texto do Retweet (de nível superior) é composto pelo texto do Post original com “RT @username: ” no início.

Em alguns casos, especialmente em contas com nomes de usuário longos, a combinação desses novos caracteres com o corpo do Post original pode facilmente exceder o limite de 140 caracteres do texto do Post. Para preservar a exibição e o armazenamento baseados em 140 caracteres, o corpo de nível superior trunca o final do corpo do Post e adiciona reticências (“…”). Consequentemente, algumas entidades de nível superior posicionadas ao final do Post original podem estar incorretas ou ausentes, por exemplo, no caso de uma hashtag ou URL truncadas.

<div id="retweets">
  ### Quote Tweets
</div>

Quote Tweets foram introduzidos em 2016 e diferem dos Retweets porque, quando você “cita” um Post, está adicionando novo conteúdo “por cima” de um Post compartilhado. Esse novo conteúdo pode incluir praticamente tudo o que um Post original pode ter, incluindo novo texto, hashtags, menções e URLs.

Quote Tweets podem conter mídia nativa (fotos, vídeos e GIFs) e aparecerão no objeto entities.

Como entidades do X podem ser adicionadas, as entidades do Quote provavelmente serão diferentes das entidades originais.

Neste exemplo, uma nova URL e hashtag foram posicionadas ao final do Quote Tweet.

#### Introdução

Se um Post contiver mídia nativa (compartilhada pela interface do usuário do Post, em vez de por meio de um link para outro lugar), também haverá uma seção extended\_entities. Para qualquer mídia nativa (foto, vídeo ou GIF), extended\_entities é a fonte de metadados preferida por vários motivos. Atualmente, até quatro fotos podem ser anexadas a um Post. Os metadados entities conterão apenas a primeira foto (até 2014, apenas uma foto podia ser incluída), enquanto a seção extended\_entities incluirá todas as fotos anexadas. Com mídia nativa, outra limitação dos metadados entities.media é que o tipo de mídia sempre indicará ‘photo’, mesmo nos casos em que a mídia anexada seja um vídeo ou GIF animado. O tipo real de mídia é especificado no atributo extended\_entities.media\[].type e pode ser _photo_, _video_ ou _animated\_gif_. Por esses motivos, se você estiver trabalhando com mídia nativa, os metadados extended\_entities são a melhor opção.

Todos os Posts com fotos, vídeos e GIFs animados anexados incluirão um objeto JSON `extended_entities`. O objeto `extended_entities` contém um único array `media` de objetos `media` (veja a seção `entities` para o dicionário de dados correspondente). Nenhum outro tipo de entidade, como hashtags e links, é incluído na seção `extended_entities`. O objeto `media` na seção `extended_entities` é idêntico em estrutura ao incluído na seção `entities`.

Posts podem ter apenas um tipo de mídia anexado. Para fotos, é possível anexar até quatro. Para vídeos e GIFs, é possível anexar apenas um. Como os metadados de `type` na seção `extended_entities` indicam corretamente o tipo de mídia (‘photo’, ‘video’ ou ‘animated\_gif’) e dão suporte a até 4 fotos, essa é a fonte de metadados preferida para mídia nativa.

```
{
      "extended_entities": {
        "media": [

        ]
      }
    }
```

Quando um anunciante opta por limitar a reprodução de vídeo apenas a plataformas de propriedade e operação do X, o objeto `video_info` será substituído por um objeto `additional_media_info`.

O `additional_media_info` conterá informações adicionais de mídia fornecidas pelo publicador, como `title`, `description` e a flag `embeddable`. O conteúdo de vídeo fica disponível apenas para clientes oficiais do X quando `embeddable=false`. Nesse caso, todas as URLs de vídeo fornecidas no payload serão baseadas no X, para que o usuário possa abrir o vídeo em uma propriedade do X ao clicar no link.

## Objetos de dados do Enterprise Activity Streams

<Note>
  Interessado em saber mais sobre como o formato de dados Activity Streams se relaciona com o formato da API do X v2?
</Note>

Confira nosso guia de comparação: [Activity Streams em comparação com a API do X v2](/pt-BR/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2)

<Note>
  Atenção: é altamente recomendável usar o formato [Enriched Native](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) para as APIs de dados enterprise.

  - O formato Enriched Native inclui todos os metadados adicionados desde 2017, como [metadados de enquetes](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata), e métricas adicionais como reply\_count e quote\_count.

  - O formato Activity Streams não é atualizado com novos metadados ou enriquecimentos desde a [atualização de contagem de caracteres](https://blog.x.com/official/en_us/topics/product/2017/Giving-you-more-characters-to-express-yourself.html), em 2017.
</Note>

### Objeto Activity

Activity Streams é uma tradução do esquema de objetos do [formato de dados original](https://developer.x.comhttps://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) do X, criada pela Gnip para “normalizar o formato” dos dados de Post e de outras mídias sociais, usando o [Activity Base Schema de terceiros descrito aqui](https://activitystrea.ms/head/activity-schema.html). Os Posts são normalizados no esquema de Activity Streams, incluindo os tipos de objeto note, person, place e service como objetos aninhados. Os Posts podem incluir outros objetos de atividade de Post aninhados para Retweets ou outros, como twitter\_quoted\_status e long\_object.

O tipo de objeto de nível básico “activity” é semelhante ao [objeto de nível básico do Post](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) do formato nativo enriquecido. Exemplos de payloads no formato Activity Streams podem ser encontrados [aqui](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#activity-streams-payload-examples).

Em diversos casos, um objeto de Post incluirá outros Posts aninhados. ### Objeto Gnip

O objeto gnip, no formato Activity Streams, contém os metadados adicionados pelos enriquecimentos ativos, bem como a indicação das regras de correspondência para a atividade.

#### Dicionário de dados

|     |     |     |
| :--- | :--- | :--- |
| **Campo** | **Tipo** | **Descrição** |
| matching\_rules | array | Contém um array de objetos de regra de correspondência que indicam a regra com a qual a atividade corresponde.  <br /> **"matching\_rules": \[**<br /><br />\*\*      \{**<br /><br />**        "tag": null,**<br /><br />**"id":\*\* 1026514022567358500\*\*,**<br /><br />**"id\_str":\*\* "1026514022567358464"<br /><br />\*\*      }**<br /><br />**    ]\*\* |

### Introdução\*\*

Em sua essência, o X é uma rede de comunicação pública, em tempo real e global. Desde 2006, a evolução do X tem sido impulsionada tanto pelos padrões de uso e convenções dos usuários quanto por novos recursos e aprimoramentos de produto. Se você usa dados do X para pesquisa histórica, entender a linha do tempo dessa evolução é importante para encontrar Posts de interesse no arquivo de dados.

O X foi lançado como um aplicativo móvel simples via SMS e evoluiu para uma plataforma de comunicação abrangente. Uma plataforma com um conjunto completo de APIs. As APIs sempre foram um pilar da rede do X. A [primeira API foi lançada pouco depois do lançamento do X](https://blog.x.com/2006/introducing-the-twitter-api). Quando a marcação geográfica de Posts foi introduzida pela primeira vez em 2009, ela foi disponibilizada por meio de uma [Geo API](https://blog.x.com/2009/think-globally-tweet-locally) (e, posteriormente, a possibilidade de “marcar geograficamente” um Post foi integrada à interface do usuário do X.com). Hoje, as APIs do X impulsionam a rede de comunicação bidirecional que se tornou fonte de notícias de última hora e de compartilhamento de informações. As oportunidades de construir em cima desse canal de comunicação global e em tempo real são infinitas.

O X disponibiliza duas APIs históricas que fornecem acesso a todos os Posts publicamente disponíveis: Historical PowerTrack e a [Full-Archive Search API](https://developer.x.com/en/docs/x-api/v1/tweets/search/overview). Ambas as APIs fornecem um conjunto de _operadores_ usados para consultar e coletar Posts de interesse. Esses operadores fazem correspondência com uma variedade de atributos associados a cada Post — centenas de atributos, como o conteúdo de texto do Post, o nome da conta do autor e os links compartilhados no Post. Os Posts e seus atributos são codificados em JSON, um formato comum de intercâmbio de dados baseado em texto. Assim, à medida que novos recursos eram introduzidos, novos atributos JSON surgiam e, normalmente, novos operadores de API eram introduzidos para corresponder a esses atributos. Se seu caso de uso inclui a necessidade de _ouvir_ o que o mundo disse no X, quanto melhor você entender quando os operadores passaram a ter metadados JSON com os quais fazer correspondência, mais eficazes serão seus filtros históricos do PowerTrack.

A seguir, apresentaremos alguns conceitos-chave que preparam o terreno para entender como atualizações nos metadados de Posts afetam a identificação do seu sinal de dados de interesse.

### Conceitos principais\*\*

#### De convenções de usuários a _objetos de primeira classe_ no X

Usuários do X introduziram organicamente novos padrões de comunicação, hoje fundamentais, na rede do X. Um exemplo seminal é a hashtag, hoje quase universalmente usada em todas as redes sociais. As hashtags foram introduzidas como uma forma de organizar conversas e tópicos. Em uma rede com centenas de milhões de mensagens por dia, ferramentas para encontrar Posts de interesse são essenciais, e as hashtags se tornaram um método fundamental. Logo após o crescimento do uso de hashtags, elas receberam status e suporte oficiais do X. Quando as hashtags se tornaram um _“objeto de primeira classe”_, isso significou muitas coisas. Significou que as hashtags passaram a ser clicáveis/pesquisáveis na interface do usuário do X.com. Também significou que as hashtags passaram a integrar a família de _entities_ do X, junto com @mentions, mídias anexadas, símbolos de ações e links compartilhados. Essas entidades são convenientemente codificadas em um array JSON pré-processado, o que facilita para desenvolvedores processá-las, analisá-las e armazená-las.

Retweets são outro exemplo de convenções criadas pelos usuários que se tornaram objetos oficiais. O Retweet surgiu como uma forma de “encaminhar” conteúdo para outras pessoas. Começou como um processo manual de copiar/colar um Post e prefixá-lo com o padrão “RT @”. Esse processo acabou sendo automatizado por meio de um novo botão de Retweet, com novos metadados JSON. Nascia o Retweet “oficial”. Outros exemplos incluem “mentions”, compartilhamento de mídia e links da web, e compartilhamento de uma localização com seu Post. Cada um desses padrões de uso resultou em novos recursos na interface do usuário do [x.com](https://x.com/), novo JSON de suporte e, portanto, novas maneiras de fazer correspondência em Posts. Todos esses atributos fundamentais de Post resultaram em Operadores do PowerTrack usados para correspondê-los.

<div id="from-user-conventions-to-x-first-class-objects">
  #### Metadados de Posts, mutabilidade, atualizações e atualidade
</div>

Embora mensagens de Post possam ter até um número fixo de caracteres, a descrição JSON de um Post consiste em mais de 100 atributos. Atributos como quem publicou, em que horário, se é um Post original ou um Retweet, e um array de objetos de primeira classe, como hashtags, menções e links compartilhados. Para a conta que publicou, há um objeto User (ou Actor) com uma variedade de atributos que fornecem o perfil do usuário e outros metadados da conta. Perfis incluem uma breve descrição biográfica, um local de residência (texto livre), idioma preferido e um link opcional para um site.

Alguns metadados da conta nunca mudam (por exemplo, ID numérico do usuário e data de criação), alguns mudam lentamente ao longo do tempo, enquanto outros atributos mudam com mais frequência. As pessoas mudam de emprego e de cidade. As empresas atualizam suas informações. Ao coletar Posts históricos, é importante entender que alguns metadados são _como eram quando o Post foi publicado_ e outros metadados são _como estão quando a consulta é enviada_.

Em todas as APIs históricas, a descrição do perfil do usuário, o nome de exibição e os atributos de “home” do perfil são atualizados para os valores no momento da consulta.

<div id="post-metadata-mutability-updates-and-currency">
  #### Mídia “nativa”
</div>

O X.com e os apps móveis do X permitem adicionar fotos e vídeos a um Post clicando em um botão e navegando pela sua galeria. Agora que essas ações são recursos de primeira classe, os vídeos e fotos compartilhados dessa forma são chamados de mídia “nativa”.

Muitos operadores de consulta funcionam com esses recursos “nativos”, incluindo `has:videos`, `has:images` e `has:media`. Eles corresponderão apenas a conteúdo de mídia compartilhado por meio de recursos do X. Para corresponder a outras mídias hospedadas fora da plataforma do X, use operadores que atuem sobre metadados de URL.

Então, antes de entrarmos nos detalhes dos produtos Historical PowerTrack e Busca em todo o arquivo, vamos fazer um tour de como o X, como produto e plataforma, evoluiu ao longo do tempo.

**Linha do tempo do X**

Abaixo você encontrará uma linha do tempo selecionada do X. A maioria dessas atualizações do X, de alguma forma, afetou fundamentalmente o comportamento dos usuários, o conteúdo JSON dos Posts, os operadores de consulta ou os três. Considerando o X como uma plataforma de API, os eventos a seguir afetaram, de algum modo, os payloads JSON usados para codificar Posts. Por sua vez, esses detalhes em JSON afetam como as APIs históricas do X fazem a correspondência.

**Dicas de filtragem**

Estar familiarizado com a linha do tempo do X sobre quando e como novos recursos foram adicionados pode ajudar você a criar consultas mais eficazes. Aqui, uma consulta significa um filtro ou regra aplicado pelas APIs históricas do X ao arquivo de Posts, usando operadores do PowerTrack para corresponder ao JSON do Post. Um exemplo é o operador `lang:`, usado para corresponder Posts em um idioma específico. O X fornece um serviço de classificação de idioma (com suporte a mais de 50 idiomas), e as APIs do X fornecem esses metadados no JSON gerado para cada Post. Assim, se um Post for escrito em espanhol, o atributo JSON “lang” é definido como “es”. Portanto, se você criar um filtro com a cláusula `lang:es`, ele corresponderá apenas a mensagens de Post classificadas como espanhol.

As informações da linha do tempo também podem ajudar a interpretar melhor os dados de Post recebidos. Digamos que você estivesse pesquisando o compartilhamento de conteúdo sobre as Olimpíadas de Verão de 2008 e 2012. Se você aplicasse apenas o operador `is:retweet` para corresponder a Retweets, nenhum dado corresponderia em 2008. No entanto, em 2012 provavelmente haveria milhões de Retweets. A partir disso, você poderia concluir erroneamente que, em 2008, os Retweets não eram uma convenção de uso ou que simplesmente ninguém deu Retweet sobre aquelas Olimpíadas. Como os Retweets se tornaram um objeto de primeira classe em 2009, você precisa adicionar uma cláusula de regra `“RT @”` para ajudar a identificá-los em 2008.

Tanto os Retweets quanto a classificação de idioma do Post são exemplos de atributos de Post com longa história e muitos detalhes de produto. A seguir, discutiremos mais detalhes desses e de outros tipos de atributos importantes para corresponder e entender os Dados do X.

<div id="2017">
  #### Reconhecendo falsos negativos
</div>

Ao escrever filtros, é importante ter em mente que todos os Operadores que atuam sobre metadados têm “datas de criação”. Se você criar um filtro com um Operador que depende de metadados introduzidos após a publicação do Post, terá um falso negativo. Por exemplo, suponha que você esteja interessado em todos os Posts que mencionam ‘snow’ e compartilham um vídeo. Se você criar uma regra com o Operador `has:videos`, que corresponde a Posts com vídeos nativos, essa cláusula não corresponderá a nenhum Post anterior a 2015.

No entanto, o compartilhamento de vídeos já era comum no X muito antes de 2015. Antes disso, os usuários compartilhavam links para vídeos hospedados em outros lugares, mas em 2015 o X incorporou novos recursos de “compartilhar vídeo” diretamente na plataforma. Para encontrar esses Posts anteriores de interesse, você incluiria uma cláusula de regra como `url:"youtube.com"`.

Observe que, com as APIs de Busca, há casos em que metadados foram “preenchidos retroativamente” quando o índice foi reconstruído. Um bom exemplo são os $cashtags, que passaram a ser amplamente usados para discutir símbolos de ações em 2009. Após a introdução do operador $cashtag em 2015, o índice de Busca foi reconstruído e, nesse processo, a entidade de símbolo foi extraída de todos os corpos de Post, inclusive nos primeiros anos (2006), quando `$` era usado principalmente como gíria; “I hope it $now$ $oon!”.

<div id="recognizing-false-negatives">
  #### Identificando e filtrando atributos de Post importantes para o seu caso de uso
</div>

Alguns metadados, como IDs numéricos de contas do X, existem desde o primeiro dia (e são um exemplo de metadados de conta que nunca mudam). Outros metadados só foram introduzidos bem depois que o X começou, em 2006. Exemplos de novos metadados incluem informações de Retweets, localizações de Posts, títulos e descrições de URLs e mídia “nativa”. Abaixo estão alguns dos tipos mais comuns de atributos de Post que foram profundamente afetados por essas atualizações da plataforma X.

O comportamento de filtragem/correspondência desses atributos depende, na maioria dos casos, de qual API histórica de Posts é usada. Para ajudar a determinar qual produto é o mais adequado para sua pesquisa e caso de uso, os detalhes dos atributos fornecidos abaixo incluem informações de produto em alto nível.

<div id="identifying-and-filtering-on-post-attributes-important-to-your-use-case">
  #### Perfis do X
</div>

Como o X é, em sua essência, um canal global de comunicação em tempo real, pesquisas com dados de Post costumam enfatizar quem está se comunicando. Muitas vezes é útil saber onde um usuário do X considera ser sua casa. Saber que a bio de uma conta inclui menções a interesses e hobbies também pode levar você a Posts relevantes. É muito comum querer monitorar Posts de contas de interesse. Os atributos de perfil são fundamentais para todos esses casos de uso.

Toda conta no X tem um Perfil que inclui metadados como @handle do X, nome de exibição, uma breve bio, localização de residência (texto livre inserido pelo usuário), número de seguidores e muitos outros. Alguns atributos nunca mudam, como o ID numérico do usuário e a data de criação da conta. Outros costumam mudar no dia a dia, de semana a semana ou de mês a mês, como o número de Posts publicados e o número de contas seguidas e de seguidores. Outros atributos da conta também podem mudar a qualquer momento, mas tendem a mudar com menos frequência: nome de exibição, localização de residência e bio.

O payload JSON de cada Post inclui metadados de _perfil da conta_ do autor do Post. Se for um Retweet, também inclui metadados de perfil da conta que publicou o Post original.

A mutabilidade dos metadados de perfil de um Post depende inteiramente do produto histórico utilizado. As Search APIs fornecem Posts históricos com as configurações de perfil _como estão no momento da recuperação_. No Historical PowerTrack, o perfil é _como estava no momento em que o Post foi publicado_, exceto para dados anteriores a 2011. Para Posts anteriores a 2011, os metadados de perfil refletem o perfil como era em setembro de 2011.

<div id="x-profiles">
  #### Post original e Retweets
</div>

Retweets são outro exemplo de convenções impulsionadas pelos usuários que se tornaram objetos oficiais. O retweet surgiu como uma forma de “encaminhar” conteúdo para outras pessoas. Começou como um processo manual de copiar/colar um Post e adicioná-lo com o padrão “RT @” no início. Esse processo acabou sendo automatizado por meio de um novo botão Retweet, com novos metadados JSON. Nasceu o Retweet “oficial” e a ação de retweetar se tornou um evento de Post de primeira classe. Junto com o novo botão Retweet, foram introduzidos novos metadados, como o payload completo do Post original.

Saber se um Post é original ou compartilhado é um critério comum de filtragem. Em alguns casos, apenas conteúdo original é necessário. Em outros, o engajamento do Post é prioritário, então os Retweets são fundamentais. O operador PowerTrack `is:retweet` permite incluir ou excluir Retweets. Ao coletar dados anteriores a agosto de 2009, é preciso ter duas estratégias para identificar Retweets (ou não). Antes de agosto de 2009, é necessário verificar a própria mensagem do Post, usando correspondência de frase exata, para ocorrências do padrão “RT @ ”. Para períodos após agosto de 2009, o operador `is:retweet` está disponível.

<div id="original-post-and-retweets">
  #### Classificações de idioma de Post
</div>

O idioma em que um Post é escrito é de interesse comum. O idioma do Post pode ajudar a inferir a localização de um Post e, muitas vezes, apenas um idioma específico é necessário para análise ou exibição. (Perfis do X também têm uma configuração de idioma preferido.)

Para filtrar com base na classificação de idioma de um Post, os produtos históricos do X ([Search API](https://developer.x.com/en/docs/x-api/v1/tweets/search/overview/enterprise) e Historical PowerTrack) são bastante diferentes. Quando o arquivo do Search foi criado, todos os posts foram preenchidos retroativamente com a classificação de idioma do X. Portanto, o operador `lang:` está disponível para todo o arquivo de posts. Com o Historical PowerTrack, os metadados de classificação de idioma do X estão disponíveis no arquivo a partir de 26 de março de 2013. 

<div id="post-language-classifications">
  #### Georreferenciamento de Posts
</div>

Ser capaz de identificar onde um Post foi publicado (ou seja, georreferenciá-lo) é importante para muitos casos de uso. Existem três métodos principais para georreferenciar Posts:

- Referências geográficas no conteúdo do Post
- Posts com geotag adicionada pelo usuário
- Localização “home” do perfil da conta definida pelo usuário

##### Referências geográficas em uma mensagem de Post

Fazer a correspondência de referências geográficas na mensagem do Post, embora muitas vezes seja o método mais desafiador por depender de conhecimento local, é uma opção para todo o arquivo de Posts. Aqui está um exemplo de correspondência georreferenciada de 2006 para a área de San Francisco com base no filtro “golden gate”:

https://x.com/biz/statuses/28311

<div id="geographical-references-in-a-post-message">
  ##### Posts com geotag definidos pelo usuário
</div>

Em novembro de 2009, o X introduziu a Post Geotagging API, que permitia marcar Posts com uma localização exata. Em junho de 2010, o X introduziu o X Places, que representa uma área geográfica na escala de estabelecimento, bairro ou cidade. Aproximadamente 1–2% dos Posts são geotagueados por qualquer um dos métodos.

O histórico de geotags disponível depende da Historical API que você está usando. Com as Search APIs, a capacidade de começar a corresponder Posts usando alguns Geo Operators começou em março de 2010 e, para outros, em fevereiro de 2015. Se você estiver usando o Historical PowerTrack, a georreferência começa em 1º de setembro de 2011. Quando o arquivo do Historical PowerTrack foi criado, todas as geotags anteriores a essa data não foram incluídas.

<div id="posts-geo-tagged-by-the-user">
  ##### Localização “home” do perfil da conta definida pelo usuário
</div>

Todos os usuários do X podem definir a Localização do Perfil, indicando sua localização principal. Milhões de usuários do X fornecem essa informação, o que aumenta significativamente a quantidade de geodados no X Firehose. Esses metadados de localização são uma string livre, gerada pelo usuário e não normalizada. Aproximadamente 30% das contas têm metadados de Perfil Geo que podem ser resolvidos ao nível de país.

Assim como em geo de Post, os métodos de correspondência e os períodos disponíveis dependem da API Histórica que você estiver usando. O Historical PowerTrack permite que os usuários realizem seu próprio casamento personalizado nessas strings em formato livre. Para facilitar esse processo, o X também oferece o [Profile Geo Enrichment](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo), que realiza a geocodificação quando possível, fornecendo metadados normalizados e operadores correspondentes. Os operadores de Perfil Geo estão disponíveis tanto no Historical PowerTrack quanto nas Search APIs. No Historical PowerTrack, esses metadados de Perfil Geo estão disponíveis a partir de junho de 2014. Nas Search APIs, esses metadados estão disponíveis a partir de fevereiro de 2015.

#### Links e mídia compartilhados

Compartilhar links de páginas da web, fotos e vídeos sempre foi um caso de uso fundamental no X. No início, todas essas ações envolviam incluir um link de URL na própria mensagem do Post. Em 2011, o X integrou o compartilhamento de fotos diretamente em sua interface do usuário. Em 2016, foram adicionados vídeos nativos.

Dado esse histórico, há uma variedade de Operadores de filtragem usados para fazer correspondência com esse tipo de conteúdo. Há um conjunto de Operadores que verificam se os Posts têm links, fotos e vídeos compartilhados. Além disso, como a maioria das URLs compartilhadas no X é encurtada para consumir menos caracteres de um Post (por exemplo, geradas por serviços como bitly ou tinyurl), o X fornece enriquecimentos de dados que geram uma URL completa e expandida que pode ser usada na correspondência. Por exemplo, se você quiser corresponder Posts que incluam links sobre o X e sistemas de alerta precoce, um filtro que faça referência a “severe weather communication” corresponderia a um Post contendo esta URL [http://bit.ly/1XV1tG4](http://bit.ly/1XV1tG4).

Em março de 2012, o [enriquecimento de URL expandida](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls) foi introduzido. Antes disso, os payloads de Post incluíam apenas a URL conforme fornecida pelo usuário. Portanto, se o usuário incluísse uma URL encurtada, poderia ser desafiador corresponder às URLs (expandidas) de interesse. Com o Historical PowerTrack e as Search APIs, esses metadados estão disponíveis a partir de março de 2012.

Em julho de 2016, foi introduzido o enriquecimento de URL aprimorada. Essa versão aprimorada fornece o título e a descrição HTML de um site no payload do Post, juntamente com Operadores para correspondência com esses elementos. Com o Historical PowerTrack, esses metadados ficam disponíveis em julho de 2016. Com as Search APIs, esses metadados começam a surgir em dezembro de 2014.

Em setembro de 2016, o X introduziu “anexos nativos”, nos quais um link compartilhado ao final não é contabilizado no limite de 140 caracteres do Post. Ambos os enriquecimentos de URL ainda se aplicam a esses links compartilhados.

Para outros detalhes específicos do produto sobre filtragem de URLs, consulte os artigos correspondentes para mais informações.
