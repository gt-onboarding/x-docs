<div id="comparing-x-apis-posts-lookup-endpoints">
  ## Comparando os endpoints de consulta de posts da API do X
</div>

Os endpoints de consulta de posts da v2 substituem os endpoints padrão da v1.1 [GET statuses/lookup](https://developer.x.com/en/docs/twitter-api/v1/tweets/post-and-engage/api-reference/get-statuses-lookup) e [GET statuses/show](https://developer.x.com/en/docs/twitter-api/v1/tweets/post-and-engage/api-reference/get-statuses-show-id). Este guia é destinado a desenvolvedores que estão migrando dessas versões mais antigas para a API do X v2.

---

  ## Tabela de comparação de endpoints
</div>

| Descrição | Standard v1.1 | API do X v2 |
|:-------------|:---------------|:----------|
| **Métodos HTTP compatíveis** | `GET` | `GET` |
| **Domínio do host** | `https://api.x.com` | `https://api.x.com` |
| **Caminho do endpoint** | `/1.1/statuses/show.json`, `/1.1/statuses/lookup.json` | `/2/tweets` |

| **[Formato JSON](/pt-BR/x-api/fundamentals/data-dictionary) do post** | Formato Standard v1.1 | [Formato da API do X v2](/pt-BR/x-api/fundamentals/data-dictionary), determinado pelos parâmetros `fields` e `expansions` (não compatível retroativamente com v1.1) |

| **Compatível com o campo `conversation_id`** |  | ✔ |

| **Exige credenciais de um [App de desenvolvedor](/pt-BR/resources/fundamentals/developer-apps) associado a um [Project](pt-BR/resources/fundamentals/projects)** |  | ✔ |

<div id="standard-v11-compared-to-x-api-v2">
  ## Padrão v1.1 comparado à API do X v2
</div>

Se você tem trabalhado com os endpoints padrão v1.1 GET statuses/show e GET statuses/lookup, este guia ajudará você a entender as semelhanças e diferenças entre os endpoints de consulta de Posts do padrão e da API do X v2.

Você também pode se interessar pela nossa [ferramenta visual de migração de formato de dados](/pt-BR/x-api/migrate/data-format-migration) para ajudar você a ver rapidamente as diferenças entre o [formato de dados da API do X v1.1](/pt-BR/x-api/fundamentals/data-dictionary) e o [formato da API do X v2](/pt-BR/x-api/fundamentals/data-dictionary).

- **Semelhanças**
  - Contexto de usuário OAuth 1.0a
  - Limites de Posts por solicitação
  - Suporte ao histórico de edição do Post e aos metadados

- **Diferenças**
  - URLs de endpoints
  - Requisitos de App e Projeto
  - Formato de dados da resposta
  - Parâmetros de solicitação

<div id="similarities">
  ### Semelhanças
</div>

<div id="oauth-10a-user-context-authentication-method">
  #### Método de autenticação OAuth 1.0a com contexto de usuário
</div>

O endpoint padrão oferece suporte a [OAuth 1.0a com contexto de usuário](/pt-BR/resources/fundamentals/authentication), enquanto o novo endpoint de consulta de posts da API do X v2 oferece suporte tanto a OAuth 1.0a com contexto de usuário quanto a [OAuth 2.0 somente para apps](/pt-BR/resources/fundamentals/authentication). Portanto, se você usava anteriormente um dos endpoints padrão v1.1 de consulta de posts, pode continuar usando o mesmo método de autenticação ao migrar para a versão da API do X v2.

A autenticação somente para apps provavelmente é a maneira mais fácil de começar. Para saber como gerar um token de acesso de app, consulte [este guia de OAuth 2.0 somente para apps](/pt-BR/resources/fundamentals/authentication).

<div id="posts-per-request-limits">
  #### Limites de Posts por solicitação
</div>

O endpoint v1.1 [GET statuses/lookup](https://developer.x.com/en/docs/twitter-api/v1/tweets/post-and-engage/api-reference/get-statuses-lookup) permite especificar até 100 Posts por solicitação. Isso também se aplica ao endpoint GET /tweets. Para solicitar o máximo de 100 Posts, use o parâmetro `ids` como parâmetro de consulta, com uma lista de [IDs de Post](/pt-BR/resources/fundamentals/x-ids) separada por vírgulas.

**Suporte a histórico de edição de Posts e metadados**

Ambas as versões fornecem metadados que descrevem qualquer histórico de edição. Consulte as referências da API de consulta de Posts e a [página de fundamentos sobre edição de Posts](/pt-BR/x-api/fundamentals/edit-posts) para mais detalhes.

<div id="differences">
  ### Diferenças
</div>

<div id="endpoint-urls">
  #### URLs de endpoints
</div>

- **Endpoints padrão v1.1:**
  - `https://api.x.com/1.1/statuses/show`
  - `https://api.x.com/1.1/statuses/lookup`

- **Endpoint da API do X v2:**
  - `https://api.x.com/2/tweets`
  - `https://api.x.com/2/tweets/:id`

<div id="app-and-project-requirements">
  #### Requisitos de App e Projeto
</div>

Os endpoints da API do X v2 exigem credenciais de um [App de desenvolvedor](/pt-BR/resources/fundamentals/developer-apps) associado a um [Projeto](/pt-BR/resources/fundamentals/projects) para autenticação. Os endpoints da API do X v1.1 podem usar credenciais de Apps autônomos ou de Apps associados a um Projeto.

<div id="response-data-format">
  #### Formato dos dados da resposta
</div>

Uma diferença significativa entre as versões de endpoints padrão v1.1 e API do X v2 é como os campos são selecionados no payload.

Nos endpoints padrão, muitos campos de resposta são incluídos por padrão, com a opção de usar parâmetros para especificar campos adicionais.

A API do X v2, porém, retorna apenas os campos `id` e `text` do Post por padrão. Campos e objetos adicionais exigem o uso dos parâmetros [fields](/pt-BR/x-api/fundamentals/fields) e [expansions](/pt-BR/x-api/fundamentals/expansions). Os campos expandidos são retornados em um objeto `includes` dentro da resposta, que pode ser associado ao objeto principal do Post por meio da correspondência de IDs.

Para mais detalhes sobre o uso de fields e expansions, consulte o [guia sobre como usar fields e expansions](/pt-BR/x-api/fundamentals/data-dictionary). Um [guia de migração de formato de dados](/pt-BR/x-api/fundamentals/fields) também mapeia os campos da v1.1 padrão para os campos mais recentes da v2.

Além disso, a API do X v2 introduz novos modelos JSON para objetos, incluindo os objetos Post e [user](/pt-BR/x-api/fundamentals/fields):

- Endpoints padrão retornam objetos Post em um array `statuses`, enquanto a API do X v2 usa um array `data`.
- Retweeted e Quoted Tweets na API do X v2 substituem a terminologia “statuses”.
- Nova terminologia como `like` substitui termos como `favorites` e `favourites`.
- Atributos sem valor (por exemplo, `null`) não são incluídos nos payloads da API do X v2.

O objeto Post na API do X v2 inclui novos campos, como:

- `conversation_id`
- Dois novos campos de [annotations](/pt-BR/x-api/fundamentals/post-annotations) (`context` e `entities`)
- Novos campos de [metrics](/pt-BR/x-api/fundamentals/metrics)
- Campo `reply_setting` que indica quem pode responder a um determinado Post

<div id="request-parameters">
  #### Parâmetros de solicitação
</div>

Os seguintes parâmetros padrão da v1.1 têm equivalentes na API do X v2:

| Padrão | API do X v2 |
|:----------|:----------|
| `id`     | `ids`    |

Alguns parâmetros padrão da v1.1 **não** são compatíveis na API do X v2:

| Padrão            | Comentário                                                                                                                      |
|:---------------------|:------------------------------------------------------------------------------------------------------------------------------|
| `tweet_mode`        | Substituído pelos recursos de fields e expansions.                                                                             |
| `trim_user`         | Substituído por fields e expansions. Use a expansion `author_id` e `user.fields` para obter dados do usuário.                 |
| `include_my_retweet`| Fornece o ID do Post de origem para Posts retuitados pelo usuário autenticado.                                                 |
| `include_entities`  | Use fields e expansions para controlar as entidades no payload.                                                                |
| `include_ext_alt_text` | Adiciona o campo `ext_alt_text` na entidade de mídia se houver texto alternativo.                                           |
| `include_card_uri`  | Adiciona `card_uri` quando um card de anúncios está anexado.                                                                   |
| `map`               | Retorna o ID do Post e a mensagem de erro para Posts indisponíveis na API do X v2, em vez de campos definidos como nulos na v1.1. |

<div id="curl-requests">
  ### Requisições cURL
</div>

As requisições cURL a seguir mostram endpoints padrão da v1.1 e seus equivalentes na v2. Substitua `ACCESS_TOKEN` no cabeçalho pelo token de acesso do seu app. Para endpoints v2, o token deve pertencer a um [App de desenvolvedor](/pt-BR/resources/fundamentals/developer-apps/overview) dentro de um [Projeto](/pt-BR/resources/fundamentals/projects).

Os payloads de resposta da v1.1 diferem dos da v2. Na v2, você pode solicitar diferentes campos com os parâmetros [fields](/pt-BR/x-api/fundamentals/fields) e [expansions](/pt-BR/x-api/fundamentals/expansions).

**Endpoints padrão v1.1 `GET statuses/lookup` e v2 `GET /tweets`**

```bash
curl --request GET \
  --url 'https://api.x.com/1.1/statuses/lookup.json?id=1460323737035677698%2C1460323743339741184' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```

```
  curl --request GET \
  --url 'https://api.x.com/2/tweets?ids=1460323737035677698%2C1460323743339741184&tweet.fields=created_at&expansions=author_id&user.fields=created_at' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```

**Endpoints padrão v1.1 `GET statuses/show/:id` e v2 `GET /tweets/:id`**

```
curl --request GET \
  --url 'https://api.x.com/1.1/statuses/show.json?id=1460323737035677698' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```

```
curl --request GET \
  --url 'https://api.x.com/2/tweets/1460323737035677698?tweet.fields=created_at&expansions=author_id&user.fields=created_at' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```
