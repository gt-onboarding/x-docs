<div id="standard-v11-compared-to-x-api-v2">
  ## Comparação entre o padrão v1.1 e a API do X v2
</div>

Se você tem trabalhado com os endpoints padrão v1.1 GET statuses/show e GET statuses/lookup, este guia ajudará você a entender as semelhanças e diferenças entre os endpoints de consulta de posts do padrão e da API do X v2.

Você também pode se interessar pela nossa [ferramenta visual de migração de formato de dados](/pt-BR/x-api/migrate/data-format-migration), que ajuda a ver rapidamente as diferenças entre o [formato de dados da API do X v1.1](/pt-BR/x-api/fundamentals/data-dictionary) e o [formato da API do X v2](/pt-BR/x-api/fundamentals/data-dictionary).

- **Semelhanças**
  - Contexto de usuário OAuth 1.0a
  - Limites de posts por solicitação
  - Suporte ao histórico de edição e metadados do post

- **Diferenças**
  - URLs de endpoint
  - Requisitos de app e projeto
  - Formato de dados da resposta
  - Parâmetros da solicitação

<div id="similarities">
  ### Semelhanças
</div>

<div id="oauth-10a-user-context-authentication-method">
  #### Método de autenticação OAuth 1.0a no contexto do usuário
</div>

O endpoint padrão oferece suporte a [OAuth 1.0a User Context](/pt-BR/resources/fundamentals/authentication), enquanto o novo endpoint de busca de posts da API do X v2 oferece suporte tanto a OAuth 1.0a User Context quanto a [OAuth 2.0 App-Only](/pt-BR/resources/fundamentals/authentication). Portanto, se você utilizava anteriormente um dos endpoints padrão de busca de posts da v1.1, pode continuar usando o mesmo método de autenticação ao migrar para a versão da API do X v2.

A autenticação App-Only provavelmente é a maneira mais simples de começar. Para saber como gerar um App Access Token, consulte [este guia de OAuth 2.0 App-Only](/pt-BR/resources/fundamentals/authentication).

<div id="posts-per-request-limits">
  #### Limites de Posts por solicitação
</div>

O endpoint v1.1 [GET statuses/lookup](https://developer.x.com/en/docs/twitter-api/v1/tweets/post-and-engage/api-reference/get-statuses-lookup) permite especificar até 100 Posts por solicitação. Isso também se aplica ao endpoint GET /tweets. Para especificar 100 Posts, use o parâmetro `ids` como parâmetro de consulta com uma lista separada por vírgulas de [IDs de Post](/pt-BR/resources/fundamentals/x-ids).

**Suporte ao histórico de edição e metadados de Post**

Ambas as versões fornecem metadados que descrevem qualquer histórico de edição. Consulte as referências da API de consulta de Post e a [página de fundamentos de edição de Posts](/pt-BR/x-api/fundamentals/edit-posts) para mais detalhes.

<div id="differences">
  ### Diferenças
</div>

<div id="endpoint-urls">
  #### URLs de endpoints
</div>

- **Endpoints padrão v1.1:**
  - `https://api.x.com/1.1/statuses/show`
  - `https://api.x.com/1.1/statuses/lookup`

- **Endpoint da API do X v2:**
  - `https://api.x.com/2/tweets`
  - `https://api.x.com/2/tweets/:id`

<div id="app-and-project-requirements">
  #### Requisitos de App e Projeto
</div>

Os endpoints da API do X v2 exigem credenciais de um [App de desenvolvedor](/pt-BR/resources/fundamentals/developer-apps) associado a um [Projeto](/pt-BR/resources/fundamentals/projects) para autenticação. Os endpoints da API do X v1.1 podem usar credenciais de Apps autônomos ou de Apps associados a um Projeto.

<div id="response-data-format">
  #### Formato de dados da resposta
</div>

Uma diferença significativa entre as versões de endpoints padrão v1.1 e API do X v2 é a forma como os campos são selecionados no payload.

Nos endpoints padrão, muitos campos de resposta são incluídos por padrão, com a opção de usar parâmetros para especificar campos adicionais.

Já a API do X v2 entrega apenas os campos `id` e `text` do Post por padrão. Campos e objetos adicionais exigem o uso dos parâmetros [fields](/pt-BR/x-api/fundamentals/fields) e [expansions](/pt-BR/x-api/fundamentals/expansions). Os campos expandidos vêm em um objeto `includes` dentro da resposta, que pode ser associado ao objeto principal do Post fazendo a correspondência dos IDs.

Para saber mais sobre o uso de fields e expansions, consulte o [guia de como usar fields e expansions](/pt-BR/x-api/fundamentals/data-dictionary). Um [guia de migração de formato de dados](/pt-BR/x-api/fundamentals/fields) também mapeia os campos padrão da v1.1 para os novos campos da v2.

Além disso, a API do X v2 apresenta novos designs JSON para objetos, incluindo os objetos Post e [user](/pt-BR/x-api/fundamentals/data-dictionary#user):

- Endpoints padrão retornam objetos Post em um array `statuses`, enquanto a API do X v2 usa um array `data`.
- Retweets e Tweets com citação na API do X v2 substituem a terminologia “statuses”.
- Nova terminologia como `like` substitui termos como `favorites` e `favourites`.
- Atributos sem valores (por exemplo, `null`) não são incluídos nos payloads da API do X v2.

O objeto Post na API do X v2 inclui novos campos, como:

- `conversation_id`
- Dois novos campos de [annotations](/pt-BR/x-api/fundamentals/post-annotations) (`context` e `entities`)
- Novos campos de [metrics](/pt-BR/x-api/fundamentals/metrics)
- Campo `reply_setting` indicando quem pode responder a um determinado Post

<div id="request-parameters">
  #### Parâmetros de requisição
</div>

Os seguintes parâmetros padrão da v1.1 têm equivalentes na API do X v2:

| Padrão | API do X v2 |
|:----------|:----------|
| `id`     | `ids`    |

Alguns parâmetros padrão da v1.1 **não** são compatíveis na API do X v2:

| Padrão              | Comentário                                                                                                                     |
|:---------------------|:-------------------------------------------------------------------------------------------------------------------------------|
| `tweet_mode`        | Substituído pela funcionalidade de fields e expansions.                                                                        |
| `trim_user`         | Substituído por fields e expansions. Use a expansion `author_id` e `user.fields` para dados do usuário.                        |
| `include_my_retweet`| Fornece o ID do Post de origem para Posts retuitados pelo usuário autenticado.                                                 |
| `include_entities`  | Use fields e expansions para controlar as entidades no payload.                                                                |
| `include_ext_alt_text` | Adiciona o campo `ext_alt_text` na entidade de mídia se houver texto alternativo.                                            |
| `include_card_uri`  | Adiciona `card_uri` quando um card de anúncios estiver anexado.                                                                 |
| `map`               | Retorna o ID do Post e a mensagem de erro para Posts indisponíveis na API do X v2, em vez de campos nulos na v1.1.             |

<div id="curl-requests">
  ### Requisições cURL
</div>

As seguintes requisições cURL mostram endpoints padrão da v1.1 e seus equivalentes na v2. Substitua `ACCESS_TOKEN` no cabeçalho pelo token de acesso do seu app. Para endpoints v2, o token deve pertencer a um [App de desenvolvedor](/pt-BR/resources/fundamentals/developer-apps/overview) dentro de um [Projeto](/pt-BR/resources/fundamentals/projects/overview).

Os payloads de resposta da v1.1 diferem dos da v2. Na v2, você pode solicitar campos diferentes com os parâmetros [fields](/pt-BR/x-api/fundamentals/fields) e [expansions](/pt-BR/x-api/fundamentals/expansions).

**Endpoints padrão da v1.1 `GET statuses/lookup` e da v2 `GET /tweets`**

```bash
curl --request GET \
  --url 'https://api.x.com/1.1/statuses/lookup.json?id=1460323737035677698%2C1460323743339741184' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```

```
  curl --request GET \
  --url 'https://api.x.com/2/tweets?ids=1460323737035677698%2C1460323743339741184&tweet.fields=created_at&expansions=author_id&user.fields=created_at' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```

**Endpoints padrão da v1.1 `GET statuses/show/:id` e da v2 `GET /tweets/:id`**

```
curl --request GET \
  --url 'https://api.x.com/1.1/statuses/show.json?id=1460323737035677698' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```

```
curl --request GET \
  --url 'https://api.x.com/2/tweets/1460323737035677698?tweet.fields=created_at&expansions=author_id&user.fields=created_at' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```
