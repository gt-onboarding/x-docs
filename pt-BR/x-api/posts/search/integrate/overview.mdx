<div id="authentication">
  #### Autenticação
</div>

Todos os endpoints da API do X v2 exigem que você autentique suas requisições com um conjunto de credenciais, também conhecidas como chaves e tokens. Você pode usar OAuth 1.0a User Context, OAuth 2.0 App-Only ou OAuth 2.0 Authorization Code com PKCE para autenticar suas requisições ao endpoint de busca recente. Você deve usar OAuth 2.0 App-Only ao utilizar o endpoint de busca em todo o arquivo.

[OAuth 2.0 App-Only](/pt-BR/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3) requer apenas que você envie um [OAuth 2.0 App Access Token](/pt-BR/resources/fundamentals/authentication#bearer-token-also-known-as-app-only) com sua requisição. Você pode gerar um App Access Token diretamente em um App de desenvolvedor ou gerá-lo usando o endpoint [POST oauth2/token](/pt-BR/resources/fundamentals/authentication#post-oauth2-token).

[OAuth 1.0a User Context](/pt-BR/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) exige que você utilize suas API Keys, Access Tokens de usuário e alguns outros parâmetros para [criar um cabeçalho de autorização](https://developer-staging.x.com/resources/fundamentals/authentication/authorizing-a-request), que você então enviará com sua requisição. Os Access Tokens devem estar associados ao usuário em nome de quem você está fazendo a requisição. Se quiser gerar um conjunto de Access Tokens para outro usuário, ele deve autorizar seu App usando o [fluxo OAuth em 3 etapas](https://developer-staging.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens).

Observe que o OAuth 1.0a pode ser difícil de usar. Se você não estiver familiarizado com esse método de autenticação, recomendamos usar uma [biblioteca](/pt-BR/x-api/tools-and-libraries/overview), uma ferramenta como o Postman ou OAuth 2.0 para autenticar suas requisições. Se você quiser solicitar um Post ou métricas privadas desses endpoints, será necessário usar OAuth 1.0a User Context ou OAuth 2.0 Authorization Code com PKCE, o que garantirá que você tenha as permissões adequadas do usuário proprietário desse conteúdo.

[OAuth 2.0 Authorization Code com PKCE](/pt-BR/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) permite maior controle sobre o escopo de uma aplicação e fluxos de autorização em vários dispositivos. O OAuth 2.0 permite escolher escopos específicos e granulares que concedem permissões específicas em nome de um usuário.

Para habilitar o OAuth 2.0 no seu App, você deve ativá-lo nas configurações de autenticação do seu App, localizadas na seção de configurações do App no portal do desenvolvedor.

<Note>
  **Observação**

  Se você estiver solicitando os seguintes [campos](/pt-BR/x-api/fundamentals/fields), é necessário OAuth 1.0a User Context ou OAuth 2.0 Authorization Code:

  - tweet.fields.non\_public\_metrics
  - tweet.fields.promoted\_metrics
  - tweet.fields.organic\_metrics
  - media.fields.non\_public\_metrics
  - media.fields.promoted\_metrics
  - media.fields.organic\_metrics
</Note>

####

Portal do desenvolvedor, Projetos e Apps de desenvolvedor

Para usar qualquer endpoint da API do X v2, você deve [criar uma conta de desenvolvedor](https://developer.x.com/en/portal/petition/essential/basic-info), configurar um Projeto nessa conta e criar um App de desenvolvedor dentro desse Projeto. As chaves e os tokens desse App de desenvolvedor funcionarão nesses endpoints de busca.

Você pode usar chaves e tokens de um Projeto com qualquer [nível de acesso](/pt-BR/x-api/getting-started/about-x-api) para fazer solicitações ao endpoint de busca recente. No entanto, será necessário usar um Projeto com nível de acesso Pro ou Enterprise para fazer solicitações ao endpoint de busca em todo o arquivo. Se você tiver [acesso Enterprise](/pt-BR/x-api/getting-started/getting-access), contará com funcionalidades adicionais, incluindo mais operadores disponíveis e consultas mais longas.
 

<div id="rate-limits">
  #### Limites de taxa
</div>

Todos os dias, muitos milhares de desenvolvedores fazem solicitações à API do X. Para ajudar a gerenciar o volume, [limites de taxa](/pt-BR/x-api/fundamentals/rate-limits) são aplicados a cada endpoint, limitando o número de solicitações que cada desenvolvedor pode fazer em nome de um app ou de um usuário autenticado.

Há diferentes limites de taxa aplicados a esses endpoints, dependendo de qual método de autenticação está sendo usado. Os limites de taxa no nível do app aplicam-se a um app que faz solicitações usando OAuth 2.0 App-Only, enquanto o limite de taxa no nível do usuário aplica-se a solicitações feitas em nome do usuário autorizador específico usando OAuth 1.0a User Context ou OAuth 2.0 Authorization Code com PKCE. Esses dois limites de taxa baseiam-se na frequência de solicitações dentro de uma janela de 15 minutos.

Por exemplo, um app usando autenticação OAuth 2.0 App-Only para fazer solicitações ao endpoint de busca recente pode fazer 450 solicitações (incluindo as de paginação) em um período de 15 minutos. Esse mesmo app, no mesmo período de 15 minutos e com dois usuários autenticados diferentes (usando OAuth 1.0a User Context ou OAuth 2.0 Authorization Code com PKCE), pode fazer até 180 solicitações (incluindo as de paginação) ao endpoint de busca recente para cada usuário autenticado.
 

<div id="fields-and-expansions">
  #### Campos e expansões
</div>

A API do X v2 permite selecionar exatamente quais dados você quer receber da API usando [campos](/pt-BR/x-api/fundamentals/fields) e [expansões](/pt-BR/x-api/fundamentals/expansions). O parâmetro expansions permite expandir objetos referenciados no payload. Por exemplo, este endpoint permite solicitar objetos de enquete (poll), local (place), mídia (media) e outros usando o parâmetro expansions.

Os parâmetros fields permitem selecionar exatamente quais campos, dentro dos diferentes objetos de dados, você deseja receber. Por padrão, o objeto principal de Post retornado por esses endpoints inclui os campos id e text (além de edit\_history\_tweet\_ids para Posts criados após o lançamento desse recurso). Para receber campos adicionais como author\_id ou public\_metrics, você precisará solicitá-los explicitamente usando os parâmetros fields. Alguns campos importantes que você pode considerar usar na sua integração são nossos dados de enquete, [métricas](/pt-BR/x-api/fundamentals/metrics), [anotações de Post](/pt-BR/x-api/fundamentals/post-annotations) e [ID de conversa](/pt-BR/x-api/fundamentals/conversation-id).

Adicionamos um guia sobre como [usar campos e expansões em conjunto](/pt-BR/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) ao nosso [dicionário de dados da API do X v2](/pt-BR/x-api/fundamentals/data-dictionary).
 

<div id="metrics">
  #### Métricas
</div>

Os endpoints da API do X v2 permitem solicitar métricas diretamente dos objetos retornados, desde que você inclua os [campos](/pt-BR/x-api/fundamentals/fields) apropriados na sua solicitação.

Há algumas limitações nas métricas de Post das quais você deve estar ciente, especificamente relacionadas à privacidade do usuário e aos seguintes campos de resposta:

- tweet.fields.non\_public\_metrics
- tweet.fields.promoted\_metrics
- tweet.fields.organic\_metrics
- media.fields.non\_public\_metrics
- media.fields.promoted\_metrics
- media.fields.organic\_metrics

Os campos citados incluem dados de métricas privadas, o que significa que você deve estar autorizado pelo autor do Post para recuperar esses dados em nome dele ao usar o endpoint de busca recente; ou seja, é necessário usar [OAuth 1.0a User Context](/pt-BR/resources/fundamentals/authentication). Como esse método de autenticação só pode ser usado na busca recente, não será possível recuperar essas métricas pelo endpoint de busca em todo o arquivo.

Por exemplo, para receber `non_public_metrics` dos Posts do usuário de ID 1234, você precisará incluir na sua solicitação tokens de acesso associados a esse usuário. Você pode fazer com que os usuários autorizem seu app e receber um conjunto de tokens de acesso associados a eles usando o [fluxo OAuth com 3 etapas](/pt-BR/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow).

Todas as `non_public_metrics`, `organic_metrics` e `promoted_metrics` estão disponíveis apenas para Posts criados nos últimos 30 dias. Isso significa que, ao solicitar os campos indicados, os resultados serão ajustados automaticamente para incluir apenas Posts dos últimos 30 dias.

Se esses campos citados forem solicitados, somente os Posts publicados pelo usuário autenticado serão retornados; todos os demais Posts receberão uma mensagem de erro.
 

<div id="building-search-queries">
  #### Construindo consultas de busca
</div>

O principal recurso desses endpoints é o uso de uma única consulta de busca para filtrar os Posts entregues a você. Essas consultas são compostas por operadores que fazem correspondência com atributos do Post e do usuário, como palavras-chave da mensagem, hashtags e URLs. Os operadores podem ser combinados em consultas com lógica booleana e parênteses para refinar o comportamento de correspondência da consulta.

Você pode usar nosso guia sobre [como criar uma consulta](/pt-BR/x-api/posts/search/integrate/build-a-query) para saber mais.

<div id="pagination">
  #### Paginação
</div>

Esses endpoints utilizam paginação para que as respostas sejam retornadas rapidamente. Quando houver mais resultados do que o possível em uma única resposta (até 100 posts para busca recente e 500 para busca em todo o arquivo), será necessário paginar. Use o parâmetro max\_results para definir quantos resultados serão retornados por página e o parâmetro pagination\_token para obter a próxima página de resultados. Saiba mais em nosso [guia de paginação](/pt-BR/x-api/fundamentals/pagination).

<div id="post-caps">
  #### Limites de posts
</div>

Os endpoints de Search Posts têm um limite de quantidade de Posts que podem retornar em um determinado mês, independentemente da paginação.

Independentemente do endpoint de busca que você use, os Posts retornados contarão para os [Limites de posts](/pt-BR/x-api/fundamentals/post-cap) no nível do Projeto. O uso é exibido no portal do desenvolvedor, e o “mês” começa no dia de renovação da sua assinatura mostrado no [dashboard do portal do desenvolvedor](https://developer.x.com/en/portal/dashboard).

**Edições de Post**

Posts elegíveis para edição podem ser editados até cinco vezes nos 30 minutos após a publicação do Post original. Os endpoints de busca sempre retornarão a versão mais recente do Post. Se você solicitar apenas Posts que foram publicados há 30 minutos ou mais, sempre receberá a versão final do Post. No entanto, se você tiver um caso de uso quase em tempo real e estiver consultando Posts publicados nos últimos 30 minutos, esses Posts podem ter sido editados depois que você os recebeu. Esses Posts podem ser reidratados com a busca ou com o endpoint de Post Lookup para confirmar seu estado final. Para saber mais sobre como funcionam as edições de Post, consulte a página [Fundamentos de edições de Post](/pt-BR/x-api/fundamentals/edit-posts).
