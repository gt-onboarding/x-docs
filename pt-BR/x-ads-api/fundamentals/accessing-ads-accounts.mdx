<div id="user-accounts-vs-ad-accounts">
  # Contas de usuário vs Contas de anúncios
</div>

Há dois tipos diferentes de contas envolvidas no uso da API de Anúncios do X: contas de anúncios e contas de usuário do X. Em toda a documentação da API de Anúncios, o termo “conta” geralmente se refere à conta de anúncios.

- Contas de anúncios são registradas em **business.x.com** e identificadas na API por account\_id. Elas se conectam diretamente a fontes de financiamento e utilizam conteúdo de uma ou mais contas de usuário do X como “usuários promovíveis”. Cada conta de anúncios pode conceder permissão a uma ou mais contas de usuário do X. A conta de anúncios, ou “conta atual”, é representada em quase todas as URLs executadas como um parâmetro em linha :account\_id.
- Contas de usuário do X (como @AdsAPI) são identificadas por user\_id na API de Anúncios. Uma ou mais dessas contas podem ser associadas a uma conta de anúncios. A conta de usuário do X autenticada que faz solicitações na API é chamada de “usuário atual”. Uma listagem das contas de anúncios às quais o usuário atual tem acesso pode ser encontrada com [GET accounts](/pt-BR/x-ads-api/campaign-management#accounts). “Usuários promovíveis” são @handles do X que podem ser promovidos por uma conta de anúncios específica. Para mais detalhes, consulte [Obtaining Ads Account Access](/pt-BR/x-ads-api/introduction).

<div id="methods-for-ad-account-access">
  ## Métodos de acesso à conta de Anúncios
</div>

Há dois métodos que você pode usar para fazer solicitações à API de Anúncios para a conta de um anunciante:

1. Fazer solicitações em nome de um anunciante (**recomendado**)
2. Fazer solicitações usando sua conta que recebeu acesso à conta de um anunciante, por exemplo, uma [agência](https://business.x.com/en/blog/agencies-manage-multiple-Twitter-Ads-accounts.html) que gerencia várias contas.

Este documento é uma breve visão geral das diferenças entre essas opções e deve ser usado em conjunto com nossos outros recursos, como o [FAQ de login multiusuário](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html).

Conforme descrito em [Autorizando uma solicitação](/pt-BR/resources/fundamentals/authentication#authorizing-a-request#authorizing-a-request), todas as solicitações à API de Anúncios exigem cabeçalhos de autorização usando [OAuth 1.0a](/pt-BR/resources/fundamentals/authentication) com um token de acesso obtido via fluxo [OAuth de 3 partes](/pt-BR/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)[OAuth](/pt-BR/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow). Seu aplicativo precisará implementar um fluxo OAuth baseado na web para [obter tokens de acesso](/pt-BR/resources/fundamentals/authentication#oauth-1-0a-2). Desenvolvedores da API de Anúncios nunca devem solicitar que os nossos anunciantes no X compartilhem suas credenciais de login.

Por padrão, cada [aplicativo de desenvolvedor do X](/pt-BR/resources/fundamentals/developer-apps) contém um **token de acesso** estático que pode ser usado para fazer solicitações à API de Anúncios para a conta que é proprietária do aplicativo. Essas credenciais são ideais para casos de uso com uma única conta, sem exigir um fluxo OAuth de 3 partes ou [baseado em PIN](/pt-BR/resources/fundamentals/authentication#pin-based-authorization). Se você não estiver acessando outra conta de Anúncios do X, use essas credenciais [de usuário único](/pt-BR/resources/fundamentals/authentication#oauth-1-0a-2) em vez das etapas a seguir.

<div id="levels-of-access">
  ## Níveis de acesso
</div>

<div id="app-level-permissions">
  ### Permissões no nível do app
</div>

Cada usuário terá um nível de acesso conforme solicitado na [solicitação de acesso à API de Anúncios](/pt-BR/x-ads-api/getting-started/step-by-step-guide):

- **Somente conversão**: O usuário tem acesso aos endpoints de [Conversões em dispositivos móveis](/pt-BR/x-ads-api/measurement/mobile-conversions) e [Conversões na Web](/pt-BR/x-ads-api/measurement/web-conversions), com permissões de [leitura e gravação](/pt-BR/resources/fundamentals/developer-apps).
- **Acesso padrão**: O usuário tem acesso aos endpoints de [Analytics](/pt-BR/x-ads-api/analytics), [Gerenciamento de campanhas](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/overview), [Criativos](https://developer.x.com/en/docs/twitter-ads-api/creatives/overview), [Públicos personalizados](https://developer.x.com/en/docs/twitter-ads-api/audiences/overview) e [Conversões](/pt-BR/x-ads-api/measurement/web-conversions), com permissões de [leitura e gravação](/pt-BR/resources/fundamentals/developer-apps).

**Observação:** Desenvolvedores da API de Anúncios que solicitaram acesso antes de julho de 2023 podem ter níveis de acesso e permissões diferentes e podem estar limitados a cinco tokens OAuth. Consulte nosso guia sobre [aumentar o acesso](/pt-BR/x-ads-api/getting-started/increasing-access) para obter acesso a endpoints adicionais ou remover limites de tokens para apps existentes.

<div id="ad-account-level-permissions">
  ### Permissões no nível da Conta de Anúncios
</div>

Cada usuário com acesso a uma Conta de Anúncios terá uma permissão específica no nível da conta: **Administrador da conta**, **Gerente de anúncios**, **Analista de campanhas**, **Analista orgânico** e **Gerente de criação**; consulte **business.x.com** para a documentação mais recente sobre permissões no nível da conta. Os aplicativos devem obter as permissões do usuário atualmente autenticado por meio do endpoint da API [Acesso do usuário autenticado](/pt-BR/x-ads-api/campaign-management#authenticated-user-access) para determinar quais endpoints da API e recursos de Anúncios estão disponíveis para ele.

**Observação:** Quaisquer tokens de usuário usados com a [API de Conversões](/pt-BR/x-ads-api/measurement/web-conversions) devem ser de usuários com permissões no nível da conta de **Administrador da conta** ou **Gerente de anúncios**.                                                                                                                                              

<div id="methods-of-obtaining-access-tokens">
  ## Formas de obter tokens de acesso
</div>

<div id="1-obtain-an-advertisers-user-access-token">
  ### 1. Obtenha o token de acesso de um anunciante (Usuário)
</div>

Há dois métodos para obter um **token de acesso** de anunciante. O método mais comum é por meio de um [fluxo OAuth de 3 etapas](/pt-BR/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) diretamente na sua interface web. Aplicativos que não têm uma interface pública acessível a anunciantes podem implementar um processo OAuth baseado em [PIN](/pt-BR/resources/fundamentals/authentication#pin-based-authorization). Após o usuário concluir o fluxo de 3 etapas, seu aplicativo terá credenciais para fazer solicitações à conta de Anúncios dele via API.

Obter credenciais de usuário por meio do fluxo OAuth é o método que **recomendamos enfaticamente** para que a maioria dos desenvolvedores da API de Anúncios do X obtenha acesso a uma conta de anunciante. Isso permite que você chame a API em nome de um usuário e execute ações como esse usuário. Esses tokens não expiram, mas podem ser revogados pelo usuário a qualquer momento.

<div id="2-obtain-your-developer-access-token">
  ### 2. Obtenha seu token de acesso (Developer)
</div>

Esta opção requer que o anunciante [conceda acesso ao seu @username](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html) (ou @usernames) à conta de Anúncios do X dele pela interface do X em business.x.com. **Tokens de acesso** obtidos pelo fluxo OAuth de 3 partes (3-legged) para a sua conta poderão acessar a conta de Anúncios do X do anunciante.

Isso permite chamar a API usando os tokens OAuth do seu próprio @username, em vez dos tokens OAuth do anunciante. A principal distinção nesta opção é que você só poderá criar Posts somente promovidos se a permissão de delegação/composição de Post tiver sido concedida ao seu @username.

Para obter acesso para criar Posts somente promovidos em nome do usuário promovível `FULL` na conta, você também deve receber acesso para criar Posts nesse fluxo. Isso habilitará o acesso por meio da permissão `TWEET_COMPOSER` no endpoint [GET accounts/:account\_id/authenticated\_user\_access](/pt-BR/x-ads-api/campaign-management#accounts).

<div id="differences-between-these-methods">
  ## Diferenças entre esses métodos
</div>

|     |     |     |
| :--- | :--- | :--- |
|     | Token OAuth do anunciante (usuário) | Token OAuth (desenvolvedor)  <br /> (@username adicionado para separar a conta) |
| Acessar conta de anúncios | ✔   | ✔   |
| Criar posts em nome do usuário | ✔   | ✔\*  |
| Gerenciar campanhas | ✔   | ✔   |
| Acessar análises | ✔   | ✔   |
| Criar cards em nome do usuário | ✔   | ✔   |
| Capacidade do desenvolvedor de acessar  <br />via interface do X Ads |     | ✔   |
| [Limites de taxa](/pt-BR/x-api/enterprise-gnip-2.0/fundamentals/rate-limits) | Distintos por anunciante | Distintos por conta de anunciante |

  ## Exemplo de caso de uso
</div>

<div id="advertisers-access-token-via-oauth-3-legged-web-flow">
  ### Token de acesso do anunciante via fluxo web OAuth de 3 etapas
</div>

O fluxo padrão é baseado na web e usa o fluxo OAuth de [autorização em 3 etapas](/pt-BR/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow). As capturas de tela descritas aqui fazem parte de um exemplo cujo código-fonte você pode consultar em [https://github.com/xdevplatform/twauth-web](https://github.com/xdevplatform/twauth-web).

Em algum momento do seu aplicativo, você precisará redirecionar para o X para autorizar seu aplicativo.

  <img src="https://cdn.cms-twdigitalassets.com/content/dam/developer-twitter/images/twauth-web-2.png" alt="image0" />
</Frame>

Quando você redirecionar para o X com o token de solicitação, será exibido ao usuário um prompt para autorizar seu aplicativo.

<Frame>
  <img src="https://cdn.cms-twdigitalassets.com/content/dam/developer-twitter/images/twauth-web-3.png" alt="image1" />
</Frame>

Após autorizar seu aplicativo, o usuário será redirecionado para a URL de callback informada quando você gerou o token de solicitação. Você usará isso para obter o token de acesso permanente desse usuário e armazená-lo localmente.
