<div id="overview">
  ## Visão geral
</div>

Todos os meses, fazemos mudanças e lançamos diversos novos recursos na API de Anúncios do X. Essas mudanças quase sempre são retrocompatíveis; no entanto, costumamos ter algumas alterações incompatíveis por ano. Recebemos feedback de desenvolvedores sobre os desafios que nosso ritmo acelerado de mudanças na API de Anúncios impõe aos seus ciclos de desenvolvimento ao implementar novos recursos, lidar com descontinuações e testar mudanças. Queremos melhorar a experiência do desenvolvedor ao usar nossa plataforma de anúncios, e por isso introduzimos o conceito de versionamento dos nossos endpoints.

Algumas definições de conceitos sobre os quais falamos:

**Versão**: Refere-se ao número de versão presente no caminho da URL de qualquer solicitação da API de Anúncios, por exemplo: GET /{version_number}/accounts. Esse estilo de versionamento é conhecido como versionamento por URI.

**Alterações incompatíveis**: Alterações incompatíveis são quaisquer mudanças que exigem esforço de desenvolvimento para manter a funcionalidade existente. Isso inclui recursos usados para investigar as mudanças necessárias, determinar recursos/endpoints a serem descontinuados e a implementação final de todas essas mudanças. Exemplos de alterações incompatíveis incluem:

- Remover um parâmetro da solicitação/resposta da API

- Modificar o nome de quaisquer parâmetros ou endpoints

- Alterar a representação de valores (preview\_url → card\_uri)

- Alterar o comportamento de endpoints (por exemplo, estatísticas assíncronas vs síncronas)

- Adicionar/alterar parâmetros opcionais ou obrigatórios (por exemplo, tornar name um campo obrigatório na solicitação)

**Descontinuação**: Versões ou produtos descontinuados deixarão de ter suporte e recomenda-se que os desenvolvedores parem de usar essas APIs.

**Encerramento**: Quando um produto ou API é encerrado, o conjunto correspondente de endpoints não ficará mais acessível via API.

<div id="versioning-strategy">
  ## Estratégia de versionamento
</div>

Os principais princípios da estratégia são:

1. Todas as mudanças incompatíveis serão agrupadas em uma nova versão

2. As descontinuações de versões existentes, sempre que uma nova versão for anunciada, terão duração de 6 meses

3. A qualquer momento, a API permitirá solicitações de duas versões simultaneamente; no entanto, a mais antiga das duas não terá suporte

4. Para acelerar a adoção de novos produtos, eles serão lançados de forma contínua (fora da cadência de versionamento)

5. Todas as respostas da API conterão um cabeçalho `x-current-api-version`, definido para a versão atual da API, além de um cabeçalho `x-api-warn` ao chamar quaisquer endpoints de API descontinuados.

Caso haja mudanças fundamentais nos requisitos do produto que exijam uma alteração incompatível na API (por exemplo, descontinuar a segmentação por múltiplos intervalos etários), enviaremos um aviso de 90 dias para anunciar essa alteração incompatível e, após pelo menos 90 dias da publicação do aviso, a alteração incompatível será implementada

<div id="new">
  #### Novo
</div>

**Como determinar quais entidades estão ativas**

O endpoint [Entidades ativas](/pt-BR/x-ads-api/analytics#active-entities-2) indica se houve alterações nas métricas de análise das entidades de anúncios. Projetado para ser usado em conjunto com os endpoints de análise, o Entidades ativas funciona mediante a especificação de um tipo de entidade e de um intervalo de datas — de no máximo 90 dias — e retorna um array de IDs de entidades para as quais sua plataforma deve solicitar análises. IDs diferentes dos retornados não devem ser consultados em solicitações de análise subsequentes.

Este endpoint oferece suporte aos seguintes tipos de entidade: `CAMPAIGN`, `FUNDING_INSTRUMENT`, `LINE_ITEM`, `MEDIA_CREATIVE` e `PROMOTED_TWEET`.

**Métricas de `MEDIA_CREATIVE`**

Os endpoints de análise da API de Anúncios do X agora fornecem métricas para entidades Media Creative. Media Creatives são a forma como anúncios in-stream ou imagens na X Audience Platform são promovidos. A interface de Anúncios do X exibe métricas de Media Creative nas abas “In-stream videos” e “Display creatives”. Tanto os endpoints de análise [síncronos](/pt-BR/x-ads-api/analytics#get-stats-accounts-account-id) quanto [assíncronos](/pt-BR/x-ads-api/analytics#asynchronous-analytics) agora oferecem suporte ao enum de entidade `MEDIA_CREATIVE`.

**Buscar vários cards**

Aprimorando a [versão v3](https://devcommunity.x.com/t/ads-api-version-3/100732) do endpoint projetado para recuperar um único card pelo seu valor de card URI, agora é possível buscar _vários_ cards usando o endpoint [GET accounts/:account\_id/cards/all](/pt-BR/x-ads-api/creatives#cards-fetch). Assim, em vez de fazer uma solicitação para cada card, você pode recuperar até 200 cards em uma única solicitação.

Duas observações:

1. O caminho da URL agora é `accounts/:account_id/cards/all`. (O caminho anterior não está mais disponível.) Isso é para manter consistência com [o endpoint](/pt-BR/x-ads-api/creatives#cards-fetch) projetado para recuperar um card por ID.
2. O parâmetro de solicitação obrigatório agora se chama card\_uris (no plural).

**Mais flexibilidade na recuperação**

O endpoint [GET accounts/:account\_id/targeting\_criteria](/pt-BR/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria) agora oferece suporte a vários IDs de line item. O parâmetro line\_item\_ids, que aceita até 200 IDs, é obrigatório. Antes, apenas um único line item era aceito, o que dificultava a sincronização. Com essa mudança, agora é possível recuperar mais segmentações em menos tempo.

Os seguintes endpoints também passam a oferecer suporte a vários IDs de line item, embora o parâmetro line\_item\_ids seja opcional neles:

- [GET accounts/:account\_id/line\_item\_apps](https://developer.x.com/en/docs/x-ads-api/campaign-management/api-reference/line-item-apps#get-accounts-account-id-line-item-apps)
- [GET accounts/:account\_id/media\_creatives](/pt-BR/x-ads-api/campaign-management#get-accounts-account-id-media-creatives)
- [GET accounts/:account\_id/promoted\_accounts](/pt-BR/x-ads-api/campaign-management#get-accounts-account-id-promoted-accounts)
- [GET accounts/:account\_id/preroll\_call\_to\_actions](/pt-BR/x-ads-api/creatives#preroll-call-to-actions)

<div id="changed">
  #### Alterado
</div>

**Recuperação de campanhas e itens de linha em rascunho**

A maneira de recuperar campanhas e itens de linha em rascunho foi atualizada. Agora, o parâmetro with\_draft(boolean), quando definido como true, retorna entidades _tanto_ em rascunho quanto não em rascunho. Isso é consistente com a forma como entidades excluídas são recuperadas (isto é, usando with\_deleted). Antes, buscar entidades em rascunho e não em rascunho exigia pelo menos duas solicitações. Agora, isso pode ser feito em uma única chamada de API.

| **v4** | **v5** |
| :--- | :--- | :--- |
| `draft_only` | `with_draft` |     |

**Direcionamento por duração de ativação de rede**

A API de Anúncios do X corrigiu um problema de exibição em que, após adicionar o direcionamento Network Activation Duration, o tipo de direcionamento na resposta incluía o sufixo \_IN\_SEC. Ter uma referência a segundos era confuso, pois Network Activation Duration é sempre representado em meses. Esta correção torna a representação consistente e reduz a confusão.

| **v4** | **v5** |
| :--- | :--- | :--- |
| `NETWORK_ACTIVATION_DURATION_IN_SEC` | `NETWORK_ACTIVATION_DURATION` |     |

**Contagens totais e cursores**

No v5, with\_total\_count e cursor são mutuamente exclusivos. Especificar ambos em uma solicitação retornará o código de erro EXCLUSIVE\_PARAMETERS. Antes do v5, with\_total\_count era ignorado quando cursor era especificado. Essa mudança torna a relação explícita.

<div id="removed">
  #### Removidos
</div>

Três campos estão sendo removidos das respostas da API de Anúncios: preview\_url, account\_id e parent\_ids. O esforço de engenharia necessário para esses três é mínimo.

- Na v4, foi anunciado que o parâmetro de resposta preview\_url para cards era sempre nulo. A etapa final dessa migração é remover preview\_url de todas as respostas de cards.
- O atributo de resposta account\_id está sendo removido dos seguintes recursos, uma vez que o ID da conta de anúncios já está presente na URL e em request.params. (É intencional excluir instrumentos de financiamento desta lista, pois os IDs dos elementos superiores devem estar presentes nos objetos de resposta, sempre que possível, e os IDs de conta são entidades superiores aos instrumentos de financiamento.)
  - Mídia da conta
  - Provedores de eventos de app
  - Tags de eventos de app
  - Campanhas
  - Cards
  - Itens de linha
  - Usuários promovíveis
  - Critérios de segmentação
- Para solicitações [GET accounts/:account\_id/targeting\_criteria](/pt-BR/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria), não retornamos mais o campo parent\_ids, pois ele sempre foi um array vazio.

**Cards de app sem mídia**

Na v5, cards de app sem mídia não são mais compatíveis. [Anteriormente](https://devcommunity.x.com/t/deprecation-announcement-app-download-cards/87807), a capacidade de criar ou editar cards de app sem mídia foi removida. Agora, os endpoints restantes para esse recurso estão sendo descontinuados.

- Observação: isso **não** afeta cards de download de app de imagem e vídeo.

**Criação de mídia da conta**

O endpoint POST accounts/:account\_id/account\_media não está mais disponível na v5. Outros endpoints para esse recurso _não_ são afetados. O motivo dessa mudança é que, ao adicionar mídia à [Biblioteca de Mídia](/pt-BR/x-ads-api/creatives#media-library), há casos em que esses assets são adicionados _automaticamente_ como entidades de Mídia da Conta e tentar adicionar um asset já existente ao recurso de Mídia da Conta resulta em erro. Isso ocorre nos seguintes casos:

- Assets `AMPLIFY_VIDEO` adicionados à Biblioteca de Mídia são automaticamente adicionados como asset de Mídia da Conta com o tipo criativo `PREROLL`.
- Imagens com dimensões específicas adicionadas à Biblioteca de Mídia são automaticamente adicionadas como assets de Mídia da Conta. O tipo criativo (por exemplo, `INTERSTITIAL`) depende das dimensões da imagem. (Para dimensões, consulte nossa página de [Enumerações](/pt-BR/x-ads-api/introduction).)

<div id="new">
  #### Novo
</div>

_**API de Audiências**_

A nova API de Audiências foi construída sobre nosso novo backend de processamento de audiências, que oferece maior robustez e confiabilidade. Esse novo endpoint permitirá que parceiros forneçam vários tipos de identificadores para um mesmo usuário, o que nos permite usar sinais adicionais para correspondência. A documentação de referência do novo endpoint de Audiências pode ser encontrada [aqui](/pt-BR/x-ads-api/audiences). Pretendemos continuar lançando atualizações e melhorias para este produto ao longo do restante do ano.

Os seguintes endpoints deixarão de estar disponíveis na v4 devido a funcionalidade redundante (eles continuarão funcionando na v3 e serão totalmente descontinuados quando a v3 não estiver mais disponível):

- TON Upload:
  - GET accounts/:account\_id/tailored\_audience\_changes
  - GET accounts/:account\_id/tailored\_audience\_changes/:tailored\_audience\_change\_id
  - POST accounts/:account\_id/tailored\_audience\_changes
  - PUT accounts/:accounti\_d/tailored\_audiences/global\_opt\_out
- Real Time Audiences:
  - POST tailored\_audience\_memberships

Por fim, o parâmetro `list_type` será removido da requisição e da resposta em _todos_ os [endpoints de Tailored Audiences](/pt-BR/x-ads-api/audiences) na versão 4.

_**Endpoints de Configurações**_

Agora oferecemos a possibilidade de administradores de contas definirem e atualizarem configurações de usuário, conta e fiscais. As [configurações de usuário](/pt-BR/x-ads-api/campaign-management#get-accounts-account-id-user-settings-user-id) correspondem às preferências de contato específicas do usuário para uma determinada conta de anúncios. Usando o endpoint [PUT accounts/:account\_id](/pt-BR/x-ads-api/campaign-management#put-accounts-account-id), os anunciantes podem atualizar o nome da conta e o setor da empresa. Por fim, os endpoints de [configurações fiscais](/pt-BR/x-ads-api/campaign-management#get-accounts-account-id-tax-settings) permitem que anunciantes em países onde há cobrança de imposto sobre valor agregado (VAT) atualizem informações como o nome da empresa, endereço, ID de VAT e se a conta pertence ao anunciante ou a uma agência que anuncia em nome de um anunciante.

**Usando `country_code` em todos os lugares**

Como parte de um esforço maior em prol da [consistência](https://devcommunity.x.com/t/feedback-ads-api-consistency/109145) na API de Anúncios, estamos renomeando os parâmetros nos seguintes endpoints de `app_country_code` para `country_code`.

- [POST accounts/:account\_id/cards/image\_app\_download](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [PUT accounts/:account\_id/cards/image\_app\_download/:card\_id](https://developer.x.com/content/developer-twitterhttps://developer.x.com/en/docs/ads/creatives/api-reference/image-app-download#put-accounts-account-id-cards-image-app-download-card-id)
- [POST accounts/:account\_id/cards/video\_app\_download](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/video-app-download#post-accounts-account-id-cards-video-app-download)
- [PUT accounts/:account\_id/cards/video\_app\_download/:card\_id](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/video-app-download#put-accounts-account-id-cards-video-app-download-card-id)

Isso não altera o comportamento nem os valores aceitos para esses parâmetros; é apenas uma mudança de nomenclatura.

**`preview_url` sempre nulo**

Conforme prometido no anúncio da v3, todos os cards existentes agora têm um `card_uri`. Como resultado, o valor de `preview_url` será sempre `null`.

Como lembrete, associe um card a um Tweet usando o valor de `card_uri`. Veja o exemplo de solicitação a seguir.

$ twurl -X POST -H ads-api.x.com "/4/accounts/18ce54d4x5t/tweet?text=Version 4\&card\_uri=card://958225772740714496"

<div id="removed">
  #### Removidos
</div>

**Endpoints de vídeo**

Os endpoints accounts/:account\_id/videos não estarão mais disponíveis na v4. Esse endpoint se tornou obsoleto com a introdução dos endpoints da [Biblioteca de Mídia](/pt-BR/x-ads-api/creatives#get-accounts-account-id-media-library). Veja a comparação de uso a seguir.

- endpoint de vídeos da v3: `twurl -H ads-api.x.com "/3/accounts/18ce54d4x5t/videos"`

- endpoint da biblioteca de mídia da v4 para vídeos: `twurl -H ads-api.x.com "/4/accounts/18ce54d4x5t/media_library?media_type=VIDEO"`

Os endpoints da Biblioteca de Mídia têm paridade total com os endpoints de vídeos e também oferecem funcionalidades adicionais, como a capacidade de lidar com imagens e GIFs. Solicitamos que os parceiros usem exclusivamente a Biblioteca de Mídia para qualquer gerenciamento de mídia.

**`as_user_id` na visualização de Tweet**

O parâmetro `as_user_id` disponível no endpoint [GET accounts/:account\_id/tweet/preview/:tweet\_id](/pt-BR/x-ads-api/creatives#get-accounts-account-id-tweets) não será mais aceito. A visualização será sempre renderizada como o autor do Tweet.

Também estamos anunciando a descontinuação do endpoint [PUT accounts/:account\_id/targeting\_criteria](/pt-BR/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria). Por fim, a versão 3 inclui algumas pequenas mudanças em parâmetros e respostas e um limite menor para o tamanho de lotes.

Assim como na [versão 2](https://devcommunity.x.com/t/ads-api-version-2/90360), estamos dando aos parceiros **6 meses** para a transição. Em **2018-08-01**, a v2 da API de Anúncios do X será desativada. Incentivamos todos os parceiros e desenvolvedores a migrarem para a v3 o quanto antes.

**Audience Intelligence**

O Audience Intelligence fornece insights em tempo real sobre as principais hashtags, @handles e eventos mais relevantes para um determinado público do X. Por exemplo, insira Masculino 18–34 nos EUA e você verá #nintendoswitch, #cardinal e @ricegum em alta entre esse público.

Os [endpoints](/pt-BR/x-ads-api/audiences) do Audience Intelligence oferecem as seguintes funcionalidades:

- Dado um público de entrada, recuperar as principais hashtags, @handles e eventos relevantes.
- Dado um público de entrada, recuperar informações demográficas essenciais (como idade, gênero e renda familiar).
- Dada uma palavra-chave, recuperar a série temporal do volume de Tweets

**Media Library**

A [Media Library](/pt-BR/x-ads-api/creatives#get-accounts-account-id-media-library) permite gerenciar imagens, GIFs e vídeos para contas de anúncios. Esses objetos de mídia podem ser usados em Tweets e na criação de cards. Eles também podem ser reutilizados em várias criativas, eliminando a necessidade de enviar o mesmo recurso várias vezes.

Os objetos na biblioteca são identificados por uma media\_key. As media keys são valores de string no seguinte formato: 13\_875943225764098048, por exemplo. Na API de Anúncios do X, estamos avançando para o uso de media keys para toda a mídia.

**Fluxo de trabalho de card aprimorado**

Todos os nossos endpoints de cards agora oferecem suporte a media keys. Isso permite que objetos na Media Library sejam usados para criar ou atualizar cards.

Além disso, estamos introduzindo dois novos endpoints para [recuperar detalhes de cards](/pt-BR/x-ads-api/creatives#fetching-cards). Esses endpoints podem ser usados para buscar cards usados em Tweets ou Tweets agendados, por exemplo, especificando o card\_uri ou o id. Antes, isso não era possível.

<div id="other-changes">
  #### Outras mudanças
</div>

Além desses novos recursos, estamos incluindo as seguintes alterações na versão 3.

**Novo**

- A resposta do endpoint [GET insights/keywords/search](/pt-BR/x-ads-api/audiences) agora inclui o atributo related\_keywords com 30 termos relacionados às palavras-chave de entrada.

**Alterado**

- O tamanho máximo do lote de critérios de segmentação agora é 500.
- Os atributos de resposta card\_uri e preview\_url agora são mutuamente exclusivos. Quando um card tiver card\_uri, preview\_url será null. Quando um card não tiver card\_uri, apenas preview\_url será retornado.
  - Todos os cards criados a partir de 2018-01-29 terão card\_uri.
  - Até a versão 4, todos os cards existentes terão card\_uri.
- Não é mais possível criar cards com imagens 5:2. Embora os cards existentes com imagens 5:2 ainda funcionem, incentivamos os parceiros a migrar para as proporções de aspecto de melhor desempenho 1,91:1 ou 1:1 (quando compatível).

**Removido**

- O endpoint [PUT accounts/:account\_id/targeting\_criteria](/pt-BR/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria) não está mais disponível. Decidimos fazer essa alteração porque o comportamento de substituição desse endpoint causava confusão entre anunciantes e não era consistente com nossos outros endpoints PUT, que atualizam um único recurso por vez. Em vez disso, os parceiros devem usar o endpoint [POST batch/accounts/:account\_id/targeting\_criteria](/pt-BR/x-ads-api/campaign-management#post-batch-accounts-account-id-targeting-criteria), que oferece maior flexibilidade, incluindo a capacidade de adicionar e remover segmentação em uma única solicitação.
- O atributo de resposta paused não é mais retornado para instrumentos de financiamento. Em vez disso, consulte o atributo de resposta entity\_status para determinar se um instrumento de financiamento está pausado. Além disso, como paused e cancelled correspondem ao mesmo valor, cancelled também não é mais retornado na resposta.
- Removemos o parâmetro card\_id do endpoint [GET accounts/:account\_id/tweet/preview](/pt-BR/x-ads-api/creatives#get-accounts-account-id-tweets).
- Como não é possível recuperar Tweets agendados excluídos, o parâmetro with\_deleted não é mais compatível.
- O parâmetro draft\_only foi removido dos seguintes endpoints, pois essas entidades nunca podem estar em estado de rascunho:
  - [GET accounts/:account\_id/targeting\_criteria](/pt-BR/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria)
  - [GET accounts/:account\_id/promoted\_tweets](/pt-BR/x-ads-api/campaign-management#get-accounts-account-id-promoted-tweets)
  - [GET accounts/:account\_id/promoted\_accounts](/pt-BR/x-ads-api/campaign-management#get-accounts-account-id-promoted-accounts)
  - [GET accounts/:account\_id/media\_creatives](/pt-BR/x-ads-api/campaign-management#get-accounts-account-id-media-creatives)
  - [GET accounts/:account\_id/preroll\_call\_to\_actions](/pt-BR/x-ads-api/creatives#get-accounts-account-id-preroll-call-to-actions)

**Observação**

Tanto os Video Website Cards quanto os Tweets agendados agora saíram do beta. **Notas[¶](#notes "Permalink to this headline")**

  - \[novo] `campaign_daily_budget_amount_local_micro`
  - `currency`
  - `bid`
  - `objective`

  - `impressions`
  - `engagements`
  - `estimated_daily_spend_local_micro`

- O `data_type` de Tailored Audiences foi alterado de `tailored_audiences` para `tailored_audience` em todas as nossas respostas.
- Públicos personalizados compartilhados agora estão disponíveis como beta exclusivo via API. Públicos personalizados compartilhados permitem que um único público seja usado em várias contas de anúncios. Use o endpoint [POST accounts/:account\_id/tailored\_audiences/:tailored\_audience\_id/permissions](/pt-BR/x-ads-api/audiences) (e relacionados) para gerenciar as permissões de um público personalizado que você deseja compartilhar entre contas de anúncios.
- Melhorias significativas na forma de coletar [análises](/pt-BR/x-ads-api/analytics) de desempenho para contas de anunciantes:
- Para estarmos alinhados com nossas [melhores práticas](/pt-BR/x-ads-api/analytics#best-practices), passaremos a permitir a extração de no máximo **7 dias** de dados nos [endpoints de estatísticas síncronas](/pt-BR/x-ads-api/analytics#get-stats-accounts-account-id).
- Para simplificar a extração de métricas, substituímos o parâmetro `metrics` pelo novo parâmetro `metric_groups`. Os desenvolvedores devem apenas indicar quais grupos de métricas desejam receber em uma determinada solicitação.
  - Qualquer solicitação de métricas que não sejam apropriadas para uma determinada entidade será excluída da resposta e representada como valores `null`. _Essas métricas não serão contabilizadas no seu limite de custo de análises._
- A resposta foi **significativamente simplificada** e agora estará mais alinhada com a forma como as métricas são exibidas na nossa interface.
  - Anteriormente, expúnhamos uma métrica separada para cada local de Posicionamento (Tweets promovidos na Busca, Tweets promovidos nas Timelines, Tweets promovidos em Perfis e Detalhes do Tweet, X Audience Platform). Agora retornaremos um conjunto padronizado de métricas para cada um (em vez de `promoted_tweet_timeline_impressions`, `promoted_tweet_search_impressions`, `promoted_tweets_profile_impressions`, `promoted_tweets_tpn_impressions`), que passarão a ser expostas, quando solicitadas em uma das seguintes categorias, como uma métrica única, `impressions` (isso se aplica a todas as métricas):
  - `ALL_ON_TWITTER`
  - `PUBLISHER_NETWORK`
  - Ao fazer uma solicitação, você receberá uma única métrica `impressions`, o que torna mais simples a correspondência dos valores com a nossa interface.
  - Você deve fazer duas consultas para obter os dados de `ALL_ON_TWITTER` e `PUBLISHER_NETWORK`, pois eles não podem ser combinados.
- [Endpoints de estatísticas assíncronas](/pt-BR/x-ads-api/analytics#asynchronous-analytics) agora estão disponíveis, com base no feedback dos desenvolvedores!
  - Um novo conjunto de endpoints para solicitar estatísticas de forma _assíncrona_, para dados de que você não precisa imediatamente ou para extrações históricas.
  - Coloque uma tarefa de estatísticas na fila usando um novo endpoint único. Vamos coletar os dados solicitados conforme a disponibilidade de recursos.
  - Você pode consultar um endpoint de status da tarefa para verificar se os dados estão disponíveis.
  - Quando os dados estiverem disponíveis, forneceremos um ID de retirada para você baixar a resposta JSON, que refletirá a resposta do endpoint síncrono.
  - Consulte até **90 dias** de dados para até **20 entidades** em uma única tarefa.
- Confira nosso guia de migração do analytics v1, que inclui o mapeamento de métricas da v0 para a v1
- Melhorias no Sandbox \* Agora você pode [criar várias contas de anúncios de teste](/pt-BR/x-ads-api/campaign-management#post-accounts) no ambiente Sandbox. \* Agora você pode criar vários instrumentos de pagamento para uma conta de anúncios de teste, exclusivamente no ambiente Sandbox. Isso permite testar todos os nossos tipos de instrumentos de pagamento. Antes, apenas a fonte de pagamento `CREDIT_CARD` estava disponível para testes. \* Quer testar um recurso beta? Agora você pode ativar ou desativar recursos para uma conta no ambiente Sandbox, de acordo com suas necessidades de teste.
