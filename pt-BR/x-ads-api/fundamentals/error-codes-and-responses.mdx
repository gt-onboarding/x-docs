<div id="typical-response-structure">
  ## Estrutura típica de resposta
</div>

Respostas bem-sucedidas são indicadas por um código HTTP da série 200 e um payload em JSON contendo o(s) objeto(s) solicitado(s), criado(s), modificado(s) ou excluído(s), juntamente com a interpretação do servidor sobre sua solicitação.

Se você emitir uma solicitação bem-sucedida, receberá, como parte da resposta, um nó `request` que replica sua solicitação.

_Exemplo:_ [GET accounts/abcdefg/campaigns?with\_deleted=true](/pt-BR/x-ads-api/campaign-management#campaigns)

```json
{
  /* os dados da sua resposta... */,
  "request": {
    "params": {
      "account_id": "abcdefg",
      "with_deleted": "true"
    }
  }
}
```

O campo `data` nas respostas em JSON conterá os objetos específicos associados ao recurso acessado. O formato do nó `data` será um array JSON quando a resposta puder conter um ou mais resultados. Ele será retornado como um objeto JSON quando apenas um resultado for possível. Em alguns casos raros, você pode ver uma resposta que normalmente incluiria uma coleção sendo retornada como um hashmap. Nesse caso, considere que o único hashmap é um objeto do mesmo tipo especificado no campo `type`.

<div id="error-response-structure">
  ## Estrutura de resposta de erro
</div>

Respostas de erro são retornadas com um código HTTP fora da série 200. Normalmente, uma resposta JSON é incluída, mas alguns erros podem trazer outros tipos de corpo. Quando a estrutura da resposta não puder ser interpretada, considere o significado principal do código HTTP como prioritário. Por exemplo, você pode ocasionalmente ver um HTTP 404 junto com uma resposta HTML. Nesse caso, é seguro presumir que o conteúdo não foi encontrado (HTTP 404 significa “Not Found”).

Respostas de erro típicas seguem uma estrutura semelhante às respostas bem-sucedidas. A natureza do erro é informada em um nó `errors` da resposta. O nó `errors/code` indica um código de erro constante em CAPS\_CASE que você pode consumir programaticamente para tomar decisões de tratamento. O nó `errors/message` indica uma descrição (geralmente) legível por humanos do erro em inglês. Campos adicionais podem ser incluídos para indicar detalhes mais granulares sobre o erro.

```json
{
  "errors": [
    {
      "parameter": "start_time",
      "details": "invalid date",
      "code": "INVALID_PARAMETER",
      "value": "",
      "message": "Expected time, got \"\" for start_time"
    }
  ],
  "request": {
    "params": {
      "account_id": "hkk5"
    }
  }
}
```

No exemplo acima, foi feita uma solicitação a um endpoint de analytics com um valor inválido para o parâmetro `start_time`. O `errors/code` para solicitações com parâmetros inválidos é `INVALID_PARAMETER`.
