<div id="overview">
  ## Visão geral
</div>

Criativos são quaisquer entidades que podem ser promovidas em uma campanha. Posts podem incluir texto, imagens, GIFs, vídeos ou cards. Cards podem incluir imagens ou vídeos.

Criativos de imagem, GIF ou vídeo são enviados usando o [POST media/upload](/pt-BR/x-api/media/upload-media#post-media-upload) — um endpoint de upload simples que oferece suporte apenas a imagens — ou os endpoints POST media/upload (em partes). Eles podem ser adicionados a

cards:

- POST accounts/:account\_id/cards
  Tweets:
- POST accounts/:account\_id/tweets - Para adicionar cards a Tweets, use o parâmetro card\_uri.
  Tweets agendados:
- POST accounts/:account\_id/scheduled\_tweets

Para obter mais detalhes sobre cards, consulte a página Cards. A página Promoted Video fornece detalhes sobre como associar vídeos a cards ou Tweets.

<div id="cards">
  ### Cards
</div>

A API de Anúncios do X oferece suporte a vários tipos de cards que podem ser usados em Tweets, os quais podem então ser promovidos em campanhas. **Observação**: depois de publicado, os detalhes do card ficam visíveis publicamente. Isso pode incluir informações sobre o usuário proprietário do card.

<div id="image">
  #### Imagem
</div>

As seguintes especificações de imagem se aplicam a assets usados em [Cards](/pt-BR/x-ads-api/creatives#cards-2). As imagens devem ter 3 MB ou menos e largura de pelo menos 800 px. Além disso, oferecemos suporte às seguintes proporções largura:altura.

- Website: 1:1 e 1,91:1
- Image App Download: 1:1 e 1,91:1
- Poll: 1,91:1
- Image Conversation: 1,91:1
- Image Direct Message: 1,91:1

Oferecemos suporte aos seguintes formatos de imagem: .bmp, .jpeg e .png.

<div id="video">
  #### Vídeo
</div>

As seguintes especificações de vídeo se aplicam aos assets usados em [Cards](/pt-BR/x-ads-api/creatives#cards-2). Oferecemos suporte às seguintes proporções de largura:altura.

- Vídeo para site: 16:9 e 1:1
- Vídeo para download de app: 16:9 e 1:1
- Enquete: 16:9
- Conversa em vídeo: 16:9
- Mensagem direta em vídeo: 16:9

<div id="promoted-video">
  ### Vídeo Promovido
</div>

Este documento apresenta uma visão geral do processo de upload e promoção de vídeo por meio da API de Anúncios do X.

A API de Anúncios do X oferece suporte a Vídeo Promovido em [Tweets](/pt-BR/x-ads-api/creatives#tweets-2) e nos seguintes cards:

- [Vídeo para Website](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [Vídeo para Download de App](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [Vídeo de Conversa](/pt-BR/x-ads-api/creatives#video-conversation-cards)

Primeiro, faça o upload do vídeo usando o endpoint [POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html). Com o `media_id`, associe o vídeo a uma conta de anúncios usando o endpoint POST accounts/:account\_id/videos. O `id` do vídeo, às vezes chamado de `media_key`, será usado em solicitações subsequentes. Ele é uma string que começa com um número inteiro, seguida por um sublinhado e terminada por um valor longo. Exemplo: `13_875943225764098048`.

<div id="promoted-video-in-tweets">
  #### Vídeo promovido em Tweets
</div>

Para criar um Tweet, use o endpoint [POST accounts/:account\_id/tweet](/pt-BR/x-ads-api/creatives#get-accounts-account-id-tweets) junto com o `id` do vídeo. Nessa etapa, você também pode fornecer um título, uma descrição e um call to action (CTA). Esses valores são exibidos aos usuários.

<div id="promoted-video-in-cards">
  #### Vídeo promovido em cards
</div>

Os cards de Download de app com vídeo e de Conversa em vídeo permitem adicionar uma imagem de capa. Envie uma imagem para usar nesses cards pelo endpoint [POST media/upload](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload.html).

Crie o card usando um dos seguintes endpoints:

- [POST accounts/:account\_id/cards/video\_website](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [POST accounts/:account\_id/cards/video\_app\_download](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [POST accounts/:account\_id/cards/video\_conversation](https://devcommunity.x.com/t/ads-api-version-11/168814)

usando o `id` do vídeo e, opcionalmente, o `media_id` da imagem (para a imagem de capa).

Por fim, crie o post usando o endpoint POST accounts/:account\_id/tweet. Os cards são anexados aos posts usando o parâmetro `card_uri`.

<div id="general-information">
  #### Informações gerais
</div>

Para orientações detalhadas sobre upload de vídeo via API, consulte o [Guia de Upload de Vídeo](https://developer.x.com/en/docs/media/upload-media/uploading-media.html).

Os vídeos também podem ser promovidos como ativos de pre-roll. Consulte o [Guia do Objetivo de Visualizações de Vídeo Pre-roll](/pt-BR/x-ads-api/campaign-management#video-views-preroll-objective) para uma explicação detalhada.

- (A partir de 2015-10-22) Ao fazer upload de vídeos para uso em conteúdo promovido, o parâmetro `media_category` deve ser definido com o valor `amplify_video` para todas as solicitações do comando `INIT` ao endpoint [POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html). O uso desse parâmetro garante que o vídeo seja pré-processado de forma assíncrona e preparado para uso em conteúdo promovido. O comando `STATUS` pode ser usado para verificar a conclusão do processamento assíncrono após o upload do vídeo.
- O tempo de duração máximo atualmente permitido para vídeo promovido é de 10 minutos, com tamanho de arquivo de 500 MB ou menos.
- O vídeo enviado deve ser MP4 ou MOV.
- O vídeo enviado geralmente é processado rapidamente, mas o tempo de processamento pode variar conforme a duração do vídeo e o tamanho do arquivo.
- As imagens de poster enviadas devem estar nos formatos PNG ou JPG. Não há requisitos de proporção ou de tamanho, mas a imagem de poster será ajustada para se adequar ao player de vídeo.

<div id="scheduled-tweets">
  ### Posts agendados
</div>

<div id="introduction">
  #### Introdução
</div>

Tweets agendados permitem que um anunciante ou usuário crie um Tweet para ser publicado em uma data futura. Além de criar e gerenciar esses Tweets, a API permite associá-los a um line item, para que sejam promovidos quando o Tweet for ao ar. Isso permite que os anunciantes preparem Tweets nativos e planejem seus criativos de campanha com antecedência a iniciativas importantes. Por exemplo, preparar um criativo de Tweet para ir ao ar imediatamente após o anúncio de um novo produto.

O conjunto completo de funcionalidades fornecidas pelos endpoints da API de Tweets agendados está listado abaixo:

- Criar, modificar e visualizar Tweets recém-agendados
- Associar um Tweet agendado a um line item
- Consultar e gerenciar Tweets agendados existentes
- Quando um Tweet agendado for ao ar, recuperar o `id` do Tweet publicado

<div id="api-endpoints">
  #### Endpoints da API
</div>

A lista completa de endpoints relacionados à funcionalidade acima é apresentada abaixo:

<div id="scheduled-tweet-management">
  #### Gerenciamento de Tweets programados
</div>

- [GET accounts/:account\_id/scheduled\_tweets](/pt-BR/x-ads-api/creatives#get-accounts-account-id-scheduled-tweets) (recupera a lista de todos os Tweets programados)
- [GET accounts/:account\_id/scheduled\_tweets/:scheduled\_tweet\_id](/pt-BR/x-ads-api/creatives#get-accounts-account-id-scheduled-tweets) (consulta um Tweet programado específico usando seu `id`)
- [POST accounts/:account\_id/scheduled\_tweets](/pt-BR/x-ads-api/creatives#post-accounts-account-id-scheduled-tweets) (cria um novo Tweet programado)
- [PUT accounts/:account\_id/scheduled\_tweets/:scheduled\_tweet\_id](/pt-BR/x-ads-api/creatives#example-request-39) (modifica um Tweet programado existente)
- [DELETE accounts/:account\_id/scheduled\_tweets/:scheduled\_tweet\_id](/pt-BR/x-ads-api/creatives#example-request-40) (exclui um Tweet programado usando seu `id`)
- [GET accounts/:account\_id/scheduled\_tweets/preview/:scheduled\_tweet\_id](/pt-BR/x-ads-api/creatives#scheduled-tweets-2) (visualiza a prévia de um Tweet programado existente)

<div id="scheduled-promoted-tweets">
  #### Tweets promovidos agendados
</div>

- [GET accounts/:account\_id/scheduled\_promoted\_tweets](/pt-BR/x-ads-api/campaign-management#get-accounts-account-id-scheduled-promoted-tweets) (recupera a lista de todos os tweets promovidos agendados)
- [GET accounts/:account\_id/scheduled\_promoted\_tweets/:scheduled\_promoted\_tweet\_id](/pt-BR/x-ads-api/campaign-management#get-accounts-account-id-scheduled-promoted-tweets-scheduled-promoted-tweet-id) (consulta um tweet promovido agendado pelo seu `id`)
- [POST accounts/:account\_id/scheduled\_promoted\_tweets](/pt-BR/x-ads-api/campaign-management#post-accounts-account-id-scheduled-promoted-tweets) (cria um novo tweet promovido agendado)
- [DELETE accounts/:account\_id/scheduled\_promoted\_tweets/:scheduled\_promoted\_tweet\_id](/pt-BR/x-ads-api/campaign-management#delete-accounts-account-id-scheduled-promoted-tweets-scheduled-promoted-tweet-id) (exclui um tweet promovido agendado existente pelo seu `id`)

<div id="scheduled-tweet-view">
  #### Visualização de Tweet agendado
</div>

- [GET accounts/:account\_id/scheduled\_tweets/preview/:scheduled\_tweet\_id](/pt-BR/x-ads-api/creatives#scheduled-tweets-2) (visualizar um Tweet agendado existente)

Como os Tweets agendados são entidades separadas dos Tweets “ativos”, há dois conjuntos distintos de validações executadas em qualquer criação ou edição desses Tweets. O primeiro conjunto de regras de validação é executado durante a etapa de criação do Tweet agendado, especificamente:

<div id="scheduled-tweet-create">
  #### Criação de Tweet agendado:
</div>

- Validar que o usuário autenticado tem acesso para criar Tweets orgânicos para um determinado @handle. Os privilégios de criação de Tweet “somente promovido” exigem que o usuário autenticado seja um usuário da conta com [permissões do compositor de Tweet](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access).
- Validar que não haja mais de 30 Tweets agendados para serem criados em uma janela de 15 minutos em relação ao horário scheduled\_at. Uma mensagem de erro SCHEDULED\_TWEET\_LIMIT\_EXCEEDED indica que muitos Tweets foram agendados dentro do mesmo intervalo futuro de 15 minutos. Os anunciantes precisarão remover um Tweet agendado existente ou ajustar o horário scheduled\_at para mais cedo ou mais tarde.

<div id="scheduled-tweet-goes-live">
  #### Tweet agendado entra “no ar”:
</div>

- Essas regras de validação são executadas no horário de scheduled\_at e são idênticas às aplicadas na criação de um Tweet comum na API. Por exemplo, um Tweet agendado não entrará no ar e o scheduled\_status será definido como FAILED se o Tweet agendado contiver simultaneamente uma imagem e um GIF

<div id="workflow">
  #### Fluxo de trabalho
</div>

**Criar um novo Tweet agendado**

É possível criar um novo Tweet agendado usando o endpoint [POST accounts/:account\_id/scheduled\_tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets). Este endpoint exige os seguintes parâmetros: o horário em `scheduled_at` e o `text` do Tweet, caso nenhuma mídia seja incluída. Além disso, o endpoint oferece opções adicionais que permitem criar um Tweet agendado em nome de outro @handle por meio do parâmetro `as_user_id`, além da possibilidade de adicionar um card (`card_uri`) e mídias (`media_ids`). Observe que um Tweet pode conter apenas entidades de um único tipo, ou seja, Vídeo, GIF ou Imagem. O parâmetro `nullcast` controla se o Tweet é “apenas promovido” ou não. Todos os Tweets agendados recém-criados são “apenas promovidos” (`nullcast=true`) por padrão. Se `nullcast=false`, é criado um Tweet agendado orgânico.

Quando um Tweet agendado é criado com sucesso, a resposta conterá um campo `id`, que é o identificador exclusivo do próprio Tweet agendado. Além desse campo, também é retornado o campo `tweet_id`. Este campo é `null` inicialmente; no entanto, quando o Tweet for publicado, ele será preenchido com o ID do Tweet “ao vivo”.

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_tweets" -d 'scheduled_at=2017-12-24T23:59:00Z&text=Happy Holidays!'
```

Isso criará o seguinte Tweet agendado:

```json
{
  "request": {
    "params": {
      "text": "Happy Holidays!",
      "scheduled_at": "2017-12-24T23:59:00Z"
    }
  },
  "data": {
    "completed_at": null,
    "id_str": "917507899668099072",
    "text": "Happy Holidays!",
    "user_id": "3271358660",
    "scheduled_status": "SCHEDULED",
    "id": 917507899668099100,
    "nullcast": true,
    "created_at": "2017-10-09T21:51:44Z",
    "scheduled_at": "2017-12-24T23:59:00Z",
    "card_uri": null,
    "updated_at": "2017-10-09T21:51:44Z",
    "tweet_id": null,
    "media_keys": []
  }
}
```

Quando esse Tweet agendado for publicado, o campo `tweet_id` será preenchido com o ID do Tweet “ao vivo”.

**Associar um Tweet agendado a um item de linha**

Embora Tweets agendados possam ser usados para criar Tweets orgânicos, também permitimos que parceiros criem um Tweet “somente promovido” (`nullcast=true`), qualquer um dos quais pode ser associado a um item de linha. Para facilitar isso, fornecemos também o endpoint [POST accounts/:account\_id/scheduled\_promoted\_tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets). Esse endpoint permite associar apenas um único Tweet agendado promovido a um item de linha em uma única chamada de API. Para associar vários Tweets agendados ao mesmo item de linha, são necessárias várias chamadas de API.

Observe que não é possível modificar um Tweet agendado promovido existente.

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_promoted_tweets" -d 'line_item_id=a44qc&scheduled_tweet_id=917507899668099072'
```

```json
{
  "data": {
    "line_item_id": "a44qc",
    "id": "26576",
    "created_at": "2017-10-09T22:24:16Z",
    "updated_at": "2017-10-09T22:24:16Z",
    "scheduled_tweet_id": "917507899668099072",
    "tweet_id": null,
    "deleted": false
  },
  "request": {
    "params": {
      "line_item_id": "a44qc",
      "scheduled_tweet_id": 917507899668099100,
      "account_id": "aaaaa"
    }
  }
}
```

Esse endpoint apenas cria uma associação entre um determinado Tweet agendado e um item de linha. Assim que as datas de veiculação da campanha/item de linha estiverem vigentes, o item de linha começará automaticamente a veicular o Tweet “ao vivo” correspondente. Embora validemos, nesta etapa, que o Tweet agendado está no estado `SCHEDULED` e que o Tweet agendado é válido para o objetivo fornecido, nenhuma outra validação é executada. Quaisquer regras de validação restantes que se aplicam ao item de linha e ao Tweet agendado são executadas quando o Tweet ficar “ao vivo”.

Para garantir que não haja problemas com a veiculação da campanha, recomenda-se que o Tweet agendado esteja com `scheduled_at` definido para um horário anterior às datas de veiculação da campanha/item de linha.

Por exemplo, digamos que o Tweet agendado esteja definido para entrar ao vivo após a data de início da campanha (e que haja apenas um único Tweet associado a um único item de linha). Nesse caso, a campanha ficará `ACTIVE`; no entanto, como o Tweet agendado ainda não está ao vivo, não haverá criativos disponíveis para veiculação.

**Gerenciamento de Tweets agendados**

Os conjuntos restantes de endpoints permitem que os consumidores da API gerenciem todos os seus Tweets agendados e Tweets agendados promovidos. Essas APIs podem ser usadas para retornar uma lista de todos os Tweets agendados, opcionalmente filtrada por um determinado estado, bem como para buscar um determinado Tweet agendado por seu `id`.

<div id="what-happens-when-a-scheduled-tweet-goes-live">
  #### O que acontece quando um Tweet agendado entra no ar?
</div>

Quando um determinado Tweet agendado está prestes a entrar no ar, ou seja, no horário `scheduled_at`, as seguintes atualizações são feitas:

- O Tweet “ao vivo” é criado; no entanto, isso pode ter uma latência de até 1 segundo
- O `tweet_id` é adicionado às seguintes entidades:
- Tweet agendado
- Tweet agendado promovido
- Uma nova entidade de Tweet promovido é criada

<div id="best-practices">
  #### Melhores práticas
</div>

Recomendamos as seguintes melhores práticas ao criar ou promover Tweets agendados:

- Verifique se o Tweet é válido ao criar o Tweet agendado (por exemplo, um Tweet pode conter apenas uma imagem, vídeo ou GIF, e não uma combinação dos três)
- Verifique se as datas de veiculação da campanha (ou seja, `start_time` e `end_time`) estão alinhadas com o horário `scheduled_at` do Tweet agendado
- Não agende Tweets para mais de um ano no futuro (365 dias)
- A visualização de Tweet não é compatível no momento para Tweets agendados (ou seja, a capacidade de visualizar Tweets agendados antes da criação)

<div id="media-library">
  ### Biblioteca de mídia
</div>

<div id="introduction">
  #### Introdução
</div>

Os endpoints da Biblioteca de Mídia permitem gerenciar imagens, GIFs e vídeos para contas de Anúncios do X. Os recursos de mídia na biblioteca podem ser usados em posts e na criação de cards. Eles também podem ser reutilizados em várias [criativas](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/overview), eliminando a necessidade de enviar o mesmo recurso várias vezes.

<div id="api-endpoints">
  #### Endpoints da API
</div>

- [POST media/upload](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload#post-media-upload) ou [POST media/upload (em partes)](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload-init#post-media-upload-init) (fazer upload de mídia)
- [POST accounts/:account\_id/media\_library](https://developer.x.com/x-ads-api/creatives#get-accounts-account-id-media-library#post-accounts-account-id-media-library) (adicionar mídia à Biblioteca de mídia)

<div id="adding-to-the-library">
  #### Adicionando à biblioteca
</div>

<div id="request-parameters">
  #### Parâmetros da solicitação
</div>

Todas as solicitações POST da Media Library exigem um identificador de mídia. Esse valor é retornado durante a etapa de upload. Ao usar o media\_id, como no exemplo acima, também é necessário especificar uma media\_category. Existem quatro valores possíveis de categoria: AMPLIFY\_VIDEO, TWEET\_GIF, TWEET\_IMAGE e TWEET\_VIDEO.

Opcionalmente, os valores name e file\_name podem ser definidos para objetos na Media Library. Esses atributos ajudam os usuários a distinguir entre variantes de mídia na biblioteca.

Para vídeos, também é possível definir um título e uma descrição. No Tweet, esse texto aparece abaixo do vídeo.

<div id="attributes">
  #### Atributos
</div>

A Media Library introduz formalmente o conceito de media\_key. Esse é o identificador exclusivo de objetos na biblioteca. As media keys são valores de string no seguinte formato: 13\_875943225764098048. Elas são totalmente compatíveis em todos os nossos endpoints de card.

Além disso, a resposta da Media Library inclui o media\_id, representado como string. Ele é incluído para recursos que atualmente não aceitam uma media key: [Tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet)\*, [Prévia de Tweet](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#get-accounts-account-id-tweet-preview)\* e [Tweets agendados](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets). Estamos trabalhando para oferecer suporte a media keys em todos os lugares.

O atributo aspect\_ratio é retornado para GIFs e vídeos. Ele pode ser usado para filtrar mídias para uso em cards que aceitam apenas determinadas proporções.

\*Esses endpoints oferecem suporte ao parâmetro video\_id, que é uma media key.

<div id="introduction">

Cards podem ser referenciados em objetos de Tweet de duas maneiras: pelo card\_uri do card ou pelo seu preview\_url. Valores de exemplo para cada um são apresentados abaixo.

<div id="identifying-media">
  ### Identificação de mídia
</div>

<div id="introduction">
  #### Introdução
</div>

Mídia — imagens, GIFs e vídeos — pode ser adicionada a posts e cards. Além disso, vídeos podem ser usados como ativos de pré-roll e imagens podem ser promovidas na [X Audience Platform](https://developer.x.com/en/docs/ads/measurement/overview/twitter-audience-platform). Esta seção descreve como encontrar referências de mídia nessas entidades.

Existem dois tipos de identificadores de mídia: IDs e keys. Valores de exemplo para cada um são apresentados abaixo.

| **ID de mídia** | **Chave de mídia** |
| :--- | :--- |
| 1029825579531807971 | 13\_1029825579531807971 |

A chave de mídia é o ID com um prefixo numérico e um sublinhado.

<div id="images">
  #### Imagens
</div>

A tabela a seguir mostra os tipos de identificadores atualmente disponíveis na resposta de cada recurso relacionado a imagem, bem como os respectivos nomes de atributo.

| **Recurso** | **Identificador** | **Atributo(s)** |
| :--- | :--- | :--- |
| Image cards | None |  |
| Tweet | Both | `entities["media"]["id_str"]`  `entities["media"]["media_key"]` |
| Scheduled Tweet | Both | `media_ids` and `media_keys` |
| Draft Tweet | Both | `media_ids` and `media_keys` |
| Account Media | None |  |
| Media Library | Both | `media_id` and `media_key` |

As imagens de Image cards e de Account Media não incluem referência a nenhum identificador de mídia. Tweets incluem apenas IDs de mídia. Scheduled e Draft Tweets incluem tanto o ID de mídia quanto a chave de mídia. A Media Library também retorna ambos.

Para Tweets, os campos id e id\_str no objeto dentro do array entities\["media"] correspondem ao ID da mídia. Nos casos em que um Tweet inclui várias imagens, as referências a cada entidade de mídia só podem ser encontradas em extended\_entities\["media"].

Além das referências a identificadores, muitas vezes é importante ter acesso à URL da imagem.

| **Recurso** | **Atributo(s)** | **Formato** |
| :--- | :--- | :--- |
| Image cards | `image` | .jpg |
| Tweet\* | `entities["media"][0]["media_url"]` or `extended_entities["media"][i]["media_url"]` | .jpg |
| Scheduled Tweet | None |  |
| Draft Tweet | None |  |
| Account Media | `media_url` | .jpg |
| Media Library | `media_url` | .jpg |

\* A localização dessa URL depende de o Tweet conter uma única imagem ou várias imagens.

Todos os Image cards incluem um atributo de resposta image que contém a URL de imagem do X. (Para image app download cards, o nome é wide\_app\_image.)

Para Tweets, a localização da URL da mídia depende tanto do tipo de mídia quanto do endpoint utilizado. Para Tweets com uma única imagem, a URL pode ser encontrada em entities\["media"]\[0]\["media\_url"]. Isso é válido tanto para a API de Anúncios do X quanto para a API Padrão. Quando os Tweets contêm várias imagens, porém, as URLs só podem ser encontradas em extended\_entities\["media"]\[i]\["media\_url"]. Isso está disponível apenas na API Padrão.

<div id="media-library">
  #### Biblioteca de Mídia
</div>

Às vezes, é necessário recuperar informações adicionais sobre um ativo de mídia. Um caso de uso, para cards de vídeo, é obter a URL mp4 em vez da vmap. Isso está disponível na Biblioteca de Mídia. Para detalhes sobre as informações disponíveis, consulte nosso [Guia da Biblioteca de Mídia](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/guides/media-library). A maioria dos ativos pertencentes ao usuário PROMOVÍVEL COMPLETO da conta de anúncios pode ser encontrada na biblioteca. Existem, porém, algumas exceções.

**Busca de mídia**

Como mencionado acima, cards de imagem não contêm referências a media IDs nem a media keys. Como resultado, não é possível buscar seus ativos por meio da Biblioteca de Mídia. Isso também vale para imagens de [Account Media](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/account-media#account-media).

Cards de vídeo exigem que o ativo de vídeo já faça parte da Biblioteca de Mídia (ou do recurso Videos, anteriormente) antes de sua criação. Como resultado, esses ativos sempre poderão ser recuperados na Biblioteca de Mídia. Isso também vale para ativos PREROLL de Account Media.

Por fim, mídias em Tweets estão sempre garantidas na Biblioteca de Mídia.

A tabela a seguir resume quais ativos podem ser recuperados na Biblioteca de Mídia, considerando se a resposta do recurso inclui um identificador para uso na consulta.

| **Resource** | **In the Media Library** |
| :--- | :--- |
| Image cards | No |
| Video cards | Yes\* |
| Tweets (any media)\*\* | Yes |
| Scheduled Tweets | Yes |
| Draft Tweets | Yes |
| Account media images | No |
| Account media videos (`PREROLL`) | Yes |

\* Para cards em que o `video_content_id` é uma media key. Quando o valor é um media ID, o ativo ainda existe na Biblioteca de Mídia, mas recuperá-lo envolve anexar um prefixo numérico e um sublinhado a ele.
\*\* Tweets retornam apenas media IDs. Embora o ativo esteja garantido na Biblioteca de Mídia, buscá-lo envolve anexar um prefixo numérico e um sublinhado a ele.

**Interações com Account Media**

Há dois casos em que ativos de mídia adicionados à biblioteca são automaticamente adicionados ao recurso Account Media.

- Quando um ativo AMPLIFY\_VIDEO é adicionado à Biblioteca de Mídia, ele é automaticamente adicionado como um ativo de Account Media, como tipo criativo PREROLL.
- Quando imagens com dimensões específicas (veja “Creative Types” em nossa [página de enumerações](https://developer.x.com/content/developer-twitter/en/docs/ads/general/overview/enums)) são adicionadas à Biblioteca de Mídia, elas são automaticamente adicionadas como ativos de Account Media. O tipo criativo (por exemplo, INTERSTITIAL) depende das dimensões da imagem.

<div id="tweets">
  ### Posts
</div>

<div id="introduction">
  #### Introdução
</div>

A API de Anúncios do X oferece suporte a três tipos de Tweets: publicados, programados e rascunhos.

<div id="nullcasted-tweets">
  #### Tweets nullcast
</div>

Tweets podem ser nullcast (também chamados de “apenas promovidos”) ou orgânicos. Tweets nullcast, depois de publicados, não aparecem na linha do tempo pública do usuário, embora sejam públicos. Tweets orgânicos, por outro lado, são exibidos aos seguidores do usuário e aparecem na linha do tempo pública do usuário.

**Criando Tweets**

Cada um dos três endpoints de criação de Tweet oferece suporte a um parâmetro booleano nullcast que dá ao usuário da API a opção de criar Tweets nullcast ou orgânicos. Tweets nullcast podem ser criados pelo próprio usuário ou por qualquer pessoa que tenha permissão para criar Tweets em nome dele. Tweets orgânicos só podem ser criados pelo [usuário totalmente promovível](https://developer.x.com/content/developer-twitter/en/docs/tutorials/ads-api-hierarchy-terminology#promotable-users).

**Atualizando Tweets**

É possível atualizar a propriedade nullcast para Tweets agendados e rascunhos. Para Tweets agendados, as edições podem ser feitas até o horário scheduled\_at do Tweet. Tweets em rascunho podem ser editados indefinidamente. Depois de publicados, porém, não é possível mudar um Tweet de nullcast para orgânico ou vice-versa.

<div id="promoting-tweets">
  #### Promoção de Tweets
</div>

Apenas Tweets publicados e agendados podem ser promovidos. Eles podem ser nullcast ou orgânicos; não há restrição. Um anunciante pode promover seus próprios Tweets ou os Tweets de outro usuário, desde que tenha obtido permissão para isso. (Veja: [Promover os Tweets de outro usuário](https://developer.x.com/content/developer-twitter/en/docs/tutorials/promoting-another-users-content) para mais informações.)

Vários Tweets podem ser promovidos em uma única campanha. Da mesma forma, um único Tweet pode ser promovido em uma ou mais campanhas.

Para promover Tweets publicados, use o [endpoint POST accounts/:account\_id/promoted\_tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/promoted-tweets#post-accounts-account-id-promoted-tweets). Isso associa Tweets publicados a um line item. Para promover Tweets agendados, use o endpoint [POST accounts/:account\_id/scheduled\_promoted\_tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets).

<div id="tweet-ids">

<div id="carousels">
  ### Carrossel
</div>

<div id="introduction">
  #### Introdução
</div>

A API de Anúncios do X oferece suporte à criação e recuperação de carrosséis de vídeo e de imagem. O carrossel é um tipo de card que pode conter entre 2 e 6 mídias. O card de carrossel pode direcionar o usuário a um site ou incentivá-lo a instalar um app móvel. Para mais informações sobre carrosséis, seus benefícios, melhores práticas e perguntas frequentes, consulte nossa página [Anúncios em Carrossel no X](https://business.x.com/en/advertising/carousels.html).

Um carrossel, como qualquer outro tipo de card, pode ser usado em Tweets e esses Tweets podem, então, ser promovidos. O fluxo de trabalho é o mesmo ao qual você já está acostumado:

1. Enviar a mídia
2. Criar o card
3. Criar o Tweet
4. Promover o Tweet

A única diferença está em como o card é criado. Enquanto outras solicitações de criação de card aceitam parâmetros de consulta, as solicitações de criação de card de carrossel aceitam apenas corpos JSON em POST.

<div id="endpoints">
  #### Endpoints
</div>

A API de Anúncios do X é compatível com a criação e a recuperação de carrosséis.

Para criar um carrossel — de qualquer tipo — use o endpoint [POST accounts/:account\_id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#post-accounts-account-id-cards). Para recuperar carrosséis, use o endpoint [GET accounts/:account\_id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#get-accounts-account-id-cards).

<div id="json-post-body">
  #### Corpo do POST em JSON
</div>

Carrosséis são criados com dois componentes. O primeiro especifica os ativos de mídia que serão usados. O segundo especifica informações sobre o site ou o app.

Especificamente, um cartão de carrossel é criado usando os seguintes componentes, nesta ordem:

- Um componente `SWIPEABLE_MEDIA`, que aceita um array de chaves de mídia
- _Um_ dos seguintes:
- Um componente `DETAILS` para especificar informações do site
- Um componente `BUTTON` para especificar informações do app

O componente `SWIPEABLE_MEDIA` deve incluir um array `media_keys` em que você pode especificar entre 2 e 6 imagens ou vídeos. A ordem em que as chaves de mídia são passadas determina a ordem em que serão renderizados.

```
{
  "type": "SWIPEABLE_MEDIA",
  "media_keys": [
    "13_1089771253848666112",
    "13_1191948012077092867"
  ]
}
```

Como lembrete, você pode obter chaves de mídia fazendo uma solicitação ao endpoint [GET accounts/:account\_id/media\_library](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/media-library#get-accounts-account-id-media-library).

A composição do segundo objeto de componente depende de você querer direcionar o usuário a um site ou incentivá-lo a instalar um app. A tabela abaixo resume as duas opções. (**Observação**: todas as chaves listadas são obrigatórias.)

|  | **Site** | **App** |
| :--- | :--- | :--- |
| Especificar o tipo de componente | `"type": "DETAILS"` | `"type": "BUTTON"` |
| Título/Rótulo | `"title": "X"` | `"label": { "type": "ENUM", "value": "INSTALL" }` |
| Destino | `"destination": { "type": "WEBSITE", "url": "https://www.x.com" }` | `"destination": { "type": "APP", ... }` |

Juntando tudo, abaixo está um exemplo de corpo de POST em JSON para um carrossel de site.

```
{
  "name": "website carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "DETAILS",
      "title": "X",
      "destination": {
        "type": "WEBSITE",
        "url": "https://www.x.com"
      }
    }
  ]
}
```

Objetos de destino de app dentro de componentes `BUTTON` exigem um código de país e pelo menos um identificador de app. Eles podem aceitar deep links opcionalmente. Para uma descrição desses campos, consulte a [documentação de referência](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#app-destination).

Juntando tudo, abaixo está um exemplo de corpo de POST em JSON para um carrossel de app.

```
{
  "name": "app carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        "type": "APP",
        "country_code": "US",
        "googleplay_app_id": "com.twitter.android",
        "iphone_app_id": "333903271"
      }
    }
  ]
}
```

<div id="creative-metadata-tagging">
  ### marcação-de-metadados-de-criativo
</div>

<div id="introduction">
  #### Introdução
</div>

Este guia é destinado a parceiros criativos, agências e desenvolvedores criativos para rotular ativos usados em campanhas no X, a fim de entender melhor o valor e o desempenho de cada ativo.

**Observação:** Ativos de mídia devem ser rotulados apenas pelo parceiro ou desenvolvedor que criou o ativo de mídia. Se o usuário do ativo de mídia não o criou, não implemente a marcação de metadados.

A Marcação de Metadados Criativos fornece atribuição a imagens e vídeos criados por Parceiros Criativos, independentemente de onde o ativo seja carregado no X ou de qual entidade esteja fazendo o upload. Para criar a conexão entre o ativo criativo e o parceiro criativo, é utilizado o padrão [XMP](https://exiftool.org/TagNames/XMP.html).

<div id="tagging-creative-assets">
  #### Marcação de ativos criativos
</div>

A tabela a seguir mostra os tipos de identificadores atualmente disponíveis na resposta de cada recurso relacionado a imagens, bem como o(s) nome(s) de atributo correspondente(s). É necessária uma ferramenta de marcação para rotular ativos criativos. O [ExifTool](https://exiftool.org/), uma biblioteca Perl multiplataforma juntamente com um [aplicativo de linha de comando](https://exiftool.org/exiftool_pod.html) para leitura, gravação e edição de metadados, é recomendado. Veja todos os [tipos de arquivo](https://exiftool.org/#supported) compatíveis.

Siga as [instruções para instalar o ExifTool](https://exiftool.org/install.html). Também há pacotes de software oferecidos pelo [Homebrew](https://formulae.brew.sh/) para simplificar ainda mais a instalação, fornecendo o [comando de instalação do exiftool](https://formulae.brew.sh/formula/exiftool) para macOS e Linux. Confirme que a ferramenta foi instalada corretamente digitando exiftool -ver na linha de comando para obter o número da versão. Saiba mais sobre os parâmetros de comando do ExifTool na [documentação do ExifTool](https://exiftool.org/).

Parceiros criativos podem atribuir tags de metadados a ativos criativos novos ou existentes, definindo seu X app\_id na tag XMP contributor e a data na tag date. Os ativos criativos seguirão as restrições de tamanho existentes ao [fazer upload de mídia](https://developer.x.com/en/docs/twitter-api/v1/media/upload-media/overview).

**Observação:** O uso da tag XMP contributor pelo X garante que os metadados capturem valores exclusivamente para campanhas no X.

`exiftool -contributor="<YOUR APP ID>" -creative_file.jpg`

`exiftool -date="<date>" -creative_file.jpg`

O app\_id pode ser encontrado no [Portal do Desenvolvedor](https://developer.x.com/en/portal/dashboard) em Projects & Apps. Exemplo: 16489123

```

```

```

```

```

```

```

```

```

```

<div id="components">
  #### Componentes
</div>

Todo componente deve incluir um campo `type` que determina o esquema do objeto. A API de Anúncios do X oferece suporte aos seguintes tipos de componentes, agrupados em componentes baseados em mídia e em descrição.

- Mídia:
- `MEDIA`: vídeo ou imagem única
- `SWIPEABLE_MEDIA`: entre 2 e 6 vídeos ou imagens
- Descrição:
- `DETAILS`
- `BUTTON`

Cada componente tem um conjunto de campos obrigatórios (além da chave `type`). Eles estão listados na tabela a seguir.

| `type` do componente | Campo | Tipo de valor |
| :--- | :--- | :--- |
| `MEDIA` | `media_key` | string |
| `SWIPEABLE_MEDIA` | `media_keys` | array de strings |
| `DETAILS` | `title`   `destination` | string   objeto |
| `BUTTON` | `label`   `destination` | objeto   objeto |

A seguir está um exemplo de um componente `BUTTON` no contexto do array `components` (omitindo intencionalmente a chave `name`). (As reticências indicam locais onde mais informações precisariam ser especificadas.)

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        ...
      },
      "destination": {
        ...
      }
    }
  ]
}
```

A ordem em que os objetos de componente são especificados define a ordem de cima para baixo em que serão renderizados. Os cards devem ser criados usando um componente baseado em mídia e um componente `DETAILS` ou `BUTTON`. Componentes baseados em descrição são renderizados abaixo da mídia e têm destinos associados, seja URLs ou aplicativos móveis.

**Label**

Labels definem o texto exibido nos botões e, portanto, aplicam-se apenas ao componente `BUTTON`. Objetos de label têm duas chaves obrigatórias: `type` e `value`. O `type` deve ser definido como `ENUM` e o `value` pode ser um dentre: `BOOK`, `CONNECT`, `INSTALL`, `OPEN`, `ORDER`, `PLAY` ou `SHOP`.

Aproveitando o exemplo anterior, a seguir é mostrado o objeto `label` dentro do componente `BUTTON`.

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        ...
      }
    }
  ]
}
```

**Destination**

Destinations são para onde os anunciantes pretendem levar os usuários. Elas são sempre obrigatórias dentro dos componentes `DETAILS` ou `BUTTON`. Há dois tipos de destino: `WEBSITE` ou `APP`.

**Observação**: Destinos de website só podem ser usados com componentes `DETAILS` e destinos de app só podem ser usados com componentes `BUTTON`.

**Website Destination**

| Nome | Descrição |
| :--- | :--- |
| type   _obrigatório_ | O tipo de destino, que determina seu esquema.  Tipo: enum  Valores possíveis: `WEBSITE` |
| url   _obrigatório_ | A URL do site para o qual o usuário será redirecionado.  Tipo: string  Exemplo: `https://devcommunity.x.com/c/advertiser-api` |

**App Destination**

| Nome | Descrição |
| :--- | :--- |
| type   _obrigatório_ | O tipo de destino, que determina seu esquema.  Tipo: enum  Valores possíveis: `APP` |
| country\_code   _obrigatório_ | O código de duas letras [ISO 3166-1 alpha-2](https://en.wikipedia.org/wiki/ISO_3166-1_alpha-2) do país onde o app é vendido.  Tipo: string  Exemplo: `US` |
| googleplay\_app\_id   _às vezes obrigatório_ | O nome do pacote do aplicativo na Google Play.  **Observação**: Pelo menos um dos seguintes é obrigatório: `ios_app_store_identifier` ou `googleplay_app_id`.  Tipo: string  Exemplo: `com.twitter.android` |
| ios\_app\_store\_identifier   _às vezes obrigatório_ | O identificador da App Store do iOS.  **Observação**: Pelo menos um dos seguintes é obrigatório: `ios_app_store_identifier` ou `googleplay_app_id`.  Tipo: string  Exemplo: `333903271` |
| googleplay\_deep\_link   _opcional_ | Um deep link para o app Android que você está promovendo.  **Observação**: Só pode ser usado se um `googleplay_app_id` tiver sido fornecido.  Tipo: string |
| ios\_deep\_link   _opcional_ | Um deep link para o app iOS que você está promovendo.  **Observação**: Só pode ser usado se um `ios_app_store_identifier` tiver sido fornecido.  Tipo: string |

#### POST accounts/:account\_id/tweet

Crie um Tweet para o usuário totalmente promovível da conta (padrão) ou para o usuário especificado no parâmetro `as_user_id`. A criação de Tweets nullcasted (padrão) e orgânicos é compatível. Tweets nullcasted não aparecem na linha do tempo pública e não são exibidos para seguidores. Qualquer um dos tipos pode ser usado em campanhas.

Se o usuário autenticado não for o usuário promovível `FULL` nesta conta, verifique se ele tem permissão para publicar Tweets em nome desse usuário fazendo uma solicitação ao endpoint [GET accounts/:account\_id/authenticated\_user\_access](https://developer.x.com/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access). A permissão `TWEET_COMPOSER` indica que o usuário pode usar este endpoint para criar Tweets nullcasted em nome do [usuário promovível](https://developer.x.com/en/docs/ads/campaign-management/api-reference/promotable-users#get-accounts-account-id-promotable-users) `FULL`.

Ao usar o [endpoint upload.x.com](https://developer.x.com/rest/reference/post/media/upload) para mídia, passe o mesmo valor de `user_id` para o parâmetro `additional_owners` que o valor de `as_user_id` que você enviar para este endpoint.
