<div id="decahose-stream">
  ### Decahose ストリーム
</div>

[エンタープライズ](https://developer.x.com/en/products/x-api/enterprise)

_これは、当社のマネージドアクセスレベルでのみ利用可能なエンタープライズ向け API です。この API を使用するには、まずエンタープライズ営業チームとのアカウント設定が必要です。 [詳細はこちら](https://developer.x.com/en/products/x-api/enterprise)_

Decahose は、ストリーミング接続を通じてリアルタイムの X Firehose から 10% のランダムサンプルを提供します。これは、データをランダムに選択しつつ、X が Firehose 経由で送信するデータを期待どおりの低遅延で配信できるリアルタイムサンプリングアルゴリズムによって実現されています。

以下は、Decahose で利用可能な機能の一部です。

- **拡張・強化された URL:** - 短縮 URL を完全に展開し、追加のメタデータ（ページタイトルと説明）を提供
- **ストリームのパーティショニング** - 2 つのパーティション（各パーティションが Decahose ストリームのボリュームの 50% を保持）
- **信頼性の強化** - バックエンドシステムの地理的多様性

注: このデータは一括配信され、追加のフィルタリング（例: キーワード）には対応していません。

`ENTERPRISE`

<div id="streaming-likes">
  ### いいねのストリーミング
</div>

_これはマネージドアクセスレベルでのみ利用可能なエンタープライズ API です。この API を使用するには、まず当社のエンタープライズセールスチームと契約（アカウントの開設）が必要です。 [詳細はこちら](https://developer.x.com/en/products/x-api/enterprise)_

「いいね」は、誰が投稿に反応したかの把握と、正確ないいね数の取得を可能にします。Gnip の Firehose と Decahose は、Gnip 経由で配信される投稿に関連する公開のいいねを配信できます。これにより、投稿に紐づくリアルタイムの公開エンゲージメントおよびオーディエンスのメトリクスが得られます。
 

**いいねの利用開始**

いいねデータを取り込むにあたり、次の点を把握しておく必要があります。

- いいねは独立した専用のストリームで配信されます
- いいねは歴史的には「Favorites」と呼ばれてきました。拡張されたネイティブ形式のペイロードでもこの名称が維持されています
- ストリームには公開のいいねのみが含まれます
  - 「公開」とは、いいねしたユーザー、投稿作成者、投稿のすべてがプラットフォーム上で公開であることを意味します
- いいねはリツイートと非常によく似ており、エンゲージメントの公開シグナルを表します
- ペイロード要素には以下が含まれます:
  - 元の投稿オブジェクト
  - 元の投稿を作成したアクターオブジェクト
  - いいねアクションを実行したアクターオブジェクト
- オリジナルのコンテンツにのみ「いいね」できます
  - リツイートには「いいね」できません。リツイートへのいいねは元の投稿に適用されます
  - 引用ツイートには「いいね」できます
- いいねアクティビティには、該当する Gnip Enrichments（購入／適用されている場合）が含まれます
- サポートされる製品／機能
  - いいねストリームは Backfill をサポートします（購入／適用されている場合）
  - いいねストリームに Replay のサポートはありません
  - いいねに対する検索または履歴のサポートはありません
  - 現時点で PowerTrack にいいねのサポートを追加する計画はありません

**Decahose**

- Decahose で配信される 10% サンプル投稿に対して、ストリームには該当する公開のいいねが 100% 含まれます
- **パーティション:** 2
- **URL 構造**
  - https://gnip-stream.x.com/stream/sample10-likes/accounts/\<accountName>/publishers/twitter/\<streamLabel>.json?partition=1

<div id="recovery-and-redundency">
  ### リカバリーと冗長性
</div>

**概要** 

大量のリアルタイム投稿をストリーミングする際には、データの信頼性と完全性の双方を高めるためのベストプラクティスがあります。リアルタイムデータを取り込む場合、接続時間を最大化することが基本目標です。切断が発生したら、自動検知して再接続することが重要です。再接続後は、バックフィルが必要な期間がないか評価する必要があります。こうした詳細を管理し、リアルタイム投稿を取り込むコンポーネントは、ネットワーク、データストア、サーバー、ストレージといった懸念を抱えるシステム全体の一部にすぎません。これらのシステムが複雑であることを踏まえると、別のベストプラクティスとして、ストリーミング環境を分け、少なくとも開発／テスト用と本番用でストリームを分離することが挙げられます。

Decahose には、これらの取り組みを支援する機能が用意されています。

1. 複数環境をサポートするために、お客様のアカウントに[Additional Streams](#AdditionalStreams)をデプロイできます。これらのストリームは相互に独立しており、識別のために異なる stream\_label を持ちます。
2. 接続維持を支援するため、各 Decahose ストリームは[Redundant Connections](/ja/x-api/enterprise-gnip-2.0/fundamentals/decahose-api#additional-streams)をサポートします。一般的なアーキテクチャは、1 つのストリームに 2 つの接続を持たせ、クライアント側に 2 つの独立したコンシューマー（理想的には異なるネットワーク上）を配置する構成です。これにより、クライアント側のネットワーク、サーバー、データストア経路にわたって冗長性を確保できます。各接続ではデータの完全コピーが配信されるため、クライアント側は重複データを許容し、適切に管理する必要がある点に注意してください。
3. 「**ハートビート**」は 10 秒ごとに送信されます。ただし、Decahose ストリームではデータ量が十分に多いため、短時間（例：数秒）でも投稿がない場合は接続問題を示す可能性があります。したがって、「データの無音」およびハートビートの欠如の両方で切断を検知できます。

切断は発生し得るため、Decahose ストリームには、切断やその他の運用上の問題によって取り逃したデータの回収を支援する専用の[Recovery](/ja/x-api/enterprise-gnip-2.0/fundamentals/decahose-api#overview)および[Backfill](/ja/x-api/enterprise-gnip-2.0/fundamentals/decahose-api#backfill)機能が用意されています。

<div id="additional-streams">
  #### 追加ストリーム
</div>

追加の Decahose ストリームを用意することは、ソリューションの信頼性を高める別の方法です。追加ストリームは完全に独立しており、それぞれ固有のエンドポイントを持ちます。各ストリームには独自の stream\_label が割り当てられ、このラベルとアカウント名がそのストリームの URL の一部になります。以下の例を参照してください：

https://gnip-stream.x.com/stream/sample10/accounts/:account\_name/publishers/twitter/:stream\_label.json

一般的なパターンとしては、本番システム専用のリアルタイムストリームに加えて、開発・テスト用のストリームを用意します。テスト／開発ストリームがあることで、Decahose のお客様はクライアントコンシューマの更新を検証できます。ストリームには任意の（ユニークな）ラベルを割り当てられますが、慣例として本番ストリームには「prod」、開発用には「dev」または「sandbox」を使用します。

ストリームの数とそれぞれの固有ラベルは、担当アカウント担当者によって設定可能です。

**冗長接続**

冗長接続では、データストリームに対して複数の同時接続を確立できます。これにより、同一ストリームに 2 つの別個のコンシューマで接続し、両方の接続から同じデータを受信することで冗長性が確保されます。したがって、例えば一方のストリームが切断された場合やアプリのプライマリサーバーが障害を起こした場合でも、アプリはホットフェイルオーバーを実現できます。

任意のストリームに許可される接続数は、担当アカウント担当者によって設定可能です。冗長ストリームを利用するには、プライマリ接続で使用しているのと同じ URL に接続するだけです。ストリームのデータは両方の接続を通じて送信され、両方の接続はストリームダッシュボードに表示されます。

課金上は、複数の接続を通じて受信したアクティビティ数を重複排除し、各ユニークなアクティビティに対して 1 回のみ課金します。Decahose には 2 つのパーティションがあるため、接続数のカウント例を以下に示します：

Connect to decahose partition=1
Connect to decahose partition=1
Connect to decahose partition=2

上記の状況では合計 3 接続になります。すなわち、partition=1 に 2 接続、partition=2 に 1 接続です。通常は各パーティションに同数の接続を持つことが望ましいため、この例は partition=2 への冗長接続がドロップしており、さらなる調査が必要な状況を示しています。

**リカバリ**

#### 概要 

Recovery は、直近の X の履歴データに対して、直近5日間のローテーションウィンドウでストリーミングアクセスを提供するデータ復旧ツールです（主要なデータ収集には使用しないでください）。リアルタイムストリームでデータの取り込みに漏れが生じた場合の復旧に使用します。短時間の切断が発生した場合や、その他の理由で一定期間リアルタイムデータの取り込みに失敗した場合などが該当します。

<div id="using-recovery">
  #### Recovery の使用
</div>

Recovery ストリームでは、リアルタイムストリームと同様の方法でリクエストを行えます。ただし、リクエストする時間範囲を示すパラメータを URL に指定する必要があります。言い換えると、Recovery リクエストは API に「時間 A から時間 B までの投稿」を要求します。これらの投稿は、リアルタイムストリームに近い形でストリーミング接続を通じて配信されますが、リアルタイムよりわずかに遅い速度で配信されます。以下は例です:

"https://stream-data-api.x.com/stream/powertrack/accounts/someAccountName/publishers/twitter/powertrack.json?startTime=2023-07-05T17:09:12.070Z"

投稿は、指定された期間の最初（最も古い）分から配信が始まり、最後の分が配信されるまで時系列で続きます。その時点で、Recovery Request Completed というメッセージが接続を通じて送信され、その後サーバーによって接続が閉じられます。リクエストが一致する結果がほとんどない、または全くない時間帯から始まる場合、最初の結果が配信されるまでに時間が生じる可能性があります。これは、Recovery がその時点で処理しているアーカイブの部分で一致を検出した時点でデータが配信されるためです。配信可能な結果がない場合でも、ストリームはタイムアウトを防ぐために接続を通じてキャリッジリターン、つまり「ハートビート」を送り続けます。

Recovery は、短時間の切断によって取り逃したデータを容易に回収するためのツールとして想定されており、1 日全体のような非常に長い時間範囲には適していません。長期間のデータ回収が必要な場合は、インターネットの不安定さやその他の理由によりリクエストの途中で切断される可能性を減らし、長時間のリクエストの進捗をより把握しやすくするために、長いリクエストを（例: 2 時間などの）短い時間ウィンドウに分割することをおすすめします。

<div id="data-availability">
  #### データ可用性
</div>

5分間のバックフィルウィンドウ内に再接続できない場合でも、Recovery 機能を使用して直近24時間の取り逃したデータを復旧できます。

ストリーミングの Recovery 機能により、バックフィルウィンドウを24時間まで拡張できます。Recovery は、取り逃したデータの期間を“復旧”することを可能にします。'start\_time' と 'end\_time' のリクエストパラメータを指定して接続リクエストを送ると、リカバリーストリームが開始されます。接続されると、Recovery は指定された期間を再配信し、その後切断されます。  

同時に Recovery へのリクエストを2件まで、すなわち「2つのリカバリージョブ」を実行できます。Recovery は技術的にはバックフィルと同様に動作しますが、開始時刻と終了時刻を指定する点が異なります。リカバリー期間は単一の時間範囲です。

<div id="backfill">
  #### バックフィル
</div>

バックフィルをリクエストするには、接続リクエストに backfillMinutes=N パラメータを追加します。ここで N は、接続時にバックフィルする分数（1〜5、整数のみ）です。例えば、90 秒間切断された場合は、接続リクエストに backfillMinutes=2 を追加します。このリクエストは 2 分間のバックフィルを提供し、切断前の 30 秒間も含むため、_コンシューマーアプリは重複データを許容できる必要があります_。

Decahose の接続リクエスト URL の例として、パーティション 1 に 5 分のバックフィルを要求する場合は次のようになります:

https://gnip-stream.x.com/stream/sample10/accounts/:account\_name/publishers/twitter/:stream\_label.json?partition=1\&backfillMinutes=5

**注意事項:**

- 接続時に常に「backfillMinutes=5」を使用し、提供される重複データを処理する運用も可能です。

- 5 分以上切断された場合は、Recovery を使用してデータを復旧できます。

**切断からの復旧**

切断からの再起動と復旧には、以下の手順が必要です:

- 切断時間の長さを判定する。
  - 5 分以下か？
    - ストリームで Backfill が有効な場合は、適切な「backfillMinutes」パラメータで接続リクエストを準備する。
  - 5 分より長いか？
    - Recovery ストリームがある場合は、切断された期間に対して Recovery リクエストを実行する（必要に応じて Rules API を使用し、可能であれば現在のリアルタイムのルールセットを用いる）。
- 新しい接続をリクエストする。

切断やダウンタイムが発生した場合、以下の戦略で緩和および復旧してください:

1. **バックフィルの実装**
   バックフィルにより、ストリーム接続から切断される前の時点から再接続でき、最大 5 分の切断をカバーします。これは接続リクエストにパラメータを含めることで実装します。

2. **別拠点から冗長ストリームを取り込む**
   冗長ストリームを同じ本番環境にストリーミングし、データの重複排除が可能であれば、通常ストリームと冗長ストリームの両方が同時にダウンタイムまたは切断しない限り、復旧は不要になります。冗長ストリームを本番環境にライブで流せない場合は、別の「緊急」データストアに書き込むこともできます。そうすれば、プライマリのストリーム接続で切断やダウンタイムが発生した際に、欠落期間のデータをプライマリデータベースに補完するためのデータを手元に確保できます。

3. **Recovery の実装**
   切断やダウンタイムがプライマリストリームと冗長ストリームの双方に影響する場合は、Decahose Recovery を使用して取り逃したデータを復旧します。API はアーカイブの 5 日分をカバーするローリングウィンドウを提供しており、そのうち 1 時間を超えない範囲でリクエストし、ストリーミングで取り込むのが最適です。これはリアルタイムストリームと並行して行います。Recovery が提供する 5 日のウィンドウより前の Decahose データを復旧する手段はないため、重大なダウンタイムに備えて冗長ストリームを活用し、貴組織側で完全なデータコピーを保持しておくことが重要です。

異常な保存データ量を検知したとき—
切断やダウンタイムが発生していないのにデータ欠落を検知するための方法:

1. 受信した投稿のカウント
   取り込みアプリの最初の段階で受信した生の投稿数をカウントし、その数と最終的なデータストアに到達した投稿数を比較できるようにします。差分を監視し、受信後にデータがドロップされる原因となる問題をチームにアラートできます。

2. 異常な保存量の分析
   最終データベースに保存されたデータ量を分析し、異常な減少がないか確認することも推奨します。これも問題を示す場合がありますが、状況によってはボリュームの低下が正常であることもあります（例: X プラットフォームが利用できず、一定期間ユーザーが投稿を作成できない場合）。

Volume Stream API へのすべてのリクエストでは、console.gnip.com のアカウントにログインする際の有効なメールアドレスとパスワードの組み合わせからなる HTTP Basic 認証を使用する必要があります。各リクエストで、認証情報は Authorization ヘッダーとして送信してください。クライアントがすべての API リクエストに「Authorization: Basic」HTTP ヘッダー（HTTPS 上でエンコードされた認証情報）を付与していることを確認してください。

#### GET \{stream-type}:stream

リアルタイムデータが配信される Firehose ストリームへの永続接続を確立します。

<div id="responses">
  #### 応答
</div>

これらのリクエストに対して、API は次の応答を返す可能性があります。エラーコードの多くは、ボディに追加の詳細を含む文字列とともに返されます。200 以外の応答の場合、クライアントは再接続を試みてください。

| ステータス | テキスト | 説明 |
| :--- | :--- | :--- |
| 200 | 成功 | 接続が正常に確立され、以後、新しいアクティビティが到着次第送信されます。 |
| 401 | 認証エラー | 無効な認証情報により HTTP 認証に失敗しました。console.gnip.com に認証情報でログインし、リクエストで正しく使用していることを確認してください。 |
| 406 | 受理不可 | 一般的には、クライアントがストリームの gzip 圧縮を受け入れるためのヘッダーを正しく含めていない場合に発生しますが、他の状況でも発生する可能性があります。  <br />  <br />次のような JSON メッセージが含まれます: "This connection requires compression. To enable compression, send an 'Accept-Encoding: gzip' header in your request and be ready to uncompress the stream as it is read on the client end." |
| 429 | レート制限 | アプリが接続リクエストの上限を超えました。 |
| 503 | サービス利用不可 | X のサーバー側の問題です。指数バックオフ方式で再接続してください。この問題に関する通知が[X API Status Page](https://api.twitterstat.us/)に掲載されていない場合、10 分経過しても接続できないときはサポートまたは緊急窓口に連絡してください。 |

<div id="replay-api">
  #### Replay API

Replay API は、リアルタイムの Volume ストリームを補完する重要な機能です。Replay は、直近の X の履歴データに対するローリングウィンドウ型のストリーミングアクセスを提供するデータ復旧ツールです。
