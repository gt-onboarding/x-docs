<div id="introduction">
  ### はじめに
</div>

`Enterprise`

投稿は、X におけるすべての機能の基本となる最小単位です。投稿を返すすべての X API は、そのデータを JavaScript Object Notation（JSON）でエンコードして提供します。JSON は名前付き属性とそれに対応する値からなるキーと値のペアで構成されます。API から取得される Post オブジェクトには X ユーザーの「ステータス更新」が含まれますが、リポスト、返信、引用投稿もすべて Post オブジェクトです。投稿がリポスト、返信、引用投稿として別の投稿に関連している場合、それぞれが識別されるか、Post オブジェクト内に埋め込まれます。ネイティブな X データ形式における最も単純な投稿であっても、投稿の他の属性（投稿者、メンションされたユーザー、タグ付けされた場所、ハッシュタグ、キャッシュタグ、メディア、URL リンクなど）を表現するために入れ子の JSON オブジェクトを持ちます。X データを扱ううえで、これは重要な概念です。X API から受け取る Post データの形式は、受信する投稿の種類、利用している X API、そしてフォーマット設定によって異なります。

Post オブジェクトを返す Enterprise エンドポイントは、投稿の編集履歴を理解するために必要なメタデータを提供するよう更新されています。これらのメタデータの詳細は、["投稿の編集" の基本](/ja/x-api/fundamentals/edit-posts)をご覧ください。

ネイティブなX形式では、JSONペイロードにはルートレベルの属性と、ネストされたJSONオブジェクト（ここでは`{}`記法で表現）が含まれます。

```JSON
{
	"created_at": "Fri Feb 14 19:00:55 +0000 2020",
	"id_str": "1228393702244134912",
	"text": "開発者はバレンタインカードに何て書いた？\n  \nwhile(true) {\n    I = Love(You);  \n}",
	"entities": {
		"hashtags": [],
		"symbols": [],
		"user_mentions": [],
		"urls": []
	},
	"user": {
		"entities": {
			"url": {}
		}
	},
	"place": {}
}
```

<div id="available-data-formats">
  ### 利用可能なデータ形式
</div>

> ご注意：エンタープライズデータ API には [Enriched Native](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) 形式の使用を強く推奨します。
>
> - Enriched Native 形式には、[投票メタデータ](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata) など 2017 年以降の新しいメタデータに加え、reply\_count や quote\_count などの追加メトリクスが含まれます。
>
> - Activity Streams 形式は、2017 年の[文字数アップデート](https://blog.x.com/official/en_us/topics/product/2017/Giving-you-more-characters-to-express-yourself.html)以降、新しいメタデータやエンリッチメントでの更新は行われていません。

エンタープライズデータ API は 2 種類の形式でデータを提供します。標準の v1.1 ネイティブ形式に最も近いエンタープライズ形式は Enriched Native です。レガシーのエンタープライズデータ形式は Activity Streams で、当時 X および他のソーシャルメディアデータ提供者間での正規化形式として Gnip により実装・利用されていました。この形式は現在も利用可能ですが、X は 2017 年以降、Enriched Native 形式にのみ新機能や開発投資を行っています。

Enriched Native 形式は名称のとおり、ネイティブな X オブジェクトに加え、URL 展開メタデータ、プロフィール位置情報、投票メタデータ、追加のエンゲージメントメトリクスなど、エンタープライズデータ製品で利用可能な追加エンリッチメントを含みます。

- [URL の拡張・強化エンリッチメント](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls)
- [マッチングルールのエンリッチメント](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#matching-rules)
- [投票メタデータのエンリッチメント](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata)
- [プロフィール位置情報のエンリッチメント](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo)

### 解析のベストプラクティス

- X の JSON は UTF-8 でエンコードされています。
- パーサーはフィールドの並び順の違いに容易に対処できる必要があります。投稿の JSON は順不同のハッシュ（連想配列）として提供されると想定してください。
- パーサーは「新規」フィールドの追加を許容する必要があります。
- JSON パーサーは「欠落」フィールドに寛容でなければなりません。すべてのコンテキストで全フィールドが出現するわけではないためです。
- null のフィールド、空集合、フィールドが存在しない状態は、同一とみなしても概ね安全です。

## エンタープライズネイティブ拡充データオブジェクト

### ネイティブ拡張 Tweet オブジェクト

> ネイティブ拡張データ形式が X API v2 の形式にどのようにマッピングされるか、詳しく知りたい方はこちら。
>
> 比較ガイドをご覧ください: [Native Enriched compared to X API v2](/ja/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2)

#### Post オブジェクト

エンタープライズ向けデータ製品を使用すると、データディクショナリの多くがPostデータのネイティブ形式と共通しており、さらに拡張メタデータが付与されていることがわかります。ネイティブ拡張形式の基本レベルでは、[X API v1.1 data format](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview.html) と同じオブジェクト名が多く使われています。Postオブジェクトには、`id`、`created_at`、`text` といった基本的な属性を含む「ルートレベル」の属性が多数あります。さらに、`user`、`entities`、`extended_entities` といった入れ子のオブジェクトも持ちます。加えて、retweeted\_status、quoted\_status、extended\_tweet のような[入れ子の Post オブジェクト](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#nested-post-objects)も含まれます。ネイティブ拡張形式には、さらに matching\_rules オブジェクトも含まれます。

##### X データ辞書

以下に、これらの「ルートレベル」属性のデータ辞書と、子オブジェクトのデータ辞書へのリンクを掲載しています。

| 属性 | 型 | 説明 |

| created\_at | 文字列 | この投稿が作成された UTC 時刻。例:<br /><br />"created\_at": "Wed Oct 10 20:19:24 +0000 2018" |
| id  | Int64 | この投稿の一意の識別子の整数表現。この数値は 53 ビットを超えるため、一部のプログラミング言語では解釈時に問題が発生したり、気づかれにくい欠陥が生じる可能性があります。この識別子の保存には符号付き 64 ビット整数を使用すると安全です。識別子の取得には **`id_str`** を使用するのが安全です。詳しくは [X IDs](/ja/resources/fundamentals/x-ids) を参照してください。例:<br /><br />"id":1050118621198921728 |
| id\_str | 文字列 | この投稿の一意の識別子の文字列表現。実装では **`id`** の大きな整数ではなく、こちらを使用してください。例:<br /><br />"id\_str":"1050118621198921728" |
| text | 文字列 | ステータス更新の実際の UTF-8 テキスト。[twitter-text](https://github.com/twitter/twitter-text/blob/master/rb/lib/twitter-text/regex.rb) を参照し、現在有効と見なされる文字の詳細を確認してください。例:<br /><br />"text":"表現の幅を広げるため、絵文字はすべて同じにカウントします—性別や肌の色のバリエーションを含むものでも同様です… https://t.co/MkGjXf9aXm" |
| source | 文字列 | 投稿に使用されたユーティリティ（HTML 形式の文字列）。X のウェブサイトからの投稿では、source の値は **`web`** になります。<br /><br />例:<br /><br />"source":"X Web Client" |

| in\_reply\_to\_status\_id | Int64 | ヌル可。対象の投稿が返信である場合、元の投稿の ID の整数表現が入ります。例:<br /><br />"in\_reply\_to\_status\_id":1051222721923756032 |
| in\_reply\_to\_status\_id\_str | 文字列 | ヌル可。対象の投稿が返信である場合、元の投稿の ID の文字列表現が入ります。例:<br /><br />"in\_reply\_to\_status\_id\_str":"1051222721923756032" |
| in\_reply\_to\_user\_id | Int64 | ヌル可。対象の投稿が返信である場合、元の投稿の作成者 ID の整数表現が入ります。これは投稿で直接言及されているユーザーと必ずしも一致しません。例:<br /><br />"in\_reply\_to\_user\_id":6253282 |
| in\_reply\_to\_user\_id\_str | 文字列 | ヌル可。対象の投稿が返信である場合、元の投稿の作成者 ID の文字列表現が入ります。これは投稿で直接言及されているユーザーと必ずしも一致しません。例:<br /><br />"in\_reply\_to\_user\_id\_str":"6253282" |
| in\_reply\_to\_screen\_name | 文字列 | ヌル可。対象の投稿が返信である場合、元の投稿の作成者のスクリーンネームが入ります。例:<br /><br />"in\_reply\_to\_screen\_name":"xapi" |

| coordinates | [Coordinates](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-geo-objects) | null 許容。この投稿の地理位置（ユーザーまたはクライアントアプリが送信）を表します。内側の座標配列は [GeoJSON](http://www.geojson.org/) 形式（経度が先、緯度が後）です。例：<br /><br />"coordinates":<br />\<br/>    "coordinates":<br />    \[<br />        -75.14310264,<br />        40.05701649<br />    ],<br />    "type":"Point"<br />} |

#### ネストされた Post オブジェクト

いくつかのケースでは、Post オブジェクトに他のオブジェクトがネストされます。ネストされたオブジェクトを扱う場合、その JSON ペイロードには複数の Post オブジェクトが含まれ、各 Post オブジェクトがさらに独自のオブジェクトを含むことがあります。ルートレベルのオブジェクトには、Retweet なのか Quote Tweet なのかといった実行されたアクションの種類に関する情報が含まれ、共有されている「元の」Post を記述するオブジェクトが含まれる場合もあります。Extended Posts には、2017 年の更新時に非互換な変更を避けるために導入された、140 文字を超える内容を表す拡張オブジェクトがネストされます。各ネストされたオブジェクトの辞書については以下で説明します。

**Retweets**

Retweet には常に 2 つの Post オブジェクトが含まれます。Retweet される「元の」Post は "retweeted\_status" オブジェクトで提供されます。ルートレベルのオブジェクトは Retweet 自体を表しており、Retweet を実行したアカウントの User オブジェクトと Retweet の実行時刻を含みます。Retweet はフォロワーと Post を共有するためのアクションであり、新たなコンテンツを追加することはできません。また、Retweet に（新たな）位置情報を付与することもできません。「元の」Post にジオタグが付いている場合でも、Retweet の "geo" および "place" オブジェクトは常に null になります。

Extended Posts の導入以前であっても、ルートレベルの "entities" オブジェクトは、Retweet 対象の Post メッセージに "RT @username " という文字列が付加されるため、場合によっては切り詰められて不完全になることがありました。なお、Retweet がさらに Retweet された場合でも、"retweet\_status" は元の Post を指し続け、中間の Retweet は含まれません。x.com で Retweet を「表示」した場合も同様の挙動になります。Retweet の「アクション」に割り当てられた固有の Post ID をコピーすると、元の Post が表示されます。

以下は Retweet の構造例です。繰り返しになりますが、Retweet をパースする際は、完全な（元の）Post メッセージとエンティティのメタデータを取得するために "retweeted\_status" オブジェクトをパースすることが重要です。

```json
{
	"tweet": {
		"text": "RT @author original message",
		"user": {
			"screen_name": "Retweeter"
		},
		"retweeted_status": {
			"text": "original message",
			"user": {
				"screen_name": "OriginalTweeter"
			},
			"place": {},
			"entities": {},
			"extended_entities": {}
		}
	},
	"entities": {},
	"extended_entities": {}
}
```

##### 引用投稿

引用投稿はリポストに似ていますが、新たな投稿メッセージが含まれる点が異なります。これらの新しいメッセージには、独自のハッシュタグ、リンク、その他の「entities」メタデータを含められます。投稿者が共有した位置情報や、GIF・動画・写真といったメディアを含めることもできます。

引用投稿には少なくとも2つの Post オブジェクトが含まれ、場合によっては3つになることもあります。引用されている投稿（それ自体が引用投稿である可能性もあります）は、「quoted\_status」オブジェクトとして提供されます。ルートレベルのオブジェクトは引用投稿そのものを表し、共有アクションを行うアカウントの User オブジェクトおよび引用投稿の時刻を含みます。

引用投稿には、写真・GIF・動画を「Post」ユーザーインターフェイスから追加できるようになった点に注意してください。引用投稿のメッセージに外部ホスティングのメディアへのリンクが含まれている場合、それらはルートレベルの「entities.urls」に記述されます。引用投稿に添付されたメディアは、ルートレベルの「extended\_entities」メタデータに表示されます。

引用投稿が最初に導入された当初は、短縮リンク（t.co URL）が「元の」投稿メッセージに付加され、ルートレベルの「text」フィールドに含まれていました。さらに、その t.co URL のメタデータはルートレベルの「entities.urls」配列にも含まれていました。2018年5月、この仕様を変更し、引用された投稿への短縮 t.co URL はルートレベルの「text」フィールドに含めないようにしました。次に、引用された投稿のメタデータも「entities.urls」メタデータには含めないようにしました。代わりに、引用された投稿の URL メタデータは、ルートレベル（トップレベル）に追加された新しい「quoted\_status\_permalink」オブジェクトに格納され、「quoted\_status」オブジェクトと同じ階層になります。

<div id="quote-tweets">
  ##### 拡張投稿
</div>

「拡張投稿」を記述する JSON は、2017 年 11 月に 280 文字の投稿が導入された際に追加されました。多数のアプリがこれらの基本的な X オブジェクトを引き続き解析できるよう、投稿の JSON は後方互換性を保ちつつ、より長いメッセージを内包できるよう拡張されました。完全な後方互換性を確保するため、元の 140 文字の 'text' フィールドと、そこから解析されるエンティティオブジェクトは維持されました。140 文字を超える投稿の場合、ルートレベルの 'text' フィールドは切り詰められて不完全になります。ルートレベルの 'entities' オブジェクトには、'text' メッセージから解析されたハッシュタグやリンクなどの主要メタデータの配列が含まれるため、これらのコレクションも不完全になります。たとえば、投稿メッセージが 200 文字で末尾にハッシュタグが含まれている場合、従来のルートレベルの 'entities.hashtags' 配列にはそれが含まれません。

より長い投稿メッセージと完全なエンティティメタデータを保持するために、新たに 'extended\_tweet' フィールドが導入されました。"extended\_tweet" オブジェクトは、140 文字を超える場合に完全で切り詰めのない投稿メッセージを格納する "full\_text" フィールドを提供します。また、"extended\_tweet" オブジェクトには、ハッシュタグ、リンク、メンションなどの完全な配列を持つ "entities" オブジェクトも含まれます。

拡張投稿は、ルートレベルの "truncated" ブール値で識別されます。true（"truncated": true）の場合は、ルートレベルのフィールドではなく "extended\_tweet" のフィールドを解析してください。

以下の JSON 例では、投稿メッセージに 3 つのハッシュタグが含まれているにもかかわらず、ルートレベルの "text" フィールドは切り詰められており、ルートレベルの "entities.hashtags" 配列は空であることに注意してください。これは拡張投稿であるため、"truncated" フィールドは true に設定され、"extended\_tweet" オブジェクトが完全な "full\_text" と "entities" の投稿メタデータを提供します。

### ネイティブ拡張ジオオブジェクト

投稿には位置情報を関連付けることができ、いわゆる「ジオタグ付き」の投稿になります。位置情報は、X のユーザーインターフェースから、または API で投稿する際に設定できます。位置情報は、正確な「点」の位置、または会場から地域全体までの広い範囲を表す「バウンディングボックス」を持つ X Place のいずれかです。

投稿に関連付けられた位置情報を表すために使用される「ルートレベル」の JSON オブジェクトは 3 つあります: [place](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#place-data-dictionary)、[geo](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-geo-objects)、[coordinates](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#coordinates-object-data-dictionary)。

さらに、ネイティブ拡張フォーマットには、ユーザーオブジェクト内に [profile geo enrichment](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo) の [derived location](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#derived-locations) が含まれます。

`place` オブジェクトは、投稿が場所でジオタグ付けされている場合は常に存在します。Place は、対応する地理座標を持つ特定の名称付きの場所です。ユーザーが投稿に場所を付与する場合、候補となる X Place の一覧が提示されます。API を使用して投稿する場合は、投稿時に place\_id を指定することで X Place を添付できます。Place に関連付けられた投稿は、必ずしもその場所から発信されたものとは限らず、その場所に「関する」投稿である可能性もあります。

geo と `coordinates` オブジェクトは、投稿に「正確な位置」が割り当てられている場合にのみ（非 null で）存在します。正確な位置が提供されている場合、`coordinates` オブジェクトは地理座標を表す \[long, lat] の配列を返し、その位置に対応する X Place が割り当てられます。

<div id="x-entities">
  ## Introduction
</div>

エンティティは、X に投稿されたコンテンツに関するメタデータと追加のコンテキスト情報を提供します。`entities` セクションには、投稿に含まれうる一般的な要素（ハッシュタグ、ユーザーのメンション、リンク、株式ティッカー（シンボル）、X の投票、添付メディア）の配列が格納されます。X 側で本文テキストが事実上前処理（前解析）されているため、開発者が投稿を取り込む際に便利です。投稿本文からこれらのエンティティを明示的に検索・抽出するのではなく、この JSON セクションを直接参照すれば済みます。

解析を容易にするだけでなく、`entities` セクションは有用な付加的メタデータも提供します。たとえば、[Enhanced URLs enrichment](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls) を使用している場合、URL メタデータには完全展開された URL のほか、関連するウェブサイトのタイトルや説明が含まれます。別の例として、ユーザーメンションがある場合、エンティティのメタデータには数値のユーザー ID が含まれ、これは多数の X API にリクエストする際に有用です。

すべての投稿 JSON ペイロードには、`hashtags`、`urls`、`user_mentions`、`symbols` という最小限の属性を備えた `entities` セクションが含まれます。これらのエンティティが投稿メッセージに含まれていない場合でも同様です。たとえば、本文が「Hello World!」で添付メディアのない投稿の JSON を確認すると、エンティティ配列が要素数 0 の次の内容が含まれます。

```
"entities": {
        "hashtags": [
        ],
        "urls": [
        ],
        "user_mentions": [
        ],
        "symbols": [
        ]
      }
```

Notes:

- media と polls のエンティティは、その種類のコンテンツが投稿に含まれる場合にのみ出現します。
- ネイティブメディア（写真、動画、GIF）を扱う場合は、[Extended Entities object](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#x-extended-entities) を使用してください。

### エンティティオブジェクト

`entities` と `extended_entities` セクションはいずれも、エンティティのオブジェクト配列で構成されています。以下では、各エンティティオブジェクトの説明に加え、オブジェクトの属性名、型、短い説明を記載したデータディクショナリを示します。さらに、これらの属性にマッチする PowerTrack オペレーターを示し、サンプルの JSON ペイロードも掲載します。

これは、ハッシュタグ、リンク、ユーザーのメンションなど、投稿内で一般的に見られるエンティティの集合です。`entities` オブジェクトには `media` 属性も含まれますが、`entities` セクションにおけるその実装が完全に正確なのは、写真が1枚のみの投稿に限られます。複数の写真、動画、またはアニメーション GIF を含む投稿については、`extended_entities` セクションを参照してください。

### ハッシュタグオブジェクト  

`entities` セクションには、Post 本文に含まれる各ハッシュタグのオブジェクトを要素とする `hashtags` 配列が含まれます。ハッシュタグが存在しない場合は空配列になります。

PowerTrack の `#` 演算子は `text` 属性に対してマッチします。`has:hashtags` 演算子は、配列に少なくとも 1 つの項目がある場合にマッチします。

|     |     |     |
| :--- | :--- | :--- |
| フィールド | 型 | 説明 |
| indices | Array of Int | Post テキスト内でハッシュタグが開始および終了する位置を示す整数配列。最初の整数は Post テキスト文字列内の # 文字の位置を表します。2 番目の整数はハッシュタグの直後の最初の文字の位置を表します。したがって、2 つの数値の差はハッシュタグ名の長さに 1（「#」文字の分）を加えた値になります。例:<br /><br />"indices":\[32,38] |
| text | String | 先頭の「#」文字を除いたハッシュタグ名。例:<br /><br />"text":"nodejs" |

### メディアオブジェクト  

`entities` セクションには、投稿にメディアオブジェクトが「添付」されている場合、単一のメディアオブジェクトを含む `media` 配列が含まれます。ネイティブメディアが添付されていない場合、`entities` に `media` 配列は含まれません。次の理由から、投稿のネイティブメディアを処理する際は `extended_entities` セクションを使用してください:
\+ メディアの `type` は、投稿に動画や GIF が添付されている場合でも常に「photo」となります。
\+ 写真は最大で 4 枚添付できますが、`entities` セクションには最初の 1 枚のみが記載されます。

### 写真メディア URL の書式設定

X 上の写真メディアはさまざまなサイズで読み込めます。特定の画像ビューポートに収まる範囲で、できるだけ小さいサイズの画像を読み込むのが最適です。異なるサイズを読み込むには、[Size Object](#size) と [media\_url](#media)（または [media\_url\_https](#media)）を所定の形式で組み合わせる必要があります。写真メディア URL を構築する例として、すでに提供されている [media entity](#entitiesobject) のサンプルオブジェクトを使用します。

`media_url` または `media_url_https` は単体でも読み込め、その場合はデフォルトで medium 変種が読み込まれます。可能であれば、完全に書式化された写真メディア URL を提供することを推奨します。

写真メディア URL は次の3つの要素で構成されます。

|     |     |
| :--- | :--- |
| Base URL | Base URL は、ファイル拡張子を除いたメディア URL です。<br /><br />例:  <br /><br />"media\_url\_https": "https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg",<br /><br />このときの Base URL は次のとおりです:  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq_ |
| Format | Format は画像のファイル形式を示します。利用可能な形式は _jpg_ または _png_ で、メディア URL の拡張子として示されます。<br /><br />例:  <br /><br />"media\_url\_https": "https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg",<br /><br />このときの Format は: _jpg_ |
| Name | Name は読み込むサイズのフィールド名です。<br /><br />例:<br /><br />  <br /> "sizes":   <br />   "thumb":   <br />     "h": 150,  <br />     "resize": "crop",  <br />     "w": 150  <br />   ,  <br />   "large": <br />     "h": 1366,  <br />     "resize": "fit",  <br />     "w": 2048  <br />   },  <br />   "medium": <br />     "h": 800,  <br />     "resize": "fit",  <br />     "w": 1200  <br />   },  <br />   "small": <br />     "h": 454,  <br />     "resize": "fit",  <br />     "w": 680  <br />   }  <br /> }  <br />}<br /><br />大サイズの写真を読み込む場合の Name は: _large_ |

これらの3つの要素（Base URL、Format、Name）を組み合わせて、読み込む写真メディア URL を構築します。画像をこの方法で読み込む形式には _legacy_ と _modern_ の2種類があります。すべての画像読み込みは _legacy_ 形式の使用をやめ、_modern_ 形式を使用してください。_modern_ 形式を使用すると、呼び出し側での CDN のヒット率が向上し、データセンターでメディアを生成して読み込む必要が生じにくくなるため、読み込み遅延の改善につながります。

|     |     |
| :--- | :--- |
| 旧形式 | 旧形式は廃止予定です。写真メディアの読み込みはすべてモダン形式に移行してください。<br /><br />_\<base\_url>.\<format>:\<name>_  <br /><br />例:  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg:large  <br />_ |
| モダン形式 | 写真読み込みのモダン形式は2015年にXで導入され、2017年以降は事実上の標準となっています。写真メディアの読み込みはすべてこの形式に移行してください。<br /><br />_\<base\_url>?format=\<format>\&name=\<name>_  <br /><br />例:  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq?format=jpg\&name=large  <br />_<br /><br />注: 写真メディアのURLにおけるクエリ文字列内の項目はアルファベット順です。メディア読み込みで追加のクエリ項目が加わる場合でも、アルファベット順を維持する必要があります。例えば、仮に\_preferred\_format\_という新しいクエリ項目が追加される場合、クエリ文字列では\_format\_および\_name\_の後に配置されます。 |

### URL オブジェクト 

`entities` セクションには、Post 本文に含まれる各リンクに対応するオブジェクトを格納した `urls` 配列が含まれ、リンクが存在しない場合は空配列が含まれます。

`has:links` 演算子は、配列に少なくとも 1 件の要素がある場合にマッチします。`url:` 演算子は `expanded_url` 属性に対してマッチします。[Expanded URL enrichment](http://support.gnip.com/enrichments/expanded_urls.html) を使用している場合、`url:` 演算子は `unwound.url`（完全に展開された URL）属性に対してマッチします。[Enhanced URL enrichment](http://support.gnip.com/enrichments/enhanced_urls.html) を使用している場合、`url_title:` および `url_description:` 演算子は `unwound.title` および `unwound.description` 属性に対してマッチします。

|     |     |     |
| :--- | :--- | :--- |
| フィールド | 型 | 説明 |
| display\_url | String | Post に貼り付け／入力された URL。例:<br /><br />"display\_url":"bit.ly/2so49n2" |
| expanded\_url | String | `display_url` の展開版。例:<br /><br />"expanded\_url":"http://bit.ly/2so49n2" |
| indices | Array of Int | URL が Post テキスト内で開始・終了する位置（オフセット）を表す整数配列。最初の整数は Post テキスト内で URL の最初の文字の位置を示します。2 番目の整数は、URL の末尾直後にある最初の非 URL 文字の位置を示します。例:<br /><br />"indices":\[30,53] |
| url | String | ラップされた URL。生の Post テキストに直接埋め込まれた値で、indices パラメータの値に対応します。例:<br /><br />"url":"https://t.co/yzocNFvJuL" |

Expanded および／または Enhanced URL enrichment を使用している場合、以下のメタデータが `unwound` 属性の下で利用できます:

|     |     |     |
| :--- | :--- | :--- |
| フィールド | 型 | 説明 |
| url | String | Post に含まれるリンクの完全に展開されたバージョン。例:<br /><br />"url":"https://blog.x.com/en\_us/topics/insights/2016/using-twitter-as-a-go-to-communication-channel-during-severe-weather-events.html" |
| status | Int | 展開処理の最終的な HTTP ステータス。200 は成功を示します。例:<br /><br />200 |
| title | String | リンクの HTML タイトル。例:<br /><br />"title":"Using X as a ‘go-to’ communication channel during severe weather" |
| description | String | リンクの HTML 説明。例:<br /><br />"description":"Using X as a ‘go-to’ communication channel during severe weather" |

PowerTrack の `@` 演算子は `screen_name` 属性に対するマッチに使用されます。`has:mentions` 演算子は、配列に少なくとも 1 件の要素がある場合にマッチします。

| id\_str | String | 言及されたユーザーの ID（文字列）。例:<br /><br />"id\_str":"6253282" |
| indices | Array of Int | 投稿テキスト内でユーザー参照が開始・終了する位置（オフセット）を表す整数配列。最初の整数はユーザー言及における「@」文字の位置を表します。2 番目の整数は、ユーザー言及の直後に続く最初の非スクリーンネーム文字の位置を表します。例:<br /><br />"indices":\[4,15] |

### Symbol オブジェクト  

`entities` セクションには、投稿本文に含まれる各 $cashtag ごとに 1 つのオブジェクトを持つ `symbols` 配列が含まれ、シンボルがない場合は空配列になります。

PowerTrack の `$` オペレーターは `text` 属性に対してマッチします。`has:symbols` オペレーターは、配列に少なくとも 1 件の要素がある場合にマッチします。

|     |     |     |
| :--- | :--- | :--- |
| フィールド | 型 | 説明 |
| indices | Array of Int | シンボル／cashtag が投稿テキスト内で開始・終了する位置のオフセットを示す整数配列。1 つ目の整数は投稿テキスト文字列内の $ 文字の位置を表します。2 つ目の整数は cashtag の直後の最初の文字の位置を表します。したがって、2 つの数の差は cashtag 名の長さに 1（「$」文字分）を加えた値になります。例:<br /><br />"indices":\[12,17] |
| text | String | 先頭の「$」文字を除いた cashtag の名称。例:<br /><br />"text":"twtr" |

<div id="symbol-object">
  ### Poll オブジェクト
</div>

`entities` セクションは、投稿に投票が含まれている場合、単一の `poll` オブジェクトを要素とする `polls` 配列を含みます。投票が含まれていない場合は、`entities` セクションに `polls` 配列は存在しません。

これらの投票メタデータは、以下の Enterprise API でのみ利用可能です。

- ボリュームストリーム（[Decahose](/ja/x-api/enterprise-gnip-2.0/fundamentals/decahose-api)）
- [Real-time PowerTrack](/ja/x-api/enterprise-gnip-2.0/powertrack-api)
- X 検索 API（[Full-Archive Search](/ja/x-api/enterprise-gnip-2.0/fundamentals/search-api) と [30-Day Search](/ja/x-api/enterprise-gnip-2.0/fundamentals/search-api)）

|     |     |     |
| :--- | :--- | :--- |
| フィールド | 型 | 説明 |
| options | Option オブジェクトの配列 | 各選択肢の位置（position）とそのテキストを持つオプションの配列。例:<br /><br />"options": \[<br />          <br />            "position": 1,<br />            "text": "I read documentation once."<br />          }<br />      ]<br />} |
| end\_datetime | String | 投票が終了する時刻（UTC）のタイムスタンプ。例:<br /><br />"end\_datetime": "Thu May 25 22:20:27 +0000 2017" |
| duration\_minutes | String | 投票の実施時間（分）。例:<br /><br />"duration\_minutes": 60 |

## リツイートと引用ツイートの詳細

X APIの観点では、リツイートと引用ツイートは、元の投稿を埋め込みオブジェクトとして含む特殊な種類の投稿です。つまり、リツイートおよび引用ツイートのオブジェクトは、子である「元の」投稿の親にあたります（そのためサイズは2倍になります）。リツイートにはトップレベルの "retweeted\_status" オブジェクトがあり、引用ツイートには "quoted\_status" オブジェクトがあります。整合性のため、これらトップレベルのリツイートおよび引用ツイートのオブジェクトにも text プロパティと関連する entities が存在します。ただし、トップレベルの entities は、埋め込まれた「元の」entities と異なる場合があります。リツイートの場合は、新しいテキストが元の投稿本文の前に追加されます。引用投稿の場合は、新しいテキストが投稿本文の後ろに追加されます。

一般的なベストプラクティスとしては、retweeted\_status が存在する場合は常に、その中の元の投稿からテキスト、entities、元の投稿者、日付を取得することを推奨します。例外は、追加された引用の一部となっている X のエンティティを取得する場合です。詳細とヒントについては以下を参照してください。

### リツイート

リツイートに関する重要な点として、投稿には新たな X の\_エンティティ\_を追加できません。ユーザーはリツイート時にハッシュタグ、URL、その他の情報を追加できません。ただし、リツイート（トップレベル）の text 属性は、元の投稿テキストの先頭に「RT @username: 」を付与したものから構成されます。  

場合によっては、特にユーザー名が長いアカウントでは、この追加文字列と元の投稿本文の組み合わせによって、元の投稿テキストの 140 文字制限を容易に超えてしまうことがあります。140 文字ベースの表示と保存への対応を維持するため、トップレベルの本文は投稿本文の末尾を切り詰め、三点リーダー（「…」）を追加します。結果として、元の投稿末尾に位置していた一部のトップレベルのエンティティが不正確になったり欠落したりする場合があります。例えば、途中で切り詰められたハッシュタグや URL エントリなどです。

この投稿 https://x.com/FloodSocial/status/907974220298125312 には、次の投稿テキストがあります:

<div id="retweets">
  ### 引用ツイート
</div>

引用ツイートは2016年に導入されました。リツイートと異なり、投稿を「引用」すると、共有された投稿の「上に」新しいコンテンツを追加します。この新しいコンテンツには、元の投稿に含められるほぼすべての要素（新しいテキスト、ハッシュタグ、メンション、URL）が含まれます。

引用ツイートにはネイティブメディア（写真、動画、GIF）を含めることができ、entities オブジェクト内に表示されます。

X の entities は追加され得るため、引用側の entities は元の entities と異なる可能性が高いです。

この例では、新しい URL とハッシュタグが引用ツイートの末尾に配置されています。

この投稿 https://x.com/FloodSocial/status/907983973225160704 の投稿テキストは次のとおりです:

                  島が洪水に見舞われるのは奇妙で同じくらい悲劇的… 引用ツイートの大西洋横断テスト | [@thisuser](https://x.com/andypiper) [@thatuser](https://x.com/johnd)[http://bit.ly/2vMMDuu](https://t.co/fzFgMhWlPO "http://bit.ly/2vMMDuu")  [#testing](https://x.com/hashtag/testing?src=hash)

この場合、最上位の entities には引用の詳細が反映されていません。 

一方で、extended\_tweet 内の投稿テキストと entities は、切り捨てや誤った entities なしに引用ツイートを完全に反映しているため、引用ツイートについてはネストされた _extended\_tweet_ オブジェクトを参照することを推奨します。

#### はじめに

投稿にネイティブメディア（外部リンクではなく、投稿のUIを通じて共有されたもの）が含まれている場合は、extended\_entities セクションも含まれます。ネイティブメディア（写真、動画、GIF）については、いくつかの理由から extended\_entities が推奨されるメタデータの参照元です。現在、1つの投稿には最大4枚の写真を添付できます。entities メタデータには最初の写真しか含まれません（2014年までは1枚しか添付できませんでした）が、extended\_entities セクションには添付されたすべての写真が含まれます。さらに、ネイティブメディアに関して entities.media メタデータには、添付メディアが動画やアニメーションGIFであってもメディアタイプが常に「photo」と示されるという欠点があります。実際のメディアタイプは extended\_entities.media\[].type 属性で指定され、_photo_、_video_、_animated\_gif_ のいずれかに設定されます。これらの理由から、ネイティブメディアを扱う場合は extended\_entities メタデータを使用するのが最適です。

写真、動画、アニメーションGIFが添付された投稿にはすべて `extended_entities` JSON オブジェクトが含まれます。`extended_entities` オブジェクトには、`media` オブジェクトの単一の `media` 配列が含まれます（データディクショナリは `entities` セクションを参照）。ハッシュタグやリンクなどの他のエンティティタイプは `extended_entities` セクションには含まれません。`extended_entities` セクション内の `media` オブジェクトの構造は、`entities` セクションに含まれるものと同一です。

投稿に添付できるメディアタイプは1種類のみです。写真は最大4枚まで、動画とGIFは各1件のみ添付できます。`extended_entities` セクションのメディア `type` メタデータはメディアタイプ（「photo」「video」「animated\_gif」）を正しく示し、最大4枚の写真をサポートするため、ネイティブメディアのメタデータソースとして推奨されます。

```
{
      "extended_entities": {
        "media": [

        ]
      }
    }
```

## エンタープライズ向け Activity Streams データオブジェクト

<Note>
  Activity Streams のデータ形式が X API v2 の形式にどのように対応するか、詳しく知りたいですか？
</Note>

比較ガイドをご覧ください: [Activity Streams compared to X API v2](/ja/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2)

<Note>
  注意: エンタープライズ向けデータ API には、[Enriched Native](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) 形式の使用を強く推奨します。 

  - Enriched Native 形式には、[poll metadata](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata) など、2017 年以降に追加された新しいメタデータや、reply\_count、quote\_count といった追加メトリクスが含まれます。

  - Activity Streams 形式は、2017 年の [character update](https://blog.x.com/official/en_us/topics/product/2017/Giving-you-more-characters-to-express-yourself.html) 以降、新しいメタデータやエンリッチメントで更新されていません。
</Note>

### アクティビティオブジェクト

Activity Streams は、Gnip によって作成された X の[元のデータ形式](https://developer.x.comhttps://developer.x.com/en/docs/x-api/v1/data-dictionary/overview)のオブジェクトスキーマ版で、サードパーティの[Activity Base Schema（こちらで説明）](https://activitystrea.ms/head/activity-schema.html)を用いて、Post データおよび他のソーシャルメディアデータの「形式の正規化」を目的としています。投稿はアクティビティストリームのスキーマに正規化され、note、person、place、service といったオブジェクトタイプがネストされたオブジェクトとして含まれます。投稿には、Retweet や twitter\_quoted\_status、long\_object などのネストされた Post アクティビティオブジェクトが含まれる場合があります。

ベースレベルのオブジェクトタイプ「activity」は、ネイティブ拡張形式の[Post のベースレベルオブジェクト](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object)に類似しています。アクティビティストリーム形式のペイロード例は[こちら](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#activity-streams-payload-examples)をご覧ください。

### Location Object

Location オブジェクトは、X アカウントレベルで設定された actor オブジェクト内、または [gnip object](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#gnip-object) の profileLocations オブジェクト内に存在する場合があります。Location オブジェクトは place のオブジェクトタイプを持ち、name、address、geo coordinates のいずれかを含むことができます。Location オブジェクトは、ネイティブ拡張形式の [Geo](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-geo-objects) に類似しています。

#### 位置データ辞書

| フィールド | 型 | 説明 |
| :--- | :--- | :--- |
| objectType | string | 詳細は[こちら](http://activitystrea.ms/head/activity-schema.html#place)を参照。例：<br /><br />**"objectType":** "place" |
| displayName | string | 位置の正式名称。<br /><br />**"displayName":** "United States" |
| name | string | X の place JSON 形式における位置の名称。 |
| link | string | その位置の完全な X の JSON 表現へのリンク。<br /><br />**"link":** "https://api.x.com/1.1/geo/id/27c45d804c777999.json" |
| geo | object | X の地理座標オブジェクト。ポリゴンまたはポイント。<br /><br />[geo](https://developer.x.com/en/docs/x-api/v1/data-dictionary/object-model/geo#coordinates)を参照 |
| countryCode | String | この位置を含む国を表す2文字の国コード。例：<br /><br />**"countryCode": "US"** |
| country | String | この位置を含む国名。例：<br /><br />**"country": "United States"** |

### イントロダクション\*\*

X の本質は、公開・リアルタイム・グローバルなコミュニケーションネットワークです。2006 年以降、X の進化はユーザーの利用パターンや慣習、新機能や機能拡張の双方によって推し進められてきました。X のデータを歴史研究に用いる場合、この進化のタイムラインを理解することは、データアーカイブから関心のある投稿を見つけ出すうえで重要です。

X はシンプルな SMS モバイルアプリとして始まり、包括的なコミュニケーションプラットフォームへと成長しました。完全な API 群を備えたプラットフォームです。API は常に X ネットワークの柱でした。[最初の API は X のローンチ直後に公開されました](https://blog.x.com/2006/introducing-the-twitter-api)。2009 年に投稿へのジオタグ付与が初めて導入された際には、[Geo API](https://blog.x.com/2009/think-globally-tweet-locally) を通じて提供され（のちに投稿への「ジオタグ」機能は X.com のユーザーインターフェイスに統合されました）、今日では X の API が速報や情報共有の源となる双方向ネットワークを支えています。このグローバルでリアルタイムなコミュニケーションチャネルの上に構築できる機会は無限です。

X は、公開されているすべての投稿にアクセスできる 2 つの履歴 API、Historical PowerTrack と [Full-Archive Search API](https://developer.x.com/en/docs/x-api/v1/tweets/search/overview) を提供しています。いずれの API も、関心のある投稿を検索・収集するための一連の「オペレーター」を提供します。これらのオペレーターは、投稿本文、投稿者のアカウント名、投稿で共有されたリンクなど、各投稿に付随する多数（数百）の属性にマッチします。投稿とその属性は、一般的なテキストベースのデータ交換フォーマットである JSON でエンコードされています。したがって、新機能が導入されると新たな JSON 属性が追加され、通常それらに対応する新しい API オペレーターも導入されます。もしあなたのユースケースに、X 上で世界がこれまでに発した声を「傾聴」する必要があるなら、各オペレーターがマッチ対象とする JSON メタデータがいつから存在するのかを理解するほど、Historical PowerTrack のフィルターをより効果的にできます。

続いて、投稿メタデータの更新が関心のあるデータシグナルの発見にどのように影響するかを理解するための、前提となる主要な概念を紹介します。

### 重要な概念\*\*

#### ユーザー主導の慣習から X の「第一級オブジェクト」へ

X のユーザーは、X ネットワークにおいて新たな、そして現在では基礎的なコミュニケーションパターンを自然発生的に生み出してきました。その代表例がハッシュタグで、いまやほぼすべてのソーシャルネットワークで広く使われています。ハッシュタグは、会話やトピックを整理する手段として導入されました。1 日に数億件のメッセージが流れるネットワークでは、関心のある投稿を見つけるためのツールが重要であり、ハッシュタグはその基本的な方法となりました。ハッシュタグの利用が拡大するとほどなくして、X によって公式なステータスとサポートが与えられました。ハッシュタグが「第一級」オブジェクトになったことには多くの意味があります。これは、X.com のユーザーインターフェースでハッシュタグがクリック可能／検索可能になったことを意味します。また、ハッシュタグが @mentions、添付メディア、株式シンボル、共有リンクと並ぶ X の entities ファミリーの一員になったことも意味します。これらのエンティティは、事前にパースされた JSON 配列としてエンコードされており、開発者が処理・スキャン・保存しやすくなっています。

リツイートも、ユーザー主導の慣習が公式オブジェクトになった例の一つです。リツイートは、コンテンツを他者に「転送」する手段として登場しました。もともとは、投稿をコピー＆ペーストし、その先頭に「RT @」というパターンを付ける手動の手順として始まりました。やがてこの手順は、新しいリツイートボタンによって自動化され、新たな JSON メタデータも付与されました。こうして「公式の」リツイートが誕生しました。その他の例としては、「メンション」、メディアやウェブリンクの共有、投稿への位置情報の添付などがあります。これらの各利用パターンは、[x.com](https://x.com/) の新しいユーザーインターフェース機能、新たな対応する JSON、そして投稿をマッチングする新たな方法へと発展しました。これらすべての基礎的な投稿属性は、それらにマッチさせるために使用される PowerTrack Operators の基盤となりました。

<div id="from-user-conventions-to-x-first-class-objects">
  #### 投稿のメタデータ、可変性、更新、鮮度
</div>

投稿メッセージは文字数に上限がありますが、投稿の JSON 記述は100を超える属性で構成されます。誰がいつ投稿したか、オリジナルの投稿かリポストかといった属性に加え、ハッシュタグ、メンション、共有リンクといった一次オブジェクトの配列も含まれます。投稿したアカウントにはユーザー（または Actor）オブジェクトがあり、プロフィールやその他のアカウントメタデータを示す多様な属性が含まれます。プロフィールには、短い自己紹介、ホームロケーション（自由記述）、優先言語、任意のウェブサイトリンクが含まれます。

アカウントメタデータの中には（数値のユーザーIDや作成日など）決して変わらないものもあれば、時間をかけてゆっくり変化するもの、より頻繁に変化するものもあります。人は転職したり引っ越したりします。企業は情報を更新します。過去の投稿を収集する際は、メタデータの一部は「投稿当時の状態」であり、別のメタデータは「クエリ送信時点の状態」であることを理解することが重要です。

すべての履歴 API では、ユーザーのプロフィールの説明、表示名、プロフィールの「ホーム」属性は、クエリ時点の値に更新されます。

<div id="post-metadata-mutability-updates-and-currency">
  #### 「ネイティブ」メディア
</div>

X.com と X モバイルアプリでは、ボタンをクリックしてフォトギャラリーから選択することで、投稿に写真や動画を追加できます。これらが第一級のアクションとして統合されたことで、この方法で共有された動画や写真は「ネイティブ」メディアと呼ばれます。

多くのクエリオペレーターは、これらの「ネイティブ」リソースに対応しており、`has:videos`、`has:images`、`has:media` などがあります。これらは、X の機能を通じて共有されたメディアコンテンツにのみマッチします。X プラットフォーム外でホストされているメディアにマッチさせるには、URL メタデータに基づいて一致させるオペレーターを使用してください。

したがって、Historical PowerTrack と全アーカイブ検索の製品詳細に入る前に、製品かつプラットフォームとしての X がどのように進化してきたかを振り返ってみましょう。

**フィルタリングのヒント**

X における新機能がいつ、どのように追加されたかというタイムラインを把握しておくと、より効果的なクエリの作成に役立つ。ここでいうクエリとは、PowerTrack の Operator を使って Post の JSON にマッチさせるために、X のヒストリカル API が Post アーカイブに適用する「フィルター」や「ルール」を意味する。例として `lang:` Operator があり、これは指定した言語の Post にマッチさせるために使用される。X は言語分類サービス（50 以上の言語に対応）を提供しており、X の API は各 Post ごとに生成される JSON 内でこのメタデータを提供する。したがって、ある Post がスペイン語で書かれていれば、JSON の「lang」属性は「es」に設定される。よって、`lang:es` という句でフィルターを作成すれば、スペイン語として分類された Post メッセージにのみマッチする。

このタイムライン情報は、受け取った Post データの解釈にも役立つ。例えば、2008 年と 2012 年の夏季オリンピックに関するコンテンツの共有を調査していたとする。`is:retweet` Operator だけを適用してリツイートにマッチさせた場合、2008 年にはまったくマッチしない。しかし、2012 年にはおそらく数百万件のリツイートがあるだろう。これだけを見ると、2008 年にはリツイートがユーザーの慣習ではなかった、または単に誰もそのオリンピックについてリツイートしなかった、と誤って結論づけてしまう可能性がある。リツイートは 2009 年にファーストクラスのオブジェクトになったため、2008 年にはそれらを特定するために `"RT @"` のルール句を追加する必要がある。

リツイートと言語分類は、長い歴史と多くのプロダクト上の詳細を持つ Post 属性の例である。以下では、X データにマッチさせ理解するうえで重要な、これらおよびその他の属性クラスの詳細について説明する。

<div id="2017">
  #### 偽陰性を見極める
</div>

フィルター作成で重要なのは、メタデータに基づいて照合する各オペレーターには「導入時期」があるという点です。投稿が行われた後になって導入されたメタデータを前提とするオペレーターでフィルターを組むと、偽陰性が発生します。たとえば、「snow」に言及し、動画を共有しているすべての投稿に関心があるとします。ネイティブ動画を持つ投稿にマッチする `has:videos` オペレーターでルールを作成した場合、この条件は2015年以前の投稿には一致しません。

しかし、X での動画共有自体は2015年よりずっと前から一般的でした。当時は外部でホストされた動画へのリンクが共有されており、2015年に X はプラットフォーム上で直接「動画共有」機能を実装しました。こうした以前の対象投稿を見つけるには、`url:"youtube.com"` のようなルール句を含めるとよいでしょう。

なお、Search API では、インデックス再構築時に一部メタデータが「バックフィル」された例があります。分かりやすい例としては、2009年に株式シンボルの議論で広く使われるようになった $cashtag があります。2015年に $cashtag オペレーターが導入された後、Search インデックスが再構築され、その過程で、2006年初期を含むすべての投稿本文からシンボルエンティティが抽出されました。当時は `$` が主にスラングとして使われていました。“I hope it $now$ $oon!”。

<div id="recognizing-false-negatives">
  #### ユースケースに重要な投稿属性の特定とフィルタリング
</div>

X アカウントの数値 ID のような一部のメタデータはサービス開始当初から存在しており（変更されないアカウントメタデータの一例です）、他のメタデータは 2006 年の X の開始からしばらく後になって導入されました。新たに導入されたメタデータの例としては、リポスト（Retweets）のメタデータ、投稿の位置情報、URL のタイトルと説明、「ネイティブ」メディアなどがあります。以下は、これらの X プラットフォームの更新によって大きく影響を受けた代表的な投稿属性です。

これらに対するフィルタリング／マッチングの挙動は、多くの場合、どの履歴投稿 API を使用するかに依存します。調査やユースケースに最適なプロダクトを選定する一助となるよう、以下の属性の詳細には概要レベルのプロダクト情報を含めています。

<div id="identifying-and-filtering-on-post-attributes-important-to-your-use-case">
  #### X プロフィール
</div>

X の本質はグローバルなリアルタイム通信チャネルであるため、Post データを用いたリサーチでは、誰が発信しているかに重点が置かれることが一般的です。X ユーザーの「ホーム」（拠点）がどこかを把握できると役立つ場面は多くあります。アカウントの自己紹介に興味や趣味の記載があるかを知ることで、関心のある Post にたどり着けることもあります。関心のあるアカウントからの Post をモニタリングしたいというニーズも非常に一般的です。プロフィール属性は、これらすべてのユースケースにおいて鍵となります。

X 上のすべてのアカウントにはプロフィールがあり、X の @handle、表示名、短い自己紹介、ホームロケーション（ユーザーが入力する自由記述テキスト）、フォロワー数などのメタデータが含まれます。数値のユーザー ID やアカウント作成日時のように、決して変わらない属性もあります。一方で、投稿数、フォロー数やフォロワー数といった属性は、日々・週単位・月単位で変化するのが一般的です。その他のアカウント属性もいつでも変更可能ですが、比較的変更頻度が低い傾向にあります（例：表示名、ホームロケーション、自己紹介）。

すべての Post の JSON ペイロードには、その Post の投稿者に関する「アカウントプロフィール」メタデータが含まれます。リツイートの場合は、元の Post を投稿したアカウントのプロフィールメタデータも含まれます。

Post のプロフィールメタデータがどの程度変化し得るかは、使用する過去データ製品に完全に依存します。Search API は、取得時点のプロフィール設定で過去の Post を提供します。Historical PowerTrack では、2011 年以前のデータを除き、Post が投稿された時点のプロフィールが反映されます。2011 年より前の Post については、プロフィールメタデータは 2011 年 9 月時点のプロフィールを反映します。

<div id="x-profiles">
  #### オリジナル投稿とリツイート
</div>

リツイートは、ユーザー主導の慣習が公式オブジェクトになったもう一つの例です。リツイートはコンテンツを他者へ「転送」する手段として生まれました。最初は、投稿をコピー＆ペーストし、先頭に「RT @」のパターンを付ける手作業でした。のちにこの手順は新しいリツイートボタンで自動化され、対応する JSON メタデータも追加されました。こうして「公式」リツイートが誕生し、リツイートという行為は第一級の投稿イベントになりました。新しいリツイートボタンとともに、オリジナル投稿の完全なペイロードなどの新たなメタデータが導入されました。

投稿がオリジナルか共有かは、一般的なフィルタリングの「切り替え要素」です。ケースによってはオリジナルコンテンツのみが必要ですし、別のケースでは投稿のエンゲージメントが最優先でリツイートが重要になります。PowerTrack の `is:retweet` 演算子により、リツイートを含めるか除外するかを制御できます。2009年8月より前のデータを取得する場合は、リツイートの一致（または非一致）に備えて戦略を2つ用意する必要があります。2009年8月以前は、投稿本文自体を正確なフレーズ一致でチェックし、「RT @」パターンに一致するかを確認する必要があります。2009年8月以降の期間は、`is:retweet` 演算子が利用可能です。

<div id="original-post-and-retweets">
  #### 投稿の言語分類
</div>

投稿がどの言語で書かれているかは多くの利用者にとって関心の高い情報です。投稿の言語は、投稿の所在地の推測に役立ち、分析や表示で特定の言語だけが必要になることもよくあります。（X のプロフィールには優先言語の設定もあります。）

投稿の言語分類でフィルタリングする場合、X のヒストリカル製品（[Search API](https://developer.x.com/en/docs/x-api/v1/tweets/search/overview/enterprise) と Historical PowerTrack）は大きく異なります。Search のアーカイブが構築された際、すべての投稿に X の言語分類が後付けされました。したがって、`lang:` 演算子は投稿アーカイブ全体で利用できます。Historical PowerTrack では、X の言語分類メタデータは 2013 年 3 月 26 日以降のアーカイブで利用可能です。 

<div id="post-language-classifications">
  #### 投稿の位置参照
</div>

投稿がどこで行われたか（すなわち位置参照すること）を把握できることは、多くのユースケースで重要です。投稿を位置参照する主な方法は3つあります。

- 投稿本文内の地理的な言及
- ユーザーによってジオタグが付けられた投稿
- ユーザーが設定したアカウントプロフィールの「ホーム」位置

##### 投稿メッセージ内の地理的参照

投稿メッセージ内の地理的参照に基づくマッチングは、地域の知識に依存するため最も難しい方法になりがちですが、全投稿アーカイブに対して利用できる手法です。以下は、サンフランシスコ地域を対象に「golden gate」というフィルターで抽出された、2006年の地理参照マッチの例です:

https://x.com/biz/statuses/28311

<div id="geographical-references-in-a-post-message">
  ##### ユーザーによってジオタグされた投稿
</div>

2009年11月、X は投稿に正確な位置情報のジオタグを付与できる Post Geotagging API を導入しました。2010年6月には、会場、近隣エリア、町といったスケールの地理的領域を表現する X Places を導入しました。投稿の約1～2%が、いずれかの方法でジオタグされています。

利用可能なジオタグ履歴は、利用する Historical API に依存します。Search API では、一部のジオ演算子による投稿マッチングが2010年3月から、その他は2015年2月から可能になりました。Historical PowerTrack を使用する場合、位置情報の参照は2011年9月1日から開始されます。Historical PowerTrack のアーカイブ構築時、この日付より前のジオタグはすべて含まれていません。

<div id="posts-geo-tagged-by-the-user">
  ##### ユーザーが設定するアカウントプロフィールの「ホーム」所在地
</div>

すべてのXユーザーは、ホーム所在地を示すプロフィールの位置情報を設定できます。何百万人ものXユーザーがこの情報を提供しており、X Firehoseに含まれる地理データ量の大幅な増加につながっています。この位置情報メタデータは、正規化されていないユーザー生成の自由形式文字列です。約30%のアカウントには、国レベルまで解決可能なプロフィールジオのメタデータがあります。

投稿のジオ情報と同様に、照合方法や利用可能な期間は、使用しているHistorical APIによって異なります。Historical PowerTrackでは、これらの自由形式文字列に対して独自のカスタム照合を試すことができます。これを容易にするため、Xは可能な場合にジオコーディングを実行し、正規化されたメタデータと対応するオペレーターを提供する[Profile Geo Enrichment](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo)も提供しています。Profile Geoオペレーターは、Historical PowerTrackとSearch APIの両方で利用可能です。Historical PowerTrackでは、これらのProfile Geoメタデータは2014年6月から利用可能です。Search APIでは、このメタデータは2015年2月から利用可能です。

#### 共有リンクとメディア

Webページのリンク、写真、動画の共有は、Xにおける基本的なユースケースです。サービス初期には、これらはすべて投稿本文内にURLリンクを含める形で行われていました。2011年には、Xのユーザーインターフェースに写真の直接共有機能が統合され、2016年にはネイティブ動画が追加されました。

この経緯により、これらのコンテンツに一致させるための各種フィルタリング演算子が用意されています。投稿に共有リンク、写真、動画が含まれているかどうかで一致させる演算子群が存在します。また、Xで共有されるURLの多くは、投稿の文字数を節約するために短縮（例：bitlyやtinyurlなどのサービスで生成）されるため、Xは一致対象として利用できる完全な展開URLを生成するデータエンリッチメントを提供しています。例えば、Xと早期警戒システムに関するリンクを含む投稿に一致させたい場合、「severe weather communication」を参照するフィルターは、この[http://bit.ly/1XV1tG4](http://bit.ly/1XV1tG4)というURLを含む投稿に一致します。

2012年3月に、[expanded URL enrichment](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls)が導入されました。これ以前は、投稿のペイロードにはユーザーが提供したURLのみが含まれていました。そのため、ユーザーが短縮URLを含めた場合、関心のある（展開後の）URLでのマッチングは困難になり得ました。Historical PowerTrackおよびSearch APIでは、これらのメタデータは2012年3月から利用可能です。

2016年7月には、enhanced URL enrichmentが導入されました。この拡張版では、投稿のペイロードにWebサイトのHTMLタイトルと説明が付与され、これらに対してマッチングするための演算子も提供されます。Historical PowerTrackでは、これらのメタデータは2016年7月から利用可能です。Search APIでは、これらのメタデータは2014年12月から順次提供開始されています。

2016年9月、Xは「ネイティブ添付」を導入し、末尾の共有リンクが投稿の140文字制限にカウントされないようにしました。これらの共有リンクには、両方のURLエンリッチメントが引き続き適用されます。

URLフィルタリングに関する他の製品固有の詳細については、該当する記事を参照してください。
