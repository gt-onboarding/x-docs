---
title: レート制限
---

毎日、何千人もの開発者が X API にリクエストを送信しています。膨大なリクエスト量を管理するため、送信可能なリクエスト数に制限を設けています。これらの制限により、開発者コミュニティが頼る信頼性が高くスケーラブルな API を提供できます。

許可される最大リクエスト数は時間間隔に基づき、通常は特定の期間（ウィンドウ）で定義されます。最も一般的な間隔は 15 分です。たとえば、15 分あたり 900 リクエストの制限があるエンドポイントでは、任意の 15 分間に最大 900 リクエストを行えます。

レート制限は認証方式によって異なります。たとえば、[OAuth 1.0a User Context](/ja/resources/fundamentals/authentication) を使用する場合、各ユーザーの[Access Tokens](/ja/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) ごとに期間単位のレート制限が適用されます。これに対し、[OAuth 2.0 Bearer Token](/ja/resources/fundamentals/authentication#oauth-2-0) を使用する場合は、アプリごとに時間期間単位の個別の制限が適用されます。これらの制限を超えると、エラーが返されます。

<div id="rate-limits-and-authentication-method">
  ## レート制限と認証方法
</div>

レート制限は、開発者アプリおよびユーザーアクセス・トークンの両方のレベルで設定されています。

- **[OAuth 2.0 Bearer Token](/ja/resources/fundamentals/authentication/oauth-2-0/application-only): アプリのレート制限**
  この方法では、開発者アプリに代わって一定数のリクエストを実行できます。この認証方法を使用する場合、制限は Bearer Token を用いて行われたリクエスト数によって決まります。
  - 例: 15分間に450リクエストの制限がある場合、その時間内にアプリに代わって450リクエストを実行できます。

- **[OAuth 1.0a User Context](/ja/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens): ユーザーのレート制限**
  この方法では、ユーザーアクセス・トークンによって識別される X ユーザーに代わってリクエストを実行できます。たとえば、投稿のプライベートメトリクスを取得する場合は、[3-legged OAuth flow](/ja/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) で生成したそのユーザーのユーザーアクセス・トークンで認証します。
  - 例: ユーザーごとに15分あたり900リクエストの制限がある場合、その時間枠内でユーザーごとに最大900リクエストを実行できます。

<div id="http-headers-and-response-codes">
  ## HTTP ヘッダーとレスポンスコード
</div>

直近のリクエストに基づき、特定のレート制限内でアプリケーションの状況を把握するために、HTTP ヘッダーを使用します。

- `x-rate-limit-limit`: 当該エンドポイントのレート制限上限
- `x-rate-limit-remaining`: 15 分間のウィンドウで残っているリクエスト数
- `x-rate-limit-reset`: レート制限がリセットされるまでの残り時間（UTC のエポック秒）

<div id="error-responses">
  ### エラーレスポンス
</div>

アプリケーションがエンドポイントのレート制限を超過した場合、API はレスポンス本文に次のエラーメッセージを含む [HTTP 429 “Too Many Requests”](http://tools.ietf.org/html/rfc6585) レスポンスを返します：

```json
{ "errors": [ { "code": 88, "message": "Rate limit exceeded" } ] }
```

<div id="recovering-from-a-rate-limit">
  ## レート制限からの復旧
</div>

これらのレート制限を超えると、エンドポイントは 429 “Too Many Requests” エラーを返します。以下で説明するように、レート制限エラーが発生した場合のベストプラクティスは、リセット時刻を示す HTTP ヘッダーを確認し、それまでリクエストを停止することです。<br />“Too Many Requests” もしくはレート制限エラーが発生した場合は、リクエスト頻度を下げる必要があります。レート制限エラーに到達したら、`x-rate-limit-reset:` HTTP ヘッダーを確認して、いつレート制限がリセットされるかを把握できます。<br /><br />もう一つの一般的な手法は指数バックオフで、リクエスト間隔を小さく（例：数秒）始め、再試行ごとに倍増させます。これは、リクエストが成功するか、あるいは合理的な最大間隔（例：数分）に達するまで続けます。<br /><br />理想的には、クライアント側が既存のレート制限を把握しており、現在超過しているウィンドウが終了するまでリクエストを一時停止できます。15 分の制限を超えた場合は、再試行前に 1～2 分待つのが妥当です。<br /><br />なお、リクエスト数の上限とは別に、Standard Basic レベルのアクセスでは、recent search および filtered stream エンドポイントから月間最大 500,000 件の投稿を取得できます。投稿数の月間上限を超えた場合は、アプリで通知を出し、月ごとの登録日を把握したうえで、その日までリクエストを控えるのが適切です。

<div id="prioritize-active-users">
  #### アクティブユーザーを優先する
</div>

サイトで多数の X ユーザーを管理している場合（例：最新のステータスや X 上での利用状況の統計を取得するなど）、直近にサイトへサインインしたユーザーのみを対象にデータをリクエストすることを検討してください。

<div id="adapt-to-the-search-results">
  #### 検索結果に合わせて調整する
</div>

アプリケーションが多数の検索語を監視している場合、結果が得られない検索については、結果が得られる検索より照会頻度を下げてください。バックオフを用いることで、人気のあるクエリは最新の状態を維持しつつ、めったに変化しないクエリへのリクエストに計算資源を浪費せずに済みます。あるいは、フィルタードストリームのエンドポイントを使用し、検索クエリでフィルタリングすることも検討してください。

<div id="denylist">
  #### 拒否リスト
</div>

アプリケーションがレート制限を悪用した場合、拒否されます。拒否されたアプリは X API からのレスポンスを受け取れません。ご自身またはご使用のアプリケーションが拒否され、誤りだと思われる場合は、[Platform Support forms](https://support.x.com/forms/platform) からサポートを依頼してください。以下の情報を含めてください。

1. アプリケーションが拒否されたと考える理由を説明してください。
2. すでにレート制限の対象外になっている場合は、どのように問題を解決したかを詳細に記載してください。
