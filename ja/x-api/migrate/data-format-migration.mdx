<div id="introduction">
  ## はじめに
</div>

X API v2 の提供開始に伴い、新しいデータレスポンス形式と、各種オブジェクトおよびフィールドのリクエスト方法を採用しました。これを総称して「X API v2 形式」と呼びます。 

「一般的な相違点」セクションでは、Standard および Enterprise ユーザーに関係する変更点を確認できます。さらに、[standard v1.1 の Native 形式](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview)、Enterprise 向けの [Native Enriched 形式](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object)、Enterprise 向けの [Activity Streams 形式](/ja/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#activity-object)についての個別ガイドも用意しており、フィールドのマッピングや、新しい v2 フィールドをリクエストする際に使用すべきフィールドおよび拡張（expansions）を説明しています。 

- [Native 形式から X API v2 へ（standard v1.1）](/ja/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2) 
- [Native Enriched から X API v2 へ（enterprise）](/ja/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2)
- [Activity Streams から X API v2 へ（enterprise）](/ja/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2)

また、[X API v1.1 のデータ形式](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview)と [X API v2 形式](/ja/x-api/introduction)の差分を素早く可視化できる[ビジュアルデータ形式移行ツール](/ja/x-api/migrate/data-format-migration#visual-data-format-migration-tool)もぜひご活用ください。

<div id="general-differences">
  ### 全般的な相違点
</div>

<div id="requesting-objects-and-fields">
  #### オブジェクトとフィールドのリクエスト
</div>

pre-v2 エンドポイントと v2 の大きな違いの一つは、v2 ではデフォルトで返されるフィールドがごく一部に限られるのに対し、standard、premium、enterprise の各エンドポイントではデフォルトでほとんどのフィールドが返される点です。v2 では、デフォルトを超える追加データを明示的に取得するために、[fields](/ja/x-api/fundamentals/fields) と [expansions](/ja/x-api/fundamentals/expansions) というパラメータを使用します。これにより、不要なフィールドを取り込むことなく、必要なデータだけをリクエストできます。

プライマリのデータオブジェクトに関連するフィールドをリクエストした場合、それらはデフォルトの値とともにそのプライマリオブジェクト内で返されます。一方、expansions パラメータで拡張オブジェクトをリクエストした場合は、セカンダリのオブジェクトが新しい includes オブジェクト内で返されます。両方に含まれる ID フィールドを使用することで、includes 内の拡張オブジェクトをプライマリオブジェクトに対応付けられます。

たとえば、v2 の [Post lookup](/ja/x-api/posts/lookup/introduction) エンドポイントを使用し、リクエストに expansions=author\_id パラメータを含めた場合、プライマリの Post オブジェクト内に author\_id フィールドが返されるとともに、includes オブジェクト内には各 Post につき 1 つの user オブジェクトが返されます。各 user オブジェクトには、Post オブジェクトと照合するために使用できるデフォルトの id フィールドが含まれます。以下はその例です:

```
{
  "data": [
    {
      "author_id": "2244994945",
      "id": "1397568983931392004",
      "text": "The Twitter Developer Platform. Ooh la la! https://t.co/iGTdPXBfOv https://t.co/Ze8z8EODdg"
    }
  ],
  "includes": {
    "users": [
      {
        "id": "2244994945",
        "name": "Twitter Dev",
        "username": "TwitterDev"
      }
    ]
  }
}
```

<div id="updated-json-design">
  #### 更新された JSON 設計
</div>

特定のフィールドのリクエスト方法の変更に加え、X API v2 では、API が返すオブジェクト（[Post](/ja/x-api/fundamentals/data-dictionary#tweet) や [user](/ja/x-api/fundamentals/data-dictionary#user) オブジェクトを含む）の JSON 設計も刷新しています。

- JSON のルートレベルでは、従来のエンドポイントは Post オブジェクトを **statuses** 配列で返しますが、X API v2 は **data** 配列で返します。
- リツイートおよび引用の「statuses」を参照する代わりに、X API v2 の JSON ではリツイートおよび引用の Tweet を参照します。**contributors** や **user.translator\_type** など、多くのレガシーおよび非推奨フィールドは削除されています。
- **favorites**（Post オブジェクト内）と **favourites**（user オブジェクト内）の両方を使用する代わりに、X API v2 では用語として like を使用します。
- X では、値を持たない JSON 値（例: null）はペイロードに含めないという慣例を採用しています。Post と user の属性は、非 null の値がある場合にのみ含まれます。 
   

<div id="new-v2-fields">
  #### 新しい v2 フィールド
</div>

[Post オブジェクト](/ja/x-api/fundamentals/data-dictionary#tweet) に、次の新しいフィールド群を追加しました。

- [conversation\_id](/ja/x-api/fundamentals/conversation-id) フィールド
- 新たに追加された [annotations](/ja/x-api/fundamentals/post-annotations) フィールド（context と entities）の 2 種類
- いくつかの新しい [metrics](/ja/x-api/fundamentals/metrics) フィールド
- 新しい reply\_setting フィールド（特定の投稿に誰が返信できるかを示します）

<div id="migrating-from-standard-v11s-data-format-to-v2">
  ### 標準 v1.1 のデータ形式から v2 への移行
</div>

まだお読みでなければ、まずは[データ形式の移行](/ja/x-api/migrate/data-format-migration)のイントロダクションをご覧になることをおすすめします。[X API v1.1 のデータ形式](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview)と[X API v2 の形式](/ja/x-api/fundamentals/data-dictionary)の違いを素早く確認できる[ビジュアルデータ形式移行ツール](/ja/x-api/migrate/data-format-migration#visual-data-format-migration-tool)もご活用ください。

標準 v1.1 のデータ形式はネイティブ形式とも呼ばれ、[standard v1.1](https://developer.x.com/en/docs/twitter-api/v1) の各エンドポイントで提供される主要な形式です。

プレミアム製品をご利用の場合は、[ネイティブ拡張（enriched）ガイド](/ja/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2)をご参照ください。エンタープライズのお客様は、Gnip コンソールでの設定に応じて、ネイティブ拡張またはアクティビティストリームをご利用の場合があります。

<div id="migrating-from-native-enriched-data-format-to-v2">
  ### ネイティブ拡張データ形式から v2 への移行
</div>

ネイティブ拡張データ形式は、当社の[エンタープライズ](/ja/x-api/enterprise-gnip-2.0/enterprise-gnip)製品で使用されています。

ネイティブ拡張データ形式は、編集済みツイートのメタデータを提供できるよう更新されました。編集機能に関するメタデータの詳細は、[Edit Tweets の基礎](/ja/x-api/enterprise-gnip-2.0/fundamentals/edit-tweets)ページをご覧ください。

標準の v1.1 エンドポイントを利用している場合は、[standard v1.1 から v2 へのガイド](/ja/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2)を参照してください。Activity Streams を利用するエンタープライズ製品をご利用の場合は、[Activity Streams から v2](/ja/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2)へのガイドも用意しています。

X API v2 は、[Tweet](/ja/x-api/fundamentals/data-dictionary#tweet)および[ユーザー](/ja/x-api/fundamentals/data-dictionary#user)オブジェクト向けに新しい JSON 設計を導入しています。

- JSON のルートレベルでは、ネイティブ拡張形式は Tweet オブジェクトを results 配列で返しますが、X API v2 は data 配列で返します。
- Tweet オブジェクト内の favorites と user オブジェクト内の favourites の両方を用いる代わりに、X API v2 では like という用語を使用します。
- X では、値を持たない JSON 値（例: null）はペイロードに含めないという規約を採用しています。Tweet と user の属性は、非 null の値がある場合にのみ含まれます。
- v2 のすべての id フィールドは文字列形式になります
   

新しい JSON 形式への変更に加えて、Tweet オブジェクトに以下の新しいフィールド群を導入しました。

- conversation\_id
- reply\_settings
- メディアの alt\_text
- 2 つの新しい[アノテーション](/ja/x-api/fundamentals/post-annotations)フィールド（context と entities）
- 複数の新しい[メトリクス](/ja/x-api/fundamentals/metrics)フィールド
- 複数の新しい[投票](/ja/x-api/fundamentals/data-dictionary#poll)フィールド
   

多くのレガシーおよび非推奨フィールドは削除されます。

- contributors
- 一部の entities.media および extended\_entities.media フィールド
- filter\_level
- timestamp\_ms
- truncated

<div id="migrating-from-activity-streams-data-format-to-v2">
  ### Activity Streams データ形式から v2 への移行
</div>

Activity Streams データ形式は、当社の[エンタープライズ](/ja/x-api/enterprise-gnip-2.0/enterprise-gnip)製品で利用可能です。

Activity Streams データ形式は、編集済み投稿のメタデータを提供できるよう更新されました。編集投稿のメタデータについては、[Edit Tweets の基礎](/ja/x-api/enterprise-gnip-2.0/fundamentals/edit-tweets)をご覧ください。

標準 v1.1 エンドポイントをご利用の場合は、[標準 v1.1 から v2 へのガイド](/ja/x-api/migrate/overview)を参照してください。プレミアムエンドポイント、またはエンタープライズ向け Native Enriched 形式をご利用の場合は、[Native Enriched から v2 へのガイド](/ja/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2)を参照してください。

X API v2 では、[Post](/ja/x-api/fundamentals/data-dictionary#tweet) および [user](/ja/x-api/fundamentals/data-dictionary#user) オブジェクトの新しい JSON 設計が導入されています。

- JSON のルートレベルでは、Activity Streams 形式は results 配列内に Tweet オブジェクトを返しますが、X API v2 は data 配列を返します。
- リツイートおよび引用の「アクティビティ」と表現する代わりに、X API v2 の JSON ではリツイートおよび引用投稿として扱います。
- Tweet オブジェクトの favorites と user オブジェクトの favourites を併用する代わりに、X API v2 では like を用語として使用します。
- X では、値がない JSON 値（例：null）はペイロードに書き出さないという規約を採用しています。Tweet および user の属性は、非 null の値がある場合にのみ含まれます。
- v2 のすべての id フィールドは文字列形式になります。
   

新しい JSON 形式への変更に加えて、Tweet オブジェクトに以下の新しいフィールドを導入しました。

- conversation\_id
- reply\_settings
- メディアの alt\_text
- 2 つの新しい[アノテーション](/ja/x-api/fundamentals/post-annotations)フィールド（context と entities）
- 複数の新しい[メトリクス](/ja/x-api/fundamentals/metrics)フィールド
- 複数の新しい[投票](/ja/x-api/fundamentals/data-dictionary#poll)フィールド
   

多くのレガシーおよび非推奨フィールドは削除または置換されます。

- display\_text\_range
- generator
- gnip
- link
- objectType
- provider
- twitter\_entities.symbols は data.entities.cashtags に置換
- 一部の twitter\_extended\_entities.media および twitter\_entities.media フィールド
- twitter\_filter\_level
- twitterTimeZone
- verb
