  ## 役立つツール
</div>

このエンドポイントの統合に役立つ主要な概念に入る前に、以下に挙げる内容にあらかじめ慣れておくことをおすすめします。

<div id="postman">
  ### Postman
</div>

Postman は、エンドポイントのテストに便利な優れたツールです。各 Postman リクエストには、利用可能な内容をすばやく把握できるよう、すべてのパスおよびボディパラメータが含まれています。Postman コレクションの詳細については、["Using Postman"](/ja/tutorials/postman-getting-started) のページをご覧ください。

</div>

お好みのプログラミング言語でこのエンドポイントをセットアップしたい方へ。出発点として使える各種コードサンプルを[GitHub ページ](https://github.com/xdevplatform/Twitter-API-v2-sample-code)で公開しています。
 

<div id="third-party-libraries">
  ### サードパーティライブラリ
</div>

<div id="authentication">
  ### 認証
</div>

すべての X API v2 エンドポイントでは、キーやトークンとも呼ばれる一式の認証情報でリクエストに認証を付与する必要があります。

これらのエンドポイントには [OAuth 1.0a User Context](/ja/resources/fundamentals/authentication) の使用が必要です。つまり、リクエストを成功させるには、API キーとユーザーの Access Token のセットを使用する必要があります。Access Token は、あなたが代理でリクエストを行うユーザーに紐づいている必要があります。別のユーザー向けの Access Token を生成する場合は、そのユーザーが [3-legged OAuth flow](/ja/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) を用いてあなたの App を承認または認証する必要があります。

OAuth 1.0a は扱いが難しい場合があります。この認証方式に不慣れな場合は、[ライブラリ](/ja/x-api/tools-and-libraries/overview) を使用する、Postman のようなツールを使う、または OAuth 2.0 でリクエストを認証することをおすすめします。

[OAuth 2.0 Authorization Code with PKCE](/ja/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) は、アプリケーションのスコープに対するきめ細かな制御や、複数デバイス間にまたがる認可フローを可能にします。OAuth 2.0 では、ユーザーに代わって特定の権限を付与するための細粒度なスコープを個別に選択できます。

App で OAuth 2.0 を有効にするには、デベロッパーポータルの App 設定セクション内にある認証設定で有効化してください。

<div id="developer-portal-projects-and-developer-apps">
  ### 開発者ポータル、プロジェクト、および開発者アプリ
</div>

X API v2エンドポイントで動作する認証資格情報のセットを取得するには、\[開発者アカウント]、そのアカウント内で[プロジェクト](/ja/resources/fundamentals/projects)を設定し、そのプロジェクト内で[開発者アプリ](/ja/resources/fundamentals/developer-apps)を作成する必要があります。その後、開発者アプリ内でキーとトークンを確認できます。 
 

<div id="rate-limits">
  ### レート制限
</div>

毎日、何万もの開発者が X API にリクエストを送信しています。これらのリクエストの膨大な量を管理するため、各エンドポイントには[レート制限](https://developer.x.com/x-api/fundamentals/rate-limits)が設定されており、アプリまたは認証済みユーザーに代わって行えるリクエスト数が制限されています。

これらのエンドポイントはユーザーレベルでレート制限されています。つまり、あなたが代行してリクエストを行う認証済みユーザーは、どの開発者アプリ経由であっても、そのエンドポイントを呼び出せる回数が一定数に制限されます。POST メソッドのユーザー単位のレート制限は 15 分あたり 200 リクエストです。DELETE メソッドのレート制限は 15 分あたり 50 リクエストです。さらに、[3 時間あたり 300 リクエスト](https://blog.x.com/developer/en_us/topics/tools/2018/new-developer-requirements-to-protect-our-platform)という制限があり、「manage Posts」または「manage Retweets」の権限で作成された投稿（Post）も含まれます。
 

<div id="source-labels">
  ### ソースラベル
</div>

アプリ名とウェブサイトのURLは、アプリケーションによってプログラムで作成された投稿のメタデータ内に[ソースラベル](https://help.x.com/en/using-twitter/how-to-tweet#source-labels)として表示されます。X アプリのユースケースを変更する場合は、これらの設定でユースケースを必ず更新し、[開発者規約](https://developer.x.com/content/developer-twitter/en/developer-terms/agreement-and-policy)への準拠を確保してください。
 

<div id="profile-settings">
  ### プロフィール設定
</div>

プロフィール設定で位置情報（ジオ）を有効にしている場合にのみ、投稿に位置情報を追加できます。位置情報を有効にしていない場合でも、リクエスト本文に location パラメータを含めること自体は可能ですが、投稿には反映されません。画像へのユーザーのタグ付けについても同様です。タグ付け対象のユーザーが写真のタグ付けを有効にしていない場合、投稿自体は正常に作成されても、タグ付けされたユーザー一覧にそのユーザー名は表示されません。 
 

<div id="adding-media-to-a-post">
  ### ポストへのメディアの追加
</div>

現在、X API v2 ではメディアを完全にアップロードする手段は提供されていません。ただし、過去にアップロード済みのメディアをポストに添付することは可能です。すでに [v1 のメディアエンドポイント](https://developer.x.com/en/docs/twitter-api/v1/media/upload-media/api-reference/post-media-upload)や [X Media Studio](https://media.x.com/en/articles/products/2018/media-studio) を使ってアップロードされたメディア ID を使用できます。これらのメディア ID は、ご自身のもの、または認証済みユーザーのものである必要があります。
