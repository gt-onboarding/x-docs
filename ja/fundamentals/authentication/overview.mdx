X の API は膨大なデータを扱います。このデータを開発者とユーザー双方にとって安全に保つ手段が認証です。認証にはいくつかの方式があり、以下に一覧で示します。

ほとんどの開発者は、クライアントライブラリがこれらの手間や複雑さを自動的に処理するため、認証の細かな実装に対応する必要はありません。

利用可能なクライアントライブラリの一覧は、[Tools and libraries](/ja/resources/tools-and-libraries) ページで確認できます。

<div id="authentication-methods">
  ## 認証方法
</div>

<CardGroup cols={2}>
  <Card title="OAuth 1.0a User Context" href="/ja/resources/fundamentals/authentication/oauth-1-0a/api-key-and-secret">
    OAuth 1.0a は、認可された X Developer アプリが非公開のアカウント情報にアクセスしたり、X アカウントに代わって操作を実行したりすることを可能にします。

    <br />

    [**詳細はこちら**](/ja/resources/fundamentals/authentication/oauth-1-0a/api-key-and-secret)
  </Card>

  <Card title="App only" href="/ja/resources/fundamentals/authentication/oauth-2-0/overview">
    App only Access Token により、X Developer アプリは X 上で公開されている情報にアクセスできます。

    <br />

    [**詳細はこちら**](/ja/resources/fundamentals/authentication/oauth-2-0/overview)
  </Card>

  <Card title="Basic authentication" href="/ja/resources/fundamentals/authentication/basic-auth">
    多くの X のエンタープライズ API では、HTTP ベーシック認証の使用が必要です。

    <br />

    [**詳細はこちら**](/ja/resources/fundamentals/authentication/basic-auth)
  </Card>

  <Card title="OAuth 2.0 Authorization Code Flow with PKCE" href="/ja/resources/fundamentals/authentication/oauth-2-0/authorization-code">
    OAuth 2.0 のユーザーコンテキストを使用すると、アプリケーションのスコープをより細かく制御しつつ、他のアカウントに代わって認証し、複数デバイスにまたがる認可フローを実行できます。

    <br />

    [**詳細はこちら**](/ja/resources/fundamentals/authentication/oauth-2-0/authorization-code)
  </Card>
</CardGroup>

<Note>
  **注記:**
  アプリの API Key と App only Access Token、ならびに個人の Access Token と Access Token Secret は、[developer portal](/ja/resources/fundamentals/developer-portal) の [X developer Apps](/ja/resources/fundamentals/developer-apps) セクションから取得できます。

  **他のユーザーに代わってリクエストを行う場合**は、そのユーザー向けに [3-legged OAuth フロー](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens) を使用して別途アクセストークン一式を生成し、そのユーザーのトークンを OAuth 1.0a のユーザーコンテキストまたは OAuth 2.0 のユーザーコンテキストのリクエストに添えて送信する必要があります。
</Note>
