私たちは、プライバシーは特権ではなく権利であり、会社の根幹に組み込まれていると考えています。X Developer Platform を利用し、開発者ポリシーを順守いただくことで、X 上の公共の会話に資するだけでなく、プライバシーへの当社のコミットメントを守るうえで重要な役割を担っていただきます。

ご自身とご自身のアプリのユーザー双方のデータを守るため、セキュアに構築することの重要性を改めてお伝えします。セキュリティ侵害の脅威から守る責任は開発者であるあなたにありますが、X を利用する人々を守る責任は私たちと共有しています。本ページでは、セキュアなアプリケーションの構築、およびデータとアクセスを可能な限り安全に保つための指針と期待事項を説明します。

<Note>X Developer Platform で利用できるセキュリティ技術（[認証](/ja/resources/fundamentals/authentication)、TLS、[アプリ権限](/ja/resources/fundamentals/developer-apps#app-permissions)）に加え、X ユーザーの観点における[サードパーティアプリやセッションの利用](https://help.x.com/en/managing-your-account/connect-or-revoke-access-to-third-party-apps)についてもご留意ください。</Note>

<div id="reporting-security-issues">
  ## セキュリティ問題の報告
</div>

X Developer Platform のユーザーは、セキュリティインシデントの発生が疑われる最初の時点から48時間以内に、[X の脆弱性報告プログラム](https://hackerone.com/twitter)を通じて X に通知する必要があります。

<div id="security-best-practices">
  ## セキュリティのベストプラクティス
</div>

X Developer Platform上やインターネット全般で開発する際には、常にこれらを念頭に置いてください。

<div id="security-by-design">
  ### 設計段階からのセキュリティ
</div>

脅威モデリングの監査や侵入テストの実施について、セキュリティ専門家の起用を検討してください。優れたセキュリティ企業は、問題を洗い出すために徹底的に調査します。X がこの考え方をどのように採用しているかは、[こちらのブログ記事](https://blog.x.com/en_us/topics/company/2019/privacy_data_protection.html)をご覧ください。

また、X はすべてのパートナーに対して次の事項の遵守を求めています。

- セキュアなリポジトリでコードを管理する。
- システム開発ライフサイクル（SDLC）全体を通じてリスク分析を実施する。
- SDLC の各工程でセキュリティ課題を特定し、対策を講じる。
- SDLC 全体を通じて環境を適切に分離する。
- すべてのテスト不具合を修正し、再テストして、クローズする。

<div id="monitor-and-get-alerted">
  ### 監視してアラートを受け取る
</div>

Webアプリに問題があると感じたとき、どのように確実に判別できますか？適切にログを収集し、重大な例外やエラーが発生した際には通知されるようにしておきましょう。主要な指標を可視化したダッシュボードを用意しておくと、異常の有無を一目で確認できます。

<div id="create-a-reporting-channel">
  ### レポート用の連絡経路を用意する
</div>

ユーザーがあなたのアプリで遭遇した潜在的なセキュリティ問題について、容易に連絡できるようにしましょう。X のユーザーやデータに影響を及ぼす問題が見つかった場合は、[その問題を X に報告](#report)する責任もあります。セキュリティインシデントが発生した際に、影響を受けたユーザーへ通知するためのアクションプラン／プロセスをあらかじめ用意しておいてください。

<div id="adequate-testing">
  ### 適切なテスト
</div>

エンドツーエンドテストが十分に網羅的で、権限のないアクセスなどのセキュリティシナリオにおける想定される失敗ケースを取り込むよう最新化されていることを確認してください。攻撃者の視点に立ち、X のデータや正当な機能への不正アクセスを阻止できることを検証するシステムテストを設計・実施してください。

<div id="securing-api-keys-and-tokens">
  ### API キーとトークンの保護
</div>

X プラットフォームの開発者は（開発者アプリがユーザーから許可を得ていることを前提に）、X に保存されている自分のデータおよびユーザーのデータにプログラムでアクセスできます。すべての API リクエストは、開発者アプリのキーとシークレット、場合によっては認可したユーザーの[アクセストークン](/ja/resources/fundamentals/authentication)を用いて、OAuth により[認証](/ja/resources/fundamentals/authentication)される必要があります。これらの認証情報を安全に管理する責任は開発者にあります。

推奨されるベストプラクティスは以下のとおりです。

- パスワード／トークンのローテーション（定期的な更新）を実施する。
- 必要に応じて機密データを常に暗号化し、処理の早い段階での復号は避ける。
- ユーザーのアクセストークンは暗号化ストアに保存する。
- 侵害の可能性がある場合は、キーやトークンを[再生成](/ja/resources/fundamentals/authentication#regenerate-api-keys-and-tokens)または[無効化](/ja/resources/fundamentals/authentication#post-oauth2-invalidate-token)する。

X 向け OAuth のデバッグや実装に関する詳細は、コミュニティフォーラムの[セキュリティカテゴリ](https://devcommunity.x.com/c/oauth/12)をご覧ください。

<div id="input-validation">
  ### 入力の検証
</div>

ユーザーが有効で信頼できるデータを提供すると想定しないでください。X API に渡される可能性のある、ユーザー由来のデータはすべてサニタイズしましょう。アプリケーションで受け入れる入力タイプを許可リストで厳格に定義し、リストにないものはすべて破棄してください。

<div id="encrypted-communication">
  ### 暗号化通信
</div>

X では、すべての API リクエストは TLS 経由で行う必要があります。可能な限り、自社サーバーとの通信も暗号化してください。

<div id="exposed-debugging-information">
  ### 露出しているデバッグ情報
</div>

デバッグ画面やログを通じて機密の X データや資格情報が露出していないことを必ず確認してください。アプリケーションが適切に構成されていない場合、Web フレームワークによってはデバッグ情報に容易にアクセスできてしまいます。デスクトップやモバイルの開発では、デバッグフラグやシンボルを有効にしたまま誤ってビルドを出荷してしまうことがあります。これらの設定に対するビルドチェックをデプロイ／ビルドプロセスに組み込んでください。さらに、レポート目的でスタックトレースやクラッシュダンプを共有する場合は、非公開の X ユーザーのデータが適切にマスキング（秘匿化）されていることを確認してください。

<div id="unfiltered-input-unescaped-output">
  ### フィルタなしの入力、エスケープなしの出力
</div>

入力検証を覚えやすくするアプローチのひとつに、FIEO（Filter Input, Escape Output）があります。

アプリケーション外部から入ってくるあらゆるデータ（X API のデータ、Cookie データ、ユーザーが入力したフォーム値、URL パラメータ、データベースからのデータなど）はフィルタし、アプリケーションが送信するすべての出力（データベースサーバーに送る SQL、ユーザーのブラウザに送る HTML、他システムに送る JSON 出力、シェルプログラムに送るコマンドなど）はエスケープしてください。

<div id="cross-site-scripting-xss">
  ### クロスサイトスクリプティング（XSS）
</div>

[XSS](https://owasp.org/) 攻撃は、多くの観点で、ウェブにおける最も一般的なセキュリティ問題です。攻撃者が自分の JavaScript コードをアプリケーションに注入できると、さまざまな不正行為が可能になります。信頼できないユーザー提供データを保存・表示する箇所はすべて、検証、サニタイズ、HTML エスケープを行う必要があります。これを正しく行うのは難しく、ハッカーには [XSS 攻撃を成立させる多様な手口](https://www.owasp.org/index.php/Types_of_Cross-Site_Scripting) があります。使用している言語やウェブ開発フレームワークには、クロスサイトスクリプティング対策として広く使われ実績のある仕組みが用意されているはずです。必ず利用してください。

<div id="sql-injection">
  ### SQLインジェクション
</div>

アプリケーションでデータベースを利用する場合は、[SQLインジェクション](http://en.wikipedia.org/wiki/SQL_injection)に注意が必要です。入力を受け付けるあらゆる箇所は、攻撃者が入力フィールドの想定範囲を抜け出してデータベースに侵入する可能性のある標的となり得ます。SQLインジェクションに体系的に対策したデータベースライブラリを使用してください。その枠組みから外れて独自のSQLを書く場合は、この種の攻撃にさらされないよう、厳密なテストを用意してください。SQLインジェクションへの主な対策は、SQL文の組み立て前にエスケープする方法と、パラメータ化された入力でステートメントを生成する方法の2つです。後者の方がプログラミング上のミスが起きにくいため、推奨されます。

<div id="cross-site-request-forgery-csrf">
  ### クロスサイトリクエストフォージェリ（CSRF）
</div>

あなたのアプリケーションへのリクエストは、本当にあなたのアプリケーションから送信されていますか？[CSRF](http://en.wikipedia.org/wiki/Cross-site_request_forgery)攻撃は、この不確実性を突き、サイトにログインしているユーザーに、処理を実行するURLを黙って開かせます。開発者向けのアプリの場合、攻撃者があなたのアプリを悪用して、ユーザーに望まない投稿をさせたり、スパムアカウントをフォローさせたりする可能性があります。

CSRFへの最も確実な対策は、信頼できる場所に保存したランダムなトークンをすべてのフォームに含め、正しいトークンがないフォームはエラーにすることです。最新のWebフレームワークにはこれを体系的に扱う方法があり、運が良ければデフォルトで有効になっていることもあります。簡易的な予防策（ただし、これだけで十分ではありません）としては、データの作成・変更・削除を伴う操作はPOSTリクエストを必須にすることが挙げられます。

<div id="lack-of-rate-limiting">
  ### レート制限の欠如
</div>

潜在的なスパマーや攻撃者の活動を遅らせるため、適切な場面では CAPTCHA の導入を検討してください。
