<div id="typical-response-structure">
  ## 典型的なレスポンス構造
</div>

成功したレスポンスは 200 番台の HTTP ステータスコードと、要求したオブジェクトの取得・作成・更新・削除の結果に加え、サーバーがリクエストをどのように解釈したかの表現を含む JSON ベースのペイロードで示されます。

リクエストが成功すると、レスポンスの一部として、送信したリクエスト内容を反映する `request` ノードが返されます。

_例:_ [GET accounts/abcdefg/campaigns?with\_deleted=true](/ja/x-ads-api/campaign-management#campaigns)

```json
{
  /* レスポンスのデータ... */,
  "request": {
    "params": {
      "account_id": "abcdefg",
      "with_deleted": "true"
    }
  }
}
```

JSON レスポンスの `data` フィールドには、参照したリソースに紐づく対象オブジェクトが含まれます。`data` ノードの形式は、結果が複数件になり得る場合は JSON 配列、結果が必ず 1 件に限られる場合は JSON オブジェクトとして返されます。まれに、通常はコレクションが含まれるはずのレスポンスが、代わりにハッシュマップを返すことがあります。この場合、その単一のハッシュマップは `type` フィールドで指定された型のオブジェクトであると見なしてください。

<div id="error-response-structure">
  ## エラーレスポンスの構造
</div>

エラーレスポンスは 200 系以外の HTTP コードで返されます。通常は JSON レスポンスが返却されますが、エラーによっては別種のボディが返されることもあります。レスポンス構造をパースできない状況では、HTTP コード本来の意味を優先してください。たとえば、HTML レスポンスとともに HTTP 404 が返される場合があります。このときは、コンテンツが見つからない（HTTP 404 は「Not Found」を意味します）と判断して問題ありません。

一般的なエラーレスポンスは、成功レスポンスと同様の構造です。エラーの内容はレスポンスの `errors` ノードで示されます。`errors/code` ノードには、解決判断をプログラムで行うために利用できる CAPS\_CASE の定数エラーコードが入ります。`errors/message` ノードには、（通常は）人間が読める英語のエラー説明が入ります。より詳細な情報を示すために、追加フィールドが付与される場合があります。

```json
{
  "errors": [
    {
      "parameter": "start_time",
      "details": "invalid date",
      "code": "INVALID_PARAMETER",
      "value": "",
      "message": "Expected time, got \"\" for start_time"
    }
  ],
  "request": {
    "params": {
      "account_id": "hkk5"
    }
  }
}
```

上記の例では、アナリティクス用エンドポイントに対し、`start_time` パラメータに無効な値を指定してリクエストしています。無効なパラメータを含むリクエストの `errors/code` は `INVALID_PARAMETER` です。
