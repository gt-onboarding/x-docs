<div id="overview">
  ## Descripción general
</div>

Si ya tienes Apps, puedes verlas, editarlas o eliminarlas a través de la [página de Apps del portal para desarrolladores](https://developer.x.com/en/portal/projects-and-apps).

Acceder a la API de X y a la API de Anuncios de X requiere un conjunto de credenciales de [autenticación](/es/resources/fundamentals/authentication), también conocidas como claves y tokens, que debes incluir con cada solicitud. Estas credenciales pueden presentarse en diferentes formas según el tipo de autenticación que requiera el endpoint específico que estés utilizando.

Estos son los diferentes tipos de credenciales que puedes generar en tu App y cómo utilizarlas:

- **[API Key and Secret](/es/resources/fundamentals/authentication#api-key-and-secret):** Básicamente, el nombre de usuario y la contraseña de tu App. Las usarás para autenticar solicitudes que requieran [OAuth 1.0a User Context](/es/resources/fundamentals/authentication#oauth-1-0a-2) o para generar otros tokens, como los Access Tokens de usuario o el App Access Token.

- **[Access Token and Secret](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow):** En general, los Access Tokens representan al usuario en cuyo nombre realizas la solicitud. Los que puedes generar a través del portal para desarrolladores representan al usuario propietario de la App. Los usarás para autenticar solicitudes que requieran [OAuth 1.0a User Context](/es/resources/fundamentals/authentication#oauth-1-0a-2). Si deseas realizar solicitudes en nombre de otro usuario, deberás usar el flujo OAuth con 3 participantes (3-legged OAuth) para que te autorice.

- **[Client ID and Client Secret](/es/resources/fundamentals/authentication#oauth-2-0):** Estas credenciales se usan para obtener un Access Token de usuario con autenticación OAuth 2.0. Al igual que con OAuth 1.0a, los Access Tokens de usuario se emplean para autenticar solicitudes que proporcionan información privada de la cuenta o realizan acciones en nombre de otra cuenta, pero con un alcance de permisos más granular para un mayor control sobre el acceso que tiene la aplicación cliente al usuario.

- **[App only Access Token](/es/resources/fundamentals/authentication#bearer-token-also-known-as-app-only):** Usarás este token al realizar solicitudes a endpoints que responden con información disponible públicamente en X.

Además de generar las claves y tokens necesarios para realizar solicitudes a la API de X, también podrás configurar [permisos](/es/resources/fundamentals/developer-apps#app-permissions), documentar el caso de uso o propósito de la App, definir una [callback URL](/es/resources/fundamentals/developer-apps#callback-urls) y modificar otras configuraciones relacionadas con tu entorno de desarrollo de Apps desde el [management dashboard](https://developer.x.com/en/portal/projects-and-apps).

<div id="apps-and-projects">
  ### Apps y Proyectos
</div>

Puede usar Apps y [Proyectos](/es/resources/fundamentals/projects) para organizar su trabajo con la Plataforma de Desarrolladores de X según el caso de uso. Cada Proyecto puede incluir una App con el plan gratuito de la API de X, hasta dos Apps con el plan Básico y hasta tres Apps con el plan Pro.

Si desea acceder a los nuevos endpoints de la [API de X v2](/es/x-api/introduction), deberá usar las claves y los tokens de una App asociada a un Proyecto.

Si tiene Apps creadas antes del lanzamiento de Proyectos, serán visibles en la sección titulada "Standalone Apps". Las Standalone Apps son Apps fuera de la estructura de Proyectos. Si adjunta una Standalone App a un Proyecto, podrá realizar solicitudes a los endpoints v2.

<div id="developer-portal-dashboard">
  ### Panel del portal para desarrolladores
</div>

Puede [visitar el panel](https://developer.x.com/content/developer-twitter/en/apps) para gestionar las Apps asociadas a su cuenta. Para obtener más información, visite nuestra página de documentación sobre el [portal para desarrolladores](/es/resources/fundamentals/developer-portal). El panel permite a los desarrolladores realizar de forma rápida y sencilla las siguientes tareas:

- Ver sus Apps independientes existentes y su App ID asociado.
- Crear un nuevo Proyecto, App o App independiente.
- Eliminar un Proyecto o App que no se utilice.
- Revisar o actualizar la configuración de una App específica, incluidos el nombre, la descripción, el sitio web, la URL de callback y los [permisos](/es/resources/fundamentals/developer-apps#app-permissions).
- Regenerar credenciales específicas de la App como la API Key y Secret, el App Access Token y los Access Tokens de usuario del propietario de la App.

<div id="signing-up-for-access">
  ### Registro para obtener acceso
</div>

Si ya tienes Apps de X, puedes ver y editar tus Apps en el [panel de Apps](https://developer.x.com/content/developer-twitter/en/portal/projects-and-apps) de X si has iniciado sesión en tu cuenta de X en developer.x.com. Ten en cuenta que **no** necesitas crear una cuenta nueva para administrar todas las Apps que se crearon anteriormente en developer.x.com.

Si necesitas crear una nueva App de X, debes [registrarte para obtener una cuenta de desarrollador](https://developer.x.com/en/portal/petition/essential/basic-info). Si eres miembro de una [cuenta de equipo](/es/resources/fundamentals/developer-portal#team-management), debes tener asignado el rol de administrador para crear una nueva App.

<div id="automated-account-labeling-for-bots">
  ### Etiquetado de Cuenta automatizada para bots
</div>

Puedes añadir una etiqueta de Cuenta automatizada a las cuentas de tu bot para informar a los usuarios de X que tu bot es una cuenta automatizada. Estos bots realizan acciones programadas a través de la API de X. Al añadir una etiqueta de Cuenta automatizada a tu bot, generas confianza con tu audiencia, legitimas tu cuenta y te diferencias de los bots de spam. Esto ayuda a que las personas en X comprendan mejor el propósito de tu cuenta al interactuar con tu bot.

Para adjuntar una etiqueta a la cuenta de tu bot, sigue estos pasos:

1. Ve a la configuración de tu cuenta
2. Selecciona "Tu cuenta"
3. Selecciona "Automatización"
4. Selecciona "Administrar cuenta"
5. A continuación, selecciona la cuenta de X que ejecuta tu bot
6. Introduce tu contraseña para iniciar sesión
7. Por último, deberías ver la confirmación de que la etiqueta se ha aplicado a tu cuenta.

  ## Gestión de Apps

<div id="introduction">
  ### Introducción
</div>

El [panel de X App](https://developer.x.com/en/portal/projects-and-apps) permite a los desarrolladores realizar de forma rápida y sencilla las siguientes tareas:

- Ver tus Apps y [Proyectos](/es/resources/fundamentals/projects) existentes y su App ID asociado.
- Crear un nuevo Proyecto o una App independiente.
- Eliminar un Proyecto, una App o una App independiente.
- Abrir la configuración de una App específica haciendo clic en la configuración de la App. Dentro de la configuración, puedes ver los detalles de la App, las claves y los tokens, y los permisos.
- Actualizar la configuración de autenticación de usuario de tu App para usar OAuth 1.0a u OAuth 2.0.

<Note>
  **Nota:**

  Todas las claves y tokens de la App ya no son visibles en el Portal para Desarrolladores y deben guardarse de forma segura en cuanto se generen. Recomendamos usar un gestor de contraseñas para almacenar tus claves y tokens.

  Puedes mostrar una pista de API Key para ayudarte a vincular tu credencial con su App correspondiente.
</Note>

<div id="app-settings">
  ### Configuración de la app
</div>

<div id="app-details">
  #### Detalles de la App
</div>

Aquí puedes editar el ícono de la App, el nombre de la App, la descripción de la App, la URL de tu sitio web, [callback URLs/redirect URIs](/es/resources/fundamentals/developer-apps#callback-urls), la URL de los términos del servicio, la URL de la política de privacidad, el nombre de la organización, la URL de la organización y el propósito o caso de uso de la App.

OAuth 2.0 y OAuth 1.0a son métodos de autenticación que permiten a los usuarios iniciar sesión en tu App con X. También permiten que tu App realice solicitudes específicas en nombre de usuarios autenticados. Puedes activar uno o ambos métodos para tu App.

Es importante mantener esta información actualizada. El nombre de tu App y la URL de tu sitio web se mostrarán como la fuente dentro de los metadatos de cualquier tuit creado de forma programática por tu aplicación. Si cambias el caso de uso de una App de X, asegúrate de actualizarlo en esta configuración para garantizar el cumplimiento de los [Términos para Desarrolladores](https://developer.x.com/content/developer-twitter/en/developer-terms/agreement-and-policy).

Si tu aplicación tiene una etiqueta que muestra “suspended”, esto se debe a que tu App infringe uno o más de los [Términos para Desarrolladores](https://developer.x.com/en/developer-terms.html) de X, como nuestras [reglas de automatización](https://help.x.com/en/rules-and-policies/twitter-automation). El equipo de políticas de la plataforma para desarrolladores se comunicará con los desarrolladores a través de la dirección de correo electrónico configurada en la cuenta de X del propietario de la App; para revisar esa dirección, consulta la [configuración de tu cuenta de X](https://x.com/settings/your_twitter_data/account). Los correos electrónicos de notificación sobre suspensiones se enviarán a esa dirección con un asunto similar a “Application suspension notice” e incluirán información específica sobre qué hacer a continuación. Para trabajar con el equipo de X y resolver las suspensiones, revisa tu correo electrónico para obtener instrucciones específicas o usa nuestro [formulario de ayuda de la plataforma](https://help.x.com/forms/platform).

<div id="keys-and-tokens">
  #### Claves y tokens
</div>

Dentro de una App en un Proyecto o mediante una App independiente, puedes ver, regenerar o revocar los siguientes tokens:

- [API Key (Consumer Key) y API Secret (Consumer Secret)](/es/resources/fundamentals/authentication#api-key-and-secret)
- [App Access Token](/es/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)
- [User Access Token y Access Token Secret](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) -
  El Access Token y el Access Token Secret disponibles en el portal para desarrolladores corresponden al usuario propietario de la App.

**Ten en cuenta**: si deseas realizar solicitudes en nombre de otro usuario (es decir, distinto del propietario de la App), deberás usar el [flujo OAuth 1.0a de 3 patas](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens) o el flujo de OAuth 2.0 [Authorization Code with PKCE](/es/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) para generar un conjunto de Access Tokens de usuario. Luego, usarás estos tokens específicos del usuario en tu solicitud a la API.

<div id="user-authentication-settings">
  #### Configuración de autenticación de usuario
</div>

Puedes seleccionar la configuración de autenticación de tu App como OAuth 1.0a u OAuth 2.0. OAuth 2.0 solo se puede usar con la API de X v2. OAuth 2.0 te permite elegir ámbitos específicos y granulares que otorgan permisos precisos en nombre de un usuario. OAuth 1.0a puede usarse con la API de X v1.1 y v2 y utiliza una autorización amplia con ámbitos generales.

<div id="oauth-10a-application-user-permissions">
  #### Permisos de OAuth 1.0a de aplicación-usuario
</div>

Si eres el propietario de la App, puedes ajustar los permisos de la App (solo lectura; lectura y escritura; o lectura, escritura y mensajes directos). Esto controla a qué recursos y eventos tienes acceso a través de las API de X (nota: algunos recursos requieren permisos adicionales directamente de X).

También puedes activar o desactivar en esta página la capacidad de tus Apps para solicitar direcciones de correo electrónico de los usuarios (esto requiere que en la página “Detalles de la App” se incluyan una URL de Términos del servicio y una URL de Política de privacidad).

<div id="oauth-20-type-of-app">
  #### Tipo de App para OAuth 2.0
</div>

Si selecciona OAuth 2.0 como método de autenticación de usuario, deberá elegir el tipo de App que está creando. Sus opciones son App nativa, App de una sola página (SPA), App web, App automatizada o bot. Las Apps nativas y las Apps de una sola página son clientes públicos, y las Apps web y las Apps automatizadas o los bots son clientes confidenciales.

Los clientes confidenciales se autentican de forma segura con el servidor de autorización y mantienen protegido su secreto de cliente. Los clientes públicos son aplicaciones que normalmente se ejecutan en un navegador o en un dispositivo móvil y no pueden usar secretos de cliente. Si selecciona un tipo de App que sea cliente confidencial, se le proporcionará un secreto de cliente.

<div id="app-permissions">
  ## Permisos de la aplicación
</div>

<div id="oauth-10a-app-permissions">
  ### Permisos de la app OAuth 1.0a
</div>

Los permisos de la app describen el nivel de acceso para la autenticación aplicación-usuario de OAuth 1.0a. Los permisos de la app se configuran por aplicación en la configuración de tu [X App](/es/resources/fundamentals/developer-apps).

Hay tres niveles de permiso disponibles:

1. Solo lectura
2. Lectura y escritura
3. Lectura, escritura y acceso a Mensajes Directos

Existe un permiso adicional para solicitar la visibilidad de la dirección de correo electrónico de un usuario; puede combinarse con cualquiera de los tres niveles mencionados.

Si se cambia un nivel de permiso, cualquier token de usuario ya emitido para esa X App debe descartarse y los usuarios deben volver a autorizar la app para que el token herede los permisos actualizados.

Una buena práctica es solicitar solo el nivel mínimo de acceso a los datos de la cuenta de un usuario que requiera una aplicación o servicio.

<div id="read-only">
  ### Solo lectura
</div>

Este nivel de permiso permite el acceso de lectura a recursos de X, incluidos (por ejemplo) las publicaciones de un usuario, su cronología de inicio y la información de su perfil. No permite leer los Mensajes Directos de un usuario ni actualizar ningún elemento u objeto.

<div id="read-and-write">
  ### Lectura y escritura
</div>

Este nivel de permisos permite acceso de lectura y escritura a los recursos de X. Además de permitir la lectura, también permite publicar publicaciones, seguir a usuarios o actualizar elementos de la información del perfil de un usuario. También permite ocultar respuestas en nombre del usuario autenticado. Este nivel de permisos no permite ningún acceso a los Mensajes Directos (incluidos lectura, escritura o eliminación).

<div id="read-write-and-access-direct-messages">
  ### Leer, escribir y acceder a los Mensajes Directos
</div>

Este nivel de permiso incluye todo lo anterior y añade la capacidad de leer, escribir y eliminar Mensajes Directos en nombre de un usuario.

- [POST /2/dm\_conversations/:dm\_conversation\_id/messages](/es/x-api/direct-messages/send-a-new-message-to-a-dm-conversation)
- [POST /2/dm\_conversations/](/es/x-api/direct-messages/create-a-new-dm-conversation)
- [POST /2/dm\_conversations/with/:participant\_id/messages](/es/x-api/direct-messages/send-a-new-message-to-a-user)
- [GET /2/dm\_conversations/with/:user\_id/dm\_events](/es/x-api/direct-messages/get-dm-events-for-a-dm-conversation)
- [GET /2/dm\_conversations/:dm\_conversation\_id/dm\_events](/es/x-api/direct-messages/get-dm-events-for-a-dm-conversation-1)
- [GET /2/dm\_events](/es/x-api/direct-messages/get-recent-dm-events)

<div id="determining-permissions">
  ### Determinación de permisos
</div>

Todas las solicitudes autenticadas a la API incluyen un encabezado `x-access-level` en la respuesta HTTP. El valor de este encabezado indica el nivel de permisos en uso. Los valores posibles son read, read-write y read-write-directmessages.

<div id="callback-urls">
  ## URLs de callback
</div>

Los métodos de autenticación OAuth 1.0a User Context y OAuth 2.0 Authorization Code con PKCE permiten a los desarrolladores realizar solicitudes en nombre de distintos usuarios de X que han completado un flujo de inicio de sesión específico. Actualmente, hay dos flujos que puede usar para permitir que los usuarios autoricen su aplicación:

- [Flujo de código de autorización de OAuth 2.0 con PKCE](/es/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2)
- [Flujo OAuth 1.0a de 3 pasos](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) (y, por separado, el [flujo de Inicio de sesión con X](/es/resources/fundamentals/authentication#log-in-with-x))

A medida que los usuarios avanzan por estos flujos, necesitan una página web o ubicación a la que ser redirigidos después de iniciar sesión correctamente y otorgar autorización a la App del desarrollador. Esta página o ubicación posterior se denomina URL de callback.

Al configurar estos flujos para que sus potenciales usuarios los completen, los desarrolladores deben incluir una URL de callback en sus solicitudes a los endpoints de autenticación que componen los flujos mencionados. Por ejemplo, los desarrolladores que usan OAuth 1.0a User Context deben incluir el parámetro `callback_url` al realizar una solicitud al endpoint [GET oauth/request\_token](/es/resources/fundamentals/authentication#post-oauth-request-token). De manera similar, los desarrolladores que usan OAuth 2.0 Authorization Code con PKCE deben incluir el parámetro `redirect_uri` en su solicitud al endpoint GET oauth2/authorize.

Además de usar estos parámetros, el desarrollador debe asegurarse de que la URL de callback también se haya agregado a la lista de permitidos de URLs de callback de su [App](/es/resources/fundamentals/developer-apps), que se encuentra en la página de configuración de la App del [portal para desarrolladores](/es/resources/fundamentals/developer-portal).

Si se configura correctamente, los desarrolladores serán redirigidos a la URL de callback después de iniciar sesión en X como parte de estos flujos.

<div id="things-to-keep-in-mind">
  ### Cosas a tener en cuenta
</div>

Al configurar sus URLs de callback, hay algunas cosas que debe considerar:

**Codifique en HTTP sus parámetros de consulta**

Como está pasando la URL de callback como un parámetro de consulta mediante `callback_url` o `redirect_uri`, asegúrese de codificar la URL en HTTP.

**Límites de la URL de callback**

Existe un límite estricto de 10 URLs de callback en el panel de X Apps. Su URL de callback siempre debe coincidir exactamente entre la URL de callback incluida en la lista de permitidos que agregue en el panel de Apps y el parámetro que incluya en el flujo de autorización.

Si desea incluir datos específicos de la solicitud en la URL de callback, puede usar el parámetro `state` para almacenar datos que se incluirán después de que el usuario sea redirigido. Puede codificar los datos en el propio parámetro `state` o usar el parámetro como un ID de sesión para almacenar el estado en el servidor.

**No use localhost como URL de callback**
En lugar de usar localhost, use un host personalizado en local o http(s)://127.0.0.1

**URLs de protocolo personalizado**
Si desea aprovechar el deep linking en dispositivos móviles, puede utilizar URLs de protocolo personalizado con una parte de ruta y dominio, como twitter://callback/path. Sin embargo, existe una lista de protocolos no permitidos que deberá evitar. Puede revisar la lista de protocolos no permitidos a continuación.

<div id="error-example">
  ### Ejemplo de error
</div>

Si usas una URL de callback que no se haya agregado correctamente a la configuración de tu App en el portal de desarrolladores, recibirás el siguiente mensaje de error:

**OAuth 1.0a**

```json HTTP 403 - Forbidden

{
  "errors": [
    {
      "code":415,"message":"Callback URL not approved for this client application. Approved callback URLs can be adjusted in your application settings."
    }
  ]
}
```

O

```xml
<?xml version="1.0" encoding="UTF-8"?>
<hash>
<error>Callback URL not approved for this client application. Approved callback URLs can be adjusted in your application settings</error>
<request>/oauth/request_token</request>
</hash>
```

**OAuth 2.0**

```json HTTP 400

{
  "error": "invalid_request",
  "error_description": "Value passed for the redirect uri did not match the uri of the authorization code."
}
```

**Solución de problemas**

Si recibes un error, asegúrate de que la URL de callback que estás usando en tus solicitudes de autenticación esté codificada para HTTP y que coincida con una URL de callback que se haya agregado a la allowlist de la App cuyas claves y tokens estás utilizando en tu solicitud.
