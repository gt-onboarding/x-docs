<div id="introduction">
  ## Introducción
</div>

Con el lanzamiento de la versión v2 de la API de X, adoptamos un nuevo formato de respuesta de datos y un método de solicitud de distintos objetos y campos, al que llamamos simplemente formato de la API de X v2.

En la sección de diferencias generales, puede conocer algunos cambios relevantes para usuarios estándar y empresariales. Además, preparamos una guía específica para el [formato nativo de la v1.1 estándar](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview), el [formato Native Enriched empresarial](/es/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) y el [formato Activity Streams empresarial](/es/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#activity-object), que ayuda a mapear campos y explica qué campos y expansiones debe usar para solicitar los nuevos campos de la v2.

- [De formato nativo a API de X v2 (v1.1 estándar)](/es/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2)
- [De Native Enriched a API de X v2 (empresarial)](/es/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2)
- [De Activity Streams a API de X v2 (empresarial)](/es/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2)

También puede interesarle nuestra [herramienta visual de migración de formato de datos](/es/x-api/migrate/data-format-migration#visual-data-format-migration-tool), que le ayuda a ver rápidamente las diferencias entre el [formato de datos de la API de X v1.1](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) y el [formato de la API de X v2](/es/x-api/introduction).

<div id="general-differences">
  ### Diferencias generales
</div>

<div id="requesting-objects-and-fields">
  #### Solicitud de objetos y campos
</div>

Uno de los mayores cambios entre los endpoints anteriores a v2 y v2 es que la versión más reciente solo devuelve algunos campos de forma predeterminada, mientras que los endpoints estándar, premium y empresariales devuelven la mayoría de los campos por defecto. La nueva versión utiliza parámetros llamados [fields](/es/x-api/fundamentals/fields) y [expansions](/es/x-api/fundamentals/expansions) para solicitar de forma explícita datos adicionales más allá de los predeterminados, lo que significa que puede solicitar solo los datos que necesita sin tener que ingerir campos que no le interesan.

Cualquier campo que solicite y que esté relacionado con el objeto de datos principal se devolverá en ese objeto principal junto con los valores predeterminados. Sin embargo, si solicita objetos expandidos usando el parámetro expansions, los objetos secundarios se devolverán en un nuevo objeto includes. Puede hacer coincidir los objetos expandidos en el objeto includes con el objeto principal utilizando el campo id, que se devuelve en ambos.

Por ejemplo, si está usando el endpoint de v2 [Post lookup](/es/x-api/posts/lookup/introduction) e incluye el parámetro expansions=author\_id en su solicitud, recibirá el campo author\_id dentro del objeto Post principal, así como un objeto de usuario por Post en el objeto includes; cada uno incluirá el campo id predeterminado que puede usarse para hacer coincidir el objeto de usuario con el objeto Post. Aquí hay un ejemplo de cómo se ve:

```
{
  "data": [
    {
      "author_id": "2244994945",
      "id": "1397568983931392004",
      "text": "The Twitter Developer Platform. Ooh la la! https://t.co/iGTdPXBfOv https://t.co/Ze8z8EODdg"
    }
  ],
  "includes": {
    "users": [
      {
        "id": "2244994945",
        "name": "Twitter Dev",
        "username": "TwitterDev"
      }
    ]
  }
}
```

<div id="updated-json-design">
  #### Diseño JSON actualizado
</div>

Además de los cambios en cómo se solicitan ciertos campos, la API de X v2 también introduce nuevos diseños JSON para los objetos que devuelven las API, incluidos los objetos [Post](/es/x-api/fundamentals/data-dictionary#tweet) y [user](/es/x-api/fundamentals/data-dictionary#user).

- En el nivel raíz de JSON, los endpoints estándar devuelven objetos Post en un arreglo **statuses**, mientras que la API de X v2 devuelve un arreglo **data**.
- En lugar de referirse a “statuses” retweeted y quoted, el JSON de la API de X v2 se refiere a Tweets retweeted y quoted. Muchos campos heredados y en desuso, como **contributors** y **user.translator\_type**, se están eliminando.
- En lugar de usar tanto **favorites** (en el objeto Post) como **favourites** (en el objeto user), la API de X v2 usa el término like.
- X adopta la convención de que los campos sin valor en JSON (por ejemplo, null) no se incluyen en la carga útil. Los atributos de Post y user solo se incluyen si tienen valores distintos de null. 
   

<div id="new-v2-fields">
  #### Nuevos campos de v2
</div>

También incorporamos un nuevo conjunto de campos en el [objeto Publicación](/es/x-api/fundamentals/data-dictionary#tweet) que incluye lo siguiente:

- Un campo [conversation\_id](/es/x-api/fundamentals/conversation-id)
- Dos nuevos campos de [annotations](/es/x-api/fundamentals/post-annotations), incluidos context y entities
- Varios campos nuevos de [metrics](/es/x-api/fundamentals/metrics)
- Un nuevo campo reply\_setting, que indica quién puede responder a una publicación determinada

<div id="migrating-from-standard-v11s-data-format-to-v2">
  ### Migración del formato de datos estándar de v1.1 a v2
</div>

Si aún no lo has hecho, te recomendamos leer primero la introducción a la [migración de formatos de datos](/es/x-api/migrate/data-format-migration). También podría interesarte nuestra [herramienta visual de migración de formatos de datos](/es/x-api/migrate/data-format-migration#visual-data-format-migration-tool), que te ayudará a ver rápidamente las diferencias entre el [formato de datos de la API de X v1.1](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) y el [formato de la API de X v2](/es/x-api/fundamentals/data-dictionary).

El formato de datos estándar de v1.1, también conocido como formato nativo, es el formato principal que se entrega con los endpoints de [standard v1.1](https://developer.x.com/en/docs/twitter-api/v1).

Si utilizas el producto premium, consulta la [guía de formato nativo enriquecido](/es/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2). Los clientes empresariales podrían estar usando el formato nativo enriquecido o activity streams, según cómo estén configurados en la consola de Gnip.

<div id="migrating-from-native-enriched-data-format-to-v2">
  ### Migración del formato de datos Native Enriched a v2
</div>

El formato de datos Native Enriched es utilizado por nuestros productos de [enterprise](/es/x-api/enterprise-gnip-2.0/enterprise-gnip).

El formato de datos Native Enriched se ha actualizado para proporcionar metadatos de tuits editados. Para obtener más información sobre los metadatos de edición de tuits, consulta la página [Fundamentos de edición de tuits](/es/x-api/enterprise-gnip-2.0/fundamentals/edit-tweets).

Si utilizas los endpoints estándar de v1.1, consulta la [guía de migración de estándar v1.1 a v2](/es/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2). Si utilizas los productos enterprise con Activity Streams, también tenemos una guía de [Activity Streams a v2](/es/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2) para ti.

La API de X v2 introduce nuevos diseños JSON para los objetos [Tweet](/es/x-api/fundamentals/data-dictionary#tweet) y [user](/es/x-api/fundamentals/data-dictionary#user).

- En el nivel raíz de JSON, el formato Native Enriched devuelve objetos Tweet en un array results, mientras que la API de X v2 devuelve un array data.
- En lugar de usar tanto favorites (en el objeto Tweet) como favourites (en el objeto user), la API de X v2 usa el término like.
- X adopta la convención de que los valores JSON sin valor (por ejemplo, null) no se incluyen en la carga útil. Los atributos de Tweet y user solo se incluyen si tienen valores no nulos.
- Todos los campos id en v2 estarán en formato string

Además de los cambios realizados en el nuevo formato JSON, también introdujimos un nuevo conjunto de campos en el objeto Tweet, incluidos los siguientes:

- conversation\_id
- reply\_settings
- alt\_text en media
- Dos campos nuevos de [annotations](/es/x-api/fundamentals/post-annotations), incluidos context y entities
- Varios campos nuevos de [metrics](/es/x-api/fundamentals/metrics)
- Varios campos nuevos de [polls](/es/x-api/fundamentals/data-dictionary#poll)

Se están eliminando muchos campos heredados y obsoletos:

- contributors
- Ciertos campos de entities.media y extended\_entities.media
- filter\_level
- timestamp\_ms
- truncated

<div id="migrating-from-activity-streams-data-format-to-v2">
  ### Migración del formato de datos Activity Streams a v2
</div>

El formato de datos Activity Streams está disponible con nuestros productos [enterprise](/es/x-api/enterprise-gnip-2.0/enterprise-gnip).

El formato de datos Activity Streams se ha actualizado para proporcionar metadatos de tuits editados. Para obtener más información sobre los metadatos de edición de tuits, consulta la página [Fundamentos de edición de tuits](/es/x-api/enterprise-gnip-2.0/fundamentals/edit-tweets).

Si usas los endpoints estándar v1.1, consulta la [guía de estándar v1.1 a v2](/es/x-api/migrate/overview). Si usas los endpoints premium o el formato Native Enriched para enterprise, consulta la [guía de Native Enriched a v2](/es/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2).

La API de X v2 introduce nuevos diseños JSON para los objetos de [publicación](/es/x-api/fundamentals/data-dictionary#tweet) y [usuario](/es/x-api/fundamentals/data-dictionary#user).

- En el nivel raíz de JSON, el formato Activity Streams devuelve objetos de tuit en un arreglo results, mientras que la API de X v2 devuelve un arreglo data.
- En lugar de referirse a las “actividades” Retweeted y Quoted, el JSON de la API de X v2 se refiere a tuits Retweeted y Quoted.
- En lugar de usar tanto favorites (en el objeto de tuit) como favourites (en el objeto de usuario), la API de X v2 usa el término like.
- X adopta la convención de que los valores JSON sin valor (por ejemplo, null) no se escriben en la carga. Los atributos de tuit y usuario solo se incluyen si tienen valores no nulos.
- Todos los campos id en v2 estarán en formato de cadena.
   

Además de los cambios realizados en el nuevo formato JSON, también introdujimos un nuevo conjunto de campos en el objeto tuit, incluidos los siguientes:

- conversation\_id
- reply\_settings
- alt\_text en medios
- Dos campos nuevos de [anotaciones](/es/x-api/fundamentals/post-annotations), incluidos context y entities
- Varios campos nuevos de [métricas](/es/x-api/fundamentals/metrics)
- Varios campos nuevos de [encuestas](/es/x-api/fundamentals/data-dictionary#poll)
   

Muchos campos heredados y obsoletos se están eliminando o reemplazando:

- display\_text\_range
- generator
- gnip
- link
- objectType
- provider
- twitter\_entities.symbols reemplazado por data.entities.cashtags
- Ciertos campos de twitter\_extended\_entities.media y twitter\_entities.media
- twitter\_filter\_level
- twitterTimeZone
- verb

La herramienta visual de migración del formato de datos es una aplicación web que muestra los campos que se asignan del [formato de datos de la API de X v1.1](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) al [formato de la API de X v2](/es/x-api/fundamentals/data-dictionary) para un objeto de Tweet o de usuario determinado. Puede proporcionar un ID de Tweet o un ID de usuario a la aplicación para ver esta correspondencia.

Tenga en cuenta que deberá iniciar sesión con su cuenta de Twitter para usar la aplicación.
