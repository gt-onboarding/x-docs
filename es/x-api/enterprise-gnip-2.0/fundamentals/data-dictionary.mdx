<div id="introduction">
  ### Introducción
</div>

`Enterprise`

Las publicaciones son el componente atómico básico de todo en X. Todas las API de X que devuelven publicaciones proporcionan esos datos codificados en JavaScript Object Notation (JSON). JSON se basa en pares clave-valor, con atributos con nombre y valores asociados. Los objetos de publicación recuperados desde la API incluyen la “actualización de estado” de un usuario de X, y los Retweets, las respuestas y las publicaciones citadas también son objetos de publicación. Si una publicación está relacionada con otra, como un Retweet, una respuesta o una publicación citada, cada una se identificará o se incrustará en el objeto de la publicación. Incluso la publicación más simple en el formato de datos nativo de X tendrá objetos JSON anidados para representar otros atributos de una publicación, como el autor, los usuarios mencionados, la ubicación etiquetada, los hashtags, los símbolos de cashtag, el contenido multimedia o los enlaces URL. Al trabajar con datos de X, este es un concepto importante que comprender. El formato de los datos de la publicación que reciba de la API de X depende del tipo de publicación, de la API de X que esté utilizando y de la configuración de formato.

Los endpoints de Enterprise que devuelven objetos de publicación se han actualizado para proporcionar los metadatos necesarios para comprender el historial de ediciones de la publicación. Obtenga más información sobre estos metadatos en la página de [fundamentos “Editar publicaciones”](/es/x-api/fundamentals/edit-posts).

<div id="available-data-formats">
  ### Formatos de datos disponibles
</div>

> Tenga en cuenta: se recomienda encarecidamente usar el formato [Enriched Native](/es/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) para las API de datos empresariales.
>
> - El formato Enriched Native incluye todos los metadatos incorporados desde 2017, como los [metadatos de encuestas](/es/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata), y métricas adicionales como reply\_count y quote\_count.
>
> - El formato Activity Streams no se ha actualizado con nuevos metadatos ni enriquecimientos desde la [actualización del límite de caracteres](https://blog.x.com/official/en_us/topics/product/2017/Giving-you-more-characters-to-express-yourself.html) de 2017.

Las API de datos empresariales entregan información en dos formatos diferentes. El formato empresarial más cercano al formato nativo estándar v1.1 es Native Enriched. El formato empresarial heredado es Activity Streams, implementado originalmente por Gnip y utilizado como un formato normalizado en X y en otros proveedores de datos de redes sociales de la época. Aunque este formato sigue disponible, X solo ha invertido en nuevas funciones y mejoras en el formato nativo enriquecido desde 2017.

El formato nativo enriquecido es exactamente lo que su nombre indica: incluye objetos nativos de X, así como enriquecimientos adicionales disponibles para productos de datos empresariales, como metadatos de expansión de URL, geolocalización de perfil, metadatos de encuestas y métricas adicionales de interacción.

- [Enriquecimiento de URL expandidas y mejoradas](/es/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls)
- [Enriquecimiento de reglas de coincidencia](/es/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#matching-rules)
- [Enriquecimiento de metadatos de encuestas](/es/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata)
- [Enriquecimiento de geolocalización de perfil](/es/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo)

### Mejores prácticas de análisis sintáctico

- El JSON de X está codificado en UTF-8.
- Los analizadores deben tolerar con facilidad variaciones en el orden de los campos. Debe asumirse que el JSON de publicaciones se entrega como un mapa/diccionario de datos no ordenado.
- Los analizadores deben tolerar la incorporación de campos "nuevos".
- Los analizadores de JSON deben tolerar campos ausentes, ya que no todos los campos aparecen en todos los contextos.
- Por lo general, es seguro considerar un campo nulo, un conjunto vacío y la ausencia de un campo como equivalentes.

#### Objeto de publicación

Al utilizar productos de datos empresariales, notará que gran parte del diccionario de datos es similar al formato nativo de los datos de publicación, con algunos metadatos enriquecidos adicionales. El nivel base del formato nativo enriquecido utiliza muchos de los mismos nombres de objeto que el [formato de datos de la API de X v1.1](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview.html). El objeto de publicación tiene una larga lista de atributos de nivel raíz, incluidos atributos fundamentales como `id`, `created_at` y `text`. Los objetos de publicación también tendrán objetos anidados que incluyen `user`, `entities` y `extended_entities`. Los objetos de publicación también tendrán otros [objetos de publicación anidados](/es/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#nested-post-objects) como `retweeted_status`, `quoted_status` y `extended_tweet`. El formato nativo enriquecido además tendrá un objeto `matching_rules`.

<div id="x-data-dictionary">
  ##### Atributos adicionales de la Publicación
</div>

Las API de X que proporcionan Publicaciones (p. ej., el endpoint GET statuses/lookup) pueden incluir estos atributos adicionales de la Publicación:

| Attribute | Type | Description |

#### Objetos de Publicación anidados

En varios casos, un objeto de Publicación incluirá otros objetos anidados. Si trabaja con objetos anidados, esa carga útil JSON contendrá múltiples objetos de Publicación, y cada objeto de Publicación puede incluir sus propios objetos. El objeto de nivel raíz contendrá información sobre el tipo de acción realizada, es decir, si es un Retuit o una Cita de tuit, y también puede incluir un objeto que describa la Publicación “original” que se comparte. Las Publicaciones extendidas incluirán un objeto extendido anidado que supera los 140 caracteres, lo cual se utilizó para evitar cambios incompatibles cuando se realizó la actualización en 2017. A continuación se describe cada diccionario de objeto anidado.

**Retuits**

Los Retuits siempre contienen dos objetos de Publicación. La Publicación “original” que se retuitea se proporciona en un objeto "retweeted\_status". El objeto de nivel raíz encapsula el Retuit en sí, incluyendo un objeto de Usuario para la cuenta que realiza la acción de retuitear y la hora del Retuit. Retuitear es una acción para compartir una Publicación con sus seguidores, y no se puede agregar ningún contenido nuevo. Además, no se puede proporcionar una ubicación (nueva) con un Retuit. Aunque la Publicación “original” pueda estar georreferenciada, los objetos "geo" y "place" del Retuit siempre serán nulos.

Incluso antes de la introducción de las Publicaciones extendidas, el objeto "entities" de nivel raíz a veces se truncaba y quedaba incompleto debido a que se agregaba la cadena "RT @username " al mensaje de la Publicación que se retuiteaba. Tenga en cuenta que, si un Retuit se vuelve a retuitear, "retweet\_status" seguirá apuntando a la Publicación original, lo que significa que el Retuit intermedio no se incluye. Se observa un comportamiento similar al usar x.com para “mostrar” un Retuit. Si copia el ID de Publicación único asignado a la acción de retuitear, se muestra la Publicación original.

A continuación se muestra un ejemplo de estructura para un Retuit. Nuevamente, al analizar Retuits, es fundamental analizar el objeto "retweeted\_status" para obtener el mensaje de la Publicación completo (original) y los metadatos de entidades.

```json
{
	"tweet": {
		"text": "RT @author original message",
		"user": {
			"screen_name": "Retweeter"
		},
		"retweeted_status": {
			"text": "original message",
			"user": {
				"screen_name": "OriginalTweeter"
			},
			"place": {},
			"entities": {},
			"extended_entities": {}
		}
	},
	"entities": {},
	"extended_entities": {}
}
```

##### Tweets citados

Los Tweets citados son muy similares a los Retweets, salvo que incluyen un nuevo mensaje de Publicación. Estos mensajes pueden contener su propio conjunto de hashtags, enlaces y otros metadatos de “entidades”. Los Tweets citados también pueden incluir información de ubicación compartida por la persona que publica el Tweet citado, junto con medios como GIF, videos y fotos.

Los Tweets citados contienen al menos dos objetos de Publicación y, en algunos casos, tres. La Publicación que se cita, que a su vez puede ser un Tweet citado, se proporciona en un objeto "quoted\_status". El objeto de nivel raíz encapsula el propio Tweet citado, incluido un objeto de Usuario para la cuenta que realiza la acción de compartir y la hora del Tweet citado.

Ten en cuenta que ahora los Tweets citados pueden tener fotos, GIF o videos añadidos mediante la interfaz de usuario de ‘Post’. Cuando se incluyen enlaces a medios alojados externamente en el mensaje del Tweet citado, el "entities.urls" de nivel raíz los describe. Los medios adjuntos a los Tweets citados aparecerán en los metadatos "extended\_entities" de nivel raíz.

Cuando se lanzaron por primera vez los Tweets citados, se añadía un enlace acortado (URL de t.co) al mensaje de la Publicación “original” y se incluía en el campo "text" de nivel raíz. Además, los metadatos de esa URL de t.co se incluían en la matriz "entities.urls" de nivel raíz. En mayo de 2018, cambiamos esto para que la URL acortada de t.co al Tweet citado no se incluya en el campo "text" de nivel raíz. En segundo lugar, los metadatos del Tweet citado no se incluirán en los metadatos de "entities.urls". En su lugar, los metadatos de la URL del Tweet citado estarán en un nuevo objeto "quoted\_status\_permalink" en el nivel raíz (o de nivel superior), es decir, al mismo nivel que el objeto "quoted\_status".

<div id="quote-tweets">
  ##### Publicaciones extendidas
</div>

El JSON que describe las _Publicaciones extendidas_ se introdujo cuando se lanzaron las publicaciones de 280 caracteres en noviembre de 2017. El JSON de la publicación se amplió para encapsular estos mensajes más largos sin afectar a las miles de apps que analizan estos objetos fundamentales de X. Para mantener la compatibilidad total con versiones anteriores, se conservaron el campo original de 140 caracteres "text" y los objetos de entidad derivados de ese campo. En el caso de publicaciones de más de 140 caracteres, este campo "text" a nivel raíz pasaba a estar truncado y, por lo tanto, incompleto. Dado que los objetos "entities" a nivel raíz contienen arreglos de metadatos clave extraídos del mensaje "text", como hashtags y enlaces incluidos, estas colecciones quedarían incompletas. Por ejemplo, si un mensaje de publicación tuviera 200 caracteres, con un hashtag al final, el arreglo heredado "entities.hashtags" a nivel raíz no lo incluiría.

Se introdujo un nuevo campo "extended\_tweet" para contener los mensajes de publicación más largos y los metadatos completos de entidades. El objeto "extended\_tweet" proporciona el campo "full\_text", que contiene el mensaje completo de la publicación, sin truncar, cuando supera los 140 caracteres. El objeto "extended\_tweet" también incluye un objeto "entities" con arreglos completos de hashtags, enlaces, menciones, etc.

Las publicaciones extendidas se identifican con un booleano "truncated" a nivel raíz. Cuando sea true ("truncated": true), se deben analizar los campos de "extended\_tweet" en lugar de los campos a nivel raíz.

Observe en el ejemplo JSON a continuación que el campo "text" a nivel raíz está truncado y el arreglo "entities.hashtags" a nivel raíz está vacío, a pesar de que el mensaje de la publicación incluye tres hashtags. Dado que se trata de una publicación extendida, el campo "truncated" está establecido en true y el objeto "extended\_tweet" proporciona metadatos completos de la publicación en "full\_text" y "entities".

### Objetos geográficos enriquecidos nativos

Las publicaciones pueden asociarse a una ubicación, generando una publicación “etiquetada geográficamente”. Las ubicaciones de las publicaciones pueden asignarse mediante la interfaz de usuario de X o al publicar a través de la API. Las ubicaciones pueden ser un “punto” exacto o un X Place con una “caja delimitadora” que describe un área mayor, desde un recinto hasta una región completa.

Hay tres objetos JSON de “nivel raíz” que se utilizan para describir la ubicación asociada a una publicación: [place](/es/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#place-data-dictionary), [geo](/es/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-geo-objects) y [coordinates](/es/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#coordinates-object-data-dictionary).

Además, el formato nativo enriquecido incluye la [ubicación derivada](/es/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#derived-locations) del [profile geo enrichment](/es/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo) dentro del objeto de usuario.

El objeto `place` siempre está presente cuando una publicación está etiquetada geográficamente con un lugar. Los lugares son ubicaciones específicas, con nombre, y con coordenadas geográficas correspondientes. Cuando los usuarios deciden asignar una ubicación a su publicación, se les presenta una lista de X Places candidatos. Al usar la API para publicar, se puede adjuntar un X Place especificando un place\_id en la publicación. Las publicaciones asociadas con Places no necesariamente se originan en esa ubicación, sino que también pueden tratar _sobre_ esa ubicación.

Los objetos geo y `coordinates` solo están presentes (no nulos) cuando a la publicación se le asigna una ubicación exacta. Si se proporciona una ubicación exacta, el objeto `coordinates` entregará un arreglo \[long, lat] con las coordenadas geográficas, y se asignará un X Place que corresponda a esa ubicación.

<div id="x-entities">
  ## Introducción
</div>

Las entidades proporcionan metadatos e información contextual adicional sobre el contenido publicado en X. La sección `entities` ofrece arreglos de elementos comunes incluidos en las publicaciones: hashtags, menciones de usuarios, enlaces, símbolos bursátiles, encuestas de X y medios adjuntos. Estos arreglos son prácticos para los desarrolladores al ingerir publicaciones, ya que X básicamente ha preprocesado, o preanalizado, el cuerpo del texto. En lugar de tener que buscar explícitamente estas entidades en el cuerpo de la publicación, su analizador puede ir directamente a esta sección JSON y encontrarlas allí.

Además de facilitar el procesamiento, la sección `entities` también aporta metadatos útiles de “valor añadido”. Por ejemplo, si utiliza el [enriquecimiento Enhanced URLs](/es/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls), los metadatos de URL incluyen direcciones completamente expandidas, así como títulos y descripciones del sitio web asociados. Otro ejemplo: cuando hay menciones de usuarios, los metadatos de entidades incluyen el ID numérico del usuario, lo cual es útil al realizar solicitudes a muchas API de X.

Cada carga útil JSON de una publicación incluye una sección `entities` con el conjunto mínimo de atributos `hashtags`, `urls`, `user_mentions` y `symbols`, incluso si ninguna de esas entidades forma parte del mensaje de la publicación. Por ejemplo, si examina el JSON de una publicación cuyo cuerpo es “Hello World!” y sin medios adjuntos, el JSON de la publicación incluirá el siguiente contenido con arreglos de entidades que contienen cero elementos:

```
"entities": {
        "hashtags": [
        ],
        "urls": [
        ],
        "user_mentions": [
        ],
        "symbols": [
        ]
      }
```

Notas:

- las entidades de media y de polls solo aparecerán cuando ese tipo de contenido forme parte de la publicación.
- si trabaja con medios nativos (fotos, videos o GIF), el [objeto Extended Entities](/es/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#x-extended-entities) es la opción recomendada.

### Objeto entities

Las secciones `entities` y `extended_entities` están compuestas por arreglos de objetos de entidad. A continuación, encontrará descripciones de cada uno de estos objetos de entidad, incluidos diccionarios de datos que describen los nombres de los atributos del objeto, sus tipos y una breve descripción. También indicaremos qué operadores de PowerTrack coinciden con estos atributos e incluiremos algunos ejemplos de cargas JSON.

Una colección de entidades comunes que se encuentran en las publicaciones, incluidas las etiquetas (hashtags), los enlaces y las menciones de usuarios. Este objeto `entities` sí incluye un atributo `media`, pero su implementación en la sección `entities` solo es completamente precisa para publicaciones con una sola foto. Para todas las publicaciones con más de una foto, un video o un GIF animado, consulte la sección `extended_entities`.

### Objeto de hashtag  

La sección `entities` contendrá un arreglo `hashtags` que incluirá un objeto por cada hashtag presente en el cuerpo de la Publicación, y un arreglo vacío si no hay hashtags.

El operador `#` de PowerTrack se usa para hacer coincidencias en el atributo `text`. El operador `has:hashtags` hará coincidencia si hay al menos un elemento en el arreglo.

|     |     |     |
| :--- | :--- | :--- |
| Campo | Tipo | Descripción |
| indices | Array of Int | Un arreglo de enteros que indica los desplazamientos dentro del texto de la Publicación donde comienza y termina el hashtag. El primer entero representa la ubicación del carácter # en la cadena de texto de la Publicación. El segundo entero representa la ubicación del primer carácter después del hashtag. \+ El `type` de los medios siempre indicará “photo”, incluso cuando se adjunte un video o un GIF a la Publicación.
\+ Aunque se pueden adjuntar hasta cuatro fotos, solo la primera aparecerá listada en la sección `entities`.

### Formato de URL de medios de foto

Los medios de foto en X pueden cargarse en diferentes tamaños. Es mejor cargar la imagen más pequeña que sea lo suficientemente grande para ajustarse a una determinada ventana de visualización. Para cargar distintos tamaños, el [Size Object](#size) y [media\_url](#media) (o [media\_url\_https](#media)) deben combinarse en un formato específico. Usaremos el objeto de ejemplo de la [media entity](#entitiesobject) ya proporcionado para construir una URL de medio de foto.

Los `media_url` o `media_url_https` pueden cargarse por sí solos, lo que hará que se cargue por defecto la variante mediana. Sin embargo, es preferible proporcionar una URL de medios de foto completamente formateada siempre que sea posible.

Hay tres partes de una URL de medios de foto:

|     |     |
| :--- | :--- |
| Base URL | La URL base es la URL de medios sin la extensión de archivo.<br /><br />Por ejemplo:  <br /><br />"media\_url\_https": "https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg",<br /><br />La URL base sería:  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq_ |
| Format | El formato es el tipo de imagen en el que está codificada la foto. Los formatos posibles son _jpg_ o _png_, que se indican como la extensión de la URL de medios.<br /><br />Por ejemplo:  <br /><br />"media\_url\_https": "https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg",<br /><br />El formato sería: _jpg_ |
| Name | El nombre es el campo del tamaño a cargar.<br /><br />Por ejemplo:<br /><br />  <br /> "sizes":   <br />   "thumb":   <br />     "h": 150,  <br />     "resize": "crop",  <br />     "w": 150  <br />   ,  <br />   "large": <br />     "h": 1366,  <br />     "resize": "fit",  <br />     "w": 2048  <br />   },  <br />   "medium": <br />     "h": 800,  <br />     "resize": "fit",  <br />     "w": 1200  <br />   },  <br />   "small": <br />     "h": 454,  <br />     "resize": "fit",  <br />     "w": 680  <br />   }  <br /> }  <br />}<br /><br />El nombre al cargar la foto en tamaño grande sería: _large_ |

Tomamos estas tres partes (URL base, formato y nombre) y las combinamos para formar la URL de medios de foto que se va a cargar. Hay 2 formatos para cargar imágenes de esta manera, _legacy_ y _modern_. Todas las cargas de imágenes deben dejar de usar el formato _legacy_ y utilizar el formato _modern_. Usar el formato _modern_ dará como resultado una mejor tasa de aciertos en la CDN para el solicitante, mejorando así la latencia de carga al ser menos probable que deba generarse y cargarse el medio desde el centro de datos.

|     |     |
| :--- | :--- |
| Legacy format | El formato heredado está en desuso. Todas las cargas de medios de fotos deben migrar al formato moderno.<br /><br />_\<base\_url>.\<format>:\<name>_  <br /><br />Por ejemplo:  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg:large  <br />_ |
| Modern format | El formato moderno para cargar fotos se estableció en X en 2015 y es el de facto desde 2017. Todas las cargas de medios de fotos deben migrar a este formato.<br /><br />_\<base\_url>?format=\<format>\&name=\<name>_  <br /><br />Por ejemplo:  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq?format=jpg\&name=large  <br />_<br /><br />Nota: los elementos de la cadena de consulta de la URL de medios de fotos están en orden alfabético. Si la carga de medios añadiera elementos de consulta adicionales, habría que mantener el orden alfabético. Por ejemplo, si existiera el elemento de consulta hipotético llamado _preferred\_format_, iría después de _format_ y _name_ en la cadena de consulta. |

## Detalles de Retweet y Tweet citado

Desde la perspectiva de la API de X, los Retweets y los Tweets citados son tipos especiales de Publicaciones que contienen la Publicación original como un objeto incrustado. Por lo tanto, los objetos de Retweet y de Tweet citado son padres de una Publicación hija "original" (y, por ende, duplican el tamaño). Los Retweets tienen un objeto de nivel superior "retweeted\_status", y los Tweets citados tienen un objeto "quoted\_status". Por coherencia, estos objetos de nivel superior de Retweet y de Tweet citado también tienen una propiedad text y entities asociadas. Sin embargo, las entities en el nivel superior pueden diferir de las entities proporcionadas por las entities "originales" incrustadas. En el caso de los Retweets, se antepone texto nuevo al cuerpo de la Publicación original. Para las Publicaciones citadas, se agrega texto nuevo al final del cuerpo de la Publicación.

En general, la mejor práctica es recuperar el texto, las entities, el autor original y la fecha de la Publicación original en retweeted\_status siempre que exista. Una excepción es obtener las entities de X que forman parte del texto adicional de la cita. Consulte a continuación para más detalles y consejos.

### Retweets

Un detalle importante sobre los Retweets es que no se pueden añadir _entidades_ adicionales de X a la Publicación. Los usuarios no pueden añadir hashtags, URL u otros elementos cuando hacen Retweet. Sin embargo, el atributo de texto (de nivel superior) del Retweet se compone del texto de la Publicación original con “RT @username: ” antepuesto.  

En algunos casos, especialmente con cuentas con nombres de usuario largos, la combinación de estos nuevos caracteres y el cuerpo de la Publicación original puede superar fácilmente el límite original de 140 caracteres del texto de la Publicación. Para preservar la compatibilidad con la visualización y el almacenamiento basados en 140 caracteres, el cuerpo de nivel superior trunca el final del cuerpo de la Publicación y añade puntos suspensivos (“…”). En consecuencia, algunas entidades de nivel superior ubicadas al final de la Publicación original podrían ser incorrectas o faltar, por ejemplo, en el caso de un hashtag o una URL truncados.

Esta Publicación,  https://x.com/FloodSocial/status/907974220298125312, tiene el siguiente texto de la Publicación:

</div>

Los Tweets citados se introdujeron en 2016 y se diferencian de los Retweets en que, cuando “citas” una Publicación, agregas contenido nuevo “encima” de una Publicación compartida. #### Introducción

Si una Publicación contiene contenido multimedia nativo (compartido mediante la interfaz de usuario de la Publicación, en lugar de a través de un enlace externo), también habrá una sección extended\_entities. Para cualquier medio nativo (foto, video o GIF), extended\_entities es la fuente de metadatos preferida por varias razones. Actualmente, se pueden adjuntar hasta cuatro fotos a una Publicación. Los metadatos entities solo contendrán la primera foto (hasta 2014 solo se podía incluir una), mientras que la sección extended\_entities incluirá todas las fotos adjuntas. Con medios nativos, otra limitación de los metadatos entities.media es que el tipo de medio siempre indicará ‘photo’, incluso cuando el contenido adjunto sea un video o un GIF animado. El tipo real de medio se especifica en el atributo extended\_entities.media\[].type y puede ser _photo_, _video_ o _animated\_gif_. Por estas razones, si trabaja con medios nativos, los metadatos extended\_entities son la opción recomendada.

Todas las Publicaciones con fotos, videos y GIF animados adjuntos incluirán un objeto JSON `extended_entities`. El objeto `extended_entities` contiene un único arreglo `media` de objetos `media` (consulte la sección `entities` para su diccionario de datos). No se incluyen otros tipos de entidades, como hashtags y enlaces, en la sección `extended_entities`. El objeto `media` en la sección `extended_entities` es idéntico en estructura al incluido en la sección `entities`.

Las Publicaciones solo pueden tener un tipo de medio adjunto. Para fotos, se pueden adjuntar hasta cuatro. Para videos y GIF, se puede adjuntar uno. Dado que los metadatos `type` del medio en la sección `extended_entities` indican correctamente el tipo (‘photo’, ‘video’ o ‘animated\_gif’) y admiten hasta cuatro fotos, es la fuente de metadatos preferida para medios nativos.

```
{
      "extended_entities": {
        "media": [

        ]
      }
    }
```

Cuando un anunciante decide limitar la reproducción de video únicamente a plataformas propiedad y operación de X, el objeto `video_info` se sustituirá por un objeto `additional_media_info`.

El objeto `additional_media_info` incluirá información adicional del medio proporcionada por el editor, como `title`, `description` y la marca `embeddable`. El contenido de video solo está disponible para los clientes oficiales de X cuando `embeddable=false`. En este caso, todas las URL de video incluidas en la carga serán de X, para que el usuario pueda abrir el video en una propiedad de X al hacer clic en el enlace.

## Objetos de datos de Activity Streams para empresas

<Note>
  ¿Te interesa conocer mejor cómo se mapea el formato de datos de Activity Streams al formato de la API de X v2?
</Note>

Consulta nuestra guía comparativa: [Activity Streams comparado con la API de X v2](/es/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2)

<Note>
  Ten en cuenta: se recomienda encarecidamente usar el formato [Enriched Native](/es/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) para las API de datos empresariales. 

  - El formato Enriched Native incluye todos los metadatos nuevos desde 2017, como los [metadatos de encuestas](/es/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata), y métricas adicionales como reply\_count y quote\_count.

  - El formato Activity Streams no se ha actualizado con nuevos metadatos ni enriquecimientos desde la [actualización de caracteres](https://blog.x.com/official/en_us/topics/product/2017/Giving-you-more-characters-to-express-yourself.html) de 2017.
</Note>

### Objeto de actividad

Activity Streams es una traducción del esquema de objetos del [formato de datos original](https://developer.x.comhttps://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) de X, creada por Gnip para “normalizar el formato” de los datos de publicaciones y otros datos de redes sociales utilizando el [Activity Base Schema de terceros descrito aquí](https://activitystrea.ms/head/activity-schema.html). Las publicaciones se normalizan al esquema de Activity Streams e incluyen los tipos de objeto note, person, place y service como objetos anidados. Las publicaciones pueden tener otros objetos de actividad de la publicación anidados para retweets u otros, incluidos twitter\_quoted\_status y long\_object.

El tipo de objeto de nivel base “activity” es similar al [objeto de nivel base de la publicación](/es/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) del formato enriquecido nativo. Ejemplos de cargas en formato Activity Streams pueden encontrarse [aquí](/es/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#activity-streams-payload-examples).

<div id="deprecated-attributes">
  #### Objetos de actividad de publicaciones anidadas
</div>

En varios casos, un objeto de Publicación incluirá otras Publicaciones anidadas. Si trabaja con objetos anidados, esa carga útil JSON contendrá múltiples objetos y cada objeto de Publicación puede contener sus propios objetos. El objeto de nivel raíz contendrá información sobre el tipo de acción realizada, es decir, si es un Retuit o una cita de tuit, y también puede contener un objeto que describe la Publicación «original» que se está compartiendo. Las Publicaciones extendidas incluirán un objeto extendido anidado que supera los 140 caracteres, lo cual se utilizó para evitar cambios disruptivos cuando se hizo la actualización en 2017. Cada diccionario de objeto anidado se describe a continuación.

### Objeto de ubicación

Los objetos de ubicación pueden existir dentro del conjunto de objetos actor definido a nivel de la cuenta de X o dentro del objeto profileLocations del [objeto gnip](/es/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#gnip-object). Los objetos de ubicación son similares a [Geo](/es/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-geo-objects) en el formato enriquecido nativo.

### Introducción\*\*

En esencia, X es una red de comunicación pública, en tiempo real y global. Desde 2006, la evolución de X ha estado impulsada tanto por los patrones y las convenciones de uso de los usuarios como por nuevas funcionalidades y mejoras de producto. Si utiliza datos de X para investigación histórica, comprender la cronología de esta evolución es importante para identificar Publicaciones de interés en el archivo de datos.

X se lanzó como una sencilla app móvil de SMS y ha crecido hasta convertirse en una plataforma de comunicación integral. Una plataforma con un conjunto completo de API. Las API siempre han sido un pilar de la red de X. La [primera API se lanzó poco después del inicio de X](https://blog.x.com/2006/introducing-the-twitter-api). Cuando se introdujo por primera vez el etiquetado geográfico de Publicaciones en 2009, se puso a disposición a través de una [API de Geo](https://blog.x.com/2009/think-globally-tweet-locally) (y más tarde la capacidad de “geolocalizar” una Publicación se integró en la interfaz de usuario de X.com). Hoy en día, las API de X impulsan la red de comunicación bidireccional que se ha convertido en una fuente de noticias de última hora y de intercambio de información. Las oportunidades para desarrollar sobre este canal de comunicación global y en tiempo real son infinitas.

X pone a disposición dos API históricas que proporcionan acceso a todas las Publicaciones disponibles públicamente: Historical PowerTrack y la [Full-Archive Search API](https://developer.x.com/en/docs/x-api/v1/tweets/search/overview). Ambas API proporcionan un conjunto de operadores utilizados para consultar y recopilar Publicaciones de interés. Estos operadores hacen coincidir una variedad de atributos asociados a cada Publicación, cientos de atributos como el contenido de texto de la Publicación, el nombre de la cuenta del autor y los enlaces compartidos en la Publicación. Las Publicaciones y sus atributos se codifican en JSON, un formato de intercambio de datos común basado en texto. Así que, a medida que se introducían nuevas funciones, aparecían nuevos atributos JSON y, por lo general, se incorporaban nuevos operadores de API para hacer coincidir esos atributos. Si su caso de uso incluye la necesidad de escuchar lo que el mundo ha dicho en X, cuanto mejor comprenda cuándo los operadores empezaron a contar con metadatos JSON con los que hacer coincidencia, más eficaces podrán ser sus filtros históricos de PowerTrack.

A continuación, presentaremos algunos conceptos clave que sientan las bases para comprender cómo las actualizaciones en los metadatos de las Publicaciones afectan a la identificación de su señal de datos de interés.

### Conceptos clave\*\*

#### De convenciones de usuarios a objetos _de primera clase_ en X

Los usuarios de X introdujeron orgánicamente patrones de comunicación nuevos y ahora fundamentales en la red de X. Un ejemplo seminal es el hashtag, hoy casi universal en todas las redes sociales. Los hashtags se introdujeron como una forma de organizar conversaciones y temas. En una red con cientos de millones de mensajes al día, las herramientas para encontrar publicaciones de interés son clave, y los hashtags se han convertido en un método fundamental. Poco después de que creciera su uso, los hashtags recibieron estatus y soporte oficial por parte de X. Al convertirse en un objeto _“de primera clase”_, esto implicó varias cosas. Implicó que los hashtags pasaran a ser clicables y buscables en la interfaz de usuario de X.com. También significó que los hashtags pasaran a formar parte de la familia de _entidades_ de X, junto con las @menciones, el contenido multimedia adjunto, los símbolos bursátiles y los enlaces compartidos. Estas entidades están convenientemente codificadas en un arreglo JSON preprocesado, lo que facilita que los desarrolladores las procesen, examinen y almacenen.

Los retuits son otro ejemplo de convenciones impulsadas por los usuarios que se convierten en objetos oficiales. El retuit surgió como una forma de “reenviar” contenido a otros. Comenzó como un proceso manual de copiar y pegar una publicación y anteponerle el patrón “RT @”. Este proceso finalmente se automatizó mediante un nuevo botón de Retuit, con nuevos metadatos JSON. Nació el retuit “oficial”. Otros ejemplos incluyen las “menciones”, el uso compartido de contenido multimedia y enlaces web, y compartir una ubicación con tu publicación. Cada uno de estos patrones de uso dio lugar a nuevas funciones en la interfaz de usuario de [x.com](https://x.com/), nuevo JSON de soporte y, por lo tanto, nuevas formas de hacer coincidencia en publicaciones. Todos estos atributos fundamentales de la publicación han dado lugar a operadores de PowerTrack utilizados para hacer coincidencia sobre ellos.

<div id="from-user-conventions-to-x-first-class-objects">
  #### Metadatos de publicaciones, mutabilidad, actualizaciones y vigencia
</div>

Si bien los mensajes de Publicación pueden tener hasta un número fijo de caracteres, la descripción JSON de una Publicación consta de más de 100 atributos. Atributos como quién publicó, en qué momento, si es una Publicación original o un Retuit, y un arreglo de objetos de primera clase como hashtags, menciones y enlaces compartidos. Para la cuenta que publicó, hay un objeto User (o Actor) con una variedad de atributos que proporcionan el Perfil del usuario y otros metadatos de la cuenta. Los perfiles incluyen una breve descripción biográfica, una ubicación (texto libre), el idioma preferido y un enlace opcional a un sitio web.

Algunos metadatos de la cuenta nunca cambian (p. ej., el ID numérico de usuario y la fecha de creación), otros cambian lentamente con el tiempo, mientras que otros atributos cambian con más frecuencia. Las personas cambian de trabajo y se mudan. Las empresas actualizan su información. Al recopilar Publicaciones históricas, es importante entender que algunos metadatos reflejan _cómo eran en el momento de la Publicación_, mientras que otros reflejan _cómo son cuando se envía la consulta_.

Con todas las API históricas, la descripción del perfil del usuario, el nombre para mostrar y los atributos “home” del perfil se actualizan a los valores vigentes en el momento de la consulta.

<div id="post-metadata-mutability-updates-and-currency">
  #### Contenido multimedia “nativo”
</div>

X.com y las apps móviles de X permiten agregar fotos y videos a una Publicación al hacer clic en un botón y explorar tus galerías de fotos. Ahora que estas funciones están integradas como acciones de primer nivel, los videos y fotos compartidos de esta manera se denominan contenido multimedia “nativo”.

Muchos operadores de consulta funcionan con estos recursos “nativos”, incluidos `has:videos`, `has:images` y `has:media`. Estos solo coinciden con contenido multimedia compartido mediante funciones de X. Para coincidir con otro contenido multimedia alojado fuera de la plataforma de X, deberás usar operadores que coincidan con metadatos de URL.

Así que, antes de profundizar en los detalles de los productos Historical PowerTrack y Búsqueda de archivo completo, hagamos un recorrido por cómo X, como producto y plataforma, ha evolucionado con el tiempo.

**Cronología de X**

A continuación encontrarás una _cronología_ seleccionada de X. La mayoría de estas actualizaciones de X afectaron de manera fundamental el comportamiento de los usuarios, los contenidos JSON de las Publicaciones, los operadores de consulta o los tres. Considerando a X como una plataforma de API, los siguientes eventos afectaron de algún modo las cargas útiles JSON usadas para codificar las Publicaciones. A su vez, esos detalles JSON afectan cómo las API históricas de X hacen coincidir dichos datos.

Ten en cuenta que esta lista cronológica es generalmente precisa, pero no exhaustiva.

<div id="native-media">
  #### 2006
</div>

- Octubre
  - @replies se convierten en una convención.
  - Los $cashtags aparecen por primera vez, pero su uso para mencionar símbolos bursátiles no se vuelve común hasta principios de 2009. Los $cashtags pasaron a ser enlaces clicables/buscables en junio de 2012.
- Noviembre - Se introducen los Favoritos.

<div id="2006">
  #### 2007
</div>

- Enero: las @respuestas pasan a ser un objeto de primera clase con un botón de responder en la interfaz y metadatos `in_reply_to`.
- Abril: los retuits se convierten en una convención.
- Agosto: los hashtags (#) surgen como herramienta principal para buscar y organizar publicaciones.

<div id="2007">
  #### 2009
</div>

- Febrero: los $cashtags se convierten en una convención común para hablar de símbolos bursátiles.
- Mayo: se introduce la versión beta de los retuits con “Via @” antepuesto al cuerpo de la publicación.
- Junio: se introducen las cuentas verificadas.
- Agosto: los retuits pasan a ser un objeto de primera clase con el patrón “RT @” y nuevos metadatos `retweet_status`.
- Octubre: se lanza la función de listas.
- Noviembre: [Se lanza la API de geolocalización de publicaciones](https://blog.x.com/2009/think-globally-tweet-locally), que proporciona el primer método para que los usuarios compartan su ubicación a través de aplicaciones de terceros.

<div id="2009">
  #### 2010
</div>

**Consejos de filtrado**

Conocer la cronología de X sobre cuándo y cómo se añadieron nuevas funciones puede ayudarte a crear consultas más efectivas. Aquí, una consulta significa un _filtro_ o _regla_ que las API históricas de X aplican al archivo de Publicaciones, utilizando Operadores de PowerTrack para hacer coincidencias con el JSON de la Publicación. Un ejemplo es el operador `lang:`, que se usa para hacer coincidir Publicaciones en un idioma especificado. X proporciona un servicio de clasificación de idiomas (compatible con más de 50 idiomas) y las API de X incluyen estos metadatos en el JSON que se genera para cada Publicación. Así, si una Publicación está escrita en español, el atributo JSON “lang” se establece en “es”. Por lo tanto, si creas un filtro con la cláusula `lang:es`, solo coincidirá con mensajes de Publicaciones clasificados como español.

La información de la cronología también puede ayudar a interpretar mejor los datos de la Publicación recibidos. Supongamos que investigas la difusión de contenido sobre los Juegos Olímpicos de Verano de 2008 y 2012. Si aplicas únicamente el operador `is:retweet` para hacer coincidir Retweets, no habrá datos en 2008. Sin embargo, para 2012 probablemente haya millones de Retweets. A partir de esto, podrías concluir erróneamente que en 2008 los Retweets no eran una práctica de los usuarios o que simplemente nadie hizo Retweet sobre esos Juegos Olímpicos. Dado que los Retweets pasaron a ser un objeto de primera clase en 2009, debes añadir una cláusula de regla `”RT @”` para ayudar a identificarlos en 2008.

Tanto los Retweets como la clasificación del idioma de la Publicación son ejemplos de atributos de Publicación con una larga trayectoria y muchos detalles del producto. A continuación, trataremos más detalles de estas y otras clases de atributos importantes para hacer coincidencias y comprender los Datos de X.

<div id="2017">
  #### Reconocer falsos negativos
</div>

Al escribir filtros, es importante tener en cuenta que todos los Operadores que actúan sobre metadatos tienen una fecha de “nacimiento”. Si crea un filtro con un Operador que depende de metadatos introducidos después de que se publicó la publicación, obtendrá un falso negativo. Por ejemplo, supongamos que le interesan todas las publicaciones que mencionan ‘snow’ y comparten un video. Si crea una regla con el Operador `has:videos`, que coincide con publicaciones con videos nativos, esa cláusula no coincidirá con ninguna publicación anterior a 2015.

Sin embargo, compartir videos era común en X mucho antes de 2015. Antes de esa fecha, los usuarios compartían enlaces a videos alojados en otros sitios, pero en 2015 X incorporó nuevas funciones de “compartir video” directamente en la plataforma. Para encontrar esas publicaciones anteriores de interés, incluiría una cláusula de regla como `url:"youtube.com"`.

Tenga en cuenta que, con las APIs de Búsqueda, hay casos en los que se “completaron retrospectivamente” metadatos a medida que se reconstruía el índice. Un buen ejemplo son los $cashtags, que se popularizaron para hablar de símbolos bursátiles en 2009. Después de que se introdujo el operador $cashtag en 2015, se reconstruyó el índice de Búsqueda y, en el proceso, se extrajo la entidad de símbolo de todos los cuerpos de las publicaciones, incluyendo las de principios de 2006 cuando `$` se usaba principalmente como jerga; “I hope it $now$ $oon!”.

<div id="recognizing-false-negatives">
  #### Identificar y filtrar atributos de publicación importantes para su caso de uso
</div>

Algunos metadatos, como los ID numéricos de cuentas de X, existen desde el primer día (y son un ejemplo de metadatos de cuenta que nunca cambian). Otros metadatos no se introdujeron hasta bastante después del inicio de X en 2006. Entre los nuevos metadatos introducidos se incluyen los relativos a Retweets, ubicaciones de publicaciones, títulos y descripciones de URL, y contenido multimedia “nativo”. A continuación se muestran algunos de los tipos más comunes de atributos de publicación que se han visto afectados de manera fundamental por estas actualizaciones de la plataforma de X.

El comportamiento de filtrado/empate de estos atributos depende, en la mayoría de los casos, de qué API histórica de publicaciones se utilice. Para ayudar a determinar qué producto se ajusta mejor a su investigación y caso de uso, los detalles de los atributos que se proporcionan a continuación incluyen información general sobre el producto.

<div id="identifying-and-filtering-on-post-attributes-important-to-your-use-case">
  #### Perfiles de X
</div>

Dado que, en esencia, X es un canal global de comunicación en tiempo real, la investigación con datos de Publicaciones suele enfocarse en quién se comunica. A menudo es útil saber de dónde es un usuario de X. Con frecuencia, identificar que la biografía de una cuenta incluye intereses y aficiones puede llevarte a Publicaciones relevantes. Es muy común querer captar Publicaciones de cuentas de interés. Los atributos del perfil son clave para todos estos casos de uso.

Cada cuenta en X tiene un perfil que incluye metadatos como el @handle de X, el nombre para mostrar, una breve biografía, la ubicación (texto libre introducido por el usuario), el número de seguidores y muchos otros. Algunos atributos nunca cambian, como el ID numérico de usuario y la fecha de creación de la cuenta. Otros suelen cambiar día a día, semana a semana o mes a mes, como el número de Publicaciones realizadas y el número de cuentas seguidas y de seguidores. Otros atributos de la cuenta también pueden cambiar en cualquier momento, pero tienden a cambiar con menos frecuencia: nombre para mostrar, ubicación y biografía.

La carga útil JSON de cada Publicación incluye metadatos de _perfil de cuenta_ del autor de la Publicación. Si es un Retweet, también incluye metadatos de perfil de la cuenta que publicó la Publicación original.

La mutabilidad de los metadatos de perfil de una Publicación depende completamente del producto histórico utilizado. Las API de Búsqueda devuelven Publicaciones históricas con la configuración del perfil _tal como está en el momento de la recuperación_. Para Historical PowerTrack, el perfil es _tal como era en el momento en que se publicó la Publicación_, excepto para datos anteriores a 2011. Para Publicaciones anteriores a 2011, los metadatos del perfil reflejan el perfil tal como era en septiembre de 2011.

<div id="x-profiles">
  #### Publicación original y Retuits
</div>

Los retuits son otro ejemplo de convenciones impulsadas por los usuarios que se convierten en objetos oficiales. Retuitear surgió como una forma de “reenviar” contenido a otros. Empezó como un proceso manual de copiar y pegar una Publicación y anteponerle el patrón “RT @”. Este proceso finalmente se automatizó mediante un nuevo botón de Retuit, con nuevos metadatos JSON. Nació el Retuit “oficial” y la acción de retuitear se convirtió en un evento de Publicación de primera clase. Junto con el nuevo botón de Retuit, se introdujeron nuevos metadatos como la carga útil completa de la Publicación original.

Determinar si una Publicación es original o compartida es un criterio de filtrado habitual. En algunos casos, solo se necesita contenido original. En otros, la interacción con la Publicación es prioritaria, por lo que los retuits son clave. El operador de PowerTrack `is:retweet` permite incluir o excluir retuits. Si se extraen datos anteriores a agosto de 2009, se necesitan dos estrategias para detectar (o no) retuits. Antes de agosto de 2009, es necesario revisar el propio mensaje de la Publicación, usando coincidencia de frase exacta, para encontrar coincidencias con el patrón “RT @ ”. Para periodos posteriores a agosto de 2009, el operador `is:retweet` está disponible.

<div id="original-post-and-retweets">
  #### Clasificaciones de idioma de las publicaciones
</div>

El idioma en que está escrita una publicación suele ser de interés. El idioma de la publicación puede ayudar a inferir su ubicación y, a menudo, solo se requiere un idioma específico para el análisis o la visualización. (Los perfiles de X también tienen una configuración de idioma preferido).

Para filtrar por la clasificación de idioma de una publicación, los productos históricos de X ([Search API](https://developer.x.com/en/docs/x-api/v1/tweets/search/overview/enterprise) y Historical PowerTrack) difieren notablemente. Cuando se creó el archivo de búsqueda, todas las publicaciones se completaron con la clasificación de idioma de X. Por lo tanto, el operador `lang:` está disponible para todo el archivo de publicaciones. Con Historical PowerTrack, los metadatos de clasificación de idioma de X están disponibles en el archivo a partir del 26 de marzo de 2013. 

<div id="post-language-classifications">
  #### Georreferenciación de publicaciones
</div>

Poder indicar dónde se publicó una publicación (es decir, georreferenciarla) es importante para muchos casos de uso. Existen tres métodos principales para georreferenciar publicaciones:

- Referencias geográficas en el contenido de una publicación
- Publicaciones etiquetadas con ubicación por el usuario
- Ubicación «principal» del perfil de la cuenta establecida por el usuario

##### Referencias geográficas en un mensaje de Publicación

La coincidencia basada en referencias geográficas dentro del mensaje de la Publicación, aunque a menudo es el método más desafiante porque depende del conocimiento local, es una opción para todo el archivo de Publicaciones. Aquí tienes un ejemplo de coincidencia georreferenciada de 2006 para el área de San Francisco basada en un filtro de “golden gate”:

https://x.com/biz/statuses/28311

<div id="geographical-references-in-a-post-message">
  ##### Publicaciones con etiqueta geográfica por parte del usuario
</div>

En noviembre de 2009, X introdujo la Post Geotagging API, que permitió etiquetar publicaciones con una ubicación exacta. En junio de 2010, X presentó X Places, que representan un área geográfica a nivel de recinto, vecindario o ciudad. Aproximadamente entre el 1% y el 2% de las publicaciones incluyen etiqueta geográfica mediante cualquiera de estos métodos.

El historial de geoetiquetado disponible depende de la API histórica que utilice. Con las Search APIs, la posibilidad de empezar a hacer coincidencias en publicaciones usando algunos operadores geográficos comenzó en marzo de 2010 y, con otros, en febrero de 2015. Si utiliza Historical PowerTrack, la georreferenciación comienza el 1 de septiembre de 2011. Cuando se construyó el archivo de Historical PowerTrack, no se incluyó ningún geoetiquetado anterior a esa fecha.

<div id="posts-geo-tagged-by-the-user">
  ##### Ubicación «home» del perfil de la cuenta establecida por un usuario
</div>

Todos los usuarios de X pueden establecer su Ubicación de perfil, indicando su ubicación principal. Millones de usuarios de X proporcionan esta información, lo que incrementa significativamente la cantidad de geodatos en X Firehose. Estos metadatos de ubicación son una cadena no normalizada, generada por el usuario y de formato libre. Aproximadamente el 30% de las cuentas tienen metadatos de Profile Geo que pueden resolverse a nivel de país.

Al igual que con la geolocalización de las publicaciones, los métodos de coincidencia y los periodos disponibles dependen de la API histórica que se esté utilizando. Historical PowerTrack permite a los usuarios realizar su propia coincidencia personalizada en estas cadenas de formato libre. Para facilitar ese proceso, X también proporciona un [Profile Geo Enrichment](/es/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo) que realiza la geocodificación cuando es posible, proporcionando metadatos normalizados y operadores correspondientes. Los operadores de Profile Geo están disponibles tanto en Historical PowerTrack como en las Search APIs. Con Historical PowerTrack, estos metadatos de Profile Geo están disponibles desde junio de 2014. Con las Search APIs, estos metadatos están disponibles desde febrero de 2015.

#### Enlaces compartidos y contenido multimedia

Compartir enlaces de páginas web, fotos y videos siempre ha sido un caso de uso fundamental en X. En sus primeros años, todas estas acciones implicaban incluir un enlace URL en el mensaje de la Publicación. En 2011, X integró el uso compartido de fotos directamente en su interfaz de usuario. En 2016 se añadieron los videos nativos.

Dado este historial, existe una variedad de Operadores de filtrado utilizados para hacer coincidir este contenido. Hay un conjunto de Operadores que determinan si las Publicaciones incluyen enlaces, fotos y videos. Además, dado que la mayoría de las URL compartidas en X se acortan para consumir menos caracteres de una Publicación (p. ej., generadas por servicios como bitly o tinyurl), X proporciona enriquecimientos de datos que generan una URL completa y expandida con la que se puede hacer coincidencia. Por ejemplo, si quisiera hacer coincidir Publicaciones que incluyan enlaces sobre X y sistemas de alerta temprana, un filtro que haga referencia a “comunicación sobre fenómenos meteorológicos severos” coincidiría con una Publicación que contenga esta URL [http://bit.ly/1XV1tG4](http://bit.ly/1XV1tG4).

En marzo de 2012 se introdujo el [enriquecimiento de URL expandida](/es/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls). Antes de ese momento, las cargas útiles de las Publicaciones incluían solo la URL proporcionada por el usuario. Por lo tanto, si el usuario incluía una URL acortada, puede resultar difícil hacer coincidir las URL (expandidas) de interés. Con Historical PowerTrack y las API de búsqueda, estos metadatos están disponibles a partir de marzo de 2012.

En julio de 2016 se introdujo el enriquecimiento de URL mejorada. Esta versión mejorada proporciona el título y la descripción HTML de un sitio web en la carga útil de la Publicación, junto con Operadores para hacer coincidir esos campos. Con Historical PowerTrack, estos metadatos pasan a estar disponibles en julio de 2016. Con las API de búsqueda, estos metadatos comienzan a aparecer en diciembre de 2014.

En septiembre de 2016, X introdujo “adjuntos nativos”, en los que un enlace compartido al final no cuenta para el límite de 140 caracteres de la Publicación. Ambos enriquecimientos de URL siguen aplicándose a estos enlaces compartidos.

Para otros detalles específicos del producto sobre el filtrado de URL, consulte los artículos correspondientes para obtener más información.
