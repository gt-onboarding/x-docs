La API de X proporciona los siguientes códigos de respuesta y error para ayudar a comprender y depurar en el momento. Usa la guía de depuración y el índice de errores a continuación para obtener contexto adicional.

Las respuestas correctas se indican con un código HTTP de la serie 200 y una carga útil en formato JSON que contiene los objetos solicitados, creados, modificados o eliminados, junto con la interpretación del servidor de tu solicitud.

Las respuestas de error se devuelven con un código HTTP que no pertenece a la serie 200. Los distintos códigos de error indican diferentes motivos del error. La API de X intenta devolver [códigos de estado HTTP](https://en.wikipedia.org/wiki/List_of_HTTP_status_codes) apropiados para cada solicitud.

<div id="x-api-v2-http-status-codes">
  ## Códigos de estado HTTP de la API de X v2
</div>

| Código | Texto | Descripción | Consejos para la resolución de problemas |
| :--- | :--- | :---------- | :------------------- |
| 200 | OK | ¡La solicitud se realizó correctamente! | |
| 400 | Bad Request | La solicitud no es válida o no puede procesarse. Un mensaje de error adjunto brindará más detalles. Las solicitudes sin autenticación o con parámetros de consulta no válidos se consideran no válidas y devolverán esta respuesta. | Verifique el formato de su consulta JSON. Por ejemplo, si su regla contiene comillas dobles asociadas con una coincidencia exacta u otro operador, puede que deba escaparlas con una barra invertida para distinguirlas de la estructura del formato JSON. |
| 401 | Unauthorized | Hubo un problema al autenticar su solicitud. Esto podría deberse a credenciales de autenticación faltantes o incorrectas. También puede devolverse en otras circunstancias no definidas. | Verifique que esté usando el [método de autenticación](https://docs.x.com/es/resources/fundamentals/authentication/overview) correcto y que sus credenciales sean correctas. |
| 403 | Forbidden | La solicitud se entiende, pero fue rechazada o no se permite el acceso. Un mensaje de error adjunto explicará por qué. | Verifique que su cuenta de desarrollador incluya acceso al endpoint que intenta usar. También puede que necesite que su App sea incluida en la allowlist (p. ej., Engagement API o API de Anuncios de X) o solicitar acceso. |
| 404 | Not Found | El URI solicitado no es válido o el recurso solicitado, como un usuario, no existe. | Verifique que esté usando parámetros válidos y el URI correcto para el endpoint que utiliza. |
| 409 | Connection Exception | Se devuelve al intentar conectarse a un flujo filtrado que no tiene reglas. | Verifique que haya creado al menos una regla en el flujo al que se conecta. El flujo filtrado solo devolverá publicaciones que coincidan con una regla activa. Si no hay reglas, el flujo no devolverá publicaciones. |
| 429 | Too Many Requests | Se devuelve cuando no se puede atender una solicitud debido a que se agotó el [límite de velocidad](/es/resources/fundamentals/rate-limits) de la App o el [límite de publicaciones](/es/x-api/fundamentals/post-cap). Consulte [Limitación de velocidad](/es/resources/fundamentals/rate-limits). | Verifique la cantidad de solicitudes por ventana temporal permitida para el endpoint que usa. Espere a que la ventana se reinicie. Espacie sus solicitudes para no alcanzar los límites de velocidad o actualice al siguiente plan de datos disponible. |
| 500 | Internal Server Error | Algo está fallando. Suele ser un error temporal, por ejemplo, en una situación de alta carga o si un endpoint tiene problemas temporales. | Revise la [página de estado](/es/status) de la API de X o el [foro de la comunidad de desarrolladores](https://devcommunity.x.com/) por si otros tienen problemas similares, o simplemente espere e intente nuevamente más tarde. |
| 501 | Unimplemented | La API de X no admite este endpoint y no puede cumplir la solicitud. |  |
| 503 | Service Unavailable | Los servidores de X están activos, pero sobrecargados con solicitudes. Inténtelo de nuevo más tarde. | Revise la [página de estado](/es/status) de la API de X o el [foro de la comunidad de desarrolladores](https://devcommunity.x.com/) por si otros tienen problemas similares, o simplemente espere e intente nuevamente más tarde. |

Cuando se produce un error durante una solicitud, se devuelve información detallada sobre el error en el cuerpo de la respuesta para ayudar a diagnosticar el problema. Un campo type, que es un URI, indica la naturaleza del problema, mientras que campos adicionales brindan más detalles. Los campos type, title y detail siempre se devolverán en estos cuerpos (vea la tabla a continuación). Cualquier campo adicional, como en el ejemplo siguiente, variará según el tipo de error.

```json
{
  "client_id": "101010101",
  "required_enrollment": "Standard Basic",
  "registration_url": "https://developer.twitter.com/en/account",
  "title": "Client Forbidden",
  "detail": "This request must be made using an approved developer account that is enrolled in the requested endpoint. Learn more by visiting our documentation.",
  "reason": "client-not-enrolled",
  "type": "https://api.x.com/2/problems/client-forbidden"
}
```

<div id="partial-errors">
  ## Errores parciales
</div>

En algunos casos puede ver los errores descritos arriba en una respuesta que devolvió un código de estado 200. En esos casos, el endpoint está diseñado para devolver los datos que pueda, a la vez que proporciona errores detallados sobre aquello que no pudo devolver.

Por ejemplo, el [endpoint de consulta de Posts](/es/x-api/posts/lookup/introduction) permite que una [App de desarrollador](/es/resources/fundamentals/developer-apps#overview) de X solicite más de un ID. Si algunos de esos Posts están disponibles, pero uno de ellos ha sido eliminado, los Posts disponibles se devolverían en el campo data de la respuesta. En la carga útil se devolvería un campo adicional errors, que indica qué Post(s) solicitados no se pudieron devolver. Se utiliza el mismo formato que en los errores de solicitud completa para facilitar el diagnóstico de problemas.

<div id="error-information">
  ### Información sobre errores
</div>

Cada tipo de problema indica la naturaleza del error encontrado. También puedes consultar una lista completa de posibles problemas en nuestra [especificación de la API](https://api.x.com/2/openapi.json).

La API de X intenta devolver [códigos de estado HTTP](https://en.wikipedia.org/wiki/List_of_HTTP_status_codes) adecuados para cada solicitud.

| Título | Tipo | Descripción |
| :--- | :--- | :--- |
| Problema genérico | `about:blank` | Un problema genérico sin información adicional más allá de la proporcionada por el código de estado HTTP. |
| Problema de solicitud no válida | `https://api.X.com/2/problems/invalid-request` | Un problema que indica que esta solicitud no es válida. Si tu solicitud incluye un cuerpo POST, asegúrate de que el contenido sea JSON válido y cumpla con la [especificación de OpenAPI](https://api.x.com/2/openapi.json). |
| Problema de recurso no encontrado | `https://api.X.com/2/problems/resource-not-found` | Un problema que indica que una Publicación, Usuario, etc. no existe. |
| Problema de recurso no autorizado | `https://api.X.com/2/problems/not-authorized-for-resource` | Un problema que indica que no tienes permiso para ver una Publicación, Usuario, etc. en particular. |
| Problema de cliente prohibido | `https://api.X.com/2/problems/client-forbidden` | Un problema que indica que tu cliente tiene prohibido realizar esta solicitud. |
| Problema de recurso no permitido | `https://api.X.com/2/problems/disallowed-resource` | Un problema que indica que el recurso solicitado infringe los preceptos de esta API. |
| Problema de autenticación no compatible | `https://api.X.com/2/problems/unsupported-authentication` | Un problema que indica que la autenticación utilizada no es compatible. |
| Problema de límite de uso excedido | `https://api.X.com/2/problems/usage-capped` | Un problema que indica que se ha superado un límite de uso. |
| Problema de excepción de conexión | `https://api.X.com/2/problems/streaming-connection` | Un problema que indica que hay algo mal con la conexión. |
| Problema de cliente desconectado | `https://api.X.com/2/problems/client-disconnected` | Tu cliente se ha desconectado. |
| Problema de desconexión operativa | `https://api.X.com/2/problems/operational-disconnect` | Has sido desconectado por razones operativas. |
| Problema de límite de reglas | `https://api.X.com/2/problems/rule-cap` | Has superado el número máximo de reglas. |
| Problema de longitud de regla | `https://api.X.com/2/problems/rule-length` | Has superado el número máximo de caracteres permitidos en tu consulta o regla según tu nivel de acceso. Consulta los [niveles de acceso](/es/x-api/getting-started/about-x-api#x-api-access-levels-and-versions). |
| Problema de reglas no válidas | `https://api.X.com/2/problems/invalid-rules` | La regla que has enviado no es válida. |
| Problema de reglas duplicadas | `https://api.X.com/2/problems/duplicate-rules` | La regla que has enviado está duplicada. |

<div id="troubleshooting-tips">
  ## Consejos para la resolución de problemas
</div>

<Accordion title="Depuración de tu código">
  Al crear una aplicación, es normal encontrarse con errores o incidencias inesperadas de vez en cuando. A continuación, encontrarás algunos consejos para depurar tu código:

  Empieza descomponiendo el problema en partes más pequeñas para identificar su origen. Por ejemplo, si estás integrando un endpoint REST o de streaming, el problema podría estar en cualquiera de los siguientes elementos:

  - El endpoint requiere [autenticación](/es/resources/fundamentals/authentication/overview) adecuada.
  - El endpoint requiere parámetros y encabezados válidos. Cualquier regla de filtrado debe construirse con los operadores correctos y la [sintaxis](/es/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#building-rules-and-queries) adecuada.
  - El URI del endpoint debe ser correcto y, en el caso de los endpoints REST, se debe usar el método HTTP correspondiente.
  - Es posible que los datos o el recurso al que intentas acceder no estén disponibles para ti (por ejemplo, los datos privados solo están disponibles para usuarios autenticados).
  - Tu paquete de datos actual solo te da acceso a determinados endpoints y a límites de tasa específicos. Consulta tu [portal de desarrolladores](https://developer.x.com/en/portal/products) para más detalles.
  - Tu código utiliza una biblioteca de terceros para integrar el endpoint.
  - Tu código necesita analizar correctamente la respuesta del endpoint.

  Lee el mensaje de error. Este debería darte una buena indicación de cuál es el problema. Usa las tablas de la [sección de códigos de error y respuesta](#x-api-http-status-codes) para obtener consejos de resolución de problemas específicos para cada código.

  Para endpoints REST, puedes usar un cliente REST como [Postman](https://www.postman.com/) o [Insomnia](https://insomnia.rest/) para validar los pasos del (1) al (5) anteriores (consulta nuestra guía ["Primeros pasos con Postman"](/es/tutorials/postman-getting-started)). Si la solicitud con el cliente REST devuelve un código de estado 200 (éxito), puedes suponer que el problema está en tu código o en la biblioteca que estás usando, y no en la solicitud al endpoint en sí.

  Para endpoints de streaming, puedes usar cURL (una herramienta de línea de comandos para realizar o enviar solicitudes usando la sintaxis de URL) para comprobar si el problema está en la solicitud al endpoint (pasos del (1) al (5) anteriores) o en tu propio código (pasos del (6) al (7) anteriores).
</Accordion>

<Accordion title="Problemas de autenticación">
  - Verifica que estás usando el [método de autenticación](/es/resources/fundamentals/authentication/overview) adecuado requerido por el endpoint. Puedes identificarlo en la página de referencia del endpoint.
  - Verifica que tus credenciales de autenticación sean correctas. Puedes revisar o regenerar las claves y los tokens de tu App en la [sección Apps](https://developer.x.com/en/portal/projects-and-apps) de tu panel de desarrollador (en “Details”).
  - Verifica que has [autorizado correctamente tu solicitud OAuth 1.0a](/es/resources/fundamentals/authentication/oauth-1-0a/authorizing-a-request) con `oauth_nonce`, `oauth_signature` y `oauth_timestamp`.
  - Si el problema persiste, considera usar una biblioteca de OAuth, un cliente REST como [Postman](https://www.postman.com/) o [Insomnia](https://insomnia.rest/), o [xurl](https://github.com/xdevplatform/xurl).

  Revisa nuestra [guía sobre autenticación](/es/resources/fundamentals/authentication/overview) para obtener más información sobre todo lo anterior.
</Accordion>

<Accordion title="Problemas de limitación de tasa y límite de publicaciones">
  Se puede devolver un error 429 si alcanzas la limitación de tasa para un endpoint determinado o si alcanzas el límite de publicaciones.

  Si se devuelve el error específico “Too many requests”, alcanzaste la limitación de tasa del endpoint. En otras palabras, has excedido el número máximo de solicitudes permitidas para un endpoint por un período de tiempo especificado.

  Ten en cuenta que las limitaciones de tasa se establecen a nivel de App y a nivel de usuario.

  - **Nivel de X App** indica la cantidad de solicitudes permitidas cuando utilizas OAuth 2.0 App-Only, donde las limitaciones de tasa se determinan globalmente para toda la App. Por ejemplo, si un método permite 15 solicitudes por ventana de limitación de tasa, te permite hacer 15 solicitudes por ventana en nombre de tu X App. Este límite se considera completamente por separado del límite a nivel de usuario. Lee más en nuestra guía sobre [OAuth 2.0 App-Only](/es/resources/fundamentals/authentication/oauth-2-0/application-only).
  - **Nivel de contexto de usuario** indica la cantidad de solicitudes que se pueden realizar por token de acceso de usuario cuando se utiliza OAuth 1.0a User Context u OAuth 2.0 Authorization Code con PKCE. Por ejemplo, si un método permite 15 solicitudes por ventana de limitación de tasa, entonces permite 15 solicitudes por ventana y por token de acceso de usuario. Lee más en nuestra guía sobre cómo obtener los tokens de acceso de un usuario con [OAuth 1.0a](/es/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens) y [OAuth 2.0](/es/resources/fundamentals/authentication/oauth-2-0/user-access-token).

  Comienza por verificar las limitaciones de tasa del endpoint que estás utilizando. Puedes encontrar esta información en la página de referencia del API del endpoint y en el nuevo panel del portal para desarrolladores.

  Revisa nuestra documentación para obtener información adicional sobre las limitaciones de tasa, incluido cómo usar encabezados HTTP para rastrear la posición de tu App respecto a una limitación de tasa determinada, cómo recuperarte de un código de error 429 por limitación de tasa y consejos para evitar llegar a la limitación de tasa en primer lugar:

  - [Limitaciones de tasa para endpoints v2](/es/x-api/fundamentals/rate-limits)

  Si recibiste el error específico “Usage cap exceeded: Monthly product cap”, significa que alcanzaste el límite de publicaciones para tu nivel de acceso. Tenemos muchos detalles sobre qué son estos [límites de publicaciones](/es/x-api/fundamentals/post-cap) en nuestra página de documentación.
</Accordion>

<Accordion title="Problemas de publicaciones faltantes">
  Sigue los pasos a continuación si esperabas que se devolviera una publicación, pero el endpoint no la entregó.

  - Verifica tu regla para asegurarte de que estás usando los [operadores](/es/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#enterprise-operators) y la [sintaxis](/es/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#building-rules-and-queries) adecuados. Divide la regla en cláusulas más pequeñas para asegurarte de que estás usando la sintaxis correcta.
  - Si la cuenta desde la cual se envió la publicación estaba [protegida](https://help.twitter.com/en/safety-and-security/public-and-protected-tweets) en el momento en que se creó la publicación, la publicación no se devolverá, incluso si la cuenta es pública en el momento de la solicitud al endpoint. Normalmente puedes comprobar esto usando [X Advanced Search](https://x.com/search-advanced): si una publicación no aparece utilizando la funcionalidad de X Advanced Search, debes asumir que no será devuelta por el endpoint.

  Los siguientes pasos aplican solo a endpoints de streaming:

  - ¿Estabas conectado al flujo cuando se envió la publicación? Recuerda que la marca de tiempo entregada en el objeto de la publicación indica la hora en UTC. Si experimentaste una desconexión cuando se envió la publicación, revisa las [funciones de recuperación y redundancia](/es/x-api/posts/filtered-stream/integrate/recovery-and-redundancy-features#recovery-and-redundancy) disponibles para rellenar cualquier dato perdido.
  - ¿Tu regla estaba activa cuando se envió la publicación? Recuerda que la marca de tiempo entregada en el objeto de la publicación indica la hora en UTC.
</Accordion>

<Accordion title="Problemas de desconexión por búfer completo">
  Puede recibir uno de los siguientes errores cuando su flujo no mantiene el ritmo de la velocidad a la que entregamos datos y su App no consume el flujo con la suficiente rapidez:

  ```
  This stream has been disconnected because your client was unable to keep up with us.
  ```

  ```
  This stream has been disconnected for operational reasons.
  ```

  Permitimos cierto retraso en la entrega durante un tiempo y contamos con un búfer temporal de preparación para cada flujo de nuestro lado; pero si no se pone al día, iniciamos una desconexión para permitirle reconectarse en el punto temporal actual. Tenga en cuenta que esto puede provocar pérdida de datos (para los datos que estén en el búfer en el momento de la desconexión por búfer completo).

  Esto puede ocurrir ante picos grandes de datos. En general, recomendamos usar un proceso de intermediación de mensajes para consumir datos rápidamente, separado del proceso de procesamiento.

  Si es un cliente empresarial que usa endpoints v1.1, puede obtener más información sobre cómo optimizar su App para evitar desconexiones como esta en nuestros artículos sobre [conexión](/es/x-api/enterprise-gnip-2.0/powertrack-api#disconnections-explained) y sobre cómo consumir datos de streaming [aquí](https://developer.x.com/en/docs/tutorials/consuming-streaming-data) y [aquí](/es/x-api/enterprise-gnip-2.0/powertrack-api#planning-for-high-volume-social-data-events).

  Existe una variedad de herramientas disponibles para recuperar publicaciones perdidas debido a una desconexión, incluidas las siguientes. Tenga en cuenta que estas herramientas solo están disponibles con endpoints v1.1 y nivel de acceso empresarial.

  - [**Conexiones redundantes**](/es/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - Con múltiples conexiones, consuma el flujo desde varios servidores para evitar pérdida de datos cuando uno se desconecte.
  - [**Reproducción**](/es/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - Recupere datos de los últimos 5 días mediante un flujo separado.
  - [**Reabastecimiento**](/es/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - Vuelva a conectarse dentro de 5 minutos y continúe desde donde se quedó.
  - **Historical PowerTrack** - Recupere datos de todo el archivo de X.
</Accordion>

<div id="before-you-post-a-question">
  ### Antes de publicar una pregunta
</div>

- Busca en la documentación para desarrolladores de X información relacionada con tu problema
- Busca en el foro de la comunidad preguntas similares de otros desarrolladores
- Revisa las directrices del foro de la comunidad

<div id="when-you-post-a-question-make-sure-to-include-the-following-information">
  ### Cuando publiques una pregunta, asegúrate de incluir la siguiente información
</div>

- Una descripción del problema
- La llamada a la API que se está realizando (incluye los encabezados, si es posible)
- La respuesta de X devuelta (incluye cualquier mensaje de error)
- Qué esperabas recibir en su lugar
- Lista de pasos realizados para solucionar el problema
- Lista de pasos necesarios para reproducir el problema
- Si corresponde, el período durante el cual ocurrió el problema
- Si corresponde, el ID de la App, el ID de la publicación, etc.
- Cualquier fragmento de código o captura de pantalla relevantes

Incluye solo un tema/pregunta por publicación.

Si tienes solicitudes de funcionalidades o comentarios, envíalos a través del [Formulario de comentarios de la Plataforma para Desarrolladores de X](https://t.co/devfeedback).

Para problemas relacionados con las políticas, como la suspensión de Apps, contacta al [soporte de Políticas](https://help.twitter.com/forms/platform).

Para problemas relacionados con X, como el inicio de sesión y el soporte de cuentas, utiliza el [Centro de ayuda de X](https://help.twitter.com/en/contact-us).
