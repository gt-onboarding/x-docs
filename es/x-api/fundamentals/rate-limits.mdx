---
title: Límites de frecuencia
---

Cada día, muchos miles de desarrolladores realizan solicitudes a la API de X. Para gestionar el alto volumen de estas solicitudes, se establecen límites en la cantidad de solicitudes que se pueden realizar. Estos límites ayudan a garantizar la confiabilidad y escalabilidad de la API en la que confía nuestra comunidad de desarrolladores.

El número máximo de solicitudes permitidas se basa en un intervalo de tiempo, normalmente durante un período o ventana especificada. El intervalo más común es de quince minutos. Por ejemplo, un endpoint con un límite de 900 solicitudes por 15 minutos permite hasta 900 solicitudes en cualquier intervalo de 15 minutos.

Los límites de frecuencia dependen del método de autenticación. Por ejemplo, si se usa [OAuth 1.0a User Context](/es/resources/fundamentals/authentication), cada conjunto de [Access Tokens](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) de los usuarios tiene su propio límite de frecuencia por período. Alternativamente, si se usa [OAuth 2.0 Bearer Token](/es/resources/fundamentals/authentication#oauth-2-0), tu app tendrá su propio límite independiente por período de tiempo. Cuando se superan estos límites, se devuelve un error.

<div id="rate-limits-and-authentication-method">
  ## Límites de velocidad y método de autenticación
</div>

Los límites de velocidad se establecen tanto a nivel de la App de desarrollador como del token de acceso de usuario:

- **[Token Bearer de OAuth 2.0](/es/resources/fundamentals/authentication/oauth-2-0/application-only): Límite de velocidad de la App**
  Este método permite realizar una cierta cantidad de solicitudes en nombre de tu App de desarrollador. Al usar este método de autenticación, los límites se determinan por las solicitudes realizadas con un token Bearer.
  - Ejemplo: Con un límite de 450 solicitudes por intervalo de 15 minutos, puedes realizar 450 solicitudes en nombre de tu App dentro de ese intervalo.

- **[Contexto de usuario de OAuth 1.0a](/es/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens): Límite de velocidad por usuario**
  Este método permite realizar solicitudes en nombre de un usuario de X identificado por el token de acceso de usuario. Por ejemplo, si necesitas obtener métricas privadas de publicaciones, autentícate con tokens de acceso de usuario para ese usuario, generados mediante el [flujo OAuth de 3 fases](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow).
  - Ejemplo: Con un límite de 900 solicitudes por 15 minutos por usuario, puedes realizar hasta 900 solicitudes por usuario en ese período.

<div id="http-headers-and-response-codes">
  ## Encabezados HTTP y códigos de respuesta
</div>

Use los encabezados HTTP para entender la situación de su aplicación dentro de un límite de solicitudes determinado, según la solicitud más reciente realizada.

- `x-rate-limit-limit`: límite máximo de solicitudes para el endpoint
- `x-rate-limit-remaining`: solicitudes restantes en la ventana de 15 minutos
- `x-rate-limit-reset`: tiempo restante antes de que se restablezca el límite de solicitudes (en segundos de época UTC)

<div id="error-responses">
  ### Respuestas de error
</div>

Si una aplicación supera el límite de solicitudes para un endpoint, la API devolverá una respuesta [HTTP 429 “Too Many Requests”](http://tools.ietf.org/html/rfc6585) con el siguiente mensaje de error en el cuerpo de la respuesta:

```json
{ "errors": [ { "code": 88, "message": "Rate limit exceeded" } ] }
```

<div id="recovering-from-a-rate-limit">
  ## Recuperarse de un límite de tasa
</div>

Cuando se exceden estos límites de tasa, el endpoint devuelve un error 429 “Too Many Requests”. Como se indica a continuación, cuando ocurren errores por límite de tasa, una práctica recomendada es examinar los encabezados HTTP que indican cuándo se restablece el límite y pausar las solicitudes hasta entonces.<br />Cuando ocurre un error de “Too Many Requests” o de limitación de tasa, es necesario reducir la frecuencia de las solicitudes. Cuando se produce un error de límite de tasa, se puede comprobar el encabezado HTTP `x-rate-limit-reset:` para saber cuándo se restablecerá la limitación de tasa<br /><br />. Otro patrón común se basa en el backoff exponencial, donde el tiempo entre solicitudes comienza siendo pequeño (por ejemplo, unos segundos) y luego se duplica antes de cada reintento. Esto continúa hasta que una solicitud tenga éxito o se alcance un tiempo máximo razonable entre solicitudes (por ejemplo, unos minutos).<br /><br /> Idealmente, el cliente es consciente de los límites de tasa existentes y puede pausar las solicitudes hasta que expire la ventana actualmente excedida. Si excedes un límite de 15 minutos, tiene sentido esperar uno o dos minutos antes de reintentar.<br /><br /> Ten en cuenta que, más allá de estos límites en el número de solicitudes, el nivel de acceso Standard Basic proporciona hasta 500.000 publicaciones por mes desde los endpoints de búsqueda reciente y flujo filtrado. Si has excedido el límite mensual de publicaciones, tiene más sentido que tu app muestre una notificación, conozca su día de renovación del mes y posponga las solicitudes hasta ese día.

<div id="tips-to-avoid-being-rate-limited">
  ### Consejos para evitar restricciones por límite de tasa
</div>

Los siguientes consejos te ayudarán a programar de forma defensiva y a reducir la posibilidad de que se apliquen límites de tasa. Algunas funciones de la aplicación que podrías querer ofrecer son simplemente inviables debido a estas restricciones, especialmente en lo relativo a la actualidad de los resultados. Si tu aplicación requiere información en tiempo real, consulta los endpoints de Flujo filtrado y de muestreo.

<div id="caching">
  #### Caché
</div>

Guarde las respuestas de la API si prevé un uso frecuente. En lugar de invocar la API en cada carga de página, almacene la respuesta en caché de forma local.

<div id="prioritize-active-users">
  #### Priorice a los usuarios activos
</div>

Si su sitio realiza un seguimiento de muchos usuarios de X (por ejemplo, al obtener su estado actual o estadísticas sobre su uso de X), considere solicitar datos únicamente de los usuarios que hayan iniciado sesión recientemente en su sitio.

<div id="adapt-to-the-search-results">
  #### Adáptate a los resultados de búsqueda
</div>

Si tu aplicación supervisa un volumen alto de términos de búsqueda, consulta con menor frecuencia aquellas búsquedas que no devuelven resultados que aquellas que sí lo hacen. Al aplicar un backoff puedes mantenerte al día con las consultas populares sin malgastar recursos solicitando consultas que rara vez cambian. Como alternativa, considera usar el endpoint de Flujo filtrado y aplicar filtros con tus consultas de búsqueda.

<div id="denylist">
  #### Lista de denegación
