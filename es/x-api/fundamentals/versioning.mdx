title: "Control de versiones"

<p>Junto con la API de X v2, lanzamos una nueva estrategia de versionado que permite a los desarrolladores entender mejor cuándo esperar cambios en las API públicas de X y cuándo deberán migrar a nuevas versiones. </p>
<p>Los desarrolladores serán notificados sobre desaprobaciones, retiros, cambios y adiciones a la API de X a través de nuestros <a href="https://developer.x.com/en/updates/stay-informed">canales de comunicación</a>, para que puedan planificar adecuadamente la incorporación de estos cambios en su hoja de ruta de desarrollo. Todos los cambios en la API se registrarán en el registro de cambios.</p>
<p>Actualmente, la API de X tiene tres versiones. Recomendamos encarecidamente utilizar la API de X v2 al planificar su integración, a menos que aún no hayamos publicado en v2 la funcionalidad que requiere su caso de uso. </p>
<p>Para obtener más información sobre cada versión, visite las siguientes páginas:</p>

<ul>
  <li><a href="/es/x-api/introduction">API de X v2</a></li>
  <li>Gnip 2.0 (<a href="/es/x-api/enterprise-gnip-2.0/enterprise-gnip">empresa</a>)</li>
</ul>

<div id="versioning-strategy">
  ## Estrategia de versionado
</div>

<p>El versionado de la API de X se representará mediante números de versión declarados en la ruta de nuestros endpoints:</p>
<p>`https://api.x.com/2/tweets`</p>
<p>Nuestro objetivo es lanzar versiones mayores de la API de X según sea necesario, pero no con una frecuencia superior a una cada 12 meses. Se publicará una versión mayor cuando se introduzcan cambios disruptivos (breaking changes) en la API. Publicaremos guías de migración al lanzar una nueva versión mayor para ayudar a los desarrolladores a migrar a la nueva versión. </p>
<p>Un cambio disruptivo requiere que los desarrolladores modifiquen su código para mantener la funcionalidad existente en su aplicación. Los cambios no disruptivos serán aditivos y se implementarán en la versión más reciente cuando estén listos, sin requerir trabajo por parte del desarrollador a menos que desee aprovechar la nueva funcionalidad.</p>
<p>Si debe implementarse un cambio disruptivo a mitad de ciclo (por razones de seguridad o privacidad), este cambio se aplicará a la versión más reciente.</p>

<div id="breaking-changes">
  ### Cambios incompatibles
</div>

<p>Estos cambios requieren que los desarrolladores modifiquen su código para mantener la funcionalidad existente de su aplicación.</p>

<ul>
  <li>Incorporación de un nuevo parámetro obligatorio</li>
  <li>Eliminación de un endpoint existente</li>
  <li>Eliminación de cualquier campo en la respuesta (obligatorio u opcional)</li>
  <li>Eliminación de un parámetro de consulta</li>
  <li>Reestructuración del formato de entrada o salida (por ejemplo, convertir un campo de nivel superior en un subcampo o mover los errores para que aparezcan en línea)</li>
  <li>Cambio del nombre o del tipo de datos de un parámetro de entrada existente o de un valor de salida</li>
  <li>Cambio del nombre de un campo</li>
  <li>Cambio del nombre de un recurso</li>
  <li>Cambio de un código de respuesta</li>
  <li>Cambio de tipos de error</li>
  <li>Cambios en los ámbitos de autorización existentes</li>
</ul>

<div id="non-breaking-changes">
  ### Cambios retrocompatibles
</div>

<ul>
  <li>Incorporación de un nuevo endpoint</li>
  <li>Incorporación de un nuevo parámetro opcional</li>
  <li>Incorporación de un nuevo campo en la respuesta</li>
  <li>Modificación del texto de los mensajes de error</li>
  <li>Disponibilidad de nuevos ámbitos</li>
  <li>“Anulación” de campos (cambiar el valor de un campo a null por motivos de privacidad/seguridad como alternativa a eliminarlo por completo)</li>
</ul>

<div id="deprecation-and-retirement">
  ### Obsolescencia y retiro
</div>

<p>En primer lugar, esta es nuestra definición de lo que significan obsolescencia y retiro para la API de X:</p>

<ul>
  <li><b>Obsolescencia:</b> La función deja de recibir soporte por parte del equipo. No se lanzará nueva funcionalidad asociada a esa función y, si hay errores o problemas con el producto, la probabilidad de que exploremos una corrección es extremadamente baja. </li>
  <li><b>Retiro:</b> La función dejará de estar accesible.</li>
</ul>

<p>En la mayoría de los casos, en cuanto se lanza una nueva versión, la versión anterior se marca como obsoleta. Las versiones permanecerán en estado de obsolescencia durante un período de tiempo, tras el cual se retirarán. </p>
<p>Le recomendamos <a href="https://developer.x.com/en/updates/stay-informed">mantenerse informado</a> para conocer más sobre futuras obsolescencias y retiros.</p>
