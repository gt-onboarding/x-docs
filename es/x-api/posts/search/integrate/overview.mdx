  ### Conceptos clave

<div id="authentication">
  #### Autenticación
</div>

Todos los endpoints de la API de X v2 requieren que autentiques tus solicitudes con un conjunto de credenciales, también conocidas como claves y tokens. Puedes usar OAuth 1.0a User Context, OAuth 2.0 App-Only o OAuth 2.0 Authorization Code con PKCE para autenticar tus solicitudes al endpoint de búsqueda reciente. Debes usar OAuth 2.0 App-Only cuando uses el endpoint de búsqueda de archivo completo.

[OAuth 2.0 App-Only](/es/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3) solo requiere que envíes un [OAuth 2.0 App Access Token](/es/resources/fundamentals/authentication#bearer-token-also-known-as-app-only) con tu solicitud. Puedes generar un App Access Token directamente en una App de desarrollador o generar uno usando el endpoint [POST oauth2/token](/es/resources/fundamentals/authentication#post-oauth2-token).

[OAuth 1.0a User Context](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) requiere que utilices tus API Keys, Access Tokens de usuario y algunos otros parámetros para [crear un encabezado de autorización](https://developer-staging.x.com/resources/fundamentals/authentication/authorizing-a-request), que luego enviarás con tu solicitud. Los Access Tokens deben estar asociados con el usuario en cuyo nombre haces la solicitud. Si deseas generar un conjunto de Access Tokens para otro usuario, ese usuario debe autorizar tu App mediante el [flujo OAuth de 3 patas](https://developer-staging.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens).

Ten en cuenta que OAuth 1.0a puede ser difícil de usar. Si no estás familiarizado con este método de autenticación, te recomendamos usar una [librería](/es/x-api/tools-and-libraries/overview), una herramienta como Postman o OAuth 2.0 para autenticar tus solicitudes. Si deseas solicitar una publicación o métricas privadas desde estos endpoints, necesitarás usar OAuth 1.0a User Context u OAuth 2.0 Authorization Code con PKCE, lo que garantizará que tengas los permisos adecuados del usuario propietario de ese contenido.

[OAuth 2.0 Authorization Code con PKCE](/es/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) permite un mayor control sobre el alcance (scope) de una aplicación y flujos de autorización en múltiples dispositivos. OAuth 2.0 te permite elegir scopes específicos y granulares que te otorgan permisos específicos en nombre de un usuario.

Para habilitar OAuth 2.0 en tu App, debes activarlo en la configuración de autenticación de tu App, ubicada en la sección de configuración de la App en el portal para desarrolladores.

<Note>
  **Ten en cuenta**

  Si estás solicitando los siguientes [fields](/es/x-api/fundamentals/fields), se requiere OAuth 1.0a User Context u OAuth 2.0 Authorization Code:

  - tweet.fields.non\_public\_metrics
  - tweet.fields.promoted\_metrics
  - tweet.fields.organic\_metrics
  - media.fields.non\_public\_metrics
  - media.fields.promoted\_metrics
  - media.fields.organic\_metrics
</Note>

####

Portal para desarrolladores, Proyectos y Apps de desarrollador

Para trabajar con cualquier endpoint de la API de X v2, debes [registrarte para obtener una cuenta de desarrollador](https://developer.x.com/en/portal/petition/essential/basic-info), configurar un Proyecto dentro de esa cuenta y crear una App de desarrollador dentro de ese Proyecto. Las claves y los tokens de esa App de desarrollador funcionarán con estos endpoints de búsqueda.

Puedes usar claves y tokens de un Proyecto con cualquier [nivel de acceso](/es/x-api/getting-started/about-x-api) para realizar solicitudes al endpoint de búsqueda reciente. Sin embargo, necesitarás usar un Proyecto con nivel de acceso Pro o Enterprise para realizar solicitudes al endpoint de búsqueda de archivo completo. Si tienes [acceso Enterprise](/es/x-api/getting-started/getting-access), contarás con funcionalidades adicionales, incluida la disponibilidad de operadores adicionales y consultas de mayor longitud.
 

<div id="rate-limits">
  #### Límites de solicitudes
</div>

Cada día, muchos miles de desarrolladores realizan solicitudes a la API de X. Para ayudar a gestionar el volumen, se aplican [límites de solicitudes](/es/x-api/fundamentals/rate-limits) a cada endpoint que restringen la cantidad de solicitudes que cada desarrollador puede realizar en nombre de una app o de un usuario autenticado.

Se aplican distintos límites de solicitudes a estos endpoints según el método de autenticación utilizado. Los límites a nivel de app aplican cuando una app realiza solicitudes usando OAuth 2.0 App-Only, mientras que el límite a nivel de usuario aplica a solicitudes realizadas en nombre del usuario autorizador específico usando OAuth 1.0a User Context u OAuth 2.0 Authorization Code con PKCE. Ambos límites se basan en la frecuencia de solicitudes dentro de una ventana de 15 minutos.

Por ejemplo, una app que usa OAuth 2.0 App-Only para realizar solicitudes al endpoint de búsqueda reciente puede efectuar 450 solicitudes (incluidas las de paginación) dentro de un período de 15 minutos. Esa misma app, en el mismo período de 15 minutos y con dos usuarios autenticados diferentes (usando OAuth 1.0a User Context u OAuth 2.0 Authorization Code con PKCE), puede efectuar hasta 180 solicitudes (incluidas las de paginación) al endpoint de búsqueda reciente por cada usuario autenticado.
 

<div id="fields-and-expansions">
  #### Campos y expansiones
</div>

La API de X v2 le permite seleccionar con precisión qué datos desea recibir de la API usando [campos](/es/x-api/fundamentals/fields) y [expansiones](/es/x-api/fundamentals/expansions). El parámetro expansions le permite incluir objetos referenciados en la carga. Por ejemplo, este endpoint permite solicitar objetos de encuesta, lugar, medios y otros mediante el parámetro expansions.

Los parámetros de fields le permiten elegir exactamente qué atributos de los distintos objetos de datos desea recibir. De forma predeterminada, el objeto principal de Publicación que devuelven estos endpoints incluye los campos id y text (además de edit\_history\_tweet\_ids para las publicaciones creadas después del lanzamiento de esa función). Para recibir campos adicionales como author\_id o public\_metrics, deberá solicitarlos explícitamente mediante los parámetros de fields. Algunos campos importantes que quizá desee considerar en su integración son los datos de encuesta, las [métricas](/es/x-api/fundamentals/metrics), las [anotaciones de publicaciones](/es/x-api/fundamentals/post-annotations) y el [ID de conversación](/es/x-api/fundamentals/conversation-id).

Hemos añadido una guía sobre cómo [usar campos y expansiones juntos](/es/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) en nuestro [diccionario de datos de la API de X v2](/es/x-api/fundamentals/data-dictionary).
 

<div id="metrics">
  #### Métricas
</div>

Los endpoints de la API de X v2 le permiten solicitar métricas directamente de los objetos devueltos, siempre que incluya los [campos](/es/x-api/fundamentals/fields) adecuados en su solicitud.

Existen algunas limitaciones con las métricas de Publicaciones que debe tener en cuenta, específicamente relacionadas con la privacidad del usuario y los siguientes campos de respuesta:

- tweet.fields.non\_public\_metrics
- tweet.fields.promoted\_metrics
- tweet.fields.organic\_metrics
- media.fields.non\_public\_metrics
- media.fields.promoted\_metrics
- media.fields.organic\_metrics

Los campos mencionados incluyen datos de métricas privadas, lo que significa que debe estar autorizado por el autor de la Publicación para recuperar estos datos en su nombre cuando use el endpoint de búsqueda reciente; es decir, debe usar [OAuth 1.0a User Context](/es/resources/fundamentals/authentication). Dado que solo puede usar este método de autenticación con la búsqueda reciente, no podrá recuperar estas métricas mediante el endpoint de búsqueda de archivo completo.

Por ejemplo, para recibir `non_public_metrics` de las Publicaciones del usuario con ID 1234, deberá incluir en su solicitud tokens de acceso asociados con ese usuario. Puede hacer que los usuarios autoricen su app y obtener un conjunto de tokens de acceso asociados a ellos mediante el [flujo de OAuth de 3 partes](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow).

Todas las `non_public_metrics`, `organic_metrics` y `promoted_metrics` solo están disponibles para Publicaciones creadas en los últimos 30 días. Esto significa que, cuando solicite los campos mencionados, los resultados se ajustarán automáticamente para incluir únicamente Publicaciones de los últimos 30 días.

Si se solicitan estos campos, solo se devolverán las Publicaciones creadas por el usuario autenticado; todas las demás Publicaciones producirán un mensaje de error.
 

<div id="building-search-queries">
  #### Creación de consultas de búsqueda
</div>

La función principal de estos endpoints es el uso de una sola consulta de búsqueda para filtrar las publicaciones que recibes. Estas consultas se componen de operadores que hacen coincidir atributos de la publicación y del usuario, como palabras clave del mensaje, hashtags y URL. Los operadores pueden combinarse en consultas con lógica booleana y paréntesis para refinar el comportamiento de coincidencia de la consulta.

Puedes consultar nuestra guía sobre [cómo crear una consulta](/es/x-api/posts/search/integrate/build-a-query) para obtener más información.

<div id="pagination">
  #### Paginación
</div>

Estos endpoints utilizan paginación para que las respuestas se devuelvan rápidamente. Cuando haya más resultados de los que pueden enviarse en una sola respuesta (hasta 100 publicaciones en la búsqueda reciente y 500 en la búsqueda de archivo completo), deberá paginar. Use el parámetro max\_results para indicar cuántos resultados se devolverán por página y el parámetro pagination\_token para obtener la siguiente página de resultados. Puede obtener más información en nuestra [guía de paginación](/es/x-api/fundamentals/pagination).

<div id="post-caps">
  #### Límites de publicaciones
</div>

Los endpoints de Search Posts están limitados en la cantidad de publicaciones que pueden devolver en un mes determinado, independientemente de la paginación.

Independientemente del endpoint de búsqueda que uses, las publicaciones devueltas contarán para los [Límites de tuits (límite de publicaciones)](/es/x-api/fundamentals/post-cap) a nivel de proyecto. El uso se muestra en el portal para desarrolladores y el “mes” comienza en el día de renovación de tu suscripción que se muestra en el [panel del portal para desarrolladores](https://developer.x.com/en/portal/dashboard).

**Ediciones de publicaciones**

Las publicaciones que sean elegibles para edición pueden editarse hasta cinco veces en los 30 minutos posteriores a la publicación original. Los endpoints de búsqueda siempre proporcionarán la última versión de la publicación. Si solo solicitas publicaciones que se publicaron hace 30 minutos o más, siempre recibirás la versión final de la publicación. Sin embargo, si tienes un caso de uso casi en tiempo real y estás consultando publicaciones de los últimos treinta minutos, esas publicaciones podrían haberse editado después de que las recibiste. Estas publicaciones pueden rehidratarse con la búsqueda o con el endpoint de Post Lookup para confirmar su estado final. Para obtener más información sobre cómo funcionan las ediciones, consulta la página [Fundamentos de ediciones de publicaciones](/es/x-api/fundamentals/edit-posts).
