<div id="standard-v11-compared-to-x-api-v2">
  ## Standard v1.1 comparado con la API de X v2
</div>

Si has estado trabajando con los endpoints standard v1.1 GET statuses/show y GET statuses/lookup, esta guía te ayudará a comprender las similitudes y diferencias entre los endpoints de consulta de publicaciones de Standard y de la API de X v2.

También puede interesarte nuestra [herramienta visual de migración de formatos de datos](/es/x-api/migrate/data-format-migration) para ayudarte a ver rápidamente las diferencias entre el [formato de datos de la API de X v1.1](/es/x-api/fundamentals/data-dictionary) y el [formato de la API de X v2](/es/x-api/fundamentals/data-dictionary).

- **Similitudes**
  - Contexto de usuario de OAuth 1.0a
  - Límites de publicaciones por solicitud
  - Compatibilidad con el historial de edición de la publicación y metadatos

- **Diferencias**
  - URL de los endpoints
  - Requisitos de App y Proyecto
  - Formato de datos de la respuesta
  - Parámetros de la solicitud

<div id="similarities">
  ### Similitudes
</div>

<div id="oauth-10a-user-context-authentication-method">
  #### Método de autenticación de contexto de usuario de OAuth 1.0a
</div>

El endpoint estándar admite [OAuth 1.0a User Context](/es/resources/fundamentals/authentication), mientras que el nuevo endpoint de consulta de publicaciones de la API de X v2 admite tanto OAuth 1.0a User Context como [OAuth 2.0 App-only](/es/resources/fundamentals/authentication). Por lo tanto, si antes utilizaba uno de los endpoints estándar v1.1 de consulta de publicaciones, puede seguir usando el mismo método de autenticación si migra a la versión de la API de X v2.

La autenticación App-only probablemente sea la forma más sencilla de comenzar. Para aprender a generar un App Access Token, consulte [esta guía de OAuth 2.0 App-only](/es/resources/fundamentals/authentication).

<div id="posts-per-request-limits">
  #### Límites de publicaciones por solicitud
</div>

El endpoint v1.1 [GET statuses/lookup](https://developer.x.com/en/docs/twitter-api/v1/tweets/post-and-engage/api-reference/get-statuses-lookup) permite especificar hasta 100 publicaciones por solicitud. Esto también se aplica al endpoint GET /tweets. Para indicar las 100 publicaciones completas, usa el parámetro `ids` como parámetro de consulta con una lista de [IDs de publicaciones](/es/resources/fundamentals/x-ids) separada por comas.

**Compatibilidad con el historial de edición de publicaciones y metadatos**

Ambas versiones proporcionan metadatos que describen el historial de edición. Consulta las referencias de la API de búsqueda de publicaciones y la [página de fundamentos sobre edición de publicaciones](/es/x-api/fundamentals/edit-posts) para más detalles.

<div id="differences">
  ### Diferencias
</div>

<div id="endpoint-urls">
  #### URLs de endpoints
</div>

- **Endpoints estándar v1.1:**
  - `https://api.x.com/1.1/statuses/show`
  - `https://api.x.com/1.1/statuses/lookup`

- **Endpoint de la API de X v2:**
  - `https://api.x.com/2/tweets`
  - `https://api.x.com/2/tweets/:id`

<div id="app-and-project-requirements">
  #### Requisitos de Apps y Proyectos
</div>

Los endpoints de la API de X v2 requieren credenciales de una [App de desarrollador](/es/resources/fundamentals/developer-apps) asociada a un [Proyecto](/es/resources/fundamentals/projects) para la autenticación. Los endpoints de la API de X v1.1 pueden usar credenciales de Apps independientes o de Apps asociadas a un Proyecto.

<div id="response-data-format">
  #### Formato de los datos de respuesta
</div>

Una diferencia significativa entre las versiones de endpoints estándar v1.1 y API de X v2 es cómo se seleccionan los campos en la carga útil.

En los endpoints estándar, muchos campos de respuesta se incluyen de forma predeterminada, con opciones para usar parámetros que especifiquen campos adicionales.

Sin embargo, la API de X v2 solo entrega de forma predeterminada los campos `id` y `text` de la Publicación. Los campos y objetos adicionales requieren el uso de los parámetros [fields](/es/x-api/fundamentals/fields) y [expansions](/es/x-api/fundamentals/expansions). Los campos expandidos se devuelven en un objeto `includes` dentro de la respuesta, que puede vincularse con el objeto principal de la Publicación haciendo coincidir los ID.

Para obtener más información sobre el uso de fields y expansions, consulta la [guía sobre cómo usar fields y expansions](/es/x-api/fundamentals/data-dictionary). Una [guía de migración de formatos de datos](/es/x-api/fundamentals/fields) también mapea los campos estándar de v1.1 a los nuevos campos de v2.

Además, la API de X v2 introduce nuevos diseños JSON para objetos, incluidos los objetos de Publicación y [user](/es/x-api/fundamentals/data-dictionary#user):

- Los endpoints estándar devuelven objetos de Publicación en un arreglo `statuses`, mientras que la API de X v2 usa un arreglo `data`.
- En la API de X v2, la terminología “statuses” se reemplaza por Retweets y Tweets citados.
- Nueva terminología como `like` reemplaza términos como `favorites` y `favourites`.
- Los atributos sin valores (p. ej., `null`) no se incluyen en las cargas útiles de la API de X v2.

El objeto de Publicación en la API de X v2 incluye nuevos campos como:

- `conversation_id`
- Dos nuevos campos de [annotations](/es/x-api/fundamentals/post-annotations) (`context` y `entities`)
- Nuevos campos de [metrics](/es/x-api/fundamentals/metrics)
- Campo `reply_setting` que indica quién puede responder a una Publicación determinada

<div id="request-parameters">
  #### Parámetros de solicitud
</div>

Los siguientes parámetros estándar de v1.1 tienen equivalentes en la API de X v2:

| Estándar | API de X v2 |
|:----------|:----------|
| `id`     | `ids`    |

Ciertos parámetros estándar de v1.1 **no** son compatibles en la API de X v2:

| Estándar            | Comentario                                                                                                                     |
|:---------------------|:------------------------------------------------------------------------------------------------------------------------------|
| `tweet_mode`        | Sustituido por la funcionalidad de campos y expansiones.                                                                      |
| `trim_user`         | Sustituido por campos y expansiones. Use la expansión `author_id` y `user.fields` para obtener datos de usuario.             |
| `include_my_retweet`| Proporciona el ID de la publicación de origen para publicaciones retuiteadas por el usuario autenticado.                      |
| `include_entities`  | Use campos y expansiones para controlar las entidades en la carga útil.                                                       |
| `include_ext_alt_text` | Agrega el campo `ext_alt_text` en la entidad de medios si hay texto alternativo.                                           |
| `include_card_uri`  | Agrega `card_uri` cuando se adjunta una tarjeta de anuncios.                                                                  |
| `map`               | Devuelve el ID de la publicación y el mensaje de error para publicaciones no disponibles en la API de X v2, en lugar de campos anulados en v1.1. |

<div id="curl-requests">
  ### Solicitudes cURL
</div>

Las siguientes solicitudes cURL muestran endpoints estándar de v1.1 y sus equivalentes en v2. Reemplace `ACCESS_TOKEN` en el encabezado con el token de acceso de su app. Para los endpoints de v2, el token debe pertenecer a una [App de desarrollador](/es/resources/fundamentals/developer-apps/overview) dentro de un [Proyecto](/es/resources/fundamentals/projects/overview).

Las cargas de respuesta de v1.1 diferirán de las de v2. Con v2, puede solicitar distintos campos con los parámetros [fields](/es/x-api/fundamentals/fields) y [expansions](/es/x-api/fundamentals/expansions).

**Endpoints estándar de v1.1 `GET statuses/lookup` y de v2 `GET /tweets`**

```bash
curl --request GET \
  --url 'https://api.x.com/1.1/statuses/lookup.json?id=1460323737035677698%2C1460323743339741184' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```

```
  curl --request GET \
  --url 'https://api.x.com/2/tweets?ids=1460323737035677698%2C1460323743339741184&tweet.fields=created_at&expansions=author_id&user.fields=created_at' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```

**Endpoints estándar de v1.1 `GET statuses/show/:id` y de v2 `GET /tweets/:id`**

```
curl --request GET \
  --url 'https://api.x.com/1.1/statuses/show.json?id=1460323737035677698' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```

```
curl --request GET \
  --url 'https://api.x.com/2/tweets/1460323737035677698?tweet.fields=created_at&expansions=author_id&user.fields=created_at' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```
