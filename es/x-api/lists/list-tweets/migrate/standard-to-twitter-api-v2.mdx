<div id="list-posts-lookup-standard-v11-compared-to-x-api-v2">
  ### Consulta de publicaciones de Listas: estándar v1.1 en comparación con la API de X v2
</div>

Si ha estado trabajando con el endpoint estándar v1.1 [GET lists/statuses](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-statuses), el objetivo de esta guía es ayudarle a comprender las similitudes y diferencias entre los endpoints del estándar v1.1 y de la API de X v2.

- **Similitudes**
  - Métodos de autenticación
  - Límites de solicitudes
- **Diferencias**
  - URL de endpoints
  - Requisitos de Apps y Proyectos
  - Límite de objetos de datos por solicitud
  - Formatos de datos de respuesta
  - Parámetros de solicitud

<div id="similarities">
  #### Similitudes
</div>

**Autenticación**

Ambas versiones del endpoint admiten [OAuth 1.0a User Context](/es/resources/fundamentals/authentication#oauth-2-0). Por lo tanto, si antes utilizabas uno de los endpoints estándar de consulta de publicaciones de Listas v1.1, puedes seguir usando el mismo método de autenticación si migras a la versión de la API de X v2.

Según la biblioteca o paquete de autenticación que elijas, la autenticación de solo App probablemente sea la forma más sencilla de empezar y puede configurarse con un encabezado de solicitud simple. Para aprender a generar un token de acceso de solo App, consulta [esta guía de App only](/es/resources/fundamentals/authentication#bearer-token-also-known-as-app-only).

**Límites de frecuencia**

|     |     |
| :--- | :--- |
| **Standard v1.1** | **API de X v2** |
| /1.1/lists/statuses.json<br /><br />900 solicitudes por ventana de 15 minutos con OAuth 1.0a User Context<br /><br />900 solicitudes por ventana de 15 minutos con App only | /2/lists/:id/tweets<br /><br />900 solicitudes por ventana de 15 minutos con OAuth 1.0a User Context<br /><br />900 solicitudes por ventana de 15 minutos con OAuth 2.0 Authorization Code with PKCE<br /><br />900 solicitudes por ventana de 15 minutos con App only |

<div id="differences">
  #### Diferencias
</div>

**URLs de endpoint**

- Endpoints estándar v1.1:
  - GET https://api.x.com/1.1/lists/statuses.json
    (Buscar tuits de una Lista específica)
- Endpoint de la API de X v2:
  - GET https://api.x.com/2/lists/:id/tweets
    (Buscar tuits de una Lista específica)

**Requisitos de App y Proyecto**

Los endpoints de la API de X v2 requieren que utilices credenciales de una [App de desarrollador](/es/resources/fundamentals/developer-apps) asociada a un [Proyecto](/es/resources/fundamentals/projects) al autenticar tus solicitudes. Todos los endpoints de la API de X v1.1 pueden usar credenciales de Apps independientes o Apps asociadas a un proyecto.

**Límites de objetos de datos por solicitud**

El endpoint estándar v1.1 /lists/statuses permite devolver hasta 5000 Publicaciones por solicitud. Los nuevos endpoints v2 permiten devolver hasta 100 Publicaciones por solicitud. De forma predeterminada, se devolverán 100 objetos de usuario; para cambiar la cantidad de resultados, debes pasar el parámetro de consulta max\_results= con un número entre 1 y 100; luego puedes pasar el next\_token devuelto en el cuerpo de la respuesta al parámetro de consulta pagination\_token en tu siguiente solicitud.

**Formato de datos de respuesta**

Una de las mayores diferencias entre las versiones de endpoints estándar v1.1 y de la API de X v2 es cómo seleccionas qué campos se devuelven en la carga.

En los endpoints estándar, recibes muchos de los campos de respuesta de forma predeterminada y luego tienes la opción de usar parámetros para identificar qué campos adicionales o conjuntos de campos deben devolverse en la carga.

La versión de la API de X v2 solo entrega, por defecto, los campos id y text de la Publicación. Para solicitar campos u objetos adicionales, debes usar los parámetros [fields](/es/x-api/fundamentals/fields) y [expansions](/es/x-api/fundamentals/expansions). Cualquier campo de la Publicación que solicites desde este endpoint se devolverá en el objeto principal de la Publicación. Cualquier campo de objeto expandido se devolverá en un objeto includes dentro de tu respuesta. Luego puedes volver a asociar cualquier objeto expandido con el objeto principal de la Publicación haciendo coincidir los IDs del objeto principal y de los objetos expandidos. 

Aquí hay ejemplos de posibles campos y expansiones de Publicación:

- attachments

- author\_id

- context\_annotations

- created\_at

- geo

- lang

|     |     |
| :--- | :--- |
| **Endpoint** | **Expansión** |
| /2/lists/:id/tweets | author\_id |

Te recomendamos leer más sobre estos nuevos parámetros en sus respectivas guías, o consultar nuestra guía sobre [cómo usar fields y expansions](/es/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions). 

También hemos preparado una [guía de migración de formato de datos](/es/x-api/migrate/data-format-migration) que puede ayudarte a mapear campos estándar v1.1 a los campos más recientes de v2. Esta guía también te proporcionará la expansión específica y el parámetro de campo que deberás pasar con tu solicitud v2 para devolver campos específicos. 

Además de los cambios en cómo solicitas ciertos campos, la API de X v2 también introduce nuevos diseños JSON para los objetos devueltos por las APIs, incluidos los objetos de [Publicación](/es/x-api/fundamentals/data-dictionary#tweet) y [usuario](/es/x-api/fundamentals/data-dictionary#user).

- En el nivel raíz de JSON, los endpoints estándar devuelven objetos de Publicación en un arreglo **statuses**, mientras que la API de X v2 devuelve un arreglo **data**. 

- En lugar de referirse a “statuses” Retweeted y Quoted, el JSON de la API de X v2 se refiere a tuits Retweeted y Quoted. Muchos campos heredados y en desuso, como **contributors** y **user.translator\_type**, se están eliminando. 

- En lugar de usar tanto **favorites** (en el objeto de Publicación) como **favourites** (en el objeto de usuario), la API de X v2 utiliza el término **like**.

- X adopta la convención de no incluir en la carga los valores JSON vacíos (por ejemplo, **null**). Los atributos de Publicación y de usuario solo se incluyen si tienen valores no nulos.
   

**Parámetros de solicitud**

Los siguientes parámetros de solicitud estándar v1.1 tienen equivalentes en la API de X v2:

|     |     |
| :--- | :--- |
| Estándar v1.1 | API de X v2 |
| list\_id | id  |
| slug | Sin equivalente |
| owner\_screen\_name | Sin equivalente |
| owner\_id | Se solicita con el parámetro expansions con el valor author\_id |
| since\_id | Sin equivalente |
| max\_id | Sin equivalente |
| include\_entities | Se solicita con el parámetro tweet.fields con el valor entities |
| include\_rts | Sin equivalente |
| count | max\_results |
