<div id="mutes-lookup-standard-v11-compared-to-x-api-v2">
  ### Consulta de silenciados: Standard v1.1 frente a API de X v2
</div>

Si ha estado trabajando con los endpoints estándar de v1.1 [GET mutes/users/ids](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/mute-block-report-users/api-reference/get-mutes-users-ids) y [GET mutes/users/list](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/mute-block-report-users/api-reference/get-mutes-users-list), el objetivo de esta guía es ayudarle a comprender las similitudes y diferencias entre los endpoints de consulta de silenciados de Standard v1.1 y la API de X v2.

- **Similitudes**
  - Autenticación
- **Diferencias**
  - URL de los endpoints

  - Límite de usuarios por solicitud

  - Requisitos de App y Proyecto

  - Formatos de datos de la respuesta

  - Parámetros de la solicitud

<div id="similarities">
  #### Similitudes
</div>

**Autenticación**

Tanto los endpoints de consulta de silenciamientos de la versión estándar v1.1 como los de la API de X v2 utilizan [OAuth 1.0a User Context](/es/resources/fundamentals/authentication#oauth-1-0a-2). Por lo tanto, si antes usabas uno de los endpoints de consulta de silenciamientos de la versión estándar v1.1, puedes seguir utilizando el mismo método de autenticación si migras a la versión de la API de X v2. 

<div id="differences">
  #### Diferencias
</div>

**URLs de endpoints**

- Endpoints estándar v1.1:
  - GET https://api.x.com/1.1/mutes/users/ids.json
    (lista de IDs de usuarios que el usuario especificado silenció)
  - GET https://api.x.com/1.1/mutes/users/lists.json
    (lista de usuarios silenciados por el usuario especificado)
- Endpoint de la API de X v2:
  - GET https://api.x.com/2/users/:id/muting
    (lista de usuarios silenciados por el ID de usuario especificado)
     

**Límites de usuarios por solicitud**

Los endpoints estándar v1.1 permiten devolver hasta 5000 usuarios por solicitud. Los nuevos endpoints v2 permiten devolver hasta 1000 usuarios por solicitud. Para devolver el máximo de 1000 usuarios, deberá pasar max\_results=1000 como parámetro de consulta; luego puede pasar el next\_token devuelto en el cuerpo de la respuesta al parámetro de consulta pagination\_token en su siguiente solicitud.
 

**Requisitos de App y Proyecto**

Los endpoints de la API de X v2 requieren que use credenciales de una [App de desarrollador](/es/resources/fundamentals/developer-apps) asociada a un [Proyecto](/es/resources/fundamentals/developer-apps) al autenticar sus solicitudes. Todos los endpoints de la API de X v1.1 pueden usar credenciales de Apps independientes o Apps asociadas a un proyecto.

**Formato de datos de la respuesta**

Una de las mayores diferencias entre las versiones de endpoints estándar v1.1 y la API de X v2 es cómo selecciona qué campos se devuelven en la carga útil.

En los endpoints estándar, recibe muchos de los campos de la respuesta de forma predeterminada y luego tiene la opción de usar parámetros para identificar qué campos o conjuntos de campos deben incluirse en la carga útil.

La versión de la API de X v2 solo entrega de forma predeterminada los campos id de usuario, name y username. Para solicitar campos u objetos adicionales, deberá usar los parámetros [fields](/es/x-api/fundamentals/fields) y [expansions](/es/x-api/fundamentals/expansions). Cualquier campo de usuario que solicite desde este endpoint se devolverá en el objeto de usuario principal. Cualquier objeto de Post expandido y sus campos se devolverán en un objeto includes dentro de su respuesta. Luego puede asociar cualquier objeto expandido con el objeto de usuario haciendo coincidir los IDs presentes tanto en el objeto de usuario como en el objeto de Post expandido. 

Le recomendamos leer más sobre estos nuevos parámetros en sus respectivas guías o en nuestra guía sobre [cómo usar fields y expansions](/es/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions). 

También hemos preparado una [guía de migración de formato de datos](/es/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2) que puede ayudarle a mapear los campos estándar v1.1 a los campos más recientes de v2. Esta guía también le proporcionará el parámetro específico de expansión y de campo que deberá pasar con su solicitud v2 para devolver campos específicos. 
 

Además de los cambios en cómo solicita ciertos campos, la API de X v2 también introduce nuevos diseños JSON para los objetos devueltos por las APIs, incluidos los objetos [Post](/es/x-api/fundamentals/data-dictionary#tweet) y [user](/es/x-api/fundamentals/data-dictionary#user).

- En el nivel raíz de JSON, los endpoints estándar devuelven objetos Post en un arreglo statuses, mientras que la API de X v2 devuelve un arreglo data. 
- En lugar de referirse a “statuses” retuiteados y citados, el JSON de la API de X v2 se refiere a Tweets retuiteados y citados. Muchos campos heredados y obsoletos, como contributors y user.translator\_type, se están eliminando. 
- En lugar de usar tanto favorites (en el objeto Post) como favourites (en el objeto user), la API de X v2 usa el término like. 
- X adopta la convención de que los valores JSON sin valor (por ejemplo, null) no se escriben en la carga útil. Los atributos de Post y user solo se incluyen si tienen valores distintos de null. 
   

También introdujimos un nuevo conjunto de campos en el [objeto Post](/es/x-api/fundamentals/data-dictionary#tweet), incluidos los siguientes:

- Un campo [conversation\_id](/es/x-api/fundamentals/conversation-id)
- Dos campos nuevos de [annotations](/es/x-api/fundamentals/post-annotations), incluidos context y entities
- Varios campos nuevos de [metrics](/es/x-api/fundamentals/metrics) 
- Un nuevo campo reply\_setting, que muestra quién puede responder a una determinada publicación

**Parámetros de solicitud**

Los siguientes parámetros de solicitud estándar v1.1 tienen equivalentes en la API de X v2:

| **Estándar** | **API de X v2** |
| :--- | :--- |
| stringify\_ids | Sin equivalente |
| cursor | pagination\_token |
| skip\_status | Sin equivalente |

También hay un conjunto de parámetros estándar de solicitud de consulta de silenciamientos de la v1.1 que **no** están disponibles en la API de X v2:

| Estándar | Comentario |
| :--- | :--- |
| include\_entities | Este parámetro se usa para quitar el nodo entities del payload de la publicación. Se ha reemplazado por la funcionalidad de campos aditivos y expansiones. |
