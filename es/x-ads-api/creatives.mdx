<div id="overview">
  ## Descripción general
</div>

Los creativos son cualquier entidad que se puede promocionar en una campaña. Las publicaciones pueden incluir texto, imágenes, GIF, videos o tarjetas. Las tarjetas pueden incluir imágenes o videos.

Los creativos de imagen, GIF o video se cargan usando [POST media/upload](/es/x-api/media/upload-media#post-media-upload), un endpoint de carga simple que solo admite imágenes, o los endpoints POST media/upload (en fragmentos). Estos se pueden agregar a

tarjetas:

- POST accounts/:account\_id/cards
  Tweets:
- POST accounts/:account\_id/tweets - Para agregar tarjetas a los Tweets, use el parámetro card\_uri.
  Tweets programados:
- POST accounts/:account\_id/scheduled\_tweets

Para obtener más detalles sobre las tarjetas, consulte la página Cards. La página Promoted Video proporciona detalles sobre cómo asociar videos con tarjetas o Tweets.

<div id="cards">
  ### Tarjetas
</div>

La API de Anuncios de X admite varios tipos de tarjetas que pueden usarse en tuits, los cuales luego pueden promocionarse en campañas. Nota: una vez publicado el tuit, los detalles de la tarjeta son visibles públicamente. Esto puede incluir información sobre el usuario propietario de la tarjeta.

<div id="image">
  #### Imagen
</div>

Las siguientes especificaciones de imagen se aplican a los recursos usados en [Cards](/es/x-ads-api/creatives#cards-2). Las imágenes deben tener un tamaño de 3 MB o menos y un ancho mínimo de 800 px. Además, admitimos las siguientes relaciones de aspecto (ancho:alto).

- Sitio web: 1:1 y 1.91:1
- Descarga de app con imagen: 1:1 y 1.91:1
- Encuesta: 1.91:1
- Conversación con imagen: 1.91:1
- Mensaje directo con imagen: 1.91:1

Admitimos los siguientes formatos de imagen: .bmp, .jpeg y .png.

<div id="video">
  #### Video
</div>

Las siguientes especificaciones de video se aplican a los recursos usados en [tarjetas](/es/x-ads-api/creatives#cards-2). Admitimos las siguientes relaciones de aspecto (ancho:alto).

- Video de sitio web: 16:9 y 1:1
- Video de descarga de apps: 16:9 y 1:1
- Encuesta: 16:9
- Conversación en video: 16:9
- Mensaje directo en video: 16:9

<div id="promoted-video">
  ### Video promocionado
</div>

Este documento ofrece una breve descripción del proceso para subir y promocionar video a través de la API de Anuncios.

La API de Anuncios admite video promocionado en [Tweets](/es/x-ads-api/creatives#tweets-2) y en las siguientes tarjetas:

- [Video para sitio web](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [Descarga de app en video](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [Conversación en video](/es/x-ads-api/creatives#video-conversation-cards)

Primero, sube el video usando el endpoint [POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html). Con el `media_id`, asocia el video con una cuenta de anuncios usando el endpoint POST accounts/:account\_id/videos. El `id` del video, a veces denominado `media_key`, se utilizará en solicitudes posteriores. Es una cadena que comienza con un entero, seguida de un guion bajo y que termina con un valor largo. Por ejemplo: `13_875943225764098048`.

<div id="promoted-video-in-tweets">
  #### Video promocionado en Tweets
</div>

Para crear un Tweet, utiliza el endpoint [POST accounts/:account\_id/tweet](/es/x-ads-api/creatives#get-accounts-account-id-tweets) junto con el `id` del video. En este paso, también puedes proporcionar un título del video, una descripción y una llamada a la acción (CTA). Estos valores son visibles para el usuario.

<div id="promoted-video-in-cards">
  #### Video promocionado en tarjetas
</div>

Las tarjetas de Descarga de app de video y de Conversación en video admiten agregar una imagen de portada. Sube una imagen para usar en estas tarjetas mediante el endpoint [POST media/upload](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload.html).

Crea la tarjeta usando uno de los siguientes endpoints:

- [POST accounts/:account\_id/cards/video\_website](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [POST accounts/:account\_id/cards/video\_app\_download](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [POST accounts/:account\_id/cards/video\_conversation](https://devcommunity.x.com/t/ads-api-version-11/168814)

usando el `id` del video y, opcionalmente, el `media_id` de la imagen (para la imagen de portada).

Por último, crea el Post usando el endpoint POST accounts/:account\_id/tweet. Las tarjetas se adjuntan a los Posts mediante el parámetro `card_uri`.

<div id="general-information">
  #### Información general
</div>

Para obtener instrucciones detalladas sobre la carga de videos mediante la API, consulta la [Guía de carga de video](https://developer.x.com/en/docs/media/upload-media/uploading-media.html).

Los videos también pueden promoverse como recursos pre‑roll. Consulta la [Guía del objetivo de vistas de video pre‑roll](/es/x-ads-api/campaign-management#video-views-preroll-objective) para una explicación detallada.

- (Desde 2015-10-22) Al cargar videos para usar en contenido promocionado, el parámetro `media_category` debe configurarse con el valor `amplify_video` en todas las solicitudes del comando `INIT` al endpoint [POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html). Usar este parámetro garantiza que el video se preprocese de forma asíncrona y quede listo para su uso en contenido promocionado. El comando `STATUS` puede usarse para verificar la finalización del procesamiento asíncrono después de cargar el video.
- La duración máxima actualmente permitida para videos promocionados es de 10 minutos, con un tamaño de archivo de 500 MB o menos.
- El video cargado debe ser MP4 o MOV.
- El video cargado generalmente se procesa rápido, pero los tiempos de procesamiento pueden variar según la duración del video y el tamaño del archivo.
- Las imágenes de póster cargadas deben estar en formato PNG o JPG. No hay requisitos de relación de aspecto ni de tamaño, pero la imagen de póster se ajustará para adaptarse al reproductor de video.

<div id="scheduled-tweets">
  ### Tuits programados
</div>

<div id="introduction">
  #### Introducción
</div>

Los Tweets programados permiten a un anunciante o usuario crear un Tweet que puede programarse para publicarse en una fecha posterior. Además de crear y gestionar estos Tweets, la API permite asociarlos a un line item para que se promocionen una vez que el Tweet esté activo. Esto permite a los anunciantes preparar Tweets nativos y planificar con antelación las creatividades de sus campañas para iniciativas clave. Por ejemplo, preparar una creatividad de Tweet para que se publique inmediatamente tras el anuncio de un nuevo producto.

A continuación se enumera el conjunto completo de funcionalidades que ofrecen los endpoints de la API de Tweets programados:

- Crear, modificar y ver Tweets recién programados
- Asociar un Tweet programado a un line item
- Consultar y gestionar Tweets programados existentes
- Una vez que un Tweet programado esté activo, recuperar el `id` del Tweet publicado

<div id="api-endpoints">
  #### Endpoints de la API
</div>

A continuación se muestra el conjunto completo de endpoints relacionados con la funcionalidad anterior:

<div id="scheduled-tweet-management">
  #### Gestión de tuits programados
</div>

- [GET accounts/:account\_id/scheduled\_tweets](/es/x-ads-api/creatives#get-accounts-account-id-scheduled-tweets) (obtener la lista de todos los tuits programados)
- [GET accounts/:account\_id/scheduled\_tweets/:scheduled\_tweet\_id](/es/x-ads-api/creatives#get-accounts-account-id-scheduled-tweets) (consultar un tuit programado específico usando su `id`)
- [POST accounts/:account\_id/scheduled\_tweets](/es/x-ads-api/creatives#post-accounts-account-id-scheduled-tweets) (crear un nuevo tuit programado)
- [PUT accounts/:account\_id/scheduled\_tweets/:scheduled\_tweet\_id](/es/x-ads-api/creatives#example-request-39) (modificar un tuit programado existente)
- [DELETE accounts/:account\_id/scheduled\_tweets/:scheduled\_tweet\_id](/es/x-ads-api/creatives#example-request-40) (eliminar un tuit programado usando su `id`)
- [GET accounts/:account\_id/scheduled\_tweets/preview/:scheduled\_tweet\_id](/es/x-ads-api/creatives#scheduled-tweets-2) (previsualizar un tuit programado existente)

<div id="scheduled-promoted-tweets">
  #### Tweets promocionados programados
</div>

- [GET accounts/:account\_id/scheduled\_promoted\_tweets](/es/x-ads-api/campaign-management#get-accounts-account-id-scheduled-promoted-tweets) (obtener la lista de todos los Tweets promocionados programados)
- [GET accounts/:account\_id/scheduled\_promoted\_tweets/:scheduled\_promoted\_tweet\_id](/es/x-ads-api/campaign-management#get-accounts-account-id-scheduled-promoted-tweets-scheduled-promoted-tweet-id) (consultar un Tweet promocionado programado mediante su `id`)
- [POST accounts/:account\_id/scheduled\_promoted\_tweets](/es/x-ads-api/campaign-management#post-accounts-account-id-scheduled-promoted-tweets) (crear un nuevo Tweet promocionado programado)
- [DELETE accounts/:account\_id/scheduled\_promoted\_tweets/:scheduled\_promoted\_tweet\_id](/es/x-ads-api/campaign-management#delete-accounts-account-id-scheduled-promoted-tweets-scheduled-promoted-tweet-id) (eliminar un Tweet promocionado programado existente mediante su `id`)

<div id="scheduled-tweet-view">
  #### Vista de tuit programado
</div>

- [GET accounts/:account\_id/scheduled\_tweets/preview/:scheduled\_tweet\_id](/es/x-ads-api/creatives#scheduled-tweets-2) (ver un tuit programado existente)

Dado que los tuits programados son entidades separadas de los tuits “en vivo”, se ejecutan dos conjuntos distintos de validaciones al crear o editar estos tuits. El primer conjunto de reglas de validación se ejecuta durante el paso de creación del tuit programado, específicamente:

<div id="scheduled-tweet-create">
  #### Creación de Tweet programado:
</div>

- Validar que el usuario autenticado tenga acceso para crear Tweets orgánicos para un @handle dado. Los privilegios para crear Tweets Solo Promocionados requieren que el usuario autenticado sea un usuario de la cuenta con [permisos del compositor de Tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access).
- Validar que no haya más de 30 Tweets programados para crearse dentro de una ventana de 15 minutos respecto al tiempo scheduled\_at. Un mensaje de error SCHEDULED\_TWEET\_LIMIT\_EXCEEDED indica que se han programado demasiados Tweets dentro del mismo intervalo futuro de 15 minutos. Los anunciantes deberán eliminar un Tweet programado existente o mover el tiempo scheduled\_at hacia antes o hacia después.

<div id="scheduled-tweet-goes-live">
  #### El Tweet programado pasa a estar “publicado”:
</div>

- Estas reglas de validación se ejecutan en el momento scheduled\_at y son idénticas a las aplicadas en la creación normal de un Tweet en la API. Por ejemplo, un Tweet programado no se publicará y el scheduled\_status se establecerá en FAILED si el Tweet programado contiene tanto una imagen como un GIF.

<div id="workflow">
  #### Flujo de trabajo
</div>

**Crear un nuevo Tweet programado**

Se puede crear un nuevo Tweet programado usando el endpoint [POST accounts/:account\_id/scheduled\_tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets). Este endpoint tiene los siguientes parámetros obligatorios: la hora `scheduled_at`, junto con el `text` del Tweet si no se incluyen entidades multimedia en el Tweet. Además, este endpoint proporciona algunas opciones adicionales que permiten crear un Tweet programado en nombre de otro @handle mediante el parámetro `as_user_id`, así como agregar una tarjeta (`card_uri`) y cualquier medio (`media_ids`). Tenga en cuenta que un Tweet solo puede contener entidades del mismo tipo; es decir, Video, GIF o Imagen. El parámetro `nullcast` controla si el Tweet es “solo promocionado” o no. Todos los Tweets programados recién creados son “solo promocionados” (`nullcast=true`) de forma predeterminada. Si `nullcast=false`, entonces se crea un Tweet orgánico programado.

Una vez que se crea correctamente un Tweet programado, la respuesta contendrá un campo `id`, que se refiere al identificador único del propio Tweet programado. Además de este campo, también se devuelve otro campo llamado `tweet_id`. Este campo es `null` inicialmente; sin embargo, una vez que el Tweet se publique, este campo se completará con el ID del Tweet “en vivo”.

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_tweets" -d 'scheduled_at=2017-12-24T23:59:00Z&text=Happy Holidays!'
```

Esto creará el siguiente Tweet programado:

```json
{
  "request": {
    "params": {
      "text": "Happy Holidays!",
      "scheduled_at": "2017-12-24T23:59:00Z"
    }
  },
  "data": {
    "completed_at": null,
    "id_str": "917507899668099072",
    "text": "Happy Holidays!",
    "user_id": "3271358660",
    "scheduled_status": "SCHEDULED",
    "id": 917507899668099100,
    "nullcast": true,
    "created_at": "2017-10-09T21:51:44Z",
    "scheduled_at": "2017-12-24T23:59:00Z",
    "card_uri": null,
    "updated_at": "2017-10-09T21:51:44Z",
    "tweet_id": null,
    "media_keys": []
  }
}
```

Una vez que este Tweet programado esté en vivo, el campo `tweet_id` se completará con el ID del Tweet “en vivo”.

**Asociar un Tuit programado con un elemento de línea**

Si bien los Tuits programados se pueden usar para crear Tuits orgánicos, también permitimos a los socios crear un Tuit “Solo promocionado” (`nullcast=true`), y cualquiera de ellos puede asociarse con un elemento de línea. Para facilitar esto, también proporcionamos el endpoint [POST accounts/:account\_id/scheduled\_promoted\_tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets). Este endpoint solo permite asociar un único Tuit programado promocionado con un elemento de línea en una sola llamada a la API. Para asociar varios Tuits programados al mismo elemento de línea, son necesarias múltiples llamadas a la API.

Tenga en cuenta que no es posible modificar un Tuit programado promocionado existente.

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_promoted_tweets" -d 'line_item_id=a44qc&scheduled_tweet_id=917507899668099072'
```

```json
{
  "data": {
    "line_item_id": "a44qc",
    "id": "26576",
    "created_at": "2017-10-09T22:24:16Z",
    "updated_at": "2017-10-09T22:24:16Z",
    "scheduled_tweet_id": "917507899668099072",
    "tweet_id": null,
    "deleted": false
  },
  "request": {
    "params": {
      "line_item_id": "a44qc",
      "scheduled_tweet_id": 917507899668099100,
      "account_id": "aaaaa"
    }
  }
}
```

Este endpoint solo crea una asociación entre un Tuit programado y un elemento de línea. Una vez que las fechas de vuelo de la campaña/elemento de línea estén vigentes, el elemento de línea comenzará automáticamente a publicar el Tuit “en vivo” correspondiente. Si bien en este paso verificamos que el Tuit programado esté en el estado `SCHEDULED` y que el Tuit programado indicado sea válido para el objetivo previsto, no se ejecutan otras validaciones. Cualquier regla de validación restante que aplique al elemento de línea y al Tuit programado se ejecutará cuando el Tuit pase a estar “en vivo”.

Para garantizar que no haya problemas con la entrega de la campaña, se recomienda que el Tuit programado tenga `scheduled_at` en un momento anterior a las fechas de vuelo de la campaña/elemento de línea.

Por ejemplo, supongamos que el Tuit programado está configurado para entrar en vivo después de la fecha de inicio de la campaña (y que solo hay un único Tuit asociado con un único elemento de línea); entonces la campaña estará `ACTIVE`. Sin embargo, dado que el Tuit programado aún no está en vivo, no habrá creatividades disponibles para su entrega.

**Administración de Tuits programados**

Los conjuntos restantes de endpoints permiten a los consumidores de la API administrar todos sus Tuits programados y Tuits programados promocionados. Estas API se pueden usar para devolver una lista de todos los Tuits programados, opcionalmente filtrados por un estado dado, así como para buscar un Tuit programado por su `id`.

<div id="what-happens-when-a-scheduled-tweet-goes-live">
  #### ¿Qué ocurre cuando un Tuit programado se publica?
</div>

Una vez que un Tuit programado está a punto de publicarse, es decir, en el momento `scheduled_at`, se realizan las siguientes actualizaciones:

- Se crea el Tuit “publicado”; sin embargo, puede haber una latencia de hasta 1 segundo
- Se añade el `tweet_id` a las siguientes entidades:
- Tuit programado
- Tuit programado promocionado
- Se crea una nueva entidad de Tuit promocionado

<div id="best-practices">
  #### Mejores prácticas
</div>

Se recomiendan las siguientes mejores prácticas al crear o promocionar tuits programados:

- Asegúrate de que el tuit sea válido al crear el tuit programado (por ejemplo, un tuit solo puede tener una imagen, un video o un GIF, y no una combinación de los tres)
- Asegúrate de que las fechas de vuelo de la campaña (es decir, `start_time` y `end_time`) se alineen con la hora `scheduled_at` del tuit programado
- Los tuits programados no deben programarse con más de un año de antelación (365 días)
- Actualmente no se admite la vista previa de tuits programados (es decir, la posibilidad de previsualizarlos antes de crearlos)

<div id="media-library">
  ### Biblioteca multimedia
</div>

<div id="introduction">
  #### Introducción
</div>

Los endpoints de la Biblioteca de medios permiten gestionar imágenes, GIF y videos para cuentas de anuncios en X. Los recursos multimedia de la biblioteca pueden usarse en publicaciones (Tweets) y para crear tarjetas. También pueden reutilizarse en múltiples [creatives](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/overview), lo que elimina la necesidad de cargar el mismo recurso varias veces.

<div id="api-endpoints">
  #### Endpoints de la API
</div>

- [POST media/upload](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload#post-media-upload) o [POST media/upload (chunked)](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload-init#post-media-upload-init) (subir contenido multimedia)
- [POST accounts/:account\_id/media\_library](https://developer.x.com/x-ads-api/creatives#get-accounts-account-id-media-library#post-accounts-account-id-media-library) (añadir contenido multimedia a la Biblioteca multimedia)

<div id="adding-to-the-library">
  #### Agregar a la biblioteca
</div>

Agregar contenido multimedia a la biblioteca es un proceso de dos pasos. <div id="request-parameters">
  #### Parámetros de la solicitud
</div>

Todas las solicitudes POST de Media Library requieren un identificador de medios. Este valor se devuelve durante el paso de carga. Al usar media\_id, como en el ejemplo anterior, también se debe especificar una media\_category. Hay cuatro valores posibles para la categoría: AMPLIFY\_VIDEO, TWEET\_GIF, TWEET\_IMAGE y TWEET\_VIDEO.

De forma opcional, se pueden establecer los valores name y file\_name para los objetos en la Media Library. Estos atributos ayudan a los usuarios a distinguir entre variantes de medios en la biblioteca.

Para los videos, también es posible establecer un título y una descripción. Estos valores están pensados para pasarse como los parámetros de solicitud video\_title y video\_description con el endpoint [POST accounts/:account\_id/tweet](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet). En el tuit, este texto aparece debajo del video.

<div id="attributes">
  #### Atributos
</div>

La Biblioteca de medios introduce formalmente el concepto de media\_key. Este es el identificador único de los objetos en la biblioteca. Las claves de medios son valores de cadena con el siguiente formato: 13\_875943225764098048. Son totalmente compatibles en todos nuestros endpoints de tarjetas.

Además, la respuesta de la Biblioteca de medios incluye el media\_id, representado como cadena. Se incluye para los recursos que actualmente no aceptan una media\_key: [Tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet)\*, [Vista previa de Tweet](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#get-accounts-account-id-tweet-preview)\* y [Tweets programados](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets). Estamos trabajando para admitir media\_keys en todas partes.

El atributo aspect\_ratio se devuelve para GIF y videos. Se puede usar para filtrar medios destinados a tarjetas que solo aceptan relaciones de aspecto específicas.

\*Estos endpoints admiten el parámetro video\_id, que es una media\_key.

<div id="introduction">

| **ID de medio** | **Clave de medio** |
| :--- | :--- |
| 1029825579531807971 | 13\_1029825579531807971 |

La clave de medio es el ID con un prefijo numérico y un guion bajo.

<div id="images">

</div>

La siguiente tabla muestra los tipos de identificadores disponibles actualmente en la respuesta de cada recurso relacionado con imágenes, así como los nombres de atributo correspondientes.

| **Recurso** | **Identificador** | **Atributo(s)** |

<div id="media-library">
  #### Biblioteca de medios
</div>

A veces es necesario obtener información adicional sobre un recurso multimedia. Un caso de uso, para tarjetas de video, es recuperar la URL de MP4 en lugar de la de VMAP. Esto está disponible en la Biblioteca de medios. Para obtener detalles sobre la información disponible, consulta nuestra [Guía de la Biblioteca de medios](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/guides/media-library). La mayoría de los recursos pertenecientes al usuario promocionable FULL de la cuenta de anuncios se pueden encontrar en la biblioteca. Sin embargo, hay algunas excepciones.

**Obtención de medios**

Como se indicó arriba, las tarjetas de imagen no contienen referencias ni a media IDs ni a media keys. Por lo tanto, no es posible obtener sus recursos a través de la Biblioteca de medios. Esto también aplica a las imágenes de [Account Media](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/account-media#account-media).

Las tarjetas de video requieren que el recurso de video forme parte de la Biblioteca de medios (o del recurso Videos anterior) antes de crearlas. Como resultado, estos recursos siempre podrán recuperarse en la Biblioteca de medios. Esto también aplica a los recursos PREROLL de Account Media.

Por último, los medios en Tweets siempre están garantizados en la Biblioteca de medios.

La siguiente tabla resume qué recursos se pueden recuperar en la Biblioteca de medios, teniendo en cuenta si la respuesta del recurso incluye un identificador para usar en la búsqueda.

| **Resource** | **In the Media Library** |
| :--- | :--- |
| Image cards | No |
| Video cards | Yes\* |
| Tweets (any media)\*\* | Yes |
| Scheduled Tweets | Yes |
| Draft Tweets | Yes |
| Account media images | No |
| Account media videos (`PREROLL`) | Yes |

\* Para tarjetas donde el `video_content_id` es una media key. Cuando el valor es un media ID, el recurso aún existe en la Biblioteca de medios, pero recuperarlo implica anteponerle un prefijo numérico y un guion bajo.
\*\* Los Tweets solo devuelven media IDs. Si bien se garantiza que el recurso existe en la Biblioteca de medios, obtenerlo implica anteponerle un prefijo numérico y un guion bajo.

**Interacciones con Account Media**

Hay dos casos en los que los recursos multimedia agregados a la biblioteca se añaden automáticamente al recurso Account Media.

- Cuando se agrega un recurso AMPLIFY\_VIDEO a la Biblioteca de medios, se agrega automáticamente como un recurso de Account Media con el tipo creativo PREROLL.
- Cuando se agregan a la Biblioteca de medios imágenes con dimensiones específicas (consulta “Creative Types” en nuestra [página de enumeraciones](https://developer.x.com/content/developer-twitter/en/docs/ads/general/overview/enums)), se agregan automáticamente como recursos de Account Media. El tipo creativo (p. ej., INTERSTITIAL) depende de las dimensiones de la imagen.

<div id="tweets">
  ### Tuits
</div>

<div id="introduction">
  #### Introducción
</div>

La API de Anuncios de X admite tres tipos de tuits: publicados, programados y borradores.

<div id="nullcasted-tweets">
  #### Tweets nullcast
</div>

Los Tweets pueden ser nullcast (también llamados “solo promocionados”) u orgánicos. Los Tweets nullcast, una vez publicados, no aparecen en la cronología pública del usuario, aunque son públicos. Los Tweets orgánicos, por otro lado, se muestran a los seguidores del usuario y sí aparecen en la cronología pública del usuario.

**Creación de Tweets**

Cada uno de los tres endpoints de creación de Tweets admite un parámetro booleano nullcast que ofrece al usuario de la API la opción de crear Tweets nullcast u orgánicos. Los Tweets nullcast pueden ser creados por el usuario o por cualquier persona que tenga permiso para crear Tweets en nombre del usuario. Los Tweets orgánicos solo pueden ser creados por el [full promotable user](https://developer.x.com/content/developer-twitter/en/docs/tutorials/ads-api-hierarchy-terminology#promotable-users).

**Actualización de Tweets**

Es posible actualizar la propiedad nullcast para los Tweets programados y los borradores. Para los Tweets programados, se pueden realizar ediciones hasta la hora scheduled\_at del Tweet. Los Tweets en borrador se pueden editar indefinidamente. Sin embargo, una vez publicados, no es posible cambiar un Tweet de nullcast a orgánico o viceversa.

<div id="promoting-tweets">
  #### Promoción de Tweets
</div>

Solo se pueden promocionar Tweets publicados y programados. Estos pueden ser nullcast o orgánicos; no hay restricciones. Un anunciante puede promocionar sus propios Tweets o los de otro usuario siempre que haya obtenido permiso para hacerlo. (Consulta: [Promoting another user's Tweets](https://developer.x.com/content/developer-twitter/en/docs/tutorials/promoting-another-users-content) para obtener más información).

Se pueden promocionar varios Tweets en una sola campaña. Del mismo modo, un solo Tweet puede promocionarse en una o más campañas.

Para promocionar Tweets publicados, usa el [endpoint POST accounts/:account\_id/promoted\_tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/promoted-tweets#post-accounts-account-id-promoted-tweets). Esto asocia los Tweets publicados con un line item. Para promocionar Tweets programados, usa el endpoint [POST accounts/:account\_id/scheduled\_promoted\_tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets).

<div id="tweet-ids">
  #### IDs de Tweet
</div>

Los IDs de Tweets publicados, programados y en borrador son [numéricos](https://developer.x.com/content/developer-twitter/en/docs/basics/twitter-ids): son enteros sin signo de 64 bits. Por ejemplo, el ID del siguiente Tweet publicado es 1166476031668015104.

<div id="carousels">
  ### Carruseles
</div>

<div id="introduction">
  #### Introducción
</div>

La API de Anuncios de X permite crear y recuperar carruseles de video e imagen. El carrusel es un tipo de tarjeta que puede contener entre 2 y 6 elementos multimedia. La tarjeta de carrusel puede dirigir a una persona a un sitio web o incentivar la instalación de una app móvil. Para obtener más información sobre los carruseles, sus beneficios, mejores prácticas y preguntas frecuentes, consulta nuestra página [Carousel Ads on X](https://business.x.com/en/advertising/carousels.html).

Un carrusel, como cualquier otro tipo de tarjeta, puede usarse en Tweets y esos Tweets pueden promocionarse después. El flujo de trabajo es el mismo al que ya estás acostumbrado:

1. Subir contenido multimedia
2. Crear la tarjeta
3. Crear el Tweet
4. Promocionar el Tweet

La única diferencia es cómo se crea la tarjeta. Mientras que otras solicitudes de creación de tarjetas aceptan parámetros de consulta, las solicitudes de creación de tarjetas de carrusel solo aceptan cuerpos POST en JSON.

<div id="endpoints">
  #### Endpoints
</div>

La API de Anuncios de X admite crear y obtener carruseles.

Para crear un carrusel —de cualquier tipo—, usa el endpoint [POST accounts/:account\_id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#post-accounts-account-id-cards). Para obtener carruseles, usa el endpoint [GET accounts/:account\_id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#get-accounts-account-id-cards).

<div id="json-post-body">
  #### Cuerpo de la solicitud POST en JSON
</div>

Los carruseles se crean con dos componentes. El primero especifica los recursos multimedia que se utilizarán. El segundo especifica información sobre el sitio web o la app.

En concreto, una tarjeta de carrusel se crea utilizando los siguientes componentes, en este orden:

- Un componente `SWIPEABLE_MEDIA`, que acepta una matriz de claves de medios
- _Uno_ de los siguientes:
- Un componente `DETAILS` para especificar información del sitio web
- Un componente `BUTTON` para especificar información de la app

El componente `SWIPEABLE_MEDIA` debe incluir una matriz `media_keys` donde puedes especificar entre 2 y 6 imágenes o videos. El orden en que se pasen las claves de medios determina el orden en que se renderizarán.

```
{
  "type": "SWIPEABLE_MEDIA",
  "media_keys": [
    "13_1089771253848666112",
    "13_1191948012077092867"
  ]
}
```

Como recordatorio, puedes obtener claves de medios haciendo una solicitud al endpoint [GET accounts/:account\_id/media\_library](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/media-library#get-accounts-account-id-media-library).

La composición del objeto del segundo componente depende de si deseas dirigir al usuario a un sitio web o alentarlo a instalar una app. La tabla a continuación resume las dos opciones. (Nota: todas las claves listadas son obligatorias).

|  | **Sitio web** | **App** |
| :--- | :--- | :--- |
| Especificar el tipo de componente | `"type": "DETAILS"` | `"type": "BUTTON"` |
| Título/Etiqueta | `"title": "X"` | `"label": { "type": "ENUM", "value": "INSTALL" }` |
| Destino | `"destination": { "type": "WEBSITE", "url": "https://www.x.com" }` | `"destination": { "type": "APP", ... }` |

Con esto, a continuación se muestra un ejemplo de cuerpo de solicitud POST en JSON para un carrusel de sitio web.

```
{
  "name": "website carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "DETAILS",
      "title": "X",
      "destination": {
        "type": "WEBSITE",
        "url": "https://www.x.com"
      }
    }
  ]
}
```

Los objetos de destino de app dentro de componentes `BUTTON` requieren un código de país y al menos un identificador de app. De forma opcional, aceptan deep links. Para una descripción de estos campos, consulta la [documentación de referencia](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#app-destination).

Con esto, a continuación se muestra un ejemplo de cuerpo de solicitud POST en JSON para un carrusel de app.

```
{
  "name": "app carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        "type": "APP",
        "country_code": "US",
        "googleplay_app_id": "com.twitter.android",
        "iphone_app_id": "333903271"
      }
    }
  ]
}
```

<div id="creative-metadata-tagging">
  ### etiquetado-de-metadatos-de-creativos
</div>

<div id="introduction">
  #### Introducción
</div>

Esta guía está dirigida a socios creativos, agencias y desarrolladores creativos para etiquetar los recursos utilizados en campañas de X y así comprender mejor el valor y el rendimiento de cada recurso.

**Nota:** Los recursos multimedia solo deben ser etiquetados por el socio o desarrollador que los crea. Si quien usa el recurso multimedia no lo creó, no implemente el etiquetado de metadatos.

El etiquetado de metadatos creativos permite atribuir imágenes y videos creados por socios creativos, independientemente de dónde se cargue el recurso en X o de quién lo cargue. Para vincular el recurso creativo con el socio creativo, se utiliza el estándar [XMP](https://exiftool.org/TagNames/XMP.html).

<div id="tagging-creative-assets">
  #### Etiquetado de recursos creativos
</div>

La siguiente tabla muestra los tipos de identificadores disponibles actualmente en la respuesta de cada recurso relacionado con imágenes, así como los nombres de los atributos correspondientes. Se necesita una herramienta de etiquetado para etiquetar recursos creativos. Se recomienda [ExifTool](https://exiftool.org/), una biblioteca de Perl multiplataforma junto con una [aplicación de línea de comandos](https://exiftool.org/exiftool_pod.html) para leer, escribir y editar metadatos. Consulta todos los [tipos de archivos](https://exiftool.org/#supported) compatibles.

Sigue las [instrucciones para instalar ExifTool](https://exiftool.org/install.html). También hay paquetes de software ofrecidos por [Homebrew](https://formulae.brew.sh/) que simplifican la instalación al proporcionar el [comando de instalación de exiftool](https://formulae.brew.sh/formula/exiftool) para macOS y Linux. Confirma que la herramienta esté correctamente instalada ejecutando exiftool -ver en la línea de comandos para obtener el número de versión. Obtén más información sobre los parámetros de comando de ExifTool en la [documentación de ExifTool](https://exiftool.org/).

Los socios creativos pueden asignar etiquetas de metadatos a recursos creativos nuevos o existentes con su app\_id de X en la etiqueta XMP contributor y la etiqueta date. Los recursos creativos seguirán las restricciones de tamaño existentes al [cargar medios](https://developer.x.com/en/docs/twitter-api/v1/media/upload-media/overview).

**Nota:** El uso por parte de X de la etiqueta XMP contributor garantiza que los metadatos capturen valores exclusivamente para campañas en X.

`exiftool -contributor="<YOUR APP ID>" -creative_file.jpg`

`exiftool -date="<date>" -creative_file.jpg`

El app\_id se puede encontrar en el [Portal para desarrolladores](https://developer.x.com/en/portal/dashboard), en Proyectos y Apps. Ejemplo: 16489123

El siguiente ejemplo agrega app\_id como la etiqueta contributor y date como la etiqueta date para una imagen:

```
app_id:858382169
```

```
date:2022-03-13
```

```
creative_file: eiffel_tower.jpg
```

```
exiftool -contributor=858382169 eiffel_tower.jpg
```

```
   1 image files updated
```

```
exiftool -date=2022-03-13 eiffel_tower.jpg
```

```
  1 image files updated
```

Verifica que tu imagen haya sido etiquetada correctamente:

`exiftool -xmp:all -G1 <filename>`

Ejemplo:

`exiftool -xmp:all -G1 eiffel_tower.jpg`

<div id="components">
  #### Componentes
</div>

Cada componente debe incluir un campo `type` que determina el esquema del objeto. La API de Anuncios de X admite los siguientes tipos de componentes, agrupados en componentes basados en medios y en descripción.

- Medios:
- `MEDIA`: vídeo o imagen individual
- `SWIPEABLE_MEDIA`: entre 2 y 6 vídeos o imágenes
- Descripción:
- `DETAILS`
- `BUTTON`

Cada componente tiene un conjunto de campos obligatorios (además de la clave `type`). Estos se enumeran en la siguiente tabla.

| `type` de componente | Campo | Tipo de valor |
| :--- | :--- | :--- |
| `MEDIA` | `media_key` | string |
| `SWIPEABLE_MEDIA` | `media_keys` | array of strings |
| `DETAILS` | `title`   `destination` | string   object |
| `BUTTON` | `label`   `destination` | object   object |

A continuación se muestra un ejemplo de un componente `BUTTON` en el contexto del arreglo `components` (omitiendo intencionalmente la clave `name`). (Los puntos suspensivos indican lugares donde haría falta especificar más información).

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        ...
      },
      "destination": {
        ...
      }
    }
  ]
}
```

El orden en que se especifican los objetos de componente define el orden de renderizado de arriba hacia abajo. Las tarjetas deben crearse usando un componente basado en medios y, además, un componente `DETAILS` o `BUTTON`. Los componentes basados en descripción se renderizan debajo de los medios y tienen destinos asociados, ya sea URLs o apps móviles.

**Etiqueta**

Las etiquetas definen el texto que se muestra en los botones y, por lo tanto, solo aplican al componente `BUTTON`. Los objetos de etiqueta tienen dos claves obligatorias: `type` y `value`. El `type` debe establecerse en `ENUM` y el `value` puede ser uno de: `BOOK`, `CONNECT`, `INSTALL`, `OPEN`, `ORDER`, `PLAY` o `SHOP`.

Basándose en el ejemplo anterior, a continuación se muestra el objeto `label` dentro del componente `BUTTON`.

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        ...
      }
    }
  ]
}
```

**Destino**

Los destinos son el lugar adonde los anunciantes quieren llevar a los usuarios. Siempre son obligatorios dentro de los componentes `DETAILS` o `BUTTON`. Hay dos tipos de destino: `WEBSITE` o `APP`.

**Nota**: Los destinos de sitio web solo se pueden usar con componentes `DETAILS` y los destinos de app solo se pueden usar con componentes `BUTTON`.

**Destino de sitio web**

| Nombre | Descripción |
| :--- | :--- |
| type   _obligatorio_ | El tipo de destino, que determina su esquema.  Tipo: enum  Valores posibles: `WEBSITE` |
| url   _obligatorio_ | La URL del sitio web a la que redirigir a un usuario.  Tipo: string  Ejemplo: `https://devcommunity.x.com/c/advertiser-api` |

**Destino de app**

| Nombre | Descripción |
| :--- | :--- |
| type   _obligatorio_ | El tipo de destino, que determina su esquema.  Tipo: enum  Valores posibles: `APP` |
| country\_code   _obligatorio_ | El código de dos letras [ISO 3166-1 alpha-2](https://en.wikipedia.org/wiki/ISO_3166-1_alpha-2) del país donde se vende la app.  Tipo: string  Ejemplo: `US` |
| googleplay\_app\_id   _a veces obligatorio_ | El nombre de paquete de la aplicación de Google Play.  **Nota**: Se requiere al menos uno de los siguientes: `ios_app_store_identifier` o `googleplay_app_id`.  Tipo: string  Ejemplo: `com.twitter.android` |
| ios\_app\_store\_identifier   _a veces obligatorio_ | El identificador de la App Store de iOS.  **Nota**: Se requiere al menos uno de los siguientes: `ios_app_store_identifier` o `googleplay_app_id`.  Tipo: string  Ejemplo: `333903271` |

#### POST accounts/:account\_id/tweet

Cree un tuit para el usuario totalmente promocionable de la cuenta (predeterminado) o para el usuario especificado en el parámetro `as_user_id`. Se admite la creación tanto de tuits nullcast (predeterminada) como orgánicos. Los tuits nullcast no aparecen en la cronología pública ni se muestran a los seguidores. Cualquiera de los dos tipos puede usarse en campañas.

Si el usuario autenticado no es el usuario promocionable `FULL` en esta cuenta, determine si tiene permiso para tuitear en nombre de ese usuario realizando una solicitud al endpoint [GET accounts/:account\_id/authenticated\_user\_access](https://developer.x.com/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access). Cuando use el [endpoint upload.x.com](https://developer.x.com/rest/reference/post/media/upload) para archivos multimedia, pase el mismo valor de `user_id` para el parámetro `additional_owners` que el valor de `as_user_id` que envía a este endpoint.
