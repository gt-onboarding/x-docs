<div id="user-accounts-vs-ad-accounts">
  # Cuentas de usuario vs Cuentas de anuncios
</div>

Hay dos tipos de cuentas involucradas en el uso de la API de Anuncios: cuentas publicitarias y cuentas de usuario de X. A lo largo de la documentación de la API de Anuncios, el término “cuenta” generalmente se refiere a la cuenta publicitaria.

- Las cuentas publicitarias se registran en **business.x.com** y se identifican en la API por account\_id. Las cuentas publicitarias se vinculan directamente a fuentes de financiación y utilizan contenido de una o más cuentas de usuario de X como ‘usuarios promocionables’. Cada cuenta publicitaria puede otorgar permisos a una o más cuentas de usuario de X. La cuenta publicitaria, o “cuenta actual”, está representada en casi todas las URL ejecutadas como un parámetro en línea :account\_id.
- Las cuentas de usuario de X (como @AdsAPI) se identifican por user\_id en la API de Anuncios. Una o más de estas cuentas pueden asociarse a una cuenta publicitaria. La cuenta de usuario de X autenticada que realiza solicitudes en la API se denomina el ‘usuario actual’. La lista de cuentas publicitarias a las que el usuario actual tiene acceso se puede obtener con [GET accounts](/es/x-ads-api/campaign-management#accounts). Los ‘usuarios promocionables’ son handles de X que pueden ser promocionados por una cuenta publicitaria específica. Para más detalles al respecto, consulta [Obtaining Ads Account Access](/es/x-ads-api/introduction).

<div id="methods-for-ad-account-access">
  ## Métodos para el acceso a la cuenta de Anuncios
</div>

Hay dos métodos que puede usar para realizar solicitudes a la API de Anuncios para la cuenta de un anunciante:

1. Realizar solicitudes en nombre de un anunciante (**recomendado**)
2. Realizar solicitudes usando su cuenta, a la que se le ha concedido acceso a la cuenta de un anunciante; por ejemplo, una [agencia](https://business.x.com/en/blog/agencies-manage-multiple-Twitter-Ads-accounts.html) que gestiona múltiples cuentas.

Este documento es un breve resumen de las diferencias entre estas opciones y debe utilizarse junto con nuestros otros recursos, como las [preguntas frecuentes sobre inicio de sesión multiusuario](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html). 

Como se describe en [Autorizar una solicitud](/es/resources/fundamentals/authentication#authorizing-a-request#authorizing-a-request), todas las solicitudes a la API de Anuncios requieren encabezados de autorización que usen [OAuth 1.0a](/es/resources/fundamentals/authentication) con un token de acceso obtenido mediante el flujo [OAuth](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) [de 3 participantes](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow). Su aplicación deberá implementar un flujo OAuth basado en la web para [obtener tokens de acceso](/es/resources/fundamentals/authentication#oauth-1-0a-2). Los desarrolladores de la API de Anuncios nunca deben solicitar que nuestros anunciantes de X compartan sus credenciales de inicio de sesión.

De forma predeterminada, cada [aplicación de desarrollador de X](/es/resources/fundamentals/developer-apps) contiene un **token de acceso** estático que puede usarse para realizar solicitudes a la API de Anuncios para la cuenta propietaria de la aplicación. Estas credenciales son ideales para casos de uso de una sola cuenta, sin requerir un flujo OAuth de 3 participantes o [basado en PIN](/es/resources/fundamentals/authentication#pin-based-authorization). Si no va a acceder a otra cuenta de Anuncios de X, use estas credenciales [de un solo usuario](/es/resources/fundamentals/authentication#oauth-1-0a-2) en lugar de los pasos siguientes.

<div id="levels-of-access">
  ## Niveles de acceso
</div>

<div id="app-level-permissions">
  ### Permisos a nivel de App
</div>

Cada usuario tendrá un nivel de acceso según lo solicitado en la [solicitud de acceso a la API de Anuncios](/es/x-ads-api/getting-started/step-by-step-guide):

- **Solo conversiones**: El usuario tiene acceso a los endpoints de [móvil](/es/x-ads-api/measurement/mobile-conversions) y [conversiones web](/es/x-ads-api/measurement/web-conversions) con permisos de [lectura y escritura](/es/resources/fundamentals/developer-apps).
- **Acceso estándar**: El usuario tiene acceso a los endpoints de [analítica](/es/x-ads-api/analytics), [gestión de campañas](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/overview), [creativos](https://developer.x.com/en/docs/twitter-ads-api/creatives/overview), [audiencias personalizadas](https://developer.x.com/en/docs/twitter-ads-api/audiences/overview) y [conversiones](/es/x-ads-api/measurement/web-conversions) con permisos de [lectura y escritura](/es/resources/fundamentals/developer-apps).

**Nota:** Los desarrolladores de la API de Anuncios que solicitaron acceso antes de julio de 2023 pueden tener distintos niveles de acceso y permisos, y pueden estar limitados a cinco tokens de OAuth. Consulte nuestra guía sobre [cómo aumentar el acceso](/es/x-ads-api/getting-started/increasing-access) para obtener acceso a endpoints adicionales o eliminar los límites de tokens en aplicaciones existentes.

<div id="ad-account-level-permissions">
  ### Permisos a nivel de cuenta de anuncios
</div>

Cada usuario que tenga acceso a una Cuenta de Anuncios tendrá un permiso específico a nivel de cuenta: **Administrador de cuenta**, **Administrador de anuncios**, **Analista de campañas**, **Analista orgánico** y **Administrador creativo**; consulte **business.x.com** para la documentación más reciente sobre permisos a nivel de cuenta. Las aplicaciones deben obtener los permisos del usuario actualmente autenticado mediante el endpoint de la API [Authenticated User Access](/es/x-ads-api/campaign-management#authenticated-user-access) para determinar a qué endpoints de la API y a qué funciones de Anuncios pueden acceder.

**Nota:** Cualquier token de usuario utilizado con la [Conversion API](/es/x-ads-api/measurement/web-conversions) debe corresponder a usuarios con permisos a nivel de cuenta de **Administrador de cuenta** o **Administrador de anuncios**.                                                                                                                                              

<div id="methods-of-obtaining-access-tokens">
  ## Métodos para obtener tokens de acceso
</div>

<div id="1-obtain-an-advertisers-user-access-token">
  ### 1. Obtener el token de acceso de un anunciante (Usuario)
</div>

Hay dos métodos para obtener un **token de acceso** de un anunciante. El más común es mediante un [flujo OAuth de 3 patas](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) directamente desde tu interfaz web. Las aplicaciones que no tengan una interfaz pública accesible para anunciantes pueden implementar un proceso OAuth [basado en PIN](/es/resources/fundamentals/authentication#pin-based-authorization). Cuando el usuario complete el flujo de 3 patas, tu aplicación tendrá credenciales para realizar solicitudes a su cuenta de Anuncios a través de la API.

Obtener credenciales de usuario mediante el flujo OAuth es el método que **recomendamos encarecidamente** para la mayoría de los desarrolladores de la API de Anuncios para obtener acceso a una cuenta de anunciante. Te permite llamar a la API en nombre de un usuario y realizar acciones como ese usuario. Estos tokens no caducan, pero pueden ser revocados por el usuario en cualquier momento.

<div id="2-obtain-your-developer-access-token">
  ### 2. Obtén tu token de acceso (Desarrollador)
</div>

Esta opción requiere que el anunciante [conceda acceso a tu @username](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html) (o @usernames) a su cuenta de X Ads a través de la interfaz de X en business.x.com. Los **tokens de acceso** obtenidos mediante el flujo OAuth de 3 partes para tu cuenta podrán acceder a la cuenta de X Ads del anunciante.

Esto te permite llamar a la API utilizando los tokens OAuth de tu propio @username en lugar de los tokens OAuth del anunciante. La diferencia clave en esta opción es que solo podrás crear Publicaciones solo promocionadas si se te ha otorgado a tu @username el permiso de delegación/composición de publicaciones.

Para obtener acceso para crear Publicaciones solo promocionadas en nombre del usuario promocionable `FULL` de la cuenta, también se te debe conceder acceso para crear Publicaciones en este flujo. Eso habilitará el acceso mediante el permiso `TWEET_COMPOSER` en el endpoint [GET accounts/:account\_id/authenticated\_user\_access](/es/x-ads-api/campaign-management#accounts).

<div id="differences-between-these-methods">
  ## Diferencias entre estos métodos
</div>

|     |     |     |
| :--- | :--- | :--- |
|     | Token OAuth de anunciante (usuario) | Token OAuth (desarrollador)  <br /> (@username agregado para separar la cuenta) |
| Acceso a la cuenta de Anuncios | ✔   | ✔   |
| Crear publicaciones en nombre del usuario | ✔   | ✔\*  |
| Gestionar campañas | ✔   | ✔   |
| Acceder a analíticas | ✔   | ✔   |
| Crear Cards en nombre del usuario | ✔   | ✔   |
| Posibilidad del desarrollador de acceder  <br />a través de la IU de X Ads |     | ✔   |
| [Límites de frecuencia](/es/x-api/enterprise-gnip-2.0/fundamentals/rate-limits) | Diferentes por anunciante | Diferentes por cuenta de anunciante |
