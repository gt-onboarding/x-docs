<div id="overview">
  ## Descripción general
</div>

Cada mes, realizamos cambios y lanzamos varias funciones nuevas en la API de Anuncios de X. Estos cambios casi siempre son compatibles con versiones anteriores; sin embargo, solemos tener un puñado de cambios incompatibles cada año. Hemos recibido comentarios de desarrolladores sobre los desafíos que nuestro rápido ritmo de cambios en la API de Anuncios supone para sus ciclos de desarrollo, especialmente al implementar nuevas funciones, gestionar deprecaciones y probar cambios. Queremos mejorar la experiencia del desarrollador al usar nuestra plataforma de Anuncios, por lo que introdujimos el concepto de versionar nuestros endpoints.

Algunas definiciones de los conceptos de los que hablamos:

**Versión**: Se refiere al número de versión que se encuentra en la ruta de la URL de cualquier solicitud a la API de Anuncios, por ejemplo: GET /{version_number}/accounts. Este estilo de versionado se conoce como versionado por URI.

**Cambios incompatibles**: Los cambios incompatibles son aquellos que requieren dedicar recursos de desarrollo para mantener la funcionalidad existente. Esto incluye recursos utilizados para investigar los cambios necesarios, determinar las funciones/endpoints que se deprecarán e implementar finalmente todos estos cambios. Una lista de cambios incompatibles incluye cosas como:

- Eliminar un parámetro de la solicitud/respuesta de la API

- Modificar el nombre de cualquier parámetro o endpoint

- Cambio en la representación de valores (preview\_url → card\_uri)

- Cambio en el comportamiento de los endpoints (p. ej., estadísticas asíncronas vs. síncronas)

- Agregar o cambiar parámetros opcionales u obligatorios (p. ej., hacer que name sea un campo obligatorio en la solicitud)

**Deprecación**: Las versiones o productos en estado deprecado dejarán de recibir soporte y se recomienda que los desarrolladores dejen de usar estas API.

**Retiro**: Una vez que se retire un producto o una API, el conjunto correspondiente de endpoints ya no será accesible a través de la API.

<div id="versioning-strategy">
  ## Estrategia de versionado
</div>

Los principios principales de la estrategia son:

1. Todos los cambios que rompan compatibilidad se agruparán en una nueva versión

2. Las descontinuaciones para versiones existentes cuando se anuncie una nueva versión tendrán un período de 6 meses

3. En cualquier momento, la API permitirá solicitudes de dos versiones simultáneamente; sin embargo, la más antigua de las dos no contará con soporte

4. Para facilitar una adopción más rápida de nuevos productos, estos se lanzarán de forma continua (fuera de la cadencia de versionado)

5. Todas las respuestas de la API contendrán un `x-current-api-version`, establecido en la versión vigente de la API, además de un encabezado `x-api-warn` al invocar cualquier endpoint de la API en desuso.

Si hubiera cambios fundamentales en los requisitos del producto que exigieran un cambio incompatible en la API (p. ej., descontinuar la segmentación por múltiples franjas de edad), enviaremos un aviso con 90 días de antelación para anunciar este cambio incompatible y, transcurridos al menos 90 días desde la emisión del aviso, se implementará dicho cambio.

<div id="v5">
  ### v5
</div>

Hoy, 28 de febrero de 2019, X presenta la API de Anuncios de X v5, con actualizaciones centradas en habilitar la escala y la eficiencia.

Esta versión incluye un nuevo endpoint para determinar qué entidades estuvieron activas en un intervalo de tiempo determinado, estadísticas para Media Creatives (es decir, videos in-stream e imágenes en la X Audience Platform), la capacidad de obtener _múltiples_ tarjetas mediante el URI de la tarjeta y mayor flexibilidad al recuperar criterios de segmentación y otras entidades. Recomendamos a todos los socios migrar a la versión más reciente de la API lo antes posible para evitar interrupciones del servicio. _La versión 3 de la API de Anuncios de X ha llegado al final de su vida útil y ya no está disponible._

<div id="new">
  #### Nuevo
</div>

**Determinación de qué entidades estuvieron activas**

El endpoint de [Entidades activas](/es/x-ads-api/analytics#active-entities-2) indica si las métricas de analítica de entidades de anuncios han cambiado. Diseñado para usarse junto con los endpoints de analítica, Entidades activas funciona especificando un tipo de entidad y un rango de fechas (máximo 90 días) y devuelve un array de IDs de entidad para las que su plataforma debería solicitar analítica. No se deben consultar IDs distintos de los devueltos en solicitudes de analítica posteriores.

Este endpoint admite los siguientes tipos de entidad: `CAMPAIGN`, `FUNDING_INSTRUMENT`, `LINE_ITEM`, `MEDIA_CREATIVE` y `PROMOTED_TWEET`.

**Estadísticas de `MEDIA_CREATIVE`**

Los endpoints de analítica de la API de Anuncios de X ahora proporcionan métricas para entidades Media Creative. Las Media Creative son el mecanismo mediante el cual se promocionan los anuncios in-stream o las imágenes en la X Audience Platform. La interfaz de X Ads muestra las métricas de Media Creative en las pestañas “In-stream videos” y “Display creatives”. Los endpoints de analítica tanto [sincrónicos](/es/x-ads-api/analytics#get-stats-accounts-account-id) como [asíncronos](/es/x-ads-api/analytics#asynchronous-analytics) ahora admiten el enum de entidad `MEDIA_CREATIVE`.

**Obtener múltiples cards**

Mejorando la [versión v3](https://devcommunity.x.com/t/ads-api-version-3/100732) del endpoint diseñado para recuperar una sola card por su valor de URI de card, ahora es posible obtener múltiples cards usando el endpoint [GET accounts/:account\_id/cards/all](/es/x-ads-api/creatives#cards-fetch). Ahora, en lugar de hacer una solicitud por cada card, puede recuperar hasta 200 cards en una sola solicitud.

Dos cosas a tener en cuenta:

1. La ruta de URL ahora es `accounts/:account_id/cards/all`. (La ruta anterior ya no está disponible). Esto es para mantener la coherencia con [el endpoint](/es/x-ads-api/creatives#cards-fetch) diseñado para recuperar una card por ID.
2. El parámetro de solicitud obligatorio ahora se llama card\_uris (en plural).

**Flexibilidad al recuperar**

El endpoint [GET accounts/:account\_id/targeting\_criteria](/es/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria) ahora admite múltiples IDs de line item. El parámetro line\_item\_ids, que acepta hasta 200 IDs, es obligatorio. Anteriormente solo se aceptaba un único line item, lo que dificultaba la sincronización. Con este cambio, ahora es posible recuperar más segmentación en menos tiempo.

Los siguientes endpoints ahora también admiten múltiples IDs de line item, aunque el parámetro line\_item\_ids es opcional en estos casos.

- [GET accounts/:account\_id/line\_item\_apps](https://developer.x.com/en/docs/x-ads-api/campaign-management/api-reference/line-item-apps#get-accounts-account-id-line-item-apps)
- [GET accounts/:account\_id/media\_creatives](/es/x-ads-api/campaign-management#get-accounts-account-id-media-creatives)
- [GET accounts/:account\_id/promoted\_accounts](/es/x-ads-api/campaign-management#get-accounts-account-id-promoted-accounts)
- [GET accounts/:account\_id/preroll\_call\_to\_actions](/es/x-ads-api/creatives#preroll-call-to-actions)

<div id="changed">
  #### Cambiado
</div>

**Recuperación de campañas y partidas en borrador**

Se actualizó la forma de recuperar campañas y partidas en borrador. Ahora, el parámetro with\_draft(boolean), cuando se establece en true, devuelve _tanto_ entidades en borrador como no en borrador. Esto es coherente con la forma en que se recuperan las entidades eliminadas (es decir, usando with\_deleted). Antes, obtener tanto entidades en borrador como no en borrador requería al menos dos solicitudes. Ahora puede hacerse en una sola llamada a la API.

| **v4** | **v5** |
| :--- | :--- | :--- |
| `draft_only` | `with_draft` |     |

**Segmentación por duración de activación de red**

La API de Anuncios de X resolvió un problema de visualización en el que, tras agregar la segmentación por duración de activación de red, el tipo de segmentación en la respuesta incluía el sufijo \_IN\_SEC. Hacer referencia a segundos era confuso, ya que la duración de activación de red siempre se expresa en meses. Esta corrección unifica la representación y reduce la confusión.

| **v4** | **v5** |
| :--- | :--- | :--- |
| `NETWORK_ACTIVATION_DURATION_IN_SEC` | `NETWORK_ACTIVATION_DURATION` |     |

**Conteos totales y cursores**

En v5, with\_total\_count y cursor son mutuamente exclusivos. Especificar ambos en una solicitud devolverá el código de error EXCLUSIVE\_PARAMETERS. Antes de v5, with\_total\_count se ignoraba cuando se especificaba cursor. Este cambio deja explícita la relación.

<div id="removed">
  #### Eliminado
</div>

Se están eliminando tres campos de las respuestas de la API de Anuncios: preview\_url, account\_id y parent\_ids. El esfuerzo de ingeniería para estos tres es mínimo.

- En la v4, se anunció que el parámetro de respuesta preview\_url para las tarjetas siempre era null. El paso final de esta migración es eliminar preview\_url de todas las respuestas de tarjetas.
- El atributo de respuesta account\_id se está eliminando para los siguientes recursos, dado que el ID de la cuenta de anuncios ya está presente en la URL, así como en request.params. (Es intencional excluir los instrumentos de financiación de esta lista, ya que los IDs padre deben estar presentes en los objetos de respuesta cuando sea posible, y los IDs de cuenta son entidades padre de los instrumentos de financiación).
  - Medios de la cuenta
  - Proveedores de eventos de apps
  - Etiquetas de eventos de apps
  - Campañas
  - Tarjetas
  - Elementos de línea
  - Usuarios promocionables
  - Criterios de segmentación
- Para las solicitudes [GET accounts/:account\_id/targeting\_criteria](/es/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria), ya no devolvemos el campo parent\_ids, ya que siempre era un arreglo vacío.

**Tarjetas de apps sin contenido multimedia**

En la v5, las tarjetas de apps sin contenido multimedia dejan de ser compatibles. [Anteriormente](https://devcommunity.x.com/t/deprecation-announcement-app-download-cards/87807), se eliminó la posibilidad de crear o editar tarjetas de apps sin contenido multimedia. Ahora, los endpoints restantes para este recurso quedan en desuso.

- Nota: Esto **no** afecta a las tarjetas de descarga de apps con imagen y video.

**Creación de medios de la cuenta**

El endpoint POST accounts/:account\_id/account\_media ya no está disponible en la v5. Otros endpoints para este recurso _no_ se ven afectados. El motivo de este cambio es que, al agregar medios a la [Biblioteca de medios](/es/x-ads-api/creatives#media-library), hay casos en los que esos recursos se agregan _automáticamente_ como entidades de Medios de la cuenta, y tratar de volver a agregar un recurso ya existente al recurso de Medios de la cuenta provoca un error. Esto ocurre en los siguientes casos:

- Los recursos `AMPLIFY_VIDEO` agregados a la Biblioteca de medios se añaden automáticamente como recursos de Medios de la cuenta con el tipo creativo `PREROLL`.
- Las imágenes con dimensiones específicas agregadas a la Biblioteca de medios se añaden automáticamente como recursos de Medios de la cuenta. El tipo creativo (p. ej., `INTERSTITIAL`) depende de las dimensiones de la imagen. (Para las dimensiones, consulta nuestra página de [Enumeraciones](/es/x-ads-api/introduction).)

<div id="new">
  #### Nuevo
</div>

_**API de Audiencias**_

La nueva API de Audiencias se basa en nuestro nuevo backend de procesamiento de audiencias, que ofrece mayor solidez y confiabilidad. Este nuevo endpoint permitirá a los socios proporcionar múltiples tipos de identificadores para un mismo usuario, lo que nos permite utilizar señales adicionales para la coincidencia. La documentación de referencia del nuevo endpoint de Audiencias se encuentra [aquí](/es/x-ads-api/audiences). Planeamos seguir publicando actualizaciones y mejoras a este producto durante el resto del año.

Los siguientes endpoints dejarán de estar disponibles en la v4 debido a funcionalidad redundante (seguirán funcionando en la v3 y se retirarán por completo cuando la v3 ya no esté disponible):

- Carga en TON:
  - GET accounts/:account\_id/tailored\_audience\_changes
  - GET accounts/:account\_id/tailored\_audience\_changes/:tailored\_audience\_change\_id
  - POST accounts/:account\_id/tailored\_audience\_changes
  - PUT accounts/:accounti\_d/tailored\_audiences/global\_opt\_out
- Audiencias en tiempo real:
  - POST tailored\_audience\_memberships

Por último, el parámetro `list_type` se eliminará de la solicitud y de la respuesta en _todos_ los [endpoints de Tailored Audiences](/es/x-ads-api/audiences) en la versión 4.

_**Endpoints de configuración**_

Ahora ofrecemos la posibilidad de que los administradores de cuenta configuren y actualicen la configuración de usuario, de cuenta y fiscal. La [configuración de usuario](/es/x-ads-api/campaign-management#get-accounts-account-id-user-settings-user-id) corresponde a las preferencias de contacto específicas del usuario para una cuenta de anuncios determinada. Con el endpoint [PUT accounts/:account\_id](/es/x-ads-api/campaign-management#put-accounts-account-id), los anunciantes ahora pueden actualizar el nombre de su cuenta y el sector o industria. Por último, los endpoints de [configuración fiscal](/es/x-ads-api/campaign-management#get-accounts-account-id-tax-settings) permiten a los anunciantes en países donde se cobra el impuesto al valor agregado (IVA) actualizar información como el nombre de la empresa, la dirección, el ID de IVA y si la cuenta pertenece al anunciante o a una agencia que anuncia en nombre de un anunciante.

**`preview_url` siempre null**

Como se anunció en la versión v3, todas las tarjetas existentes ahora tienen un `card_uri`. Como resultado, el valor de `preview_url` siempre será `null`.

Como recordatorio, asocie una tarjeta a un Tweet usando su valor `card_uri`. Vea el siguiente ejemplo de solicitud.

$ twurl -X POST -H ads-api.x.com "/4/accounts/18ce54d4x5t/tweet?text=Version 4\&card\_uri=card://958225772740714496"

<div id="removed">
  #### Eliminado
</div>

**Endpoints de video**

Los endpoints accounts/:account\_id/videos ya no estarán disponibles en v4. Este endpoint ha quedado obsoleto con la introducción de los endpoints de la [Biblioteca de medios](/es/x-ads-api/creatives#get-accounts-account-id-media-library). Consulte la siguiente comparación de uso.

- endpoint de videos v3: `twurl -H ads-api.x.com "/3/accounts/18ce54d4x5t/videos"`

- endpoint de biblioteca de medios v4 para videos: `twurl -H ads-api.x.com "/4/accounts/18ce54d4x5t/media_library?media_type=VIDEO"`

Los endpoints de la Biblioteca de medios tienen paridad total con los endpoints de videos y además admiten funcionalidades adicionales, como la posibilidad de manejar imágenes y GIF. Se solicita a los partners que utilicen exclusivamente la Biblioteca de medios para cualquier gestión de medios.

**`as_user_id` en vista de Tweet**

El parámetro `as_user_id` disponible en el endpoint [GET accounts/:account\_id/tweet/preview/:tweet\_id](/es/x-ads-api/creatives#get-accounts-account-id-tweets) ya no será aceptado. La vista previa siempre se renderizará como el autor del Tweet.

<div id="v3">
  ### v3
</div>

La versión 3 de la API de Anuncios [se lanzó](https://devcommunity.x.com/t/ads-api-version-3/100732) el 1 de febrero de 2018. La versión 2 de la API de Anuncios alcanzará su fin de vida el 1 de agosto de 2018.

Esta versión incluye nuestro nuevo producto Audience Intelligence, acceso a la Media Library y flujos de trabajo de tarjetas mejorados. También anunciamos la obsolescencia del endpoint [PUT accounts/:account\_id/targeting\_criteria](/es/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria). Finalmente, la versión 3 incluye algunos cambios menores en parámetros y respuestas y un límite inferior para el tamaño de lote.

Como con la [versión 2](https://devcommunity.x.com/t/ads-api-version-2/90360), damos a los socios **6 meses** para la transición. El **2018-08-01**, se desactivará la v2 de la API de Anuncios. Recomendamos a todos los socios y desarrolladores migrar a v3 lo antes posible.

**Audience Intelligence**

Audience Intelligence ofrece información en tiempo real sobre los principales hashtags, @handles y eventos más relevantes para una audiencia específica de X. Por ejemplo, ingrese Hombre 18-34 en EE. UU. y verá #nintendoswitch, #cardinal y @ricegum en tendencia entre esta audiencia.

Los [endpoints](/es/x-ads-api/audiences) de Audience Intelligence proporcionarán la siguiente funcionalidad:

- Dada una audiencia de entrada, recuperar los hashtags, @handles y eventos más relevantes.
- Dada una audiencia de entrada, recuperar información demográfica clave (como edad, género e ingresos del hogar).
- Dada una palabra clave, recuperar la serie temporal del volumen de tweets

**Media Library**

La [Media Library](/es/x-ads-api/creatives#get-accounts-account-id-media-library) permite gestionar imágenes, GIF y videos para cuentas de anuncios. Estos objetos multimedia pueden usarse en tweets y para crear tarjetas. También pueden reutilizarse en múltiples creatividades, lo que elimina la necesidad de cargar el mismo recurso varias veces.

Los objetos de la biblioteca se identifican mediante una media\_key. Las media keys son valores de cadena con el siguiente formato, por ejemplo: 13\_875943225764098048. En la API de Anuncios, estamos avanzando hacia el uso de media keys para todos los medios.

**Flujo de trabajo de tarjetas mejorado**

Todos nuestros endpoints de tarjetas ahora admiten media keys. Esto permite usar los objetos de la Media Library para crear o actualizar tarjetas.

Además, presentamos dos nuevos endpoints para [recuperar detalles de tarjetas](/es/x-ads-api/creatives#fetching-cards). Estos endpoints pueden utilizarse para buscar tarjetas usadas en tweets o tweets programados, por ejemplo, especificando el card\_uri o el id. Anteriormente, esto no era posible.

Además de estas nuevas funciones, incluimos los siguientes cambios en la versión 3.

**Novedades**

- La respuesta del endpoint [GET insights/keywords/search](/es/x-ads-api/audiences) ahora incluye un atributo related\_keywords con 30 términos relacionados con las palabras clave de entrada.

**Cambios**

- El tamaño máximo del lote de criterios de segmentación ahora es 500.
- Los atributos de respuesta card\_uri y preview\_url ahora son mutuamente excluyentes. Cuando una tarjeta tiene card\_uri, preview\_url será null. Cuando una tarjeta no tiene card\_uri, solo se devolverá preview\_url.
  - Todas las tarjetas creadas a partir de 2018-01-29 tendrán card\_uri.
  - Para la versión 4, todas las tarjetas existentes tendrán card\_uri.
- Ya no es posible crear tarjetas con imágenes 5:2. Si bien las tarjetas existentes basadas en imágenes 5:2 seguirán funcionando, recomendamos a los socios cambiar a las relaciones de aspecto de mejor rendimiento 1.91:1 o 1:1 (cuando se admitan).

**Eliminados**

- El endpoint [PUT accounts/:account\_id/targeting\_criteria](/es/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria) ya no está disponible. Hemos decidido realizar este cambio porque el comportamiento de reemplazo con este endpoint causaba confusión a los anunciantes y no era coherente con nuestros otros endpoints PUT que actualizan un solo recurso a la vez. En su lugar, los socios deben usar el endpoint [POST batch/accounts/:account\_id/targeting\_criteria](/es/x-ads-api/campaign-management#post-batch-accounts-account-id-targeting-criteria), que proporciona mayor flexibilidad, incluida la capacidad de agregar y eliminar segmentación en una sola solicitud.
- El atributo de respuesta paused ya no se devuelve para los instrumentos de financiamiento. En su lugar, consulta el atributo de respuesta entity\_status para determinar si un instrumento de financiamiento está en pausa o no. Además, dado que paused y cancelled corresponden al mismo valor, cancelled tampoco se devuelve en la respuesta.
- Hemos eliminado el parámetro card\_id del endpoint [GET accounts/:account\_id/tweet/preview](/es/x-ads-api/creatives#get-accounts-account-id-tweets).
- Debido a que no es posible recuperar Tweets programados eliminados, el parámetro with\_deleted ya no es compatible.
- El parámetro draft\_only se ha eliminado de los siguientes endpoints, ya que estas entidades nunca pueden estar en estado de borrador:
  - [GET accounts/:account\_id/targeting\_criteria](/es/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria)
  - [GET accounts/:account\_id/promoted\_tweets](/es/x-ads-api/campaign-management#get-accounts-account-id-promoted-tweets)
  - [GET accounts/:account\_id/promoted\_accounts](/es/x-ads-api/campaign-management#get-accounts-account-id-promoted-accounts)
  - [GET accounts/:account\_id/media\_creatives](/es/x-ads-api/campaign-management#get-accounts-account-id-media-creatives)
  - [GET accounts/:account\_id/preroll\_call\_to\_actions](/es/x-ads-api/creatives#get-accounts-account-id-preroll-call-to-actions)

**Notas[¶](#notes "Permalink a este encabezado")**

- Para ofrecer una mejor estimación del alcance, ahora el endpoint tiene en cuenta el presupuesto. Los siguientes parámetros pasan a ser obligatorios:
  - \[nuevo] `campaign_daily_budget_amount_local_micro`
  - `currency`
  - `bid`
  - `objective`

- `infinite_count` se ha renombrado a `infinite_bid_count` para evitar confusiones sobre su propósito
- Además de `count` e `infinite_bid_count`, ahora se devolverán estos nuevos datos:
  - `impressions`
  - `engagements`
  - `estimated_daily_spend_local_micro`

- El `data_type` de Audiencias personalizadas se ha cambiado de `tailored_audiences` a `tailored_audience` en todas nuestras respuestas.
- Las audiencias personalizadas compartidas ahora están disponibles como beta solo por API. Las audiencias personalizadas compartidas permiten usar una única audiencia en varias cuentas de anuncios. Use el endpoint [POST accounts/:account\_id/tailored\_audiences/:tailored\_audience\_id/permissions](/es/x-ads-api/audiences) (y los relacionados) para administrar los permisos de una audiencia personalizada que desee compartir entre cuentas de anuncios.
- Mejoras significativas en la forma de recopilar [analíticas](/es/x-ads-api/analytics) de rendimiento para cuentas de anunciantes:
- Para alinearnos con nuestras [mejores prácticas](/es/x-ads-api/analytics#best-practices), a partir de ahora solo permitiremos extraer datos de hasta **7 días** para los [endpoints de estadísticas síncronas](/es/x-ads-api/analytics#get-stats-accounts-account-id).
- Para simplificar la recopilación de métricas, hemos reemplazado el parámetro `metrics` por un nuevo parámetro `metric_groups`. Los desarrolladores solo deben indicar qué grupos de métricas desean que se devuelvan para una solicitud determinada.
  - Cualquier solicitud de métricas que no correspondan a una entidad determinada se excluirá de la respuesta y se representará con valores `null`. _Estas métricas no se contabilizarán en tu límite de costo de analítica._
- La respuesta se ha **simplificado considerablemente** y ahora se alineará más estrechamente con la forma en que las métrricas se muestran en nuestra interfaz.
  - Anteriormente exponíamos una métrica separada por cada Ubicación (Promoted Tweets in Search, Promoted Tweets in Timelines, Promoted Tweets in Profiles & Tweet Details, X Audience Platform). Ahora devolveremos un conjunto estandarizado de métricas para cada una (en lugar de `promoted_tweet_timeline_impressions`, `promoted_tweet_search_impressions`, `promoted_tweets_profile_impressions`, `promoted_tweets_tpn_impressions`); estas se expondrán, cuando se soliciten en una de las siguientes categorías, como una única métrica, `impressions` (esto aplica a todas las métricas):
  - `ALL_ON_TWITTER`
  - `PUBLISHER_NETWORK`
  - Cuando realices una solicitud, obtendrás una única métrica `impressions` para facilitar la concordancia de valores con nuestra interfaz.
  - Debes hacer dos consultas para obtener tanto los datos de `ALL_ON_TWITTER` como de `PUBLISHER_NETWORK`, ya que no se pueden combinar.
- Los [endpoints de estadísticas asíncronas](/es/x-ads-api/analytics#asynchronous-analytics) ya están disponibles, gracias a los comentarios de nuestros desarrolladores.
  - Un nuevo conjunto de endpoints para solicitar estadísticas de forma _asíncrona_, para datos que no necesitas de inmediato o para extracciones de datos históricos.
  - Coloca en cola un trabajo de estadísticas usando un endpoint único. Obtendremos los datos que solicitaste conforme lo permitan los recursos.
  - Puedes consultar un endpoint de estado del trabajo para determinar si los datos están disponibles.
  - Una vez que los datos estén disponibles, te proporcionaremos un ID de retiro para que descargues la respuesta JSON, la cual reflejará la del endpoint sincrónico.
  - Consulta hasta **90 días** de datos de hasta **20 entidades** en un solo trabajo.
- Consulta nuestra guía de migración de analytics v1, que incluye la correspondencia de métricas de v0 a v1
- Mejoras del entorno Sandbox \* Ahora puede [crear varias cuentas de anuncios de prueba](/es/x-ads-api/campaign-management#post-accounts) en el entorno Sandbox. \* Ahora puede crear varios instrumentos de financiación para una cuenta de anuncios de prueba únicamente en el entorno Sandbox. Esto le permite probar todos nuestros tipos de instrumentos de financiación. Antes, solo había disponible una fuente de financiación `CREDIT_CARD` para pruebas. \* ¿Desea probar una función beta? Ahora puede activar o desactivar funciones para una cuenta en el entorno Sandbox, según sus necesidades de prueba.
