<div id="typical-response-structure">
  ## Estructura típica de la respuesta
</div>

Las respuestas correctas se indican con un código HTTP de la serie 200 y una carga útil en JSON que contiene los objetos solicitados, creados, modificados o eliminados, junto con una representación de cómo el servidor interpretó la solicitud.

Si realizas una solicitud exitosa, recibirás como parte de la respuesta un nodo `request` que refleja tu solicitud.

_Ejemplo:_ [GET accounts/abcdefg/campaigns?with\_deleted=true](/es/x-ads-api/campaign-management#campaigns)

```json
{
  /* los datos de tu respuesta... */,
  "request": {
    "params": {
      "account_id": "abcdefg",
      "with_deleted": "true"
    }
  }
}
```

El campo `data` en las respuestas JSON contiene los objetos específicos asociados al recurso utilizado. El nodo `data` será un array JSON cuando la respuesta pueda incluir uno o más resultados. Se devolverá como un objeto JSON cuando solo sea posible un resultado en la respuesta. En algunos casos poco frecuentes, podrías ver una respuesta que normalmente incluiría una colección, pero en su lugar devuelve un objeto (hashmap). En ese caso, asume que dicho objeto único es del mismo tipo indicado en el campo `type`.

<div id="error-response-structure">
  ## Estructura de la respuesta de error
</div>

Las respuestas de error se devuelven con un código HTTP fuera de la serie 200. Por lo general, se adjunta una respuesta JSON, pero algunos errores devolverán cuerpos de otro tipo. En estas circunstancias, cuando no se puede analizar la estructura de la respuesta, considere que prevalece el significado fundamental del código HTTP. Por ejemplo, ocasionalmente puede ver un HTTP 404 junto con una respuesta HTML. En este caso, es razonable asumir que el contenido no se encuentra (HTTP 404 significa “Not Found”).

Las respuestas de error típicas siguen una estructura similar a la de las respuestas correctas. La naturaleza del error se comunica en un nodo `errors` de la respuesta. El nodo `errors/code` indicará un código de error constante en MAYÚSCULAS\_CON\_GUIONES\_BAJOS que puede consumir de forma programática para tomar decisiones de resolución. El nodo `errors/message` indicará una descripción (generalmente) legible para humanos del error en inglés. Se pueden adjuntar campos adicionales para indicar detalles más específicos sobre el error.

```json
{
  "errors": [
    {
      "parameter": "start_time",
      "details": "invalid date",
      "code": "INVALID_PARAMETER",
      "value": "",
      "message": "Expected time, got \"\" for start_time"
    }
  ],
  "request": {
    "params": {
      "account_id": "hkk5"
    }
  }
}
```

En el ejemplo anterior, se hizo una solicitud a un endpoint de analítica con un valor no válido para el parámetro `start_time`. El `errors/code` para solicitudes con parámetros no válidos es `INVALID_PARAMETER`.
