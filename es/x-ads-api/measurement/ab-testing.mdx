<div id="introduction">
  ### Introducción
</div>

Las pruebas A/B permiten a los anunciantes segmentar a los usuarios a los que llegan en X para comprender mejor cómo optimizar el rendimiento de la campaña y recopilar aprendizajes que informen sus estrategias de marketing.

Estos segmentos —denominados divisiones de grupos de usuarios— son aleatorios y mutuamente excluyentes. Con la aleatorización, los factores que influyen en los resultados se distribuyen equitativamente. En otras palabras, no existen diferencias inherentes entre los grupos ni en sus comportamientos esperados. Por ello, cuando se aplica una única variación a un grupo de usuarios y no a los demás, la diferencia en el rendimiento de la campaña puede atribuirse a esa variación.

Si bien es posible probar muchas variaciones a la vez, recomendamos encarecidamente probar una sola variación a la vez. Esto aísla el factor causal de la diferencia observada en el rendimiento de la campaña.

Las variaciones se definen a nivel de campaña. Por ejemplo, si el anunciante desea evaluar la eficacia de un nuevo creativo, debe crear dos campañas idénticas en las que la única diferencia sea el creativo. En el futuro, planeamos admitir variaciones a nivel de partida de pedido.

<div id="use-cases">
  ### Casos de uso
</div>

Las pruebas A/B se utilizan con mayor frecuencia para respaldar (1) casos de uso de optimización para clientes orientados al rendimiento que quieren entender qué funciona mejor en X a fin de optimizar su inversión y (2) casos de uso de aprendizaje para anunciantes de marca que quieren aplicar esos aprendizajes para orientar su estrategia de marketing.

La API admitirá pruebas A/B para cualquier variable de campaña, incluidas:

- Creatividades
- Segmentación
- Tipo de puja
- Unidad de puja

<div id="ab-testing">
  ## Pruebas A/B
</div>

Las pruebas A/B permiten a los anunciantes segmentar a los usuarios a los que llegan en X para entender cómo optimizar mejor el desempeño de las campañas y obtener aprendizajes que orienten sus estrategias de marketing.

Estos segmentos —denominados divisiones de grupos de usuarios— se asignan de manera aleatoria y son mutuamente excluyentes. Con la aleatorización, los factores que influyen en los resultados se distribuyen por igual. En otras palabras, no existen diferencias inherentes entre los grupos ni en sus comportamientos esperados. Por ello, cuando se aplica una única variación a un grupo de usuarios y no a los demás, la diferencia en el desempeño de la campaña puede atribuirse a esa variación.

Si bien es posible probar muchas variaciones a la vez, recomendamos encarecidamente probar una sola variación por vez. Esto aísla el factor causal de la diferencia observada en el desempeño de la campaña.

Las variaciones se configuran a nivel de campaña o a nivel de grupo de anuncios. El grupo de anuncios se define mediante [line item](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/line-items) en la API de Anuncios de X. Como ejemplo de una variación a nivel de grupo de anuncios, si el anunciante quiere evaluar la eficacia de una nueva creatividad, debe crear una campaña con 2 grupos de anuncios idénticos donde la única diferencia sea la creatividad.

<div id="use-cases">
  ### Casos de uso
</div>

Las pruebas A/B se utilizan con mayor frecuencia para respaldar: (1) casos de optimización para clientes orientados al rendimiento que desean comprender qué funciona mejor en X para optimizar su inversión y (2) casos de aprendizaje para anunciantes de marca que buscan aprovechar esos aprendizajes para orientar su estrategia de marketing.

La API admitirá pruebas A/B para cualquier variable de campaña, incluidas:

- Creatividad

- Segmentación

- Tipo de puja

- Unidad de puja

<div id="attributes">
  ### Atributos
</div>

Las pruebas A/B se representan como estructuras anidadas. Hay campos de nivel superior para la propia prueba A/B y un arreglo de objetos de grupos de usuarios, cada uno con un conjunto de campos que lo describen.

A alto nivel, toda prueba A/B debe incluir la siguiente información:

- La duración de la prueba, representada por los campos start\_time y end\_time

- El nivel en el que se realizará la división, representado por el campo entity\_type

- Al menos dos (y como máximo 30) grupos de usuarios, cada uno representado como un objeto en el arreglo user\_groups

Cada grupo de usuarios debe incluir la siguiente información:

- El porcentaje de usuarios que se debe asignar al grupo de usuarios indicado, representado por el campo size

- Los IDs de campaña/IDs de línea de pedido que deben conformar el conjunto de usuarios para el grupo de usuarios indicado, representados por el arreglo entity\_ids

Opcionalmente, se pueden establecer valores de name y description para las pruebas A/B y para los grupos de usuarios. A continuación se puede encontrar información sobre reglas de validación y otras restricciones.

Otros metadatos, como el ID o la hora de creación, también se incluyen, pero los establece automáticamente X.

<div id="usage">
  ### Uso
</div>

Las subsecciones a continuación describen cómo crear y actualizar pruebas A/B. La lectura y la eliminación funcionan igual que en los demás endpoints de la API de Anuncios de X.

<div id="creating">
  #### Creación
</div>

Cree una prueba A/B usando el endpoint [POST accounts/:account\_id/ab\_tests](https://developer.x.com/en/docs/x-ads-api/measurement/api-reference/ab-tests). El endpoint solo acepta cuerpos POST en JSON. El Content-Type debe establecerse en application/json.

Después de que el anunciante configure dos o más campañas, se puede crear una prueba A/B. Como se indicó arriba, las pruebas A/B _deben_ incluir: duración de la prueba, nivel de división y al menos dos grupos de usuarios. Cada grupo de usuarios debe declarar el porcentaje de usuarios que se le asignará, así como los ID de campaña que conformarán su conjunto de usuarios. Cada uno de estos se describe con más detalle a continuación.

Duración de la prueba:

- Los valores start\_time y end\_time deben

  - Estar en el futuro (en relación con el momento en que se crea la prueba A/B)

  - Superponerse con las fechas de vuelo de la campaña/line item

- La prueba debe durar al menos un día para campañas no basadas en apps y al menos cinco días para campañas basadas en apps

Nivel de división:

- El entity\_type puede establecerse en CAMPAIGN o LINE\_ITEM

Grupos de usuarios:

- Cada grupo de usuarios se representa como un objeto en el arreglo user\_groups

  - Se requieren al menos dos grupos de usuarios

  - Se permiten como máximo 30 grupos de usuarios

- El tamaño de cada grupo de usuarios se establece usando una representación en cadena de un valor numérico entre 1.00 y 99.00

  - Nota: Los valores de tamaño entre objetos deben sumar 100.00

- Los ID de campaña deben especificarse en el arreglo entity\_ids de cada grupo de usuarios

Opcionalmente, establezca el nombre y la descripción de la prueba A/B o de uno o más grupos de usuarios.

**Para pruebas A/B a nivel de ítem de línea**

La principal diferencia entre las pruebas A/B a nivel de campaña y a nivel de ítem de línea es el entity\_type. Debemos configurarlo como ‘entity\_type’ = ‘LINE\_ITEM’ para las pruebas A/B a nivel de ítem de línea. Esto aplica a todas las acciones sobre una prueba A/B ya creada indicadas a continuación.

Requisitos:

1. Todos los ítems de línea de la campaña de prueba A/B deben incluirse en el test dividido.
2. Solo se permite una división equitativa a nivel de ítem de línea.
3. La cantidad de ítems de línea de grupos de usuarios permitidos en 1 test dividido debe ser menor o igual a 5.
4. Solo 1 ítem de línea por grupo de usuarios.

<div id="updating">
  ### Actualización
</div>

Actualice una Prueba A/B usando el endpoint [PUT accounts/:account\_id/ab\_tests/:ab\_test\_id](https://developer.x.com/en/docs/x-ads-api/measurement/api-reference/ab-tests). El endpoint requiere enviar un blob JSON en la solicitud. El Content-Type debe establecerse en application/json.

Como otros endpoints de actualización, el endpoint [PUT accounts/:account\_id/ab\_tests/:ab\_test\_id](/es/x-ads-api/measurement/ab-testing#get-accounts-account-id-ab-tests) requiere que el ID de la Prueba A/B se incluya en la URL. En general, las Pruebas A/B solo pueden actualizarse cuando el estado es SCHEDULED. Hay una excepción: es posible actualizar el end\_time de la Prueba A/B mientras está LIVE.

Este endpoint admite JSON parcial con IDs de objetos. Se aplican los siguientes principios:

- Para agregar o eliminar objetos o elementos, envíe el arreglo completo (y sus subestructuras); esta es una operación de **reemplazo**

- De lo contrario, modifique (cambie, agregue, elimine) los _campos_ existentes haciendo referencia a nombres de clave o IDs

  - Para eliminar un campo, establezca su valor en null

  - Los campos que no se envían no se modifican

Por ejemplo, agregar un tercer grupo de usuarios a la Prueba A/B creada anteriormente requeriría enviar el arreglo user\_groups con los dos objetos de grupo de usuarios existentes, así como con el nuevo que desea agregar. Piense en esto como _recrear_ el arreglo user\_groups; envíe los datos como si lo estuviera creando así desde el principio (no envíe IDs de objetos de grupo de usuarios). El arreglo user\_groups en la solicitud de actualización podría representarse como sigue.

```json
[
  {
    "entity_ids": [
      "f2qcw",
      "f2tht"
    ],
    "size": "30.0",
    "name": "first group"
  },
  {
    "entity_ids": [
      "f2rqi",
      "f2tws"
    ],
    "size": "30.0",
    "name": "second group",
    "description": "second AB test group"
  },
  {
    "entity_ids": [
      "i1vwr",
      "i1xre"
    ],
    "size": "40.0"
  }
]
```

Observe que los valores de size entre objetos siguen sumando 100.00. Si no los hubiéramos actualizado en los dos primeros objetos —anteriormente establecidos en 50.00 cada uno—, la solicitud habría fallado.

Si, en cambio, simplemente quisiéramos agregar una descripción al primer grupo de usuarios, el arreglo user\_groups en la solicitud de actualización se representaría como sigue.

```json
[
  {
    "id": "p1bcx",
    "description": "updated using a PUT request"
  }
]
```

Hacemos referencia al objeto de grupo de usuarios por ID y solo incluimos el campo que deseamos modificar.

#### POST accounts/:account\_id/ab\_tests

Crea una nueva prueba A/B.

Todos los parámetros se envían en el cuerpo de la solicitud y se requiere un `Content-Type` de `application/json`.

</div>

`https://ads-api.x.com/12/accounts/:account_id/ab_tests`

#### PUT accounts/:account\_id/ab\_tests/:ab\_test\_id

Actualiza la prueba A/B especificada.

Todos los parámetros se envían en el cuerpo de la solicitud y se requiere un `Content-Type` de `application/json`.

Este endpoint admite JSON parcial con IDs de objetos. Se aplican los siguientes principios:

- Para añadir o eliminar objetos o elementos, envía el arreglo completo (y sus subestructuras); esta es una operación de **reemplazo**
  - Piénsalo como recrear el arreglo
- En caso contrario, modifica (cambia, añade, elimina) los campos existentes haciendo referencia a nombres de clave o IDs
  - Para eliminar un campo, establece su valor en `null`
  - Los campos que no se envían no se modifican
