<div id="postman">
  #### Postman
</div>

Postman ist ein hervorragendes Tool, mit dem Sie Endpunkte testen können. Jede Postman-Anfrage enthält alle Pfad- und Body-Parameter, sodass Sie schnell erkennen, welche Optionen Ihnen zur Verfügung stehen. Weitere Informationen zu unseren Postman-Collections finden Sie auf der Seite ["Postman verwenden"](/de/tutorials/postman-getting-started).

<div id="authentication">
  #### Authentifizierung
</div>

Alle Endpunkte der X-API v2 erfordern, dass Sie Ihre Anfragen mit Zugangsdaten (Keys und Tokens) authentifizieren. Sie können entweder OAuth 1.0a User Context oder OAuth 2.0 Authorization Code mit PKCE verwenden, um Ihre Anfragen an diese Endpunkte zu authentifizieren.

[OAuth 1.0a User Context](/de/resources/fundamentals/authentication) erfordert die Verwendung Ihrer API-Keys, der Access Tokens des Nutzers sowie einiger weiterer Parameter, um einen [Authorization-Header zu erstellen](/de/resources/fundamentals/authentication#authorizing-a-request), den Sie dann mit Ihrer Anfrage übermitteln. Die Access Tokens müssen dem Nutzer zugeordnet sein, in dessen Namen Sie die Anfrage stellen. Wenn Sie Access Tokens für einen anderen Nutzer erzeugen möchten, muss dieser Ihre App mithilfe des [3-legged OAuth-Flows](https://developer-staging.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens) autorisieren.

Bitte beachten Sie, dass OAuth 1.0a schwierig zu verwenden sein kann. Wenn Sie mit dieser Authentifizierungsmethode nicht vertraut sind, empfehlen wir, eine [Bibliothek](/de/x-api/tools-and-libraries/overview) zu nutzen, ein Tool wie Postman zu verwenden oder OAuth 2.0 für die Authentifizierung Ihrer Anfragen zu verwenden.

[OAuth 2.0 Authorization Code mit PKCE](/de/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3) ermöglicht eine feinere Kontrolle über den Scope einer Anwendung sowie Autorisierungsabläufe über mehrere Geräte hinweg. Mit OAuth 2.0 können Sie spezifische, fein granulare Scopes wählen, die Ihnen bestimmte Berechtigungen im Namen eines Nutzers erteilen.

Um OAuth 2.0 in Ihrer App zu aktivieren, müssen Sie es in den Authentifizierungseinstellungen Ihrer App einschalten. Diese finden Sie im Bereich App-Einstellungen des Developer-Portals.

<div id="developer-portal-projects-and-developer-apps">
  #### Developer-Portal, Projekte und Entwickler-Apps
</div>

Um Authentifizierungsdaten zu erhalten, die mit den Endpunkten der X-API v2 funktionieren, müssen Sie sich für ein [Entwicklerkonto registrieren](https://developer.x.com/en/portal/petition/essential/basic-info), innerhalb dieses Kontos ein [Projekt](/de/resources/fundamentals/projects) einrichten und in diesem Projekt eine [Entwickler-App](/de/resources/fundamentals/developer-apps) erstellen. Anschließend finden Sie Ihre Keys und Tokens in Ihrer Entwickler-App.
 

<div id="rate-limits">
  #### Rate Limits
</div>

Jeden Tag stellen viele Tausend Entwickler Anfragen an die X-API. Um das enorme Volumen dieser Anfragen zu steuern, gelten für jeden Endpoint [Rate Limits](/de/x-api/fundamentals/rate-limits), die die Anzahl der Anfragen begrenzen, die Sie im Namen Ihrer App oder eines authentifizierten Nutzers stellen können. 

Diese Endpoints sind auf Nutzerebene ratebegrenzt. Das bedeutet, dass der authentifizierte Nutzer, in dessen Namen Sie die Anfrage stellen, den Endpoint nur eine bestimmte Anzahl von Malen über beliebige Developer-Apps hinweg aufrufen kann. Es gibt ein Nutzer-Rate-Limit von 50 Anfragen pro 15 Minuten und pro Endpoint für die Methoden POST und DELETE. Bei der Methode GET beträgt das Rate Limit jedoch nur 15 Anfragen pro 15 Minuten.
 

<div id="fields-and-expansions">
  #### Felder und Expansions
</div>

Die X-API v2 GET-Endpunkte ermöglichen es Nutzerinnen und Nutzern, mit einem Satz von Werkzeugen namens Felder und Expansions genau festzulegen, welche Daten die API zurückgeben soll. Der Parameter expansions ermöglicht es, im Payload referenzierte Objekte aufzulösen. Dieser Endpunkt erlaubt beispielsweise das Abrufen der folgenden [Expansions](/de/x-api/fundamentals/expansions):

- pinned\_tweet\_id

Der Parameter **fields** ermöglicht es, genau auszuwählen, welche [Felder](/de/x-api/fundamentals/fields) innerhalb der verschiedenen Datenobjekte Sie erhalten möchten. Diese Endpunkte liefern primär Post-Objekte. Standardmäßig gibt das Post-Objekt die Felder **id** und **text** zurück. Um zusätzliche Felder wie **tweet.created\_at** oder **tweet.entities** zu erhalten, müssen Sie diese ausdrücklich über den Parameter **fields** anfordern. Einige wichtige Felder, die Sie für Ihre Integration in Betracht ziehen sollten, sind unsere Umfragedaten, Metriken, Post-Anmerkungen und die Konversations-ID.

Wir haben unserer [X-API v2 Datendokumentation](/de/x-api/fundamentals/data-dictionary) eine Anleitung hinzugefügt, wie Sie [Felder und Expansions gemeinsam verwenden](/de/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions).

  #### Paginierung

Der Blocks-Lookup kann sehr viele Daten zurückliefern. Um sicherzustellen, dass wir zu keinem Zeitpunkt zu viele Ergebnisse zurückgeben, verwenden wir Paginierung. Erfahren Sie mehr darüber, wie Sie [durch Ergebnisse paginieren.](/de/x-api/fundamentals/pagination)
