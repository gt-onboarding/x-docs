<div id="mutes-lookup-standard-v11-compared-to-x-api-v2">
  ### Nachschlagen stummgeschalteter Accounts: Standard v1.1 im Vergleich zur X-API v2
</div>

Wenn Sie mit den Standard-v1.1-Endpunkten [GET mutes/users/ids](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/mute-block-report-users/api-reference/get-mutes-users-ids) und [GET mutes/users/list](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/mute-block-report-users/api-reference/get-mutes-users-list) gearbeitet haben, soll Ihnen dieser Leitfaden helfen, die Gemeinsamkeiten und Unterschiede zwischen den Standard-v1.1- und den X-API-v2-Endpunkten zum Nachschlagen stummgeschalteter Accounts zu verstehen.

- **Gemeinsamkeiten**
  - Authentifizierung
- **Unterschiede**
  - Endpoint-URLs
  - Benutzer pro Anfrage (Limits)
  - Anforderungen an App und Projekt
  - Antwortdatenformate
  - Anfrageparameter

<div id="similarities">
  #### Gemeinsamkeiten
</div>

**Authentifizierung**

Sowohl die Standard-v1.1- als auch die X-API-v2-Endpunkte für das Nachschlagen von Stummschaltungen verwenden [OAuth 1.0a User Context](/de/resources/fundamentals/authentication#oauth-1-0a-2). Wenn Sie zuvor einen der Standard-v1.1-Endpunkte für das Nachschlagen von Stummschaltungen verwendet haben, können Sie bei der Migration zur Version der X-API v2 daher weiterhin dieselbe Authentifizierungsmethode verwenden. 

<div id="differences">
  #### Unterschiede
</div>

**Endpoint-URLs**

- Standard-v1.1-Endpoints:
  - GET https://api.x.com/1.1/mutes/users/ids.json
    (Liste der Benutzer-IDs, die der angegebene Benutzer stummgeschaltet hat)
  - GET https://api.x.com/1.1/mutes/users/lists.json
    (Liste der Benutzer, die vom angegebenen Benutzer stummgeschaltet wurden)
- X-API-v2-Endpoint:
  - GET https://api.x.com/2/users/:id/muting
    (Liste der Benutzer, die von der angegebenen Benutzer-ID stummgeschaltet wurden)
     

**Limits pro Anfrage für Benutzer**

Die Standard-v1.1-Endpoints erlauben die Rückgabe von bis zu 5000 Benutzern pro Anfrage. Die neuen v2-Endpoints erlauben bis zu 1000 Benutzer pro Anfrage. Um volle 1000 Benutzer zu erhalten, müssen Sie max\_results=1000 als Query-Parameter übergeben; anschließend können Sie das in der Response-Nutzlast zurückgegebene next\_token als Wert für den pagination\_token-Query-Parameter in Ihrer nächsten Anfrage verwenden.
 

**App- und Projektanforderungen**

Die X-API-v2-Endpoints erfordern, dass Sie bei der Authentifizierung Ihrer Anfragen Anmeldedaten aus einer [Developer-App](/de/resources/fundamentals/developer-apps) verwenden, die einem [Projekt](/de/resources/fundamentals/developer-apps) zugeordnet ist. Alle X-API-v1.1-Endpoints können Anmeldedaten von eigenständigen Apps oder von Apps verwenden, die einem Projekt zugeordnet sind.

**Format der Antwortdaten**

Einer der größten Unterschiede zwischen Standard v1.1 und X-API-v2-Endpoint-Versionen ist, wie Sie auswählen, welche Felder in Ihrer Nutzlast zurückgegeben werden.

Bei den Standard-Endpoints erhalten Sie viele der Antwortfelder standardmäßig und haben dann die Möglichkeit, Parameter zu verwenden, um festzulegen, welche Felder oder Feldgruppen in der Nutzlast zurückgegeben werden sollen.

Die X-API-v2-Version liefert standardmäßig nur die Felder user id, name und username. Um zusätzliche Felder oder Objekte anzufordern, müssen Sie die Parameter [fields](/de/x-api/fundamentals/fields) und [expansions](/de/x-api/fundamentals/expansions) verwenden. Alle Benutzerfelder, die Sie von diesem Endpoint anfordern, werden im primären user-Objekt zurückgegeben. Alle erweiterten Post-Objekte und -Felder werden als includes-Objekt innerhalb Ihrer Antwort zurückgegeben. Sie können dann alle erweiterten Objekte dem user-Objekt zuordnen, indem Sie die IDs abgleichen, die sowohl im user- als auch im erweiterten Post-Objekt vorhanden sind. 

Wir empfehlen, mehr über diese neuen Parameter in den jeweiligen Leitfäden zu lesen oder unseren Leitfaden zu [how to use fields and expansions](/de/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) zu konsultieren. 

Wir haben außerdem einen [Leitfaden zur Datenformat-Migration](/de/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2) zusammengestellt, der Ihnen hilft, Standard-v1.1-Felder den neueren v2-Feldern zuzuordnen. Dieser Leitfaden nennt Ihnen auch die spezifischen Expansion- und Field-Parameter, die Sie mit Ihrer v2-Anfrage übergeben müssen, um bestimmte Felder zu erhalten. 
 

Zusätzlich zu den Änderungen bei der Anforderung bestimmter Felder führt die X-API v2 auch neue JSON-Designs für die von den APIs zurückgegebenen Objekte ein, einschließlich [Post](/de/x-api/fundamentals/data-dictionary#tweet)- und [user](/de/x-api/fundamentals/data-dictionary#user)-Objekten.

- Auf der JSON-Root-Ebene geben die Standard-Endpoints Post-Objekte in einem statuses-Array zurück, während X-API v2 ein data-Array zurückgibt. 
- Anstelle der Bezeichnung Retweeted und Quoted „statuses“ verweist das X-API-v2-JSON auf Retweeted und Quoted Tweets. Viele Legacy- und veraltete Felder wie contributors und user.translator\_type werden entfernt. 
- Anstelle der gleichzeitigen Verwendung von favorites (im Post-Objekt) und favourites (im user-Objekt) verwendet die X-API v2 den Begriff like. 
- X übernimmt die Konvention, dass JSON-Felder ohne Wert (zum Beispiel null) nicht in die Nutzlast geschrieben werden. Post- und user-Attribute werden nur aufgenommen, wenn sie einen Nicht-Null-Wert haben. 
   

Wir haben außerdem dem [Post-Objekt](/de/x-api/fundamentals/data-dictionary#tweet) einen neuen Satz von Feldern hinzugefügt, darunter:

- Ein [conversation\_id](/de/x-api/fundamentals/conversation-id)-Feld
- Zwei neue [annotations](/de/x-api/fundamentals/post-annotations)-Felder, darunter context und entities
- Mehrere neue [metrics](/de/x-api/fundamentals/metrics)-Felder 
- Ein neues reply\_setting-Feld, das anzeigt, wer auf einen bestimmten Post antworten kann

**Anfrageparameter**

Die folgenden Standard-v1.1-Anfrageparameter haben Entsprechungen in der X-API v2:

| **Standard** | **X-API v2** |
| :--- | :--- |
| stringify\_ids | Kein entsprechender Parameter |
| cursor | pagination\_token |
| skip\_status | Kein entsprechender Parameter |

Es gibt außerdem eine Reihe von Standard-Request-Parametern für die Mute-Suche in v1.1, die in der X-API v2 **nicht** unterstützt werden:

| Standard | Kommentar |
| :--- | :--- |
| include\_entities | Dieser Parameter entfernt den Knoten entities aus der Post-Payload. Er wurde durch additive Felder und die Expansionsfunktionalität ersetzt. |
