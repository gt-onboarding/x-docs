<div id="standard-v11-compared-to-x-api-v2">
  ### Standard v1.1 im Vergleich zur X-API v2
</div>

Wenn Sie mit den Standard-v1.1-Endpunkten GET users/show und GET users/lookup gearbeitet haben, soll Ihnen dieser Leitfaden helfen, die Gemeinsamkeiten und Unterschiede zwischen den Endpunkten zur Benutzerabfrage in Standard v1.1 und der X-API v2 zu verstehen.

- **Gemeinsamkeiten**
  - OAuth 1.0a User Context
  - Limits für Benutzer pro Anfrage
- **Unterschiede**
  - Endpoint-URLs
  - Anforderungen für Apps und Projekte
  - Format der Antwortdaten
  - Anfrageparameter

<div id="similarities">
  #### Gemeinsamkeiten
</div>

**Authentifizierungsmethode „OAuth 1.0a User Context“**

Der Standard-Endpunkt unterstützt [OAuth 1.0a User Context](/de/resources/fundamentals/authentication#oauth-1-0a-2), während die neuen X-API-v2-Endpunkte für die Benutzerabfrage sowohl OAuth 1.0a User Context als auch [App-only](/de/resources/fundamentals/authentication#oauth-2-0) unterstützen. Wenn Sie zuvor einen der Standard-v1.1-Endpunkte für die Benutzerabfrage verwendet haben, können Sie bei der Migration zur Version der X-API v2 dieselbe Authentifizierungsmethode weiterverwenden.

Je nach verwendeter Authentifizierungsbibliothek/-paket ist App-only-Authentifizierung wahrscheinlich der einfachste Einstieg und kann über einen einfachen Request-Header gesetzt werden. Informationen zum Generieren eines App-only-Access-Tokens finden Sie in [diesem App-only-Leitfaden](/de/resources/fundamentals/authentication#bearer-token-also-known-as-app-only).

**Benutzerlimits pro Anfrage**

Der Standard-v1.1-Endpunkt GET users/lookup erlaubt es, 100 Benutzer pro Anfrage anzugeben. Dies gilt auch für die Endpunkte GET /users und GET /users/by. Um volle 100 Benutzer anzugeben, müssen Sie den Parameter ids (GET /users) bzw. username (GET /users/by) als Query-Parameter übergeben und die Liste der Benutzer-IDs bzw. Benutzernamen als kommagetrennte Liste anfügen. 
 

<div id="differences">
  #### Unterschiede
</div>

**Endpoint-URLs**

- Standard-v1.1-Endpoints:
  - https://api.x.com/1.1/users/show (Abruf per einzelner ID oder Benutzername)
  - https://api.x.com/1.1/users/lookup (Abruf per mehreren IDs oder Benutzernamen)
- X-API-v2-Endpoints:
  - https://api.x.com/2/users (Abruf per mehreren IDs)
  - https://api.x.com/2/users/:id (Abruf per einzelner ID)
  - https://api.x.com/2/users/by (Abruf per mehreren Benutzernamen)
  - https://api.x.com/2/users/by/username/:username (Abruf per einzelnem Benutzernamen)

**App- und Projektanforderungen**

Die X-API-v2-Endpoints erfordern bei der Authentifizierung, dass Sie Anmeldedaten aus einer [Developer-App](/de/resources/fundamentals/developer-apps) verwenden, die einem [Projekt](/de/resources/fundamentals/projects) zugeordnet ist. Alle X-API-v1.1-Endpoints können Anmeldedaten von eigenständigen Apps oder von Apps verwenden, die einem Projekt zugeordnet sind.

**Antwortdatenformat**

Einer der größten Unterschiede zwischen den Standard-v1.1- und den X-API-v2-Endpunktversionen ist, wie Sie festlegen, welche Felder in Ihrer Payload zurückgegeben werden.

Bei den Standard-Endpoints erhalten Sie viele Antwortfelder standardmäßig und können dann Parameter verwenden, um festzulegen, welche Felder oder Feldgruppen in der Payload zurückgegeben werden sollen.

Die X-API v2 liefert standardmäßig nur die Felder user id, name und username. Um zusätzliche Felder oder Objekte anzufordern, müssen Sie die Parameter [fields](/de/x-api/fundamentals/fields) und [expansions](/de/x-api/fundamentals/expansions) verwenden. Alle Benutzerfelder, die Sie von diesem Endpoint anfordern, werden im primären user-Objekt zurückgegeben. Alle expandierten Post-Objekte und -Felder werden in einem includes-Objekt innerhalb Ihrer Antwort zurückgegeben. Sie können anschließend alle expandierten Objekte dem user-Objekt zuordnen, indem Sie die IDs abgleichen, die sowohl im user- als auch im expandierten Post-Objekt enthalten sind.

Wir empfehlen, mehr über diese neuen Parameter in den jeweiligen Leitfäden zu lesen oder unseren Leitfaden zu [how to use fields and expansions](/de/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) zu konsultieren.

Wir haben außerdem einen [data format migration guide](/de/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2) zusammengestellt, der Ihnen hilft, Standard-v1.1-Felder den neueren v2-Feldern zuzuordnen. Dieser Leitfaden nennt auch die spezifischen expansion- und field-Parameter, die Sie Ihrer v2-Anfrage hinzufügen müssen, um bestimmte Felder zurückzugeben.

Zusätzlich zu den Änderungen bei der Anforderung bestimmter Felder führt die X-API v2 auch neue JSON-Designs für die von den APIs zurückgegebenen Objekte ein, einschließlich [Post](/de/x-api/fundamentals/data-dictionary#tweet)- und [user](/de/x-api/fundamentals/data-dictionary#user)-Objekten.

- Auf der JSON-Root-Ebene geben die Standard-Endpoints Post-Objekte in einem statuses-Array zurück, während die X-API v2 ein data-Array zurückgibt.
- Anstatt sich auf Retweeted- und Quoted-„statuses“ zu beziehen, verwendet das X-API-v2-JSON die Begriffe Retweeted und Quoted Tweets. Viele Legacy- und veraltete Felder, wie contributors und user.translator\_type, werden entfernt.
- Anstelle von sowohl favorites (im Post-Objekt) als auch favourites (im user-Objekt) verwendet die X-API v2 den Begriff like.
- X übernimmt die Konvention, dass JSON-Felder ohne Wert (zum Beispiel null) nicht in die Payload geschrieben werden. Post- und user-Attribute werden nur einbezogen, wenn sie einen nicht-null Wert haben.

Wir haben dem [Post object](/de/x-api/fundamentals/data-dictionary#tweet) auch einen neuen Satz von Feldern hinzugefügt, darunter:

- Ein [conversation\_id](/de/x-api/fundamentals/conversation-id)-Feld
- Zwei neue [annotations](/de/x-api/fundamentals/post-annotations)-Felder, darunter context und entities
- Mehrere neue [metrics](/de/x-api/fundamentals/metrics)-Felder
- Ein neues reply\_setting-Feld, das anzeigt, wer auf einen bestimmten Post antworten kann

**Anfrageparameter**

Die folgenden Standard-v1.1-Anfrageparameter haben Entsprechungen in der X-API v2:

|     |     |
| :--- | :--- |
| **Standard** | **X API v2** |
| user\_id | ids |
| screen\_name | username |

Es gibt außerdem eine Reihe von Standard-Users-Lookup-Anfrageparametern, die in der X-API v2 **nicht** unterstützt werden:

| Standard | Kommentar |
| :--- | :--- |
| include\_entities | Dieser Parameter dient dazu, den Knoten „entities“ aus der Post-Payload zu entfernen. Er wurde durch die additive Funktionalität von Feldern und Erweiterungen ersetzt. |
