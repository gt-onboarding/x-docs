<div id="tweet">
  ### Tweet
</div>

Tweets sind der grundlegende Baustein für alles auf Twitter. Das Tweet-Objekt verfügt über eine lange Liste von „Root“-Feldern wie `id`, `text` und `created_at`. Tweet-Objekte sind außerdem das übergeordnete Objekt („Parent“) für mehrere untergeordnete Objekte, darunter `user`, `media`, `poll` und `place`. Verwenden Sie den Feldparameter `tweet.fields`, wenn Sie diese Root-Felder für das Tweet-Objekt anfordern.

Das Tweet-Objekt kann in der User-Ressource gefunden und erweitert werden. Weitere, zum angeforderten Tweet gehörende Tweets können ebenfalls in der Tweet-Ressource gefunden und erweitert werden. Das Objekt kann in der User-Ressource mit `?expansions=pinned_tweet_id` oder in der Tweet-Ressource mit `?expansions=referenced_tweets.id` erweitert werden, um das Objekt nur mit Standardfeldern zu erhalten. Verwenden Sie die Expansion zusammen mit dem Feldparameter `tweet.fields`, wenn Sie zusätzliche Felder anfordern, um das Objekt zu vervollständigen.

| Field Value           | Type   | Description                                                                                                                           | How it Can Be Used                                                  |
|:-----------------------|:--------|:---------------------------------------------------------------------------------------------------------------------------------------|:----------------------------------------------------------------------|
| **id (default)**      | string | Die eindeutige Kennung des angeforderten Tweets.                                                                                       | Verwenden Sie dies, um programmgesteuert einen bestimmten Tweet abzurufen. |
| **text (default)**    | string | Der tatsächliche UTF-8-Text des Tweets. Siehe [twitter-text](https://github.com/twitter/twitter-text/) für Details zu gültigen Zeichen. | Schlagwortextraktion und Stimmungsanalyse/-klassifizierung.          |
| **edit\_history\_tweet\_ids (default)** | object | Eindeutige Kennungen, die alle Versionen eines Tweets angeben. Bei Tweets ohne Bearbeitungen gibt es eine ID, bei Tweets mit Bearbeitungshistorie mehrere IDs. | Verwenden Sie diese Informationen, um die Bearbeitungshistorie eines Tweets zu ermitteln. |
| **attachments**       | object | Gibt die Art der Anhänge (falls vorhanden) in diesem Tweet an.                                                                        | Verständnis der für angeforderte Expansions zurückgegebenen Objekte. |
| **author\_id**         | string | Die eindeutige Kennung des Nutzers, der diesen Tweet gepostet hat.                                                                    | Anreichern des Nutzerobjekts, Teilen des Datensatzes zur Peer-Review. |
| **context\_annotations** | array | Enthält Kontextanmerkungen zum Tweet.                                                                                                 | Erkennung/Extraktion von Entitäten, thematische Analyse.             |
| **conversation\_id**   | string | Die Tweet-ID des ursprünglichen Tweets der Unterhaltung (einschließlich direkter Antworten und Antworten auf Antworten).              | Verwenden Sie dies, um die Unterhaltung ausgehend von einem Tweet zu rekonstruieren. |
| **created\_at**        | date (ISO 8601) | Erstellungszeitpunkt des Tweets.                                                                                                    | Nützlich für Zeitreihenanalysen und zum Verständnis, wann ein Tweet erstellt wurde. |
| **edit\_controls**     | object | Gibt an, wie lange der Tweet noch bearbeitet werden kann und wie viele Bearbeitungen verbleiben.                                      | Verwenden Sie dies, um festzustellen, ob ein Tweet noch bearbeitet werden kann. |
| **entities**          | object | Entitäten, die aus dem Text des Tweets extrahiert wurden. Siehe Entities in Twitter Objects.                                          | Liefert zusätzliche Informationen zu Hashtags, URLs, Erwähnungen usw. |
| **in\_reply\_to\_user\_id** | string | Wenn der dargestellte Tweet eine Antwort ist, enthält dieses Feld die Autor-ID des ursprünglichen Tweets.                            | Bestimmen, ob ein Tweet eine Antwort auf einen anderen Tweet war.    |
| **lang**              | string | Sprache des Tweets, sofern von Twitter erkannt.                                                                                       | Tweets nach gesprochener Sprache klassifizieren.                     |
| **non\_public\_metrics** | object | Nicht öffentliche Engagement-Metriken für den Tweet zum Zeitpunkt der Anfrage. Erfordert Authentifizierung im Benutzerkontext.       | Gesamtimpressionen bestimmen, die für den Tweet erzeugt wurden.      |
| **organic\_metrics**   | object | Engagement-Metriken, im organischen Kontext erfasst, zum Zeitpunkt der Anfrage. Erfordert Authentifizierung im Benutzerkontext.      | Organisches Engagement für den Tweet messen.                         |
| **possibly\_sensitive** | boolean | Gibt an, ob der Inhalt möglicherweise als sensibel erkannt wird.                                                                     | Verbreitung bestimmter Inhaltsarten untersuchen.                     |
| **promoted\_metrics**  | object | Engagement-Metriken, im beworbenen Kontext erfasst, zum Zeitpunkt der Anfrage. Erfordert Authentifizierung im Benutzerkontext.       | Engagement für den Tweet messen, wenn er beworben wurde.             |
| **public\_metrics**    | object | Öffentliche Engagement-Metriken für den Tweet zum Zeitpunkt der Anfrage.                                                              | Tweet-Engagement messen.                                             |
| **referenced\_tweets** | array  | Eine Liste von Tweets, auf die sich dieser Tweet bezieht, z. B. Retweets, zitierte Tweets oder Antworten.                             | Konversationelle Aspekte von Retweets usw. verstehen.                |
| **reply\_settings**    | string | Zeigt, wer auf einen bestimmten Tweet antworten kann. Optionen sind „everyone“, „mentioned\_users“ und „followers“.                   | Antwort-Einstellungen der Unterhaltung für den Tweet bestimmen.      |
| **withheld**          | object | Enthält Details zur Zurückhaltung für [zurückgehaltene Inhalte](https://help.x.com/en/rules-and-policies/tweet-withheld-by-country). |                                                                      |
| **media\_metadata**          | array | Enthält Metadaten zu Medienanhängen des Tweets.     | Zusätzliche Metadaten wie den `alt_text` eines Medienanhangs des Tweets abrufen. |

**Abrufen eines Tweet-Objekts**

**Beispielanforderung**

In der folgenden Anfrage fordern wir Felder für den Tweet am Endpoint [Tweets lookup](/de/resources/fundamentals/rate-limits) an. Ersetzen Sie dabei unbedingt `$BEARER_TOKEN` durch Ihr eigenes generiertes [Bearer Token](/de/resources/fundamentals/authentication).

```bash
curl --request GET 'https://api.x.com/2/tweets?ids=1212092628029698048&tweet.fields=attachments,author_id,context_annotations,created_at,entities,geo,id,in_reply_to_user_id,lang,possibly_sensitive,public_metrics,referenced_tweets,text,withheld&expansions=referenced_tweets.id' --header 'Authorization: Bearer $BEARER_TOKEN'
```

<div id="user">
  ### Benutzer
</div>

Das Benutzerobjekt enthält Metadaten zum Twitter-Benutzerkonto, die den referenzierten Benutzer beschreiben. Es ist das primäre Objekt, das vom Endpoint [users lookup](/de/x-api/users/lookup/introduction) zurückgegeben wird. Wenn Sie für diesen Endpoint zusätzliche Benutzerfelder anfordern, verwenden Sie den Parameter `user.fields`.

Das Benutzerobjekt kann auch als untergeordnetes Objekt im Tweet-Objekt vorkommen und dort erweitert werden. Es steht zur Erweiterung mit `?expansions=author_id` oder `?expansions=in_reply_to_user_id` zur Verfügung, um das komprimierte Objekt mit nur den Standardfeldern zu erhalten. Verwenden Sie die Expansion in Kombination mit dem Feldparameter `user.fields`, wenn Sie zusätzliche Felder anfordern, um das Objekt zu vervollständigen.

| name (Standard) | string | Der vom Nutzer im Profil festgelegte Name. Nicht zwingend ein Personenname. Üblicherweise auf 50 Zeichen begrenzt, kann sich jedoch ändern.<br /><br />`"name": "Twitter Dev"` |     |

**Abrufen eines User-Objekts**

**Beispielanfrage**

In der folgenden Anfrage rufen wir Felder für den User über den Endpunkt [users lookup](/de/x-api/users/lookup/introduction) ab. Ersetzen Sie unbedingt `$BEARER_TOKEN` durch Ihr eigenes generiertes [Bearer-Token](/de/resources/fundamentals/authentication#using-and-generating-an-app-only-bearer-token).
 

```{
  curl --request GET 'https://api.x.com/2/users?
  ids=2244994945&user.fields=created_at,description,entities,id,location,name,pinned_tweet_id,profile_image_url,protected,url,username,verified,withheld&expansions=pinned_tweet_id'
  --header 'Authorization: Bearer $BEARER_TOKEN'
  }
```

<div id="space">
  ### Space
</div>

Spaces ermöglichen Ausdruck und Interaktion über Live-Audiogespräche. Das Space-Datenwörterbuch enthält relevante Metadaten zu einem Space; alle Details werden in Echtzeit aktualisiert.

Benutzerobjekte sind in der Benutzerressource verfügbar und können dort erweitert werden. Diese Objekte lassen sich erweitern, indem dem Abfrageparameter `expansions` mindestens einer der folgenden Werte hinzugefügt wird: `host_ids`, `creator_id`, `speaker_ids`, `mentioned_user_ids`.

Im Gegensatz zu Tweets sind Spaces vergänglich und nicht mehr verfügbar, sobald sie beendet werden oder wenn sie von ihrer Erstellerin bzw. ihrem Ersteller abgesagt werden. Wenn Ihre App mit Spaces-Daten arbeitet, sind Sie dafür verantwortlich, die aktuellsten Informationen bereitzustellen und Daten zu entfernen, die auf der Plattform nicht mehr verfügbar sind. Die [Spaces-Lookup-Endpunkte](/de/x-api/spaces/lookup/introduction) helfen Ihnen dabei sicherzustellen, dass Sie die Erwartungen und die Intention der Nutzenden respektieren.

| Feldwert            | Typ             | Beschreibung                                                                                                 | Verwendungsmöglichkeiten                                 |

| host\_ids           | array           | Eindeutige Kennungen der Space-Hosts.<br />`"host_ids": ["2244994945", "6253282"]`                           | User-Objekte erweitern, Engagement verstehen.            |

  ### Direct-Message-Ereignisse
</div>

Direct-Message- (DM-)Konversationen bestehen aus Ereignissen. Die X-API v2 unterstützt derzeit drei Ereignistypen: MessageCreate, ParticipantsJoin und ParticipantsLeave.

DM-Ereignisobjekte werden von den Endpunkten zur [Direct-Message-Abfrage](/de/x-api/direct-messages/lookup/introduction) zurückgegeben, und ein MessageCreate-Ereignis entsteht, wenn Direct Messages erfolgreich über die Endpunkte [Direct Messages verwalten](/de/x-api/direct-messages/lookup/introduction) erstellt werden.

Bei Anfragen zu DM-Ereignissen sind standardmäßig drei Attribute (bzw. Felder) des Ereignisobjekts enthalten: id, event\_type und text. Um zusätzliche Ereignisfelder zu erhalten, verwenden Sie den Parameter [fields](/de/x-api/fundamentals/fields) dm\_event.fields, um weitere auszuwählen. Weitere verfügbare Ereignisfelder sind: dm\_conversation\_id, created\_at, sender\_id, attachments, participant\_ids und referenced\_tweets.

Mehrere dieser Felder liefern die IDs anderer X-Objekte, die mit dem Direct-Message-Ereignis in Zusammenhang stehen:

- sender\_id – Die ID des Accounts, der die Nachricht gesendet hat oder eine Person zu einer Gruppenunterhaltung eingeladen hat
- participant\_ids – Ein Array von Account-IDs. Bei ParticipantsJoin- und ParticipantsLeave-Ereignissen enthält dieses Array die einzelne ID des Accounts, der das Ereignis erstellt hat
- attachments – Liefert Media-IDs für Inhalte, die vom Absender bzw. der Absenderin auf Twitter hochgeladen wurden
- referenced\_tweets – Wenn im Feld text eine Tweet-URL gefunden wird, ist die ID dieses Tweets in der Antwort enthalten

Die [Erweiterungen](/de/x-api/fundamentals/expansions) sender\_id, participant\_ids, referenced\_tweets.id und attachments.media\_keys sind verfügbar, um diese Twitter-Objekt-IDs aufzulösen.

**Abrufen eines Direct-Message-Event-Objekts**

**Beispielanfrage**

In diesem Beispiel erstellen wir eine Anfrage, die Events abruft, die mit einer Eins-zu-eins-Konversation verknüpft sind. Diese Anfrage gibt grundlegende Direct-Message-Event-Felder zurück sowie zusätzliche Felder für referenzierte Tweets und deren Autoren. Erstellen wir eine Abfrage, die Folgendes anfordert:

- Grundlegende Event-Attribute wie den Erstellungszeitpunkt und zu welcher Konversation es gehört (dm\_conversation).
- Die Account-ID und die Beschreibung der Person, die die Direct Message gesendet hat.
- Den Text eines referenzierten Tweets und wann er veröffentlicht wurde.
- Die Account-ID und die Beschreibung der jeweiligen Autorin bzw. des jeweiligen Autors eines referenzierten Tweets.

Um diese Attribute zurückzugeben, würde Ihre Anfrage die folgenden Parameter enthalten:

`?dm_event.fields=id,sender_id,text,created_at,dm_conversation_id&expansions=sender_id,referenced_tweets.id&tweet.fields=created_at,text,author_id&user.fields=description`

```bash
    --header 'Authorization: Bearer $BEARER_TOKEN'
```

<div id="community">
  ### Community
</div>

Communities sind dedizierte Bereiche für X-Nutzer, um sich zu vernetzen, Inhalte zu teilen und an den Diskussionen teilzunehmen, die ihnen am wichtigsten sind.

Beiträge in Communities sind für alle auf X sichtbar, aber nur Mitglieder der jeweiligen Community können interagieren und an der Diskussion teilnehmen.

Das Community-Objekt enthält relevante Metadaten zu einer Community.

<div id="how-to-use-fields-and-expansions">
  ## So verwenden Sie Felder und Expansions
</div>

Standardmäßig enthalten die Datenobjekte der X-API v2 nur eine kleine Anzahl von Standardfeldern, wenn eine Anfrage ohne die Parameter [fields](/de/x-api/fundamentals/fields) oder [expansions](/de/x-api/fundamentals/expansions) gestellt wird. In diesem Leitfaden zeigen wir, wie Sie die Query-Parameter `fields` und `expansions` in Ihrer Anfrage verwenden, um zusätzliche Objekte und Felder in der Antwort zu erhalten.

In diesem Leitfaden fordern wir mehrere Felder für den folgenden Tweet-Screenshot an.
 

![Dieses Bild zeigt einen Screenshot eines Tweets, der von @X veröffentlicht wurde. Sichtbar sind der Tweet-Text, der Benutzername, Datum und Uhrzeit der Veröffentlichung, die Quelle und öffentliche Metriken. Es enthält außerdem ein Video. ](https://cdn.cms-twdigitalassets.com/content/dam/developer-twitter/ScreenShotDataDictionaryGuide.png.twimg.1920.png)

Wie Sie im Screenshot sehen, gibt es mehrere sichtbare Informationen zum Tweet, darunter der Autor des Tweets, Metriken zum Tweet, der Erstellungszeitpunkt, das Video und die Anzahl der Videoaufrufe. Es gibt auch mehrere Datenpunkte, die im Screenshot nicht sichtbar sind, aber dennoch angefordert werden können. 

Bei einer Anfrage an die API ist die Standardantwort schlank und enthält nur die Standard-Tweet-Felder (id und text). Außerdem erhalten Sie nur das primäre Objekt, das vom verwendeten Endpoint zurückgegeben wird, und nicht die zugehörigen Datenobjekte, die sich auf das primäre Objekt beziehen könnten.

Diese Schlankheit, zusammen mit den Parametern fields und expansions, ermöglicht es Ihnen, je nach Anwendungsfall nur die Felder anzufordern, die Sie benötigen.

Zunächst fordern wir mit einer Tweet-ID und dem [GET /tweets-Endpunkt](/de/x-api/posts/lookup/introduction) ein [Tweet-Objekt](/de/x-api/fundamentals/data-dictionary#tweet) an.

```bash
curl --request GET --url 'https://api.x.com/2/tweets?ids=1260294888811347969' \
```

   attachments, author\_id, created\_at, public\_metrics.

   `?tweet.fields=attachments,author_id,created_at,public_metrics`
