---
title: Rate Limits
---

Jeden Tag stellen viele Tausend Entwickler Anfragen an die X-API. Um das enorme Volumen dieser Anfragen zu bewältigen, sind Limits für die Anzahl der zulässigen Anfragen festgelegt. Diese Limits tragen dazu bei, eine zuverlässige und skalierbare API bereitzustellen, auf die sich unsere Entwickler-Community verlässt.

Die maximale Anzahl zulässiger Anfragen basiert auf einem Zeitintervall, in der Regel über einen festgelegten Zeitraum oder Zeitrahmen. Das gebräuchlichste Intervall beträgt fünfzehn Minuten. Ein Endpunkt mit einem Limit von 900 Anfragen pro 15 Minuten erlaubt beispielsweise bis zu 900 Anfragen in jedem beliebigen 15‑Minuten‑Intervall.

Rate Limits hängen von der Authentifizierungsmethode ab. Wenn Sie beispielsweise den [OAuth 1.0a User Context](/de/resources/fundamentals/authentication) verwenden, hat jeder Satz von [Access Tokens](/de/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) der Benutzer sein eigenes Rate Limit pro Zeitraum. Alternativ hat Ihre App bei Verwendung eines [OAuth 2.0 Bearer Tokens](/de/resources/fundamentals/authentication#oauth-2-0) ihr eigenes separates Limit pro Zeitraum. Wenn diese Limits überschritten werden, wird ein Fehler zurückgegeben.

<div id="rate-limits-and-authentication-method">
  ## Rate Limits und Authentifizierungsmethode
</div>

Rate Limits werden sowohl auf Ebene der Entwickler-App als auch auf Ebene des Benutzerzugriffstokens festgelegt:

- **[OAuth 2.0 Bearer Token](/de/resources/fundamentals/authentication/oauth-2-0/application-only): App-Rate-Limit**
  Diese Methode erlaubt eine bestimmte Anzahl von Anfragen im Namen Ihrer Entwickler-App. Bei Verwendung dieser Authentifizierungsmethode werden die Limits durch die mit einem Bearer-Token gestellten Anfragen bestimmt.
  - Beispiel: Bei einem Limit von 450 Anfragen pro 15-Minuten-Intervall können Sie innerhalb dieses Intervalls 450 Anfragen im Namen Ihrer App stellen.

- **[OAuth 1.0a User Context](/de/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens): Benutzer-Rate-Limit**
  Diese Methode ermöglicht Anfragen im Namen eines X-Nutzers, der durch das Benutzerzugriffstoken identifiziert wird. Wenn Sie beispielsweise private Metriken aus Posts abrufen, authentifizieren Sie sich mit Benutzerzugriffstokens für diesen Nutzer, die mithilfe des [3-legged OAuth Flow](/de/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) generiert wurden.
  - Beispiel: Bei einem Limit von 900 Anfragen pro 15 Minuten und pro Nutzer können Sie in diesem Zeitraum bis zu 900 Anfragen pro Nutzer stellen.

<div id="http-headers-and-response-codes">
  ## HTTP-Header und Antwortcodes
</div>

Verwenden Sie HTTP-Header, um auf Basis der zuletzt ausgeführten Anfrage zu bestimmen, wo Ihre Anwendung innerhalb eines bestimmten Rate Limits steht.

- `x-rate-limit-limit`: Obergrenze des Rate Limits für den Endpunkt
- `x-rate-limit-remaining`: verbleibende Anfragen im 15-Minuten-Zeitfenster
- `x-rate-limit-reset`: verbleibende Zeit bis zum Zurücksetzen des Rate Limits (in UTC-Epochensekunden)

<div id="error-responses">
  ### Fehlerantworten
</div>

Wenn eine Anwendung das Ratenlimit für einen Endpunkt überschreitet, gibt die API eine [HTTP-429-„Too Many Requests“](http://tools.ietf.org/html/rfc6585)-Antwort mit der folgenden Fehlermeldung im Antworttext zurück:

```json
{ "errors": [ { "code": 88, "message": "Rate limit exceeded" } ] }
```

<div id="recovering-from-a-rate-limit">
  ## Wiederherstellung nach Überschreitung eines Rate-Limits
</div>

Wenn diese Rate-Limits überschritten werden, gibt der Endpoint einen 429-Fehler „Too many requests“ zurück. Wie unten erläutert, ist es bei Rate-Limit-Fehlern Best Practice, die HTTP-Header zu prüfen, die angeben, wann das Limit zurückgesetzt wird, und Anfragen bis dahin zu pausieren.<br />Tritt ein „Too many requests“- oder Rate-Limiting-Fehler auf, sollte die Frequenz der Anfragen reduziert werden. Bei einem Rate-Limit-Fehler kann der HTTP-Header `x-rate-limit-reset:` geprüft werden, um zu erfahren, wann das Rate-Limiting zurückgesetzt wird<br /><br />. Ein weiteres gängiges Muster ist exponentielles Backoff: Die Zeit zwischen Anfragen beginnt klein (zum Beispiel einige Sekunden) und wird vor jedem erneuten Versuch verdoppelt. Dies wird fortgeführt, bis eine Anfrage erfolgreich ist oder eine angemessene maximale Zeit zwischen Anfragen erreicht ist (zum Beispiel einige Minuten).<br /><br />Idealerweise ist der Client sich der bestehenden Rate-Limits bewusst und kann Anfragen pausieren, bis das aktuell überschrittene Fenster abläuft. Wenn Sie ein 15‑Minuten‑Limit überschreiten, ist es sinnvoll, eine oder zwei Minuten zu warten, bevor Sie es erneut versuchen.<br /><br />Beachten Sie, dass zusätzlich zu diesen Limits für die Anzahl der Anfragen die Zugriffsstufe Standard Basic bis zu 500.000 Posts pro Monat über die Endpunkte für die jüngste Suche und den Gefilterten Stream bereitstellt. Wenn Sie das monatliche Limit für die Anzahl der Posts überschritten haben, ist es für Ihre App sinnvoller, eine Benachrichtigung auszulösen, das Einschreibedatum im Monat zu kennen und Anfragen bis dahin zurückzustellen.

<div id="tips-to-avoid-being-rate-limited">
  ### Tipps, um Rate Limits zu vermeiden
</div>

Die folgenden Hinweise sollen Ihnen helfen, defensiv zu programmieren und die Wahrscheinlichkeit zu verringern, von Rate Limits betroffen zu sein. Manche Funktionen, die Sie in Ihrer Anwendung anbieten möchten, sind angesichts von Rate Limits schlicht nicht umsetzbar, insbesondere im Hinblick auf die Aktualität der Ergebnisse. Wenn Echtzeitinformationen ein Ziel Ihrer Anwendung sind, sehen Sie sich die Endpunkte für den gefilterten und den Stichproben-Stream an.

<div id="caching">
  #### Caching
</div>

Speichern Sie API-Antworten, wenn eine häufige Nutzung zu erwarten ist. Anstatt die API bei jedem Seitenaufruf aufzurufen, speichern Sie die Antwort lokal im Cache.

<div id="prioritize-active-users">
  #### Aktive Nutzer priorisieren
</div>

Wenn Ihre Website eine große Zahl an X-Nutzern verfolgt (z. B. indem sie ihren aktuellen Status oder Nutzungsstatistiken abruft), sollten Sie erwägen, Daten nur für Nutzer anzufordern, die sich kürzlich auf Ihrer Website angemeldet haben.

<div id="adapt-to-the-search-results">
  #### An die Suchergebnisse anpassen
</div>

Wenn Ihre Anwendung viele Suchbegriffe überwacht, fragen Sie seltener nach Suchanfragen ohne Ergebnisse als nach solchen mit Ergebnissen. Mit einem Backoff bleiben Sie bei beliebten Abfragen aktuell, ohne Rechenzeit für Abfragen zu verschwenden, die sich nur sehr selten ändern. Alternativ können Sie den Endpunkt für den Gefilterten Stream verwenden und Ihre Suchabfragen als Filter einsetzen.

<div id="denylist">
  #### Denylist
</div>

Wenn eine Anwendung die Rate Limits missbraucht, wird sie auf die Denylist gesetzt. Apps auf der Denylist erhalten keine Antwort von der X-API. Wenn Sie oder Ihre Anwendung auf die Denylist gesetzt wurden und Sie glauben, dass ein Fehler vorliegt, können Sie unsere [Platform Support Forms](https://support.x.com/forms/platform) nutzen, um Unterstützung anzufordern. Bitte fügen Sie die folgenden Informationen bei:

1. Erklären Sie, warum Sie glauben, dass Ihre Anwendung auf die Denylist gesetzt wurde.
2. Wenn Sie nicht mehr von Rate Limits betroffen sind, beschreiben Sie detailliert, wie Sie das Problem behoben haben.
