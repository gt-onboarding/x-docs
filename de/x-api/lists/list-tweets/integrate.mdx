<div id="authentication">
  #### Authentifizierung
</div>

Alle X-API-v2-Endpunkte erfordern, dass Sie Ihre Anfragen mit einem Satz von Anmeldeinformationen (Keys und Tokens) authentifizieren. Sie können entweder OAuth 1.0a User Context, App only oder OAuth 2.0 Authorization Code mit PKCE verwenden, um Ihre Anfragen an diesen Endpunkt zu authentifizieren. 

[OAuth 1.0a User Context](/de/resources/fundamentals/authentication#oauth-1-0a-2) bedeutet, dass Sie einen Satz von API-Keys und Benutzer-Access-Tokens verwenden müssen, um eine erfolgreiche Anfrage zu stellen. Die Access-Tokens müssen dem Benutzer zugeordnet sein, in dessen Namen Sie die Anfrage stellen. Wenn Sie ein Set von Access-Tokens für einen anderen Benutzer generieren möchten, muss dieser Ihre App über den [3-legged OAuth Flow](/de/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) autorisieren.

Bitte beachten Sie, dass OAuth 1.0a schwer zu handhaben sein kann. Wenn Sie mit dieser Authentifizierungsmethode nicht vertraut sind, empfehlen wir, eine [Library](/de/x-api/tools-and-libraries/overview) zu verwenden, ein Tool wie Postman zu nutzen oder entweder OAuth 2.0 oder App only zu verwenden, um Ihre Anfragen zu authentifizieren.

[OAuth 2.0 Authorization Code mit PKCE](/de/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) ermöglicht eine feinere Kontrolle über den Scope einer Anwendung sowie Autorisierungsabläufe über mehrere Geräte. Mit OAuth 2.0 können Sie spezifische, fein granulare Scopes auswählen, die Ihnen bestimmte Berechtigungen im Namen eines Benutzers gewähren. 

Um OAuth 2.0 in Ihrer App zu aktivieren, müssen Sie es in den Authentifizierungseinstellungen Ihrer App aktivieren, die im Bereich App-Einstellungen des Developer-Portals zu finden sind.

[App only](/de/resources/fundamentals/authentication#app-only-authentication-and-oauth-2-0-bearer-token) erfordert lediglich, dass Sie Ihrer Anfrage ein [App-only-Access-Token](/de/resources/fundamentals/authentication#app-only-authentication-and-oauth-2-0-bearer-token) beifügen. Sie können ein App-only-Access-Token entweder direkt in einer Developer-App generieren oder eines über den Endpoint [POST oauth2/token](/de/resources/fundamentals/authentication#post-oauth2-token) erzeugen.

<div id="developer-portal-projects-and-developer-apps">
  #### Entwicklerportal, Projekte und Entwickler-Apps
</div>

Um einen Satz an Anmeldedaten zu erhalten, der mit den Endpunkten der X-API v2 funktioniert, müssen Sie sich zunächst für ein [Entwicklerkonto anmelden](https://developer.x.com/en/portal/petition/essential/basic-info), ein [Projekt](/de/resources/fundamentals/projects) innerhalb dieses Kontos einrichten und eine [Entwickler-App](/de/resources/fundamentals/developer-apps) innerhalb dieses Projekts erstellen. Anschließend finden Sie Ihre Keys und Tokens in Ihrer Entwickler-App.
 

<div id="rate-limits">
  #### Rate Limits
</div>

Jeden Tag stellen viele Tausend Entwickler Anfragen an die X-API. Um das enorme Volumen dieser Anfragen zu steuern, werden für jeden Endpunkt [Rate Limits](https://developer.x.com/x-api/fundamentals/rate-limits) festgelegt, die die Anzahl der Anfragen begrenzen, die Sie im Namen Ihrer App oder eines authentifizierten Nutzers stellen können.

Dieser Endpunkt ist sowohl auf App- als auch auf Nutzer-Ebene ratebegrenzt. Das App-Rate-Limit bedeutet, dass Sie als Entwickler innerhalb eines bestimmten Zeitraums von einer beliebigen App aus nur eine bestimmte Anzahl von Anfragen an diesen Endpunkt stellen können (unter Verwendung entweder des API Key und API Secret Key oder des Bearer Token). Das Nutzer-Rate-Limit bedeutet, dass der authentifizierte Nutzer, in dessen Namen Sie die Anfrage stellen, eine Listen-Post-Abfrage nur eine bestimmte Anzahl von Malen über alle Entwickler-Apps hinweg ausführen kann.

Die folgende Tabelle zeigt die Rate Limits für jeden Endpunkt.

|     |     |     |
| :--- | :--- | :--- |
| **Endpunkt** | **HTTP-Methode** | **Rate Limit** |
| /2/lists/:id/tweets | GET | 900 Anfragen pro 15 Minuten |

Felder und Erweiterungen

Der GET-Endpunkt der X-API v2 ermöglicht es Nutzern, mit Hilfe von Werkzeugen namens `fields` und `expansions` genau auszuwählen, welche Daten sie von der API zurückerhalten möchten. Der Parameter `expansions` ermöglicht es, Objekte zu erweitern, auf die in der Nutzlast verwiesen wird. Beispielsweise erlaubt die Abfrage von Listen-Posts das Abrufen der folgenden [Expansions](/de/x-api/fundamentals/expansions):

- `author_id`

Der Parameter `fields` ermöglicht es Ihnen, genau auszuwählen, welche [Felder](/de/x-api/fundamentals/fields) innerhalb der verschiedenen Datenobjekte Sie erhalten möchten. Dieser Endpunkt liefert hauptsächlich Post-Objekte. Standardmäßig gibt das Post-Objekt die Felder `id` und `text` zurück. Um zusätzliche Felder wie `tweet.created_at` oder `tweet.lang` zu erhalten, müssen Sie diese explizit über einen Fields-Parameter anfordern.

Wir haben unserer [X-API v2 Daten­wörterbuch](/de/x-api/fundamentals/data-dictionary) einen Leitfaden zur gemeinsamen Verwendung von [fields und expansions](/de/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) hinzugefügt.

Die folgende Tabelle zeigt die für den Lookup-Endpunkt verfügbaren Felder und Expansions:

|     |     |     |
| :--- | :--- | :--- |
| **Endpunkt** | **Felder** | **Expansions** |
| /2/lists/:id/tweets | `tweet.fields`<br /><br />`user.fields` | `author_id` |

Paginierung

Die Abfrage von Listen-Posts kann viele Daten zurückgeben. Um sicherzustellen, dass wir jederzeit konsistente, performante Ergebnisse liefern, verwenden wir Paginierung. Paginierung ist ein Feature in Endpunkten der X-API v2, die mehr Ergebnisse zurückgeben, als in einer einzelnen Antwort enthalten sein können. In diesem Fall werden die Daten in einer Reihe von „Seiten“ zurückgegeben. Erfahren Sie mehr darüber, wie Sie [durch Ergebnisse paginieren.](/de/x-api/fundamentals/pagination)
