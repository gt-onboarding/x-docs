<div id="list-posts-lookup-standard-v11-compared-to-x-api-v2">
  ### Lookup von Listen-Posts: Standard v1.1 im Vergleich zur X-API v2
</div>

Wenn Sie mit dem Standard-v1.1-Endpunkt [GET lists/statuses](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-statuses) gearbeitet haben, soll Ihnen dieser Leitfaden helfen, die Gemeinsamkeiten und Unterschiede zwischen den Endpunkten von Standard v1.1 und X-API v2 zu verstehen.

- **Gemeinsamkeiten**
  - Authentifizierungsmethoden
  - Rate Limits
- **Unterschiede**
  - Endpoint-URLs
  - Anforderungen an App und Projekt
  - Limits für Datenobjekte pro Anfrage
  - Formate der Antwortdaten
  - Anfrageparameter

<div id="similarities">
  #### Gemeinsamkeiten
</div>

**Authentifizierung**

Beide Endpunktversionen unterstützen [OAuth 1.0a User Context](/de/resources/fundamentals/authentication#oauth-2-0). Wenn Sie zuvor einen der Standard-v1.1-Endpunkte zur Listen-Post-Abfrage verwendet haben, können Sie bei der Migration zur X-API v2 weiterhin dieselbe Authentifizierungsmethode verwenden.

Je nach bevorzugter Authentifizierungsbibliothek bzw. -paket ist App-only-Authentifizierung wahrscheinlich der einfachste Einstieg und kann über einen einfachen Request-Header gesetzt werden. Wie Sie ein App-only-Access-Token erzeugen, erfahren Sie in [dieser App-only-Anleitung](/de/resources/fundamentals/authentication#bearer-token-also-known-as-app-only).

**Rate Limits**

|     |     |
| :--- | :--- |
| **Standard v1.1** | **X API v2** |
| /1.1/lists/statuses.json<br /><br />900 Anfragen pro 15-Minuten-Fenster mit OAuth 1.0a User Context<br /><br />900 Anfragen pro 15-Minuten-Fenster mit App-only | /2/lists/:id/tweets<br /><br />900 Anfragen pro 15-Minuten-Fenster mit OAuth 1.0a User Context<br /><br />900 Anfragen pro 15-Minuten-Fenster mit OAuth 2.0 Authorization Code mit PKCE<br /><br />900 Anfragen pro 15-Minuten-Fenster mit App-only |

<div id="differences">
  #### Unterschiede
</div>

**Endpoint-URLs**

- Standard-v1.1-Endpoints:
  - GET https://api.x.com/1.1/lists/statuses.json
    (Tweets aus einer angegebenen Liste abrufen)
- X-API-v2-Endpoint:
  - GET https://api.x.com/2/lists/:id/tweets
    (Tweets aus einer angegebenen Liste abrufen)

**Anforderungen für App und Projekt**

Die Endpoints der X-API v2 erfordern, dass Sie bei der Authentifizierung Ihrer Anfragen Anmeldedaten aus einer [Developer App](/de/resources/fundamentals/developer-apps) verwenden, die einem [Project](/de/resources/fundamentals/projects) zugeordnet ist. Alle X-API-v1.1-Endpoints können Anmeldedaten von eigenständigen Apps oder von Apps verwenden, die einem Projekt zugeordnet sind.

**Grenzwerte für Datenobjekte pro Anfrage**

Der Standard-v1.1-Endpoint /lists/statuses erlaubt die Rückgabe von bis zu 5000 Posts pro Anfrage. Die neuen v2-Endpoints erlauben die Rückgabe von bis zu 100 Posts pro Anfrage. Standardmäßig werden 100 User-Objekte zurückgegeben; um die Anzahl der Ergebnisse zu ändern, müssen Sie den Query-Parameter max\_results= mit einer Zahl zwischen 1–100 übergeben; anschließend können Sie das in der Response-Nutzlast zurückgegebene next\_token im Query-Parameter pagination\_token in Ihrer nächsten Anfrage übergeben.

**Format der Antwortdaten**

Einer der größten Unterschiede zwischen Standard v1.1 und den X-API-v2-Endpoint-Versionen ist, wie Sie auswählen, welche Felder in Ihrer Nutzlast zurückgegeben werden.

Bei den Standard-Endpoints erhalten Sie viele der Antwortfelder standardmäßig und haben dann die Möglichkeit, Parameter zu verwenden, um festzulegen, welche zusätzlichen Felder oder Feldsätze in der Nutzlast zurückgegeben werden sollen.

Die X-API-v2-Version liefert standardmäßig nur die Felder Post id und text. Um zusätzliche Felder oder Objekte anzufordern, müssen Sie die Parameter [fields](/de/x-api/fundamentals/fields) und [expansions](/de/x-api/fundamentals/expansions) verwenden. Alle Post-Felder, die Sie von diesem Endpoint anfordern, werden im primären Post-Objekt zurückgegeben. Alle erweiterten Objektfelder werden in einem includes-Objekt innerhalb Ihrer Antwort zurückgegeben. Sie können dann alle erweiterten Objekte dem primären Post-Objekt zuordnen, indem Sie die IDs aus dem primären Objekt und den erweiterten Objekten abgleichen. 

Hier sind Beispiele für mögliche Post-Felder und Expansions:

- attachments

- author\_id

- context\_annotations

- created\_at

- geo

- lang

|     |     |
| :--- | :--- |
| **Endpoint** | **Expansion** |
| /2/lists/:id/tweets | author\_id |

Wir empfehlen Ihnen, mehr über diese neuen Parameter in den jeweiligen Leitfäden zu lesen oder unseren Leitfaden zum [Verwenden von fields und expansions](/de/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) zu konsultieren. 

Wir haben außerdem einen [Leitfaden zur Datenformat-Migration](/de/x-api/migrate/data-format-migration) zusammengestellt, der Ihnen dabei helfen kann, Standard-v1.1-Felder den neueren v2-Feldern zuzuordnen. Dieser Leitfaden liefert Ihnen auch die spezifischen Parameter expansions und fields, die Sie mit Ihrer v2-Anfrage übergeben müssen, um bestimmte Felder zurückzugeben. 

Zusätzlich zu den Änderungen darin, wie Sie bestimmte Felder anfordern, führt die X-API v2 auch neue JSON-Designs für die von den APIs zurückgegebenen Objekte ein, darunter [Post](/de/x-api/fundamentals/data-dictionary#tweet)- und [user](/de/x-api/fundamentals/data-dictionary#user)-Objekte.

- Auf der JSON-Root-Ebene geben die Standard-Endpoints Post-Objekte in einem **statuses**-Array zurück, während die X-API v2 ein **data**-Array zurückgibt. 

- Anstatt sich auf Retweeted und Quoted „statuses“ zu beziehen, verwendet das X-API-v2-JSON die Begriffe Retweeted und Quoted Tweets. Viele Legacy- und veraltete Felder wie **contributors** und **user.translator\_type** werden entfernt. 

- Anstelle der Verwendung von sowohl **favorites** (im Post-Objekt) als auch **favourites** (im user-Objekt) verwendet die X-API v2 den Begriff **like**.

- X übernimmt die Konvention, dass JSON-Werte ohne Wert (zum Beispiel **null**) nicht in die Nutzlast geschrieben werden. Post- und user-Attribute werden nur aufgenommen, wenn sie nicht null sind.
   

**Request-Parameter**

Die folgenden Standard-v1.1-Request-Parameter haben Entsprechungen in der X-API v2:

|     |     |
| :--- | :--- |
| Standard v1.1 | X API v2 |
| list\_id | id  |
| slug | Keine Entsprechung |
| owner\_screen\_name | Keine Entsprechung |
| owner\_id | Angefordert mit dem Parameter expansions und dem Wert author\_id |
| since\_id | Keine Entsprechung |
| max\_id | Keine Entsprechung |
| include\_entities | Angefordert mit dem Parameter tweet.fields und dem Wert entities |
| include\_rts | Keine Entsprechung |
| count | max\_results |
