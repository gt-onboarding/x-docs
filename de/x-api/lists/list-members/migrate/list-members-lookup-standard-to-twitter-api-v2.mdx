<div id="list-members-lookup-standard-v11-compared-to-x-api-v2">
  ### Lookup von Listenmitgliedern: Standard v1.1 im Vergleich zur X-API v2
</div>

Wenn Sie mit den Standard-v1.1-Endpunkten [GET lists/members](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-members) und [GET lists/memberships](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-memberships) gearbeitet haben, soll Ihnen dieser Leitfaden helfen, die Gemeinsamkeiten und Unterschiede zwischen den Endpunkten für Listenmitglieder in Standard v1.1 und der X-API v2 zu verstehen.

- **Gemeinsamkeiten**
  - Authentifizierungsmethoden
- **Unterschiede**
  - Endpoint-URLs
  - Rate Limits
  - Anforderungen an App und Projekt
  - Limits für Datenobjekte pro Anfrage
  - Formate der Antwortdaten
  - Anfrageparameter

<div id="similarities">
  #### Gemeinsamkeiten
</div>

**Authentifizierung**

Beide Endpoint-Versionen unterstützen sowohl [OAuth 1.0a User Context](/de/resources/fundamentals/authentication#oauth-1-0a-2) als auch [App only](/de/resources/fundamentals/authentication#oauth-2-0). Wenn Sie zuvor einen der Standard-Endpoints für Listenmitglieder in v1.1 verwendet haben, können Sie bei der Migration auf die X-API-v2-Version dieselbe Authentifizierungsmethode weiterverwenden.

Je nach verwendeter Authentifizierungsbibliothek bzw. -paket ist die App-only-Authentifizierung wahrscheinlich der einfachste Einstieg und kann über einen einfachen Request-Header gesetzt werden. Informationen zum Erzeugen eines App-only-Access-Tokens finden Sie in [dieser App-only-Anleitung](/de/resources/fundamentals/authentication#bearer-token-also-known-as-app-only).

<div id="differences">
  #### Unterschiede
</div>

**Endpoint-URLs**

- Standard-v1.1-Endpoints:
  - GET https://api.x.com/1.1/lists/members.json
    (Abruf der Mitglieder einer angegebenen Liste)
  - GET https://api.x.com/1.1/lists/memberships.json
    (Abruf der Listen, in denen ein Nutzer Mitglied ist)
- X-API-v2-Endpoint:
  - GET https://api.x.com/2/lists/:id/members
    (Abruf der Mitglieder einer angegebenen Liste)

  - GET https://api.x.com/2/users/:id/list\_memberships
    (Abruf der Listen, in denen ein Nutzer Mitglied ist)

**Rate Limits**

|     |     |
| :--- | :--- |
| **Standard v1.1** | **X API v2** |
| /1.1/lists/members.json<br /><br />900 Anfragen pro 15-Minuten-Fenster mit OAuth 1.0a User Context<br /><br />15 Anfragen pro 15-Minuten-Fenster mit App-only | /2/lists/:id/members<br /><br />900 Anfragen pro 15-Minuten-Fenster mit OAuth 1.0a User Context<br /><br />900 Anfragen pro 15-Minuten-Fenster mit OAuth 2.0 Authorization Code mit PKCE<br /><br />900 Anfragen pro 15-Minuten-Fenster mit App-only |
| /1.1/lists/memberships.json<br /><br />15 Anfragen pro 15-Minuten-Fenster mit OAuth 1.0a User Context<br /><br />15 Anfragen pro 15-Minuten-Fenster mit App-only | /2/users/:id/list\_memberships<br /><br />15 Anfragen pro 15-Minuten-Fenster mit OAuth 1.0a User Context<br /><br />15 Anfragen pro 15-Minuten-Fenster mit OAuth 2.0 Authorization Code mit PKCE<br /><br />15 Anfragen pro 15-Minuten-Fenster mit App-only |

**App- und Projektanforderungen**

Die X-API-v2-Endpoints erfordern, dass Sie Anmeldedaten aus einer [Developer App](/de/resources/fundamentals/developer-apps) verwenden, die mit einem [Project](/de/resources/fundamentals/projects) verknüpft ist, wenn Sie Ihre Anfragen authentifizieren. Alle X-API-v1.1-Endpoints können Anmeldedaten von eigenständigen Apps oder Apps verwenden, die mit einem Projekt verknüpft sind.

**Grenzwerte für Datenobjekte pro Anfrage**

Der Standard-v1.1-Endpoint /1.1/lists/members ermöglicht die Rückgabe von bis zu 5000 Nutzern pro Anfrage. Die neuen v2-Endpoints ermöglichen die Rückgabe von bis zu 100 Nutzern pro Anfrage. Standardmäßig werden 100 Nutzerobjekte zurückgegeben; um die Anzahl der Ergebnisse zu ändern, müssen Sie den Query-Parameter max\_results= mit einer Zahl zwischen 1–100 übergeben; anschließend können Sie das in der Antwort enthaltene next\_token an den Query-Parameter pagination\_token in Ihrer nächsten Anfrage übergeben.

Zusätzlich ermöglicht der Endpoint /1.1/lists/memberships die Rückgabe von bis zu 1000 Listen pro Anfrage. Mit dem v2-Ersatz erlaubt der Endpoint bis zu 100 Listen pro Anfrage. Standardmäßig werden 100 Listenobjekte zurückgegeben; verwenden Sie die Query-Parameter max\_results= und pagination\_token in gleicher Weise wie bei /1.1/lists/members, um die Anzahl der Ergebnisse zu ändern.

**Format der Antwortdaten**

Einer der größten Unterschiede zwischen den Standard-v1.1- und den X-API-v2-Endpoint-Versionen ist, wie Sie festlegen, welche Felder in Ihrer Payload zurückgegeben werden.

Bei den Standard-Endpoints erhalten Sie viele der Antwortfelder standardmäßig und haben dann die Möglichkeit, Parameter zu verwenden, um anzugeben, welche zusätzlichen Felder oder Feldgruppen in der Payload zurückgegeben werden sollen.

Die X-API-v2-Version /users/:id/list\_memberships liefert standardmäßig die Felder List-ID und Name. Um zusätzliche Felder oder Objekte anzufordern, müssen Sie die Parameter [fields](/de/x-api/fundamentals/fields) und [expansions](/de/x-api/fundamentals/expansions) verwenden. Alle Listenfelder, die Sie von diesem Endpoint anfordern, werden im primären Listenobjekt zurückgegeben. Alle erweiterten Objekte und Felder werden in einem includes-Objekt innerhalb Ihrer Antwort zurückgegeben. Sie können dann alle erweiterten Objekte dem primären Listenobjekt zuordnen, indem Sie die IDs abgleichen, die sich sowohl im primären als auch im erweiterten Objekt befinden.

Hier sind Beispiele für mögliche Listenfelder und -erweiterungen:

- created\_at

- follower\_count

- member\_count

- owner\_id

- description

- private

|     |     |
| :--- | :--- |
| **Endpoint** | **Expansion** |
| /2/lists/:id/members | pinned\_tweet\_id |
| /2/users/:id/list\_memberships | owner\_id |

Wir empfehlen, mehr über diese neuen Parameter in den jeweiligen Leitfäden zu lesen oder unseren Leitfaden dazu zu lesen, [wie man fields und expansions verwendet](/de/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions). 

Wir haben außerdem einen [Leitfaden zur Datenformatmigration](/de/x-api/migrate/data-format-migration) zusammengestellt, der Ihnen hilft, Standardfelder aus v1.1 den neueren v2-Feldern zuzuordnen. Dieser Leitfaden nennt auch die konkreten Expansion- und Field-Parameter, die Sie mit Ihrer v2-Anfrage übergeben müssen, um bestimmte Felder zurückzugeben. 

Zusätzlich zu den Änderungen bei der Anforderung bestimmter Felder führt die X-API v2 neue JSON-Designs für die von den APIs zurückgegebenen Objekte ein, darunter [Post](/de/x-api/fundamentals/data-dictionary#tweet)- und [user](/de/x-api/fundamentals/data-dictionary#user)-Objekte.

- Auf der JSON-Root-Ebene geben die Standardendpunkte Post-Objekte in einem statuses-Array zurück, während die X-API v2 ein data-Array zurückgibt. 

- Statt von Retweeted- und Quoted-„statuses“ zu sprechen, verweist das JSON der X-API v2 auf Retweeted- und Quoted-Tweets. Viele Legacy- und veraltete Felder, wie contributors und user.translator\_type, werden entfernt. 

- Anstelle von sowohl favorites (im Post-Objekt) als auch favourites (im user-Objekt) verwendet die X-API v2 den Begriff like. 

- X übernimmt die Konvention, dass JSON-Felder ohne Wert (zum Beispiel null) nicht in die Payload geschrieben werden. Post- und user-Attribute werden nur einbezogen, wenn sie Nicht-Null-Werte haben.
