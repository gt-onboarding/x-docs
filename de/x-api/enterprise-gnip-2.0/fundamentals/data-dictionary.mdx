  ## X-API: Enterprise-Datenverzeichnis

<div id="introduction">
  ### Einführung
</div>

`Enterprise`

Posts sind die grundlegenden, atomaren Bausteine von allem auf X. Alle X-APIs, die Posts zurückgeben, liefern diese Daten in JavaScript Object Notation (JSON) kodiert. JSON basiert auf Schlüssel-Wert-Paaren mit benannten Attributen und zugehörigen Werten. Über die API abgerufene Post-Objekte umfassen den „Status-Update“ eines X-Nutzers; Retweets, Antworten und zitierte Tweets sind ebenfalls Post-Objekte. Wenn ein Post mit einem anderen Post zusammenhängt – etwa als Retweet, Antwort oder zitierter Tweet –, wird dies jeweils im Post-Objekt kenntlich gemacht oder eingebettet. Selbst der einfachste Post im nativen X-Datenformat enthält verschachtelte JSON-Objekte, um weitere Attribute eines Posts darzustellen, wie den Autor, erwähnte Nutzer, markierte Orte, Hashtags, Cashtag-Symbole, Medien oder URL-Links. Beim Arbeiten mit X-Daten ist dies ein wichtiges Konzept. Das Format der Post-Daten, die Sie von der X-API erhalten, hängt vom Typ des Posts, der verwendeten X-API und den Formateinstellungen ab.

Enterprise-Endpunkte, die Post-Objekte zurückgeben, wurden aktualisiert, um die Metadaten bereitzustellen, die zum Verständnis der Bearbeitungshistorie eines Posts erforderlich sind. Weitere Informationen zu diesen Metadaten finden Sie auf der Seite [„Posts bearbeiten“ – Grundlagen](/de/x-api/fundamentals/edit-posts).

```JSON
{
	"created_at": "Fri Feb 14 19:00:55 +0000 2020",
	"id_str": "1228393702244134912",
	"text": "Was hat der Entwickler in seine Valentinskarte geschrieben?\n  \nwhile(true) {\n    I = Love(You);  \n}",
	"entities": {
		"hashtags": [],
		"symbols": [],
		"user_mentions": [],
		"urls": []
	},
	"user": {
		"entities": {
			"url": {}
		}
	},
	"place": {}
}
```

<div id="available-data-formats">
  ### Verfügbare Datenformate
</div>

> Bitte beachten: Für Enterprise-Daten-APIs wird dringend empfohlen, das Format [Enriched Native](/de/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) zu verwenden. 
>
> - Das Enriched-Native-Format enthält alle seit 2017 hinzugekommenen Metadaten, etwa [Umfrage-Metadaten](/de/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata), sowie zusätzliche Metriken wie reply\_count und quote\_count.
>
> - Das Activity-Streams-Format wurde seit dem [Zeichen-Update](https://blog.x.com/official/en_us/topics/product/2017/Giving-you-more-characters-to-express-yourself.html) im Jahr 2017 nicht mehr mit neuen Metadaten oder Enrichments aktualisiert.

Enterprise-Daten-APIs liefern Daten in zwei Formaten. Das Enterprise-Format, das dem nativen Standardformat v1.1 am nächsten kommt, ist Enriched Native. Das ältere Enterprise-Datenformat ist Activity Streams, ursprünglich von Gnip als normalisiertes Format für X und andere Social-Media-Datenanbieter implementiert und verwendet. Obwohl dieses Format weiterhin verfügbar ist, hat X seit 2017 neue Funktionen und Weiterentwicklungen ausschließlich in das Enriched-Native-Format investiert.

Das Enriched-Native-Format ist genau das, was der Name vermuten lässt: Es umfasst native X-Objekte sowie zusätzliche Enrichments, die für Enterprise-Datenprodukte verfügbar sind, wie URL-Unwinding-Metadaten, Profil-Geodaten, Umfrage-Metadaten und zusätzliche Engagement-Metriken.  

- [Erweiterte und verbesserte URL-Enrichment](/de/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls)
- [Matching-Rules-Enrichment](/de/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#matching-rules)
- [Umfrage-Metadaten-Enrichment](/de/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata)
- [Profil-Geo-Enrichment](/de/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo)

### Best Practices für das Parsing

- X-JSON ist in UTF-8 codiert.
- Parser sollten problemlos Abweichungen in der Feldreihenfolge tolerieren. Es ist davon auszugehen, dass Post-JSON als ungeordnete Hash-Struktur bereitgestellt wird.
- Parser sollten die Ergänzung „neuer“ Felder tolerieren.
- JSON-Parser müssen gegenüber „fehlenden“ Feldern tolerant sein, da nicht alle Felder in allen Kontexten vorkommen.
- Es ist im Allgemeinen unbedenklich, ein auf null gesetztes Feld, eine leere Menge und das Fehlen eines Feldes gleichzusetzen.

> Lesen Sie unseren Vergleichsleitfaden: [Native Enriched im Vergleich zu X-API v2](/de/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2)

#### Post-Objekt

Bei der Nutzung von Enterprise-Datenprodukten werden Sie feststellen, dass ein Großteil des Data Dictionary dem nativen Format der Post-Daten ähnelt, ergänzt um zusätzliche angereicherte Metadaten. Die Basisebene des nativen angereicherten Formats verwendet viele der gleichen Objektnamen wie das [X-API-v1.1-Datenformat](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview.html). Das Post-Objekt verfügt über eine lange Liste von Attributen auf Root-Ebene, darunter grundlegende Attribute wie `id`, `created_at` und `text`. Post-Objekte enthalten außerdem verschachtelte Objekte wie `user`, `entities` und `extended_entities`. Post-Objekte beinhalten zudem weitere [verschachtelte Post-Objekte](/de/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#nested-post-objects) wie retweeted\_status, quoted\_status und extended\_tweet. Das native angereicherte Format enthält zusätzlich ein matching\_rules-Objekt.

#### Verschachtelte Post-Objekte

In mehreren Fällen enthält ein Post-Objekt weitere verschachtelte Objekte. Wenn Sie mit verschachtelten Objekten arbeiten, enthält diese JSON-Payload mehrere Post-Objekte, und jedes Post-Objekt kann eigene Objekte enthalten. Das Objekt auf Root-Ebene enthält Informationen über die Art der Aktion, also ob es sich um einen Retweet oder einen Quote-Tweet handelt, und kann auch ein Objekt enthalten, das den „ursprünglichen“ geteilten Post beschreibt. Erweiterte Posts enthalten ein verschachteltes Extended-Objekt, das über 140 Zeichen hinausgeht; dies wurde 2017 eingeführt, um Breaking Changes beim Update zu vermeiden. Jedes verschachtelte Objekt-Dictionary wird unten beschrieben.

**Retweets**

Retweets enthalten immer zwei Post-Objekte. Der „ursprüngliche“ Post, der retweetet wird, wird in einem "retweeted\_status"-Objekt bereitgestellt. Das Objekt auf Root-Ebene kapselt den Retweet selbst, einschließlich eines User-Objekts für das Konto, das die Retweet-Aktion ausführt, sowie der Zeit des Retweets. Retweeten ist eine Aktion, um einen Post mit Ihren Followern zu teilen; es kann kein neuer Inhalt hinzugefügt werden. Außerdem kann bei einem Retweet kein (neuer) Standort angegeben werden. Selbst wenn der „ursprüngliche“ Post geogetagt war, sind die "geo"- und "place"-Objekte des Retweets stets null.

Schon vor der Einführung erweiterter Posts war das "entities"-Objekt auf Root-Ebene in einigen Fällen gekürzt und unvollständig, da die Zeichenfolge "RT @username " an die retweetete Post-Nachricht angehängt wurde. Beachten Sie, dass, wenn ein Retweet erneut retweetet wird, der "retweet\_status" weiterhin auf den ursprünglichen Post zeigt, das heißt, der zwischengeschaltete Retweet ist nicht enthalten. Ein ähnliches Verhalten zeigt sich bei der Anzeige eines Retweets auf x.com. Wenn Sie die eindeutige Post-ID kopieren, die der Retweet-„Aktion“ zugewiesen ist, wird der ursprüngliche Post angezeigt.

Nachfolgend finden Sie eine Beispielstruktur für einen Retweet. Auch hier ist beim Parsen von Retweets entscheidend, das "retweeted\_status"-Objekt zu verarbeiten, um die vollständige (ursprüngliche) Post-Nachricht und die Entity-Metadaten zu erhalten.

```json
{
	"tweet": {
		"text": "RT @author original message",
		"user": {
			"screen_name": "Retweeter"
		},
		"retweeted_status": {
			"text": "original message",
			"user": {
				"screen_name": "OriginalTweeter"
			},
			"place": {},
			"entities": {},
			"extended_entities": {}
		}
	},
	"entities": {},
	"extended_entities": {}
}
```

##### Zitierte Tweets

Zitierte Tweets ähneln Retweets, enthalten jedoch eine neue Post-Nachricht. Diese neuen Nachrichten können eigene Hashtags, Links und andere „Entities“-Metadaten enthalten. Zitierte Tweets können auch vom Nutzer geteilte Standortinformationen sowie Medien wie GIFs, Videos und Fotos enthalten.

Zitierte Tweets enthalten mindestens zwei Post-Objekte, in einigen Fällen drei. Der zitierte Post, der selbst ein zitierter Tweet sein kann, wird in einem „quoted\_status“-Objekt bereitgestellt. Das Objekt auf Root-Ebene umfasst den zitierten Tweet selbst, einschließlich eines User-Objekts für das Konto, das die Teilaktion ausführt, sowie den Zeitpunkt des zitierten Tweets.

Beachten Sie, dass zitierte Tweets jetzt Fotos, GIFs oder Videos enthalten können, die über die „Post“-Benutzeroberfläche hinzugefügt werden. Wenn Links zu extern gehosteten Medien in der Nachricht des zitierten Tweets enthalten sind, beschreibt das „entities.urls“-Feld auf Root-Ebene diese. An zitierte Tweets angehängte Medien erscheinen in den „extended\_entities“-Metadaten auf Root-Ebene.

Als zitierte Tweets erstmals eingeführt wurden, wurde ein verkürzter Link (t.co-URL) an die „originale“ Post-Nachricht angehängt und im „text“-Feld auf Root-Ebene bereitgestellt. Zusätzlich wurden Metadaten für diese t.co-URL im „entities.urls“-Array auf Root-Ebene aufgenommen. Im Mai 2018 haben wir dies geändert, sodass die verkürzte t.co-URL zum zitierten Tweet nicht mehr im „text“-Feld auf Root-Ebene enthalten ist. Zweitens werden die Metadaten für den zitierten Tweet nicht mehr in den „entities.urls“-Metadaten enthalten sein. Stattdessen befinden sich die URL-Metadaten für den zitierten Tweet in einem neuen Objekt „quoted\_status\_permalink“ auf Root- (oder Top‑)Ebene, also auf derselben Ebene wie das „quoted\_status“-Objekt.

<div id="quote-tweets">
  ##### Erweiterte Posts
</div>

JSON, das _erweiterte Posts_ beschreibt, wurde eingeführt, als im November 2017 Posts mit 280 Zeichen gestartet wurden. Das Post-JSON wurde erweitert, um diese längeren Nachrichten abzubilden, ohne die Tausenden von Apps zu beeinträchtigen, die diese grundlegenden X-Objekte parsen. Um vollständige Abwärtskompatibilität zu gewährleisten, wurden das ursprüngliche 140-Zeichen-Feld „text“ und die daraus extrahierten Entity-Objekte beibehalten. Bei Posts, die länger als 140 Zeichen sind, wird dieses Root-Level-Feld „text“ jedoch abgeschnitten und ist damit unvollständig. Da die Root-Level-Objekte „entities“ Arrays mit wichtigen, aus der „text“-Nachricht extrahierten Metadaten enthalten, wie etwa enthaltene Hashtags und Links, wären diese Sammlungen unvollständig. Wenn zum Beispiel eine Post-Nachricht 200 Zeichen lang ist und am Ende ein Hashtag enthält, würde das Legacy-Root-Level-Array „entities.hashtags“ dieses nicht enthalten.

Ein neues Feld „extended\_tweet“ wurde eingeführt, um die längeren Post-Nachrichten und vollständige Entity-Metadaten aufzunehmen. Das Objekt „extended\_tweet“ stellt das Feld „full\_text“ bereit, das die vollständige, nicht abgeschnittene Post-Nachricht enthält, wenn sie länger als 140 Zeichen ist. Das Objekt „extended\_tweet“ enthält außerdem ein „entities“-Objekt mit vollständigen Arrays von Hashtags, Links, Erwähnungen usw.

Erweiterte Posts werden über ein Root-Level-Boolean „truncated“ identifiziert. Wenn true („truncated“: true), sollten die Felder von „extended\_tweet“ anstelle der Root-Level-Felder geparst werden.

Beachten Sie im JSON-Beispiel unten, dass das Root-Level-Feld „text“ abgeschnitten ist und das Root-Level-Array „entities.hashtags“ leer ist, obwohl die Post-Nachricht drei Hashtags enthält. Da es sich um einen erweiterten Post handelt, ist das Feld „truncated“ auf true gesetzt, und das Objekt „extended\_tweet“ liefert vollständige „full\_text“- und „entities“-Metadaten zum Post.

### Native erweiterte Geo-Objekte

Beiträge können mit einem Standort verknüpft sein und sind dann „geotaggt“. Standorte können über die X-Benutzeroberfläche zugewiesen werden oder beim Veröffentlichen eines Beitrags über die API. Ein Standort kann ein exakter Punkt sein oder ein X Place mit einer „Bounding Box“, die ein größeres Gebiet beschreibt – von einem Veranstaltungsort bis hin zu einer ganzen Region.

Es gibt drei JSON-Objekte auf Root-Ebene, die den einem Beitrag zugeordneten Standort beschreiben: [place](/de/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#place-data-dictionary), [geo](/de/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-geo-objects) und [coordinates](/de/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#coordinates-object-data-dictionary).

Zusätzlich enthält das native erweiterte Format innerhalb des Benutzerobjekts die [abgeleitete Position](/de/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#derived-locations) der [Profil-Geo-Anreicherung](/de/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo).

Das Objekt `place` ist immer vorhanden, wenn ein Beitrag mit einem Ort geotaggt wurde. Places sind konkrete, benannte Orte mit entsprechenden Geokoordinaten. Wenn Nutzer einem Beitrag einen Standort zuweisen, wird ihnen eine Liste möglicher X Places angezeigt. Bei Verwendung der API kann ein X Place angehängt werden, indem beim Posten eine place\_id angegeben wird. Beiträge, die mit Places verknüpft sind, müssen nicht zwingend von diesem Ort stammen, sondern können sich auch auf diesen Ort beziehen.

Die Objekte `geo` und `coordinates` sind nur vorhanden (nicht null), wenn dem Beitrag ein exakter Standort zugewiesen ist. Wenn ein exakter Standort angegeben ist, liefert das Objekt `coordinates` ein \[long, lat]-Array mit den geografischen Koordinaten, und ein entsprechender X Place wird zugewiesen.

<div id="x-entities">
  ## Einführung
</div>

Entities liefern Metadaten und zusätzliche Kontextinformationen zu Inhalten, die auf X veröffentlicht werden. Der Abschnitt `entities` enthält Arrays mit häufig vorkommenden Bestandteilen von Posts: Hashtags, Benutzererwähnungen, Links, Börsenticker (Symbole), X-Umfragen und angehängte Medien. Diese Arrays sind für Entwickler praktisch beim Einlesen von Posts, da X den Text im Wesentlichen vorverarbeitet bzw. vorgeparst hat. Anstatt diese Entities explizit im Post-Text suchen zu müssen, kann Ihr Parser direkt zu diesem JSON-Abschnitt gehen – und dort sind sie.

Über die reine Parsing-Erleichterung hinaus liefert der Abschnitt `entities` auch nützliche Mehrwert-Metadaten. Wenn Sie beispielsweise die [Enhanced URLs Enrichment](/de/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls) verwenden, enthalten die URL-Metadaten vollständig expandierte URLs sowie zugehörige Website-Titel und -Beschreibungen. Ein weiteres Beispiel: Bei Benutzererwähnungen enthalten die Entities-Metadaten die numerische Benutzer-ID, was bei Anfragen an viele X-APIs nützlich ist.

Jede Post-JSON-Nutzlast enthält einen Abschnitt `entities` mit dem Mindestumfang der Attribute `hashtags`, `urls`, `user_mentions` und `symbols`, selbst wenn keine dieser Entities Teil der Post-Nachricht sind. Wenn Sie beispielsweise das JSON für einen Post mit dem Text „Hello World!“ und ohne angehängte Medien untersuchen, enthält das JSON des Posts den folgenden Inhalt mit Entity-Arrays, die null Elemente enthalten:

```
"entities": {
        "hashtags": [
        ],
        "urls": [
        ],
        "user_mentions": [
        ],
        "symbols": [
        ]
      }
```

Hinweise:

- media- und polls-Entities erscheinen nur, wenn dieser Inhaltstyp Teil des Posts ist.
- Wenn Sie mit nativen Medien (Fotos, Videos oder GIFs) arbeiten, ist das [Extended Entities object](/de/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#x-extended-entities) der richtige Ansatz.

### Entities-Objekt

Die Abschnitte `entities` und `extended_entities` bestehen beide aus Arrays von Entity-_Objekten_. Unten finden Sie Beschreibungen zu jedem dieser Entity-Objekte, einschließlich Datendictionaries, die die Attributnamen, Typen und Kurzbeschreibungen des Objekts erläutern. Wir geben außerdem an, welche PowerTrack-Operatoren auf diese Attribute passen, und fügen einige Beispiel-JSON-Payloads bei.

Eine Sammlung gebräuchlicher Entities in Posts, einschließlich Hashtags, Links und Benutzererwähnungen. Dieses `entities`-Objekt enthält zwar ein `media`-Attribut, dessen Umsetzung im Abschnitt `entities` ist jedoch nur für Posts mit einem einzelnen Foto vollständig korrekt. Für alle Posts mit mehr als einem Foto, einem Video oder einem animierten GIF verweisen wir auf den Abschnitt `extended_entities`.

### Hashtag-Objekt  

Der Abschnitt `entities` enthält ein Array `hashtags` mit einem Objekt für jedes Hashtag, das im Post-Text vorkommt, und ein leeres Array, wenn keine Hashtags vorhanden sind.

Der PowerTrack-Operator `#` wird verwendet, um mit dem Attribut `text` abzugleichen. Der Operator `has:hashtags` trifft zu, wenn sich mindestens ein Eintrag im Array befindet.

|     |     |     |
| :--- | :--- | :--- |
| Feld | Typ | Beschreibung |
| indices | Array aus Int | Ein Array von Ganzzahlen, das die Offsets innerhalb des Post-Texts angibt, an denen das Hashtag beginnt und endet. Die erste Ganzzahl gibt die Position des #-Zeichens im Post-Text-String an. Die zweite Ganzzahl gibt die Position des ersten Zeichens nach dem Hashtag an. ### Mediaobjekt  

Der Abschnitt `entities` enthält ein `media`-Array mit genau einem Mediaobjekt, wenn ein Mediaobjekt an den Beitrag „angehängt“ wurde. Aus den folgenden Gründen sollte der Abschnitt `extended_entities` zur Verarbeitung nativer Medien eines Beitrags verwendet werden:
\+ Der Medien-`type` lautet immer „photo“, selbst wenn ein Video oder GIF an den Beitrag angehängt ist.
\+ Obwohl bis zu vier Fotos angehängt werden können, wird im Abschnitt `entities` nur das erste aufgeführt.

### Formatierung von Foto-Medien-URLs

Fotomedien auf X können in unterschiedlichen Größen geladen werden. Es ist am besten, das kleinste Bild zu laden, das noch groß genug ist, um in einen bestimmten Bild-Viewport zu passen. Um verschiedene Größen zu laden, müssen das [Size Object](#size) und [media\_url](#media) (oder [media\_url\_https](#media)) in einem bestimmten Format kombiniert werden. Wir verwenden das bereits bereitgestellte Beispielobjekt der [media entity](#entitiesobject), um eine Foto-Medien-URL zu konstruieren.

Die `media_url` oder `media_url_https` können auch allein geladen werden; standardmäßig wird dann die Variante „medium“ verwendet. Bevorzugt ist jedoch, wann immer möglich, eine vollständig formatierte Foto-Medien-URL anzugeben.

Eine Foto-Medien-URL besteht aus drei Teilen:

|     |     |
| :--- | :--- |
| Base URL | Die Basis-URL ist die Medien-URL ohne Dateierweiterung.<br /><br />Zum Beispiel:  <br /><br />"media\_url\_https": "https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg",<br /><br />Die Basis-URL ist dann:  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq_ |
| Format | Das Format ist der Bildtyp, in dem das Foto vorliegt. Mögliche Formate sind _jpg_ oder _png_; sie erscheinen als Erweiterung der Medien-URL.<br /><br />Zum Beispiel:  <br /><br />"media\_url\_https": "https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg",<br /><br />Das Format ist dann: _jpg_ |
| Name | Der Name ist das Feld der zu ladenden Größe.<br /><br />Zum Beispiel:<br /><br />  <br /> "sizes":   <br />   "thumb":   <br />     "h": 150,  <br />     "resize": "crop",  <br />     "w": 150  <br />   ,  <br />   "large": <br />     "h": 1366,  <br />     "resize": "fit",  <br />     "w": 2048  <br />   },  <br />   "medium": <br />     "h": 800,  <br />     "resize": "fit",  <br />     "w": 1200  <br />   },  <br />   "small": <br />     "h": 454,  <br />     "resize": "fit",  <br />     "w": 680  <br />   }  <br /> }  <br />}<br /><br />Der Name beim Laden des großformatigen Fotos wäre: _large_ |

Wir nehmen diese drei Teile (Basis-URL, Format und Name) und kombinieren sie zur zu ladenden Foto-Medien-URL. Es gibt zwei Formate, um Bilder auf diese Weise zu laden: _legacy_ und _modern_. Alle Bildladevorgänge sollten das _legacy_-Format nicht mehr verwenden und stattdessen das _modern_-Format nutzen. Die Verwendung des _modern_-Formats führt zu einer höheren CDN-Trefferquote für den Aufrufer und verbessert somit die Ladezeiten, da Medien seltener erst im Rechenzentrum generiert und geladen werden müssen.

|     |     |
| :--- | :--- |
| Legacy format | Das Legacy-Format ist veraltet. Alle Foto-Medienabrufe sollten auf das moderne Format umgestellt werden.<br /><br />_\<base\_url>.\<format>:\<name>_  <br /><br />Zum Beispiel:  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq.jpg:large  <br />_ |
| Modern format | Das moderne Format zum Laden von Fotos wurde 2015 bei X eingeführt und ist seit 2017 de facto Standard. Alle Foto-Medienabrufe sollten auf dieses Format umgestellt werden.<br /><br />_\<base\_url>?format=\<format>\&name=\<name>_  <br /><br />Zum Beispiel:  <br /><br />_https://pbs.twimg.com/media/DOhM30VVwAEpIHq?format=jpg\&name=large  <br />_<br /><br />Hinweis: Die Parameter in der Query-String der Foto-Medien-URL sind alphabetisch sortiert. Wenn für das Laden von Medien zusätzliche Query-Parameter hinzugefügt würden, müsste die alphabetische Sortierung beibehalten werden. Ein hypothetischer neuer Query-Parameter namens _preferred\_format_ würde in der Query-String nach _format_ und _name_ stehen. |

Wenn Sie die [Enhanced-URL-Enrichment](http://support.gnip.com/enrichments/enhanced_urls.html) verwenden, werden die Operatoren `url_title:` und `url_description:` verwendet, um auf die Attribute `unwound.title` und `unwound.description` abzugleichen.

## Details zu Retweets und Zitaten (Quote Tweets)

Aus Sicht der X-API sind Retweets und Quote Tweets spezielle Arten von Posts, die den ursprünglichen Post als eingebettetes Objekt enthalten. Retweet- und Quote-Tweet-Objekte sind also die Eltern eines untergeordneten „originalen“ Posts (und damit doppelt so groß). Retweets haben ein Top-Level-Objekt "retweeted\_status", und Quote Tweets haben ein "quoted\_status"-Objekt. Zur Konsistenz besitzen diese Top-Level-Objekte für Retweets und Quote Tweets ebenfalls eine Eigenschaft text und zugehörige entities. Allerdings können sich die Entities auf der obersten Ebene von den Entities unterscheiden, die vom eingebetteten „originalen“ Objekt bereitgestellt werden. Bei Retweets wird neuer Text dem ursprünglichen Post vorangestellt. Bei zitierten Posts wird neuer Text an den Post angehängt.

Im Allgemeinen ist es Best Practice, Text, Entities, ursprünglichen Autor und Datum aus dem ursprünglichen Post in retweeted\_status abzurufen, sofern dieser vorhanden ist. Eine Ausnahme ist das Erfassen von X-Entities, die Teil des zusätzlichen Zitats sind. Siehe unten für weitere Details und Tipps.

### Retweets

Ein wichtiges Detail bei Retweets ist, dass dem Post keine zusätzlichen X-Entities hinzugefügt werden können. Nutzende können beim Retweeten keine Hashtags, URLs oder andere Details hinzufügen. Das oberste Textattribut des Retweets besteht jedoch aus dem ursprünglichen Post-Text, dem „RT @username: “ vorangestellt wird.  

In einigen Fällen, insbesondere bei Accounts mit langen Nutzernamen, kann die Kombination dieser neuen Zeichen und des ursprünglichen Post-Texts leicht die ursprüngliche Zeichenlängenbegrenzung von 140 Zeichen überschreiten. Um die Unterstützung für eine auf 140 Zeichen basierende Anzeige und Speicherung beizubehalten, kürzt der oberste Text das Ende des Post-Texts und fügt eine Ellipse („…“) hinzu. Folglich können einige oberste Entities, die am Ende des ursprünglichen Posts positioniert sind, falsch sein oder fehlen, etwa im Fall eines abgeschnittenen Hashtags oder eines teilweise abgeschnittenen URL-Eintrags.

<div id="retweets">
  ### Zitierte Tweets
</div>

Zitierte Tweets wurden 2016 eingeführt und unterscheiden sich von Retweets dadurch, dass Sie beim „Zitieren“ eines Posts neuen Inhalt „oben auf“ einen geteilten Post legen. Dieser neue Inhalt kann nahezu alles enthalten, was ein ursprünglicher Post enthalten kann, einschließlich neuem Text, Hashtags, Erwähnungen und URLs.

Zitierte Tweets können native Medien (Fotos, Videos und GIFs) enthalten und erscheinen unter dem entities-Objekt.

Da X-Entities hinzugefügt werden können, unterscheiden sich die Quote-Entities wahrscheinlich von den ursprünglichen Entities.

In diesem Beispiel wurden eine neue URL und ein Hashtag am Ende des zitierten Tweets platziert.

#### Einführung

Wenn ein Post native Medien enthält (über die Post-Benutzeroberfläche geteilt statt über einen Link anderswo), gibt es auch einen Abschnitt extended\_entities. Bei nativen Medien (Foto, Video oder GIF) ist extended\_entities aus mehreren Gründen die bevorzugte Metadatenquelle. Derzeit können bis zu vier Fotos an einen Post angehängt werden. Die entities-Metadaten enthalten nur das erste Foto (bis 2014 konnte nur ein Foto enthalten sein), während der Abschnitt extended\_entities alle angehängten Fotos umfasst. Bei nativen Medien besteht ein weiterer Nachteil der entities.media-Metadaten darin, dass der Medientyp immer „photo“ angibt – selbst dann, wenn es sich bei den angehängten Medien um ein Video oder ein animiertes GIF handelt. Der tatsächliche Medientyp wird im Attribut extended\_entities.media\[].type angegeben und ist entweder auf _photo_, _video_ oder _animated\_gif_ gesetzt. Aus diesen Gründen ist bei der Arbeit mit nativen Medien die extended\_entities-Metadatenquelle vorzuziehen.

Alle Posts mit angehängten Fotos, Videos und animierten GIFs enthalten ein `extended_entities`-JSON-Objekt. Das `extended_entities`-Objekt enthält ein einzelnes `media`-Array aus `media`-Objekten (siehe den Abschnitt `entities` für dessen Datenwörterbuch). Keine anderen Entity-Typen, wie Hashtags und Links, sind im Abschnitt `extended_entities` enthalten. Das `media`-Objekt im Abschnitt `extended_entities` ist strukturell identisch mit dem im Abschnitt `entities`.

Posts können nur einen Medientyp enthalten. Bei Fotos können bis zu vier Bilder angehängt werden. Bei Videos und GIFs kann jeweils eines angehängt werden. Da die Metadaten `type` im Abschnitt `extended_entities` den Medientyp korrekt angeben („photo“, „video“ oder „animated\_gif“) und bis zu vier Fotos unterstützen, ist dies die bevorzugte Metadatenquelle für native Medien.

```
{
      "extended_entities": {
        "media": [

        ]
      }
    }
```

## Enterprise-Activity-Streams-Datenobjekte

<Note>
  Möchten Sie mehr darüber erfahren, wie das Activity-Streams-Datenformat auf das X-API-v2-Format abgebildet wird?
</Note>

Sehen Sie sich unseren Vergleichsleitfaden an: [Activity Streams im Vergleich zur X-API v2](/de/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2)

<Note>
  Bitte beachten Sie: Für Enterprise-Daten-APIs wird dringend empfohlen, das Format [Enriched Native](/de/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) zu verwenden. 

  - Das Enriched-Native-Format enthält alle seit 2017 hinzugekommenen Metadaten, z. B. [Umfrage-Metadaten](/de/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#poll-metadata), sowie zusätzliche Metriken wie reply\_count und quote\_count.

  - Das Activity-Streams-Format wurde seit dem [Zeichen‑Update](https://blog.x.com/official/en_us/topics/product/2017/Giving-you-more-characters-to-express-yourself.html) im Jahr 2017 nicht mehr mit neuen Metadaten oder Anreicherungen aktualisiert.
</Note>

### Aktivitätsobjekt

Activity Streams ist eine Objektschema-Übersetzung von X’ [ursprünglichem Datenformat](https://developer.x.comhttps://developer.x.com/en/docs/x-api/v1/data-dictionary/overview), erstellt von Gnip, um das Format von Post-Daten und anderen Social-Media-Daten mithilfe des Drittanbieter-[Activity Base Schema (hier beschrieben)](https://activitystrea.ms/head/activity-schema.html) zu „normalisieren“. Posts werden in das Activity-Streams-Schema überführt, einschließlich der Objekttypen note, person, place und service als verschachtelte Objekte. Posts können weitere verschachtelte Post-Aktivitätsobjekte für Retweets oder andere enthalten, darunter twitter\_quoted\_status und long\_object.

Der Basisobjekttyp „activity“ ist dem [Post-Basisobjekt](/de/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) des nativen angereicherten Formats ähnlich. Beispiel-Payloads im Activity-Streams-Format finden Sie [hier](/de/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#activity-streams-payload-examples).

<div id="deprecated-attributes">
  #### Verschachtelte Post-Aktivitätsobjekte
</div>

In mehreren Fällen enthält ein Post-Objekt weitere verschachtelte Posts. Wenn Sie mit verschachtelten Objekten arbeiten, enthält diese JSON-Nutzlast mehrere Objekte, und jedes Post-Objekt kann wiederum eigene Objekte enthalten. Das Objekt auf Root-Ebene enthält Informationen über die Art der Aktion, d. h. ob es sich um einen Retweet oder einen Quote-Tweet handelt, und kann außerdem ein Objekt enthalten, das den „ursprünglichen“ geteilten Post beschreibt. Erweiterte Posts enthalten ein verschachteltes Extended-Objekt, das über 140 Zeichen hinausgeht; dies wurde 2017 verwendet, um Breaking Changes beim Update zu vermeiden. Jedes verschachtelte Objektwörterbuch wird unten beschrieben.

Standortobjekte können entweder im actor-Objekt auf X-Kontoebene oder im profileLocations-Objekt des [gnip-Objekts](/de/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#gnip-object) vorkommen. Standortobjekte haben den Objekttyp place und können einen Namen, eine Adresse oder Geokoordinaten enthalten. Standortobjekte ähneln [Geo](/de/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-geo-objects) im nativen angereicherten Format.

#### Datenwörterbuch für Standortdaten

| Feld | Typ | Beschreibung |
| :--- | :--- | :--- |
| objectType | string | Siehe [hier](http://activitystrea.ms/head/activity-schema.html#place) für detailliertere Informationen. Beispiel:<br /><br />**"objectType":** "place" |
| displayName | string | Der vollständige Name des Standorts.<br /><br />**"displayName":** "United States" |
| name | string | Name des Standorts aus dem Place-JSON-Format von X. |
| link | string | Ein Link zur vollständigen JSON-Repräsentation des Ortes auf X.<br /><br />**"link":** "https://api.x.com/1.1/geo/id/27c45d804c777999.json" |
| geo | object | Das Geokoordinaten-Objekt von X, entweder ein Polygon oder ein Punkt.<br /><br />Siehe [geo](https://developer.x.com/en/docs/x-api/v1/data-dictionary/object-model/geo#coordinates) |
| countryCode | String | Abgekürzter Ländercode für das Land, in dem sich dieser Ort befindet. Beispiel:<br /><br />**"countryCode": "US"** |
| country | String | Name des Landes, in dem sich dieser Ort befindet. Beispiel:<br /><br />**"country": "United States"** |

### Einführung\*\*

Im Kern ist X ein öffentliches, Echtzeit- und globales Kommunikationsnetzwerk. Seit 2006 wurde die Entwicklung von X sowohl durch Nutzungsgewohnheiten und Konventionen der Nutzer als auch durch neue Produktfunktionen und -verbesserungen vorangetrieben. Wenn Sie X-Daten für historische Forschung verwenden, ist es wichtig, die zeitliche Entwicklung zu verstehen, um relevante Beiträge im Datenarchiv zu finden.

X wurde als einfache SMS-Mobile-App gestartet und hat sich zu einer umfassenden Kommunikationsplattform entwickelt – einer Plattform mit einem vollständigen Satz von APIs. APIs waren schon immer eine tragende Säule des X-Netzwerks. Die [erste API wurde kurz nach dem Start von X veröffentlicht](https://blog.x.com/2006/introducing-the-twitter-api). Als 2009 das Geotagging von Beiträgen eingeführt wurde, war dies zunächst über eine [Geo-API](https://blog.x.com/2009/think-globally-tweet-locally) verfügbar (später wurde die Möglichkeit, einen Beitrag zu „geotaggen“, in die X.com-Benutzeroberfläche integriert). Heute treiben die APIs von X das bidirektionale Kommunikationsnetzwerk an, das zur Quelle für Eilmeldungen und Informationsaustausch geworden ist. Die Möglichkeiten, auf diesem globalen Echtzeit-Kommunikationskanal aufzubauen, sind nahezu unbegrenzt.

X stellt zwei historische APIs bereit, die Zugriff auf jeden öffentlich verfügbaren Beitrag bieten: Historical PowerTrack und die [Full-Archive Search API](https://developer.x.com/en/docs/x-api/v1/tweets/search/overview). Beide APIs stellen eine Reihe von _Operatoren_ bereit, mit denen relevante Beiträge abgefragt und gesammelt werden können. Diese Operatoren prüfen eine Vielzahl von Attributen, die jedem Beitrag zugeordnet sind – Hunderte von Attributen wie der Textinhalt des Beitrags, der Kontoname des Autors und im Beitrag geteilte Links. Beiträge und ihre Attribute sind in JSON codiert, einem gängigen textbasierten Datenaustauschformat. Mit der Einführung neuer Funktionen sind neue JSON-Attribute hinzugekommen, und typischerweise wurden neue API-Operatoren eingeführt, um auf diese Attribute abzugleichen. Wenn Ihr Anwendungsfall das _Zuhören_ dessen umfasst, was die Welt auf X gesagt hat, gilt: Je besser Sie verstehen, ab wann Operatoren über die passenden JSON-Metadaten verfügen, desto effektiver können Ihre historischen PowerTrack-Filter sein.

Als Nächstes stellen wir einige Schlüsselkonzepte vor, die den Rahmen dafür bilden, wie Aktualisierungen in Beitragsmetadaten das Auffinden des für Sie relevanten Datensignals beeinflussen.

### Zentrale Konzepte\*\*

#### Von Nutzerkonventionen zu X _First-Class-Objekten_

X Nutzer führten organisch neue, inzwischen grundlegende Kommunikationsmuster im X Netzwerk ein. Ein prägendes Beispiel ist der Hashtag, der heute nahezu universell in allen sozialen Netzwerken verwendet wird. Hashtags wurden eingeführt, um Unterhaltungen und Themen zu organisieren. In einem Netzwerk mit hunderten Millionen Nachrichten pro Tag sind Werkzeuge, um interessante Posts zu finden, entscheidend, und Hashtags sind zu einer zentralen Methode geworden. Kurz nachdem die Nutzung von Hashtags zunahm, erhielten sie offiziellen Status und Unterstützung von X. Als Hashtags zu einem _„First-Class“-Objekt_ wurden, bedeutete das vieles. Es bedeutete, dass Hashtags in der X.com-Benutzeroberfläche anklickbar und durchsuchbar wurden. Es bedeutete auch, dass Hashtags Mitglied der X-_Entities_-Familie wurden, zusammen mit @Mentions, angehängten Medien, Börsensymbolen und geteilten Links. Diese Entities sind praktisch in einem vorgeparsten JSON-Array codiert, was es Entwicklern erleichtert, sie zu verarbeiten, zu scannen und zu speichern.

Retweets sind ein weiteres Beispiel dafür, wie nutzergetriebene Konventionen zu offiziellen Objekten werden. Retweeten entstand als eine Art, Inhalte an andere „weiterzuleiten“. Es begann als manueller Prozess des Kopierens/Einfügens eines Posts und des Voranstellens eines „RT @“-Musters. Dieser Prozess wurde schließlich über einen neuen Retweet-Button automatisiert, komplett mit neuen JSON-Metadaten. Der „offizielle“ Retweet war geboren. Weitere Beispiele umfassen „Mentions“, das Teilen von Medien und Weblinks sowie das Teilen eines Standorts mit Ihrem Post. Jedes dieser Nutzungsmuster führte zu neuen [x.com](https://x.com/)-Benutzeroberflächenfunktionen, neuem unterstützendem JSON und damit neuen Möglichkeiten, Posts zu erkennen. Alle diese grundlegenden Post-Attribute haben zu PowerTrack-Operatoren geführt, die verwendet werden, um genau darauf zu matchen.

<div id="from-user-conventions-to-x-first-class-objects">
  #### Post-Metadaten, Veränderlichkeit, Aktualisierungen und Aktualität
</div>

Während Posts auf eine feste Anzahl von Zeichen begrenzt sind, besteht die JSON-Beschreibung eines Posts aus über 100 Attributen. Dazu gehören unter anderem, wer gepostet hat, zu welchem Zeitpunkt, ob es sich um einen Original-Post oder einen Retweet handelt, sowie ein Array von First-Class-Objekten wie Hashtags, Erwähnungen und geteilten Links. Für das Konto, das gepostet hat, gibt es ein User- (oder Actor-) Objekt mit einer Vielzahl von Attributen, die das Profil des Nutzers und weitere Kontometadaten bereitstellen. Profile enthalten eine kurze biografische Beschreibung, einen „Home“-Standort (Freitext), die bevorzugte Sprache und einen optionalen Website-Link.

Einige Kontometadaten ändern sich nie (z. B. numerische User-ID und Erstellungsdatum), einige ändern sich langsam im Laufe der Zeit, während andere Attribute häufiger wechseln. Menschen wechseln den Job und ziehen um. Unternehmen aktualisieren ihre Informationen. Wenn Sie historische Posts erfassen, ist es wichtig zu verstehen, dass einige Metadaten so vorliegen, wie sie zum Zeitpunkt des Postings waren, und andere Metadaten so, wie sie zum Zeitpunkt der Abfrage sind. 

Bei allen historischen APIs werden die Profilbeschreibung des Nutzers, der Anzeigename und die Profil-„Home“-Attribute auf die Werte zum Zeitpunkt der Abfrage aktualisiert.

<div id="post-metadata-mutability-updates-and-currency">
  #### „Native“ Medien
</div>

X.com und die mobilen X-Apps unterstützen das Hinzufügen von Fotos und Videos zu Posts, indem Sie auf eine Schaltfläche klicken und Ihre Fotogalerien durchsuchen. Da sie jetzt als erstklassige Aktionen integriert sind, werden Videos und Fotos, die auf diese Weise geteilt werden, als „native“ Medien bezeichnet.

Viele Abfrageoperatoren arbeiten mit diesen „nativen“ Ressourcen, darunter `has:videos`, `has:images` und `has:media`. Diese greifen nur bei Medieninhalten, die über X-Funktionen geteilt wurden. Um auf andere, außerhalb der X-Plattform gehostete Medien zu matchen, sollten Sie Operatoren verwenden, die auf URL-Metadaten basieren.

Bevor wir also in die Produktdetails von Historical PowerTrack und der Vollarchivsuche eintauchen, werfen wir einen Blick darauf, wie sich X als Produkt und Plattform im Laufe der Zeit entwickelt hat.

**Tipps zum Filtern**

Wenn Sie mit der X‑Zeitleiste vertraut sind, wann und wie neue Funktionen hinzugefügt wurden, können Sie effektivere Abfragen erstellen. Hier bedeutet eine Abfrage einen _Filter_ oder eine _Regel_, die von den historischen X‑APIs auf das Post‑Archiv angewendet wird, wobei PowerTrack-Operatoren verwendet werden, um auf Post‑JSON abzugleichen. Ein Beispiel ist der Operator `lang:`, der verwendet wird, um Posts in einer angegebenen Sprache zu finden. X bietet einen Sprachklassifizierungsdienst (unterstützt über 50 Sprachen), und die X‑APIs stellen diese Metadaten im JSON bereit, das für jeden Post erzeugt wird. Wenn also ein Post auf Spanisch verfasst ist, wird das JSON‑Attribut „lang“ auf „es“ gesetzt. Wenn Sie einen Filter mit der Klausel `lang:es` erstellen, werden damit nur Post‑Nachrichten erfasst, die als Spanisch klassifiziert sind.

Die Informationen in der Zeitleiste können auch helfen, die empfangenen Post‑Daten besser zu interpretieren. Angenommen, Sie untersuchen die Verbreitung von Inhalten zu den Sommerolympischen Spielen 2008 und 2012. Wenn Sie nur den Operator `is:retweet` anwenden, um Retweets zu erfassen, würden 2008 keine Daten übereinstimmen. Für 2012 gäbe es jedoch wahrscheinlich Millionen von Retweets. Daraus könnten Sie fälschlicherweise schließen, dass Retweets 2008 nicht üblich waren oder einfach niemand zu diesen Olympischen Spielen retweetet hat. Da Retweets 2009 zu einem First-Class-Objekt wurden, müssen Sie eine Regelklausel „RT @“ hinzufügen, um sie 2008 zu identifizieren.

Sowohl Retweets als auch die Sprachklassifizierung von Posts sind Beispiele für Post‑Attribute mit einer langen Historie und vielen Produktdetails. Im Folgenden besprechen wir weitere Details zu diesen und anderen Attributklassen, die für das Matching und das Verständnis von X‑Daten wichtig sind.

<div id="2017">
  #### Erkennen von False Negatives
</div>

Beim Erstellen von Filtern ist ein wichtiger Punkt, dass die Metadaten, auf die Operatoren prüfen, jeweils ein „Einführungsdatum“ haben. Wenn Sie einen Filter mit einem Operator erstellen, der auf Metadaten abzielt, die erst nach Veröffentlichung des Posts eingeführt wurden, erhalten Sie ein False Negative. Wenn Sie beispielsweise an allen Posts interessiert sind, die „snow“ erwähnen und ein Video enthalten: Wenn Sie eine Regel mit dem Operator `has:videos` erstellen, der auf Posts mit _nativen_ Videos passt, trifft diese Klausel auf Posts vor 2015 nicht zu.

Das Teilen von Videos war auf X jedoch schon lange vor 2015 üblich. Zuvor teilten Nutzer Links zu extern gehosteten Videos; 2015 hat X dann neue Funktionen zum direkten „Teilen von Videos“ in die Plattform integriert. Um diese älteren relevanten Posts zu finden, würden Sie eine Regelklausel wie `url:"youtube.com"` hinzufügen.

Beachten Sie, dass es bei den Search-APIs einige Beispiele für nachträglich „aufgefüllte“ Metadaten gibt, als der Index neu aufgebaut wurde. Ein gutes Beispiel sind $cashtags, die seit 2009 weit verbreitet zur Diskussion von Aktiensymbolen verwendet werden. Nachdem der $cashtag-Operator 2015 eingeführt worden war, wurde der Search-Index neu aufgebaut und dabei die Symbol-Entität aus allen Post-Texten extrahiert – auch rückwirkend bis Anfang 2006, als `$` hauptsächlich als Slang verwendet wurde; „I hope it $now$ $oon!“.

<div id="recognizing-false-negatives">
  #### Identifizieren und Filtern von Post-Attributen, die für Ihren Anwendungsfall wichtig sind
</div>

Einige Metadaten, wie numerische IDs von X-Konten, gibt es seit dem ersten Tag (und sie sind ein Beispiel für Kontometadaten, die sich nie ändern). Andere Metadaten wurden erst lange nach dem Start von X im Jahr 2006 eingeführt. Beispiele für später eingeführte Metadaten sind Retweet-Metadaten, Post-Standorte, URL-Titel und -Beschreibungen sowie „native“ Medien. Nachfolgend finden Sie einige der gängigsten Arten von Post-Attributen, die durch diese Updates der X-Plattform grundlegend beeinflusst wurden.

Das Filter- bzw. Matching-Verhalten hängt in den meisten Fällen davon ab, welche historische Post-API verwendet wird. Um Ihnen die Wahl des am besten geeigneten Produkts für Ihre Forschung und Ihren Anwendungsfall zu erleichtern, enthalten die unten aufgeführten Attributedetails überblicksartige Produktinformationen.

<div id="identifying-and-filtering-on-post-attributes-important-to-your-use-case">
  #### X-Profile
</div>

Da X im Kern ein globaler Echtzeit-Kommunikationskanal ist, liegt bei der Forschung mit Post-Daten der Schwerpunkt häufig darauf, wer kommuniziert. Oft ist es hilfreich zu wissen, wo ein X Nutzer beheimatet ist. Häufig kann die Information, dass eine Account-Bio Interessen und Hobbys erwähnt, zu relevanten Posts führen. Es ist sehr üblich, Posts von interessanten Accounts gezielt zu beobachten. Profilattribute sind für all diese Anwendungsfälle entscheidend.

Jedes Konto auf X hat ein Profil, das Metadaten wie X-@Handle, Anzeigename, eine kurze Bio, Heimatort (Freitext, der von einem Benutzer eingegeben wird), Anzahl der Follower und vieles mehr enthält. Einige Attribute ändern sich nie, etwa die numerische Benutzer-ID und das Erstellungsdatum des Kontos. Andere ändern sich üblicherweise von Tag zu Tag, Woche zu Woche oder Monat zu Monat, wie die Anzahl der veröffentlichten Posts sowie die Anzahl der gefolgten Accounts und Follower. Weitere Kontoattribute können sich ebenfalls jederzeit ändern, neigen jedoch zu selteneren Anpassungen: Anzeigename, Heimatort und Bio.

Die JSON-Nutzlast für jeden Post enthält Metadaten zum _Konto-Profil_ des Autors. Handelt es sich um einen Retweet, enthält sie auch Profilmetadaten für den Account, der den ursprünglichen Post veröffentlicht hat.

Die Veränderlichkeit der Profilmetadaten eines Posts hängt vollständig vom verwendeten historischen Produkt ab. Die Search-APIs liefern historische Posts mit den Profileinstellungen _so, wie sie zum Zeitpunkt des Abrufs sind_. Für Historical PowerTrack gilt: Das Profil ist _so, wie es zum Zeitpunkt der Veröffentlichung des Posts war_, außer für Daten vor 2011. Für Posts, die älter als 2011 sind, spiegeln die Profilmetadaten das Profil wider, wie es im September 2011 war.

<div id="x-profiles">
  #### Original-Post und Retweets
</div>

Retweets sind ein weiteres Beispiel dafür, wie nutzergetriebene Konventionen zu offiziellen Objekten werden. Retweeten entstand als Methode, Inhalte an andere „weiterzuleiten“. Es begann als manueller Prozess des Kopierens/Einfügens eines Posts und des Voranstellens eines „RT @“-Musters. Dieser Prozess wurde schließlich über eine neue Retweet-Schaltfläche automatisiert, ergänzt um neue JSON-Metadaten. Der „offizielle“ Retweet war geboren, und die Aktion des Retweetens wurde zu einem erstklassigen Post-Ereignis. Zusammen mit der neuen Retweet-Schaltfläche wurden neue Metadaten eingeführt, etwa die vollständige Payload des Original-Posts.

Ob ein Post original oder geteilt ist, ist ein häufig verwendeter Filter. In manchen Fällen wird nur Originalinhalt benötigt. In anderen Fällen steht die Post-Interaktion im Vordergrund, sodass Retweets entscheidend sind. Der PowerTrack-Operator `is:retweet` ermöglicht es, Retweets entweder ein- oder auszuschließen. Beim Abrufen von Daten aus der Zeit vor August 2009 sind zwei Strategien für das Abgleichen (bzw. Nicht-Abgleichen) von Retweets erforderlich. Vor August 2009 muss die eigentliche Post-Nachricht mittels exaktem Phrasenabgleich auf Übereinstimmungen mit dem Muster „RT @ “ geprüft werden. Für Zeiträume nach August 2009 steht der Operator `is:retweet` zur Verfügung.

<div id="original-post-and-retweets">
  #### Sprachklassifizierungen von Posts
</div>

Die Sprache, in der ein Post verfasst ist, ist von allgemeinem Interesse. Die Postsprache kann helfen, den Standort eines Posts abzuleiten, und oft wird für Analyse oder Anzeige nur eine bestimmte Sprache benötigt. (X-Profile haben außerdem eine bevorzugte Spracheinstellung.)

Für das Filtern nach der Sprachklassifizierung eines Posts unterscheiden sich X’ historische Produkte ([Search API](https://developer.x.com/en/docs/x-api/v1/tweets/search/overview/enterprise) und Historical PowerTrack) erheblich. Beim Aufbau des Sucharchivs wurden alle Posts nachträglich mit der X-Sprachklassifizierung versehen. Daher ist der Operator `lang:` für das gesamte Postarchiv verfügbar. Bei Historical PowerTrack sind die Metadaten der X-Sprachklassifizierung im Archiv ab dem 26. März 2013 verfügbar. 

<div id="post-language-classifications">
  #### Georeferenzierung von Posts
</div>

Zu wissen, wo ein Post veröffentlicht wurde (also ihn zu georeferenzieren), ist für viele Anwendungsfälle wichtig. Es gibt drei Hauptmethoden zur Georeferenzierung von Posts:

- Geografische Verweise im Inhalt eines Posts
- Vom Nutzer mit Geotags versehene Posts
- Vom Nutzer im Account-Profil festgelegter „Home“-Standort

##### Geografische Verweise in einer Post-Nachricht

Das Matching über geografische Verweise in der Post-Nachricht ist zwar häufig die anspruchsvollste Methode, da es lokales Wissen erfordert, bietet aber eine Möglichkeit für das gesamte Post-Archiv. Hier ein Beispiel für einen georeferenzierten Treffer aus dem Jahr 2006 für den Großraum San Francisco auf Basis eines „golden gate“-Filters:

https://x.com/biz/statuses/28311

<div id="geographical-references-in-a-post-message">
  ##### Vom Benutzer geo-getaggte Posts
</div>

Im November 2009 führte X seine Post-Geotagging-API ein, die es ermöglichte, Posts mit einem exakten Standort zu versehen. Im Juni 2010 führte X „X Places“ ein, die ein geografisches Gebiet auf Ebene eines Veranstaltungsorts, einer Nachbarschaft oder einer Stadt darstellen. Ungefähr 1–2 % der Posts sind mit einer der beiden Methoden geo-getaggt.

Der verfügbare Geotagging‑Verlauf hängt von der verwendeten Historical API ab. Bei den Search APIs begann die Möglichkeit, Posts mit einigen Geo Operators ab März 2010 zu matchen, und mit anderen ab Februar 2015. Wenn Sie Historical PowerTrack verwenden, beginnt die Georeferenzierung am 1. September 2011. Beim Aufbau des Historical‑PowerTrack‑Archivs wurde sämtliches Geotagging vor diesem Datum nicht übernommen.

<div id="posts-geo-tagged-by-the-user">
  ##### Vom Nutzer festgelegter „Home“-Standort im Kontoprofil
</div>

Alle X Nutzer haben die Möglichkeit, ihre Profil-Standortangabe festzulegen, die ihren Heimatstandort angibt. Millionen von X Nutzern stellen diese Informationen bereit, was die Menge an Geodaten im X Firehose erheblich erhöht. Diese Standortmetadaten sind eine nicht normalisierte, nutzergenerierte, frei formulierte Zeichenfolge. Ungefähr 30 % der Konten verfügen über Profil-Geo-Metadaten, die auf Länderebene aufgelöst werden können.

Wie bei Post-Geo hängen die Abgleichmethoden und die verfügbaren Zeiträume von der verwendeten Historical API ab. Historical PowerTrack ermöglicht es Nutzern, eigene benutzerdefinierte Abgleiche für diese frei formulierten Zeichenfolgen durchzuführen. Um diesen Prozess zu erleichtern, stellt X außerdem eine [Profile Geo Enrichment](/de/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#profile-geo) bereit, die, wo möglich, die Geokodierung übernimmt und normalisierte Metadaten sowie entsprechende Operatoren bereitstellt. Profile-Geo-Operatoren sind sowohl in Historical PowerTrack als auch in den Search APIs verfügbar. Mit Historical PowerTrack sind diese Profil-Geo-Metadaten seit Juni 2014 verfügbar. Mit den Search APIs sind diese Metadaten seit Februar 2015 verfügbar.

#### Geteilte Links und Medien

Das Teilen von Links zu Webseiten, Fotos und Videos war schon immer ein grundlegender X-Anwendungsfall. Früh in der Geschichte von X beinhalteten all diese Aktionen das Einfügen eines URL-Links direkt in die Post-Nachricht. 2011 integrierte X das direkte Teilen von Fotos in die Benutzeroberfläche. 2016 kamen native Videos hinzu.

Vor diesem Hintergrund gibt es eine Vielzahl von Filter-Operatoren zum Abgleichen dieser Inhalte. Es gibt Operatoren, die prüfen, ob Posts geteilte Links, Fotos und Videos enthalten. Da die meisten auf X geteilten URLs verkürzt werden, um weniger Zeichen eines Posts zu benötigen (z. B. generiert von Diensten wie bitly oder tinyurl), stellt X Datenanreicherungen bereit, die eine vollständige, expandierte URL erzeugen, auf die abgeglichen werden kann. Wenn Sie beispielsweise Posts abgleichen möchten, die Links enthalten, in denen X und Frühwarnsysteme diskutiert werden, würde ein Filter mit Bezug auf „severe weather communication“ einen Post mit dieser URL [http://bit.ly/1XV1tG4](http://bit.ly/1XV1tG4) treffen.

Im März 2012 wurde die [Expanded-URL-Anreicherung](/de/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#expanded-and-enhanced-urls) eingeführt. Vor diesem Zeitpunkt enthielten die Post-Payloads nur die vom Nutzer bereitgestellte URL. Wenn der Nutzer also eine verkürzte URL einfügte, konnte es schwierig sein, auf (expandierte) interessierende URLs abzugleichen. Sowohl bei Historical PowerTrack als auch bei den Search-APIs sind diese Metadaten ab März 2012 verfügbar.

Im Juli 2016 wurde die Enhanced-URL-Anreicherung eingeführt. Diese erweiterte Version stellt den HTML-Titel und die Beschreibung einer Website in der Post-Payload bereit, zusammen mit Operatoren zum Abgleich darauf. Bei Historical PowerTrack sind diese Metadaten ab Juli 2016 verfügbar. Bei den Search-APIs tauchen diese Metadaten ab Dezember 2014 auf.

Im September 2016 führte X „native Attachments“ ein, bei denen ein nachgestellter geteilter Link nicht auf das Zeichenlimit von 140 pro Post angerechnet wird. Beide URL-Anreicherungen gelten weiterhin für diese geteilten Links.

Weitere produktspezifische Details zur URL-Filterung finden Sie in den entsprechenden Artikeln.
