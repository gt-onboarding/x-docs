<div id="usage-api-twitter-api">
  ## Nutzungs-API | Twitter API
</div>

<div id="overview">
  ### Übersicht
</div>

`Enterprise`

Dies ist das wichtigste und beste Tool, um die Nutzung der verschiedenen APIs unter Ihrem Konto zu überwachen und zu verwalten.

**Wichtiger Hinweis:**

Die von der Usage API zurückgegebenen Nutzungszahlen können aufgrund von Testphasen und anderen Abrechnungsanpassungen von den Angaben auf einer Rechnung abweichen. Alle Zahlen basieren auf deduplizierten Aktivitäten innerhalb eines jeweiligen Tages (in UTC).

<div id="features">
  #### Funktionen
</div>

- Programmatisches Abrufen von Nutzungsdaten, die in der Benutzeroberfläche von console.gnip.com verfügbar sind
- Nutzungsdaten auf Stream-Ebene – liefert Nutzungsdaten auf Stream-Ebene (z. B. Dev und Prod) zusätzlich zur Produktebene
- Granulare und beschreibende Daten – Suchanfragen werden nach Vollarchivsuche und 30‑Tage‑Suche aufgeschlüsselt
- Historische PowerTrack „Days“ und „Jobs“ 

<div id="supported-apis">
  ### Unterstützte APIs
</div>

Nachfolgend finden Sie eine Liste der APIs, die derzeit von der Usage-API unterstützt werden:

- PowerTrack API `enterprise`
- 30-Day Search API `enterprise`
- Full-Archive Search API `enterprise`
- Historical PowerTrack `enterprise`

<div id="limitations">
  ### Einschränkungen
</div>

- Die Usage-API ermöglicht den Zugriff auf Nutzungsdaten seit dem 1. Mai 2018. Ab dem 1. Juli 2019 ermöglicht die Usage-API den Zugriff auf Nutzungsdaten der **vergangenen 13 Kalendermonate**.

- Sie können auf Nutzungsdaten in **Dreimonatsintervallen** zugreifen, die mit fromDate und toDate definiert sind.

Nachfolgend ein Beispiel für Anfrage und Antwort:

```bash
curl -u<username>:<password> \
"https://gnip-api.x.com/metrics/usage/accounts/<account-name>.json?bucket=month"
```

<div id="get-usage">
  ### get-usage
</div>

<div id="methods">
  #### **Methoden**
</div>

| Methode | Beschreibung |
| :--- | :--- |
| [GET /metrics/usage/accounts/{ACCOUNT_NAME}.json](#GETData) | Nutzungsdaten abrufen |

Dabei gilt:

- :account\_name ist der (groß-/kleinschreibungsabhängige) Name, der Ihrem Konto zugeordnet ist, wie auf console.gnip.com angezeigt

<div id="authentication-and-rate-limit">
  #### **Authentifizierung und Rate Limits**
</div>

<div id="authentication">
  ##### **Authentifizierung**
</div>

Alle Anfragen an die Usage-API erfordern die HTTP-Authentifizierung per Basic. Verwenden Sie dazu beliebige E‑Mail-/Passwort-Anmeldedaten, die für Ihr Konto aktiviert sind, um sich bei console.gnip.com anzumelden oder eine Verbindung zu einem Gnip-Stream herzustellen.

<div id="rate-limit">
  ##### **Rate-Limit**
</div>

Die Usage-API setzt ein Rate-Limit von zwei Anfragen pro Minute durch.

<div id="best-practices-limitations">
  #### **Bewährte Verfahren & Einschränkungen**
</div>

<div id="data-availability">
  ##### **Datenverfügbarkeit**
</div>

Nutzungsdaten basieren auf deduplizierten Aktivitäten, die bis zum letzten vollständigen Zeitraum (UTC), in dem Daten verarbeitet wurden, erfasst wurden. Daten werden im Allgemeinen bis auf die Minute genau verarbeitet und aktualisiert, außer in Fällen, in denen Gnip Systeme ausrollt.

- Die Usage-API ermöglicht den Zugriff auf Nutzungsdaten seit dem 1. Mai 2018. Ab dem 1. Juli 2019 ermöglicht die Usage-API den Zugriff auf Nutzungsdaten für die **zurückliegenden 13 Kalendermonate**.

- Sie können auf Nutzungsdaten in **Drei-Monats-Intervallen** zugreifen, definiert mit einem fromDate und einem toDate

<div id="requesting-and-receiving-data">
  #### **Anfordern und Empfangen von Daten**
</div>

Die Usage-API wird verwendet, indem Sie eine HTTP-GET-Anfrage mit HTTP-BASIC-AUTH-Anmeldedaten an den API-Endpunkt Ihres Kontos senden.

<div id="get-request">
  ##### GET-Anfrage:
</div>

Senden Sie eine GET-Anfrage an den folgenden Endpunkt mit Ihren Benutzeranmeldedaten und Ihrem Kontonamen:

https://gnip-api.x.com/metrics/usage/accounts/:account\_name.json

**Zusätzliche Parameter**

|     |     |
| :--- | :--- |
| bucket | Optional. Die Zeiteinheit, für die Nutzungsdaten bereitgestellt werden. Nutzungsdaten können mit täglicher oder monatlicher Granularität zurückgegeben werden.  <br />  <br />Anfragen ohne angegebenen bucket liefern die monatliche Granularität.  <br />  <br />Optionen: 'month' oder 'day' |
| fromDate (YYYYMMDDHHMM) | Optional. Nutzungsdaten sind erst ab dem 1. Mai 2018 verfügbar. Der älteste UTC-Zeitstempel, ab dem Nutzungsdaten bereitgestellt werden. Der Zeitstempel hat Tagesgranularität und ist inklusive (d. h., 201805010000 umfasst den Tag 0501). Anfragen, die andere Werte als '0000' für die Stunden- und Minutengranularität enthalten, werden auf '0000' zurückgesetzt.  <br />  <br />Anfragen ohne fromDate oder toDate liefern Nutzungsdaten nach Monat für den aktuellen Monat und beinhalten eine historische Referenz für die vergangenen zwei Monate.  <br />  <br />**Bitte beachten:** Ab dem 1. Juni 2019 können Sie auf die vergangenen 13 Kalendermonate an Nutzungsdaten zugreifen. Wenn es beispielsweise der 10. Oktober ist, können Sie bis zum 1. September des Vorjahres zurückgreifen.  <br />**Beispiel:** 201810010000 liefert Daten ab dem 1. Oktober 2018, einschließlich des 1. Oktober. |
| toDate (YYYYMMDDHHMM) | Optional. Der neueste UTC-Zeitstempel, bis zu dem Nutzungsdaten bereitgestellt werden. Der Zeitstempel hat Tagesgranularität und ist nicht inklusive (d. h., 201703020000 enthält keine Daten für den Tag 0302). Wenn ein toDate für den aktuellen Tag oder einen zukünftigen Tag angegeben wird, werden Nutzungsdaten bis zum letzten vollen Tag (UTC) zurückgegeben. Anfragen, die andere Werte als '0000' für die Stunden- und Minutengranularität enthalten, werden auf '0000' zurückgesetzt.  <br />  <br />Eine Anfrage ohne toDate wird standardmäßig auf den nächsten bucket gesetzt (morgen für bucket=day und nächsten Monat für bucket=month). Eine Anfrage ohne fromDate und toDate wird standardmäßig auf bucket=month gesetzt und zeigt Daten für den aktuellen Monat plus die zwei unmittelbar vorherigen Monate.  <br />  <br />**Beispiel:** 201703050000 liefert Daten bis zum 5. März 2017, ohne Daten vom 5. März. |

**Beispiel-GET-Anfrage**

Diese Anfrage gibt Daten mit monatlicher Granularität vom 1. März 2017 bis zum 5. März 2017 zurück, ohne Daten vom 5. März 2017 einzuschließen.

curl -u "https://gnip-api.x.com/metrics/usage/accounts/:account\_name.json?bucket=month\&fromdate=201403010000\&toDate=201403150000"

<div id="data-format">
  #### **Datenformat**
</div>

Die folgenden Tabellen beschreiben die Datenstrukturen auf Root-Ebene für Nutzungsdaten, die von der Usage-API zurückgegeben werden. Für Felder mit mehreren Ebenen von Unterfeldern klicken Sie auf die angegebenen Links, um Details zu den Unterfeldern anzuzeigen.

Wenn Sie ein Beispiel für eine vollständige Usage-API-Payload sehen möchten, besuchen Sie bitte [diese Seite](/de/x-api/usage/introduction).

|     |     |
| :--- | :--- |
| **account** | Ein Objekt, das das Konto repräsentiert, für das Nutzungsdaten angefordert wurden. |
| **bucket** | Die Zeiteinheit, für die Nutzungsdaten bereitgestellt werden. Kann entweder „day“ oder „month“ sein. |
| **fromDate** | Der früheste UTC-Zeitstempel, für den Sie Nutzungsdaten abrufen möchten (einschließlich). |
| **toDate** | Der späteste UTC-Zeitstempel, für den Sie Nutzungsdaten abrufen möchten (ausschließlich). |
| **publishers** | Enthält drei primäre Objekte: used, projected und products. |
