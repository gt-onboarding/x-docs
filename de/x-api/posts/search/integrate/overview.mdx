<div id="authentication">
  #### Authentifizierung
</div>

Alle Endpunkte der X-API v2 erfordern, dass Sie Ihre Anfragen mit Anmeldedaten authentifizieren, also mit Keys und Tokens. Für den Endpunkt der jüngsten Suche können Sie entweder OAuth 1.0a User Context, OAuth 2.0 App-Only oder OAuth 2.0 Authorization Code mit PKCE verwenden. Beim Endpunkt der Vollarchivsuche müssen Sie OAuth 2.0 App-Only verwenden.

[OAuth 2.0 App-Only](/de/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3) erfordert lediglich, dass Sie Ihrer Anfrage ein [OAuth 2.0 App Access Token](/de/resources/fundamentals/authentication#bearer-token-also-known-as-app-only) beifügen. Sie können ein App Access Token entweder direkt in einer Developer-App erzeugen oder über den Endpunkt [POST oauth2/token](/de/resources/fundamentals/authentication#post-oauth2-token) generieren.

[OAuth 1.0a User Context](/de/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) erfordert die Verwendung Ihrer API Keys, Benutzer-Access Tokens und einiger weiterer Parameter, um einen [Authorization-Header zu erstellen](https://developer-staging.x.com/resources/fundamentals/authentication/authorizing-a-request), den Sie dann mit Ihrer Anfrage übermitteln. Die Access Tokens müssen dem Benutzer zugeordnet sein, in dessen Namen Sie die Anfrage stellen. Wenn Sie für einen anderen Benutzer ein Set von Access Tokens generieren möchten, muss dieser Ihre App über den [3-legged OAuth-Flow](https://developer-staging.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens) autorisieren.

Bitte beachten Sie, dass OAuth 1.0a schwierig zu verwenden sein kann. Wenn Sie mit dieser Authentifizierungsmethode nicht vertraut sind, empfehlen wir, eine [Bibliothek](/de/x-api/tools-and-libraries/overview) zu nutzen, ein Tool wie Postman einzusetzen oder OAuth 2.0 zur Authentifizierung Ihrer Anfragen zu verwenden. Wenn Sie einen Post oder private Metriken von diesen Endpunkten anfordern möchten, müssen Sie entweder OAuth 1.0a User Context oder OAuth 2.0 Authorization Code mit PKCE verwenden. Dadurch ist sichergestellt, dass Sie die erforderlichen Berechtigungen vom Benutzer haben, dem dieser Inhalt gehört.

[OAuth 2.0 Authorization Code mit PKCE](/de/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) ermöglicht eine feinere Kontrolle über den Scope einer Anwendung sowie Autorisierungsabläufe über mehrere Geräte hinweg. Mit OAuth 2.0 können Sie spezifische, fein granulare Scopes auswählen, die Ihnen im Namen eines Benutzers bestimmte Berechtigungen gewähren.

Um OAuth 2.0 in Ihrer App zu aktivieren, müssen Sie es in den Authentifizierungseinstellungen Ihrer App aktivieren. Diese finden Sie im Bereich App-Einstellungen des Developer-Portals.

<Note>
  **Bitte beachten**

  Wenn Sie die folgenden [Felder](/de/x-api/fundamentals/fields) anfordern, sind OAuth 1.0a User Context oder OAuth 2.0 Authorization Code erforderlich:

  - tweet.fields.non\_public\_metrics
  - tweet.fields.promoted\_metrics
  - tweet.fields.organic\_metrics
  - media.fields.non\_public\_metrics
  - media.fields.promoted\_metrics
  - media.fields.organic\_metrics
</Note>

####

Developer-Portal, Projekte und Entwickler-Apps

Um mit beliebigen X-API-v2-Endpunkten zu arbeiten, müssen Sie sich zunächst [für ein Entwicklerkonto registrieren](https://developer.x.com/en/portal/petition/essential/basic-info), ein Projekt in diesem Konto einrichten und darin eine Entwickler-App erstellen. Die in dieser Entwickler-App generierten Keys und Tokens funktionieren für diese Suchendpunkte.

Sie können Keys und Tokens aus einem Projekt mit jeder [Zugriffsstufe](/de/x-api/getting-started/about-x-api) verwenden, um Anfragen an den Endpunkt für die jüngste Suche zu stellen. Für Anfragen an den Endpunkt der Vollarchivsuche benötigen Sie jedoch ein Projekt mit der Zugriffsstufe Pro oder Enterprise. Wenn Sie [Enterprise-Zugriff](/de/x-api/getting-started/getting-access) haben, stehen Ihnen zusätzliche Funktionen zur Verfügung, darunter weitere Operatoren und längere Abfragen.
 

<div id="rate-limits">
  #### Rate Limits
</div>

Jeden Tag senden viele Tausend Entwickler Anfragen an die X-API. Um das Volumen zu steuern, gelten für jeden Endpoint [Rate Limits](/de/x-api/fundamentals/rate-limits), die die Anzahl der Anfragen begrenzen, die jeder Entwickler im Namen einer App oder eines authentifizierten Nutzers stellen kann.

Für diese Endpoints gelten unterschiedliche Rate Limits, abhängig von der verwendeten Authentifizierungsmethode. Die Rate Limits auf App-Ebene gelten für eine App, die Anfragen mit OAuth 2.0 App-Only stellt, während das Rate Limit auf Nutzer-Ebene für Anfragen gilt, die im Namen des autorisierenden Nutzers mit OAuth 1.0a User Context oder OAuth 2.0 Authorization Code mit PKCE gestellt werden. Beide Rate Limits basieren auf der Anfragerate innerhalb eines 15‑Minuten‑Fensters.

Beispielsweise kann eine App, die OAuth 2.0 App-Only verwendet, um Anfragen an den Endpoint für die jüngste Suche zu stellen, 450 Anfragen (einschließlich Paginierungsanfragen) innerhalb eines 15‑Minuten‑Zeitraums senden. Dieselbe App kann im selben 15‑Minuten‑Zeitraum und mit zwei unterschiedlichen authentifizierten Nutzern (unter Verwendung von OAuth 1.0a User Context oder OAuth 2.0 Authorization Code mit PKCE) bis zu 180 Anfragen (einschließlich Paginierungsanfragen) an den Endpoint für die jüngste Suche für jeden authentifizierten Nutzer stellen.
 

<div id="fields-and-expansions">
  #### Felder und Erweiterungen
</div>

Die X-API v2 ermöglicht es Ihnen, mit [Fields](/de/x-api/fundamentals/fields) und [Expansions](/de/x-api/fundamentals/expansions) genau festzulegen, welche Daten die API zurückgeben soll. Der expansions-Parameter ermöglicht es, in der Nutzlast referenzierte Objekte „aufzuklappen“. Dieses Endpoint erlaubt es Ihnen beispielsweise, Umfrage-, Orts-, Medien- und weitere Objekte über den expansions-Parameter anzufordern.

Die fields-Parameter ermöglichen es Ihnen, genau auszuwählen, welche Felder innerhalb der verschiedenen Datenobjekte Sie erhalten möchten. Standardmäßig enthält das primäre Post-Objekt, das von diesen Endpunkten zurückgegeben wird, die Felder id und text (zusätzlich zu edit\_history\_tweet\_ids für Posts, die nach der Einführung dieser Funktion erstellt wurden). Um zusätzliche Felder wie author\_id oder public\_metrics zu erhalten, müssen Sie diese explizit über die fields-Parameter anfordern. Einige wichtige Felder, die Sie für Ihre Integration in Betracht ziehen sollten, sind unsere Umfragedaten, [Metriken](/de/x-api/fundamentals/metrics), [Post-Anmerkungen](/de/x-api/fundamentals/post-annotations) und die [Konversations-ID](/de/x-api/fundamentals/conversation-id).

Wir haben unserer [X-API-v2-Datendefinition](/de/x-api/fundamentals/data-dictionary) eine Anleitung hinzugefügt, wie Sie [Fields und Expansions gemeinsam verwenden](/de/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions).
 

<div id="metrics">
  #### Metriken
</div>

Die Endpunkte der X-API v2 ermöglichen es Ihnen, Metriken direkt aus den zurückgegebenen Objekten anzufordern, vorausgesetzt, Sie übergeben die entsprechenden [Felder](/de/x-api/fundamentals/fields) mit Ihrer Anfrage.

Es gibt einige Einschränkungen bei Post-Metriken, derer Sie sich bewusst sein sollten, insbesondere im Zusammenhang mit dem Schutz der Privatsphäre der Nutzer und den folgenden Response-Feldern:

- tweet.fields.non\_public\_metrics
- tweet.fields.promoted\_metrics
- tweet.fields.organic\_metrics
- media.fields.non\_public\_metrics
- media.fields.promoted\_metrics
- media.fields.organic\_metrics

Die genannten Felder enthalten private Metrikdaten. Das bedeutet, dass Sie vom Publisher des Posts autorisiert sein müssen, um diese Daten in dessen Namen über den Recent-Search-Endpunkt abzurufen, das heißt, Sie müssen [OAuth 1.0a User Context](/de/resources/fundamentals/authentication) verwenden. Da Sie diese Authentifizierungsmethode nur mit Recent Search verwenden können, können Sie diese Metriken nicht über den Vollarchivsuche-Endpunkt abrufen.

Um beispielsweise `non_public_metrics` für Posts der Benutzer-ID 1234 zu erhalten, müssen Sie Zugriffstokens einbeziehen, die diesem Benutzer zugeordnet sind. Sie können Benutzer Ihre App autorisieren lassen und ein Set von Zugriffstokens erhalten, die ihnen zugeordnet sind, indem Sie den [3-legged OAuth Flow](/de/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) verwenden.

Alle `non_public_metrics`, `organic_metrics` und `promoted_metrics` sind nur für Posts verfügbar, die in den letzten 30 Tagen erstellt wurden. Das bedeutet, dass sich die Ergebnisse automatisch anpassen und nur Posts der letzten 30 Tage enthalten, wenn Sie die genannten Felder anfordern.

Wenn diese Felder angefordert werden, werden nur Posts zurückgegeben, die vom authentifizierten Nutzer verfasst wurden; alle anderen Posts führen zu einer Fehlermeldung.
 

<div id="building-search-queries">
  #### Erstellen von Suchabfragen
</div>

Die zentrale Funktion dieser Endpunkte ist die Verwendung einer einzelnen Suchabfrage, um die an Sie ausgelieferten Posts zu filtern. Diese Abfragen bestehen aus Operatoren, die auf Post- und Nutzerattributen wie Schlagwörtern im Text, Hashtags und URLs matchen. Operatoren können mithilfe boolescher Logik und Klammern zu Abfragen kombiniert werden, um das Matching-Verhalten der Abfrage zu verfeinern.

Weitere Informationen finden Sie in unserem Leitfaden zum [Erstellen einer Abfrage](/de/x-api/posts/search/integrate/build-a-query).

<div id="pagination">
  #### Paginierung
</div>

Diese Endpunkte nutzen Paginierung, damit Antworten schnell bereitgestellt werden. Wenn es mehr Ergebnisse gibt, als in einer einzelnen Antwort enthalten sein können (bis zu 100 Posts bei der jüngsten Suche und 500 bei der Vollarchivsuche), müssen Sie paginieren. Verwenden Sie den Parameter max\_results, um festzulegen, wie viele Ergebnisse pro Seite zurückgegeben werden, und den Parameter pagination\_token, um die nächste Ergebnisseite abzurufen. Weitere Informationen finden Sie in unserem [Leitfaden zur Paginierung](/de/x-api/fundamentals/pagination).

<div id="post-caps">
  #### Post-Limits
</div>

Die Search-Posts-Endpunkte sind in der Anzahl der Posts begrenzt, die sie in einem bestimmten Monat zurückgeben können, unabhängig von der Paginierung.

Unabhängig davon, welchen Search-Endpunkt Sie verwenden, werden die zurückgegebenen Posts auf die projektweiten [Tweet-Limits (Post-Limit)](/de/x-api/fundamentals/post-cap) angerechnet. Die Nutzung wird im Developer-Portal angezeigt, und der „Monat“ beginnt an Ihrem Verlängerungstag des Abonnements, der auf dem [Developer-Portal-Dashboard](https://developer.x.com/en/portal/dashboard) angezeigt wird.

**Post-Edits**

Posts, die für Edits infrage kommen, können bis zu fünfmal innerhalb von 30 Minuten nach Veröffentlichung des ursprünglichen Posts bearbeitet werden. Die Search-Endpunkte liefern stets die neueste Version des Posts. Wenn Sie nur Posts anfordern, die vor 30 oder mehr Minuten veröffentlicht wurden, erhalten Sie immer die finale Version des Posts. Wenn Sie jedoch einen Near-Real-Time-Anwendungsfall haben und Posts abfragen, die in den letzten dreißig Minuten veröffentlicht wurden, könnten diese Posts nach Erhalt noch bearbeitet worden sein. Diese Posts können über Search oder den Post-Lookup-Endpunkt rehydriert werden, um ihren endgültigen Status zu bestätigen. Weitere Informationen dazu, wie Post-Edits funktionieren, finden Sie auf der Seite [Grundlagen zu Post-Edits](/de/x-api/fundamentals/edit-posts).
