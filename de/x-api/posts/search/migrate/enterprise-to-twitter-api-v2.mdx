<div id="enterprise-compared-to-x-api-v2">
  ### Enterprise im Vergleich zur X-API v2
</div>

**Gemeinsamkeiten**

- Paginierung
- Zeitzone
- Unterstützung für Bearbeitungsverlauf und Metadaten von Posts

**Unterschiede**

- Endpoint-URLs
- Anforderungen an App und Projekt
- Verfügbare Zeiträume
- Format der Antwortdaten
- HTTP-Methoden
- Zeitformate in Anfragen
- Anfrageparameter
- Filteroperatoren

<div id="similarities">
  #### Ähnlichkeiten
</div>

**Paginierung**

Während v2 zusätzliche Paginierungsfunktionen bietet (neue Paginierungsparameter, die die Navigation über Post-IDs mittels `since_id` und `until_id` ermöglichen), unterstützen sowohl Enterprise als auch v2 die Paginierung über Zeit (`fromDate` und `toDate` bei Enterprise sowie `start_time` und `end_time` bei v2).
 

**Zeitzone**

Wie im Abschnitt zur Paginierung erwähnt, können Sie bei Enterprise wie auch bei v2 verschiedene Datenseiten über Zeitparameter abrufen. In beiden Fällen verwenden Sie UTC als Zeitzone, wenn Sie diese Parameter nutzen.

**Unterstützung für Bearbeitungshistorie und Metadaten von Posts**

Beide Versionen stellen Metadaten bereit, die eine Bearbeitungshistorie beschreiben. Weitere Details finden Sie in den [Search-API-Referenzen](/de/x-api/posts/recent-search) und auf der Seite [Grundlagen zu Post-Bearbeitungen](/de/x-api/fundamentals/edit-posts). 

<div id="differences">
  #### Unterschiede
</div>

**Endpoint-URLs**

- Enterprise-Endpoints:
  - 30 Tage – `http://gnip-api.x.com/search/30day/accounts/:account_name/:label.json`
  - Vollarchiv – `http://gnip-api.x.com/search/fullarchive/accounts/:account_name/:label.json`
- X-API-v2-Endpoints
  - Kürzlich (7 Tage) – `https://api.x.com/2/tweets/search/recent`
  - Vollarchiv – `https://api.x.com/2/tweets/search/all`

**Anforderungen für App und Projekt**

Die X-API-v2-Endpoints erfordern, dass Sie bei der Authentifizierung Ihrer Anfragen Anmeldedaten aus einem Projekt verwenden. Alle X-API-v1.1-Endpoints können Anmeldedaten von eigenständigen Apps oder von Apps verwenden, die einem Projekt zugeordnet sind.
 

**Verfügbare Zeiträume**

Sowohl die Enterprise-API als auch die X-API v2 bieten Endpoints, mit denen Sie gefilterte Post-Daten über das gesamte Vollarchiv abrufen können.

Die X-API v2 bietet jedoch keinen Endpoint für einen Zeitraum von 30 Tagen wie die Enterprise-API. Stattdessen bietet sie das oben genannte Vollarchiv oder einen Zeitraum von 7 Tagen, was mit Native Enriched zu v2 und [Activity Streams zu v2](/de/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2) übereinstimmt und Ihnen helfen kann, Enterprise-Felder den neueren v2-Feldern zuzuordnen. Dieser Leitfaden liefert Ihnen außerdem die spezifischen Expansions- und Feld-Parameter, die Sie mit Ihrer v2-Anfrage übergeben müssen, um bestimmte Felder zurückzugeben.
 

**Antwortdatenformat**

Einer der größten Unterschiede zwischen dem [Enterprise-Antwortformat](https://developer.x.com/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary) und dem [Format der X-API v2](/de/x-api/fundamentals/data-dictionary) besteht darin, wie Sie auswählen, welche Felder in Ihrer Payload zurückgegeben werden.

Bei der Enterprise Search API erhalten Sie viele der Antwortfelder standardmäßig und haben dann die Option, Parameter zu verwenden, um anzugeben, welche Felder oder Feldgruppen in der Payload zurückgegeben werden sollen.

Die X-API v2 liefert standardmäßig nur die Felder `id` und `text` eines Posts. Um zusätzliche Felder oder Objekte anzufordern, müssen Sie die Parameter [fields](/de/x-api/fundamentals/fields) und [expansions](/de/x-api/fundamentals/expansions) verwenden. Alle Post-Felder, die Sie von diesen Endpoints anfordern, werden im primären Post-Objekt zurückgegeben. Alle erweiterten User-, Media-, Poll- oder Place-Objekte und -Felder werden in einem `includes`-Objekt innerhalb Ihrer Antwort zurückgegeben. Sie können die erweiterten Objekte dem Post-Objekt zuordnen, indem Sie die IDs abgleichen, die sowohl im Post als auch im erweiterten Objekt enthalten sind.

Wir empfehlen Ihnen, mehr über diese neuen Parameter in den jeweiligen Leitfäden zu lesen oder unseren Leitfaden dazu zu verwenden, [wie man fields und expansions verwendet](/de/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions).

Zusätzlich zu den Änderungen bei der Anforderung bestimmter Felder führt die X-API v2 auch neue JSON-Designs für die von den APIs zurückgegebenen Objekte ein, einschließlich [Post](/de/x-api/fundamentals/data-dictionary#tweet)- und [User](/de/x-api/fundamentals/data-dictionary#user)-Objekten.

- Auf Root-Ebene des JSON geben die Standard-Endpoints Post-Objekte in einem `statuses`-Array zurück, während die X-API v2 ein `data`-Array zurückgibt.
- Anstatt auf retweetete und zitierte „statuses“ zu verweisen, spricht die JSON der X-API v2 von retweeteten und zitierten Tweets. Viele Legacy- und veraltete Felder wie `contributors` und `user.translator_type` werden entfernt.
- Anstelle der Verwendung von sowohl `favorites` (im Post-Objekt) als auch `favourites` (im User-Objekt) verwendet die X-API v2 den Begriff Like.
- X übernimmt die Konvention, dass JSON-Werte ohne Wert (zum Beispiel `null`) nicht in die Payload geschrieben werden. Post- und User-Attribute werden nur einbezogen, wenn sie Nicht-Null-Werte haben.
   

Wir haben dem Post-Objekt auch eine neue Reihe von Feldern hinzugefügt, darunter:

- Ein [conversation\_id](/de/x-api/fundamentals/conversation-id)-Feld
- Zwei neue [annotations](/de/x-api/fundamentals/post-annotations)-Felder, einschließlich context und entities
- Mehrere neue [metrics](/de/x-api/fundamentals/metrics)-Felder
- Ein neues `reply_setting`-Feld, das anzeigt, wer auf einen bestimmten Post antworten kann

Und noch ein letzter Hinweis. Die Premium-Antwort enthält auf Root-Ebene ein `requestParameters`-Objekt, das die Parameter enthält, die Sie in Ihrer Anfrage angegeben haben. Die v2-Version enthält stattdessen ein `meta`-Objekt auf Root-Ebene, das `newest_id`, `oldest_id`, `result_count` und `next_token` enthält, falls es eine zusätzliche Ergebnisseite gibt.

**HTTP-Methoden**

Die Enterprise-Version der API ermöglicht es Ihnen, die Anfrage entweder per HTTP-Methode POST mit einem JSON-Body oder per HTTP-Methode GET mit einer Query-String zu übermitteln.

V2 erlaubt nur die Verwendung der HTTP-Methode GET mit einer Query-String.
 

**Formate für Anforderungszeiten**

Die Enterprise-Version dieses Endpunkts verwendet folgendes Datums-/Zeitformat sowohl in den Paginierungsparametern als auch im Antwortfeld `timePeriod`: `YYYYMMDDHHmm`

Der v2-Endpunkt verwendet das ISO-8601/RFC-3339-Datums-/Zeitformat sowohl in den Paginierungsparametern als auch in den Antwortfeldern `start` und `end`: `YYYY-MM-DDTHH:mm:ssZ
 `

**Anfrageparameter**

Im Folgenden finden Sie eine Tabelle der Anfrageparameter für Enterprise und X-API v2:

| Enterprise | Search Posts v2 |
| :--- | :--- |
| query | query |
| maxResults | max\_results |
| fromDate (YYMMDDHHmm) | start\_time (YYYY-MM-DDTHH:mm:ssZ) |
| toDate (YYMMDDHHmm) | end\_time (YYYY-MM-DDTHH:mm:ssZ) |
|     | since\_id |
|     | until\_id |
| next | next\_token oder pagination\_token |

**Filteroperatoren**

Während die Operatoren zwischen Enterprise und der X-API v2 größtenteils identisch sind, gibt es einige Unterschiede in der Verfügbarkeit von Operatoren sowie einige neue Operatoren, die ausschließlich in der X-API v2 eingeführt wurden.

Eine vollständige Tabelle der Operatoren, die für X-API v2, Enterprise sowie Premium und Standard verfügbar sind, finden Sie auf der [Search Posts Migration-Landingpage](/de/x-api/posts/search/migrate/overview).
