<div id="comparing-x-apis-posts-lookup-endpoints">
  ## Vergleich der Endpunkte zur Beitragsabfrage der X-API
</div>

Die v2-Endpunkte zur Beitragsabfrage ersetzen die Standard-v1.1-Endpunkte [GET statuses/lookup](https://developer.x.com/en/docs/twitter-api/v1/tweets/post-and-engage/api-reference/get-statuses-lookup) und [GET statuses/show](https://developer.x.com/en/docs/twitter-api/v1/tweets/post-and-engage/api-reference/get-statuses-show-id). Dieser Leitfaden richtet sich an Entwickler, die von diesen älteren Versionen zur X-API v2 migrieren.

---

<div id="endpoint-comparison-table">
  ## Vergleichstabelle der Endpunkte
</div>

| Beschreibung | Standard v1.1 | X-API v2 |
|:-------------|:---------------|:----------|
| **Unterstützte HTTP-Methoden** | `GET` | `GET` |
| **Host-Domain** | `https://api.x.com` | `https://api.x.com` |
| **Endpoint-Pfad** | `/1.1/statuses/show.json`, `/1.1/statuses/lookup.json` | `/2/tweets` |
| **[Authentifizierung](de/resources/fundamentals/authentication)** | OAuth 1.0a User Context | OAuth 1.0a User Context, OAuth 2.0 App-Only, OAuth 2.0 Authorization Code mit PKCE |
| **Post-[JSON-Format](/de/x-api/fundamentals/data-dictionary)** | Standardformat v1.1 | [Format der X-API v2](/de/x-api/fundamentals/data-dictionary), bestimmt durch die Parameter `fields` und `expansions` (nicht abwärtskompatibel zu v1.1) |
| **Unterstützt die Auswahl spezifischer [Felder](/de/x-api/fundamentals/data-dictionary)** |  | ✔ |
| **Unterstützt die Felder für [Post-Annotationen](/de/x-api/fundamentals/post-annotations)** |  | ✔ |
| **Unterstützt neue [Metrik-](/de/x-api/fundamentals/metrics) Felder** |  | ✔ |
| **Unterstützt das Feld `conversation_id`** |  | ✔ |
| **Stellt Bearbeitungshistorie von Posts bereit** | ✔ | ✔ |
| **Erfordert Zugangsdaten einer [Developer App](/de/resources/fundamentals/developer-apps), die einem [Project](de/resources/fundamentals/projects) zugeordnet ist** |  | ✔ |

<div id="standard-v11-compared-to-x-api-v2">
  ## Standard v1.1 im Vergleich zur X-API v2
</div>

Wenn Sie mit den Standard-Endpunkten v1.1 GET statuses/show und GET statuses/lookup gearbeitet haben, hilft Ihnen dieser Leitfaden, die Gemeinsamkeiten und Unterschiede zwischen den Standard- und den X-API-v2-Endpunkten zur Post-Abfrage zu verstehen.

Möglicherweise interessiert Sie auch unser [visuelles Datenformat-Migrationstool](/de/x-api/migrate/data-format-migration), mit dem Sie die Unterschiede zwischen dem [Datenformat der X-API v1.1](/de/x-api/fundamentals/data-dictionary) und dem [Format der X-API v2](/de/x-api/fundamentals/data-dictionary) schnell erkennen können.

- **Gemeinsamkeiten**
  - OAuth 1.0a User Context
  - Limits für Posts pro Anfrage
  - Unterstützung für Bearbeitungsverlauf und Metadaten von Posts

- **Unterschiede**
  - Endpoint-URLs
  - Anforderungen für App und Projekt
  - Antwortdatenformat
  - Anfrageparameter

<div id="similarities">
  ### Gemeinsamkeiten
</div>

<div id="oauth-10a-user-context-authentication-method">
  #### Authentifizierungsmethode mit OAuth 1.0a im Benutzerkontext
</div>

Der Standard-Endpunkt unterstützt [OAuth 1.0a im Benutzerkontext](/de/resources/fundamentals/authentication), während der neue X-API-v2-Endpunkt zur Post-Abfrage sowohl OAuth 1.0a im Benutzerkontext als auch [OAuth 2.0 App-only](/de/resources/fundamentals/authentication) unterstützt. Wenn Sie zuvor einen der Standardendpunkte zur Post-Abfrage in v1.1 verwendet haben, können Sie bei der Migration auf die Version der X-API v2 weiterhin dieselbe Authentifizierungsmethode nutzen.

Die App-only-Authentifizierung ist wahrscheinlich der einfachste Einstieg. Hinweise zum Erzeugen eines App Access Tokens finden Sie in [diesem Leitfaden zu OAuth 2.0 App-only](/de/resources/fundamentals/authentication).

<div id="posts-per-request-limits">
  #### Beiträge pro Anfrage-Limits
</div>

Der v1.1-Endpunkt [GET statuses/lookup](https://developer.x.com/en/docs/twitter-api/v1/tweets/post-and-engage/api-reference/get-statuses-lookup) erlaubt es, bis zu 100 Beiträge pro Anfrage anzugeben. Dies gilt auch für den Endpunkt GET /tweets. Um 100 Beiträge vollständig anzugeben, verwenden Sie den Parameter `ids` als Query-Parameter mit einer kommagetrennten Liste von [Beitrags-IDs](/de/resources/fundamentals/x-ids).

**Unterstützung für Bearbeitungsverlauf und Metadaten von Beiträgen**

Beide Versionen stellen Metadaten bereit, die einen Bearbeitungsverlauf beschreiben. Weitere Details finden Sie in den API-Referenzen zur Beitragsabfrage und auf der Seite [Grundlagen zum Bearbeiten von Beiträgen](/de/x-api/fundamentals/edit-posts).

<div id="differences">
  ### Unterschiede
</div>

<div id="endpoint-urls">
  #### Endpoint-URLs
</div>

- **Standard-v1.1-Endpunkte:**
  - `https://api.x.com/1.1/statuses/show`
  - `https://api.x.com/1.1/statuses/lookup`

- **X-API-v2-Endpunkte:**
  - `https://api.x.com/2/tweets`
  - `https://api.x.com/2/tweets/:id`

<div id="app-and-project-requirements">
  #### Anforderungen an App und Projekt
</div>

X-API-v2-Endpunkte erfordern für die Authentifizierung Anmeldeinformationen aus einer [Developer-App](/de/resources/fundamentals/developer-apps), die einem [Projekt](/de/resources/fundamentals/projects) zugeordnet ist. X-API-v1.1-Endpunkte können Anmeldeinformationen entweder von eigenständigen Apps oder von Apps verwenden, die einem Projekt zugeordnet sind.

<div id="response-data-format">
  #### Antwortdatenformat
</div>

Ein wesentlicher Unterschied zwischen den Standard-Endpunktversionen v1.1 und der X-API v2 besteht darin, wie Felder im Payload ausgewählt werden.

Bei Standard-Endpunkten sind viele Antwortfelder standardmäßig enthalten; zusätzlich können über Parameter weitere Felder angefordert werden.

Die X-API v2 liefert jedoch standardmäßig nur die Felder `id` und `text` des Posts. Zusätzliche Felder und Objekte erfordern die Verwendung der Parameter [fields](/de/x-api/fundamentals/fields) und [expansions](/de/x-api/fundamentals/expansions). Die erweiterten Felder werden in einem `includes`-Objekt innerhalb der Antwort zurückgegeben und können durch Abgleich der IDs dem primären Post-Objekt zugeordnet werden.

Weitere Informationen zur Verwendung von Fields und Expansions finden Sie im [Leitfaden zur Verwendung von Fields und Expansions](/de/x-api/fundamentals/data-dictionary). Ein [Migrationsleitfaden zum Datenformat](/de/x-api/fundamentals/fields) ordnet außerdem Standard-v1.1-Felder den neueren v2-Feldern zu.

Zusätzlich führt die X-API v2 neue JSON-Designs für Objekte ein, einschließlich des Post- und des [user](/de/x-api/fundamentals/fields)-Objekts:

- Standard-Endpunkte geben Post-Objekte in einem `statuses`-Array zurück, während die X-API v2 ein `data`-Array verwendet.
- Retweetete und zitierte Tweets ersetzen in der X-API v2 die Terminologie „statuses“.
- Neue Terminologie wie `like` ersetzt Begriffe wie `favorites` und `favourites`.
- Attribute ohne Werte (z. B. `null`) sind in Payloads der X-API v2 nicht enthalten.

Das Post-Objekt in der X-API v2 enthält neue Felder wie:

- `conversation_id`
- Zwei neue [annotations](/de/x-api/fundamentals/post-annotations)-Felder (`context` und `entities`)
- Neue [metrics](/de/x-api/fundamentals/metrics)-Felder
- `reply_setting`-Feld, das angibt, wer auf einen bestimmten Post antworten kann

<div id="request-parameters">
  #### Anfrageparameter
</div>

Die folgenden Standardparameter von v1.1 haben Entsprechungen in der X-API v2:

| Standard | X-API v2 |
|:----------|:----------|
| `id`     | `ids`    |

Bestimmte Standardparameter von v1.1 werden in der X-API v2 **nicht** unterstützt:

| Standard            | Kommentar                                                                                                                      |
|:---------------------|:------------------------------------------------------------------------------------------------------------------------------|
| `tweet_mode`        | Durch Felder und Expansions ersetzt.                                                                                           |
| `trim_user`         | Durch Felder und Expansions ersetzt. Verwenden Sie die Expansion `author_id` und `user.fields` für Benutzerdaten.              |
| `include_my_retweet`| Liefert die ID des ursprünglichen Posts für Retweets des authentifizierten Nutzers.                                            |
| `include_entities`  | Verwenden Sie Felder und Expansions, um Entities im Payload zu steuern.                                                        |
| `include_ext_alt_text` | Fügt das Feld `ext_alt_text` in der Media-Entity hinzu, wenn Alt-Text vorhanden ist.                                        |
| `include_card_uri`  | Fügt `card_uri` hinzu, wenn eine Ads-Card angehängt ist.                                                                       |
| `map`               | Gibt in der X-API v2 die Post-ID und eine Fehlermeldung für nicht verfügbare Posts zurück, im Gegensatz zu nullierten Feldern in v1.1. |

<div id="curl-requests">
  ### cURL-Anfragen
</div>

Die folgenden cURL-Anfragen zeigen Standardendpunkte von v1.1 und ihre v2-Pendants. Ersetzen Sie `ACCESS_TOKEN` im Header durch das Zugriffstoken Ihrer App. Für v2-Endpunkte muss das Token zu einer [Developer-App](/de/resources/fundamentals/developer-apps/overview) innerhalb eines [Projekts](/de/resources/fundamentals/projects) gehören.

Die Antwortpayloads von v1.1 unterscheiden sich von v2. Mit v2 können Sie über die Parameter [fields](/de/x-api/fundamentals/fields) und [expansions](/de/x-api/fundamentals/expansions) gezielt zusätzliche Felder anfordern.

**Standard-v1.1-Endpunkt `GET statuses/lookup` und v2-Endpunkt `GET /tweets`**

```bash
curl --request GET \
  --url 'https://api.x.com/1.1/statuses/lookup.json?id=1460323737035677698%2C1460323743339741184' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```

```
  curl --request GET \
  --url 'https://api.x.com/2/tweets?ids=1460323737035677698%2C1460323743339741184&tweet.fields=created_at&expansions=author_id&user.fields=created_at' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```

**Standard-v1.1-Endpunkt `GET statuses/show/:id` und v2-Endpunkt `GET /tweets/:id`**

```
curl --request GET \
  --url 'https://api.x.com/1.1/statuses/show.json?id=1460323737035677698' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```

```
curl --request GET \
  --url 'https://api.x.com/2/tweets/1460323737035677698?tweet.fields=created_at&expansions=author_id&user.fields=created_at' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```
