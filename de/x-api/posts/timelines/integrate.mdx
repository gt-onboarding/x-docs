<div id="helpful-tools">
  ### Nützliche Tools
</div>

Bevor wir einige zentrale Konzepte behandeln, empfehlen wir, eines der folgenden Tools oder Codebeispiele zu verwenden, um die Funktionalität dieser Endpunkte zu testen.

**Codebeispiele**

Möchten Sie diese Endpunkte mit Code in Ihrer bevorzugten Programmiersprache einrichten? Wir stellen mehrere Codebeispiele bereit, die Sie als Ausgangspunkt auf unserer [GitHub-Seite](https://github.com/xdevplatform/Twitter-API-v2-sample-code) nutzen können.

**Bibliotheken**

Nutzen Sie eine unserer vielen [Community-Bibliotheken von Drittanbietern](/de/x-api/tools-and-libraries/overview), um schnell loszulegen. Eine mit den v2-Endpunkten kompatible Bibliothek erkennen Sie am passenden Versions-Tag.

**Postman**

Postman ist ein hervorragendes Tool, um diese Endpunkte zu testen. Jede Postman-Anfrage enthält sämtliche Pfad- und Body-Parameter, damit Sie schnell nachvollziehen können, was verfügbar ist. Weitere Informationen zu unseren Postman-Collections finden Sie auf der Seite [Using Postman](/de/tutorials/postman-getting-started).

<div id="key-concepts">
  ### Zentrale Konzepte
</div>

**Authentifizierung**

Alle Endpunkte der X-API v2 erfordern [authentifizierte](/de/resources/fundamentals/authentication) Anfragen mit einem Satz von Zugangsdaten (Keys und Tokens). Sie können entweder OAuth 1.0a User Context oder OAuth 2.0 Authorization Code mit PKCE verwenden, um Anfragen an diese Endpunkte zu authentifizieren. Für die Nutzer-Post-Timeline und die Nutzer-Erwähnungs-Timeline können Sie OAuth 2.0 App-Only verwenden.

[OAuth 1.0a User Context](/de/resources/fundamentals/authentication#oauth-1-0a-2) erfordert die Nutzung Ihrer API-Keys, der Benutzer-Access-Tokens und einiger weiterer Parameter, um einen [Authorization-Header zu erstellen](https://developer-staging.x.com/resources/fundamentals/authentication/authorizing-a-request), den Sie dann mit Ihrer Anfrage übermitteln. Die Access-Tokens müssen dem Nutzer zugeordnet sein, in dessen Namen Sie die Anfrage stellen. Wenn Sie ein Set von Access-Tokens für einen anderen Nutzer generieren möchten, muss dieser Ihre App über den [3-legged OAuth Flow](https://developer-staging.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens) autorisieren.

Bitte beachten Sie, dass OAuth 1.0a in der Anwendung anspruchsvoll sein kann. Wenn Sie mit dieser Authentifizierungsmethode nicht vertraut sind, empfehlen wir, eine [Bibliothek](/de/x-api/tools-and-libraries/overview) zu verwenden, ein Tool wie Postman zu nutzen oder OAuth 2.0 zur Authentifizierung Ihrer Anfragen zu verwenden. Wenn Sie einen Post oder private Metriken von diesen Endpunkten anfordern möchten, müssen Sie entweder OAuth 1.0a User Context oder OAuth 2.0 Authorization Code mit PKCE verwenden, wodurch sichergestellt wird, dass Sie die entsprechenden Berechtigungen des Nutzers besitzen, dem dieser Inhalt gehört.

[OAuth 2.0 App-Only](/de/resources/fundamentals/authentication#bearer-token-also-known-as-app-only) erfordert lediglich, dass Sie ein [OAuth 2.0 App Access Token](/de/resources/fundamentals/authentication#bearer-token-also-known-as-app-only) mit Ihrer Anfrage übermitteln. Sie können ein App Access Token entweder direkt in einer Developer-App generieren oder eines über den Endpunkt [POST oauth2/token](/de/resources/fundamentals/authentication#post-oauth2-token) erstellen. Sie können dies für die Nutzer-Post-Timeline oder die Nutzer-Erwähnungs-Timeline verwenden.

[OAuth 2.0 Authorization Code mit PKCE](/de/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3) ermöglicht eine feinere Kontrolle über den Scope einer Anwendung sowie Autorisierungsabläufe über mehrere Geräte hinweg. OAuth 2.0 erlaubt die Auswahl spezifischer, feingranularer Scopes, die Ihnen bestimmte Berechtigungen im Namen eines Nutzers geben.

Um OAuth 2.0 in Ihrer App zu nutzen, müssen Sie es in den Authentifizierungseinstellungen Ihrer App im Bereich App-Einstellungen des Developer-Portals aktivieren.

<Info>
  **Bitte beachten**

  Wenn Sie die folgenden [Felder](/de/x-api/fundamentals/fields) anfordern, sind OAuth 1.0a User Context oder OAuth 2.0 Authorization Code erforderlich:

  - `tweet.fields.non_public_metrics`
  - `tweet.fields.promoted_metrics`
  - `tweet.fields.organic_metrics`
  - `media.fields.non_public_metrics`
  - `media.fields.promoted_metrics`
  - `media.fields.organic_metrics`
</Info>

**[](/de/x-api/fundamentals/pagination)Developer-Portal, Projekte und Developer-Apps**

Um mit Endpunkten der X-API v2 zu arbeiten, müssen Sie sich für ein [Developer-Konto registrieren](https://developer.x.com/en/portal/petition/essential/basic-info), innerhalb dieses Kontos ein [Projekt](/de/resources/fundamentals/projects) einrichten und innerhalb dieses Projekts eine [Developer-App](/de/resources/fundamentals/developer-apps) erstellen. Ihre Keys und Tokens in dieser Developer-App funktionieren für diese Timeline-Endpunkte.

**Rate Limits**

Jeden Tag stellen viele tausend Entwickler Anfragen an die X-API. Um das Volumen zu steuern, werden für jeden Endpunkt [Rate Limits](/de/x-api/fundamentals/rate-limits) festgelegt, die die Anzahl der Anfragen begrenzen, die jeder Entwickler im Namen einer App oder eines authentifizierten Nutzers stellen kann.

Für diese Endpunkte gelten unterschiedliche Rate Limits, abhängig von der verwendeten Authentifizierungsmethode. Die Rate Limits auf App-Ebene gelten für eine App, die Anfragen mit OAuth 2.0 App-Only stellt, während das Rate Limit auf Nutzerebene für Anfragen gilt, die im Namen des spezifisch autorisierenden Nutzers mit OAuth 1.0a User Context gestellt werden. Beide Rate Limits basieren auf der Anfragerate innerhalb eines 15-Minuten-Fensters.

Zum Beispiel kann eine App, die für beide dieser Timelines-Endpunkte OAuth 2.0 App-Only-Auth verwendet, innerhalb eines 15‑Minuten-Fensters 1500 Anfragen (einschließlich Paginierungsanfragen) an die Benutzer-Post-Timeline und 450 Anfragen (einschließlich Paginierungsanfragen) an die Benutzer-Erwähnungs-Timeline stellen. Dieselbe App kann im selben 15‑Minuten-Fenster mit zwei unterschiedlichen autorisierten Nutzern (unter Verwendung von OAuth 1.0a User Context) 900 Anfragen (einschließlich Paginierungsanfragen) an die Benutzer-Post-Timeline und 180 Anfragen (einschließlich Paginierungsanfragen) an die Benutzer-Erwähnungs-Timeline pro authentifiziertem Nutzer stellen.

Die umgekehrt chronologische Home-Timeline hat ein nutzerbezogenes Rate-Limit von 180 Anfragen pro 15‑Minuten-Fenster. Mit diesem Endpunkt können Sie alle in den letzten 7 Tagen auf einer Timeline erstellten Posts sowie die neuesten 800 unabhängig vom Erstellungsdatum abrufen.

**Felder und Erweiterungen**

Die X-API v2 ermöglicht es Ihnen, mithilfe von [fields](/de/x-api/fundamentals/fields) und [expansions](/de/x-api/fundamentals/expansions) genau auszuwählen, welche Daten Sie von der API zurückerhalten möchten. Der Parameter expansions erlaubt es, im Payload referenzierte Objekte aufzulösen. Dieser Endpunkt ermöglicht es Ihnen beispielsweise, Umfrage-, Orts-, Medien- und andere Objekte über den expansions-Parameter anzufordern.

Der Parameter fields ermöglicht es Ihnen, genau auszuwählen, welche Felder innerhalb der verschiedenen Datenobjekte Sie erhalten möchten. Standardmäßig enthält das primäre Post-Objekt, das von diesen Endpunkten zurückgegeben wird, die Felder id und text. Um zusätzliche Felder wie author\_id oder public\_metrics zu erhalten, müssen Sie diese ausdrücklich über die fields-Parameter anfordern. Einige wichtige Felder, die Sie für Ihre Integration in Betracht ziehen sollten, sind unsere Umfragedaten, [Metriken](/de/x-api/fundamentals/metrics), [Post-Anmerkungen](/de/x-api/fundamentals/post-annotations) und [Konversations-ID](/de/x-api/fundamentals/conversation-id).

Wir haben unserem [X-API-v2-Datenwörterbuch](/de/x-api/fundamentals/data-dictionary) einen Leitfaden hinzugefügt, [wie man Felder und Erweiterungen gemeinsam verwendet](/de/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions).

**Post-Metriken**

Die X-API-v2-Endpunkte ermöglichen es Ihnen, Post-Metriken direkt aus dem zurückgegebenen Post-Objekt anzufordern, sofern Sie die entsprechenden Felder mit Ihrer Anfrage übermitteln.

Es gibt einige Einschränkungen bei Post-Metriken, derer Sie sich bewusst sein sollten, insbesondere in Bezug auf den Schutz der Nutzerprivatsphäre und die folgenden Antwortfelder:

- `tweet.fields.non_public_metrics`
- `tweet.fields.promoted_metrics`
- `tweet.fields.organic_metrics`
- `media.fields.non_public_metrics`
- `media.fields.promoted_metrics`
- `media.fields.organic_metrics`

Die genannten Felder enthalten private Metrikdaten. Das bedeutet, dass Sie vom Publisher des Posts autorisiert sein müssen, diese Daten in seinem Namen abzurufen, wenn Sie den Endpunkt für die Benutzer-Post-Timeline verwenden; Sie müssen daher [OAuth 1.0a User Context](/de/resources/fundamentals/authentication) oder [OAuth 2.0 Authorization Code Flow with PKCE](/de/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3) verwenden.

Um beispielsweise `non_public_metrics` für die Benutzer-Post-Timeline der Benutzer-ID 1234 zu erhalten, müssen Sie Zugriffstoken einschließen, die diesem Nutzer zugeordnet sind. Sie können Nutzer Ihre App autorisieren lassen und eine Reihe von ihnen zugeordneter Zugriffstoken erhalten, indem Sie den [3-legged OAuth-Flow](/de/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) verwenden.

Wenn Sie die Benutzer-Erwähnungs-Timeline verwenden, stehen die genannten Felder nur zur Verfügung, wenn der erwähnende Autor Ihrer App den Zugriff auf seine privaten Metrikdaten autorisiert hat und Sie bei der Anfrage mit OAuth 1.0a User Context die Zugriffstoken dieses Nutzers verwenden.

Alle `non_public_metrics`, organic\_metrics und promoted\_metrics sind nur für Posts verfügbar, die in den letzten 30 Tagen erstellt wurden. Das bedeutet, dass bei Anforderung der genannten Felder die Ergebnisse automatisch so eingeschränkt werden, dass nur Posts der letzten 30 Tage enthalten sind.

Wenn diese genannten Felder angefordert werden, werden nur Posts zurückgegeben, die vom authentifizierten Nutzer verfasst wurden; für alle anderen Posts wird eine Fehlermeldung zurückgegeben.

**Paginierung**

Diese Endpunkte verwenden Paginierung, damit Antworten schnell zurückgegeben werden. In Fällen, in denen es mehr Ergebnisse gibt, als in einer einzelnen Antwort gesendet werden können (bis zu 100 Posts für die Timelines-Endpunkte), müssen Sie paginieren. Verwenden Sie den Parameter max\_results, um festzulegen, wie viele Ergebnisse pro Seite zurückgegeben werden, und den Parameter pagination\_token, um die nächste Ergebnisseite abzurufen. Weitere Informationen finden Sie in unserem [Paginierungsleitfaden](/de/x-api/fundamentals/pagination).

**Ergebnisse filtern**

Diese Endpunkte enthalten mehrere Parameter, mit denen Sie Ergebnisse filtern können. Mit start\_date und end\_date können Sie die Ergebnisse auf einen bestimmten Zeitraum eingrenzen. Wenn Sie lieber Post-IDs verwenden, um eine bestimmte Menge an Posts auszuwählen, können Sie since\_id und until\_id nutzen. Die Benutzer-Post-Timeline verfügt außerdem über einen exclude-Parameter, mit dem Retweets und Antworten aus Ihren Ergebnissen entfernt werden können. 

**Tweet-Limits (Post-Limit) und Anzahl zurückgegebener Posts**

Die Endpunkte für die Benutzer-Post-Timeline und die Benutzer-Mentions-Timeline sind in der Anzahl der Posts begrenzt, die sie in einem bestimmten Monat zurückgeben können. Der Endpunkt für die umgekehrt chronologische Home-Timeline unterliegt dieser Begrenzung nicht.

Unabhängig davon, welchen Timeline-Endpunkt Sie verwenden, werden die zurückgegebenen Posts auf die projektbezogenen [Tweet-Limits (Post-Limit)](/de/x-api/fundamentals/post-cap) angerechnet. Die Nutzung wird im Developer-Portal angezeigt, und der „Monat“ beginnt an Ihrem Verlängerungstag des Abonnements, der auf dem [Developer-Portal-Dashboard](https://developer.x.com/en/portal/dashboard) angezeigt wird. 

Der Endpunkt für die Benutzer-Post-Timeline gibt nur die jüngsten 3200 Posts zurück, die in der Timeline eines Nutzers veröffentlicht wurden. Wenn Sie start\_time und end\_time auf einen Zeitraum setzen, der Posts umfasst, die über die jüngsten 3200 hinausgehen, erhalten Sie zwar eine erfolgreiche Antwort, jedoch keine Posts.

Es ist außerdem wichtig zu beachten, dass, wenn Sie excludes=replies bei Ihren Anfragen an die Benutzer-Post-Timeline übergeben, nur die jüngsten 800 Posts zurückgegeben werden.

Der Endpunkt für die Benutzer-Mentions-Timeline gibt nur die jüngsten 800 Post-Mentions zurück.

Der Endpunkt für die umgekehrt chronologische Home-Timeline gibt die letzten 3200 Posts zurück.

**Post-Bearbeitungen**

Posts, die für Bearbeitungen infrage kommen, können bis zu fünfmal innerhalb von 30 Minuten nach der ursprünglichen Veröffentlichung bearbeitet werden. Die Suchendpunkte liefern stets die neueste Version des Posts. Wenn Sie nur Posts anfordern, die vor 30 oder mehr Minuten veröffentlicht wurden, erhalten Sie immer die endgültige Version des Posts. Wenn Sie jedoch einen Near-Real-Time-Anwendungsfall haben und Posts abfragen, die in den letzten dreißig Minuten veröffentlicht wurden, könnten diese Posts nach dem Abruf noch bearbeitet worden sein. Diese Posts können mit der Suche oder dem Post-Lookup-Endpunkt rehydriert werden, um ihren endgültigen Status zu bestätigen. Um mehr darüber zu erfahren, wie Post-Bearbeitungen funktionieren, siehe die Seite [Grundlagen zu „Posts bearbeiten“](/de/x-api/fundamentals/edit-posts).  

**Randfälle**

- Wenn Sie nicht öffentliche Metriken über den Endpunkt der Benutzer-Post-Timeline für Posts anfordern, die älter als 30 Tage sind, kann es sein, dass Sie ein next\_token in der Antwort mit einer Ergebnisanzahl von 0 sehen. Um dieses Problem zu vermeiden, stellen Sie sicher, dass der mit dem Parameter non\_public\_metrics angeforderte Zeitraum innerhalb der letzten 30 Tage liegt. Zusätzlich sollte der Mindestwert für max\_results 10 sein. Diese Maßnahmen können helfen, dieses Szenario zu vermeiden, es kann jedoch dennoch auftreten.

- Das Anfordern von Promoted-Metriken für Posts, die nicht beworben wurden, gibt eine leere Antwort zurück, statt Post-Daten. Unser Team arbeitet derzeit an der Behebung dieses Problems.

- Bei einem Retweet mit einem Post-Text von mehr als 140 Zeichen wird das Feld text abgeschnitten, statt den vollständigen Post-Text zurückzugeben. Als kurzfristigen Workaround können Sie den referenzierten Post expandieren und den vollständigen Text aus der Expansion abrufen. Dies ist ein Bug, den wir in Zukunft beheben werden.
