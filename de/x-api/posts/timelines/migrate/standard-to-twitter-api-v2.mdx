<div id="standard-v11-timelines-to-x-api-v2-timelines">
  ## Standard-v1.1-Timelines zu X-API-v2-Timelines
</div>

Wenn Sie mit den v1.1-Timeline-Endpunkten (statuses/user\_timeline und statuses/mentions\_timeline) gearbeitet haben, hilft Ihnen dieser Leitfaden, die Gemeinsamkeiten und Unterschiede zwischen den Standard- und den X-API-v2-Timeline-Endpunkten zu verstehen, damit Sie Ihre aktuelle Integration auf die neue Version migrieren können.

- **Gemeinsamkeiten:**
  - Authentifizierung:
    - OAuth 1.0a User Context (umgekehrt chronologische Startseiten-Timeline, Nutzer-Post-Timeline und Nutzer-Erwähnungen-Timeline)
    - OAuth 2.0 App-Only (Nutzer-Post-Timeline)
  - Historisches Zugriffslimit: Nutzer-Timeline (Nutzer-Post-Timeline) bietet Zugriff auf die jüngsten 3200 Posts; Erwähnungen-Timeline (Nutzer-Erwähnungen-Timeline) bietet Zugriff auf die jüngsten 800 Erwähnungen.
  - Unterstützung für Post-Bearbeitungshistorie und Metadaten
  - Rate Limits (Nutzer-Post-Timeline)
  - Refresh-Polling: Möglichkeit, neue Ergebnisse seit der since\_id abzurufen
  - Navigieren durch Timelines anhand von Post-IDs
  - Ergebnisspezifikationen:
    - Reihenfolge der Ergebnisse: in umgekehrt chronologischer Reihenfolge
    - Möglichkeit, Antworten auszuschließen (nur Nutzer-Post-Timeline)
    - Möglichkeit, Retweets auszuschließen (nur Nutzer-Post-Timeline)
- **Unterschiede**
  - Neue Authentifizierungsoptionen:
    - OAuth 2.0 App-Only (Nutzer-Erwähnungen-Timeline)
    - OAuth 2.0 Authorization Code Flow mit PKCE (umgekehrt chronologische Startseiten-Timeline, Nutzer-Post-Timeline und Nutzer-Erwähnungen-Timeline)
  - Zugriffsanforderungen: Anforderungen für X-API-v2-App und -Projekt
  - Rate Limits (Nutzer-Erwähnungen-Timeline und umgekehrt chronologische Startseiten-Timeline)
  - Unterschiedliche Request-Limits und Volumenlimits
  - Zusätzliche Methode zur Paginierung
    - Unterschiedliche max\_results (count) pro Antwort
  - Antwortdatenformat
  - Request-Parameter
    - Anpassbares Datenformat basierend auf Request-Parametern, einschließlich v2-Feldern und Expansions
    - Zusätzlich verfügbare Daten: Metriken, Post-Anmerkungen, Umfragen

<div id="similarities">
  ### Gemeinsamkeiten
</div>

**Authentifizierung**

Die v1.1 statuses/user\_timeline und der X-API-v2-Endpunkt für die Benutzer-Post-Timeline unterstützen sowohl [OAuth 1.0a User Context](/de/resources/fundamentals/authentication) als auch [OAuth 2.0 App-Only](/de/resources/fundamentals/authentication#bearer-token-also-known-as-app-only). Daher können Sie bei der Migration zur X-API v2 weiterhin dieselbe Authentifizierungsmethode und dieselben Autorisierungstokens verwenden.

**Historischer Zugriff**

Die v1.1 statuses/user\_timeline und der X-API-v2-Endpunkt für die Benutzer-Post-Timeline geben beide die aktuellsten 3.200 Posts zurück, einschließlich Retweets.

Die v1.1 statuses/mentions\_timeline und der X-API-v2-Endpunkt für die Benutzer-Erwähnungs-Timeline können die aktuellsten 800 Posts zurückgeben.

**Unterstützung für Post-Bearbeitungshistorie und Metadaten**

Beide Versionen stellen Metadaten bereit, die eine etwaige Bearbeitungshistorie beschreiben. Weitere Details finden Sie in den [API-Referenzen zum Gefilterten Stream](/de/x-api/posts/filtered-stream#api-reference-index) und auf der Seite [Grundlagen zu Post-Bearbeitungen](/de/x-api/fundamentals/edit-posts).

**Rate Limits**

|     |     |
| :--- | :--- |
| **Standard v1.1** | **X API v2** |
| user\_timeline:<br /><br />900 Anfragen pro 15 Min mit OAuth 1.0a User Context<br /><br />1500 Anfragen pro 15 Min mit OAuth 2.0 App-Only | User-Posts-Timeline:<br /><br />900 Anfragen pro 15-Minuten-Fenster mit OAuth 1.0a User Context<br /><br />1500 Anfragen pro 15-Minuten-Fenster mit OAuth 2.0 App-Only |

**Aktualisierungspolling mit since\_id**

Beide Versionen erlauben das Abfragen aktueller Ergebnisse mit since\_id.

**Durchlaufen von Timelines anhand von Post-IDs**

Beide Endpunkte können Timelines mithilfe von Post-ID-„Zeitstempeln“ durchlaufen, basierend auf der Art, wie Post-IDs aufgebaut sind. Die Funktionalität ist im Allgemeinen gleich, mit Ausnahme der folgenden Punkte:

|     |     |
| :--- | :--- |
| **Standard-Timelines v1.1** | **Timelines v2** |
| since\_id (exklusiv)<br /><br />max\_id (inklusiv) | since\_id (exklusiv)<br /><br />until\_id (ebenfalls exklusiv, im Gegensatz zu max\_id, das inklusiv war) |

**Antwortfilter-Parameter**

|     |     |
| :--- | :--- |
| **Standard-Timelines v1.1** | **Timelines v2** |
| Antwortfilter-Parameter:<br /><br />\* include\_rts<br />\* exclude\_replies | Antwortfilter-Parameter:<br /><br />\* exclude=retweets,replies |
| Beispiel <br /><br />https://api.x.com/1.1/statuses/user\_timeline.json?user\_id=2244994945\&include\_rts=0&\&exclude\_replies=1 | Beispiel:<br /><br />https://api.x.com/2/users/2244994945/tweets?max\_results=100\&exclude=retweets,replies |
| Hinweise:<br /><br />Für user\_timeline:<br /><br />\* Die Verwendung von include\_rts=0 ändert nicht das mögliche historische Post-Limit der aktuellsten 3.200 | Hinweise:<br /><br />Für die User-Posts-Timeline:<br /><br />\* Die Verwendung von exclude=retweets ändert nicht das mögliche historische Post-Limit der aktuellsten 3.200 <br />\* Die Verwendung von exclude=replies reduziert das mögliche historische Post-Limit auf die aktuellsten 800 Antworten |

<div id="differences">
  ### Unterschiede
</div>

**Authentifizierung**

**Der v1.1-Endpoint statuses/mentions\_timeline unterstützt nur [OAuth 1.0a User Context](https://developer-staging.x.com/resources/fundamentals/authentication). Der X-API v2-Endpoint für die Benutzer-Erwähnungstimeline unterstützt [OAuth 1.0a User Context](/de/resources/fundamentals/authentication), [OAuth 2.0 App-Only](/de/resources/fundamentals/authentication#bearer-token-also-known-as-app-only) und [OAuth 2.0 Authorization Code mit PKCE](/de/resources/fundamentals/authenticationoauth-2-0/authorization-code "This method allows an authorized app to act on behalf of the user, as the user. It is typically used to access or post public information for a specific user, and it us useful when your app needs to be aware of the relationship between a user and what this endpoint returns. Click to learn how to authenticate with OAuth 2.0 Authorization Code with PKCE.")**

Wenn Sie über den X-API v2-Endpoint für die Benutzer-Post-Timeline auf private oder beworbene Metriken zugreifen möchten, müssen Sie OAuth 1.0a User Context oder OAuth 2.0 Authorization Code mit PKCE verwenden und die Benutzer-Zugriffstoken des Nutzers übergeben, der den Post erstellt hat, für den Sie Metriken abrufen möchten.
 

**Endpoint-URLs**

Beachten Sie, dass die X-API v2-Timeline-Endpoints einen Pfadparameter :id für die Benutzer-ID erfordern.

- Standard-v1.1-Endpoints:
  - https://api.x.com/1.1/statuses/home\_timeline
  - https://api.x.com/1.1/statuses/user\_timeline
  - https://api.x.com/1.1/statuses/mention\_timeline
- X-API v2-Endpoints:
  - https://api.x.com/2/users/:id/timelines/reverse\_chronological
  - https://api.x.com/2/users/:id/tweets
  - https://api.x.com/2/users/:id/mentions

**App- und Projektanforderungen**

Die X-API v2-Endpoints erfordern bei der Authentifizierung Ihrer Anfragen die Verwendung von Zugangsdaten einer [Developer-App](/de/resources/fundamentals/developer-apps), die einem [Projekt](/de/resources/fundamentals/projects) zugeordnet ist. Alle X-API v1.1-Endpoints können Zugangsdaten aus eigenständigen Apps oder Apps verwenden, die einem Projekt zugeordnet sind. 
 

**Rate Limits**

|     |     |
| :--- | :--- |
| **mentions\_timeline:**<br /><br />75 Anfragen pro 15 Minuten mit OAuth 1.0a User Context | **user mention timeline:**<br /><br />180 Anfragen pro 15-Minuten-Fenster mit OAuth 1.0a User Context<br />450 Anfragen pro 15-Minuten-Fenster mit OAuth 2.0 Bearer Token |
| **home\_timeline:**<br /><br />15 Anfragen pro 15 Minuten<br /><br />Bis zu 800 Posts sind in der Home-Timeline abrufbar | **Reverse-chronologische Home-Timeline:**<br /><br />180 Anfragen pro 15 Minuten<br /><br />Sie können alle in den letzten 7 Tagen auf einer Timeline erstellten Posts sowie die jüngsten 800, unabhängig vom Erstellungsdatum, abrufen. |

**Verbrauchsvolumenlimits**

|     |     |
| :--- | :--- |
| **Standard-Timelines v1.1** | **Timelines v2** |

**X-API-v2-JSON-Format**

Die X-API v2 führt neue JSON-Designs für die von den APIs zurückgegebenen Objekte ein, einschließlich [Post](/de/x-api/fundamentals/data-dictionary#tweet)- und [user](/de/x-api/fundamentals/data-dictionary#user)-Objekten. Weitere Informationen über das X-API-v2-Format sowie zur Verwendung von Feldern und Expansions finden Sie in unserem [Leitfaden](/de/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) und in unserem umfassenderen [Datenwörterbuch](/de/x-api/fundamentals/data-dictionary).

- Auf der JSON-Root-Ebene geben die Standard-Endpunkte Post-Objekte in einem statuses-Array zurück, während die X-API v2 ein data-Array zurückgibt. 
- Anstelle der Bezeichnung „statuses“ für Retweeted und Quoted verweist das X-API-v2-JSON auf Retweeted und Quoted Tweets. Viele Legacy- und veraltete Felder, wie contributors und user.translator\_type, werden entfernt. 
- Anstelle der Nutzung von favorites (im Post-Objekt) und favorites (im user-Objekt) verwendet die X-API v2 den Begriff like. 
- X übernimmt die Konvention, dass JSON-Werte ohne Wert (zum Beispiel null) nicht in die Payload geschrieben werden. Post- und User-Attribute werden nur einbezogen, wenn sie einen Nicht-null-Wert haben.
   

Einer der größten Unterschiede zwischen Standard v1.1 und den X-API-v2-Endpunktversionen ist, wie Sie auswählen, welche Felder in Ihrer Payload zurückgegeben werden. Für die Standard-Endpunkte gibt es mehrere Parameter, mit denen Sie festlegen konnten, welche Felder oder Feldsätze in der Payload zurückgegeben werden, während die X-API-v2-Version diese unterschiedlichen Parameter in [fields](/de/x-api/fundamentals/fields) und [expansions](/de/x-api/fundamentals/expansions) vereinfacht. 

- fields: X-API-v2-Endpunkte ermöglichen Ihnen auszuwählen, welche Felder in Ihrer Payload bereitgestellt werden. Zum Beispiel haben Post-, User-, Media-, Place- und Poll-Objekte jeweils eine Liste von Feldern, die zurückgegeben werden können (oder nicht).

- expansions: Wird verwendet, um die in Post-JSON-Payloads referenzierten ergänzenden Objekte zu erweitern. Zum Beispiel verweisen alle Retweets und Replies auf andere Posts. Durch Setzen von expansions=referenced\_tweets.id werden diese anderen Post-Objekte gemäß der tweet.fields-Einstellung erweitert. Auch andere Objekte wie Users, Polls und Media können erweitert werden.

- conversation\_id

- Zwei neue [annotations](/de/x-api/fundamentals/post-annotations)-Felder, einschließlich context und entities

- Mehrere neue [metrics](/de/x-api/fundamentals/metrics)-Felder

Wir haben einen [Migrationsleitfaden für Datenformate](/de/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2) zusammengestellt, der Ihnen helfen kann, Standard-v1.1-Felder den neueren v2-Feldern zuzuordnen. Dieser Leitfaden liefert Ihnen auch die spezifischen Parameter expansion und field, die Sie mit Ihrer v2-Anfrage übergeben müssen, um bestimmte Felder zurückzugeben.
