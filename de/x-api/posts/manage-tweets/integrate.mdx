<div id="postman">
  ### Postman
</div>

Postman ist ein hervorragendes Tool, mit dem Sie Endpunkte testen können. Jede Postman-Anfrage enthält alle Pfad- und Body-Parameter, damit Sie schnell nachvollziehen können, was verfügbar ist. Weitere Informationen zu unseren Postman-Collections finden Sie auf der Seite ["Using Postman"](/de/tutorials/postman-getting-started).

<div id="authentication">
  ### Authentifizierung
</div>

Alle Endpunkte der X-API v2 erfordern, dass Sie Ihre Anfragen mit Zugangsdaten authentifizieren, auch bekannt als Keys und Tokens.

Diese Endpunkte erfordern die Verwendung von [OAuth 1.0a User Context](/de/resources/fundamentals/authentication). Das bedeutet, dass Sie API-Keys und Benutzer-Access-Tokens verwenden müssen, um eine erfolgreiche Anfrage zu stellen. Die Access-Tokens müssen dem Benutzer zugeordnet sein, in dessen Namen Sie die Anfrage stellen. Wenn Sie Access-Tokens für einen anderen Benutzer erzeugen möchten, muss dieser Ihre App über den [3-legged OAuth-Flow](/de/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) autorisieren bzw. authentifizieren.

Bitte beachten Sie, dass OAuth 1.0a komplex sein kann. Wenn Sie mit dieser Authentifizierungsmethode nicht vertraut sind, empfehlen wir die Verwendung einer [Bibliothek](/de/x-api/tools-and-libraries/overview), eines Tools wie Postman oder die Authentifizierung Ihrer Anfragen mit OAuth 2.0.

[OAuth 2.0 Authorization Code with PKCE](/de/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) ermöglicht eine feinere Kontrolle über den Umfang (Scope) einer Anwendung sowie Autorisierungsabläufe über mehrere Geräte hinweg. Mit OAuth 2.0 können Sie spezifische, fein granulare Scopes auswählen, die Ihnen bestimmte Berechtigungen im Namen eines Nutzers gewähren.

Um OAuth 2.0 in Ihrer App zu aktivieren, müssen Sie es in den Authentifizierungseinstellungen Ihrer App aktivieren, die Sie im Bereich App-Einstellungen des Developer-Portals finden.

<div id="developer-portal-projects-and-developer-apps">
  ### Developer portal, Projects, and developer Apps
</div>

Um einen Satz von Authentifizierungsdaten zu erhalten, der mit den Endpunkten der X API v2 funktioniert, benötigen Sie ein Entwicklerkonto, richten Sie innerhalb dieses Kontos ein [Projekt](/de/resources/fundamentals/projects) ein und erstellen innerhalb dieses Projekts eine [Entwickler-App](/de/resources/fundamentals/developer-apps). Anschließend finden Sie Ihre Schlüssel und Token in Ihrer Entwickler-App. 
 

<div id="rate-limits">
  ### Rate Limits
</div>

Jeden Tag stellen viele Tausend Entwickler Anfragen an die X-API. Um das enorme Volumen dieser Anfragen zu bewältigen, gelten für jeden Endpoint [Rate Limits](https://developer.x.com/x-api/fundamentals/rate-limits), die die Anzahl der Anfragen begrenzen, die Sie im Namen Ihrer App oder eines authentifizierten Nutzers stellen können.

Diese Endpoints sind auf Nutzerebene ratebegrenzt. Das bedeutet, dass der authentifizierte Nutzer, in dessen Namen Sie die Anfrage stellen, den Endpoint über alle Developer-Apps hinweg nur eine bestimmte Anzahl von Malen aufrufen kann. Es gibt ein Nutzer-Rate-Limit von 200 Anfragen pro 15 Minuten für die POST-Methode. Die DELETE-Methode hat ein Rate-Limit von 50 Anfragen pro 15 Minuten. Zusätzlich gibt es ein Limit von [300 Anfragen pro 3 Stunden](https://blog.x.com/developer/en_us/topics/tools/2018/new-developer-requirements-to-protect-our-platform), einschließlich Posts, die entweder mit manage Posts oder manage Retweets erstellt wurden.
 

<div id="source-labels">
  ### Quell-Labels
</div>

Der Name Ihrer App und die Website-URL werden als [Quell-Label](https://help.x.com/en/using-twitter/how-to-tweet#source-labels) in den Metadaten für alle Posts angezeigt, die programmatisch von Ihrer Anwendung erstellt werden. Wenn Sie den Use Case einer X-App ändern, aktualisieren Sie unbedingt den Use Case in diesen Einstellungen, um die Einhaltung der [Entwicklerbedingungen](https://developer.x.com/content/developer-twitter/en/developer-terms/agreement-and-policy) sicherzustellen.
 

<div id="profile-settings">
  ### Profileinstellungen
</div>

Sie können einem Post nur dann einen Standort hinzufügen, wenn Geodaten in Ihren Profileinstellungen aktiviert sind. Wenn Geodaten nicht aktiviert sind, können Sie zwar weiterhin einen Standortparameter im Request-Body angeben, er wird jedoch nicht mit Ihrem Post verknüpft. Gleiches gilt für das Markieren von Nutzern in Bildern: Wenn der Nutzer, den Sie markieren möchten, die Fotomarkierung nicht aktiviert hat, erscheint sein Name nicht in der Liste der markierten Nutzer, auch wenn der Post erfolgreich erstellt wurde. 
 

<div id="adding-media-to-a-post">
  ### Medien zu einem Post hinzufügen
</div>

Derzeit gibt es in der X-API v2 keine Möglichkeit, Medien vollständig hochzuladen. Sie können jedoch bereits hochgeladene Medien an einen Post anhängen. Verwenden Sie dazu Medien-IDs, die zuvor [über den v1-Medienendpunkt hochgeladen wurden](https://developer.x.com/en/docs/twitter-api/v1/media/upload-media/api-reference/post-media-upload) oder im [X Media Studio](https://media.x.com/en/articles/products/2018/media-studio). Diese Medien-IDs müssen von Ihnen selbst oder von einem authentifizierten Nutzer stammen.
