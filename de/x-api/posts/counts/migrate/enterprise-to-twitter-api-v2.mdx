<div id="enterprise-compared-to-x-api-v2">
  ### Enterprise im Vergleich zur X-API v2
</div>

**Ähnlichkeiten**

- Granularität
- Paginierung
- Zeitzone

**Unterschiede**

- Endpoint-URLs
- Anforderungen an App und Projekt
- Verfügbare Zeiträume
- Format der Antwortdaten
- HTTP-Methoden
- Zeitformate in Anfragen
- Anforderungsparameter
- Filteroperatoren

<div id="similarities">
  #### Gemeinsamkeiten
</div>

**Granularität**

Während der Parameter zur Auswahl der Granularität der zurückgegebenen Daten unterschiedlich ist (`bucket` für die Enterprise-Version, `granularity` für die v2-Version), sind die mit diesem Parameter zulässigen Werte sowie das Standardverhalten identisch:

- `day`
- `hour` (Standard)
- `minute`

**Paginierung**

Während v2 zusätzliche Paginierungsfunktionen bietet (neue Paginierungsparameter, die es ermöglichen, mit Post-IDs über `since_id` und `until_id` zu navigieren), erlauben sowohl Enterprise als auch v2 die Paginierung über die Zeit (`fromDate` und `toDate` bei Enterprise sowie `start_time` und `end_time` bei v2).

Wenn Sie die Enterprise-Version verwenden, nutzen Sie den Parameter `next` zur Paginierung. Das Feld für das nächste Token heißt `next` und befindet sich in der Antwort auf Root-Ebene.

Wenn Sie v2 verwenden, können Sie entweder den Parameter `next_token` oder `pagination_token` zur Paginierung verwenden, und Ihr nächstes Token befindet sich in der Antwort unter `meta.next_token`.
 

**Zeitzone**

Wie im Abschnitt zur Paginierung erwähnt, können Sie sowohl bei Enterprise als auch bei v2 mit Zeit durch verschiedene Datenseiten navigieren. In beiden Fällen verwenden Sie UTC als Zeitzone, wenn Sie diese Parameter nutzen.

<div id="differences">
  #### Unterschiede
</div>

**Endpoint-URLs**

- Enterprise-Endpunkte:
  - 30 Tage – `http://gnip-api.x.com/search/30day/accounts/:account_name/:label/counts.json`
  - Vollarchiv – `http://gnip-api.x.com/search/fullarchive/accounts/:account_name/:label/counts.json`
- X-API-v2-Endpunkte
  - Aktuell (7 Tage) – `https://api.x.com/2/tweets/counts/recent`
  - Vollarchiv – `https://api.x.com/2/tweets/counts/all`

**Anforderung für App und Projekt**

Die Endpunkte der X-API v2 erfordern, dass Sie bei der Authentifizierung Ihrer Anfragen Anmeldedaten aus einer [Developer-App](/de/resources/fundamentals/developer-apps) verwenden, die einem [Projekt](/de/resources/fundamentals/developer-apps) zugeordnet ist. Alle Endpunkte der X-API v1.1 können Anmeldedaten von eigenständigen Apps oder von Apps verwenden, die einem Projekt zugeordnet sind.
 

**Verfügbare Zeiträume**

Sowohl die Enterprise-API als auch die X-API v2 bieten Endpunkte, mit denen Sie das Beitragsvolumen für das Vollarchiv der Posts abrufen können.

Die X-API v2 bietet jedoch keinen Endpunkt für einen Zeitraum von 30 Tagen wie die Enterprise-API. Stattdessen bietet sie das oben genannte Vollarchiv oder einen Zeitraum von 7 Tagen, was den [v2-Search-Posts-Endpunkten](/de/x-api/posts/search/introduction) entspricht.
 

**Format der Antwortdaten**

Es gibt einige geringfügige Unterschiede im Datenformat, das Sie über Enterprise und die X-API v2 erhalten:

- Die Count-Daten von Enterprise befinden sich in einem `results`-Objekt, während sich die v2-Count-Daten in einem `data`-Objekt befinden.
- Die Count-Felder von Enterprise heißen `timePeriod` (Startzeit) und `count`, während v2 den Zeitraum in die Felder `start` und `end` aufteilt (die ein anderes Datums-/Zeitformat als bei Enterprise verwenden, wie in den Zeitformaten für Anfragen erläutert) und das Feld `count` in `tweet_count` umbenennt.
- Die Enterprise-Metadaten enthalten `totalCount`, `next` und das Objekt `requestParameters` auf Root-Ebene. V2 enthält das Objekt `requestParameters` dagegen nicht und verschiebt/benennt Folgendes in ein `meta`-Objekt auf Root-Ebene um: `total_tweet_count` und `next_token`.
   

**HTTP-Methoden**

Die Enterprise-Version der API ermöglicht es, die Anfrage entweder per HTTP-POST mit einem JSON-Body oder per HTTP-GET mit einer Query-String zu übermitteln.

V2 erlaubt nur die Verwendung von HTTP-GET mit einer Query-String.
 

**Zeitformate der Anfrage**

Die Enterprise-Version dieses Endpunkts verwendet das folgende Datums-/Zeitformat sowohl in den Paginierungsparametern als auch im Antwortfeld `timePeriod`: `YYYYMMDDHHmm`

Der v2-Endpunkt verwendet das Datums-/Zeitformat ISO 8601/RFC 3339 sowohl in den Paginierungsparametern als auch in den Antwortfeldern `start` und `end`: `YYYY-MM-DDTHH:mm:ssZ`

**Anfrageparameter**

Im Folgenden finden Sie eine Tabelle der Anfrageparameter für Enterprise und die X-API v2:

| Enterprise | Search Posts v2 |
| :--- | :--- |
| query | query |
| bucket | granularity |
| fromDate (YYMMDDHHmm) | start\_time (YYYY-MM-DDTHH:mm:ssZ) |
| toDate (YYMMDDHHmm) | end\_time (YYYY-MM-DDTHH:mm:ssZ) |
|     | since\_id |
|     | until\_id |
| next | next\_token und pagination\_token |

**Filteroperatoren**

Während die Operatoren zwischen Enterprise und der X-API v2 größtenteils gleich sind, gibt es einige Unterschiede in der Verfügbarkeit und einige neue Operatoren, die nur in der X-API v2 eingeführt wurden.

Eine vollständige Tabelle der für die X-API v2, Enterprise und sogar Premium verfügbaren Operatoren finden Sie auf der [Übersichtsseite zur Migration der Post-Zählungen](/de/x-api/posts/counts/migrate/overview).
