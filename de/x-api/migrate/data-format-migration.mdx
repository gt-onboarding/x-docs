<div id="introduction">
  ## Einführung
</div>

Mit der Einführung der v2-Version der X-API haben wir ein neues Datenrückgabeformat und eine neue Methode zur Anforderung verschiedener Objekte und Felder eingeführt, die wir kurz das X-API-v2-Format nennen. 

Im Abschnitt zu allgemeinen Unterschieden erfahren Sie mehr über einige Änderungen, die für Standard- und Enterprise-Nutzer relevant sind. Darüber hinaus haben wir einen eigenen Leitfaden für das [Native-Format (Standard v1.1)](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview), das Enterprise-[Native-Enriched-Format](/de/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) und das Enterprise-[Activity-Streams-Format](/de/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#activity-object) erstellt, der die Feldzuordnung erleichtert und erklärt, welche Felder und Erweiterungen Sie verwenden müssen, um die neuen v2-Felder anzufordern. 

- [Native-Format zu X-API v2 (Standard v1.1)](/de/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2) 
- [Native Enriched zu X-API v2 (Enterprise)](/de/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2)
- [Activity Streams zu X-API v2 (Enterprise)](/de/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2)

Möglicherweise sind Sie auch an unserem [visuellen Tool zur Datenformat-Migration](/de/x-api/migrate/data-format-migration#visual-data-format-migration-tool) interessiert, das Ihnen hilft, die Unterschiede zwischen dem [X-API-v1.1-Datenformat](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) und dem [X-API-v2-Format](/de/x-api/introduction) schnell zu erkennen.

<div id="general-differences">
  ### Allgemeine Unterschiede
</div>

<div id="requesting-objects-and-fields">
  #### Anfordern von Objekten und Feldern
</div>

Eine der größten Änderungen zwischen den Endpunkten vor v2 und v2 ist, dass die neuere Version standardmäßig nur wenige Felder zurückgibt, während Standard-, Premium- und Enterprise-Endpunkte die meisten Felder standardmäßig liefern. Die neue Version verwendet Parameter namens [fields](/de/x-api/fundamentals/fields) und [expansions](/de/x-api/fundamentals/expansions), um gezielt zusätzliche Daten über die Standardwerte hinaus anzufordern. Das bedeutet, dass Sie genau die Daten anfordern können, die Sie benötigen, ohne Felder aufnehmen zu müssen, die für Sie irrelevant sind.

Alle von Ihnen angeforderten Felder, die sich auf das primäre Datenobjekt beziehen, werden in diesem primären Datenobjekt zusammen mit den Standardwerten zurückgegeben. Wenn Sie jedoch zusätzliche Objekte über den expansions-Parameter anfordern, werden die sekundären Objekte in einem neuen includes-Objekt zurückgegeben. Sie können die erweiterten Objekte im includes-Objekt über das ID-Feld, das in beiden enthalten ist, wieder dem primären Objekt zuordnen.

Wenn Sie beispielsweise den v2-Endpunkt [Post lookup](/de/x-api/posts/lookup/introduction) verwenden und den Parameter expansions=author\_id in Ihre Anfrage aufnehmen, erhalten Sie das Feld author\_id innerhalb des primären Post-Objekts sowie ein user-Objekt pro Post im includes-Objekt. Jedes davon enthält das Standardfeld id, mit dem das user-Objekt dem Post-Objekt zugeordnet werden kann. Hier ist ein Beispiel, wie das aussieht:

```
{
  "data": [
    {
      "author_id": "2244994945",
      "id": "1397568983931392004",
      "text": "The Twitter Developer Platform. Ooh la la! https://t.co/iGTdPXBfOv https://t.co/Ze8z8EODdg"
    }
  ],
  "includes": {
    "users": [
      {
        "id": "2244994945",
        "name": "Twitter Dev",
        "username": "TwitterDev"
      }
    ]
  }
}
```

<div id="updated-json-design">
  #### Aktualisiertes JSON-Design
</div>

Zusätzlich zu den Änderungen bei der Anforderung bestimmter Felder führt die X-API v2 auch neue JSON-Designs für die von den APIs zurückgegebenen Objekte ein, darunter [Post](/de/x-api/fundamentals/data-dictionary#tweet)- und [User](/de/x-api/fundamentals/data-dictionary#user)-Objekte.

- Auf der JSON-Root-Ebene geben die Standardendpunkte Post-Objekte in einem **statuses**-Array zurück, während die X-API v2 ein **data**-Array zurückgibt.
- Statt von Retweeted- und Quoted-„Statuses“ zu sprechen, verweist das JSON der X-API v2 auf Retweeted- und Quoted-Tweets. Viele Legacy- bzw. veraltete Felder wie **contributors** und **user.translator\_type** werden entfernt.
- Anstelle der gleichzeitigen Verwendung von **favorites** (im Post-Objekt) und **favourites** (im User-Objekt) verwendet die X-API v2 den Begriff like.
- X folgt der Konvention, dass JSON-Felder ohne Wert (zum Beispiel null) nicht in die Payload geschrieben werden. Post- und User-Attribute werden nur aufgenommen, wenn sie einen Nicht-null-Wert haben. 
   

<div id="new-v2-fields">
  #### Neue v2-Felder
</div>

Wir haben außerdem dem [Post-Objekt](/de/x-api/fundamentals/data-dictionary#tweet) eine neue Reihe von Feldern hinzugefügt, darunter:

- Ein [conversation\_id](/de/x-api/fundamentals/conversation-id)-Feld
- Zwei neue [annotations](/de/x-api/fundamentals/post-annotations)-Felder, nämlich context und entities
- Mehrere neue [metrics](/de/x-api/fundamentals/metrics)-Felder
- Ein neues reply\_setting-Feld, das angibt, wer auf einen bestimmten Post antworten kann

<div id="migrating-from-standard-v11s-data-format-to-v2">
  ### Migration vom Standard-v1.1-Datenformat zu v2
</div>

Falls noch nicht geschehen, empfehlen wir Ihnen, zunächst die Einführung zur [Datenformat-Migration](/de/x-api/migrate/data-format-migration) zu lesen. Sie könnten auch an unserem [visuellen Tool zur Datenformat-Migration](/de/x-api/migrate/data-format-migration#visual-data-format-migration-tool) interessiert sein, das Ihnen hilft, die Unterschiede zwischen dem [X-API-v1.1-Datenformat](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) und dem [X-API-v2-Format](/de/x-api/fundamentals/data-dictionary) schnell zu erkennen.

Das Standard-v1.1-Datenformat, auch als natives Format bekannt, ist das primäre Format, das über die [Standard-v1.1](https://developer.x.com/en/docs/twitter-api/v1)-Endpoints bereitgestellt wird.

Wenn Sie das Premium-Produkt verwenden, beachten Sie bitte den [Leitfaden zu Native Enriched](/de/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2). Enterprise-Kunden könnten je nach Einrichtung in der Gnip-Konsole entweder Native Enriched oder Activity Streams verwenden.

<div id="migrating-from-native-enriched-data-format-to-v2">
  ### Migration vom Native Enriched-Datenformat zu v2
</div>

Das Native Enriched-Datenformat wird von unseren [Enterprise](/de/x-api/enterprise-gnip-2.0/enterprise-gnip)-Produkten verwendet.

Das Native Enriched-Datenformat wurde aktualisiert, um Metadaten zu bearbeiteten Tweets bereitzustellen. Weitere Informationen zu Edit-Tweet-Metadaten finden Sie auf der Seite [Edit Tweets – Grundlagen](/de/x-api/enterprise-gnip-2.0/fundamentals/edit-tweets).

Wenn Sie die Standard-v1.1-Endpunkte verwenden, lesen Sie bitte den [Leitfaden von Standard v1.1 zu v2](/de/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2). Wenn Sie die Enterprise-Produkte mit Activity Streams verwenden, haben wir außerdem einen Leitfaden [von Activity Streams zu v2](/de/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2) für Sie.

Die X-API v2 führt neue JSON-Designs für [Tweet](/de/x-api/fundamentals/data-dictionary#tweet)- und [User](/de/x-api/fundamentals/data-dictionary#user)-Objekte ein.

- Auf Root-Ebene des JSON gibt das Native Enriched-Format Tweet-Objekte in einem results-Array zurück, während die X-API v2 ein data-Array zurückgibt.
- Anstatt sowohl favorites (im Tweet-Objekt) als auch favourites (im User-Objekt) zu verwenden, nutzt die X-API v2 den Begriff like.
- X übernimmt die Konvention, dass JSON-Felder ohne Wert (zum Beispiel null) nicht in die Payload geschrieben werden. Tweet- und User-Attribute werden nur aufgenommen, wenn sie nicht null sind.
- Alle id-Felder in v2 liegen im String-Format vor

Zusätzlich zu den Änderungen am neuen JSON-Format haben wir auch einen neuen Satz von Feldern für das Tweet-Objekt eingeführt, einschließlich:

- conversation\_id
- reply\_settings
- alt\_text bei Medien
- Zwei neue [annotations](/de/x-api/fundamentals/post-annotations)-Felder, darunter context und entities
- Mehrere neue [metrics](/de/x-api/fundamentals/metrics)-Felder
- Mehrere neue [polls](/de/x-api/fundamentals/data-dictionary#poll)-Felder

Viele Legacy- und veraltete Felder werden entfernt:

- contributors
- Bestimmte entities.media- und extended\_entities.media-Felder
- filter\_level
- timestamp\_ms
- truncated

<div id="migrating-from-activity-streams-data-format-to-v2">
  ### Migration vom Activity-Streams-Datenformat zu v2
</div>

Das Activity-Streams-Datenformat ist mit unseren [Enterprise](/de/x-api/enterprise-gnip-2.0/enterprise-gnip)-Produkten verfügbar.

Das Activity-Streams-Datenformat wurde aktualisiert, um Metadaten für bearbeitete Tweets bereitzustellen. Weitere Informationen zu den Metadaten für bearbeitete Tweets finden Sie auf der Seite [Edit Tweets – Grundlagen](/de/x-api/enterprise-gnip-2.0/fundamentals/edit-tweets).

Wenn Sie die Standard-v1.1-Endpunkte verwenden, lesen Sie bitte den [Leitfaden von Standard v1.1 zu v2](/de/x-api/migrate/overview). Wenn Sie die Premium-Endpunkte oder das Native-Enriched-Format für Enterprise verwenden, lesen Sie bitte den [Leitfaden von Native Enriched zu v2](/de/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2).

Die X-API v2 führt neue JSON-Designs für [Post](/de/x-api/fundamentals/data-dictionary#tweet)- und [User](/de/x-api/fundamentals/data-dictionary#user)-Objekte ein.

- Auf der JSON-Stammebene liefert das Activity-Streams-Format Tweet-Objekte in einem results-Array, während die X-API v2 ein data-Array zurückgibt.
- Anstelle der Bezeichnung „Retweeted“- und „Quoted“-Activities verweist das JSON der X-API v2 auf Retweeted- und Quoted-Tweets.
- Anstatt sowohl favorites (im Tweet-Objekt) als auch favourites (im User-Objekt) zu verwenden, verwendet die X-API v2 den Begriff like.
- Twitter übernimmt die Konvention, dass JSON-Werte ohne Inhalt (zum Beispiel null) nicht in die Payload geschrieben werden. Tweet- und User-Attribute werden nur einbezogen, wenn sie Nicht-null-Werte haben.
- Alle id-Felder in v2 liegen im String-Format vor.

Zusätzlich zu den Änderungen am neuen JSON-Format haben wir auch einen neuen Satz von Feldern für das Tweet-Objekt eingeführt, darunter:

- conversation\_id
- reply\_settings
- alt\_text bei Medien
- Zwei neue [annotations](/de/x-api/fundamentals/post-annotations)-Felder, einschließlich context und entities
- Mehrere neue [metrics](/de/x-api/fundamentals/metrics)-Felder
- Mehrere neue [polls](/de/x-api/fundamentals/data-dictionary#poll)-Felder

Viele Legacy- und veraltete Felder werden entfernt oder ersetzt:

- display\_text\_range
- generator
- gnip
- link
- objectType
- provider
- twitter\_entities.symbols ersetzt durch data.entities.cashtags
- Bestimmte Felder in twitter\_extended\_entities.media und twitter\_entities.media
- twitter\_filter\_level
- twitterTimeZone
- verb

<div id="visual-data-format-migration-tool">
  ### Visuelles Tool zur Migration von Datenformaten
</div>

Das visuelle Tool zur Migration von Datenformaten ist eine Webanwendung, die die Felder zeigt, die vom [X-API-v1.1-Datenformat](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) auf das [X-API-v2-Format](/de/x-api/fundamentals/data-dictionary) für ein bestimmtes Tweet- oder Nutzerobjekt abgebildet werden. Sie können der Anwendung entweder eine Tweet-ID oder eine Nutzer-ID übergeben, um diese Zuordnung anzuzeigen.

Bitte beachten Sie, dass Sie sich mit Ihrem Twitter-Konto anmelden müssen, um die App zu nutzen.
