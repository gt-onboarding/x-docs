### Voraussetzungen 

<div id="ads-api-access-new-applications">
  #### **Ads-API-Zugang – Neue Anwendungen**
</div>

- Die wichtigste Voraussetzung für die Conversion API ist ein [Developer Account](https://developer.x.com/en/portal/dashboard) und [Ads-API-Zugang](https://developer.x.com/en/docs/twitter-ads-api/getting-started). Der Ablauf ist im Leitfaden [Ads API Getting Started](https://developer.x.com/en/docs/twitter-ads-api/getting-started) beschrieben. Bitte beachten Sie Folgendes:

**Schritt 1: Developer Account**

- Wenn Sie einen [Developer Account](https://developer.x.com/en/portal/dashboard) beantragen, wählen Sie eines unserer Abonnement-[Pläne](https://developer.x.com/en/portal/products/) für eine sofortige Freischaltung.
- **Hinweis**: Als Best Practice empfehlen wir ausdrücklich, den offiziellen X-Handle Ihres Unternehmens zu verwenden, um ein Developer-Konto zu erstellen und den Ads-API-Zugang zu beantragen. Wenn das Developer-Konto mit einem persönlichen Entwickler-Handle verknüpft ist, können diese Anmeldedaten bei Bedarf nicht übertragen werden. Es ist besser, das Konto unter einem Unternehmenskonto zu führen, um eine fortlaufende Verwaltung zu gewährleisten, und bei Bedarf die [Multi-User-Anmeldung](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html) zu nutzen. Andernfalls sollte das Konto mindestens mit individuellen Einstellungen (Headerbild, Avatar, Bio-Beschreibung und Bio-URL) eingerichtet werden und die Zwei-Faktor-Authentifizierung verwenden.

**Schritt 2: Ads-API-Antrag**

- Stellen Sie sicher, dass Sie die korrekte **App**-**ID** für Ihren Ads-API-Antrag bereithalten. Die App-ID finden Sie im [Developer Portal](https://developer.x.com/en/portal/dashboard) unter Projekte und Apps. Beispiel: 16489123
- Fordern Sie den Zugang zur Ads-API an, indem Sie Ihren X-Ansprechpartner kontaktieren.

<div id="ads-api-access-existing-applications">
  #### **Zugriff auf die X-Ads-API – bestehende Anwendungen**
</div>

- Wenn Sie bereits eine aktiv genutzte X-Ads-API-Anwendung haben, können sowohl die Anwendung als auch bestehende Zugriffstoken für die Conversion-API verwendet werden.

<div id="access-tokens">
  #### **Zugriffstoken**
</div>

- Benutzerzugriffstoken für den Handle des Nutzers, dem die Ads-API-Anwendung gehört, können direkt im Developer-Portal erstellt und abgerufen werden. Dies wird als „persönliches Zugriffstoken“ bezeichnet, da es für den eigenen X-Handle gedacht ist. Allgemeine Informationen zur Authentifizierung und zum Developer-Portal finden Sie [hier](/de/resources/fundamentals/authentication/overview).
- Benutzerzugriffstoken für andere Handles als den, dem die Ads-API-Anwendung gehört, müssen über einen 3-legged-OAuth-Flow erzeugt werden. Optionen zum Erzeugen des Zugriffstokens mit 3-legged OAuth umfassen:
  - [Kommandozeile mit webbasiertem Autorisierungs-Flow über das twurl-Tool](https://github.com/twitter/twurl#getting-started)
  - [Kommandozeile mit PIN-basierter Autorisierung](https://developer.x.com/resources/fundamentals/authentication/pin-based-oauth)
  - [Benutzerdefinierter Web-Flow, der das 3-legged-OAuth-Muster implementiert](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)
- Alle Benutzertoken, die mit der Conversion API verwendet werden, müssen für Nutzer mit dem Zugriffslevel AD\_MANAGER oder ACCOUNT\_ADMIN ausgestellt sein; dies kann über den Endpoint [authenticated\_user\_access](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/authenticated-user-access) geprüft werden.
- **Hinweis:** Die Token selbst (nach Erstellung wie oben beschrieben) können zur Nutzung auch an Nutzer ohne das Zugriffslevel AD\_MANAGER oder ACCOUNT\_ADMIN weitergegeben werden.

<div id="creating-the-conversion-api-event">
  ### Erstellen des Conversion-API-Ereignisses
</div>

Um die Conversion API zu verwenden, müssen Sie entweder ein neues Conversion-Ereignis im Ads Manager erstellen oder ein bereits vorhandenes Ereignis nutzen, das mit dem X Pixel eingerichtet und verwendet wurde. Wenn Sie eine Deduplizierung zwischen Pixel- und Conversion-API-Ereignissen vornehmen möchten, müssen Sie das vorhandene, für das Pixel erstellte Ereignis verwenden. 

<div id="option-1-using-an-existing-conversion-event-in-ads-manager">
  ##### Option 1: Verwendung eines vorhandenen Conversion-Events im Ads Manager
</div>

Wenn Sie ein bereits mit dem X-Pixel genutztes Event weiterverwenden möchten, können Sie das tun; übernehmen Sie dazu die Event-ID dieses Events. Wenn Sie für dasselbe Event sowohl Pixel als auch Conversion API einsetzen, stellen Sie sicher, dass Sie in sowohl dem Pixel-Code-Snippet als auch der Conversion-API-Anfrage (als `conversion_id`) denselben Deduplizierungsschlüssel verwenden, um Events zwischen Pixel und Conversion API zu deduplizieren. Weitere Informationen finden Sie in Abschnitt d. Testing Events and Deduplication. 

<div id="option-2-creating-a-new-conversion-event-in-ads-manager">
  ##### Option 2: Erstellen eines neuen Conversion-Events im Ads Manager:
</div>

Es ist wichtig, vor dem Erstellen eines Events eine Eventquelle im Events Manager angelegt zu haben. Um zu überprüfen, ob Sie eine Eventquelle (X Pixel) zu Ihrem Konto hinzugefügt haben, gehen Sie zum Events Manager und prüfen Sie, ob Sie „X Pixel“ in Ihrem linken Menü sehen.

Wenn Sie noch keine Eventquelle hinzugefügt haben, folgen Sie bitte den nachstehenden Schritten, um eine Eventquelle zu erstellen.

1. Gehen Sie zu ads.x.com
2. Navigieren Sie oben links zum Bereich **Tools** und klicken Sie auf **Events Manager**
3. Wählen Sie oben rechts „Eventquelle hinzufügen“, um **eine Eventquelle hinzuzufügen**, falls Sie noch keine X‑Pixel‑Eventquelle in Ihrer linken Seitenleiste haben
   1. Die ID der X‑Pixel‑Eventquelle ist Ihre **Pixel-ID**

Nun haben Sie eine Eventquelle und eine Pixel-ID. Sie müssen innerhalb der Eventquelle ein Event für die Conversion-Events erstellen, die Sie verfolgen möchten:

1. Wählen Sie innerhalb der X‑Pixel‑Eventquelle auf der rechten Seite **Events hinzufügen**
2. Wählen Sie Installation mit **Conversion API**
3. Sie sehen die **Pixel-ID** und **Event-ID** dieses Events, die in der API verwendet werden
   1. Die ID des Events ist Ihre Event-ID
4. Klicken Sie auf Speichern, und Ihr Conversion-Event ist erstellt und einsatzbereit

### Vorbereitung von Bezeichnern für Conversion-Ereignisse 

Derzeit muss mindestens ein Bezeichner übergeben werden, z. B. Click-ID (`twclid`), E‑Mail‑Adresse oder Telefonnummer. Bei Verwendung von IP-Adresse oder User-Agent muss ein zweiter Bezeichner gesendet werden, um ein korrektes Conversion-Matching zu gewährleisten.

Die Übermittlung weiterer Bezeichner führt zu einer höheren Conversion-Matching-Rate.

|     |     |     |
| :--- | :--- | :--- |
| Feld für Kundenabgleich | Format | Hashing erforderlich? |
| X Click ID | Von X generiert ([mehr erfahren](https://business.x.com/en/help/campaign-measurement-and-analytics/conversion-tracking-for-websites.html#faq)) | Nein  |
| E‑Mail‑Adresse | Führende und nachgestellte Leerzeichen entfernen | Erforderlich (SHA256) |
| Telefonnummer | [E164-Standard](https://en.wikipedia.org/wiki/E.164) | Erforderlich (SHA256) |

|     |     |     |
| :--- | :--- | :--- |
| IP-Adresse | Führende und nachgestellte Leerzeichen entfernen | Nein  |
| User-Agent | Führende und nachgestellte Leerzeichen entfernen | Nein  |

#### 1. X Click ID-Bezeichner vorbereiten 

Es wird empfohlen, die Click ID stets in die Conversion-Anfrage aufzunehmen.

Die Click ID sollte aus dem Query-String-Parameter twclid extrahiert werden, sobald dieser verfügbar ist, nachdem der Nutzer zur Zielwebsite navigiert ist. 

Ein einfaches JavaScript-Beispiel:

```js
var queryString = document.location.search;
if (queryString.has('twclid') {
  twitterClickID = getParam(queryString, 'twclid');
  // Empfohlene nächste Schritte: Logging, in den Local Storage schreiben
}
```

Empfehlungen: 

1. Den twclid-Wert immer zu extrahieren, wenn er in den URL-Query-Parametern vorhanden ist.

2. Die Daten zusammen mit relevanten Formularfeldern oder Informationen zu Conversion-Ereignissen zu speichern.

Die Verknüpfung der Click ID mit Conversion-Ereignissen und Workflow-Informationen ermöglicht Szenarien wie Batch-Verarbeitung, Algorithmen zur Analyse und Erstellung von Conversion-Ereignissen auf Basis mehrerer Website-Navigationsabläufe sowie Massenuploads.

Die Event Source URL sollte [URL-codiert](https://en.wikipedia.org/wiki/Percent-encoding) sein und die Webseite repräsentieren, die das Ereignis ausgelöst hat.

#### 2. E-Mail-Kennung vorbereiten 

Unterstützte Felder für das Kundenabgleichen können gesendet werden, sollten jedoch normalisiert und, falls erforderlich, gehasht werden, um die Privatsphäre zu schützen.

Die Informationen müssen mit SHA256 ohne Salt gehasht werden. 

Beispielsweise sollte eine E-Mail-Adresse wie test@x.com in gehashter Form an uns gesendet werden: d360d510a224510f373931ce2d6215a799f5a9c1cef221b0149b6b6b50cced62.

#### 3. Telefon-Identifier vorbereiten 

Die Telefonnummer sollte gemäß dem [E.164-Standard](https://en.wikipedia.org/wiki/E.164) übermittelt werden, und die Daten müssen ohne Salt mit SHA256 gehasht werden. 

Beispiel: Für eine US-Telefonnummer +11234567890 sollte sie in gehashter Form an uns gesendet werden: 1fa6b8d986d9b9cd01bf36951815158bbde9f520c0567c835dfe34783d0a4231.

<div id="4-prepare-ip-address-identifier">
  #### 4. IP-Adressbezeichner vorbereiten
</div>

Die IP-Adresse muss zusammen mit einem weiteren Bezeichner (twclid, E‑Mail-Adresse, Telefonnummer oder User-Agent) übermittelt werden. Für diesen Bezeichner ist kein Hashing erforderlich.

Dieser Wert wird in punktgetrennter Dezimalschreibweise angegeben, mit vier durch Punkte getrennten Zahlen. Als Beispiel könnte die IP-Adresse eines Nutzers 8.25.197.25 lauten.

<div id="5-prepare-user-agent-identifier">
  #### 5. User-Agent-Bezeichner vorbereiten
</div>

Der User-Agent muss zusammen mit einem weiteren Bezeichner (twclid, E-Mail-Adresse, Telefonnummer oder IP-Adresse) übermittelt werden. Für diesen Bezeichner ist kein Hashing erforderlich.

Dieser Bezeichner ermöglicht es dem Server, die Anwendung, das Betriebssystem, den Anbieter und/oder die Version des anfragenden User-Agents zu identifizieren. Beispielsweise kann dieser Wert als Mozilla/5.0 (Macintosh; Intel Mac OS X 10\_15\_7) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/127.0.0.0 Safari/537.36 übermittelt werden.

<div id="constructing-the-conversion-event-request">
  ### Erstellen der Anfrage für das Conversion-Ereignis
</div>

`POST: version/measurement/conversions/:pixel_id`

Senden Sie Conversion-Ereignisse für ein bestimmtes Werbekonto. Der Antwortcode sollte auf Erfolg geprüft werden (HTTP 200 OK). Es wird empfohlen, einen Retry-Mechanismus und grundlegendes Logging vorzusehen, falls Fehlercodes zurückgegeben werden.

Ausführliche Informationen zur URL des Endpunkts und zu den POST-Body-Parametern finden Sie im Abschnitt [API Reference](https://developer.x.com/en/docs/twitter-ads-api/measurement/api-reference/conversions).

<div id="rate-limit">
  ### Rate Limit
</div>

Das Rate Limit beträgt 60.000 Events pro Konto und pro 15‑Minuten‑Intervall.

Beachten Sie, dass Ihr Servercode möglicherweise Logik außerhalb dieses Request-Aufrufs implementieren muss, darunter:

1. Instrumentierung von Nutzeraktionen (Logging), um pro Event korrekte Conversion-Daten senden zu können

2. Alle erforderlichen Mechanismen, um Conversion-Events von Nutzern herauszufiltern, die relevante Datenschutzentscheidungen getroffen haben – zum Beispiel, wenn sie dem Tracking oder dem Verkauf ihrer personenbezogenen Daten auf der Website des Werbetreibenden widersprochen haben

3. Integration mit Event-Triggern und Seiten, um Events zu erfassen und Conversions zu senden

<div id="duplication-between-pixel-and-conversion-api">
  ### Duplikation zwischen Pixel und Conversion API
</div>

Wenn Sie Konversionen zwischen Pixel- und Conversion-API-Anfragen deduplizieren möchten, dient conversion\_id als Deduplizierungsschlüssel. Die Deduplizierung erfolgt nur auf Ereignisebene. Anders ausgedrückt: Um zwischen Pixel- und CAPI-Anfragen zu deduplizieren, muss ein Werbetreibender dasselbe Ereignis in beiden Anfragen verwenden und zusätzlich dieselbe conversion\_id einsetzen. Eine Deduplizierung ist nur bei Ereignissen möglich, die innerhalb eines Zeitfensters von 48 Stunden eingehen.

<div id="conversion-tracking-overview">
  ## Conversion Tracking (Überblick)
</div>

<div id="summary">
  ### Zusammenfassung
</div>

Conversion-Tracking ermöglicht es Ihnen, die Anzahl der X Nutzer zu messen, die nach dem Ansehen und Interagieren mit Ihren Anzeigen auf X eine gewünschte Aktion ausführen. Es bietet Ihnen die Möglichkeit zu erfassen, welche Ihrer Kampagnen Aktionen wie Website-Besuche, Anmeldungen und Käufe auslösen. So erhalten Werbetreibende Off-X-Messfunktionen, um die Performance ihrer Direct-Response-Anzeigen zu verstehen und Kunden kosteneffizient zu gewinnen.

Mithilfe eines Conversion-Tags können Werbetreibende Nutzer-Conversions erfassen und ihren Werbekampagnen auf X zuordnen. Dadurch erhalten sie die notwendige Transparenz, um ihre Kampagnen auf die Erreichung ihrer Cost-per-Acquisition-(CPA)-Ziele zu optimieren.

Es gibt eine Vielzahl von Website-Aktionen, die ein Werbetreibender mit Conversion-Tracking messen kann. Er kann eine oder mehrere auswählen – entsprechend den Aktionen, die mit seiner Kampagne erzielt werden sollen:

- Website-Besuch: Nutzer besucht eine Landingpage auf der Website des Werbetreibenden
- Kauf: Nutzer schließt den Kauf eines Produkts oder einer Dienstleistung auf der Website des Werbetreibenden ab
- Download: Nutzer lädt eine Datei, etwa ein Whitepaper oder ein Softwarepaket, von der Website des Werbetreibenden herunter
- Anmeldung: Nutzer registriert sich für den Dienst, Newsletter oder die E-Mail-Kommunikation des Werbetreibenden
- Benutzerdefiniert: Sammelkategorie für eine individuelle Aktion, die in keine der oben genannten Kategorien fällt

Das Conversion-Tracking von X bietet Werbetreibenden einen vollständigen Überblick über die Conversion-Attribution. Im Vergleich zu Messsystemen von Drittanbietern, die Kunden anstelle der eigenen Conversion-Tracking-Funktion von X genutzt haben könnten (z. B. eindeutige Klick-URLs in Kombination mit Tracking-Tags von Drittanbietern), ermöglicht das Conversion-Tag von X, Conversions zu erfassen, die Interaktionen im mittleren und oberen Funnel zugeordnet sind – etwa Tweet-Aufklappungen, Retweets, Favoriten, Antworten und Follows – sowie Impressionen.

<div id="faq">
  ### FAQ
</div>

<AccordionGroup>
  <Accordion title="Wie funktioniert das Conversion-Tracking-Tag?">
    Zunächst erstellt ein Werbetreibender ein Conversion-Tag – ein von X bereitgestelltes Code-Snippet – und implementiert es auf seiner Website. Das Tag ist dann bereit, eine Conversion zu messen, sobald ein Nutzer die definierte Aktion ausführt.

    Anschließend sehen Nutzer die Anzeige des Werbetreibenden im X-Client, klicken sich zur Website durch und führen die getaggte Aktion aus. Wenn der Nutzer diese Aktion innerhalb des/der vom Werbetreibenden bei der Tag-Einrichtung festgelegten Attributionsfenster(s) abschließt, erkennt das Tag, dass der Nutzer zuvor mit einer X-Anzeige interagiert hat. Das Tag „feuert“ dann bzw. sendet eine Benachrichtigung an die Server von X, damit die Conversion der Anzeige zugeschrieben werden kann, die die Conversion ausgelöst hat.
  </Accordion>

  <Accordion title="Gibt es im Kampagneneinrichtungsprozess eine Möglichkeit, auszuwählen, welche Tracking-Pixel für diese Kampagne relevant sind?">
    Nein, unser Produkt ist nicht darauf ausgelegt, spezifische Conversion-Tags einzelnen Kampagnen zuzuordnen. Sobald ein Tag eingerichtet ist, verfolgt das System automatisch, welche Anzeige Conversions auf einem bestimmten Tag ausgelöst hat.
  </Accordion>

  <Accordion title="Was sind unsere standardmäßigen Einstellungen für Attributionsfenster bei Conversion-Tags?">
    Standard-Attributionsfenster nach View: 1 Tag

    Standard-Attribution nach Engagement: 14 Tage

    Diese Standardwerte können während der Einrichtung des Conversion-Tags oder jederzeit nach Erstellung des Tags geändert werden. Die Optionen für Attributionsfenster nach Engagement sind 1, 7, 14, 30, 60 und 90 Tage. Die Optionen für Attributionsfenster nach View sind keine, 1, 7, 14, 30, 60 und 90 Tage.
  </Accordion>

  <Accordion title="Welche Ideen gibt es für effektive DR-Creatives und Strategien, die Conversions wirksam steigern?">
    Auch wenn Ziele, Situationen und Strategien je nach Kunde variieren, finden Sie hier einige Ansätze, die sich bei Kunden in der Conversion-Tracking-Alpha bzw. -Beta bewährt haben:

    **Creative:**

    - **Angebote**: Kombinieren Sie einen Rabatt, eine Promotion oder kostenlosen Versand mit dem Promoted Tweet, um mehr Interesse an der Aktion zu wecken
    - **Gewinnspiele und Wettbewerbe**: Besonders bei bekannten Marken führten Gewinnspiele und Wettbewerbe zu Conversions
    - **Experimente mit Tweet-Text**: Testen von GROSSSCHRIFT vs. Kleinschreibung (FREE vs free bzw. NOW vs now)
    - **Fristen**: Setzen Sie eine Frist, um zu sofortiger Aktion zu motivieren (Angebot endet am 12. Dezember!)
    - **Aussagekräftige Fotos hinzufügen**: Es lohnt sich zu testen, ob visuell ansprechende Fotos im Tweet-Creative Conversions fördern; die Ergebnisse können variieren oder vom jeweiligen Angebot abhängen.

    **Targeting:**

    - @handle-Targeting und Targeting nach Interessenkategorien: Eine enge Abstimmung von Tweet-Text und @handles auf die beabsichtigte Zielgruppe des Tweets führte zu Conversions
    - Einsatz von Nischen-, aber reichweitenstarken Keywords: Im Konzertbereich erwies sich die Nutzung von Keywords zum Künstler/Musiker (z. B. dessen Name) als effektiv
    - Tailored Audiences: Kunden, die TA Web und Conversion-Tracking gemeinsam nutzten, erzielten niedrigere CPAs als Kontrollgruppen mit anderem Targeting

    Je granularer Ihre Kampagnensegmentierung, desto umsetzbarer sind die berichteten Conversion-Ergebnisse. Es ist beispielsweise deutlich einfacher, eine Kampagne mit 4 Keywords zu optimieren als eine Kampagne mit 50.
  </Accordion>
</AccordionGroup>

<div id="troubleshooting-and-support-for-conversion-api">
  ## Fehlerbehebung und Support für die Conversion API
</div>

Bei Fragen zu Fehlercodes nach dem Aufruf der API finden Sie den entsprechenden Abschnitt unten. Für alle anderen Anliegen wenden Sie sich bitte an Ihre Twitter-Ansprechperson; wir werden diese so schnell wie möglich klären. 

<div id="error-handling-and-explanation">
  ### Fehlerbehandlung und Erläuterung
</div>

Eine Anfrage ist nur erfolgreich, wenn für alle darin enthaltenen Conversions keine Fehler auftreten. Tritt in einer einzelnen Conversion ein Fehler auf, gibt der Endpoint eine Liste aller zutreffenden Fehler zurück.

<div id="x-ads-api-error-codes-overview">
  ### Übersicht der Fehlercodes der X-Ads-API
</div>

Nachfolgend finden Sie eine vollständige Liste der Fehlercodes in der X-Ads-API:

[https://developer.x.com/en/docs/twitter-ads-api/response-codes](https://developer.x.com/en/docs/twitter-ads-api/response-codes)

Erfolgreiche Antworten der Conversion-API sind durch einen HTTP-Statuscode aus der 200er‑Serie gekennzeichnet und enthalten eine JSON-basierte Nutzlast mit dem angeforderten Objekt.

<div id="when-there-is-a-500-series-http-code-its-related-to-a-server-issue-instead-of-your-request-or-account-set-up-please-check-the-x-api-status-page-or-the-developer-community-forum-in-case-others-are-having-similar-issues">
  #### **Wenn ein HTTP-Code aus der 500er-Reihe auftritt**, liegt ein Serverproblem vor – nicht bei Ihrer Anfrage oder der Einrichtung Ihres Kontos. Bitte prüfen Sie die [X API Statusseite](https://api.twitterstat.us/) oder das [Developer Community Forum](https://devcommunity.x.com/), um festzustellen, ob andere ähnliche Probleme melden.
</div>

<div id="when-there-is-a-400-series-http-code-the-common-cases-are">
  #### Wenn ein HTTP-Code aus der 400er-Reihe auftritt, sind die häufigsten Fälle:
</div>

- 400 Bad Request (Anfrage entspricht nicht den Vorgaben)

- 401 Unauthorized (Authentifizierungsprobleme)

- 403 Forbidden (API-Zugriffsprobleme im Zusammenhang mit diesem Entwicklerkonto)

- 404 Not Found (URL oder Parameter sind für den Endpunkt möglicherweise nicht korrekt)

<div id="conversion-api-error-codes">
  ### Fehlercodes der Conversion API
</div>

<div id="400-bad-request-scenarios">
  #### 400 Bad-Request-Szenarien
</div>

| **Grund** | **Typ** | **Fehlermeldung** |
| :--- | :--- | :--- |

<div id="401-unauthorized">
  ### 401 Nicht autorisiert
</div>

**Grund**: Authentifizierungsdaten fehlen oder sind falsch

**Lösung**: Befolgen Sie die Schritte zur Authentifizierung in der [Set-up-Dokumentation](https://developer.x.com/en/docs/twitter-ads-api/measurement/guides/conversion-api) mit einer der drei Authentifizierungsmethoden:

User-Access-Tokens für andere Handles als das Handle, dem die Ads-API-Anwendung gehört, müssen über einen 3-legged-OAuth-Flow generiert werden. Optionen zum Generieren des Access Tokens mit 3-legged OAuth umfassen:

- [Kommandozeile mit webbasiertem Autorisierungsprozess über das twurl-Utility](https://github.com/twitter/twurl#getting-started)

- [Kommandozeile mit PIN-basierter Autorisierung](https://developer.x.com/resources/fundamentals/authentication/pin-based-oauth)

- [Benutzerdefinierter Webflow, der das 3-legged-OAuth-Muster implementiert](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)

Alle Benutzertokens, die mit der Conversion-API verwendet werden, müssen zu Nutzern mit dem Zugriffslevel AD\_MANAGER oder ACCOUNT\_ADMIN\* gehören; dies kann über den Endpoint [authenticated\_user\_access](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/authenticated-user-access) geprüft werden.

Senden Sie Website-Conversions für eine einzelne Event-Tag-ID.
