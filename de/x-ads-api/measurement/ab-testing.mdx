<div id="introduction">
  ### Einführung
</div>

A/B-Tests ermöglichen es Werbetreibenden, die Nutzersegmente, die sie auf X erreichen, aufzuteilen, um besser zu verstehen, wie sich die Kampagnenleistung optimal verbessern lässt, und um Erkenntnisse für ihre Marketingstrategien zu gewinnen.

Diese Segmente – sogenannte Aufteilungen in Nutzergruppen – sind randomisiert und gegenseitig exklusiv. Durch die Randomisierung werden Einflussfaktoren gleichmäßig verteilt. Anders ausgedrückt: Es gibt keine grundlegenden Unterschiede zwischen den Gruppen oder ihrem erwarteten Verhalten. Wird daher eine einzelne Variante nur auf eine Nutzergruppe angewendet und nicht auf die anderen, kann der Unterschied in der Kampagnenleistung dieser Variante zugeschrieben werden.

Obwohl es möglich ist, viele Varianten gleichzeitig zu testen, empfehlen wir nachdrücklich, jeweils nur eine einzige Variante zu testen. So lässt sich der kausale Faktor für den beobachteten Unterschied in der Kampagnenleistung isolieren.

Varianten werden auf Kampagnenebene festgelegt. Möchte der Werbetreibende beispielsweise die Wirksamkeit eines neuen Creatives testen, sollte er zwei identische Kampagnen erstellen, bei denen sich lediglich das Creative unterscheidet. Künftig planen wir, Varianten auf Line-Item-Ebene zu unterstützen.

<div id="use-cases">

</div>

A/B-Tests werden am häufigsten eingesetzt, um (1) Optimierungsfälle für Performance-Kund:innen zu unterstützen, die verstehen möchten, was auf X am besten funktioniert, um ihre Investitionen zu maximieren, und (2) Lernfälle für Markenwerbetreibende, die Erkenntnisse zur Gestaltung ihrer Marketingstrategie nutzen möchten. 

Die API unterstützt A/B-Tests für beliebige Kampagnenvariablen, einschließlich:

- Creative

- Gebotsart

- Gebotseinheit

<div id="ab-testing">
  ## A/B-Tests
</div>

A/B-Tests ermöglichen es Werbetreibenden, die Nutzer, die sie auf X erreichen, zu segmentieren, um zu verstehen, wie sie die Kampagnenleistung am besten optimieren können, und Erkenntnisse zu gewinnen, die ihre Marketingstrategien informieren.

Diese Segmente – sogenannte Nutzergruppen-Splits – sind randomisiert und gegenseitig exklusiv. Durch die Randomisierung werden Einflussfaktoren gleichmäßig verteilt. Anders ausgedrückt gibt es keine grundlegenden Unterschiede zwischen den Gruppen oder ihrem erwarteten Verhalten. Daher kann, wenn eine einzelne Variante nur auf eine Nutzergruppe angewendet wird und nicht auf die anderen, der Unterschied in der Kampagnenleistung dieser Variante zugeschrieben werden.

Obwohl es möglich ist, viele Varianten gleichzeitig zu testen, empfehlen wir dringend, jeweils nur eine zu testen. Dadurch wird der ursächliche Faktor für den beobachteten Unterschied in der Kampagnenleistung isoliert.

Varianten werden entweder auf Kampagnenebene oder auf Anzeigengruppenebene festgelegt. Die Anzeigengruppe wird über [Line Item](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/line-items) in der X-Ads-API festgelegt. Als Beispiel für eine Variante auf Anzeigengruppenebene: Wenn der Werbetreibende die Wirksamkeit eines neuen Creatives testen möchte, sollte er eine Kampagne mit zwei identischen Anzeigengruppen erstellen, bei denen sich lediglich das Creative unterscheidet.

<div id="use-cases">
  ### Anwendungsfälle
</div>

A/B-Tests werden am häufigsten eingesetzt, um (1) Optimierungsanwendungsfälle für Performance-Kundinnen und -Kunden zu unterstützen, die verstehen möchten, was auf X am besten funktioniert, um ihre Investitionen zu optimieren, und (2) Lernanwendungsfälle für Markenwerbetreibende, die Erkenntnisse zur Ausrichtung ihrer Marketingstrategie nutzen möchten.

Die API unterstützt A/B-Tests für jede Kampagnenvariable, einschließlich:

- Creative

- Targeting

- Gebotsart

- Gebotseinheit

<div id="attributes">
  ### Attribute
</div>

A/B-Tests werden als verschachtelte Strukturen dargestellt. Es gibt Felder auf oberster Ebene für den A/B-Test selbst sowie ein Array von Benutzergruppenobjekten, die jeweils durch eine Reihe von Feldern beschrieben werden.

Auf hoher Ebene muss jeder A/B-Test die folgenden Informationen enthalten:

- Die Testdauer, dargestellt durch die Felder start\_time und end\_time

- Die Ebene, auf der die Aufteilung erfolgt, dargestellt durch das Feld entity\_type

- Mindestens zwei (und höchstens 30) Benutzergruppen, jeweils dargestellt als Objekt im Array user\_groups

Jede Benutzergruppe muss die folgenden Informationen enthalten:

- Den Prozentsatz der Benutzer, die der jeweiligen Benutzergruppe zugeordnet werden sollen, dargestellt durch das Feld size

- Die Campaign-IDs/Line-Item-IDs, die den Pool der Benutzer für die jeweilige Benutzergruppe bilden sollen, dargestellt durch das Array entity\_ids

Optional können für A/B-Tests und für Benutzergruppen Werte für name und description festgelegt werden. Informationen zu Validierungsregeln und anderen Einschränkungen finden Sie unten.

Weitere Metadaten, wie die ID oder der Erstellungszeitpunkt, sind ebenfalls enthalten, werden jedoch automatisch von X gesetzt.

<div id="usage">
  ### Verwendung
</div>

Die folgenden Unterabschnitte beschreiben das Erstellen und Aktualisieren von A/B-Tests. Lesen und Löschen funktionieren wie bei allen anderen X-Ads-API-Endpunkten.

<div id="creating">
  #### Erstellen
</div>

Erstellen Sie einen A/B-Test mit dem Endpoint [POST accounts/:account\_id/ab\_tests](https://developer.x.com/en/docs/x-ads-api/measurement/api-reference/ab-tests). Der Endpoint akzeptiert nur JSON-POST-Bodies. Der Content-Type muss auf application/json gesetzt sein.

Nachdem der Werbetreibende zwei oder mehr Kampagnen eingerichtet hat, kann ein A/B-Test erstellt werden. Wie oben angegeben, müssen A/B-Tests Folgendes enthalten: Testdauer, Split-Level und mindestens zwei Nutzergruppen. Jede Nutzergruppe muss den Prozentsatz der Nutzer angeben, der ihr zugeordnet werden soll, sowie die Kampagnen-IDs, aus denen ihr Nutzerpool bestehen soll. Jedes dieser Elemente wird weiter unten näher beschrieben.

Testdauer:

- Die Werte start\_time und end\_time müssen

  - in der Zukunft liegen (relativ zu dem Zeitpunkt, zu dem der A/B-Test erstellt wird)

  - sich mit den Laufzeiten der Kampagne bzw. des Line Items überschneiden

- Der Test muss bei nicht appbasierten Kampagnen mindestens einen Tag dauern und bei appbasierten Kampagnen mindestens fünf Tage

Split-Level:

- Der entity\_type kann auf CAMPAIGN oder LINE\_ITEM gesetzt werden

Nutzergruppen:

- Jede Nutzergruppe wird als Objekt im Array user\_groups dargestellt

  - Es sind mindestens zwei Nutzergruppen erforderlich

  - Es sind maximal 30 Nutzergruppen zulässig

- Die Größe jeder Nutzergruppe wird mithilfe einer String-Darstellung eines numerischen Werts zwischen 1,00 und 99,00 festgelegt

  - Hinweis: Die Größenwerte über alle Objekte hinweg müssen in der Summe 100,00 ergeben

- Die Kampagnen-IDs sollten im Array entity\_ids jeder Nutzergruppe angegeben werden

Optional können der Name und die Beschreibung für den A/B-Test oder für eine oder mehrere Nutzergruppen festgelegt werden.

Die folgende Anfrage erstellt einen A/B-Test auf Kampagnenebene, der vier Tage dauert und zwei Nutzergruppen mit jeweils 50 % der Nutzer umfasst. Für A/B-Tests auf Line-Item-Ebene

Der wesentliche Unterschied zwischen A/B-Tests auf Kampagnen- und Line-Item-Ebene ist der entity\_type. Für A/B-Tests auf Line-Item-Ebene muss ‚entity\_type‘ auf ‚LINE\_ITEM‘ gesetzt werden. Dies gilt für alle unten aufgeführten Aktionen für einen bereits erstellten A/B-Test.

Anforderungen:

1. Alle Line Items der A/B-Testkampagne müssen in den Split-Test einbezogen werden.
2. Auf Line-Item-Ebene ist nur eine gleichmäßige Aufteilung zulässig.
3. Die Anzahl der in einem Split-Test zulässigen Line Items pro User Group muss kleiner oder gleich 5 sein.
4. Pro User Group ist nur ein Line Item zulässig.

<div id="updating">
  ### Aktualisieren
</div>

Aktualisieren Sie einen A/B-Test über den Endpunkt [PUT accounts/:account\_id/ab\_tests/:ab\_test\_id](https://developer.x.com/en/docs/x-ads-api/measurement/api-reference/ab-tests). Der Endpunkt erfordert, dass ein JSON-Blob in der Anfrage gesendet wird. Der Content-Type muss auf application/json gesetzt sein.

Wie bei anderen Update-Endpunkten erfordert der Endpunkt [PUT accounts/:account\_id/ab\_tests/:ab\_test\_id](/de/x-ads-api/measurement/ab-testing#get-accounts-account-id-ab-tests), dass die A/B-Test-ID in der URL angegeben wird. Im Allgemeinen können A/B-Tests nur aktualisiert werden, solange der Status SCHEDULED ist. Es gibt eine Ausnahme: Es ist möglich, die end\_time des A/B-Tests zu aktualisieren, während er LIVE ist.

Dieser Endpunkt unterstützt partielles JSON mit Objekt-IDs. Die folgenden Grundsätze gelten:

- Um Objekte oder Elemente hinzuzufügen oder zu entfernen, übergeben Sie das gesamte Array (und seine Unterstrukturen); dies ist eine **Ersetzungs**-Operation

- Andernfalls ändern (hinzufügen, ändern, entfernen) Sie bestehende _Felder_, indem Sie Schlüsselnamen oder IDs angeben

  - Um ein Feld zu entfernen, setzen Sie seinen Wert auf null

  - Felder, die nicht übergeben werden, bleiben unverändert

Wenn wir beispielsweise einer zuvor erstellten A/B-Testkonfiguration eine dritte Benutzergruppe hinzufügen möchten, müssen wir das user\_groups-Array mit den zwei vorhandenen Benutzergruppenobjekten sowie dem neuen, das wir hinzufügen möchten, senden. Betrachten Sie dies als _Neuerstellung_ des user\_groups-Arrays; übergeben Sie die Daten so, als würden Sie sie von Anfang an auf diese Weise erstellen (übergeben Sie keine IDs von Benutzergruppenobjekten). Das user\_groups-Array in der Update-Anfrage könnte wie folgt aussehen.

```json
[
  {
    "entity_ids": [
      "f2qcw",
      "f2tht"
    ],
    "size": "30.0",
    "name": "first group"
  },
  {
    "entity_ids": [
      "f2rqi",
      "f2tws"
    ],
    "size": "30.0",
    "name": "second group",
    "description": "second AB test group"
  },
  {
    "entity_ids": [
      "i1vwr",
      "i1xre"
    ],
    "size": "40.0"
  }
]
```

Beachten Sie, dass die size-Werte über alle Objekte hinweg weiterhin 100.00 ergeben. Hätten wir sie für die ersten beiden Objekte — zuvor jeweils auf 50.00 gesetzt — nicht aktualisiert, wäre die Anfrage fehlgeschlagen.

Wenn wir stattdessen lediglich eine Beschreibung zur ersten Benutzergruppe hinzufügen möchten, sähe das user\_groups-Array in der Update-Anfrage wie folgt aus.

```json
[
  {
    "id": "p1bcx",
    "description": "updated using a PUT request"
  }
]
```

Wir referenzieren das Benutzergruppenobjekt per ID und geben nur das Feld an, das wir ändern möchten.

#### PUT accounts/:account\_id/ab\_tests/:ab\_test\_id

Aktualisiert den angegebenen A/B-Test.

Alle Parameter werden im Request-Body gesendet; der `Content-Type` muss `application/json` sein.

Dieser Endpunkt unterstützt partielles JSON mit Objekt-IDs. Es gelten die folgenden Grundsätze:

- Um Objekte oder Elemente hinzuzufügen oder zu entfernen, übergeben Sie das gesamte Array (einschließlich seiner Unterstrukturen); dies ist eine **Ersetzungsoperation**
  - Denken Sie daran wie an ein Neuaufsetzen des Arrays
- Andernfalls ändern (erweitern, anpassen, entfernen) Sie bestehende Felder, indem Sie sich auf Schlüsselnamen oder IDs beziehen
  - Um ein Feld zu entfernen, setzen Sie dessen Wert auf `null`
  - Felder, die nicht übergeben werden, bleiben unverändert

Im Allgemeinen können A/B-Tests nur aktualisiert werden, solange der `status` `SCHEDULED` ist. Eine Ausnahme: Die `end_time` eines A/B-Tests kann aktualisiert werden, während er `LIVE` ist.
