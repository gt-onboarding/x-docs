---
title: Paginierung
---

Um eine größere maximale Anzahl von Kampagnen zu unterstützen und alle mit einem Konto verknüpften Entitäten effizient abzurufen, unterstützt die Advertiser-API nun bei vielen GET-Endpunkten Paginierung. Der Paging-Mechanismus ist einfach zu verwenden und dem Cursor-basierten Paginierungsverfahren der REST-API sehr ähnlich, wie in [Using cursors to navigate collections](https://developer.x.com/en/docs/x-api/v1/pagination) beschrieben.

<div id="getting-started">
  ## Erste Schritte
</div>

Für bestimmte GET-Anfragen akzeptieren wir jetzt die optionalen Query-Parameter `cursor` und `count`. Wenn eine Antwort mehr als `count` Entities enthält, geben diese Endpunkte nun die ersten `count` Entities zurück sowie einen `next_cursor`-Schlüssel im Antwort-JSON.

`GET https://ads-api.x.com/5/accounts/abc1/campaigns?count=50`

```json
{
  "data": \[...\],
  "next_cursor": "c-3yvu1pzhd3i7",
  "request": {...}
}
```

Um die nächste Ergebnismenge abzurufen, fügen Sie `cursor` zu Ihren Query-Parametern hinzu:

`GET https://ads-api.x.com/5/accounts/abc1/campaigns?cursor=c-3yvu1pzhd3i7&count=50`

```json
{
  "data": \[...\],
  "next_cursor": "c-3w3zdyg8ywan",
  "request": {...}
}
```

<div id="going-deeper">
  ## Tiefergehende Informationen
</div>

Für die meisten Endpunkte beträgt der maximale `count`-Wert **1.000**, der minimale **1** und der Standardwert **200**.

Der von `next_cursor` zurückgegebene Wert ist immer eine Zeichenfolge und sollte als undurchsichtig betrachtet werden; die Implementierung kann sich ändern. Wenn auf der aktuellen Seite des Ergebnissatzes weniger als `count` Entities zurückgegeben werden, ist der Wert von `next_cursor` `null`.

**Hinweis:** Die Analytics-Endpunkte unterstützen diesen Paginierungsstil nicht. Das Paging bei den Stats-Endpunkten erfolgt durch die Angabe von Zeiträumen. Weitere Informationen finden Sie in der Dokumentation der jeweiligen Endpunkte.
