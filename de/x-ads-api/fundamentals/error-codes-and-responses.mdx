<div id="typical-response-structure">
  ## Typische Antwortstruktur
</div>

Erfolgreiche Antworten werden durch einen HTTP-Statuscode aus der 200er-Serie und eine JSON-basierte Nutzlast gekennzeichnet, die die angeforderten, erstellten, geänderten oder gelöschten Objekte enthält, zusammen mit einer Darstellung der Interpretation Ihrer Anfrage durch den Server.

Wenn Sie eine erfolgreiche Anfrage gestellt haben, erhalten Sie als Teil der Antwort einen `request`-Knoten, der Ihre Anfrage widerspiegelt.

_Beispiel:_ [GET accounts/abcdefg/campaigns?with\_deleted=true](/de/x-ads-api/campaign-management#campaigns)

```json
{
  /* die Daten Ihrer Antwort ... */,
  "request": {
    "params": {
      "account_id": "abcdefg",
      "with_deleted": "true"
    }
  }
}
```

Das Feld `data` in JSON-Antworten enthält die spezifischen Objekte, die der verwendeten Ressource zugeordnet sind. Der `data`-Knoten ist ein JSON-Array, wenn die Antwort ein oder mehrere Ergebnisse enthalten kann. Er wird als JSON-Objekt zurückgegeben, wenn nur ein Ergebnis möglich ist. In einigen seltenen Fällen kann eine Antwort, die typischerweise eine Sammlung enthalten würde, stattdessen eine Hashmap enthalten. In diesem Fall ist davon auszugehen, dass die einzelne Hashmap ein Objekt desselben Typs ist, wie im Feld `type` angegeben.

<div id="error-response-structure">
  ## Struktur von Fehlerantworten
</div>

Fehlerantworten werden mit einem HTTP-Statuscode außerhalb der 200er-Range zurückgegeben. In der Regel liegt eine JSON-Antwort bei, einige Fehler liefern jedoch andere Arten von Antwortkörpern. In Fällen, in denen die Antwortstruktur nicht geparst werden kann, hat die grundlegende Bedeutung des HTTP-Statuscodes Vorrang. So kann es beispielsweise vorkommen, dass ein HTTP 404 zusammen mit einer HTML-Antwort erscheint. In diesem Fall kann man davon ausgehen, dass der Inhalt nicht gefunden wurde (HTTP 404 bedeutet „Not Found“).

Typische Fehlerantworten folgen einer ähnlichen Struktur wie erfolgreiche Antworten. Die Art des Fehlers wird in einem Knoten namens `errors` mitgeteilt. Der Knoten `errors/code` enthält einen in CAPS\_CASE geschriebenen, konstanten Fehlercode, den Sie programmgesteuert auswerten können, um Entscheidungen zur Behebung zu treffen. Der Knoten `errors/message` enthält eine (in der Regel) menschenlesbare Beschreibung des Fehlers auf Englisch. Zusätzliche Felder können angehängt sein, um detailliertere Informationen zum Fehler anzugeben.

```json
{
  "errors": [
    {
      "parameter": "start_time",
      "details": "invalid date",
      "code": "INVALID_PARAMETER",
      "value": "",
      "message": "Expected time, got \"\" for start_time"
    }
  ],
  "request": {
    "params": {
      "account_id": "hkk5"
    }
  }
}
```

Im obigen Beispiel wurde eine Anfrage an einen Analytics-Endpunkt mit einem ungültigen Wert für den Parameter `start_time` gestellt. Der Wert in `errors/code` für Anfragen mit ungültigen Parametern lautet `INVALID_PARAMETER`.
