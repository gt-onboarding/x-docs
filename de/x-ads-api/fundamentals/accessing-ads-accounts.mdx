<div id="user-accounts-vs-ad-accounts">
  # Nutzerkonten vs. Werbekonten
</div>

Es gibt zwei verschiedene Arten von Konten, die bei der Verwendung der X-Ads-API beteiligt sind: Werbekonten und X-Nutzerkonten. In der gesamten Ads-API-Dokumentation bezieht sich der Begriff „Konto“ in der Regel auf das Werbekonto.

- Werbekonten werden auf **business.x.com** registriert und in der API durch account\_id identifiziert. Werbekonten sind direkt mit Finanzierungsquellen verknüpft und nutzen Inhalte von einem oder mehreren X-Nutzerkonten als „promotable users“. Jedes Werbekonto kann einem oder mehreren X-Nutzerkonten Berechtigungen erteilen. Das Werbekonto, oder „aktuelles Konto“, wird in nahezu jeder ausgeführten URL als Inline-Parameter :account\_id dargestellt.
- X-Nutzerkonten (z. B. @AdsAPI) werden in der Ads-API durch user\_id identifiziert. Eines oder mehrere dieser Konten können einem Werbekonto zugeordnet sein. Das authentifizierte X-Nutzerkonto, das Anfragen an die API stellt, wird als der „aktuelle Nutzer“ bezeichnet. Eine Liste der Werbekonten, auf die der aktuelle Nutzer Zugriff hat, finden Sie unter [GET accounts](/de/x-ads-api/campaign-management#accounts). „Promotable users“ sind X-Handles, die von einem bestimmten Werbekonto beworben werden können. Weitere Details hierzu finden Sie unter [Obtaining Ads Account Access](/de/x-ads-api/introduction).

<div id="methods-for-ad-account-access">
  ## Methoden für den Zugriff auf Werbekonten
</div>

Es gibt zwei Methoden, mit denen Sie Ads-API-Anfragen für das Konto eines Werbetreibenden stellen können:

1. Anfragen im Namen eines Werbetreibenden stellen (**empfohlen**)
2. Anfragen über Ihr eigenes Konto stellen, dem der Zugriff auf das Konto eines Werbetreibenden gewährt wurde, z. B. als [Agentur](https://business.x.com/en/blog/agencies-manage-multiple-Twitter-Ads-accounts.html), die mehrere Konten betreut.

Dieses Dokument bietet einen kurzen Überblick über die Unterschiede zwischen diesen Optionen und sollte zusammen mit unseren weiteren Ressourcen verwendet werden, z. B. den [FAQ zur Multi-User-Anmeldung](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html). 

Wie in [Autorisieren einer Anfrage](/de/resources/fundamentals/authentication#authorizing-a-request#authorizing-a-request) beschrieben, erfordern alle Anfragen an die X-Ads-API Authorization-Header mit [OAuth 1.0a](/de/resources/fundamentals/authentication) und einem Zugriffstoken, das über den [3‑legged‑](/de/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)[OAuth](/de/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)-Flow erhalten wird. Ihre Anwendung muss einen webbasierten OAuth-Flow implementieren, um [Zugriffstoken zu erhalten](/de/resources/fundamentals/authentication#oauth-1-0a-2). Ads-API-Entwickler sollten niemals X‑Werbetreibende auffordern, ihre Anmeldedaten weiterzugeben.

Standardmäßig enthält jede [X-Entwickleranwendung](/de/resources/fundamentals/developer-apps) ein statisches **Access Token**, das verwendet werden kann, um Ads-API-Anfragen für das Konto zu stellen, dem die Anwendung gehört. Diese Anmeldedaten sind ideal für Single-Account-Anwendungsfälle und erfordern keinen 3‑legged‑ oder [PIN-basierten OAuth-](/de/resources/fundamentals/authentication#pin-based-authorization)Flow. Wenn Sie nicht auf ein anderes X Ads-Konto zugreifen, verwenden Sie diese [Single-User-](/de/resources/fundamentals/authentication#oauth-1-0a-2)Anmeldedaten anstelle der folgenden Schritte.

<div id="levels-of-access">
  ## Zugriffsstufen
</div>

<div id="app-level-permissions">
  ### Berechtigungen auf App-Ebene
</div>

Jede Nutzerin und jeder Nutzer erhält eine Zugriffsebene, wie bei der [Beantragung der Ads-API](/de/x-ads-api/getting-started/step-by-step-guide) angefordert:

- **Nur Conversions**: Nutzer hat Zugriff auf die Endpunkte für [Mobile](/de/x-ads-api/measurement/mobile-conversions) und [Web-Conversions](/de/x-ads-api/measurement/web-conversions) mit [Lese- und Schreib-](/de/resources/fundamentals/developer-apps)zugriff.
- **Standardzugriff**: Nutzer hat Zugriff auf die Endpunkte für [Analytics](/de/x-ads-api/analytics), [Kampagnenverwaltung](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/overview), [Creatives](https://developer.x.com/en/docs/twitter-ads-api/creatives/overview), [Benutzerdefinierte Zielgruppen](https://developer.x.com/en/docs/twitter-ads-api/audiences/overview) und [Conversions](/de/x-ads-api/measurement/web-conversions) mit [Lese- und Schreib-](/de/resources/fundamentals/developer-apps)zugriff.

**Hinweis:** Ads-API-Entwickler, die vor Juli 2023 Zugriff beantragt haben, können abweichende Zugriffs- und Berechtigungsstufen haben und möglicherweise auf fünf OAuth-Token beschränkt sein. Siehe unseren Leitfaden zum [Erhöhen des Zugriffs](/de/x-ads-api/getting-started/increasing-access), um Zugriff auf zusätzliche Endpunkte zu erhalten oder Token-Beschränkungen für bestehende Anwendungen aufzuheben.

<div id="ad-account-level-permissions">
  ### Berechtigungen auf Anzeigenkontoebene
</div>

Jede Person mit Zugriff auf ein Anzeigenkonto hat eine spezifische Berechtigung auf Kontoebene: **Account administrator**, **Ad manager**, **Campaign analyst**, **Organic analyst** und **Creative Manager**; siehe **business.x.com** für die aktuellste Dokumentation zu Berechtigungen auf Kontoebene. Anwendungen sollten die Berechtigungen für die aktuell authentifizierte Person über den API-Endpunkt [Authenticated User Access](/de/x-ads-api/campaign-management#authenticated-user-access) abrufen, um zu bestimmen, auf welche API-Endpunkte und Anzeigenfunktionen sie zugreifen kann.

**Hinweis:** Alle Tokens, die mit der [Conversion API](/de/x-ads-api/measurement/web-conversions) verwendet werden, müssen zu Personen mit den Berechtigungen auf Kontoebene **Account administrator** oder **Ad manager** gehören.                                                                                                                                              

<div id="methods-of-obtaining-access-tokens">
  ## Methoden zum Erlangen von Zugriffstokens
</div>

<div id="1-obtain-an-advertisers-user-access-token">
  ### 1. Zugriffstoken eines Werbetreibenden (Benutzer) erhalten
</div>

Es gibt zwei Methoden, um das **Zugriffstoken** eines Werbetreibenden zu erhalten. Die gängigste Methode ist ein [3-legged OAuth-Flow](/de/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) direkt aus Ihrer Web-UI heraus. Anwendungen, die keine für Werbetreibende öffentlich zugängliche UI haben, können einen [PIN-basierten](/de/resources/fundamentals/authentication#pin-based-authorization) OAuth-Prozess implementieren. Nachdem der Benutzer den 3-legged-Flow abgeschlossen hat, verfügt Ihre Anwendung über Zugangsdaten, um über die API Anfragen für sein Ads-Konto zu stellen.

Die Beschaffung von Benutzerzugangsdaten über den OAuth-Flow ist die von uns für die meisten Entwickler der X-Ads-API ausdrücklich empfohlene Methode, um Zugriff auf ein Werbetreibendenkonto zu erhalten. Sie ermöglicht es Ihnen, die API im Namen eines Benutzers aufzurufen und in dessen Namen zu handeln. Diese Tokens laufen nicht ab, können jedoch vom Benutzer jederzeit widerrufen werden.

<div id="2-obtain-your-developer-access-token">
  ### 2. Ihren (Developer-) Zugriffstoken erhalten
</div>

Diese Option erfordert, dass der Werbetreibende [Ihrem @username](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html) (oder mehreren @usernames) über die X-UI unter business.x.com Zugriff auf sein X-Ads-Konto gewährt. **Access Tokens**, die über den 3-legged OAuth-Flow für Ihr Konto erhalten werden, können auf das X-Ads-Konto des Werbetreibenden zugreifen.

So können Sie die API mit den OAuth-Tokens Ihres eigenen @username statt mit den OAuth-Tokens des Werbetreibenden aufrufen. Der wichtige Unterschied bei dieser Option ist, dass Sie Promoted-Only Posts nur erstellen können, wenn Ihrem @username die Berechtigung für Post-Delegation/Composer erteilt wurde.

Um die Berechtigung zu erhalten, Promoted-Only Posts im Namen des `FULL` promotable user im Konto zu erstellen, müssen Sie in diesem Flow ebenfalls die Berechtigung zum Erstellen von Posts erhalten. Dadurch wird der Zugriff über die Berechtigung `TWEET_COMPOSER` am Endpoint [GET accounts/:account\_id/authenticated\_user\_access](/de/x-ads-api/campaign-management#accounts) ermöglicht.

<div id="differences-between-these-methods">
  ## Unterschiede zwischen diesen Methoden
</div>

|     |     |     |
| :--- | :--- | :--- |
|     | OAuth-Token des Werbetreibenden (Benutzer) | (Entwickler-)OAuth-Token  <br /> (@username hinzugefügt, um Konto zu unterscheiden) |
| Zugriff auf Werbekonto | ✔   | ✔   |
| Beiträge im Namen des Benutzers erstellen | ✔   | ✔\*  |
| Kampagnen verwalten | ✔   | ✔   |
| Zugriff auf Analytics | ✔   | ✔   |
| Cards im Namen des Benutzers erstellen | ✔   | ✔   |
| Zugriff des Entwicklers  <br />über die X Ads UI |     | ✔   |
| [Rate Limits](/de/x-api/enterprise-gnip-2.0/fundamentals/rate-limits) | Unterschiedlich pro Werbetreibenden | Unterschiedlich pro Werbekonto |

</Note>

<div id="advertisers-access-token-via-oauth-3-legged-web-flow">
  ### Zugriffstoken des Werbetreibenden über den 3-legged-OAuth-Webflow
</div>

Der Standardflow ist webbasiert und verwendet den OAuth-Flow für die [3-legged-Autorisierung](/de/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow). Die hier gezeigten Screenshots stammen aus einem Beispiel, dessen Quellcode Sie unter [https://github.com/xdevplatform/twauth-web](https://github.com/xdevplatform/twauth-web) einsehen können.

An einem Punkt in Ihrer Anwendung leiten Sie zu X weiter, um Ihre Anwendung zu autorisieren.

</Frame>

Wenn Sie mit dem Request-Token zu X weiterleiten, wird der Nutzer aufgefordert, Ihre Anwendung zu autorisieren.
