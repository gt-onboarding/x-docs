<div id="overview">
  ## Übersicht
</div>

Creatives sind alle Objekte, die in einer Kampagne beworben werden können. Posts können Text, Bilder, GIFs, Videos oder Cards enthalten. Cards können Bilder oder Videos enthalten.

Bild-, GIF- oder Video-Creatives werden entweder über [POST media/upload](/de/x-api/media/upload-media#post-media-upload) — ein einfacher Upload-Endpunkt, der nur Bilder unterstützt — oder über die Endpunkte POST media/upload (chunked) hochgeladen. Diese können zu

Cards hinzugefügt werden:

- POST accounts/:account\_id/cards
  Tweets:
- POST accounts/:account\_id/tweets – um Cards zu Tweets hinzuzufügen, verwenden Sie den Parameter card\_uri.
  Geplante Tweets:
- POST accounts/:account\_id/scheduled\_tweets

Weitere Details zu Cards finden Sie auf der Seite Cards. Die Seite Promoted Video enthält Details zur Verknüpfung von Videos mit Cards oder Tweets.

<div id="cards">
  ### Karten
</div>

Die X-Ads-API unterstützt mehrere Kartentypen, die in Tweets verwendet und anschließend in Kampagnen beworben werden können. Hinweis: Sobald ein Tweet veröffentlicht wurde, sind die Kartendetails öffentlich sichtbar. Dies kann Informationen über die Person enthalten, der die Karte gehört.

<div id="image">
  #### Bild
</div>

Die folgenden Bildspezifikationen gelten für Assets, die in [Cards](/de/x-ads-api/creatives#cards-2) verwendet werden. Bilder müssen 3 MB oder kleiner sein und eine Breite von mindestens 800 px haben. Außerdem unterstützen wir die folgenden Seitenverhältnisse (Breite:Höhe):

- Website: 1:1 und 1,91:1
- Bild-App-Download: 1:1 und 1,91:1
- Umfrage: 1,91:1
- Bild-Konversation: 1,91:1
- Bild-Direktnachricht: 1,91:1

Wir unterstützen die folgenden Bildformate: .bmp, .jpeg und .png.

<div id="video">
  #### Video
</div>

Die folgenden Videospezifikationen gelten für Assets, die in [Cards](/de/x-ads-api/creatives#cards-2) verwendet werden. Wir unterstützen die folgenden Seitenverhältnisse (Breite:Höhe).

- Video-Website: 16:9 und 1:1
- Video-App-Download: 16:9 und 1:1
- Umfrage: 16:9
- Video-Unterhaltung: 16:9
- Video-Direktnachricht: 16:9

<div id="promoted-video">
  ### Promoted Video
</div>

Dieses Dokument bietet einen kurzen Überblick über den Prozess zum Hochladen und Bewerben von Videos über die Ads-API.

Die Ads-API unterstützt Promoted Video in [Tweets](/de/x-ads-api/creatives#tweets-2) und in den folgenden Cards:

- [Video-Website](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [Video-App-Download](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [Video-Konversation](/de/x-ads-api/creatives#video-conversation-cards)

Laden Sie zuerst das Video über den Endpunkt [POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html) hoch. Verwenden Sie die `media_id`, um das Video einem Werbekonto über den Endpunkt POST accounts/:account\_id/videos zuzuordnen. Die `id` des Videos, manchmal auch als `media_key` bezeichnet, wird in nachfolgenden Anfragen verwendet. Dabei handelt es sich um eine Zeichenfolge, die mit einer Ganzzahl beginnt, auf einen Unterstrich folgt und mit einem langen numerischen Wert endet. Beispiel: `13_875943225764098048`.

<div id="promoted-video-in-tweets">
  #### Promoted Video in Tweets
</div>

Um einen Tweet zu erstellen, verwenden Sie den Endpunkt [POST accounts/:account\_id/tweet](/de/x-ads-api/creatives#get-accounts-account-id-tweets) zusammen mit der Video-`id`. In diesem Schritt können Sie außerdem einen Videotitel, eine Beschreibung und einen Call-to-Action (CTA) angeben. Diese Werte sind für Nutzer sichtbar.

<div id="promoted-video-in-cards">
  #### Promoted Video in Cards
</div>

Video App Download- und Video Conversation-Karten unterstützen das Hinzufügen von Posterbildern. Laden Sie ein Bild hoch, das in diesen Karten verwendet werden soll, über den Endpunkt [POST media/upload](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload.html).

Erstellen Sie die Karte über einen der folgenden Endpunkte:

- [POST accounts/:account\_id/cards/video\_website](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [POST accounts/:account\_id/cards/video\_app\_download](https://devcommunity.x.com/t/ads-api-version-11/168814)
- [POST accounts/:account\_id/cards/video\_conversation](https://devcommunity.x.com/t/ads-api-version-11/168814)

unter Verwendung der `id` des Videos und optional der `media_id` des Bildes (für das Posterbild).

Erstellen Sie anschließend den Tweet über den Endpunkt POST accounts/:account\_id/tweet. Karten werden mithilfe des Parameters `card_uri` an Tweets angehängt.

<div id="general-information">
  #### Allgemeine Informationen
</div>

Ausführliche Anleitungen zum Hochladen von Videos über die API finden Sie im [Leitfaden zum Hochladen von Videos](https://developer.x.com/en/docs/media/upload-media/uploading-media.html).

Videos können auch als Pre-Roll-Assets eingesetzt werden. Eine detaillierte Erklärung finden Sie im [Leitfaden zum Pre-Roll-Ziel „Video Views“](/de/x-ads-api/campaign-management#video-views-preroll-objective).

- (Stand: 2015-10-22) Beim Hochladen von Videos, die in beworbenen Inhalten verwendet werden sollen, muss der Parameter `media_category` bei allen `INIT`-Anfragen an den Endpunkt [POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html) auf den Wert `amplify_video` gesetzt werden. Die Verwendung dieses Parameters stellt sicher, dass das Video asynchron vorverarbeitet und für die Verwendung in beworbenen Inhalten vorbereitet wird. Mit dem Befehl `STATUS` kann nach dem Hochladen der Abschluss der asynchronen Verarbeitung überprüft werden.
- Die derzeit zulässige maximale Länge für beworbene Videos beträgt 10 Minuten bei einer Dateigröße von 500 MB oder weniger.
- Hochgeladene Videos sollten entweder im MP4- oder MOV-Format vorliegen.
- Hochgeladene Videos werden in der Regel schnell verarbeitet, die Verarbeitungszeiten können jedoch je nach Videolänge und Dateigröße variieren.
- Hochgeladene Posterbilder sollten im PNG- oder JPG-Format vorliegen. Es gibt keine Anforderungen an Seitenverhältnis oder Größe, das Posterbild wird jedoch an den Videoplayer angepasst.

<div id="scheduled-tweets">
  ### Geplante Tweets
</div>

<div id="introduction">
  #### Einführung
</div>

Geplante Tweets ermöglichen es Werbetreibenden oder Nutzern, einen Tweet zu erstellen, der zu einem späteren Zeitpunkt veröffentlicht werden kann. Zusätzlich zur Erstellung und Verwaltung dieser Tweets erlaubt die API, diese Tweets einem Line Item zuzuordnen, damit sie beworben werden können, sobald der Tweet live ist. So können Werbetreibende native Tweets im Voraus erstellen und ihre Kampagnen-Creatives vor wichtigen Maßnahmen planen. Zum Beispiel kann ein Tweet-Creative so vorbereitet werden, dass es unmittelbar zur Ankündigung eines neuen Produkts live geht.

Der vollständige Funktionsumfang, den die API-Endpunkte für Geplante Tweets bereitstellen, ist unten aufgeführt:

- Neue Geplante Tweets erstellen, ändern und anzeigen
- Einen Geplanten Tweet einem Line Item zuordnen
- Bestehende Geplante Tweets abfragen und verwalten
- Sobald ein Geplanter Tweet live ist, die Live-Tweet-`id` abrufen

</div>

- [GET accounts/:account\_id/scheduled\_tweets/preview/:scheduled\_tweet\_id](/de/x-ads-api/creatives#scheduled-tweets-2) (einen bestehenden geplanten Tweet anzeigen)

Da geplante Tweets eigenständige Entitäten gegenüber „Live“-Tweets sind, werden beim Erstellen oder Bearbeiten dieser Tweets zwei unterschiedliche Validierungssets ausgeführt. Das erste Validierungsset wird während der Erstellung des geplanten Tweets ausgeführt, konkret:

<div id="scheduled-tweet-create">
  #### Geplanten Tweet erstellen:
</div>

- Prüfen Sie, dass der authentifizierte Nutzer berechtigt ist, organische Tweets für einen bestimmten @handle zu erstellen. Für die Erstellung von „Promoted-Only“-Tweets muss der authentifizierte Nutzer ein Kontonutzer mit [Tweet Composer-Berechtigungen](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access) sein.
- Prüfen Sie, dass nicht mehr als 30 Tweets innerhalb eines 15‑Minuten‑Fensters rund um den scheduled\_at‑Zeitpunkt eingeplant sind. Die Fehlermeldung SCHEDULED\_TWEET\_LIMIT\_EXCEEDED weist darauf hin, dass zu viele geplante Tweets innerhalb desselben zukünftigen 15‑Minuten‑Zeitraums eingeplant wurden. Werbetreibende müssen einen vorhandenen geplanten Tweet entfernen oder den scheduled\_at‑Zeitpunkt nach vorne oder hinten verschieben.

<div id="scheduled-tweet-goes-live">
  #### Geplanter Tweet geht „live“:
</div>

- Diese Validierungsregeln werden zum Zeitpunkt von scheduled\_at ausgeführt und sind identisch mit denen, die bei der regulären Tweet-Erstellung in der API angewendet werden. Geht ein geplanter Tweet beispielsweise nicht live, wird der scheduled\_status auf FAILED gesetzt, wenn der geplante Tweet sowohl ein Bild als auch ein GIF enthält

<div id="workflow">
  #### Workflow
</div>

**Einen neuen geplanten Tweet erstellen**

Ein neuer geplanter Tweet kann über den Endpunkt [POST accounts/:account\_id/scheduled\_tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets) erstellt werden. Dieser Endpunkt hat die folgenden erforderlichen Parameter: den Zeitpunkt `scheduled_at` sowie den Tweet-`text`, wenn keine Media-Entities im Tweet enthalten sind. Zusätzlich bietet dieser Endpunkt einige weitere Optionen, mit denen Sie einen geplanten Tweet im Namen eines anderen @handle über den Parameter `as_user_id` erstellen können, sowie die Möglichkeit, eine Card (`card_uri`) und beliebige Medien (`media_ids`) hinzuzufügen. Beachten Sie: Ein Tweet kann nur Entities desselben Typs enthalten, d. h. entweder Video, GIF oder Bild. Der Parameter `nullcast` steuert, ob der Tweet ein „nur beworbener“ Tweet ist oder nicht. Alle neu erstellten geplanten Tweets sind standardmäßig „nur beworben“ (`nullcast=true`). Wenn `nullcast=false`, wird ein organischer geplanter Tweet erstellt.

Sobald ein geplanter Tweet erfolgreich erstellt wurde, enthält die Antwort ein Feld `id`, das den eindeutigen Bezeichner des geplanten Tweets selbst angibt. Zusätzlich wird ein weiteres Feld namens `tweet_id` zurückgegeben. Dieses Feld ist zunächst `null`, wird jedoch, sobald der Tweet live geht, mit der ID des „Live“-Tweets befüllt.

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_tweets" -d 'scheduled_at=2017-12-24T23:59:00Z&text=Happy Holidays!'
```

Dadurch wird der folgende geplante Tweet erstellt:

```json
{

      "text": "Happy Holidays!",
```

**Einen geplanten Tweet einem Line Item zuordnen**

Während geplante Tweets zur Erstellung organischer Tweets verwendet werden können, ermöglichen wir Partnern auch, einen „Nur-beworben“-Tweet (`nullcast=true`) zu erstellen – beide Varianten können einem Line Item zugeordnet werden. Um dies zu ermöglichen, stellen wir zudem einen Endpunkt [POST accounts/:account\_id/scheduled\_promoted\_tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets) bereit. Dieser Endpunkt erlaubt in einem einzelnen API-Aufruf nur die Zuordnung eines einzigen beworbenen geplanten Tweets zu einem Line Item. Um mehrere geplante Tweets demselben Line Item zuzuordnen, sind mehrere API-Aufrufe erforderlich.

Bitte beachten Sie, dass es nicht möglich ist, einen bestehenden beworbenen geplanten Tweet zu ändern.

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_promoted_tweets" -d 'line_item_id=a44qc&scheduled_tweet_id=917507899668099072'
```

```json
{
  "data": {
    "line_item_id": "a44qc",
    "id": "26576",
    "created_at": "2017-10-09T22:24:16Z",
    "updated_at": "2017-10-09T22:24:16Z",
    "scheduled_tweet_id": "917507899668099072",
    "tweet_id": null,
    "deleted": false
  },
  "request": {
    "params": {
      "line_item_id": "a44qc",
      "scheduled_tweet_id": 917507899668099100,
      "account_id": "aaaaa"
    }
  }
}
```

Dieser Endpunkt erstellt lediglich eine Zuordnung zwischen einem bestimmten geplanten Tweet und einem Line Item. Sobald die Kampagnen-/Line-Item-Flugdaten aktuell sind, beginnt das Line Item automatisch, den entsprechenden „Live“-Tweet auszuliefern. Zwar prüfen wir in diesem Schritt, dass sich der geplante Tweet im Zustand `SCHEDULED` befindet und für das angegebene Ziel gültig ist, es werden jedoch keine weiteren Validierungen durchgeführt. Alle verbleibenden Validierungsregeln, die für das Line Item und den geplanten Tweet gelten, werden ausgeführt, wenn der Tweet „live“ geht.

Um sicherzustellen, dass es keine Probleme bei der Auslieferung der Kampagne gibt, wird empfohlen, den geplanten Tweet über `scheduled_at` auf einen Zeitpunkt vor den Kampagnen-/Line-Item-Flugdaten zu setzen.

Angenommen, der geplante Tweet soll nach dem Kampagnenstartdatum live gehen (und es gibt nur einen einzelnen Tweet, der einem einzelnen Line Item zugeordnet ist), dann ist die Kampagne `ACTIVE`. Da der geplante Tweet jedoch noch nicht live ist, stehen keine Creatives zur Auslieferung zur Verfügung.

**Verwaltung geplanter Tweets**

Die verbleibenden Endpunktgruppen ermöglichen API-Nutzenden, all ihre geplanten Tweets und beworbenen geplanten Tweets zu verwalten. Diese APIs können verwendet werden, um entweder eine Liste aller geplanten Tweets zurückzugeben – optional gefiltert nach einem bestimmten Zustand – sowie um einen bestimmten geplanten Tweet anhand seiner `id` nachzuschlagen.

<div id="what-happens-when-a-scheduled-tweet-goes-live">
  #### Was passiert, wenn ein geplanter Tweet live geht?
</div>

Sobald ein bestimmter geplanter Tweet kurz davor ist, live zu gehen, bzw. zum Zeitpunkt `scheduled_at`, werden die folgenden Aktualisierungen vorgenommen:

- Der „Live“-Tweet wird erstellt; dies kann jedoch eine Latenz von bis zu 1 Sekunde aufweisen
- Die `tweet_id` wird den folgenden Entities hinzugefügt:
- Geplanter Tweet
- Promoted Geplanter Tweet
- Eine neue Promoted-Tweet-Entity wird erstellt

<div id="best-practices">
  #### Bewährte Vorgehensweisen
</div>

Die folgenden bewährten Vorgehensweisen werden empfohlen, wenn Sie geplante Tweets erstellen oder bewerben:

- Stellen Sie sicher, dass der Tweet beim Erstellen des geplanten Tweets gültig ist (z. B. kann ein Tweet nur entweder ein Bild, ein Video oder ein GIF enthalten – keine Kombination der drei)
- Stellen Sie sicher, dass die Kampagnenlaufzeiten (d. h. `start_time` und `end_time`) mit dem `scheduled_at`-Zeitpunkt des geplanten Tweets übereinstimmen
- Geplante Tweets sollten nicht mehr als ein Jahr im Voraus terminiert werden (365 Tage)
- Eine Tweet-Vorschau wird für geplante Tweets derzeit nicht unterstützt (die Möglichkeit, geplante Tweets vor der Erstellung zu prüfen)

Die Media-Library-Endpunkte ermöglichen die Verwaltung von Bildern, GIFs und Videos für X Ads-Konten. Medienassets in der Bibliothek können in Tweets und zur Erstellung von Cards verwendet werden. Sie können außerdem in mehreren [Creatives](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/overview) wiederverwendet werden, sodass dasselbe Asset nicht mehrfach hochgeladen werden muss.

  #### API-Endpunkte
</div>

- [POST media/upload](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload#post-media-upload) oder [POST media/upload (chunked)](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload-init#post-media-upload-init) (Medien hochladen)

<div id="request-parameters">
  #### Anfrageparameter
</div>

Alle POST-Anfragen der Media Library erfordern eine Medien-ID. Dieser Wert wird während des Upload-Schritts zurückgegeben. Bei Verwendung der `media_id`, wie im obigen Beispiel, muss auch eine `media_category` angegeben werden. Es gibt vier mögliche Kategorie-Werte: `AMPLIFY_VIDEO`, `TWEET_GIF`, `TWEET_IMAGE` und `TWEET_VIDEO`.

Optional können für Objekte in der Media Library die Werte `name` und `file_name` gesetzt werden. Diese Attribute helfen Nutzerinnen und Nutzern, Medienvarianten in der Bibliothek zu unterscheiden.

Für Videos können außerdem ein Titel und eine Beschreibung festgelegt werden. Diese Werte sollen als die Anfrageparameter `video_title` und `video_description` mit dem Endpoint [POST accounts/:account\_id/tweet](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet) übermittelt werden. Im Tweet erscheint dieser Text unter dem Video.

<div id="attributes">
  #### Attribute
</div>

Die Media Library führt formal das Konzept des media\_key ein. Das ist der eindeutige Bezeichner für Objekte in der Bibliothek. Media Keys sind Zeichenfolgen im folgenden Format: 13\_875943225764098048. Diese werden in allen unseren Card-Endpunkten vollständig unterstützt.

Zusätzlich enthält die Antwort der Media Library die media\_id, dargestellt als Zeichenfolge. Diese ist für Ressourcen enthalten, die derzeit keinen Media Key akzeptieren: [Tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet)\*, [Tweet-Vorschau](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#get-accounts-account-id-tweet-preview)\* und [Geplante Tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets). Wir arbeiten daran, Media Keys überall zu unterstützen.

Das Attribut aspect\_ratio wird für GIFs und Videos zurückgegeben. Es kann verwendet werden, um Medien für Cards zu filtern, die nur bestimmte Seitenverhältnisse akzeptieren.

\*Diese Endpunkte unterstützen den Parameter video\_id, der ein Media Key ist.

<div id="introduction">

</div>

Cards sind anpassbare Anzeigenformate, die Medien nutzen und mit einer Website, einer App oder mit Handlungsaufrufen verknüpft werden können, um bestimmte Nutzerinteraktionen zu fördern, etwa das Starten einer Direct Message. Sie können an Tweets, geplante Tweets oder Tweet-Entwürfe angehängt werden.

Auf Cards kann in Tweet-Objekten auf zwei Arten verwiesen werden: über die card\_uri der Card oder über ihre preview\_url. Beispielwerte für beide sind unten dargestellt.

| :--- | :--- |
| card://1043282691834048513 | https://cards.x.com/cards/18ce54d4x5t/68w3s |

**Hinweis**: Seit Version 3 der X-Ads-API wird nur die card\_uri generiert und in der cards-Antwort für neu erstellte Cards zurückgegeben.

<div id="identifying-media">
  ### Medien identifizieren
</div>

<div id="introduction">
  #### Einführung
</div>

Medien – Bilder, GIFs und Videos – können zu Tweets und Cards hinzugefügt werden. Außerdem können Videos als Pre-Roll-Assets verwendet und Bilder auf der [X Audience Platform](https://developer.x.com/en/docs/ads/measurement/overview/twitter-audience-platform) beworben werden. Dieser Abschnitt beschreibt, wie Medienreferenzen über diese Entities hinweg gefunden werden.

Es gibt zwei Arten von Medienkennungen: IDs und Keys. Beispielwerte für beide sind unten aufgeführt.

| **Media-ID** | **Media-Key** |
| :--- | :--- |
| 1029825579531807971 | 13\_1029825579531807971 |

Der Media-Key besteht aus der ID mit einem numerischen Präfix und einem Unterstrich.

<div id="images">
  #### Bilder
</div>

Die folgende Tabelle zeigt die derzeit verfügbaren Identifikatortypen in den Antworten der einzelnen bildbezogenen Ressourcen sowie die entsprechenden Attributnamen.

| **Resource** | **Identifier** | **Attribute(s)** |
| :--- | :--- | :--- |
| Image cards | None |  |
| Tweet | Both | `entities["media"]["id_str"]`  `entities["media"]["media_key"]` |
| Scheduled Tweet | Both | `media_ids` and `media_keys` |
| Draft Tweet | Both | `media_ids` and `media_keys` |
| Account Media | None |  |
| Media Library | Both | `media_id` and `media_key` |

Image cards und Account Media-Bilder enthalten keinen Verweis auf einen Medien-Identifier. Tweets enthalten nur Medien-IDs. Scheduled und Draft Tweets enthalten sowohl die Medien-ID als auch den Medien-Key. Die Media Library gibt ebenfalls beides zurück.

Bei Tweets entsprechen die Felder id und id\_str im Objekt innerhalb des Arrays entities\["media"] der Medien-ID. In Fällen, in denen ein Tweet mehrere Bilder enthält, finden sich die Verweise auf jede Medien-Entity nur in extended\_entities\["media"].

Zusätzlich zu Identifier-Verweisen ist es oft wichtig, Zugriff auf die Bild-URL zu haben.

| **Resource** | **Attribute(s)** | **Format** |
| :--- | :--- | :--- |
| Image cards | `image` | .jpg |
| Tweet\* | `entities["media"][0]["media_url"]` or `extended_entities["media"][i]["media_url"]` | .jpg |
| Scheduled Tweet | None |  |
| Draft Tweet | None |  |
| Account Media | `media_url` | .jpg |
| Media Library | `media_url` | .jpg |

\* Der Speicherort dieser URL hängt davon ab, ob der Tweet ein einzelnes oder mehrere Bilder enthält.

Alle Image cards enthalten ein image-Response-Attribut, das die X-Bild-URL enthält. (Bei Image App Download Cards lautet der Name wide\_app\_image.)

Bei Tweets hängt der Speicherort der Medien-URL sowohl vom Medientyp als auch vom verwendeten Endpoint ab. Bei Tweets mit einem einzelnen Bild ist die URL in entities\["media"]\[0]\["media\_url"] zu finden. Dies gilt sowohl für die Ads API als auch die Standard API. Enthält ein Tweet mehrere Bilder, sind die URLs jedoch nur in extended\_entities\["media"]\[i]\["media\_url"] zu finden. Dies ist nur in der Standard API verfügbar.

<div id="videos">
  #### Videos
</div>

Die folgende Tabelle zeigt die derzeit verfügbaren Kennzeichnertypen in der Antwort jedes videobezogenen Resources sowie die entsprechenden Attributnamen.

| **Resource** | **Identifier** | **Attribute(s)** |
| :--- | :--- | :--- |
| Video cards | Kann beides sein | `video_content_id` |
| Video poll cards | Keine |  |
| Tweet | Beide | `entities["media"]["id_str"]`  `entities["media"]["media_key"]` |
| Scheduled Tweet | Beide | `media_ids` und `media_keys` |
| Draft Tweet | Beide | `media_ids` und `media_keys` |
| Account Media | Media key | `video_id` |
| Media Library | Beide | `media_id` und `media_key` |

Während Video cards (mit Ausnahme von Poll cards mit Video) ein video\_content\_id-Antwortattribut enthalten, gibt es Inkonsistenzen beim Typ des zurückgegebenen Werts. In einigen Fällen ist es eine Media-ID; in anderen ist es ein Media-Key.

Informationen dazu, wie auf die URL des Videos zugegriffen werden kann, sind unten dargestellt.

| **Resource** | **Attribute(s)** | **Format** |
| :--- | :--- | :--- |
| Video cards | `video_url` und `video_hls_url` | .vmap  .m3u8 |
| Tweet with video | `extended_entities["media"][i]["video_info"]["variants"][j]["url"]` | .mp4 |
| Scheduled Tweet | Keine |  |
| Draft Tweet | Keine |  |
| Account Media | Keine |  |
| Media Library | `media_url` | .mp4 |

Video cards enthalten die Antwortattribute video\_url und video\_hls\_url mit .vmap- bzw. .m3u8-URLs.

<div id="media-library">
  #### Medienbibliothek
</div>

Mitunter ist es erforderlich, zusätzliche Informationen zu einem Medienobjekt abzurufen. Ein Anwendungsfall, insbesondere bei Video-Cards, ist das Abrufen der MP4-URL anstelle der VMAP-URL. Dies ist in der Medienbibliothek möglich. Einzelheiten zu den verfügbaren Informationen finden Sie in unserem [Leitfaden zur Medienbibliothek](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/guides/media-library). Die meisten Assets, die zum FULL promotable user des Anzeigenkontos gehören, sind in der Bibliothek zu finden. Es gibt jedoch einige Ausnahmen.

**Abrufen von Medien**

Wie oben erwähnt, enthalten Image-Cards keine Verweise auf Medien-IDs oder Media Keys. Daher ist es nicht möglich, deren Assets über die Medienbibliothek abzurufen. Dies gilt auch für Bilder aus [Account Media](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/account-media#account-media).

Video-Cards setzen voraus, dass das Video-Asset bereits vor der Erstellung Teil der Medienbibliothek (oder zuvor der Videos-Ressource) ist. Infolgedessen sind diese Assets in der Medienbibliothek immer abrufbar. Dies gilt auch für Account-Media-PREROLL-Assets.

Schließlich ist Medieninhalt in Tweets stets garantiert in der Medienbibliothek vorhanden.

Die folgende Tabelle fasst zusammen, welche Assets in der Medienbibliothek abrufbar sind, unter Berücksichtigung dessen, ob die Ressourcenantwort einen Bezeichner enthält, der für die Suche verwendet werden kann.

| **Ressource** | **In der Medienbibliothek** |
| :--- | :--- |
| Image-Cards | Nein |
| Video-Cards | Ja\* |
| Tweets (beliebige Medien)\*\* | Ja |
| Geplante Tweets | Ja |
| Tweet-Entwürfe | Ja |
| Account-Media-Bilder | Nein |
| Account-Media-Videos (`PREROLL`) | Ja |

\* Für Cards, bei denen die `video_content_id` ein Media Key ist. Wenn der Wert eine Medien-ID ist, existiert das Asset weiterhin in der Medienbibliothek, aber das Abrufen erfordert das Anhängen eines numerischen Präfixes und eines Unterstrichs.
\*\* Tweets geben nur Medien-IDs zurück. Das Asset ist zwar garantiert in der Medienbibliothek vorhanden, das Abrufen erfordert jedoch das Anhängen eines numerischen Präfixes und eines Unterstrichs.

**Interaktionen mit Account Media**

Es gibt zwei Fälle, in denen Medien-Assets, die zur Bibliothek hinzugefügt werden, automatisch zur Ressource Account Media hinzugefügt werden.

- Wenn ein AMPLIFY\_VIDEO-Asset zur Medienbibliothek hinzugefügt wird, wird es automatisch als Account-Media-Asset mit dem Creative-Typ PREROLL hinzugefügt.
- Wenn Bilder mit bestimmten Abmessungen (siehe „Creative Types“ auf unserer [Enumerations-Seite](https://developer.x.com/content/developer-twitter/en/docs/ads/general/overview/enums)) zur Medienbibliothek hinzugefügt werden, werden sie automatisch als Account-Media-Assets hinzugefügt. Der Creative-Typ (z. B. INTERSTITIAL) hängt von den Bildabmessungen ab.

<div id="tweets">
  ### Tweets
</div>

<div id="introduction">
  #### Einführung
</div>

Die X-Ads-API unterstützt drei Arten von Tweets: veröffentlicht, geplant und als Entwurf.

<div id="nullcasted-tweets">
  #### Nullcasted Tweets
</div>

Tweets können entweder nullcasted (auch „Nur beworben“) oder organisch sein. Nullcasted Tweets erscheinen nach der Veröffentlichung nicht in der öffentlichen Timeline des Nutzers, obwohl sie öffentlich sind. Organische Tweets hingegen werden den Followern des Nutzers ausgespielt und erscheinen in der öffentlichen Timeline des Nutzers.

**Tweets erstellen**

Jeder der drei Endpunkte zum Erstellen von Tweets unterstützt einen booleschen Parameter nullcast, der dem API-Nutzer die Option gibt, nullcasted oder organische Tweets zu erstellen. Nullcasted Tweets können vom Nutzer selbst oder von jedem erstellt werden, der die Berechtigung hat, Tweets im Namen des Nutzers zu erstellen. Organische Tweets können nur vom [vollständig bewerbbaren Nutzer](https://developer.x.com/content/developer-twitter/en/docs/tutorials/ads-api-hierarchy-terminology#promotable-users) erstellt werden.

**Tweets aktualisieren**

Es ist möglich, die Nullcast-Eigenschaft für geplante und Entwurfs-Tweets zu aktualisieren. Bei geplanten Tweets können Bearbeitungen bis zur scheduled\_at-Zeit des Tweets vorgenommen werden. Entwurfs-Tweets können unbegrenzt bearbeitet werden. Nach der Veröffentlichung ist es jedoch nicht möglich, einen Tweet von nullcasted zu organisch zu ändern oder umgekehrt.

<div id="promoting-tweets">
  #### Bewerben von Tweets
</div>

Nur veröffentlichte und geplante Tweets können beworben werden. Diese können entweder nullcasted oder organisch sein; es gibt keine Einschränkungen. Werbetreibende können ihre eigenen Tweets oder die Tweets anderer Nutzer bewerben, sofern sie die entsprechende Erlaubnis erhalten haben. (Siehe: [Promoting another user's Tweets](https://developer.x.com/content/developer-twitter/en/docs/tutorials/promoting-another-users-content) für weitere Informationen.)

Mehrere Tweets können in einer einzigen Kampagne beworben werden. Ebenso kann ein einzelner Tweet in einer oder mehreren Kampagnen beworben werden.

Um veröffentlichte Tweets zu bewerben, verwenden Sie den Endpoint [POST accounts/:account\_id/promoted\_tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/promoted-tweets#post-accounts-account-id-promoted-tweets). Dadurch werden veröffentlichte Tweets einem Line Item zugeordnet. Um geplante Tweets zu bewerben, verwenden Sie den Endpoint [POST accounts/:account\_id/scheduled\_promoted\_tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets).

<div id="tweet-ids">
  #### Tweet-IDs
</div>

Veröffentlichte, geplante und Entwurfs-Tweet-IDs sind [numerisch](https://developer.x.com/content/developer-twitter/en/docs/basics/twitter-ids) — es handelt sich um 64-Bit-Ganzzahlen ohne Vorzeichen. Die ID des folgenden veröffentlichten Tweets lautet beispielsweise 1166476031668015104.

Wenn veröffentlichte oder geplante Tweets beworben werden, wird eine entsprechende beworbene Tweet-Entität erstellt. Diese Entitäten haben eigene IDs, die alphanumerisch sind und als base36-codierte Werte dargestellt werden. Wird zum Beispiel der oben veröffentlichte Tweet beworben — das heißt, einem Line Item 6c62d zugeordnet — ergibt sich folgende API-Antwort.

<div id="carousels">
  ### Karussells
</div>

<div id="introduction">
  #### Einführung
</div>

Die X-Ads-API unterstützt das Erstellen und Abrufen von Video- und Bildkarussells. Das Karussell ist ein Kartentyp, der zwischen 2 und 6 Medienobjekte enthalten kann. Die Karussellkarte kann Nutzer auf eine Website leiten oder sie zum Installieren einer mobilen App motivieren. Weitere Informationen zu Karussells, ihren Vorteilen, Best Practices und FAQs finden Sie auf unserer Seite [Carousel Ads on X](https://business.x.com/en/advertising/carousels.html).

Ein Karussell kann, wie jeder andere Kartentyp, in Tweets verwendet werden, und diese Tweets können anschließend beworben werden. Der Ablauf ist derselbe, den Sie bereits kennen:

1. Medien hochladen
2. Karte erstellen
3. Tweet erstellen
4. Tweet bewerben

Der einzige Unterschied liegt in der Art und Weise, wie die Karte erstellt wird. Während andere Anforderungen zum Erstellen von Karten Query-Parameter akzeptieren, akzeptieren Anforderungen zum Erstellen von Karussellkarten ausschließlich JSON-POST-Bodies.

<div id="endpoints">
  #### Endpunkte
</div>

Die X-Ads-API unterstützt das Erstellen und Abrufen von Karussells.

Um ein Karussell – beliebiger Art – zu erstellen, verwenden Sie den Endpunkt [POST accounts/:account\_id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#post-accounts-account-id-cards). Zum Abrufen von Karussells verwenden Sie den Endpunkt [GET accounts/:account\_id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#get-accounts-account-id-cards).

<div id="json-post-body">
  #### JSON-POST-Body
</div>

Karussells werden mit zwei Komponenten erstellt. Die erste gibt die zu verwendenden Medienassets an. Die zweite enthält Informationen entweder zur Website oder zur App.

Konkret wird eine Karussellkarte in folgender Reihenfolge mit diesen Komponenten erstellt:

- Eine `SWIPEABLE_MEDIA`-Komponente, die ein Array von Media Keys akzeptiert
- _Eine_ der folgenden:
- Eine `DETAILS`-Komponente zur Angabe von Website-Informationen
- Eine `BUTTON`-Komponente zur Angabe von App-Informationen

Die `SWIPEABLE_MEDIA`-Komponente muss ein `media_keys`-Array enthalten, in dem Sie zwischen 2 und 6 Bilder oder Videos angeben können. Die Reihenfolge, in der die Media Keys übergeben werden, bestimmt die Reihenfolge, in der sie gerendert werden.

```
{
  "type": "SWIPEABLE_MEDIA",
  "media_keys": [
    "13_1089771253848666112",
    "13_1191948012077092867"
  ]
}
```

Zur Erinnerung: Sie können Media Keys erhalten, indem Sie eine Anfrage an den Endpoint [GET accounts/:account\_id/media\_library](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/media-library#get-accounts-account-id-media-library) stellen.

Die Zusammensetzung des zweiten Komponentenobjekts hängt davon ab, ob Sie Nutzer auf eine Website leiten oder sie zur Installation einer App motivieren möchten. Die folgende Tabelle fasst die beiden Optionen zusammen. (**Hinweis**: Alle aufgeführten Keys sind erforderlich.)

|  | **Website** | **App** |
| :--- | :--- | :--- |
| Komponententyp angeben | `"type": "DETAILS"` | `"type": "BUTTON"` |
| Titel/Label | `"title": "X"` | `"label": { "type": "ENUM", "value": "INSTALL" }` |
| Ziel | `"destination": { "type": "WEBSITE", "url": "https://www.x.com" }` | `"destination": { "type": "APP", ... }` |

Zusammengefügt ergibt sich der folgende Beispiel-JSON-POST-Body für ein Website-Karussell.

```
{
  "name": "website carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "DETAILS",
      "title": "X",
      "destination": {
        "type": "WEBSITE",
        "url": "https://www.x.com"
      }
    }
  ]
}
```

App-Zielobjekte innerhalb von `BUTTON`-Komponenten erfordern einen Ländercode und mindestens einen App-Bezeichner. Optional werden Deep Links unterstützt. Eine Beschreibung dieser Felder finden Sie in der [Referenzdokumentation](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#app-destination).

Zusammengefügt ergibt sich der folgende Beispiel-JSON-POST-Body für ein App-Karussell.

```
{
  "name": "app carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        "type": "APP",
        "country_code": "US",
        "googleplay_app_id": "com.twitter.android",
        "iphone_app_id": "333903271"
      }
    }
  ]
}
```

<div id="creative-metadata-tagging">
  ### Creative-Metadaten-Tagging
</div>

<div id="introduction">
  #### Einführung
</div>

Dieser Leitfaden richtet sich an Kreativpartner, Agenturen und kreative Entwickler, um Assets, die in X-Kampagnen verwendet werden, zu kennzeichnen und den individuellen Asset-Wert sowie die Performance besser zu verstehen.

**Hinweis:** Medienassets dürfen ausschließlich von dem Partner oder Entwickler gekennzeichnet werden, der das Medienasset erstellt hat. Wenn der Nutzer des Medienassets dieses nicht erstellt hat, implementieren Sie kein Metadaten-Tagging.

Creative Metadata Tagging ermöglicht die Attribution von Bildern und Videos, die von Kreativpartnern erstellt wurden – unabhängig davon, wo das Asset auf X hochgeladen wird oder von welcher Entität es hochgeladen wird. Um die Verbindung zwischen dem kreativen Asset und dem Kreativpartner herzustellen, wird der Standard [XMP](https://exiftool.org/TagNames/XMP.html) verwendet.

<div id="tagging-creative-assets">
  #### Tagging kreativer Assets
</div>

Die folgende Tabelle zeigt die in den Antworten der bildbezogenen Ressourcen derzeit verfügbaren Identifikatortypen sowie die entsprechenden Attributnamen. Zum Taggen kreativer Assets wird ein Tagging-Tool benötigt. Empfohlen wird [ExifTool](https://exiftool.org/), eine plattformunabhängige Perl-Bibliothek plus eine [Kommandozeilenanwendung](https://exiftool.org/exiftool_pod.html) zum Lesen, Schreiben und Bearbeiten von Metadaten. Alle unterstützten [Dateitypen](https://exiftool.org/#supported) anzeigen.

Folgen Sie den bereitgestellten [Anweisungen zur Installation von ExifTool](https://exiftool.org/install.html). Es gibt außerdem Softwarepakete über [Homebrew](https://formulae.brew.sh/), die die Installation weiter vereinfachen, indem sie den [Installationsbefehl für exiftool](https://formulae.brew.sh/formula/exiftool) für macOS und Linux bereitstellen. Bestätigen Sie die ordnungsgemäße Installation, indem Sie in der Kommandozeile exiftool -ver eingeben; damit wird die Versionsnummer des Tools ausgegeben. Weitere Informationen zu ExifTool-Befehlsparametern finden Sie in der [ExifTool-Dokumentation](https://exiftool.org/).

Kreativpartner können Metadaten-Tags für neue oder bestehende kreative Assets mit ihrer X app\_id dem Contributor-XMP-Tag sowie dem Datums-Tag zuweisen. Die kreativen Assets unterliegen den bestehenden Größenbeschränkungen beim [Hochladen von Medien](https://developer.x.com/en/docs/twitter-api/v1/media/upload-media/overview). 

Hinweis: Die Verwendung des Contributor-XMP-Tags durch X stellt sicher, dass Metadaten ausschließlich Werte für Kampagnen auf X erfassen.

`exiftool -contributor="<YOUR APP ID>" -creative_file.jpg`

`exiftool -date="<date>" -creative_file.jpg`

Die app\_id finden Sie im [Developer Portal](https://developer.x.com/en/portal/dashboard) unter Projects & Apps. Beispiel: 16489123

Das folgende Beispiel fügt app\_id als Contributor-Tag und date als Datums-Tag für ein Bild hinzu:

```
app_id:858382169
```

```
date:2022-03-13
```

```
creative_file: eiffel_tower.jpg
```

```
exiftool -contributor=858382169 eiffel_tower.jpg
```

```
   1 image files updated
```

```
exiftool -date=2022-03-13 eiffel_tower.jpg
```

```
  1 image files updated
```

Überprüfen Sie, ob Ihr Bild korrekt getaggt wurde: 

`exiftool -xmp:all -G1 <filename>`

Beispiel:

`exiftool -xmp:all -G1 eiffel_tower.jpg`

```
[XMP]        XMP Toolkit            : Image::ExifTool 12.30
```

```
[XMP]        Contributor            : 858382169
```

```
[XMP]        Date                   : 2022:03:13
```

<div id="components">
  #### Komponenten
</div>

Jede Komponente muss ein `type`-Feld enthalten, das das Schema des Objekts bestimmt. Die X-Ads-API unterstützt die folgenden Komponententypen, gruppiert in medien- und beschreibungsbasierte Komponenten.

- Medien:
- `MEDIA`: einzelnes Video oder Bild
- `SWIPEABLE_MEDIA`: zwischen 2–6 Videos oder Bildern
- Beschreibung:
- `DETAILS`
- `BUTTON`

Jede Komponente hat einen Satz erforderlicher Felder (zusätzlich zum Schlüssel `type`). Diese sind in der folgenden Tabelle aufgeführt.

| Komponenten-`type` | Feld | Werttyp |
| :--- | :--- | :--- |
| `MEDIA` | `media_key` | string |
| `SWIPEABLE_MEDIA` | `media_keys` | array of strings |
| `DETAILS` | `title`   `destination` | string   object |
| `BUTTON` | `label`   `destination` | object   object |

Im Folgenden sehen Sie ein Beispiel für eine `BUTTON`-Komponente im Kontext des Arrays `components` (der Schlüssel `name` wird absichtlich weggelassen). (Die Auslassungspunkte markieren Stellen, an denen weitere Informationen angegeben werden müssten.)

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        ...
      },
      "destination": {
        ...
      }
    }
  ]
}
```

Die Reihenfolge, in der die Komponentenobjekte angegeben werden, bestimmt die Reihenfolge von oben nach unten, in der sie gerendert werden. Cards müssen mit einer medienbasierten Komponente und entweder einer `DETAILS`- oder `BUTTON`-Komponente erstellt werden. Beschreibungsbasierte Komponenten werden unterhalb der Medien gerendert und haben zugehörige Ziele, entweder URLs oder mobile Apps.

**Label**

Labels definieren den auf Buttons angezeigten Text und gelten daher nur für die `BUTTON`-Komponente. Label-Objekte haben zwei erforderliche Schlüssel: `type` und `value`. Der `type` muss auf `ENUM` gesetzt sein und der `value` kann einer der folgenden sein: `BOOK`, `CONNECT`, `INSTALL`, `OPEN`, `ORDER`, `PLAY` oder `SHOP`.

Aufbauend auf dem vorherigen Beispiel zeigt das Folgende das Objekt `label` innerhalb der `BUTTON`-Komponente.

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        ...
      }
    }
  ]
}
```

**Destination**

Destinations sind die Ziele, zu denen Werbetreibende Nutzer führen möchten. Sie sind innerhalb von `DETAILS`- oder `BUTTON`-Komponenten immer erforderlich. Es gibt zwei Destination-Typen: `WEBSITE` oder `APP`.

**Hinweis**: Website-Destinations können nur mit `DETAILS`-Komponenten verwendet werden und App-Destinations nur mit `BUTTON`-Komponenten.

**Website-Destination**

| Name | Beschreibung |
| :--- | :--- |
| type   _required_ | Der Destination-Typ, der das Schema bestimmt.  Typ: enum  Mögliche Werte: `WEBSITE` |
| url   _required_ | Die URL der Website, zu der ein Nutzer weitergeleitet werden soll.  Typ: string  Beispiel: `https://devcommunity.x.com/c/advertiser-api` |

**App-Destination**

| Name | Beschreibung |

| country\_code   _required_ | Der [ISO 3166-1 alpha-2](https://en.wikipedia.org/wiki/ISO_3166-1_alpha-2) Zwei-Buchstaben-Code für das Land, in dem die App vertrieben wird.  Typ: string  Beispiel: `US` |
| googleplay\_app\_id   _sometimes required_ | Der Google-Play-Paketname der Anwendung.  **Hinweis**: Mindestens eines der folgenden Felder ist erforderlich: `ios_app_store_identifier` oder `googleplay_app_id`.  Typ: string  Beispiel: `com.twitter.android` |
| ios\_app\_store\_identifier   _sometimes required_ | Die iOS-App-Store-ID.  **Hinweis**: Mindestens eines der folgenden Felder ist erforderlich: `ios_app_store_identifier` oder `googleplay_app_id`.  Typ: string  Beispiel: `333903271` |

#### POST accounts/:account\_id/tweet

Erstellt einen Tweet für den vollständigen promotablen Nutzer (Standard) des Kontos oder den im Parameter `as_user_id` angegebenen Nutzer. Sowohl nullgecastete (Standard) als auch organische Tweet-Erstellung wird unterstützt. Nullgecastete Tweets erscheinen nicht in der öffentlichen Timeline und werden nicht an Follower ausgeliefert. Beide Typen können in Kampagnen verwendet werden.

Wenn der authentifizierte Nutzer nicht der `FULL`-promotable user dieses Kontos ist, prüfen Sie, ob er die Berechtigung hat, im Namen dieses Nutzers zu twittern, indem Sie eine Anfrage an den Endpoint [GET accounts/:account\_id/authenticated\_user\_access](https://developer.x.com/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access) stellen. Eine Berechtigung von `TWEET_COMPOSER` zeigt an, dass der Nutzer diesen Endpoint verwenden darf, um nullgecastete Tweets im Namen des `FULL` [promotable user](https://developer.x.com/en/docs/ads/campaign-management/api-reference/promotable-users#get-accounts-account-id-promotable-users) zu erstellen.

Wenn Sie den [upload.x.com Endpoint](https://developer.x.com/rest/reference/post/media/upload) für Medien verwenden, übergeben Sie für den Parameter `additional_owners` denselben `user_id`-Wert wie den `as_user_id`-Wert, den Sie an diesen Endpoint übergeben.
