---
title: X-IDs
icon: id-card-clip
---

Jedes Objekt innerhalb von X – ein Tweet, eine Direktnachricht, ein Nutzer, eine Liste und so weiter – hat eine eindeutige ID.

Ganz am Anfang der Plattform waren diese IDs klein genug, um sie fortlaufend zu generieren. Im Laufe der Zeit, um dem Wachstum gerecht zu werden, wechselten die IDs von 32-Bit zu 64-Bit. Heute sind X-IDs eindeutige 64-Bit-Unsigned-Integer, die auf der Zeit basieren, statt fortlaufend zu sein. Die vollständige ID setzt sich aus einem Zeitstempel, einer Workernummer und einer Sequenznummer zusammen. X entwickelte einen internen Dienst namens „Snowflake“, um diese IDs konsistent zu generieren ([mehr dazu im X‑Blog](https://blog.x.com/engineering/en_us/a/2010/announcing-snowflake.html)).

Zahlen mit einer Größe von bis zu 64 Bit können Probleme mit Programmiersprachen verursachen, die Integer mit weniger als 64 Bit darstellen. Ein Beispiel dafür ist JavaScript, wo Integer auf 53 Bit begrenzt sind. Um hierfür eine Lösung bereitzustellen, wurden in den ursprünglichen Designs der X-API (v1/1.1) ID‑Werte in zwei Formaten zurückgegeben: sowohl als Integer als auch als Strings.

```json
{"id": 10765432100123456789, "id_str": "10765432100123456789"}
```

Wenn Sie den Befehl `(10765432100123456789).toString()` in einer JavaScript-Konsole im Browser ausführen, lautet das Ergebnis „10765432100123458000“ – der 64‑Bit‑Integer verliert durch die Umwandlung an Genauigkeit (dies wird manchmal als „Munging“ bezeichnet – eine destruktive Änderung an einem Datenelement).

In X-APIs bis Version 1.1 sollten Sie stets die String-Darstellung der Zahl verwenden, um Genauigkeitsverluste zu vermeiden.

In neueren Versionen der API werden alle großen Integer‑Werte standardmäßig als Strings dargestellt.
