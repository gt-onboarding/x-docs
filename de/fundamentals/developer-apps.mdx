<div id="overview">
  ## Übersicht
</div>

Wenn Sie bereits Apps haben, können Sie diese über die [App-Seite im Developer-Portal](https://developer.x.com/en/portal/projects-and-apps) anzeigen, bearbeiten oder löschen.

Der Zugriff auf die X-API und die X-Ads-API erfordert einen Satz von [Authentifizierungs](/de/resources/fundamentals/authentication)-Anmeldedaten, auch bekannt als Keys und Tokens, die Sie jeder Anfrage beifügen müssen. Diese Anmeldedaten können je nach Art der Authentifizierung, die der jeweilige von Ihnen verwendete Endpoint erfordert, unterschiedliche Formen annehmen.

Hier sind die verschiedenen Anmeldedaten, die Sie in Ihrer App generieren können, und wie Sie sie verwenden:

- **[API Key und Secret](/de/resources/fundamentals/authentication#api-key-and-secret):** Im Wesentlichen der Benutzername und das Passwort Ihrer App. Sie verwenden diese zur Authentifizierung von Anfragen, die [OAuth 1.0a User Context](/de/resources/fundamentals/authentication#oauth-1-0a-2) erfordern, oder um andere Tokens wie User-Access-Tokens oder App-Access-Token zu generieren.

- **[Access Token und Secret](/de/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow):** Access-Tokens repräsentieren im Allgemeinen den Benutzer, in dessen Namen Sie die Anfrage stellen. Die Tokens, die Sie über das Developer-Portal generieren können, repräsentieren den Benutzer, dem die App gehört. Sie verwenden diese zur Authentifizierung von Anfragen, die [OAuth 1.0a User Context](/de/resources/fundamentals/authentication#oauth-1-0a-2) erfordern. Wenn Sie Anfragen im Namen eines anderen Benutzers stellen möchten, müssen Sie den 3-legged-OAuth-Flow verwenden, damit dieser Sie autorisieren kann.

- **[Client ID und Client Secret](/de/resources/fundamentals/authentication#oauth-2-0):** Diese Anmeldedaten werden verwendet, um mit OAuth 2.0 ein User-Access-Token zu erhalten. Ähnlich wie bei OAuth 1.0a werden User-Access-Tokens verwendet, um Anfragen zu authentifizieren, die private Benutzerkontoinformationen bereitstellen oder Aktionen im Namen eines anderen Kontos ausführen, jedoch mit fein abgestuften Scopes für eine bessere Kontrolle darüber, auf was die Client-Anwendung beim Benutzer zugreifen darf.

- **[App-only Access Token](/de/resources/fundamentals/authentication#bearer-token-also-known-as-app-only):** Sie verwenden dieses Token, wenn Sie Anfragen an Endpoints stellen, die mit öffentlich auf X verfügbaren Informationen antworten.

Zusätzlich zur Generierung der für X-API-Anfragen erforderlichen Keys und Tokens können Sie auch Zugriffs-[Berechtigungen](/de/resources/fundamentals/developer-apps#app-permissions) festlegen, den Use Case bzw. Zweck der App dokumentieren, eine [Callback-URL](/de/resources/fundamentals/developer-apps#callback-urls) definieren und weitere Einstellungen zu Ihrer App-Entwicklungsumgebung im [Management-Dashboard](https://developer.x.com/en/portal/projects-and-apps) vornehmen.

<div id="apps-and-projects">
  ### Apps und Projekte
</div>

Sie können Apps und [Projekte](/de/resources/fundamentals/projects) verwenden, um Ihre Arbeit mit der X Developer Platform nach Anwendungsfällen zu organisieren. Jedes Projekt kann im Free-X-API-Plan eine App enthalten, im Basic-Plan bis zu zwei Apps und im Pro-Plan bis zu drei Apps.

Wenn Sie auf die neuen [X-API v2](/de/x-api/introduction)-Endpoints zugreifen möchten, müssen Sie Schlüssel und Token aus einer App verwenden, die einem Projekt zugeordnet ist.

Wenn Sie Apps haben, die erstellt wurden, bevor wir Projekte eingeführt haben, sind diese im Abschnitt mit dem Titel „Standalone Apps“ sichtbar. Standalone-Apps sind Apps außerhalb der Projektstruktur. Wenn Sie eine Standalone-App mit einem Projekt verknüpfen, kann sie anschließend Anfragen an die v2-Endpoints stellen.

<div id="developer-portal-dashboard">
  ### Dashboard des Developer-Portals
</div>

Sie können [das Dashboard besuchen](https://developer.x.com/content/developer-twitter/en/apps), um die mit Ihrem Konto verknüpften Apps zu verwalten. Weitere Informationen finden Sie auf unserer Dokumentationsseite zum [Developer-Portal](/de/resources/fundamentals/developer-portal). Das Dashboard ermöglicht es Entwicklerinnen und Entwicklern, die folgenden Aufgaben schnell und einfach auszuführen:

- Anzeigen Ihrer vorhandenen Standalone-Apps und der zugehörigen App-ID.
- Erstellen eines neuen Projekts, einer App oder einer Standalone-App.
- Löschen eines nicht verwendeten Projekts oder einer App.
- Überprüfen oder Aktualisieren der Einstellungen einer bestimmten App, einschließlich Name, Beschreibung, Website, Callback-URL und [Berechtigungen](/de/resources/fundamentals/developer-apps#app-permissions).
- Neuerstellen appspezifischer Anmeldedaten wie API Key & Secret, App Access Token sowie der Access Tokens des App-Inhabers.

<div id="signing-up-for-access">
  ### Registrierung für den Zugriff
</div>

Wenn Sie bereits X-Apps haben, können Sie Ihre Apps über das X‑[App-Dashboard](https://developer.x.com/content/developer-twitter/en/portal/projects-and-apps) anzeigen und bearbeiten, sofern Sie auf developer.x.com in Ihrem X-Konto angemeldet sind. Bitte beachten Sie, dass Sie sich **nicht** für ein Konto registrieren müssen, um alle zuvor auf developer.x.com erstellten Apps zu verwalten.

Wenn Sie eine neue X-App erstellen möchten, benötigen Sie zuvor ein [Entwicklerkonto](https://developer.x.com/en/portal/petition/essential/basic-info). Wenn Sie Mitglied eines [Teamkontos](/de/resources/fundamentals/developer-portal#team-management) sind, müssen Sie die Admin-Rolle zugewiesen bekommen, um eine neue App zu erstellen.

<div id="automated-account-labeling-for-bots">
  ### Kennzeichnung „Automatisiertes Konto“ für Bots
</div>

Sie können Ihren Bot-Konten das Label „Automatisiertes Konto“ hinzufügen, damit Nutzer auf X erkennen, dass es sich um ein automatisiertes Konto handelt. Diese Bots führen programmierte Aktionen über die X-API aus. Wenn Sie Ihrem Bot das Label „Automatisiertes Konto“ hinzufügen, stärken Sie das Vertrauen Ihrer Zielgruppe, unterstreichen die Legitimität Ihres Kontos und grenzen sich von Spam-Bots ab. So verstehen Personen auf X den Zweck Ihres Kontos bei der Interaktion mit Ihrem Bot besser.

Gehen Sie wie folgt vor, um Ihrem Bot-Konto ein Label hinzuzufügen:

1. Öffnen Sie Ihre Kontoeinstellungen
2. Wählen Sie „Ihr Konto“
3. Wählen Sie „Automatisierung“
4. Wählen Sie „Kontoverwaltung“
5. Wählen Sie anschließend das X-Konto aus, das Ihr Bot-Konto betreibt
6. Geben Sie Ihr Passwort ein, um sich anzumelden
7. Abschließend sollten Sie eine Bestätigung sehen, dass das Label auf Ihr Konto angewendet wurde.

  ### Einführung
</div>

Das [X App-Dashboard](https://developer.x.com/en/portal/projects-and-apps) ermöglicht es Entwicklerinnen und Entwicklern, schnell und unkompliziert Folgendes zu erledigen:

- Ein Projekt, eine App oder eine eigenständige App löschen.
- Die Einstellungen einer bestimmten App öffnen, indem Sie die App auswählen. In den Einstellungen können Sie die App-Details, Schlüssel und Token sowie Berechtigungen einsehen.
- Die Benutzerauthentifizierung Ihrer App auf OAuth 1.0a oder OAuth 2.0 aktualisieren.

<Note>
  **Hinweis:**

  Alle App-Schlüssel und Token sind im Developer-Portal nicht mehr sichtbar und müssen nach der Generierung sicher gespeichert werden. Wir empfehlen, einen Passwort-Manager zum Speichern Ihrer Schlüssel und Token zu verwenden.

  Sie können einen Hinweis zum API-Schlüssel einblenden, um Ihre Zugangsdaten der entsprechenden App zuzuordnen.
</Note>

<div id="app-details">
  #### App-Details
</div>

Hier können Sie das App-Symbol, den App-Namen, die App-Beschreibung, Ihre Website-URL, [Callback-URLs/Redirect-URIs](/de/resources/fundamentals/developer-apps#callback-urls), die URL zu den Nutzungsbedingungen, die URL zur Datenschutzrichtlinie, den Organisationsnamen, die Organisations-URL sowie den Zweck bzw. Anwendungsfall der App bearbeiten.

OAuth 2.0 und OAuth 1.0a sind Authentifizierungsmethoden, die es Nutzerinnen und Nutzern ermöglichen, sich mit X bei Ihrer App anzumelden. Sie erlauben Ihrer App außerdem, bestimmte Anfragen im Namen authentifizierter Nutzerinnen und Nutzer zu stellen. Sie können eine der Methoden oder beide für Ihre App aktivieren.

Es ist wichtig, diese Informationen aktuell zu halten. Ihr App-Name und Ihre Website-URL werden als Quelle in den Metadaten für alle von Ihrer Anwendung programmgesteuert erstellten Tweets angezeigt. Wenn Sie den Anwendungsfall einer X-App ändern, aktualisieren Sie diesen in den Einstellungen, um sicherzustellen, dass Sie die [Developer Terms](https://developer.x.com/content/developer-twitter/en/developer-terms/agreement-and-policy) einhalten.

Wenn Ihre Anwendung ein Tag mit der Kennzeichnung „suspended“ aufweist, liegt dies daran, dass Ihre App gegen eine oder mehrere der [Developer Terms](https://developer.x.com/en/developer-terms.html) von X verstößt, etwa gegen unsere [Automation Rules](https://help.x.com/en/rules-and-policies/twitter-automation). Das Richtlinienteam der Developer-Plattform kommuniziert mit Entwicklerinnen und Entwicklern über die E-Mail-Adresse, die im X-Konto der App-Inhaberin bzw. des App-Inhabers hinterlegt ist. Um diese E-Mail-Adresse zu prüfen, sehen Sie bitte in Ihren [X-Kontoeinstellungen](https://x.com/settings/your_twitter_data/account) nach. Benachrichtigungs-E-Mails über Sperrungen werden an diese E-Mail-Adresse mit einem Betreff ähnlich „Application suspension notice“ gesendet und enthalten konkrete Hinweise zu den nächsten Schritten. Um mit dem X-Team an der Behebung von Sperren zu arbeiten, prüfen Sie bitte Ihre E-Mails auf spezifische Anweisungen oder nutzen Sie unser [Platform Help Form](https://help.x.com/forms/platform).

<div id="keys-and-tokens">
  #### Schlüssel und Tokens
</div>

Innerhalb einer App in einem Projekt oder über eine eigenständige App können Sie die folgenden Tokens anzeigen, neu generieren oder widerrufen:

- [API Key (Consumer Key) und API Secret (Consumer Secret)](/de/resources/fundamentals/authentication#api-key-and-secret)
- [App Access Token](/de/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)
- [User Access Token und Access Token Secret](/de/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) -
  Das im Developer-Portal verfügbare Access Token und Secret beziehen sich auf den Benutzer, dem die App gehört.

**Bitte beachten** – Wenn Sie Anfragen im Namen eines anderen Benutzers stellen möchten (also nicht des Benutzers, dem die App gehört), müssen Sie entweder den [OAuth 1.0a 3-legged OAuth Flow](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens) oder den OAuth 2.0 [Authorization Code mit PKCE Flow](/de/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) verwenden, um einen Satz von Benutzer-Access-Tokens zu generieren. Diese benutzerspezifischen Tokens verwenden Sie dann in Ihrer Anfrage an die API.

<div id="user-authentication-settings">
  #### Einstellungen zur Benutzerauthentifizierung
</div>

Sie können die Authentifizierungseinstellungen Ihrer App auf OAuth 1.0a oder OAuth 2.0 festlegen. OAuth 2.0 kann nur mit der X-API v2 verwendet werden. Mit OAuth 2.0 können Sie gezielte, fein abgestufte Scopes auswählen, die Ihnen im Namen eines Nutzers bestimmte Berechtigungen einräumen. OAuth 1.0a kann mit der X-API v1.1 und v2 verwendet werden und nutzt eine breite Autorisierung mit grobgranularen Scopes.

<div id="oauth-10a-application-user-permissions">
  #### OAuth 1.0a Berechtigungen für App und Nutzer
</div>

Wenn Sie der Eigentümer der App sind, können Sie die Berechtigungen der App anpassen (Nur-Lesen; Lesen und Schreiben; oder Lesen, Schreiben und Direktnachrichten). Dadurch wird gesteuert, auf welche Ressourcen und Ereignisse Sie über die X APIs zugreifen können (Hinweis: Einige Ressourcen erfordern zusätzliche Berechtigungen direkt von X).

Sie können auf dieser Seite außerdem die Fähigkeit Ihrer Apps ein- oder auszuschalten, die E-Mail-Adressen von Nutzern anzufordern (hierfür müssen auf der Seite „App details“ eine URL zu den Nutzungsbedingungen und eine URL zur Datenschutzrichtlinie hinterlegt sein).

<div id="oauth-20-type-of-app">
  #### OAuth 2.0 App-Typ
</div>

Wenn Sie OAuth 2.0 als Benutzerauthentifizierungsmethode auswählen, müssen Sie den Typ der App festlegen, die Sie erstellen. Zur Auswahl stehen Native App, Single-Page-App, Web App, Automated App oder Bot. Native App und Single-Page-Apps sind öffentliche Clients, während Web App sowie Automated App oder Bots vertrauliche Clients sind.

Vertrauliche Clients authentifizieren sich sicher beim Autorisierungsserver und schützen Ihr Client-Secret. Öffentliche Clients sind Anwendungen, die in der Regel in einem Browser oder auf einem mobilen Gerät laufen und keine Client-Secrets verwenden können. Wenn Sie einen App-Typ wählen, der ein vertraulicher Client ist, erhalten Sie ein Client-Secret.

<div id="app-permissions">
  ## App-Berechtigungen
</div>

<div id="oauth-10a-app-permissions">
  ### OAuth 1.0a App-Berechtigungen
</div>

App-Berechtigungen beschreiben die Zugriffsebene für die OAuth 1.0a-Authentifizierung zwischen Anwendung und Nutzer. App-Berechtigungen werden pro Anwendung in den Einstellungen Ihrer [X App](/de/resources/fundamentals/developer-apps) konfiguriert.

Es gibt drei Berechtigungsstufen:

1. Nur lesen
2. Lesen und schreiben
3. Lesen, schreiben und Zugriff auf Direktnachrichten

Es gibt eine zusätzliche Berechtigung, um die E-Mail-Adresse eines Nutzers einsehen zu dürfen – diese kann mit jeder der oben aufgeführten drei Stufen kombiniert werden.

Wenn eine Berechtigungsstufe geändert wird, müssen alle bereits für diese X‑App ausgestellten Nutzertokens verworfen werden, und die Nutzer müssen die App erneut autorisieren, damit das Token die aktualisierten Berechtigungen übernimmt.

Eine gute Praxis ist es, nur das minimale erforderliche Zugriffslevel auf die Kontodaten eines Nutzers anzufordern, das eine Anwendung oder ein Dienst benötigt.

<div id="read-only">
  ### Nur Lesen
</div>

Diese Berechtigungsstufe erlaubt Lesezugriff auf X-Ressourcen, einschließlich (beispielsweise) der Tweets eines Nutzers, der Startseiten-Timeline und der Profilinformationen. Sie erlaubt keinen Zugriff auf die Direct Messages eines Nutzers und erlaubt nicht, irgendwelche Elemente oder Objekte zu aktualisieren.

<div id="read-and-write">
  ### Lesen und Schreiben
</div>

Diese Berechtigungsstufe erlaubt Lese- und Schreibzugriff auf X-Ressourcen. Zusätzlich zum Lesezugriff ermöglicht sie das Veröffentlichen von Tweets, das Folgen von Nutzern sowie das Aktualisieren von Teilen der Profilinformationen eines Nutzers. Sie ermöglicht außerdem, im Namen des authentifizierten Nutzers Antworten auszublenden. Diese Berechtigungsstufe gewährt keinen Zugriff auf Direct Messages (einschließlich Lesen, Schreiben oder Löschen).

<div id="read-write-and-access-direct-messages">
  ### Direktnachrichten lesen, schreiben und darauf zugreifen
</div>

Diese Berechtigungsstufe umfasst den Zugriff auf alle oben genannten Punkte und erweitert ihn um die Möglichkeit, im Namen eines Nutzers Direktnachrichten zu lesen, zu schreiben und zu löschen.

- [POST /2/dm\_conversations/:dm\_conversation\_id/messages](/de/x-api/direct-messages/send-a-new-message-to-a-dm-conversation)
- [POST /2/dm\_conversations/](/de/x-api/direct-messages/create-a-new-dm-conversation)
- [POST /2/dm\_conversations/with/:participant\_id/messages](/de/x-api/direct-messages/send-a-new-message-to-a-user)
- [GET /2/dm\_conversations/with/:user\_id/dm\_events](/de/x-api/direct-messages/get-dm-events-for-a-dm-conversation)
- [GET /2/dm\_conversations/:dm\_conversation\_id/dm\_events](/de/x-api/direct-messages/get-dm-events-for-a-dm-conversation-1)
- [GET /2/dm\_events](/de/x-api/direct-messages/get-recent-dm-events)

<div id="determining-permissions">
  ### Berechtigungen ermitteln
</div>

Alle authentifizierten API-Anfragen enthalten in der HTTP-Antwort einen `x-access-level`-Header. Der Wert dieses Headers gibt die aktuell verwendete Berechtigungsstufe an. Mögliche Werte sind read, read-write und read-write-directmessages.

<div id="callback-urls">
  ## Callback-URLs
</div>

Die Authentifizierungsmethoden OAuth 1.0a User Context und OAuth 2.0 Authorization Code mit PKCE ermöglichen es Entwicklerinnen und Entwicklern, im Namen verschiedener X-Nutzerinnen und -Nutzer Anfragen zu stellen, die einen bestimmten Anmeldeablauf durchlaufen haben. Derzeit gibt es zwei Abläufe, mit denen Sie Nutzerinnen und Nutzern die Autorisierung Ihrer Anwendung ermöglichen können:

- [OAuth 2.0 Authorization Code Flow mit PKCE](/de/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2)
- [OAuth 1.0a 3-legged OAuth Flow](/de/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) (und separat der [Sign in with X Flow](/de/resources/fundamentals/authentication#log-in-with-x))

Wenn Nutzerinnen und Nutzer diese Abläufe durchlaufen, benötigen sie eine Webseite bzw. einen Zielort, zu dem sie nach erfolgreicher Anmeldung und erteilter Autorisierung für die App der Entwicklerinnen und Entwickler weitergeleitet werden. Diese nachgelagerte Webseite bzw. dieser Ort wird als Callback-URL bezeichnet.

Beim Einrichten dieser Abläufe müssen Entwicklerinnen und Entwickler eine Callback-URL in ihren Anfragen an die Authentifizierungsendpunkte übergeben, aus denen die oben genannten Abläufe bestehen. Beispielsweise müssen Entwicklerinnen und Entwickler, die OAuth 1.0a User Context verwenden, den Parameter `callback_url` übergeben, wenn sie eine Anfrage an den Endpunkt [GET oauth/request\_token](/de/resources/fundamentals/authentication#post-oauth-request-token) stellen. Ebenso müssen Entwicklerinnen und Entwickler, die OAuth 2.0 Authorization Code mit PKCE verwenden, den Parameter `redirect_uri` mit ihrer Anfrage an den Endpunkt GET oauth2/authorize übergeben.

Zusätzlich zur Verwendung dieser Parameter müssen Entwicklerinnen und Entwickler sicherstellen, dass die Callback-URL auch in die Allowlist der [App](/de/resources/fundamentals/developer-apps) für Callback-URLs aufgenommen wurde, die auf der Einstellungsseite der App im [Developer-Portal](/de/resources/fundamentals/developer-portal) zu finden ist.

Wenn alles korrekt eingerichtet ist, werden Entwicklerinnen und Entwickler nach der erfolgreichen Anmeldung bei X im Rahmen dieser Abläufe zur Callback-URL weitergeleitet.

<div id="things-to-keep-in-mind">
  ### Dinge, die Sie beachten sollten
</div>

Wenn Sie Ihre Callback-URLs einrichten, gibt es ein paar Dinge, die Sie beachten sollten:

**Query-Parameter URL-codieren**

Da Sie die Callback-URL als Query-Parameter mit `callback_url` oder `redirect_uri` übergeben, stellen Sie bitte sicher, dass Sie die URL URL-codieren.

**Grenzwerte für Callback-URLs**

Im X-Apps-Dashboard gibt es eine feste Obergrenze von 10 Callback-URLs. Ihre Callback-URL muss immer exakt mit der auf der Allowlist stehenden Callback-URL übereinstimmen, die Sie dem Apps-Dashboard hinzufügen, sowie mit dem Parameter, den Sie im Autorisierungsablauf angeben.

Wenn Sie anforderungsspezifische Daten in die Callback-URL aufnehmen möchten, können Sie den Parameter `state` verwenden, um Daten zu speichern, die nach der Weiterleitung des Nutzers enthalten sind. Entweder werden die Daten direkt im Parameter `state` codiert, oder Sie verwenden den Parameter als Sitzungs-ID, um den Zustand auf dem Server zu speichern.

**Verwenden Sie localhost nicht als Callback-URL**
Verwenden Sie statt localhost bitte lokal einen benutzerdefinierten Host oder http(s)://127.0.0.1

**Benutzerdefinierte Protokoll-URLs**
Wenn Sie Mobile Deep Linking nutzen möchten, können Sie benutzerdefinierte Protokoll-URLs mit einem Pfad- und Domain-Teil verwenden, beispielsweise twitter://callback/path. Es gibt jedoch eine Liste nicht zulässiger Protokolle, die Sie vermeiden müssen. Sie können die Liste der nicht zulässigen Protokolle unten einsehen.

<div id="error-example">
  ### Fehlerbeispiel
</div>

Wenn Sie eine Callback-URL verwenden, die nicht ordnungsgemäß in den Einstellungen Ihrer App im Developer-Portal hinzugefügt wurde, erhalten Sie die folgende Fehlermeldung:

**OAuth 1.0a**

```json HTTP 403 - Forbidden

{
  "errors": [
    {
      "code":415,"message":"Callback URL not approved for this client application. Approved callback URLs can be adjusted in your application settings."
    }
  ]
}
```

ODER

```xml
<?xml version="1.0" encoding="UTF-8"?>
<hash>
<error>Callback URL not approved for this client application. Approved callback URLs can be adjusted in your application settings</error>
<request>/oauth/request_token</request>
</hash>
```

**OAuth 2.0**

```json HTTP 400

{
  "error": "invalid_request",
  "error_description": "Value passed for the redirect uri did not match the uri of the authorization code."
}
```

**Fehlerbehebung**

Wenn ein Fehler auftritt, stellen Sie sicher, dass die von Ihnen in Authentifizierungsanfragen verwendete Callback-URL URL-codiert ist und exakt mit einer Callback-URL übereinstimmt, die der Allowlist der App hinzugefügt wurde, deren Schlüssel und Token Sie in Ihrer Anfrage verwenden.
