  ## 개요

<div id="introduction">
  ### 소개
</div>

A/B 테스트는 광고주가 X에서 도달하는 사용자를 세분화하여 캠페인 성과 최적화에 가장 효과적인 방법을 파악하고, 마케팅 전략 수립에 활용할 인사이트를 얻도록 돕습니다.

이러한 세그먼트(사용자 그룹 분할)는 무작위로 배정되며 상호 배타적입니다. 무작위화로 결과에 영향을 미치는 요인이 균등하게 분포됩니다. 즉, 그룹 간 혹은 기대되는 행동 간에 본질적인 차이가 없습니다. 따라서 특정 사용자 그룹에만 단일 변형을 적용하고 다른 그룹에는 적용하지 않았다면, 캠페인 성과 차이는 해당 변형에 기인한다고 볼 수 있습니다.

여러 변형을 동시에 테스트할 수도 있지만, 한 번에 하나의 변형만 테스트할 것을 강력히 권장합니다. 이렇게 해야 관찰된 캠페인 성과 차이의 인과 요인을 명확히 분리할 수 있습니다.

변형은 캠페인 수준에서 설정됩니다. 예를 들어 광고주가 새로운 크리에이티브의 효력을 테스트하려면, 크리에이티브만 다른 두 개의 동일한 캠페인을 만들어야 합니다. 향후에는 라인 아이템 수준의 변형도 지원할 예정입니다.

<div id="use-cases">
  ### 사용 사례
</div>

A/B 테스트는 주로 (1) X에서 어떤 방법이 가장 효과적인지 파악하여 투자를 최적화하려는 퍼포먼스 광고주를 위한 최적화 활용 사례와 (2) 학습 결과를 마케팅 전략 수립에 반영하려는 브랜드 광고주를 위한 학습 활용 사례를 지원하는 데 사용됩니다. 

이 API는 다음을 포함한 모든 캠페인 변수에 대해 A/B 테스트를 지원합니다:

- 크리에이티브 
- 타게팅 
- 입찰 유형
- 입찰 단위

<div id="ab-testing">
  ## A/B 테스트
</div>

A/B 테스트는 광고주가 X에서 도달하는 사용자를 세분화해 캠페인 성과를 최적으로 향상시키는 방법을 파악하고, 마케팅 전략 수립에 필요한 인사이트를 얻을 수 있도록 합니다.

이러한 세그먼트(사용자 그룹 분할)는 무작위로 배정되며 상호 배타적입니다. 무작위화를 통해 결과에 영향을 주는 요인이 각 그룹에 균등하게 분포합니다. 즉, 그룹 간이나 그들의 예상 행동에는 본질적인 차이가 없습니다. 따라서 특정 사용자 그룹에만 하나의 단일 변형을 적용하고 다른 그룹에는 적용하지 않았을 때 나타나는 캠페인 성과 차이는 해당 변형에 기인한다고 볼 수 있습니다.

한 번에 여러 변형을 테스트할 수도 있지만, 한 번에 하나의 변형만 테스트할 것을 강력히 권장합니다. 이렇게 하면 관찰된 캠페인 성과 차이의 인과 요인을 분리할 수 있습니다.

변형은 캠페인 수준 또는 광고 그룹 수준에서 설정합니다. 광고 그룹은 Ads API의 [line item](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/line-items)으로 설정합니다. 광고 그룹 수준 변형의 예로, 광고주가 새로운 크리에이티브의 효과를 테스트하려는 경우, 크리에이티브만 다른 동일한 광고 그룹 2개를 포함한 하나의 캠페인을 생성하면 됩니다.

<div id="use-cases">
  ### 활용 사례
</div>

A/B 테스트는 주로 (1) 성과 고객이 X에서 가장 효과적인 요소를 파악해 투자 효율을 극대화하려는 최적화 용도와 (2) 브랜드 광고주가 얻은 인사이트를 마케팅 전략에 반영하려는 학습 용도를 지원하는 데 사용됩니다.

API는 다음을 포함한 모든 캠페인 변수에 대해 A/B 테스트를 지원합니다:

- 크리에이티브

- 타기팅

- 입찰 유형

- 입찰 단위

<div id="attributes">
  ### 속성
</div>

A/B 테스트는 중첩 구조로 표현됩니다. 최상위에는 A/B 테스트 자체에 대한 필드가 있고, 각 사용자 그룹을 설명하는 필드 집합을 가진 사용자 그룹 객체 배열이 있습니다.

개략적으로, 모든 A/B 테스트에는 다음 정보가 포함되어야 합니다.

- 테스트 기간: start\_time 및 end\_time 필드로 표현됨

- 분할이 이루어지는 수준: entity\_type 필드로 표현됨

- 최소 2개(최대 30개)의 사용자 그룹: 각 그룹은 user\_groups 배열의 객체로 표현됨

각 사용자 그룹에는 다음 정보가 반드시 포함되어야 합니다.

- 해당 사용자 그룹에 할당할 사용자 비율: size 필드로 표현됨

- 해당 사용자 그룹의 사용자 풀을 구성할 캠페인 ID/라인 아이템 ID: entity\_ids 배열로 표현됨

선택적으로 A/B 테스트와 사용자 그룹에 대해 name 및 description 값을 설정할 수 있습니다. 유효성 검사 규칙 및 기타 제약 조건에 대한 정보는 아래에서 확인할 수 있습니다.

ID나 생성 시각과 같은 기타 메타데이터도 포함되지만, 이는 X에서 자동으로 설정됩니다.

  ### 사용

아래 하위 섹션에서는 A/B 테스트를 생성하고 업데이트하는 방법을 설명합니다. 읽기 및 삭제는 다른 모든 X Ads API 엔드포인트와 동일하게 작동합니다.

<div id="creating">
  #### 생성
</div>

[POST accounts/:account\_id/ab\_tests](https://developer.x.com/en/docs/x-ads-api/measurement/api-reference/ab-tests) 엔드포인트를 사용해 A/B 테스트를 생성합니다. 이 엔드포인트는 JSON POST 본문만 허용합니다. Content-Type은 application/json으로 설정해야 합니다.

광고주가 두 개 이상의 캠페인을 설정한 후 A/B 테스트를 생성할 수 있습니다. 위에서 언급했듯이 A/B 테스트에는 반드시 테스트 기간, 분할 수준, 그리고 최소 두 개의 사용자 그룹이 포함되어야 합니다. 각 사용자 그룹은 자신에게 할당할 사용자 비율과 해당 그룹의 사용자 풀을 구성할 캠페인 ID를 지정해야 합니다. 아래에서 각 항목을 자세히 설명합니다.

테스트 기간:

- start\_time과 end\_time 값은 다음을 충족해야 합니다
  - (A/B 테스트 생성 시점을 기준으로) 미래여야 합니다
  - 캠페인/라인 아이템의 집행 기간과 겹쳐야 합니다
- 테스트는 앱 기반이 아닌 캠페인의 경우 최소 1일, 앱 기반 캠페인의 경우 최소 5일 이상 지속되어야 합니다

분할 수준:

- entity\_type은 CAMPAIGN 또는 LINE\_ITEM으로 설정할 수 있습니다

사용자 그룹:

- 각 사용자 그룹은 user\_groups 배열의 객체로 표현됩니다
  - 최소 두 개의 사용자 그룹이 필요합니다
  - 최대 30개의 사용자 그룹까지 허용됩니다
- 각 사용자 그룹의 크기는 1.00과 99.00 사이의 숫자 값을 문자열로 설정합니다
  - 참고: 각 객체의 size 값 합계는 반드시 100.00이 되어야 합니다
- 각 사용자 그룹의 entity\_ids 배열에 캠페인 ID를 지정해야 합니다

선택적으로 A/B 테스트 또는 하나 이상의 사용자 그룹에 이름과 설명을 설정할 수 있습니다.

다음 요청은 캠페인 수준에서 4일 동안 지속되며, 각 그룹에 50%의 사용자가 배정된 두 개의 사용자 그룹을 갖는 A/B 테스트를 생성합니다. 첫 번째 사용자 그룹은 캠페인 f2qcw 및 f2tht를 기반으로 하고, 두 번째 사용자 그룹은 캠페인 f2rqi 및 f2tws를 기반으로 합니다. 이 요청은 또한 엔터티의 일부에 이름과 설명을 추가합니다.

**라인 아이템 수준 A/B 테스트**

캠페인 수준과 라인 아이템 수준 A/B 테스트의 주요 차이는 entity\_type입니다. 라인 아이템 수준 A/B 테스트의 경우 ‘entity\_type’ 값을 ‘LINE\_ITEM’으로 설정해야 합니다. 아래에 나오는, 이미 생성된 A/B 테스트에 대한 모든 작업에도 동일하게 적용됩니다.

요구 사항:

1. A/B 테스트 캠페인의 모든 라인 아이템이 분할 테스트에 포함되어야 합니다.
2. 라인 아이템 수준에서는 균등 분할만 허용됩니다.
3. 하나의 분할 테스트에서 허용되는 사용자 그룹 라인 아이템 수는 5개 이하여야 합니다.
4. 사용자 그룹당 라인 아이템은 1개만 허용됩니다.

<div id="updating">
  ### 업데이트
</div>

[PUT accounts/:account\_id/ab\_tests/:ab\_test\_id](https://developer.x.com/en/docs/x-ads-api/measurement/api-reference/ab-tests) 엔드포인트를 사용해 A/B 테스트를 업데이트합니다. 이 엔드포인트는 요청 본문에 JSON 블롭을 포함해야 하며, Content-Type은 application/json으로 설정해야 합니다.

다른 업데이트 엔드포인트와 마찬가지로 [PUT accounts/:account\_id/ab\_tests/:ab\_test\_id](/ko/x-ads-api/measurement/ab-testing#get-accounts-account-id-ab-tests) 엔드포인트는 URL에 A/B 테스트 ID를 포함해야 합니다. 일반적으로 A/B 테스트는 상태가 SCHEDULED일 때만 업데이트할 수 있습니다. 단, 예외적으로 LIVE 상태일 때 end\_time은 업데이트할 수 있습니다.

이 엔드포인트는 객체 ID가 포함된 부분 JSON을 지원합니다. 다음 원칙이 적용됩니다.

- 객체나 요소를 추가·제거하려면 전체 배열(및 그 하위 구조)을 전달하세요. 이는 교체 작업입니다.

- 그 외에는 키 이름이나 ID를 참조해 기존 필드를 변경(추가·수정·삭제)하세요.
  - 필드를 제거하려면 값을 null로 설정하세요.
  - 전달되지 않은 필드는 변경되지 않습니다.

예를 들어, 이전에 생성한 A/B 테스트에 세 번째 사용자 그룹을 추가하려면 기존 두 사용자 그룹 객체와 새로 추가할 객체를 포함해 user\_groups 배열을 전송해야 합니다. 이는 user\_groups 배열을 재생성하는 것으로 보면 됩니다. 처음부터 이렇게 생성하듯 데이터를 전달하세요(사용자 그룹 객체 ID는 전달하지 마세요). 업데이트 요청의 user\_groups 배열은 다음과 같이 표현할 수 있습니다.

```json
[
  {
    "entity_ids": [
      "f2qcw",
      "f2tht"
    ],
    "size": "30.0",
    "name": "first group"
  },
  {
    "entity_ids": [
      "f2rqi",
      "f2tws"
    ],
    "size": "30.0",
    "name": "second group",
    "description": "second AB test group"
  },
  {
    "entity_ids": [
      "i1vwr",
      "i1xre"
    ],
    "size": "40.0"
  }
]
```

객체 전반의 size 값 합은 여전히 100.00이어야 합니다. 첫 두 객체의 값을 업데이트하지 않았다면(이전에는 각각 50.00으로 설정됨) 요청은 실패했을 것입니다.

반대로 첫 번째 사용자 그룹에 설명만 추가하려는 경우, 업데이트 요청의 user\_groups 배열은 다음과 같이 표현됩니다.

```json
[
  {
    "id": "p1bcx",
    "description": "updated using a PUT request"
  }
]
```

ID로 사용자 그룹 객체를 참조하고, 수정하려는 필드만 포함합니다.
