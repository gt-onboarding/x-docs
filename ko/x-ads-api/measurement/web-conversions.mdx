### 사전 준비사항 

<div id="ads-api-access-new-applications">
  #### **Ads API 액세스 - 신규 애플리케이션**
</div>

- Conversion API의 기본 요건은 [개발자 계정](https://developer.x.com/en/portal/dashboard)과 [Ads API 액세스](https://developer.x.com/en/docs/twitter-ads-api/getting-started) 보유입니다. 절차는 [Ads API 시작하기](https://developer.x.com/en/docs/twitter-ads-api/getting-started) 가이드에 설명되어 있습니다. 다음 사항을 참고하세요:

**1단계: 개발자 계정**

- [개발자 계정](https://developer.x.com/en/portal/dashboard)을 신청할 때 즉시 승인을 받으려면 구독 [플랜](https://developer.x.com/en/portal/products/) 중 하나를 신청하세요. 
- **참고**: 모범 사례로, 공식 회사 X 핸들로 개발자 계정을 생성하고 Ads API 액세스를 신청할 것을 강력히 권장합니다. 개발자 계정이 개인 개발자 핸들과 연결된 경우, 필요 시 해당 자격 증명을 이전할 방법이 없습니다. 지속적인 관리를 위해 회사 계정에서 운영하고 필요할 때 [Multi-user login](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html)을 활용하는 것이 가장 좋습니다. 그렇지 않다면 최소한 계정을 기본값이 아닌 설정(헤더 이미지, 아바타, 바이오 설명, 바이오 URL)으로 구성하고 2단계 인증을 사용해야 합니다.

**2단계: Ads API 애플리케이션**

- Ads API 애플리케이션에 사용할 올바른 **App ID**를 준비하세요. App ID는 [Developer Portal](https://developer.x.com/en/portal/dashboard)의 프로젝트와 앱(Projects & Apps)에서 확인할 수 있습니다. 예: 16489123
- X 담당자에게 연락하여 Ads API 액세스를 요청하세요.

<div id="ads-api-access-existing-applications">
  #### **Ads API 액세스 - 기존 애플리케이션**
</div>

- 이미 적극적으로 사용 중인 Ads API 애플리케이션이 있다면, 해당 애플리케이션과 기존 액세스 토큰을 모두 Conversion API에 사용할 수 있습니다.

<div id="access-tokens">
  #### **액세스 토큰**
</div>

- Ads API 애플리케이션을 보유한 사용자 핸들의 사용자 액세스 토큰은 개발자 포털에서 직접 생성하고 조회할 수 있습니다. 이는 자신의 X 핸들에 사용하도록 되어 있으므로 “개인 액세스 토큰”이라고 합니다. 인증 및 개발자 포털에 대한 전반적인 정보는 [여기](/ko/resources/fundamentals/authentication/overview)에서 확인할 수 있습니다.
- Ads API 애플리케이션을 보유한 핸들이 아닌 다른 사용자 핸들의 사용자 액세스 토큰은 3-legged OAuth 플로우로 생성해야 합니다. 3-legged OAuth를 사용해 액세스 토큰을 생성하는 방법은 다음과 같습니다:
  - [twurl 유틸리티를 사용한 웹 기반 인증의 명령줄 방식](https://github.com/twitter/twurl#getting-started)
  - [PIN 기반 인증의 명령줄 방식](https://developer.x.com/resources/fundamentals/authentication/pin-based-oauth)
  - [3-legged OAuth 패턴을 구현한 커스텀 웹 플로우](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)
- Conversion API와 함께 사용하는 모든 사용자 토큰은 AD\_MANAGER 또는 ACCOUNT\_ADMIN 액세스 레벨을 가진 사용자여야 하며, 이는 [authenticated\_user\_access](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/authenticated-user-access) 엔드포인트로 확인할 수 있습니다.
- **참고:** 위와 같이 생성된 토큰 자체는 사용을 위해 AD\_MANAGER 또는 ACCOUNT\_ADMIN 액세스 레벨이 없는 사용자에게도 공유할 수 있습니다.

<div id="creating-the-conversion-api-event">
  ### Conversion API 이벤트 생성
</div>

Conversion API를 사용하려면 Ads Manager에서 새 전환 이벤트를 생성하거나, 이미 생성되어 X Pixel과 함께 사용 중인 기존 이벤트를 사용할 수 있습니다. 픽셀 이벤트와 Conversion API 이벤트 간 중복 제거를 하려면 Pixel용으로 생성해 둔 기존 이벤트를 사용해야 합니다. 

<div id="option-1-using-an-existing-conversion-event-in-ads-manager">
  ##### 옵션 1: Ads Manager에서 기존 전환 이벤트 사용
</div>

이미 X 픽셀과 함께 사용 중인 기존 이벤트를 계속 사용할 수 있습니다. 이 경우 해당 이벤트의 Event ID를 가져와야 합니다. 동일한 이벤트에 픽셀과 Conversion API를 모두 사용하는 경우, 동일 이벤트의 픽셀과 Conversion API 간 중복을 방지하기 위해 Pixel 코드 스니펫과 Conversion API 요청 모두에 중복 제거 키를 사용하세요(Conversion API에서는 `conversion_id`). 자세한 내용은 d. 이벤트 테스트 및 중복 제거 섹션을 참조하세요. 

<div id="option-2-creating-a-new-conversion-event-in-ads-manager">
  ##### 옵션 2: Ads Manager에서 새 전환 이벤트 생성:
</div>

이벤트를 생성하기 전에 Events Manager에서 이벤트 소스(Event Source)를 먼저 만들어 두는 것이 중요합니다. 계정에 이벤트 소스(X Pixel)가 추가되어 있는지 확인하려면 Events Manager로 이동하여 왼쪽 메뉴에 X Pixel이 표시되는지 확인하세요.

아직 이벤트 소스를 추가하지 않았다면 아래 단계에 따라 이벤트 소스를 생성하세요.

1. ads.x.com으로 이동
2. 왼쪽 상단의 **Tools** 섹션으로 이동하여 **Events Manager** 클릭
3. 왼쪽 사이드바에 X Pixel 이벤트 소스가 없다면, 오른쪽 상단의 Add event source를 선택하여 **Add an event source** 실행
   1. X Pixel 이벤트 소스의 ID가 **Pixel ID**입니다

이제 이벤트 소스와 Pixel ID가 있습니다. 추적하려는 전환 이벤트를 위해 이벤트 소스 내에 이벤트를 생성해야 합니다:

1. X Pixel 이벤트 소스에서 오른쪽의 **Add events** 선택
2. **Conversion API**로 설치 선택
3. API에서 사용될 이 이벤트의 **Pixel ID**와 **Event ID**가 표시됩니다
   1. 이 이벤트의 ID가 Event ID입니다
4. Save를 클릭하면 전환 이벤트가 생성되어 바로 사용할 수 있습니다

### 전환 이벤트용 식별자 준비 

현재 Click ID(`twclid`), 이메일 주소, 전화번호 등 식별자 중 최소 하나를 전달해야 합니다. IP 주소 또는 사용자 에이전트를 사용하는 경우 올바른 전환 매칭을 위해 두 번째 식별자를 함께 전송해야 합니다.

더 많은 식별자를 제공할수록 전환 매칭률이 높아집니다.

|     |     |     |
| :--- | :--- | :--- |
| 고객 매칭 필드 | 형식 | 해시 필요 여부 |
| X Click ID | X에서 생성됨 ([자세히 알아보기](https://business.x.com/en/help/campaign-measurement-and-analytics/conversion-tracking-for-websites.html#faq)) | 아니요  |
| 이메일 주소 | 앞뒤 공백 제거 | 필요 (SHA256) |
| 전화번호 | [E.164 표준](https://en.wikipedia.org/wiki/E.164) | 필요 (SHA256) |

|     |     |     |
| :--- | :--- | :--- |
| IP 주소 | 앞뒤 공백 제거 | 아니요  |
| 사용자 에이전트 | 앞뒤 공백 제거 | 아니요  |

#### 1. X Click ID 식별자 준비 

전환 요청에는 항상 Click ID를 포함할 것을 권장합니다.

사용자가 대상 웹사이트로 이동한 후 사용할 수 있게 되면, 쿼리 문자열 매개변수 twclid에서 Click ID를 파싱해야 합니다. 

기본 JavaScript 코드 예시:

```js
var queryString = document.location.search;
if (queryString.has('twclid') {
  twitterClickID = getParam(queryString, 'twclid');
  // 권장되는 다음 단계: 로깅, 로컬 스토리지에 저장
}
```

다음과 같이 할 것을 권장합니다: 

1. URL 쿼리 매개변수에 twclid 값이 존재할 때마다 항상 파싱합니다.

2. 관련 양식 필드 또는 전환 이벤트 정보와 함께 데이터를 저장합니다.

Click ID를 전환 이벤트 및 워크플로 정보와 연계하면 배치 처리, 여러 웹사이트 내비게이션 흐름을 기반으로 전환 이벤트를 분석·생성하는 알고리즘, 대량 업로드 등의 시나리오를 구현할 수 있습니다.

Event Source URL은 [URL 인코딩](https://en.wikipedia.org/wiki/Percent-encoding)되어야 하며, 이벤트를 트리거한 웹페이지를 나타냅니다.

#### 2. 이메일 식별자 준비 

지원되는 고객 매칭 필드는 전송할 수 있으나, 표준화하고 필요 시 개인정보 보호를 위해 해시 처리해야 합니다.

해당 정보는 솔트 없이 SHA256으로 해시해야 합니다. 

예를 들어, 이메일 주소 test@x.com은 다음과 같은 해시 형식으로 전송되어야 합니다: d360d510a224510f373931ce2d6215a799f5a9c1cef221b0149b6b6b50cced62.

#### 3. 전화 번호 식별자 준비 

전화 번호는 [E164 표준](https://en.wikipedia.org/wiki/E.164)에 따라 전달해야 하며, 솔트를 사용하지 않고 SHA-256으로 해시해야 합니다. 

예를 들어 미국 전화 번호 +11234567890의 경우, 다음과 같이 해시된 값으로 전송해야 합니다: 1fa6b8d986d9b9cd01bf36951815158bbde9f520c0567c835dfe34783d0a4231.

<div id="4-prepare-ip-address-identifier">
  #### 4. IP 주소 식별자 준비
</div>

IP 주소는 다른 식별자(twclid, 이메일 주소, 전화번호 또는 사용자 에이전트)와 함께 전달해야 합니다. 이 식별자에는 해싱이 필요하지 않습니다.

이 값은 마침표로 구분된 네 개의 숫자로 이루어진 점-십진 표기법으로 표기합니다. 예를 들어, 사용자의 IP 주소는 8.25.197.25일 수 있습니다.

<div id="5-prepare-user-agent-identifier">
  #### 5. 사용자 에이전트 식별자 준비
</div>

사용자 에이전트는 다른 식별자(twclid, 이메일 주소, 전화번호 또는 IP 주소)와 함께 전달해야 합니다. 이 식별자에는 해싱이 필요하지 않습니다.

이 식별자를 통해 서버는 요청한 사용자 에이전트의 애플리케이션, 운영체제, 공급업체 및/또는 버전을 식별할 수 있습니다. 예를 들어, 이 값은 Mozilla/5.0 (Macintosh; Intel Mac OS X 10\_15\_7) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/127.0.0.0 Safari/537.36처럼 전달할 수 있습니다.

<div id="constructing-the-conversion-event-request">
  ### 전환 이벤트 요청 구성
</div>

`POST: version/measurement/conversions/:pixel_id`

특정 광고 계정에 대한 전환 이벤트를 전송합니다. 응답 코드에서 성공 여부(HTTP 200 OK)를 확인해야 합니다. 오류 코드가 반환될 경우를 대비해 재시도 메커니즘과 기본 로깅을 마련할 것을 권장합니다.

엔드포인트의 URL과 POST 본문 매개변수에 대한 자세한 내용은 [API Reference](https://developer.x.com/en/docs/twitter-ads-api/measurement/api-reference/conversions) 섹션을 참조하세요.

<div id="rate-limit">
  ### 속도 제한
</div>

속도 제한은 계정당 15분마다 이벤트 60,000건입니다.

다음과 같이 이 요청 호출 외부에서 별도의 로직을 구현해야 할 수 있다는 점에 유의하세요. 예를 들어:

1. 이벤트별로 올바른 전환 데이터를 전송할 수 있도록 사용자 행동을 계측(로깅)하는 처리

2. 관련 개인정보 선택권을 행사한 사용자의 전환 이벤트를 필터링하기 위한 필요한 로직 — 예를 들어 광고주 웹사이트에서 추적 또는 개인 정보 판매를 거부(opt-out)한 경우

3. 이벤트를 포착하고 전환을 전송하기 위한 이벤트 트리거 및 페이지와의 연동

<div id="testing-events">
  ### 이벤트 테스트
</div>

이벤트가 전환 이벤트를 정상적으로 수신하면 12~24시간 내에 Ads Manager의 전환 추적 페이지에서 ‘단일 이벤트 웹 태그’의 상태가 TRACKING으로 표시됩니다. 전환 API를 통해 전환을 전송해도 진행 중인 캠페인에는 영향을 주지 않습니다.

다음 방법으로 태그 ID별 전환 이벤트의 분석 결과를 확인할 수도 있습니다.

- Ads Manager 데이터 내보내기([웹사이트 전환 추적 분석 도움말 페이지](https://business.x.com/en/help/campaign-measurement-and-analytics/conversion-tracking-for-websites.html))

- Ads API를 통한 데이터 내보내기(segmentation\_type=CONVERSION\_TAGS) 

<div id="duplication-between-pixel-and-conversion-api">
  ### Pixel과 Conversion API 간 중복 제거
</div>

Pixel과 Conversion API 요청 간 전환 중복을 제거하려면 중복 제거 키로 conversion\_id를 사용하세요. 중복 제거는 이벤트 수준에서만 수행됩니다. 다시 말해, 픽셀과 CAPI 요청 간 중복을 제거하려면 동일한 conversion\_id를 사용할 뿐만 아니라 픽셀과 CAPI 요청 모두에서 동일한 이벤트를 보내야 합니다. 중복 제거는 수신 시점이 48시간 이내인 이벤트에만 적용됩니다.

  ### 요약
</div>

전환 추적은 X에서 광고를 본 후 상호작용한 사용자가 원하는 행동을 수행한 X 사용자 수를 측정할 수 있게 해줍니다. 이를 통해 어떤 캠페인이 사이트 방문, 가입, 구매 등의 행동을 유도하는지 파악할 수 있습니다. 이는 광고주가 직접 반응형 광고의 성과를 이해하고 비용 효율적으로 고객을 확보할 수 있도록 X 외부에서도 측정할 수 있는 기능을 제공합니다.

전환 태그를 사용하면 광고주는 사용자 전환을 추적하고 이를 X의 광고 캠페인과 연계할 수 있습니다. 이를 통해 캠페인을 최적화하여 획득당 비용(CPA) 목표를 달성하는 데 필요한 가시성을 확보할 수 있습니다.

전환 추적을 통해 광고주가 측정할 수 있는 웹사이트 내 다양한 행동이 있습니다. 광고 캠페인을 통해 유도하려는 행동에 따라 하나 이상을 선택할 수 있습니다.

- 사이트 방문: 사용자가 광고주의 사이트 내 랜딩 페이지를 방문함
- 구매: 사용자가 광고주의 사이트에서 제품 또는 서비스를 구매 완료함
- 다운로드: 사용자가 광고주의 사이트에서 백서나 소프트웨어 패키지 등의 파일을 다운로드함
- 가입: 사용자가 광고주의 서비스, 뉴스레터 또는 이메일 수신에 가입함
- 사용자 지정: 위 범주에 속하지 않는 맞춤 행동을 위한 포괄적 범주

X의 전환 추적은 광고주에게 전환 기여도를 종합적으로 보여줍니다. 고유 클릭 URL과 서드파티 추적 태그 조합과 같은, X의 자체 전환 추적 기능 대신 사용되었을 수 있는 서드파티 측정 시스템과 비교할 때, X의 전환 태그는 트윗 펼치기, 리트윗, 즐겨찾기, 답글, 팔로우와 같은 퍼널 중상단 참여뿐만 아니라 노출에 기인한 전환까지 추적할 수 있습니다.

<div id="401-unauthorized">
  ### 401 인증 실패(Unauthorized)
</div>

**이유**: 인증 자격 증명이 없거나 올바르지 않음

**해결 방법**: 3가지 인증 방식 중 하나를 사용하여 [설정 문서](https://developer.x.com/en/docs/twitter-ads-api/measurement/guides/conversion-api)의 인증 단계를 따르세요:

Ads API 애플리케이션 소유자 핸들이 아닌 다른 사용자 핸들의 사용자 액세스 토큰은 3-legged OAuth 플로우로 생성해야 합니다. 3-legged OAuth로 액세스 토큰을 생성하는 방법은 다음과 같습니다

- [twurl 유틸리티를 사용한 웹 기반 인증(명령줄)](https://github.com/twitter/twurl#getting-started)

- [PIN 기반 인증(명령줄)](https://developer.x.com/resources/fundamentals/authentication/pin-based-oauth)

- [3-legged OAuth 패턴을 구현한 커스텀 웹 플로우](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)

Conversion API에 사용하는 모든 사용자 토큰은 AD\_MANAGER 또는 ACCOUNT\_ADMIN 액세스 수준을 가진 사용자에 대한 것이어야 하며, 이는 [authenticated\_user\_access](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/authenticated-user-access) 엔드포인트에서 확인할 수 있습니다.

응답 코드가 성공(HTTP 200 OK)인지 확인해야 합니다. 오류 코드가 반환될 경우를 대비해 재시도 메커니즘과 기본 로깅을 갖추는 것을 권장합니다.

요청 한도는 계정당 15분마다 100,000건입니다(요청당 최대 500개 이벤트).
