---
title: 인증된 요청 수행
---

X Ads API 엔드포인트에 액세스하려면 애플리케이션이 TLS를 사용해 https://ads-api.x.com 으로 보안 인증 웹 요청을 전송해야 합니다.

다음 섹션에서는 인증된 API 요청 수행 개요, API와 상호작용하기 위한 [Twurl](https://github.com/twitter/twurl#getting-started) 설정 방법, 그리고 애플리케이션을 확장해 OAuth 1.0a를 지원하고 Ads 계정에 요청을 보내는 방법을 설명합니다.

<div id="requirements">
  ## 요구 사항
</div>

X Ads API에 인증된 요청을 보내기 전에 다음이 필요합니다.

- [승인된 개발자 계정](/ko/resources/fundamentals/developer-portal)
- [X Ads API](/ko/x-ads-api/introduction) 액세스가 승인된 애플리케이션
- [앱 관리 UI](/ko/resources/fundamentals/developer-apps)를 통해 발급받은 API 키와 시크릿, 그리고
- X Ads 계정에 접근 권한이 있는 사용자의 [액세스 토큰](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)

<div id="using-the-api">
  ## API 사용
</div>

Advertising API는 https://ads-api.x.com에서 사용할 수 있습니다. [표준 REST API](https://developer.x.com/en/docs/x-api/v1/tweets/post-and-engage/overview)와 Advertising API는 동일한 클라이언트 앱에서 함께 사용할 수 있습니다. Advertising API는 HTTPS 사용을 강제하므로, HTTP로 엔드포인트에 접근하면 오류 메시지가 반환됩니다.

Ads API의 출력은 JSON입니다. 모든 식별자는 문자열이며, 모든 문자열은 UTF-8입니다. Advertising API는 [버전이 관리](/ko/x-ads-api/fundamentals/versioning)되며, 버전은 모든 리소스 URL의 첫 번째 경로 요소로 지정됩니다.

`https://ads-api.x.com/<version>/accounts`

<div id="http-verbs-typical-response-codes">
  ## HTTP 동사 및 일반적인 응답 코드
</div>

X Ads API에서 사용되는 HTTP 동사는 네 가지입니다:

- **GET**은 데이터를 조회합니다
- **POST**는 캠페인 등의 새 데이터를 생성합니다
- **PUT**은 라인 아이템 등 기존 데이터를 업데이트합니다
- **DELETE**는 데이터를 삭제합니다.

삭제는 영구적이지만, 리소스를 요청할 때 `with_deleted=true` 매개변수를 명시적으로 포함하면 대부분의 GET 기반 메서드에서 삭제된 데이터를 계속 조회할 수 있습니다. 그렇지 않으면 삭제된 레코드는 HTTP 404를 반환합니다.

요청이 성공하면 리소스를 생성, 삭제 또는 업데이트할 때 객체를 나타내는 JSON과 함께 HTTP 200대 응답이 반환됩니다.

HTTP PUT으로 데이터를 업데이트할 때는 지정한 필드만 변경됩니다. 선택적 값을 해제(unset)하려면 해당 매개변수를 빈 문자열로 지정하면 됩니다. 예를 들어, 다음 매개변수 묶음은 이미 지정된 `end_time`을 해제합니다: `&end_time=&paused=false`.

오류 응답에 대한 자세한 내용은 [오류 코드 및 응답](/ko/x-ads-api/fundamentals/error-codes-and-responses)을 참조하세요.

<div id="in-line-parameters">
  ## 인라인 매개변수
</div>

대부분의 리소스 URL에는 하나 이상의 인라인 매개변수가 포함됩니다. 많은 URL은 쿼리 문자열에 명시적으로 선언된 매개변수를 받거나, POST 또는 PUT 요청의 경우 본문으로 매개변수를 받습니다.

인라인 매개변수는 각 리소스의 **Resource Path** 섹션에서 앞에 콜론(”:”)을 붙여 표시됩니다. 예를 들어, 작업 중인 계정이 `"abc1"`로 식별되고 [계정과 연관된 캠페인을 조회](/ko/x-ads-api/campaign-management#campaigns)하려는 경우, `https://ads-api.x.com/6/accounts/abc1/campaigns` URL을 사용해 해당 목록에 액세스합니다. 리소스 URL(`https://ads-api.x.com/6/accounts/:account_id/campaigns`)에 설명된 인라인 `account_id` 매개변수를 지정하면, 해당 계정과 연관된 객체로만 요청 범위가 한정됩니다.

<div id="using-access-tokens">
  ## 액세스 토큰 사용
</div>

X Ads API는 서명된 HTTPS 요청을 사용해 애플리케이션의 신원을 검증하고, 애플리케이션이 대리로 API 요청을 수행하는 최종 사용자가 부여한 권한을 사용자 액세스 토큰으로 확인합니다. Ads API에 대한 모든 HTTP 호출은 HTTPS 프로토콜을 통해 Authorization 요청 헤더(OAuth 1.0a 사용)를 포함해야 합니다.

X Ads API와 연동하려면 애플리케이션에 OAuth 1.0a Authorization 요청 헤더 생성 기능을 추가해야 합니다. 다만 서명 요청 생성이 복잡하므로, X API를 지원하거나 OAuth 1.0a 요청 처리를 구현한 기존 라이브러리를 사용할 것을 강력히 권장합니다. 다음은 [권장 OAuth 라이브러리](/ko/resources/fundamentals/authentication#oauth-1-0a-2)와 [인증 코드 샘플](/ko/resources/fundamentals/authentication#oauth-1-0a-2) 목록입니다.

검증된 라이브러리를 사용하는 과정에서 인증 오류가 발생한 경우에는 지원해 드릴 수 있으나, 커스텀 OAuth 구현은 지원하지 않습니다.

<div id="http-oauth">
  ## HTTP 및 OAuth
</div>

X REST API v1.1과 마찬가지로 광고 API는 [OAuth 1.0A](/ko/resources/fundamentals/authentication)와 HTTPS의 동시 사용을 요구합니다. API 키는 [앱 관리 콘솔](/ko/resources/fundamentals/developer-apps)에서 발급받을 수 있습니다. 액세스 토큰은 “현재 사용자”를 나타내기 위해서도 반드시 사용해야 합니다. 현재 사용자는 광고 기능이 활성화된 X 계정입니다.

직접 구현하기보다는 OAuth 라이브러리를 사용하는 것을 강력히 권장합니다. 검증된 라이브러리를 사용할 경우에는 디버깅을 지원할 수 있지만, 자체 구현한 OAuth는 지원할 수 없습니다. 사용할 수 있는 [라이브러리](/ko/resources/fundamentals/authentication#oauth-1-0a-2)를 확인하세요.

이 API는 HTTP/1.1과 OAuth 준수에 엄격합니다. OAuth 서명 기반 문자열을 준비하기 전에 URL과 POST 본문 내 [예약 문자 인코딩](https://tools.ietf.org/html/rfc3986#section-2.2)을 적절히 처리했는지 확인하세요. 특히 광고 API는 시간을 지정할 때 “:” 문자, 옵션 집합을 제공할 때 “,” 문자를 사용합니다. 두 문자는 다음의 예약 문자 집합에 포함됩니다:

| Symbol | URL Encoded |
| :--- | :--- |
| !   | %21 |
| #   | %23 |
| $   | %24 |
| &   | %26 |
| '   | %27 |
| (   | %28 |
| )   | %29 |
| \*   | %2A |
| +   | %2B |
| ,   | %2C |
| /   | %2F |
| :   | %3A |
| ;   | %3B |
| =   | %3D |
| ?   | %3F |
| @   | %40 |
| \[   | %5B |
| ]   | %5D |

<div id="making-your-first-api-request-with-twurl">
  ## Twurl로 첫 번째 API 요청 보내기
</div>

X는 [cURL](https://en.wikipedia.org/wiki/CURL)의 대안으로 OAuth 1.0a 인증 헤더를 지원하는 명령줄 도구 [Twurl](https://developer.x.com/en/docs/tutorials/using-twurl)을 유지 관리합니다. Twurl은 인증된 API 요청을 간편하게 보내고, 애플리케이션에 인증을 추가하기 전에 Ads API를 탐색해 볼 수 있는 방법을 제공합니다.

[Twurl 설치 및 권한 부여](https://github.com/twitter/twurl#getting-started)를 완료하면 액세스 토큰을 빠르게 생성하고 Ads API에 인증된 요청을 보낼 수 있습니다.

```bash
twurl -H "ads-api.x.com" "/5/accounts/"
```

API를 통해 캠페인을 생성하는 이 [단계별](/ko/x-ads-api/campaign-management#creating-a-campaign-step-by-step) 튜토리얼을 따라 Twurl과 API에 익숙해지세요.

<div id="extending-your-application-to-make-authenticated-requests">
  ### 인증된 요청을 수행하도록 애플리케이션 확장하기
</div>

Twurl로 X Ads API에 요청을 보내는 방법에 익숙해졌다면, 이제 애플리케이션에 OAuth 1.0a 인증 헤더 생성을 지원하도록 추가할 차례입니다.

[OAuth 1.0a](/ko/resources/fundamentals/authentication) 인증 헤더에는 애플리케이션과 사용자의 신원을 모두 검증하고 요청 변조를 방지하는 정보가 포함됩니다. 애플리케이션은 각 API 요청마다 새로운 Authorization 헤더를 생성해야 합니다. 많은 언어에서 X에 API 요청을 보낼 때 이 Authorization 헤더 생성을 지원하는 오픈 소스 라이브러리를 제공합니다.

다음은 C#, PHP, Ruby, Python을 사용하는 예입니다: [코드 샘플](/ko/resources/fundamentals/authentication#oauth-1-0a-2).

<div id="custom-implementation">
  ## 사용자 지정 구현
</div>

오픈 소스 라이브러리의 지원 없이 OAuth 1.0a 인증을 구현해야 하는 경우가 있습니다. [요청 승인](/ko/resources/fundamentals/authentication#authorizing-a-request)에서는 Authorization 헤더를 생성할 수 있도록 지원을 구현하는 방법을 자세히 안내합니다. 커뮤니티에서 지원하는 라이브러리 사용을 강력히 권장합니다.

일반적인 절차:

1. oauth\_로 시작하는 헤더용 키/값 쌍 7개 수집
2. 해당 키/값 쌍을 사용하여 [OAuth 1.0a HMAC-SHA1 서명](/ko/resources/fundamentals/authentication#creating-a-signature) 생성
3. 위 값을 사용하여 [Authorization 헤더](/ko/resources/fundamentals/authentication#authorizing-a-request) 구성
