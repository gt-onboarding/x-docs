<div id="typical-response-structure">
  ## 일반적인 응답 구조
</div>

성공한 응답은 200번대 HTTP 코드와 함께, 서버가 요청을 어떻게 해석했는지에 대한 설명과 요청된 객체(생성, 수정, 삭제된 객체 포함)를 담은 JSON 기반 페이로드로 표시됩니다.

요청이 성공했다면, 응답에 원 요청을 반영한 `request` 노드가 포함됩니다.

_예시:_ [GET accounts/abcdefg/campaigns?with\_deleted=true](/ko/x-ads-api/campaign-management#campaigns)

```json
{
  /* 응답 데이터... */,
  "request": {
    "params": {
      "account_id": "abcdefg",
      "with_deleted": "true"
    }
  }
}
```

JSON 응답의 `data` 필드는 사용한 리소스와 연관된 구체적인 객체를 포함합니다. 응답에 하나 이상의 결과가 포함될 수 있는 경우 `data` 노드는 JSON 배열 형식입니다. 하나의 결과만 가능한 경우 JSON 해시로 반환됩니다. 드물게, 일반적으로 컬렉션이 포함되어야 하는 응답이 해시맵으로 제공되는 경우가 있습니다. 이때 단일 해시맵은 `type` 필드에 지정된 것과 동일한 유형의 객체로 간주하면 됩니다.

<div id="error-response-structure">
  ## 오류 응답 구조
</div>

오류 응답은 200대가 아닌 HTTP 코드로 반환됩니다. 일반적으로 JSON 응답이 포함되지만, 일부 오류는 다른 형태의 본문을 반환하기도 합니다. 이처럼 응답 구조를 파싱할 수 없는 경우에는 HTTP 코드의 본래 의미를 우선 고려하십시오. 예를 들어, HTML 응답과 함께 HTTP 404가 반환되는 경우가 있을 수 있습니다. 이때는 콘텐츠를 찾을 수 없다고(HTTP 404는 “Not Found”를 의미) 판단해도 안전합니다.

일반적인 오류 응답은 성공 응답과 유사한 구조를 따릅니다. 오류의 성격은 응답의 `errors` 노드에 포함됩니다. `errors/code` 노드는 해석 로직에 활용할 수 있는 CAPS\_CASE 상수형 오류 코드를 나타냅니다. `errors/message` 노드는 (보통) 사람이 읽을 수 있는 영어 오류 설명을 제공합니다. 오류에 대한 더 세밀한 정보를 제공하기 위해 추가 필드가 포함될 수 있습니다.

```json
{
  "errors": [
    {
      "parameter": "start_time",
      "details": "invalid date",
      "code": "INVALID_PARAMETER",
      "value": "",
      "message": "Expected time, got \"\" for start_time"
    }
  ],
  "request": {
    "params": {
      "account_id": "hkk5"
    }
  }
}
```

위 예시에서는 분석 엔드포인트에 대한 요청에서 `start_time` 매개변수의 값이 유효하지 않습니다. 잘못된 매개변수가 있는 요청의 `errors/code`는 `INVALID_PARAMETER`입니다.
