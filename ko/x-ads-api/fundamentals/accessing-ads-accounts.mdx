<div id="user-accounts-vs-ad-accounts">
  # 사용자 계정과 광고 계정
</div>

Ads API에는 두 가지 유형의 계정이 있습니다: 광고 계정과 X 사용자 계정. Ads API 문서 전반에서 “account”라는 용어는 보통 광고 계정을 의미합니다.

- 광고 계정은 **business.x.com**에서 등록되며 API에서는 account\_id로 식별됩니다. 광고 계정은 자금원과 직접 연결되어 있고, 하나 이상의 X 사용자 계정의 콘텐츠를 ‘프로모션 가능한 사용자’로 활용합니다. 각 광고 계정은 하나 이상의 X 사용자 계정에 권한을 부여할 수 있습니다. 광고 계정, 즉 “현재 계정”은 거의 모든 요청의 URL에서 인라인 :account\_id 매개변수로 표시됩니다.
- X 사용자 계정(예: @AdsAPI)은 Ads API에서 user\_id로 식별됩니다. 이러한 계정은 하나 이상이 특정 광고 계정과 연결될 수 있습니다. API 요청을 수행하는 인증된 X 사용자 계정은 ‘현재 사용자’라고 합니다. 현재 사용자가 액세스할 수 있는 광고 계정 목록은 [GET accounts](/ko/x-ads-api/campaign-management#accounts)에서 확인할 수 있습니다. ‘프로모션 가능한 사용자’는 특정 광고 계정이 프로모션할 수 있는 X 핸들입니다. 자세한 내용은 [Obtaining Ads Account Access](/ko/x-ads-api/introduction)를 참고하세요.

<div id="methods-for-ad-account-access">
  ## 광고 계정에 액세스하는 방법
</div>

광고주의 계정에 대해 Ads API 요청을 수행하는 방법은 두 가지가 있습니다.

1. 광고주를 대신하여 요청하기(권장)
2. [대행사](https://business.x.com/en/blog/agencies-manage-multiple-Twitter-Ads-accounts.html)처럼 여러 계정을 지원하는 경우와 같이, 광고주의 계정에 대한 액세스 권한이 부여된 본인 계정을 사용해 요청하기

이 문서는 두 옵션 간 차이를 간략히 요약한 것이며, [다중 사용자 로그인 FAQ](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html) 등 다른 리소스와 함께 참고하시기 바랍니다.

[요청 승인](/ko/resources/fundamentals/authentication#authorizing-a-request#authorizing-a-request)에 설명된 대로, Ads API에 대한 모든 요청에는 [OAuth 1.0a](/ko/resources/fundamentals/authentication) 기반의 Authorization 헤더가 필요하며, [3-legged](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)[OAuth](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) 플로우를 통해 획득한 액세스 토큰을 사용합니다. 애플리케이션은 웹 기반 OAuth 플로우를 구현하여 [액세스 토큰을 획득](/ko/resources/fundamentals/authentication#oauth-1-0a-2)해야 합니다. Ads API 개발자는 X 광고주에게 로그인 자격 증명 공유를 요청해서는 절대 안 됩니다.

기본적으로 각 [X 개발자 애플리케이션](/ko/resources/fundamentals/developer-apps)에는 애플리케이션 소유 계정에 대해 Ads API 요청을 수행하는 데 사용할 수 있는 정적 **액세스 토큰**이 포함되어 있습니다. 이러한 자격 증명은 3-legged 또는 [PIN 기반 OAuth](/ko/resources/fundamentals/authentication#pin-based-authorization) 플로우 없이 단일 계정 사용 사례에 적합합니다. 다른 X Ads 계정에 액세스하지 않는 경우, 다음 단계 대신 이러한 [단일 사용자](/ko/resources/fundamentals/authentication#oauth-1-0a-2) 자격 증명을 사용하십시오.

<div id="levels-of-access">
  ## 접근 수준
</div>

<div id="app-level-permissions">
  ### 앱 수준 권한
</div>

각 사용자는 [Ads API 신청](/ko/x-ads-api/getting-started/step-by-step-guide) 시 요청한 대로 접근 수준을 부여받습니다:

- **Conversion Only**: 사용자는 [읽기 및 쓰기](/ko/resources/fundamentals/developer-apps) 권한으로 [Mobile](/ko/x-ads-api/measurement/mobile-conversions) 및 [Web Conversion](/ko/x-ads-api/measurement/web-conversions) 엔드포인트에 접근할 수 있습니다.
- **Standard Access**: 사용자는 [읽기 및 쓰기](/ko/resources/fundamentals/developer-apps) 권한으로 [Analytics](/ko/x-ads-api/analytics), [Campaign Management](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/overview), [Creatives](https://developer.x.com/en/docs/twitter-ads-api/creatives/overview), [Custom Audiences](https://developer.x.com/en/docs/twitter-ads-api/audiences/overview), [Conversion](/ko/x-ads-api/measurement/web-conversions) 엔드포인트에 접근할 수 있습니다.

**참고:** 2023년 7월 이전에 액세스를 요청한 Ads API 개발자는 액세스 수준과 권한이 다를 수 있으며, OAuth 토큰이 5개로 제한될 수 있습니다. 기존 애플리케이션에 추가 엔드포인트 접근 권한을 부여하거나 토큰 한도를 해제하려면 [액세스 확대](/ko/x-ads-api/getting-started/increasing-access) 가이드를 참조하세요.

<div id="ad-account-level-permissions">
  ### 광고 계정 수준 권한
</div>

Ads 계정에 액세스 권한이 있는 각 사용자는 특정 계정 수준 권한을 갖습니다: **Account administrator**, **Ad manager**, **Campaign analyst**, **Organic analyst**, **Creative Manager**. 계정 수준 권한에 대한 최신 문서는 **business.x.com**을 참조하세요. 애플리케이션은 현재 인증된 사용자의 권한을 [Authenticated User Access](/ko/x-ads-api/campaign-management#authenticated-user-access) API 엔드포인트로 조회해, 접근 가능한 API 엔드포인트와 Ads 기능을 판별해야 합니다.

**참고:** [Conversion API](/ko/x-ads-api/measurement/web-conversions)를 사용할 때의 모든 사용자 토큰은 **Account administrator** 또는 **Ad manager** 계정 수준 권한을 가진 사용자여야 합니다.                                                                                                                                              

<div id="methods-of-obtaining-access-tokens">
  ## 액세스 토큰 획득 방법
</div>

<div id="1-obtain-an-advertisers-user-access-token">
  ### 1. 광고주(사용자) 액세스 토큰 받기
</div>

광고주의 **액세스 토큰**을 받는 방법은 두 가지가 있습니다. 가장 일반적인 방법은 웹 UI에서 직접 [3-legged OAuth 플로우](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)를 사용하는 것입니다. 광고주가 사용할 수 있는 공개 UI가 없는 애플리케이션은 [PIN 기반](/ko/resources/fundamentals/authentication#pin-based-authorization) OAuth 프로세스를 구현할 수 있습니다. 사용자가 3-legged 플로우를 완료하면 애플리케이션은 API를 통해 해당 사용자의 Ads 계정에 요청을 보낼 수 있는 자격 증명을 얻게 됩니다.

OAuth 플로우를 통해 사용자 자격 증명을 받는 방법은 대부분의 X Ads API 개발자에게 광고주 계정에 접근하기 위한 **강력 권장** 방식입니다. 이를 통해 사용자를 대신해 API를 호출하고 그 사용자로서 작업을 수행할 수 있습니다. 이 토큰은 만료되지 않지만 사용자가 언제든지 취소할 수 있습니다.

<div id="2-obtain-your-developer-access-token">
  ### 2. (개발자) 액세스 토큰 받기
</div>

이 옵션을 사용하려면 광고주가 business.x.com의 X UI에서 [귀하의 @username](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html) (또는 여러 @username)에 자신의 X Ads 계정 접근 권한을 부여해야 합니다. 귀하의 계정에 대해 3-legged OAuth 플로우로 발급받은 **액세스 토큰**은 광고주의 X Ads 계정에 접근할 수 있습니다.

이는 광고주의 OAuth 토큰 대신 귀하 자신의 @username의 OAuth 토큰으로 API를 호출할 수 있게 해줍니다. 이 옵션의 핵심 차이는, 귀하의 @username에 게시물 위임/작성 권한이 부여된 경우에만 Promoted-Only Posts를 생성할 수 있다는 점입니다.

해당 계정의 `FULL` 홍보 가능 사용자를 대신해 Promoted-Only Posts를 생성하려면, 이 플로우에서 게시물을 생성할 수 있는 권한도 추가로 부여받아야 합니다. 그러면 [GET accounts/:account\_id/authenticated\_user\_access](/ko/x-ads-api/campaign-management#accounts) 엔드포인트에서 `TWEET_COMPOSER` 권한을 통해 접근이 가능합니다.

<div id="differences-between-these-methods">
  ## 이러한 방법 간의 차이점
</div>

|     |     |     |
| :--- | :--- | :--- |
|     | 광고주(사용자) OAuth 토큰 | (개발자) OAuth 토큰  <br /> (@username이 별도 계정으로 추가됨) |
| 광고 계정에 액세스 | ✔   | ✔   |
| 사용자를 대신해 게시물 생성 | ✔   | ✔\*  |
| 캠페인 관리 | ✔   | ✔   |
| 분석 액세스 | ✔   | ✔   |
| 사용자를 대신해 카드 생성 | ✔   | ✔   |
| 개발자의 X Ads UI  <br />액세스 가능 여부 |     | ✔   |
| [Rate Limits](/ko/x-api/enterprise-gnip-2.0/fundamentals/rate-limits) | 광고주별로 구분 | 광고주 계정별로 구분 |

</Note>

<div id="sample-use-case">
  ## 예시 사용 사례
</div>

<div id="advertisers-access-token-via-oauth-3-legged-web-flow">
  ### 광고주의 액세스 토큰: OAuth 3-legged 웹 플로우
</div>

표준 플로우는 웹 기반이며 [3-legged authorization](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) OAuth 플로우를 사용합니다. 여기의 스크린샷은 [https://github.com/xdevplatform/twauth-web](https://github.com/xdevplatform/twauth-web)에서 소스를 확인할 수 있는 샘플의 일부입니다.

애플리케이션의 적절한 시점에 애플리케이션을 승인받기 위해 X로 리디렉션해야 합니다.

<Frame>
  <img src="https://cdn.cms-twdigitalassets.com/content/dam/developer-twitter/images/twauth-web-2.png" alt="image0" />
</Frame>

요청 토큰과 함께 X로 리디렉션하면 사용자는 애플리케이션 승인 요청을 받게 됩니다.

<Frame>
  <img src="https://cdn.cms-twdigitalassets.com/content/dam/developer-twitter/images/twauth-web-3.png" alt="image1" />
</Frame>

애플리케이션이 승인되면 사용자는 요청 토큰을 생성할 때 제공한 콜백 URL로 리디렉션됩니다. 이를 사용해 이 사용자의 영구 액세스 토큰을 획득하고 로컬에 저장합니다.
