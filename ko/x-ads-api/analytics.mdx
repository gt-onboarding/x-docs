<div id="introduction">
  ## 소개
</div>

분석 지표는 파트너와 광고주가 X에서 프로모션하는 콘텐츠의 성과를 파악하는 데 도움을 줍니다. 여기에는 노출, 클릭, 동영상 조회, 지출액 등의 정보가 포함됩니다. 또한 파트너와 광고주는 도달한 잠재고객을 다양한 세그먼트로 구분해 상세 지표를 확인할 수 있습니다.

X Ads API는 캠페인 성과 지표를 상세히 조회하는 두 가지 방식을 지원합니다: 동기식과 비동기식. 동기식 분석 호출은 요청한 지표를 응답 본문으로 즉시 반환합니다. 비동기식 분석 엔드포인트는 연결된 “작업” 처리가 완료된 후 다운로드 가능한 결과 파일에서 요청한 지표를 제공합니다. 동기식 엔드포인트는 짧은 기간을 지원하여 실시간 캠페인 최적화에 적합합니다. 비동기식 엔드포인트는 훨씬 긴 기간을 지원하므로 더 많은 데이터를 수집하는 데 적합하며, 보고서 생성이나 과거 데이터 보강에 이상적입니다.

<div id="details">
  ## 상세 정보
</div>

<div id="synchronous-vs-asynchronous">
  ### 동기식 vs. 비동기식
</div>

동기식과 비동기식 분석 엔드포인트의 차이점은 아래 표에 요약되어 있습니다. 이 정보는 개발자가 어떤 엔드포인트 집합을 사용할지 선택하는 데 도움을 주기 위한 것입니다.

| 기능 | 동기식 | 비동기식 |
|:---------|:------------|:--------------|
| 할당량 제한 | 사용자 수준: 15분당 요청 250건 | 계정 수준: 동시 작업 100건\* |
| 기간 범위 | 7일 | 90일(비세그먼트)<br />45일(세그먼트) |
| 세그먼트 분할 | 아니요 | 예 |
| 응답 내용 | 지표 데이터 | 작업의 처리 상태\*\* |
| 권장 사용 사례 | 실시간 최적화<br />UI 요청 | 정기 동기화<br />과거 데이터 백필 |

<Note>
  \* 임의의 시점에 처리 상태에 있을 수 있는 최대 동시 작업 수를 의미합니다.

  \*\* 작업 처리가 성공적으로 완료되면 URL이 반환됩니다. 이 URL에서 압축(gzip)된 결과 파일을 다운로드할 수 있습니다.

  이 외에는 두 엔드포인트는 동일한 기능을 제공합니다.
</Note>

<div id="use-cases">
  ### 사용 사례
</div>

주요 분석 사용 사례는 세 가지입니다.

1. 실시간 최적화: 성과 지표를 활용하여 활성 캠페인 업데이트
2. 동기화: 정기적으로 실행되는 백그라운드 동기화
3. 신규 계정 온보딩: 과거 데이터 백필

동기식 분석 엔드포인트는 최근 5~15분 내 지표 변화에 따라 캠페인을 업데이트하는 실시간 최적화에 사용할 수 있습니다. 분석 동기화에는 어느 엔드포인트든 사용할 수 있습니다. 원하는 시간 범위와 세분화가 필요한지 여부에 따라 사용할 엔드포인트가 달라진다는 점을 유의하세요. 신규 계정 온보딩은 비동기 분석 엔드포인트만 사용해야 합니다. (동기식 분석 엔드포인트는 대량의 데이터를 가져오는 용도로 사용해서는 안 됩니다.)

비동기 분석 엔드포인트는 지표를 백엔드 프로세스와 동기화하는 경우 대시보드 및 기타 UI 요소에 활용할 수 있습니다. 구현 시 사용자 인터페이스 요청을 직접 처리하기 위해 비동기 분석 엔드포인트를 호출하는 것은 피해야 합니다.

<div id="request-options">
  ### 요청 옵션
</div>

분석 요청은 광고 계정 단위로 범위가 정해지므로 리소스 경로에 계정 ID가 필요합니다. 아래 요청 옵션은 쿼리 매개변수로 지정됩니다. 다음 값들이 필요합니다.

- 엔터티: 엔터티 유형과, 분석을 요청하려는 최대 20개의 엔터티 ID
- 시간 범위: ISO 8601로 표현한 시작 및 종료 시간
  - **참고:** 정시(시간 단위)로 맞춰야 합니다
- 지표 그룹: 관련 지표의 하나 이상의 집합(각 지표 그룹의 지표 목록은 Metrics and Segmentation을 참조)
- 세분화 수준: 지표가 반환될 집계 수준을 지정
- 게재 위치: X 온사이트/오프사이트에서 게재된 광고의 지표를 가져올지 결정
  - **참고:** 요청당 하나의 게재 위치 값만 지정할 수 있습니다

`start_time` 및 `end_time` 요청 매개변수로 시간 범위를 지정하세요. 이 값들은 다음과 같이 지정한 세분화 수준과 정렬되어야 합니다.

1. `TOTAL`: 임의의 시간 범위를 지정(엔드포인트 제한 내)
2. `DAY`: 시작 시간과 종료 시간 모두 계정의 표준시간대 기준 자정에 맞춰야 함
3. `HOUR`: 임의의 시간 범위를 지정(엔드포인트 제한 내)

종료 시간은 배타적입니다. 예를 들어, `start_time=2019-01-01T00:00:00Z`, `end_time=2019-01-02T00:00:00Z`로 요청하면 이 시간 범위가 24시간만 포함하므로 하루치 분석 지표만 반환됩니다(이틀치가 아님).

**세분화**

비동기 분석 엔드포인트에서만 제공되는 세분화는 파트너와 광고주가 특정 타기팅 값별로 분리된 지표를 조회할 수 있게 합니다. 세분화된 지표를 요청하려면 `segmentation_type` 요청 매개변수를 사용하세요. 세분화 옵션에 대한 자세한 내용은 [Metrics and Segmentation](/ko/x-ads-api/analytics#metrics-and-segmentation)을 참조하세요.

<div id="faqs">
  ## 자주 묻는 질문
</div>

왜 X Ads UI에 표시되는 숫자와 Ads API 숫자가 일치하지 않나요?

- 다음 모든 게재 위치에 대한 데이터를 요청했는지 확인하세요: `ALL_ON_TWITTER`, `PUBLISHER_NETWORK`, `SPOTLIGHT`, `TREND`.
- Ads API의 종료 시간은 배타적(exclusive)이며, Ads UI에서는 포함(inclusive)입니다.

왜 데이터를 요청하는 시점에 따라 숫자가 달라지나요?

- 보고 지표가 공개되는 즉시 조회할 수 있으며, 거의 실시간으로 제공됩니다. 다만 초기 결과는 추정치이므로 변경될 수 있습니다. 지표는 지출 데이터를 제외하고 24시간 후에 확정됩니다.
- 지출 지표는 일반적으로 이벤트 발생 후 3일 이내에 확정됩니다. 다만 청구 데이터는 이벤트 발생일로부터 최대 14일 동안 처리됩니다(예: 스팸 필터링).

특정 기간에 대해 어떤 엔터티 ID를 요청해야 하는지 어떻게 알 수 있나요?

- [활성 엔터티 엔드포인트](/ko/x-ads-api/analytics#active-entities-2)를 사용하세요

왜 애널리틱스 응답의 모든 값이 `null`인가요?

- 요청한 기간에 캠페인이 집행되지 않았을 가능성이 큽니다
- 어떤 엔터티의 어떤 기간 데이터를 조회해야 하는지 판단하려면 [활성 엔터티 엔드포인트](/ko/x-ads-api/analytics#active-entities-2)를 사용하세요

API에는 `null` 값이 표시되는데 UI에는 0으로 표시되는 이유는 무엇인가요?

- UI는 이러한 값을 0으로 표시할 뿐이며, 의미상으로는 동일합니다

X 타임라인처럼 더 세분화된 게재 위치의 지표는 어떻게 요청하나요?

- 애널리틱스에서 다음 게재 위치 값을 지원합니다: `ALL_ON_TWITTER`, `PUBLISHER_NETWORK`, `SPOTLIGHT`, `TREND`(즉, [X Audience Platform](/ko/x-ads-api/audiences))

삭제되었거나 일시 중지된 엔터티의 지표를 조회할 수 있나요?

- 가능합니다. 엔터티 상태는 애널리틱스 지표의 제공 여부에 영향을 주지 않습니다.

세분화된 값이 비세분화 값과 일치하지 않는 이유는 무엇인가요?

- 산출 방식에 따라 세분화된 데이터가 비세분화 데이터와 100% 일치하도록 집계되지는 않습니다.

여러 차원으로 세분화된 데이터를 요청할 수 있나요?

- 다중 세분화는 지원하지 않습니다.

<div id="best-practices">
  ## 모범 사례
</div>

X Ads API에서 [analytics](/ko/x-ads-api/analytics) 데이터를 수집할 때 권장되는 모범 사례입니다.

<div id="rate-limiting-and-retries">
  ### 요청 제한 및 재시도
</div>

- 요청 제한이 적용된 쿼리(HTTP 429 상태 코드를 반환하는 경우)에서는 `x-rate-limit-reset` 헤더를 확인하고, 표시된 시각에 또는 그 이후에만 재시도해야 합니다.
- HTTP 503 Service Unavailable 상태 코드를 반환하는 쿼리에서는 `retry-after` 헤더를 확인하고, 표시된 시각 이후에만 재시도해야 합니다.
- 재시도 시각을 준수하지 않는 애플리케이션은 사전 고지 없이 X Ads API에 대한 액세스가 취소되거나 제한될 수 있습니다.

<div id="analytics-metrics-in-a-nutshell">
  ### 분석 지표 한눈에 보기
</div>

- `billed_charge_local_micro`를 제외한 모든 분석 지표는 24시간이 지나면 확정되며 더 이상 변경되지 않습니다.
- `billed_charge_local_micro` 지표는 데이터가 반환된 후 최대 3일 동안 추정치로 제공됩니다.
- 24시간 이후에는, 이 지표가 과다 지출에 대한 크레딧(지정된 `end_time` 이후에 집행된 광고)과 정크로 판정된 과금 대상 이벤트에 대한 크레딧으로 인해 감소할 수 있습니다. 이 지표는 24시간 이후에는 변화 폭이 매우 제한적입니다.
- 자세한 내용은 [Analytics](/ko/x-ads-api/analytics)를 참고하세요.

<div id="fetching-real-time-non-segmented-data">
  ### 실시간 비분할 데이터 가져오기
</div>

- 항상 `start_time`과 `end_time`를 모두 제공하세요.
- 7일을 초과해 오래된 엔터티의 데이터는 가져오지 마세요.
- 가능하면 `HOUR` 세분화 수준으로 데이터를 요청하세요. 언제든지 집계해 `DAY` 및 `TOTAL` 세분화 수준으로 롤업할 수 있습니다.
- 가능하면 `line_items` 및 `promoted_tweets` 수준에서 데이터를 요청하세요. 언제든지 이러한 지표를 집계해 캠페인, 자금 조달 수단, 계정 등 전체 광고 엔터티 계층의 합계를 얻을 수 있습니다.
- 분석 지표 값을 로컬에 저장하세요.
- 30일을 초과해 오래된 데이터를 반복해서 조회하지 마세요. 이 데이터는 변경되지 않으므로 로컬에 저장해야 합니다.
- 모든 비분할 데이터는 실시간이며, 이벤트 발생 후 수초 내에 확인할 수 있습니다.
- 전환 지표와 비전환 지표는 별도의 요청으로 나누어 요청하세요.

<div id="fetching-segmented-data">
  ### 세분화된 데이터 가져오기
</div>

- 위의 “실시간 비세분화 데이터 가져오기” 지침을 참조하세요. 아래에 추가 안내가 있습니다.
- 대부분의 세분화된 데이터 유형은 경우에 따라 최대 1시간 동안 완전하지 않을 수 있습니다. `INTERESTS`로 세분화된 데이터는 최대 12시간까지 지연될 수 있습니다.
- 해당 정보의 산출 방식상, 세분화된 데이터가 비세분화 데이터와 100% 일치하여 합산되지 않을 수 있습니다.

<div id="fetching-historical-data">
  ### 과거 데이터 가져오기
</div>

- 과거 데이터를 보충(예: 새 광고주 계정 추가)할 때는 `start_time`과 `end_time` 범위를 더 작은 구간으로 나눠 여러 번 요청해야 할 수 있습니다.
- 조회 기간을 30일 단위의 날짜 창으로 제한하세요.
- 이 요청들은 속도를 조절하고 시간에 분산해, 해당 조회의 레이트 리밋을 소진하지 않도록 하세요.

<div id="metrics-by-objective">
  ## 목표별 지표
</div>

엔터티에 어떤 지표가 적용되는지는 [캠페인 목표](/ko/x-ads-api/campaign-management)에 따라 달라집니다. 이 가이드를 참고하여 각 목표 유형별로 가져올 관련 지표 그룹을 결정하고, 추가로 도출 지표를 계산하는 방법을 확인하세요.

<div id="engagements">
  ### `ENGAGEMENTS`
</div>

**관련 지표 그룹:** `ENGAGEMENT` 및 `BILLING`. 크리에이티브에 미디어가 포함된 경우 `MEDIA`도 해당됩니다.

|     |     |
| :--- | :--- |
| 파생 지표 | 노출 지표 계산식 |
| 참여율 | `engagements/impressions` |
| CPE | `billed_charge_local_micro/engagements` |
| 미디어 조회율 | `media_views/impressions` |

<div id="website_clicks-and-website_conversions">
  ### `WEBSITE_CLICKS` 및 `WEBSITE_CONVERSIONS`
</div>

**관련 지표 그룹:** `ENGAGEMENT`, `BILLING`, `WEB_CONVERSION`. 크리에이티브에 미디어가 사용된 경우 `MEDIA`도 해당됩니다.

|     |     |
| :--- | :--- |
| 파생 지표 | 공개 지표 계산식 |
| CPM | `billed_charge_local_micro/impressions/1000` |
| 클릭률 | `clicks/impressions` |
| CPLC | `billed_charge_local_micro/clicks` |
| 총 전환 수 | `conversion_custom` + `conversion_site_visits` + `conversion_sign_ups` + `conversion_downloads` + `conversion_purchases` |
| 전환율 | 총 전환 수 / `impressions` |
| CPA | `billed_charge_local_micro` / 총 전환 수 |

<div id="app_installs-and-app_engagements">

  ## 가이드

<div id="active-entities">
  ### 활성 엔터티
</div>

<div id="introduction">
  #### 소개
</div>

[활성 엔터티 엔드포인트](/ko/x-ads-api/analytics#get-stats-accounts-account-id-active-entities)는 어떤 캠페인에 대해 분석을 요청할지 판단하는 데 필요한 정보를 제공하므로, 당사의 [동기식](/ko/x-ads-api/analytics#get-stats-accounts-account-id) 및 [비동기식](/ko/x-ads-api/analytics#asynchronous-analytics) 분석 엔드포인트와 함께 사용하도록 설계되었습니다. 이 엔드포인트는 광고 엔터티와 해당 지표의 변경 시점에 대한 세부 정보를 반환합니다. 이를 활용하면 코드와 분석 수집 로직을 크게 단순화할 수 있습니다.

이 가이드는 엔드포인트와 그 데이터 소스에 대한 정보와 배경을 제공합니다. 또한 [사용 지침](#usage)과 [요청 예시](#example)를 통해 분석 엔드포인트와 함께 활성 엔터티를 활용하는 방법을 보여줍니다. [요약 섹션](#summary)에서는 권장 접근 방식에 대한 상위 수준 개요를 제공합니다.

<div id="data">
  #### 데이터
</div>

광고 엔터티의 지표가 변경될 때마다 해당 변경에 대한 정보를 기록합니다. 이러한 변경 이벤트는 시간 단위 버킷으로 저장되며, 엔터티에 대한 세부 정보와 변경이 적용되는 시점에 대한 정보를 포함합니다. 변경 이벤트는 항상 기록된 시점과 일치하지 않을 수 있으므로, 후자의 정보가 필요합니다. 청구 조정이 흔한 이유이지만 다른 경우도 있습니다.

<div id="endpoint">
  #### 엔드포인트
</div>

<div id="request">
  ### 요청
</div>

활성 엔터티 요청은 광고 계정 범위에서 수행되며, `entity`, `start_time`, `end_time`의 세 가지 필수 쿼리 매개변수가 필요합니다.

`twurl -H ads-api.x.com "/11/stats/accounts/18ce54d4x5t/active_entities?entity=PROMOTED_TWEET&start_time=2019-03-05T00:00:00Z&end_time=2019-03-06T00:00:00Z"`

지원되는 `entity` 값은 다음과 같습니다: `CAMPAIGN`, `FUNDING_INSTRUMENT`, `LINE_ITEM`, `MEDIA_CREATIVE`, `PROMOTED_ACCOUNT`, `PROMOTED_TWEET`. 이는 당사의 분석 엔드포인트가 지원하는 엔터티 유형을 반영합니다.

`start_time` 및 `end_time` 값은 [ISO 8601](https://en.wikipedia.org/wiki/ISO_8601) 형식으로 표기해야 하며, 조회할 시간 단위 구간을 지정합니다. 값은 반드시 정시(정각)로 지정해야 합니다.

이 엔드포인트는 결과를 필터링하는 데 사용할 수 있는 선택적 매개변수 `funding_instrument_ids`, `campaign_ids`, `line_item_ids`도 지원합니다. 이는 광고 계층 구조의 모든 수준에서, 어떤 `entity` 유형과도 함께 사용할 수 있습니다.

<div id="response">
  ### 응답
</div>

위 요청에 대한 활성 엔터티 응답은 아래에 표시되어 있습니다.

```json
    {
      "request": {
        "params": {
          "account_id": "18ce54d4x5t",
          "entity": "PROMOTED_TWEET",
          "start_time": "2019-03-05T00:00:00Z",
          "end_time": "2019-03-06T00:00:00Z"
        }
      },
      "data": [
        {
          "entity_id": "2r0wxw",
          "activity_start_time": "2019-03-04T20:55:20Z",
          "activity_end_time": "2019-03-05T03:43:56Z",
          "placements": [
            "ALL_ON_TWITTER"
          ]
        },
        {
          "entity_id": "2r30fn",
          "activity_start_time": "2019-03-05T08:11:08Z",
          "activity_end_time": "2019-03-05T14:40:59Z",
          "placements": [
            "ALL_ON_TWITTER",
            "PUBLISHER_NETWORK"
          ]
        }
      ]
    }
```

`data` 배열에는 이후 분석 요청에 포함해야 하는 각 엔터티의 객체가 담겨 있습니다. 이 집합에 없는 ID에 대해 분석을 요청해서는 안 됩니다.

각 객체에는 네 가지 필드가 있습니다: `entity_id`, `activity_start_time`, `activity_end_time`, `placements`. 활동 시작 및 종료 시간은 해당 엔터티의 변경 이벤트가 적용되는 기간을 나타내며, 이에 따라 이후 분석 요청에서 지정해야 할 날짜가 결정됩니다. `placements` 배열에는 다음 값이 포함될 수 있습니다: `ALL_ON_TWITTER`, `PUBLISHER_NETWORK`, `SPOTLIGHT`, `TREND`. 이는 주어진 엔터티 ID에 대해 어떤 게재 위치를 요청해야 하는지를 나타냅니다.

<div id="usage">
  #### 사용
</div>

활성 엔터티 엔드포인트는 분석 요청을 어떻게 수행할지에 대한 기준이 되어야 합니다. 다음 사용 지침은 분석 동기화를 지원하기 위해 작성되었으며, 파트너가 데이터 저장소를 Twitter와 동기화 상태로 유지하도록 돕습니다. 즉, 정기적으로 예약된 백그라운드 동기화를 수행하는 방법을 설명합니다.

개발자는 다음 두 가지를 결정해야 합니다.

1. 활성 엔터티 정보를 얼마나 자주 요청할지, 그리고 그에 따라 분석을 얼마나 자주 가져올지
2. 활동 시작 및 종료 시간을 어떻게 사용하여 분석 요청의 `start_time` 및 `end_time` 값을 정할지

요약 뒤에 이어지는 아래 두 개의 하위 섹션에서 각 항목을 더 자세히 설명합니다.

<div id="summary">
  ### 요약
</div>

다음과 같은 방식으로 활성 엔터티 엔드포인트를 사용해 분석 요청 방식을 결정하세요. 활성 엔터티 정보를 요청할 주기와 그에 따른 분석 수집 주기를 정한 뒤 아래 절차를 따르세요.

1. 활성 엔터티 요청을 보냅니다.
2. 응답을 게재 위치별로 분리합니다. `ALL_ON_TWITTER`, `PUBLISHER_NETWORK`, `SPOTLIGHT`, `TREND` 각각을 하나의 그룹으로 나눕니다.
3. 각 게재 위치 그룹에 대해 다음을 수행합니다.
   1. 엔터티 ID를 추출합니다.
   2. 분석용 `start_time` 및 `end_time` 값을 결정합니다.
      - 최소 `activity_start_time`을 찾고, 이 값을 내림 처리합니다.
      - 최대 `activity_end_time`을 찾고, 이 값을 올림 처리합니다.
   3. 분석 요청을 보냅니다.
      - 엔터티 ID를 20개 단위의 배치로 묶습니다.
      - 3-b에서 결정한 `start_time`과 `end_time` 값을 사용합니다.
      - 적절한 `placement` 값을 지정합니다.
   4. 데이터 저장소에 기록합니다.

Python SDK 사용 예시는 [active\_entities.py](https://github.com/xdevplatform/twitter-python-ads-sdk/blob/master/examples/active_entities.py)를 참고하세요.

<div id="frequency">
  ### 빈도
</div>

첫 번째 질문에 대한 답은 활성 엔터티 요청에 사용할 시간 범위를 결정합니다. 예를 들어 매시간 활성 엔터티 정보를 요청한다면 시간 범위는 1시간이어야 합니다. 하루에 한 번 활성 엔터티 정보를 요청한다면 시간 범위는 1일이어야 합니다. 즉, 현재 요청의 `start_time`이 이전 요청의 `end_time`과 같도록 시간 범위를 선택해야 합니다.

<Note>
  **참고**: 하나의 시간 구간은 한 번만 요청해야 합니다. 동일한 시간 구간을 두 번 이상 요청하면 불필요한 분석 요청이 발생합니다. (아래 예외 참고.)
</Note>

파트너가 _현재_ 시간에 대해 한 시간 동안 여러 차례 분석을 요청하려는 경우에도 동일한 패턴이 적용됩니다. 즉, 요청 빈도가 시간 범위를 결정합니다. 아래 표는 이 시나리오에서 활성 엔터티의 시작 및 종료 타임스탬프 예시입니다.

|     |     |     |
| :--- | :--- | :--- |
| **요청 시각** | **`start_time` 타임스탬프** | **`end_time` 타임스탬프** |
| 00:15:00 | 00:00:00 | 00:15:00 |
| 00:30:00 | 00:15:00 | 00:30:00 |
| 00:45:00 | 00:30:00 | 00:45:00 |
| 01:00:00 | 00:45:00 | 01:00:00 |

변경 이벤트의 저장 방식상 위 네 가지 활성 엔터티 요청은 모두 동일한 시간 단위 버킷을 조회하며, 이는 이 사용 사례에 필요합니다. 다만 현재 시간이 지나면 해당 시간 단위 버킷은 더 이상 조회하지 않아야 합니다.

<div id="activity-times">
  ### 활동 시간
</div>

활동 시작 및 종료 시간을 처리할 때는 다음 접근 방식을 권장합니다. 활성 엔터티 응답의 모든 오브젝트에서 최소 `activity_start_time`과 최대 `activity_end_time`을 찾습니다. 최소 활동 시작 시간은 내림, 최대 활동 종료 시간은 올림으로 반올림해 값을 조정합니다. 구체적으로, 두 타임스탬프의 시·분·초를 0으로 설정하고 종료 시간에 하루를 추가합니다. 이는 아래 표에 설명되어 있습니다. 이렇게 도출된 시간이 이후 분석 요청에서 지정해야 할 시작 및 종료 시간입니다.

|     |     |
| :--- | :--- |
| **최소/최대 활동 시간** | **도출된 시간** |
| 2019-03-04T20:55:20Z  <br />  <br />2019-03-05T14:40:59Z | 2019-03-04T00:00:00Z  <br />  <br />2019-03-06T00:00:00Z |

**참고**: 시·분·초가 0으로 설정된 타임스탬프를 포함하는 것이 중요합니다. 그렇지 않고 날짜만 전달되면, 광고 계정의 시간대에서 자정에 시작해 자정에 종료하는 분석을 요청한 것으로 간주되어 원치 않는 결과가 발생할 수 있습니다. 예를 들어, 최소 활동 시작 시간이 2019-02-28T01:30:07Z인데 오프셋이 -08:00:00인 광고 계정에서 타임스탬프가 생략되면, 분석 요청은 01:30과 08:00 사이에 발생한 변경 사항을 놓치게 됩니다.

또는 하루 단위로 확장하지 않고 반환된 활동 시간 구간에 대해서만 분석을 요청할 수도 있습니다. 이 접근 방식을 사용하면 도출된 시작 및 종료 시간은 각각 2019-03-04T20:00:00Z와 2019-03-05T15:00:00Z가 됩니다. (분석 요청에서 `DAY` 세분화 수준을 지정한 경우에는 이러한 범위가 허용되지 않습니다.)

<div id="introduction">
  #### 소개
</div>

비동기 분석 엔드포인트는 파트너와 광고주가 생성 요청을 제출하면 서버가 이를 비동기적으로 처리하여 지표를 반환하도록 합니다. (이를 비동기 분석 “작업”이라 부릅니다.) 이 방식에서는 요청이 완료될 때까지 클라이언트 연결을 유지할 필요가 없습니다.

이 엔드포인트는 동기식 엔드포인트와 마찬가지로 파트너와 광고주가 캠페인 성과에 대한 상세 통계를 요청할 수 있게 합니다. 계정, 결제 수단, 캠페인, 라인 아이템, 프로모션된 트윗, 미디어 크리에이티브에 대한 데이터 요청을 지원합니다. 동기식 엔드포인트와의 차이는 비동기 분석 엔드포인트가 최대 90일의 더 긴 날짜 범위와 세분화를 지원한다는 점입니다. 두 방식 간 차이에 대한 자세한 내용은 [Analytics Overview](/ko/x-ads-api/analytics) 페이지에서 확인하실 수 있습니다.

동기식 엔드포인트와 달리 레이트 리밋은 특정 계정의 동시 작업 수를 기준으로 적용됩니다. 즉, 특정 시점에 처리 상태일 수 있는 작업 수를 기준으로 합니다. 이는 광고 계정 수준에서 계산합니다.

<div id="usage">
  #### 사용 방법
</div>

비동기 분석 엔드포인트를 사용해 캠페인 지표를 가져오는 과정은 여러 단계로 이루어집니다. 작업을 생성하고, 해당 작업의 처리가 완료됐는지 확인한 뒤, 데이터를 다운로드합니다. 데이터 파일은 압축을 해제해야 합니다. 아래에 네 가지 구체적인 단계를 정리했습니다.

1. [POST stats/jobs/accounts/:account\_id](/ko/x-ads-api/analytics#asynchronous-analytics) 엔드포인트로 작업을 생성합니다.
2. [GET stats/jobs/accounts/:account\_id](/ko/x-ads-api/analytics#asynchronous-analytics) 엔드포인트에 일정 간격으로 요청을 보내 작업 처리 완료 여부를 확인합니다.
3. 작업 처리가 완료되면 데이터 파일을 다운로드합니다.
4. 데이터 파일의 압축을 해제합니다.

데이터 파일에 포함된 response 객체는 동기식 분석 엔드포인트의 response와 동일한 JSON 스키마를 따릅니다.

세분화된 캠페인 지표는 비동기 분석 엔드포인트로만 제공됩니다. 캠페인 지표는 위치, 성별, 관심사, 키워드 등으로 분할할 수 있습니다. 전체 옵션 목록은 [Metrics and Segmentation](/ko/x-ads-api/analytics#metrics-and-segmentation) 페이지를 참조하세요. 세분화된 지표를 요청하려면 작업을 생성할 때 `segmentation_type` 요청 매개변수를 사용하세요.
