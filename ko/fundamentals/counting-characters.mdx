이 페이지에서는 트윗 작성과 X API 전반에서 문자를 어떻게 처리하는지 설명합니다. 구현 세부 정보는 X에서 제공하는 오픈 소스 [twitter-text](http://github.com/twitter/twitter-text) 라이브러리를 참고하세요. 라이브러리는 [GitHub](https://github.com/twitter)에서 확인할 수 있습니다.

<div id="background">
  ## 배경
</div>

X는 SMS 문자 기반 서비스로 시작했습니다. 이 때문에 초기 트윗 길이는 140자로 제한되었는데(160자 제한의 SMS에서 명령어와 사용자 이름에 20자를 할당했기 때문입니다), 시간이 지나 X가 진화하면서 최대 트윗 길이는 280자로 확대되었습니다. 여전히 짧고 간결하지만 더 많은 표현이 가능해졌습니다.

<div id="definition-of-a-character">
  ## 문자 정의
</div>

대부분의 경우 트윗의 텍스트 콘텐츠는 최대 280개의 문자 또는 [Unicode](https://unicode.org/) 글리프를 포함할 수 있습니다. 일부 글리프는 한 문자를 초과해 계산됩니다.

특정 글리프가 한 문자로 계산되는지, 그 이상으로 계산되는지를 우리는 그 글리프의 가중치라고 부릅니다. 어떤 문자가 한 문자보다 큰 가중치를 갖는지에 대한 정확한 정의는 [twitter-text 트윗 파싱 라이브러리](https://github.com/twitter/twitter-text)의 [구성 파일](https://github.com/twitter/twitter-text/tree/master/config)에 있습니다.

현재 버전의 구성 파일은 기본 두 문자 가중치와, 가중치가 다르게 적용되는 [Unicode 코드 포인트](https://unicode.org/charts/About.html)의 네 가지 범위를 정의합니다. 현재 이 범위의 코드 포인트는 모두 한 문자로 계산됩니다.

- 첫 번째 범위는 Latin-1 코드 페이지 전반의 문자를 포함합니다(U+0000 - U+10FF).
- 두 번째 범위는 Zero Width Joiner(이모지 및 기타 글리프 결합에 사용)를 포함한 일반 구두점입니다(U+2000-U+200D).
- 세 번째 범위는 Unicode 방향 표식인 U+200E 및 U+200F를 제외한 일반 구두점입니다(U+2010-U+201F).
- 마지막 범위는 따옴표 기호를 포함합니다(U+2032-U+2037).

트윗 텍스트의 예시와 twitter-text 라이브러리가 계산한 길이는 라이브러리의 [validate.yml](https://github.com/twitter/twitter-text/blob/master/conformance/validate.yml) 테스트 구성 파일에서 확인할 수 있습니다.

**예시**

| 표시된 문자 | 길이 | 설명 | Unicode 시퀀스 |
|:---------------------|:--------|:-------------|:------------------|
| a | 1 | 라틴 소문자 a | U+0061 |
| á | 1 | 라틴 소문자 a(어큐트 악센트) | U+00E1 |
| ӑ | 1 | 키릴 소문자 a(브레베) | U+04D1 |
| Ồ | 1 | 라틴 대문자 O(서컴플렉스+악센트) | U+1ED2 |

<div id="emojis">
  ## 이모지
</div>

[twemoji](https://twemoji.x.com/)에서 지원되는 이모지는 결합 수정자 여부와 관계없이 항상 두 글자로 계산됩니다. 여기에는 [Fitzpatrick 피부 톤](https://emojipedia.org/modifiers/) 또는 [성별 수정자](https://blog.emojipedia.org/unicode-and-the-emoji-gender-gap/)가 적용된 이모지도 포함되며, 유니코드 코드 포인트가 훨씬 더 많더라도 동일합니다. 이모지의 가중치는 표준 이모지가 하나 이상의 유니코드 Zero Width Joiner(U+200D)로 결합된 시퀀스를 찾는 정규식으로 twitter-text에 정의되어 있습니다.

**예시**

| 표시된 이모지 | 길이 | 설명 | 유니코드 시퀀스 |
|:-----------------|:--------|:-------------|:------------------|
| 👾 | 2 | 알려진 이모지의 기본 길이 | -- |
| 🙋🏽 | 2 | 피부 톤 수정자가 있는 이모지 | 🙋 U+1F64B, 🏽 U+1F3FD |
| 👨‍🎤 | 2 | 결합용 글리프(제로 폭 조이너)를 사용하는 이모지 시퀀스 | [👨 U+1F468](https://emojipedia.org/emoji/%F0%9F%91%A8/), [U+200D](https://emojipedia.org/emoji/%E2%80%8D/), [🎤 U+1F3A4](https://emojipedia.org/emoji/%F0%9F%8E%A4/) |
| 👨‍👩‍👧‍👦 | 2 | 여러 결합용 글리프(제로 폭 조이너)를 사용하는 이모지 시퀀스 | [👨 U+1F468](https://emojipedia.org/emoji/%F0%9F%91%A8/), [U+200D](https://emojipedia.org/emoji/%E2%80%8D/), [👩 U+1F469](https://emojipedia.org/emoji/%F0%9F%91%A9/), [U+200D](https://emojipedia.org/emoji/%E2%80%8D/), [👧 U+1F467](https://emojipedia.org/emoji/%F0%9F%91%A7/), [U+200D](https://emojipedia.org/emoji/%E2%80%8D/), [👦 U+1F466](https://emojipedia.org/emoji/%F0%9F%91%A6/) |

<div id="chinese-japanese-korean-glyphs">
  ## 중국어/일본어/한국어 글리프
</div>

CJK(중국어/일본어/한국어) 언어에서 사용되는 글리프는 두 글자로 계산됩니다. 따라서 CJK 텍스트로만 구성된 트윗은 이러한 유형의 글리프를 최대 140개까지만 포함할 수 있습니다.

<div id="entity-objects">
  ## 엔터티 객체
</div>

트윗에는 [엔터티 객체](/ko/x-api/fundamentals/data-dictionary)가 포함될 수 있으며, 이 중 일부는 트윗의 길이에 영향을 줍니다.

URL: [모든 URL은 t.co 링크로 변환됩니다](https://developer.x.com/content/developer-twitter/en/docs/basics/tco). 이는 URL의 길이가 [twitter-text 구성 파일](https://github.com/twitter/twitter-text/tree/master/config)의 `transformedURLLength` 매개변수로 정의됨을 의미합니다. 트윗에서 URL은 실제 길이가 더 짧더라도 현재 23자로 계산됩니다.

답글: 답글 트윗의 시작 부분에 자동으로 채워지는 @이름은 글자 수 제한에 포함되지 않습니다. @멘션으로 시작하는 새로운 비답글 트윗은 포함되며, 사용자가 트윗 본문에 직접 추가한 @멘션도 포함됩니다.

미디어: 공식 클라이언트에서 게시된 경우 pic.x.com URL로 표시되는, 트윗에 첨부된 미디어는 글자 수 0으로 계산됩니다.

엔터티 객체에 대한 자세한 내용은 [개발자 문서](/ko/x-api/fundamentals/data-dictionary)를 참조하세요.

<div id="x-character-encoding">
  ## X 문자 인코딩
</div>

X API 엔드포인트는 UTF-8로 인코딩된 텍스트만 허용합니다. 다른 모든 인코딩은 텍스트를 API로 보내기 전에 UTF-8로 변환해야 합니다.

X는 트윗의 길이를 텍스트의 정규화 형식 C(NFC) 버전을 기준으로 계산합니다.

예: 단어 "café". 겉보기에는 동일하지만 사용하는 바이트 수가 다른 두 가지 바이트 시퀀스가 있습니다.

|     | | |
|:-----|:---|:---|
| café | 0x63 0x61 0x66 0xC3 0xA9 | "é" 문자를 사용하는, 이른바 "합성 문자"입니다. |
| café | 0x63 0x61 0x66 0x65 0xCC 0x81 | 결합 분음 부호를 사용하며, "e" 위에 겹칩니다. |

정규화 형식 C는 장형 버전(0x65 0xCC 0x81)보다 완전히 결합된 문자(café 예시의 0xC3 0xA9) 사용을 선호합니다.

X는 UTF-8 바이트 수가 아니라 텍스트의 코드 포인트 수를 계산합니다. café 예시의 0xC3 0xA9는 UTF-8에서 두 바이트로 인코딩되는 하나의 코드 포인트(U+00E9)인 반면, 0x65 0xCC 0x81는 세 바이트로 인코딩되는 두 개의 코드 포인트입니다.
