<div id="overview">
  ## 개요
</div>

기존 앱이 있는 경우 [developer portal의 App 페이지](https://developer.x.com/en/portal/projects-and-apps)에서 확인, 수정 또는 삭제할 수 있습니다.

X API 및 X Ads API에 액세스하려면 각 요청에 함께 전달해야 하는 [인증](/ko/resources/fundamentals/authentication) 자격 증명(키와 토큰이라고도 함) 세트가 필요합니다. 사용하는 특정 엔드포인트가 요구하는 인증 유형에 따라 자격 증명의 형태가 달라질 수 있습니다.

아래는 앱에서 생성할 수 있는 자격 증명의 종류와 사용 방법입니다.

- **[API Key and Secret](/ko/resources/fundamentals/authentication#api-key-and-secret):** 사실상 앱의 사용자 이름과 비밀번호에 해당합니다. [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication#oauth-1-0a-2)가 필요한 요청을 인증하거나, 사용자 액세스 토큰 또는 앱 액세스 토큰 등 다른 토큰을 생성하는 데 사용합니다.

- **[Access Token and Secret](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow):** 일반적으로 액세스 토큰은 귀하가 대신하여 요청을 수행하는 사용자를 나타냅니다. developer portal에서 생성하는 토큰은 앱 소유자(해당 사용자)를 나타냅니다. [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication#oauth-1-0a-2)가 필요한 요청을 인증하는 데 사용합니다. 다른 사용자를 대신해 요청하려면, 해당 사용자가 승인을 부여할 수 있도록 3-legged OAuth 플로우를 사용해야 합니다.

- **[Client ID and Client Secret](/ko/resources/fundamentals/authentication#oauth-2-0):** 이 자격 증명은 OAuth 2.0 인증으로 사용자 액세스 토큰을 발급받는 데 사용됩니다. OAuth 1.0a와 유사하게, 사용자 액세스 토큰은 비공개 사용자 계정 정보를 제공하거나 다른 계정을 대신해 작업을 수행하는 요청을 인증하는 데 사용되며, 클라이언트 애플리케이션이 사용자에 대해 갖는 접근 권한을 보다 세밀한 스코프로 제어할 수 있습니다.

- **[App only Access Token](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only):** X에서 공개적으로 제공되는 정보를 응답하는 엔드포인트에 요청할 때 이 토큰을 사용합니다.

X API 요청에 필요한 키와 토큰을 생성하는 것 외에도, [permissions](/ko/resources/fundamentals/developer-apps#app-permissions)을 설정하고, 앱의 사용 사례 또는 목적을 문서화하며, [callback URL](/ko/resources/fundamentals/developer-apps#callback-urls)을 정의하고, [management dashboard](https://developer.x.com/en/portal/projects-and-apps) 내에서 앱 개발 환경과 관련된 기타 설정을 수정할 수 있습니다.

<div id="apps-and-projects">
  ### 앱과 프로젝트
</div>

X Developer Platform에서 사용 사례별로 작업을 체계화하기 위해 앱과 [프로젝트](/ko/resources/fundamentals/projects)를 사용할 수 있습니다. 각 프로젝트에는 Free X API 요금제는 앱 1개, Basic 요금제는 최대 2개, Pro 요금제는 최대 3개의 앱을 포함할 수 있습니다.

새로운 [X API v2](/ko/x-api/introduction) 엔드포인트에 액세스하려면 프로젝트에 연결된 앱의 키와 토큰을 사용해야 합니다.

프로젝트를 도입하기 전에 생성된 앱이 있는 경우 "Standalone Apps" 섹션에서 확인할 수 있습니다. Standalone Apps는 프로젝트 구조 밖에 있는 앱을 의미합니다. Standalone App을 프로젝트에 연결하면 v2 엔드포인트로 요청을 보낼 수 있습니다.

<div id="developer-portal-dashboard">
  ### 개발자 포털 대시보드
</div>

계정과 연결된 앱을 관리하려면 [대시보드](https://developer.x.com/content/developer-twitter/en/apps)를 방문하세요. 자세한 내용은 [개발자 포털](/ko/resources/fundamentals/developer-portal) 문서 페이지를 참조하세요. 대시보드는 개발자가 다음 작업을 빠르고 쉽게 수행할 수 있도록 해줍니다:

- 기존 독립형 앱과 해당 App ID를 확인합니다.
- 새 프로젝트, 앱 또는 독립형 앱을 생성합니다.
- 사용하지 않는 프로젝트 또는 앱을 삭제합니다.
- 이름, 설명, 웹사이트, 콜백 URL, [권한](/ko/resources/fundamentals/developer-apps#app-permissions) 등 특정 앱의 설정을 검토하거나 업데이트합니다.
- API Key & Secret, App Access Token, 앱 소유자의 사용자 Access Token 등 앱 전용 자격 증명을 다시 생성합니다.

<div id="signing-up-for-access">
  ### 액세스 신청
</div>

기존 X 앱이 있는 경우 developer.x.com에서 X 계정으로 로그인한 상태라면 X [앱 대시보드](https://developer.x.com/content/developer-twitter/en/portal/projects-and-apps)에서 앱을 조회하고 편집할 수 있습니다. 이전에 developer.x.com에서 생성된 모든 앱을 관리하기 위해 새 계정을 등록할 필요는 **없습니다**.

새로운 X 앱을 만들어야 하는 경우 [개발자 계정에 가입](https://developer.x.com/en/portal/petition/essential/basic-info)해야 합니다. [팀 계정](/ko/resources/fundamentals/developer-portal#team-management)의 구성원인 경우, 새 앱을 생성하려면 관리자 역할이 부여되어 있어야 합니다.

<div id="automated-account-labeling-for-bots">
  ### 봇용 자동화 계정 라벨 지정
</div>

X 사용자에게 귀하의 봇이 자동화된 계정임을 알리기 위해 봇 계정에 자동화 계정 라벨을 추가할 수 있습니다. 이러한 봇은 X API를 통해 프로그래밍된 작업을 수행합니다. 봇에 자동화 계정 라벨을 추가하면 대상 사용자와의 신뢰를 구축하고 계정의 신뢰성을 높이며 스팸성 봇과 구분할 수 있습니다. 이는 사용자가 봇과 상호작용할 때 귀하의 계정 목적을 더 잘 이해하는 데 도움이 됩니다.

봇 계정에 라벨을 추가하려면 다음 단계를 따르세요:

1. 계정 설정으로 이동합니다
2. "Your account"을 선택합니다
3. "Automation"을 선택합니다
4. "Managing account"를 선택합니다
5. 다음으로, 봇을 운영하는 X 계정을 선택합니다
6. 비밀번호를 입력해 로그인합니다
7. 마지막으로 라벨이 계정에 적용되었음을 확인하는 메시지가 표시됩니다.

<div id="app-management">
  ## 앱 관리
</div>

<div id="introduction">
  ### 소개
</div>

[X App 대시보드](https://developer.x.com/en/portal/projects-and-apps)를 통해 개발자는 다음 작업을 빠르고 쉽게 수행할 수 있습니다.

- 보유 중인 앱과 [프로젝트](/ko/resources/fundamentals/projects), 그리고 각 앱에 연결된 App ID를 확인합니다.
- 새 프로젝트 또는 독립 실행형 앱을 생성합니다.
- 프로젝트, 앱, 또는 독립 실행형 앱을 삭제합니다.
- 특정 앱을 클릭해 해당 앱의 설정을 엽니다. 설정에서 앱 세부 정보, 키와 토큰, 권한을 확인할 수 있습니다.
- 앱의 사용자 인증 설정을 OAuth 1.0a 또는 OAuth 2.0으로 업데이트합니다.

<Note>
  **참고:**

  모든 앱 키와 토큰은 이제 Developer Portal에서 더 이상 조회할 수 없으므로, 생성 시 안전하게 보관해야 합니다. 키와 토큰은 비밀번호 관리자를 사용해 저장하는 것을 권장합니다.

  자격 증명이 어떤 앱에 해당하는지 식별할 수 있도록 API 키 힌트를 표시할 수 있습니다.
</Note>

<div id="app-settings">
  ### 앱 설정
</div>

<div id="app-details">
  #### 앱 세부 정보
</div>

여기에서 앱 아이콘, 앱 이름, 앱 설명, 웹사이트 URL, [callback URL/redirect URI](/ko/resources/fundamentals/developer-apps#callback-urls), 서비스 약관 URL, 개인정보 처리방침 URL, 조직 이름, 조직 URL, 그리고 앱의 목적 또는 사용 사례를 편집할 수 있습니다.

OAuth 2.0과 OAuth 1.0a는 사용자가 X 계정으로 귀하의 앱에 로그인할 수 있게 해주는 인증 방식입니다. 또한 인증된 사용자를 대신해 앱이 특정 요청을 수행할 수 있도록 합니다. 앱에서 한 가지 방식만 켜거나 두 가지 방식을 모두 사용할 수 있습니다.

이 정보는 항상 최신 상태로 유지하는 것이 중요합니다. 앱 이름과 웹사이트 URL은 귀하의 애플리케이션이 프로그래밍 방식으로 생성한 트윗의 메타데이터 내 소스로 표시됩니다. X 앱의 사용 사례를 변경하는 경우, [Developer Terms](https://developer.x.com/content/developer-twitter/en/developer-terms/agreement-and-policy)를 준수하기 위해 이 설정에서 사용 사례를 반드시 업데이트하십시오.

애플리케이션에 ‘suspended’ 태그가 표시된다면, 이는 앱이 X의 [developer terms](https://developer.x.com/en/developer-terms.html)(예: [automation rules](https://help.x.com/en/rules-and-policies/twitter-automation)) 중 하나 이상을 위반했기 때문입니다. 개발자 플랫폼 정책팀은 앱 소유자의 X 계정에 등록된 이메일 주소로 개발자에게 안내를 전달합니다. 이 이메일 주소를 확인하려면 [X 계정 설정](https://x.com/settings/your_twitter_data/account)을 검토하십시오. 정지 관련 알림 메일은 “Application suspension notice”와 유사한 제목으로 해당 주소로 전송되며, 이후 조치에 대한 구체적인 안내가 포함됩니다. 정지 문제 해결을 위해 X 팀과 협력하려면 이메일의 세부 지침을 확인하시거나 [platform help form](https://help.x.com/forms/platform)을 사용하십시오.

<div id="keys-and-tokens">
  #### 키와 토큰
</div>

프로젝트의 앱 또는 독립 실행형 앱에서 다음 토큰을 조회, 재발급, 또는 취소할 수 있습니다:

- [API Key(Consumer Key) 및 API Secret(Consumer Secret)](/ko/resources/fundamentals/authentication#api-key-and-secret)
- [App Access Token](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)
- [User Access Token 및 Access Token Secret](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) -
  개발자 포털에서 제공되는 Access Token과 Secret은 해당 앱의 소유자 사용자에 연결됩니다.

**유의하세요** - 앱 소유자가 아닌 다른 사용자를 대신해 요청하려면 [OAuth 1.0a 3-legged OAuth flow](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens) 또는 OAuth 2.0 [Authorization Code with PKCE flow](/ko/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2)를 사용해 사용자용 Access Token 세트를 발급받아야 합니다. 이후 API 요청 시 이러한 사용자 전용 토큰을 사용하세요.

<div id="user-authentication-settings">
  #### 사용자 인증 설정
</div>

앱의 인증 설정으로 OAuth 1.0a 또는 OAuth 2.0을 선택할 수 있습니다. OAuth 2.0은 X API v2에서만 사용할 수 있습니다. OAuth 2.0은 사용자 대신 특정 권한을 부여하기 위해 세밀한 범위(scope)를 선택할 수 있게 해줍니다. OAuth 1.0a는 X API v1.1 및 v2에서 사용할 수 있으며, 상대적으로 큰 범위의 포괄적 권한 부여를 사용합니다.

<div id="oauth-10a-application-user-permissions">
  #### OAuth 1.0a 애플리케이션-사용자 권한
</div>

앱 소유자는 앱의 권한(읽기 전용, 읽기 및 쓰기, 또는 읽기·쓰기·다이렉트 메시지)을 조정할 수 있습니다. 이는 X API를 통해 접근 가능한 리소스와 이벤트의 범위를 제어합니다(참고: 일부 리소스는 X로부터의 추가 권한이 필요합니다).

또한 이 페이지에서 앱이 사용자 이메일 주소 요청 권한을 켜거나 끌 수 있습니다(이를 위해서는 “App details” 페이지에 서비스 약관 URL과 개인정보 처리방침 URL을 입력해야 합니다).

<div id="oauth-20-type-of-app">
  #### OAuth 2.0 앱 유형
</div>

사용자 인증 방식으로 OAuth 2.0을 선택하면, 생성하려는 앱의 유형을 선택해야 합니다. 선택지는 Native App, Single Page App, Web App, Automated App 또는 bot입니다. Native App과 Single Page App은 공개 클라이언트이고, Web App과 Automated App 또는 bot은 기밀 클라이언트입니다.

기밀 클라이언트는 인증 서버와 안전하게 상호 인증하며, 클라이언트 시크릿을 안전하게 보관합니다. 공개 클라이언트는 일반적으로 브라우저나 모바일 기기에서 실행되는 애플리케이션으로, 클라이언트 시크릿을 사용할 수 없습니다. 기밀 클라이언트 유형의 앱을 선택하면 클라이언트 시크릿이 제공됩니다.

<div id="app-permissions">
  ## 앱 권한
</div>

<div id="oauth-10a-app-permissions">
  ### OAuth 1.0a 앱 권한
</div>

앱 권한은 OAuth 1.0a 애플리케이션-사용자 인증의 액세스 수준을 설명합니다. 앱 권한은 [X App](/ko/resources/fundamentals/developer-apps) 설정에서 앱별로 구성됩니다.

사용 가능한 권한 수준은 세 가지입니다:

1. 읽기 전용
2. 읽기 및 쓰기
3. 읽기, 쓰기 및 다이렉트 메시지 접근

추가로, 사용자의 이메일 주소 보기 권한을 요청하는 옵션이 있으며, 이는 위의 세 가지 수준 중 어느 것과도 결합할 수 있습니다.

권한 수준이 변경되면 해당 X 앱에 이미 발급된 모든 사용자 토큰을 폐기해야 하며, 업데이트된 권한이 토큰에 반영되도록 사용자는 앱을 다시 승인해야 합니다.

모범 사례로, 애플리케이션이나 서비스에 필요한 사용자 계정 데이터에 대해 최소한의 액세스만 요청하는 것이 좋습니다.

<div id="read-only">
  ### 읽기 전용
</div>

이 권한 수준은 사용자의 트윗, 홈 타임라인, 프로필 정보 등 X 리소스에 대한 읽기 권한을 제공합니다. 사용자의 다이렉트 메시지를 읽을 수 없으며, 어떤 요소나 오브젝트도 업데이트할 수 없습니다.

<div id="read-and-write">
  ### 읽기 및 쓰기
</div>

이 권한 수준은 X 리소스에 대한 읽기 및 쓰기 권한을 부여합니다. 읽기 권한 외에 트윗 게시, 사용자 팔로우, 사용자 프로필 정보 요소 업데이트도 가능합니다. 또한 인증된 사용자를 대신해 답글을 숨길 수 있습니다. 이 권한 수준은 다이렉트 메시지에 대한 접근을 전혀 허용하지 않습니다(읽기, 쓰기, 삭제 포함).

<div id="read-write-and-access-direct-messages">
  ### 다이렉트 메시지 읽기, 쓰기 및 접근
</div>

<div id="callback-urls">
  ## 콜백 URL
</div>

OAuth 1.0a 사용자 컨텍스트와 OAuth 2.0 Authorization Code with PKCE 인증 방식은 특정 로그인 플로우를 완료한 서로 다른 X 사용자를 대신해 개발자가 요청을 보낼 수 있도록 합니다. 현재 사용자가 애플리케이션을 승인하도록 할 수 있는 두 가지 플로우가 있습니다:

- [OAuth 2.0 authorization code flow with PKCE](/ko/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2)
- [OAuth 1.0a 3-legged OAuth flow](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) (그리고 별도로 [Sign in with X flow](/ko/resources/fundamentals/authentication#log-in-with-x))

사용자가 이 플로우들을 진행하는 동안, 성공적으로 로그인하고 개발자의 앱에 권한을 부여한 뒤 다시 이동할 웹페이지 또는 위치가 필요합니다. 이 후속 웹페이지 또는 위치를 콜백 URL이라고 합니다.

사용자가 이러한 플로우를 진행할 수 있도록 설정할 때, 개발자는 앞서 언급한 플로우를 구성하는 인증 엔드포인트로의 요청에 콜백 URL을 함께 전달해야 합니다. 예를 들어, OAuth 1.0a 사용자 컨텍스트를 사용하는 개발자는 [GET oauth/request\_token](/ko/resources/fundamentals/authentication#post-oauth-request-token) 엔드포인트에 요청을 보낼 때 `callback_url` 매개변수를 전달해야 합니다. 마찬가지로, OAuth 2.0 Authorization Code with PKCE를 사용하는 개발자는 GET oauth2/authorize 엔드포인트로의 요청에 `redirect_uri` 매개변수를 포함해야 합니다.

이러한 매개변수를 사용하는 것 외에도, [개발자 포털](/ko/resources/fundamentals/developer-portal)의 앱 설정 페이지에서 확인할 수 있는 [앱](/ko/resources/fundamentals/developer-apps)의 콜백 URL 허용 목록에 해당 콜백 URL이 추가되어 있는지 확인해야 합니다.

올바르게 설정된 경우, 이러한 플로우의 일환으로 X에 성공적으로 로그인하면 콜백 URL로 리디렉션됩니다.

<div id="things-to-keep-in-mind">
  ### 유의 사항
</div>

콜백 URL을 설정할 때 다음 사항을 염두에 두세요:

**쿼리 매개변수는 반드시 URL 인코딩하세요**

`callback_url` 또는 `redirect_uri` 매개변수로 콜백 URL을 쿼리 매개변수로 전달하므로, 해당 URL을 반드시 URL 인코딩해야 합니다.

**콜백 URL 제한**

X Apps 대시보드에서는 콜백 URL을 최대 10개까지 등록할 수 있습니다. 콜백 URL은 Apps 대시보드에 허용 목록으로 추가한 콜백 URL과 권한 부여 플로우에서 전달하는 매개변수 값이 항상 정확히 일치해야 합니다.

콜백 URL에 요청별 데이터를 포함하려면, 사용자가 리디렉션된 후 포함될 데이터를 저장하기 위해 `state` 매개변수를 사용할 수 있습니다. 데이터 자체를 `state` 매개변수에 인코딩하거나, 해당 매개변수를 세션 ID로 사용해 서버에 상태를 저장할 수 있습니다.

**콜백 URL로 localhost를 사용하지 마세요**
localhost 대신 로컬 환경에서는 사용자 지정 호스트 또는 http(s)://127.0.0.1을 사용하세요

**커스텀 프로토콜 URL**
모바일 딥링크를 활용하려면 twitter://callback/path처럼 경로와 도메인 부분을 포함한 커스텀 프로토콜 URL을 사용할 수 있습니다. 다만 사용이 불가한 프로토콜 목록이 있으므로 아래 목록을 반드시 확인하고 피하세요.

<div id="error-example">
  ### 오류 예시
</div>

개발자 포털의 앱 설정에 올바르게 추가되지 않은 콜백 URL을 사용하는 경우 다음과 같은 오류 메시지가 표시됩니다:

**OAuth 1.0a**

```json HTTP 403 - Forbidden

{
  "errors": [
    {
      "code":415,"message":"Callback URL not approved for this client application. Approved callback URLs can be adjusted in your application settings."
    }
  ]
}
```

또는

```xml
<?xml version="1.0" encoding="UTF-8"?>
<hash>
<error>Callback URL not approved for this client application. Approved callback URLs can be adjusted in your application settings</error>
<request>/oauth/request_token</request>
</hash>
```

**OAuth 2.0**

```json HTTP 400

{
  "error": "invalid_request",
  "error_description": "Value passed for the redirect uri did not match the uri of the authorization code."
}
```

**문제 해결**

오류가 발생했다면, 인증 요청에서 사용하는 콜백 URL이 URL 인코딩되어 있는지 확인하고, 요청에 사용하는 키와 토큰이 속한 앱의 허용 목록에 추가된 콜백 URL과 정확히 일치하는지 확인하세요.
