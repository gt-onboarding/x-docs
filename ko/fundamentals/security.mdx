우리는 프라이버시가 특권이 아닌 권리이며 회사의 토대에 내재되어 있다고 믿습니다. X 개발자 플랫폼을 사용하고 개발자 정책을 준수함으로써, 여러분은 플랫폼이 X의 공적 대화를 뒷받침하고 우리의 프라이버시에 대한 약속을 지키도록 하는 데 중요한 역할을 합니다.

여러분 자신과 앱 사용자들의 데이터를 보호하기 위해 보안을 고려하여 구축하는 것이 얼마나 중요한지 상기드립니다. 보안 침해 위협으로부터 방어하는 것은 여러분의 책임이며, X를 사용하는 사람들을 보호하는 것은 우리의 공동 책임입니다. 이 페이지는 안전한 애플리케이션을 구축하고 데이터 및 접근 권한을 가능한 한 안전하게 유지하는 데 대한 기대 사항을 설명합니다.

<Note>X Developer Platform에서 제공되는 보안 기술(예: [인증](/ko/resources/fundamentals/authentication), TLS, [앱 권한](/ko/resources/fundamentals/developer-apps#app-permissions))은 물론, X 사용자 관점에서의 [서드파티 애플리케이션 및 세션 사용](https://help.x.com/en/managing-your-account/connect-or-revoke-access-to-third-party-apps) 관련 안내도 숙지하시기 바랍니다.</Note>

<div id="reporting-security-issues">
  ## 보안 문제 보고
</div>

X Developer Platform 사용자는 보안 사고가 발생했을 가능성을 처음 인지한 시점부터 48시간 이내에 [X의 취약점 신고 프로그램](https://hackerone.com/twitter)을 통해 X에 반드시 통지해야 합니다.

<div id="security-best-practices">
  ## 보안 모범 사례
</div>

X 개발자 플랫폼을 포함해 인터넷 전반에서 구축 작업을 수행할 때, 다음 사항을 유념해 주세요.

<div id="security-by-design">
  ### 설계 단계에서부터 보안 내재화
</div>

위협 모델링 감사 및/또는 침투 테스트를 수행할 보안 전문가 고용을 고려하세요. 신뢰할 수 있는 보안 업체는 문제를 철저히 파악합니다. X가 이러한 관점을 어떻게 채택했는지는 [블로그 게시물](https://blog.x.com/en_us/topics/company/2019/privacy_data_protection.html)에서 자세히 확인하세요.

또한 X는 모든 파트너에게 다음을 요구합니다.

- 보안이 보장된 리포지토리에서 코드를 관리할 것
- 시스템 개발 생명주기(SDLC) 전반에 걸쳐 위험 분석을 수행할 것
- SDLC 전 단계에서 보안 문제가 식별되고 완화되도록 보장할 것
- SDLC 전 과정에서 환경 분리를 보장할 것
- 모든 테스트 결함이 수정되고, 재검증되며, 종결되도록 보장할 것

<div id="monitor-and-get-alerted">
  ### 모니터링하고 알림 받기
</div>

웹 앱에 문제가 있다고 의심되면 어떻게 확실히 확인할 수 있을까요? 양질의 로그를 유지하고 중요한 예외와 오류에 대한 알림을 받도록 설정하세요. 문제가 발생하는지 한눈에 파악할 수 있도록 핵심 지표를 모은 대시보드를 구성하는 것도 좋습니다.

<div id="create-a-reporting-channel">
  ### 보고 채널 만들기
</div>

사용자가 앱에서 겪은 잠재적 보안 문제를 귀하에게 쉽게 신고할 수 있도록 하십시오. X 사용자와 데이터에 영향을 미치는 문제가 발견된 경우 [해당 문제를 X에 보고](#report)하는 것도 귀하의 책임입니다. 보안 사고 발생 시 영향을 받은 사용자에게 통지하기 위한 실행 계획과 절차를 미리 마련해 두십시오.

<div id="adequate-testing">
  ### 적절한 테스트
</div>

엔드투엔드 테스트가 충분히 철저하며 최신 상태인지 확인하고, 무단 접근과 같은 보안 시나리오에서의 예상 실패를 포함하세요. 공격자의 관점에서 사고하여, 공격자가 X 데이터나 권한이 필요한 기능에 무단으로 접근하려는 시도를 차단하도록 설계된 시스템 테스트를 구성하세요.

<div id="securing-api-keys-and-tokens">
  ### API 키와 토큰 보호
</div>

X 플랫폼의 개발자는, 사용자가 여러분의 개발자 앱을 승인했다는 전제하에, X에 저장된 본인의 데이터와 사용자 데이터를 프로그래밍 방식으로 액세스할 수 있습니다. 모든 API 요청은 개발자 앱의 키와 시크릿, 그리고 경우에 따라 권한을 부여한 사용자의 [액세스 토큰](/ko/resources/fundamentals/authentication)을 사용하여 OAuth로 [인증](/ko/resources/fundamentals/authentication)해야 합니다. 자격 증명을 안전하게 보호하는 것은 여러분의 책임입니다.

권장 모범 사례는 다음과 같습니다.

- 비밀번호/토큰 갱신 주기(로테이션)를 운영하세요.
- 필요한 경우 항상 민감한 데이터를 암호화하고, 너무 이른 단계에서 복호화하지 마세요.
- 사용자의 액세스 토큰은 암호화된 저장소에 보관하세요.
- 키와 토큰이 노출되었거나 손상되었다고 판단되면 [재생성](/ko/resources/fundamentals/authentication#regenerate-api-keys-and-tokens)하거나 [무효화](/ko/resources/fundamentals/authentication#post-oauth2-invalidate-token)하세요.

X의 OAuth로 디버깅하고 개발하는 방법에 대한 더 많은 논의는 커뮤니티 포럼의 [보안 카테고리](https://devcommunity.x.com/c/oauth/12)를 방문하세요.

<div id="input-validation">
  ### 입력 검증
</div>

사용자가 유효하고 신뢰할 수 있는 데이터를 제공한다고 가정하지 마세요. X API 요청으로 이어질 수 있는 사용자 입력 데이터는 모두 정제하세요. 애플리케이션에서 허용 가능한 입력 유형을 허용 목록에 명시하고, 목록에 없는 입력은 모두 폐기하세요.

<div id="encrypted-communication">
  ### 암호화된 통신
</div>

X는 모든 API 요청이 TLS를 통해 이루어져야 합니다. 가능하다면 자체 서버와의 통신도 암호화해야 합니다.

<div id="exposed-debugging-information">
  ### 노출된 디버깅 정보
</div>

디버깅 화면/로그를 통해 민감한 X 데이터나 자격 증명이 노출되지 않도록 하세요. 일부 웹 프레임워크는 애플리케이션이 올바르게 구성되지 않은 경우 디버깅 정보에 손쉽게 접근할 수 있게 합니다. 데스크톱 및 모바일 개발 환경에서는 디버깅 플래그나 심벌이 활성화된 빌드를 실수로 배포하기 쉽습니다. 이러한 설정에 대한 빌드 검증을 배포/빌드 프로세스에 포함하세요. 또한 리포팅을 위해 스택 트레이스나 크래시 덤프를 공유할 때는 비공개 X 사용자의 데이터가 마스킹(비식별화)되었는지 확인하세요.

<div id="unfiltered-input-unescaped-output">
  ### 필터링되지 않은 입력, 이스케이프되지 않은 출력
</div>

입력 유효성 검사를 기억하기 쉬운 방법 중 하나는 FIEO입니다: 입력은 필터링하고, 출력은 이스케이프하세요.

애플리케이션 외부에서 들어오는 모든 데이터를 필터링하세요. 여기에는 X API 데이터, 쿠키 데이터, 사용자가 제공한 폼 입력, URL 매개변수, 데이터베이스의 데이터 등이 포함됩니다. 또한 애플리케이션이 내보내는 모든 출력을 이스케이프하세요. 여기에는 데이터베이스 서버로 보내는 SQL, 사용자 브라우저로 보내는 HTML, 다른 시스템으로 전송하는 JSON 출력, 셸 프로그램으로 전달하는 명령 등이 포함됩니다.

<div id="cross-site-scripting-xss">
  ### 크로스사이트 스크립팅(XSS)
</div>

[XSS](https://owasp.org/) 공격은 대체로 웹에서 가장 흔한 보안 문제입니다. 공격자가 자신의 JavaScript 코드를 애플리케이션에 주입할 수 있다면, 악의적인 행위를 수행할 수 있습니다. 신뢰할 수 없는 사용자 제공 데이터를 저장하고 표시하는 모든 곳은 점검하고, 정규화(소독)하고, HTML 이스케이프 처리해야 합니다. 이를 올바르게 처리하는 것은 어렵습니다. 해커들이 [XSS 공격을 성공시키는 다양한 기법](https://www.owasp.org/index.php/Types_of_Cross-Site_Scripting)을 보유하고 있기 때문입니다. 사용 중인 프로그래밍 언어나 웹 개발 프레임워크에는 교차 사이트 스크립팅을 방어하기 위한 널리 쓰이고 검증된 메커니즘이 있을 가능성이 큽니다. 반드시 이를 활용하시기 바랍니다.

<div id="sql-injection">
  ### SQL 인젝션
</div>

애플리케이션에서 데이터베이스를 사용하는 경우 [SQL 인젝션](http://en.wikipedia.org/wiki/SQL_injection)을 반드시 인지해야 합니다. 입력을 받는 모든 지점은 공격자가 입력 필드를 벗어나 데이터베이스에 침투하려는 잠재적 표적이 됩니다. SQL 인젝션을 체계적으로 방어하는 데이터베이스 라이브러리를 사용하세요. 그 방식을 벗어나 커스텀 SQL을 작성해야 한다면, 이 유형의 공격에 노출되지 않도록 강력한 테스트를 마련하세요. SQL 인젝션을 방어하는 두 가지 주요 방법은 SQL 문을 구성하기 전에 이스케이프 처리를 하는 것과 매개변수화된 입력을 사용해 문을 생성하는 것입니다. 후자가 권장되며, 개발자 실수에 덜 취약합니다.

<div id="cross-site-request-forgery-csrf">
  ### 교차 사이트 요청 위조(CSRF)
</div>

애플리케이션으로 전송되는 요청이 정말로 귀하의 애플리케이션에서 비롯된 것인지 확신하십니까? [CSRF](http://en.wikipedia.org/wiki/Cross-site_request_forgery) 공격은 로그인한 사용자가 모르게 동작을 수행하는 URL을 열도록 강제함으로써 이러한 검증 부재를 악용합니다. 개발자 앱의 경우, 이는 공격자가 귀하의 앱을 통해 사용자가 원치 않는 트윗을 게시하게 하거나 스팸 계정을 팔로우하게 만들 수 있음을 뜻합니다.

CSRF에 가장 철저히 대응하는 방법은 신뢰할 수 있는 위치에 저장된 난수 토큰을 모든 폼에 포함하는 것입니다. 폼에 올바른 토큰이 없다면 오류를 발생시키십시오. 최신 웹 프레임워크는 이를 체계적으로 처리하는 방법을 제공하며, 운이 좋다면 기본적으로 활성화되어 있을 수도 있습니다. 간단한 예방 조치(하지만 결코 유일하게 취해야 할 조치는 아님)는 데이터를 생성, 수정 또는 삭제하는 모든 작업에 대해 POST 요청을 요구하도록 하는 것입니다.

<div id="lack-of-rate-limiting">
  ### 레이트 리밋 부족
</div>

스패머와 공격자의 활동을 지연시키기 위해 적절한 경우 CAPTCHA를 사용하세요.
