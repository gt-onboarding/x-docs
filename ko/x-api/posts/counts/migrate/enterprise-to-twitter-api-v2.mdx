<div id="enterprise-compared-to-x-api-v2">
  ### Enterprise와 X API v2 비교
</div>

**유사점**

- 세분화 수준
- 페이지네이션
- 표준시대

**차이점**

- 엔드포인트 URL
- 앱 및 프로젝트 요건
- 사용 가능한 기간
- 응답 데이터 형식
- HTTP 메서드
- 요청 시간 형식
- 요청 매개변수
- 필터링 연산자

<div id="similarities">
  #### 유사점
</div>

**세분화 수준**

반환 데이터의 세분화 수준을 선택하는 매개변수 이름은 다르지만(enterprise 버전은 `bucket`, v2 버전은 `granularity`), 해당 매개변수에 전달할 수 있는 값과 기본 동작은 동일합니다:

- `day`
- `hour` (기본값)
- `minute`

**페이지네이션**

v2에는 추가 페이지네이션 기능이 있습니다(`since_id` 및 `until_id`를 사용해 게시물 ID 기반으로 탐색할 수 있는 새로운 페이지네이션 매개변수). 하지만 enterprise와 v2 모두 시간 기준으로 페이지네이션할 수 있습니다(enterprise에서는 `fromDate` 및 `toDate`, v2에서는 `start_time` 및 `end_time`).

enterprise 버전을 사용하는 경우 페이지네이션에는 `next` 매개변수를 사용하며, 다음 토큰 필드 이름은 `next`이고 응답의 루트에 위치합니다.

v2를 사용하는 경우 페이지네이션에는 `next_token` 또는 `pagination_token` 매개변수 중 하나를 사용할 수 있으며, 다음 토큰은 응답의 `meta.next_token`에 위치합니다.
 

**시간대**

페이지네이션 섹션에서 언급했듯이 enterprise와 v2 모두 시간 기준으로 서로 다른 데이터 페이지를 탐색할 수 있습니다. 두 경우 모두 이러한 매개변수를 사용할 때 시간대는 UTC를 사용합니다.

<div id="differences">
  #### 차이점
</div>

**엔드포인트 URL**

- Enterprise 엔드포인트:
  - 30일 - `http://gnip-api.x.com/search/30day/accounts/:account_name/:label/counts.json`
  - 전체 아카이브 - `http://gnip-api.x.com/search/fullarchive/accounts/:account_name/:label/counts.json`
- X API v2 엔드포인트
  - 최근(7일) - `https://api.x.com/2/tweets/counts/recent`
  - 전체 아카이브 - `https://api.x.com/2/tweets/counts/all`

**App 및 Project 요구 사항**

X API v2 엔드포인트는 요청 인증 시 [developer App](/ko/resources/fundamentals/developer-apps)의 자격 증명을 사용해야 하며, 해당 App은 [Project](/ko/resources/fundamentals/developer-apps)와 연결되어 있어야 합니다. 모든 X API v1.1 엔드포인트는 독립형 App 또는 Project와 연결된 App의 자격 증명을 사용할 수 있습니다.
 

**사용 가능한 기간**

Enterprise API와 X API v2 모두 전체 아카이브의 게시물에 대한 게시물 볼륨 데이터를 검색할 수 있는 엔드포인트를 제공합니다.

다만 X API v2에는 Enterprise API와 같은 30일 기간 엔드포인트가 없습니다. 대신 앞서 언급한 전체 아카이브 또는 7일 기간을 제공하며, 이는 [v2 Search Posts 엔드포인트](/ko/x-api/posts/search/introduction)와 일치합니다.
 

**응답 데이터 형식**

Enterprise와 X API v2를 통해 수신하는 데이터 형식에는 약간의 차이가 있습니다:

- Enterprise의 카운트 데이터는 `results` 객체에, v2의 카운트 데이터는 `data` 객체에 위치합니다.
- Enterprise의 카운트 필드는 `timePeriod`(시작 시간)와 `count`이며, v2는 기간을 `start`와 `end` 필드로 분리합니다(Enterprise와는 다른 날짜/시간 형식을 사용하며, 요청 시간 형식에서 설명됨). 또한 카운트 필드 이름을 `tweet_count`로 변경했습니다.
- Enterprise 메타데이터에는 루트 레벨에 `totalCount`, `next`, `requestParameters` 객체가 포함됩니다. 반면 v2는 `requestParameters` 객체를 포함하지 않으며, 다음 항목을 루트 레벨의 `meta` 객체로 이동/이름 변경합니다: `total_tweet_count`, `next_token`.
   

**HTTP 메서드**

이 API의 Enterprise 버전은 JSON 본문을 포함한 POST HTTP 메서드 또는 쿼리 문자열을 포함한 GET HTTP 메서드로 요청을 보낼 수 있습니다.

v2는 쿼리 문자열이 있는 GET HTTP 메서드만 사용할 수 있습니다.
 

**요청 시간 형식**

이 엔드포인트의 Enterprise 버전은 페이지네이션 매개변수와 `timePeriod` 응답 필드 모두에서 다음 날짜/시간 형식을 사용합니다: `YYYYMMDDHHmm`

v2 엔드포인트는 페이지네이션 매개변수와 `start` 및 `end` 응답 필드 모두에서 ISO 8601/RFC 3339 날짜/시간 형식을 사용합니다: `YYYY-MM-DDTHH:mm:ssZ`

**요청 매개변수**

다음은 Enterprise와 X API v2의 요청 매개변수 표입니다:

| Enterprise | Search Posts v2 |
| :--- | :--- |
| query | query |
| bucket | granularity |
| fromDate (YYMMDDHHmm) | start\_time (YYYY-MM-DDTHH:mm:ssZ) |
| toDate (YYMMDDHHmm) | end\_time (YYYY-MM-DDTHH:mm:ssZ) |
|     | since\_id |
|     | until\_id |
| next | next\_token 및 pagination\_token |

**필터링 연산자**

Enterprise와 X API v2의 연산자는 대부분 동일하지만, 연산자 가용성에 일부 차이가 있으며 X API v2에만 도입된 새로운 연산자도 있습니다.

X API v2, Enterprise, 그리고 Premium에서 사용 가능한 연산자의 전체 표를 보려면 [Post counts migration landing page](/ko/x-api/posts/counts/migrate/overview)를 방문하세요.
