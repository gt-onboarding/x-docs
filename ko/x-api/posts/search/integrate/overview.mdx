<div id="postman">
  #### Postman
</div>

Postman은 이러한 엔드포인트를 테스트하는 데 유용한 도구입니다. 각 Postman 요청에는 해당 엔드포인트의 모든 매개변수가 포함되어 있어, 사용할 수 있는 옵션을 빠르게 파악하는 데 도움이 됩니다. Postman 컬렉션에 대해 더 알아보려면 [Using Postman](/ko/tutorials/postman-getting-started) 페이지를 방문하세요.

<div id="authentication">
  #### 인증
</div>

모든 X API v2 엔드포인트는 키와 토큰으로도 알려진 자격 증명 세트를 사용해 요청을 인증해야 합니다. 최근 검색 엔드포인트에 대한 요청 인증에는 OAuth 1.0a 사용자 컨텍스트, OAuth 2.0 앱 전용, 또는 OAuth 2.0 Authorization Code with PKCE 중 하나를 사용할 수 있습니다. 전체 아카이브 검색 엔드포인트를 사용할 때는 반드시 OAuth 2.0 앱 전용을 사용해야 합니다.

[OAuth 2.0 앱 전용](/ko/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3)은 요청에 [OAuth 2.0 App Access Token](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)만 함께 전달하면 됩니다. 개발자 앱 내에서 직접 App Access Token을 생성하거나 [POST oauth2/token](/ko/resources/fundamentals/authentication#post-oauth2-token) 엔드포인트로 생성할 수 있습니다.

[OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)는 API 키, 사용자 액세스 토큰, 그리고 몇 가지 추가 매개변수를 사용해 [authorization 헤더를 생성](https://developer-staging.x.com/resources/fundamentals/authentication/authorizing-a-request)하고 이를 요청에 포함해야 합니다. 액세스 토큰은 요청을 수행하는 당사자인 해당 사용자와 연관되어 있어야 합니다. 다른 사용자의 액세스 토큰 세트를 생성하려면, 해당 사용자가 [3-legged OAuth 플로우](https://developer-staging.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)를 통해 귀하의 앱을 승인해야 합니다. 

OAuth 1.0a는 사용하기 어려울 수 있습니다. 이 인증 방식에 익숙하지 않다면 [라이브러리](/ko/x-api/tools-and-libraries/overview)를 사용하거나 Postman과 같은 도구를 사용하거나, OAuth 2.0으로 요청을 인증할 것을 권장합니다. 이러한 엔드포인트에서 포스트 또는 비공개 지표를 요청하려면 OAuth 1.0a 사용자 컨텍스트 또는 OAuth 2.0 Authorization Code with PKCE를 사용해야 하며, 이를 통해 해당 콘텐츠 소유 사용자로부터 적절한 권한을 보유하게 됩니다.

[OAuth 2.0 Authorization Code with PKCE](/ko/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2)는 애플리케이션 범위에 대한 더 세밀한 제어와, 여러 디바이스에 걸친 인증 플로우를 제공합니다. OAuth 2.0은 사용자 대신 특정 권한을 부여하는 세분화된 스코프를 선택할 수 있게 합니다. 

앱에서 OAuth 2.0을 사용하려면 개발자 포털의 앱 설정 섹션에 있는 앱 인증 설정에서 이를 활성화해야 합니다.

<Note>
  **유의 사항**

  다음 [fields](/ko/x-api/fundamentals/fields)를 요청하는 경우, OAuth 1.0a 사용자 컨텍스트 또는 OAuth 2.0 Authorization Code가 필요합니다: 

  - tweet.fields.non\_public\_metrics
  - tweet.fields.promoted\_metrics
  - tweet.fields.organic\_metrics
  - media.fields.non\_public\_metrics
  - media.fields.promoted\_metrics
  - media.fields.organic\_metrics
</Note>

####

개발자 포털, 프로젝트와 개발자 앱

X API v2의 어떤 엔드포인트를 사용하려면 [개발자 계정에 가입](https://developer.x.com/en/portal/petition/essential/basic-info)하고, 해당 계정 내에 프로젝트를 설정한 뒤, 그 프로젝트 안에 개발자 앱을 만들어야 합니다. 해당 개발자 앱의 키와 토큰은 이 검색 엔드포인트들에 사용할 수 있습니다.

프로젝트의 키와 토큰은 어떤 [액세스 수준](/ko/x-api/getting-started/about-x-api)이든 최근 검색 엔드포인트에 요청을 보내는 데 사용할 수 있습니다. 다만 전체 아카이브 검색 엔드포인트에 요청하려면 Pro 또는 Enterprise 액세스 수준의 프로젝트가 필요합니다. [Enterprise 액세스](/ko/x-api/getting-started/getting-access)가 있는 경우, 추가 연산자 사용 및 더 긴 쿼리 길이 등 추가 기능을 이용할 수 있습니다.
 

<div id="rate-limits">
  #### 레이트 리밋
</div>

매일 수많은 개발자가 X API에 요청을 보냅니다. 트래픽을 관리하기 위해 각 엔드포인트에는 [레이트 리밋](/ko/x-api/fundamentals/rate-limits)이 적용되어, 앱 또는 인증된 사용자를 대신해 각 개발자가 보낼 수 있는 요청 수를 제한합니다.

사용 중인 인증 방식에 따라 엔드포인트별 레이트 리밋이 달라집니다. 앱 수준 레이트 리밋은 OAuth 2.0 App-Only로 요청을 보내는 앱에 적용되며, 사용자 수준 레이트 리밋은 OAuth 1.0a User Context 또는 OAuth 2.0 Authorization Code with PKCE를 사용해 특정 승인 사용자를 대신해 이루어지는 요청에 적용됩니다. 이 두 레이트 리밋은 15분 단위 윈도우 내 요청 빈도를 기준으로 합니다.

예를 들어, 최근 검색 엔드포인트에 요청하기 위해 OAuth 2.0 App-Only 인증을 사용하는 앱은 15분 동안 450회의 요청(페이지네이션 요청 포함)을 보낼 수 있습니다. 동일한 앱은 같은 15분 동안 서로 다른 두 명의 인증된 사용자(OAuth 1.0a User Context 또는 OAuth 2.0 Authorization Code with PKCE 사용)를 통해 각 사용자마다 최근 검색 엔드포인트에 최대 180회의 요청(페이지네이션 요청 포함)을 보낼 수 있습니다.
 

<div id="fields-and-expansions">
  #### 필드와 확장
</div>

X API v2에서는 [필드](/ko/x-api/fundamentals/fields)와 [확장](/ko/x-api/fundamentals/expansions)을 사용해 API에서 반환받을 데이터를 정확히 선택할 수 있습니다. 확장 매개변수는 페이로드에서 참조되는 객체를 펼쳐 포함하도록 합니다. 예를 들어, 이 엔드포인트는 확장 매개변수를 통해 투표, 장소, 미디어 등 관련 객체를 함께 요청할 수 있습니다.

필드 매개변수는 각 데이터 객체 내에서 수신할 필드를 정확히 지정할 수 있게 해줍니다. 기본적으로 이들 엔드포인트가 반환하는 기본 게시물(Post) 객체에는 id와 text 필드가 포함됩니다(해당 기능 출시 이후 생성된 게시물의 경우 edit\_history\_tweet\_ids도 포함). author\_id나 public\_metrics 같은 추가 필드를 받으려면 필드 매개변수로 해당 필드를 명시적으로 요청해야 합니다. 통합 시 활용을 권장하는 중요한 필드로는 투표 데이터, [지표](/ko/x-api/fundamentals/metrics), [게시물 주석](/ko/x-api/fundamentals/post-annotations), [대화 ID](/ko/x-api/fundamentals/conversation-id) 등이 있습니다.

[X API v2 데이터 사전](/ko/x-api/fundamentals/data-dictionary)에 [필드와 확장을 함께 사용하는 방법](/ko/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions)에 대한 가이드를 추가했습니다.
 

<div id="metrics">
  #### 지표
</div>

X API v2 엔드포인트에서는 요청에 적절한 [fields](/ko/x-api/fundamentals/fields)를 전달하면 반환된 객체에서 지표를 직접 조회할 수 있습니다.

사용자 프라이버시와 아래 응답 필드와 관련하여 게시물 지표에는 알아두어야 할 몇 가지 제한이 있습니다:

- tweet.fields.non\_public\_metrics
- tweet.fields.promoted\_metrics
- tweet.fields.organic\_metrics
- media.fields.non\_public\_metrics
- media.fields.promoted\_metrics
- media.fields.organic\_metrics

위 필드에는 비공개 지표가 포함되므로, 최근 검색 엔드포인트를 사용할 때 게시물 게시자를 대신해 이 데이터를 가져오려면 해당 게시자의 승인이 필요하며, 즉 [OAuth 1.0a User Context](/ko/resources/fundamentals/authentication)를 사용해야 합니다. 이 인증 방식은 최근 검색에서만 사용할 수 있으므로 전체 아카이브 검색 엔드포인트를 통해서는 이러한 지표를 조회할 수 없습니다.

예를 들어 사용자 ID 1234의 게시물에 대한 `non_public_metrics`를 받으려면 해당 사용자와 연결된 액세스 토큰을 요청에 포함해야 합니다. [3-legged OAuth flow](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)를 사용하면 사용자가 앱을 승인하도록 하고, 그 사용자와 연결된 액세스 토큰 세트를 발급받을 수 있습니다.

`non_public_metrics`, `organic_metrics`, 그리고 `promoted_metrics`는 최근 30일 이내에 생성된 게시물에만 제공됩니다. 따라서 위 필드를 요청하면 결과는 자동으로 최근 30일의 게시물만 포함하도록 조정됩니다.

위 필드를 요청하는 경우 인증된 사용자가 작성한 게시물만 반환되며, 다른 모든 게시물에 대해서는 오류가 반환됩니다.
 

<div id="building-search-queries">
  #### 검색 쿼리 빌드
</div>

이 엔드포인트의 핵심 기능은 단일 검색 쿼리를 사용해 전달받는 게시물을 필터링하는 것입니다. 쿼리는 메시지 키워드, 해시태그, URL 등 게시물 및 사용자 속성과 일치시키는 연산자들로 구성됩니다. 연산자는 불리언 논리와 괄호를 사용해 결합하여 쿼리의 매칭 동작을 정교하게 조정할 수 있습니다.

[쿼리 빌드 방법](/ko/x-api/posts/search/integrate/build-a-query) 가이드를 참고해 자세히 알아보세요.

<div id="pagination">
  #### 페이지네이션
</div>

이 엔드포인트는 응답을 신속하게 반환하기 위해 페이지네이션을 사용합니다. 단일 응답으로 보낼 수 있는 결과 수를 초과하는 경우(최근 검색은 최대 100개, 전체 아카이브 검색은 최대 500개) 페이지네이션이 필요합니다. 페이지당 반환할 결과 수를 지정하려면 max\_results 매개변수를, 다음 페이지의 결과를 받으려면 pagination\_token 매개변수를 사용하세요. 자세한 내용은 [페이지네이션 가이드](/ko/x-api/fundamentals/pagination)를 참고하세요.

<div id="post-caps">
  #### 게시 한도
</div>

Search Posts 엔드포인트는 페이지네이션과 무관하게 해당 월에 반환할 수 있는 게시물 수에 제한이 있습니다.

어떤 검색 엔드포인트를 사용하더라도 반환된 게시물은 프로젝트 수준의 [트윗 한도(게시 한도)](/ko/x-api/fundamentals/post-cap)에 산정됩니다. 사용량은 개발자 포털에 표시되며, ‘월’의 시작일은 [developer portal dashboard](https://developer.x.com/en/portal/dashboard)에 표시된 구독 갱신일입니다.

**게시물 편집**

편집이 가능한 게시물은 최초 게시 후 30분 동안 최대 다섯 번까지 편집할 수 있습니다. 검색 엔드포인트는 항상 해당 게시물의 최신 버전을 제공합니다. 게시된 지 30분 이상 지난 게시물만 요청하면 항상 최종 버전을 받게 됩니다. 다만, 거의 실시간에 가까운 사용 사례로 최근 30분 이내에 게시된 게시물을 조회하는 경우, 수신 이후에 게시물이 편집되었을 수 있습니다. 이러한 게시물은 검색 또는 Post Lookup 엔드포인트로 재수화(rehydrate)하여 최종 상태를 확인할 수 있습니다. 게시물 편집 작동 방식에 대해 자세히 알아보려면 [Post edits fundamentals](/ko/x-api/fundamentals/edit-posts) 페이지를 참조하세요.
