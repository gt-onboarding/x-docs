몇 가지 핵심 개념을 살펴보기 전에, 아래 도구나 코드 샘플 중 하나를 사용해 해당 엔드포인트의 기능을 테스트해 보시기 바랍니다.

**코드 샘플**

선호하는 프로그래밍 언어로 이 엔드포인트를 설정하고 싶으신가요? 시작점으로 활용할 수 있는 다양한 코드 샘플을 [GitHub 페이지](https://github.com/xdevplatform/Twitter-API-v2-sample-code)에서 제공합니다.

**라이브러리**

시작에 도움이 되도록 다양한 [커뮤니티 서드파티 라이브러리](/ko/x-api/tools-and-libraries/overview) 중 하나를 활용해 보세요. 올바른 버전 태그를 확인해 v2 엔드포인트와 호환되는 라이브러리를 찾을 수 있습니다.

**Postman**

Postman은 이 엔드포인트를 테스트하는 데 매우 유용한 도구입니다. 각 Postman 요청에는 사용 가능한 항목을 빠르게 파악할 수 있도록 모든 경로 및 본문 파라미터가 포함되어 있습니다. Postman 컬렉션에 대한 자세한 내용은 [Using Postman](/ko/tutorials/postman-getting-started) 페이지를 방문하세요.

<div id="key-concepts">
  ### 핵심 개념
</div>

**인증**

모든 X API v2 엔드포인트는 키와 토큰으로도 불리는 자격 증명 세트로 [인증](/ko/resources/fundamentals/authentication)된 요청이 필요합니다. 이러한 엔드포인트에 대한 요청 인증에는 OAuth 1.0a User Context 또는 OAuth 2.0 Authorization Code with PKCE를 사용할 수 있습니다. 사용자 게시물 타임라인과 사용자 멘션 타임라인에는 OAuth 2.0 App-Only를 사용할 수 있습니다.

[OAuth 1.0a User Context](/ko/resources/fundamentals/authentication#oauth-1-0a-2)는 요청에 함께 보낼 [authorization 헤더 생성](https://developer-staging.x.com/resources/fundamentals/authentication/authorizing-a-request)을 위해 API Key, 사용자 Access Token 및 몇 가지 추가 매개변수를 사용해야 합니다. Access Token은 대리 요청 대상 사용자의 계정과 연결되어 있어야 합니다. 다른 사용자의 Access Token 세트를 생성하려면 해당 사용자가 [3-legged OAuth 흐름](https://developer-staging.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)을 통해 귀하의 App을 승인해야 합니다. 

OAuth 1.0a는 사용하기 까다로울 수 있습니다. 이 인증 방식에 익숙하지 않다면 [라이브러리](/ko/x-api/tools-and-libraries/overview)나 Postman 같은 도구를 사용하거나, OAuth 2.0으로 요청을 인증할 것을 권장합니다. 이러한 엔드포인트에서 게시물 또는 비공개 지표를 요청하려면 OAuth 1.0a User Context 또는 OAuth 2.0 Authorization Code with PKCE 중 하나를 사용해야 하며, 이를 통해 해당 콘텐츠 소유 사용자로부터 적절한 권한을 확보할 수 있습니다.

[OAuth 2.0 App-Only](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)는 요청에 [OAuth 2.0 App Access Token](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)만 포함하면 됩니다. 개발자 App 내에서 직접 App Access Token을 생성하거나 [POST oauth2/token](/ko/resources/fundamentals/authentication#post-oauth2-token) 엔드포인트로 생성할 수 있습니다. 이는 사용자 게시물 타임라인 또는 사용자 멘션 타임라인에 사용할 수 있습니다.

[OAuth 2.0 Authorization Code with PKCE](/ko/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3)는 애플리케이션의 스코프에 대한 더 세밀한 제어와 다중 기기 간 인증 플로우를 제공합니다. OAuth 2.0을 사용하면 사용자 대리 권한에 대해 세밀한 스코프를 선택할 수 있습니다. 

App에서 OAuth 2.0을 사용하려면 개발자 포털의 App 설정 섹션에 있는 인증 설정에서 이를 활성화해야 합니다.

<Info>

  다음 [fields](/ko/x-api/fundamentals/fields)를 요청하는 경우 OAuth 1.0a User Context 또는 OAuth 2.0 Authorization Code가 필요합니다: 

  - `tweet.fields.non_public_metrics`
  - `tweet.fields.promoted_metrics`
  - `tweet.fields.organic_metrics`
  - `media.fields.non_public_metrics`
  - `media.fields.promoted_metrics`
  - `media.fields.organic_metrics`
</Info>

**[](/ko/x-api/fundamentals/pagination)개발자 포털, 프로젝트와 앱, 개발자 앱**

X API v2 엔드포인트를 사용하려면 [개발자 계정에 가입](https://developer.x.com/en/portal/petition/essential/basic-info)하고 해당 계정 내에 [프로젝트](/ko/resources/fundamentals/projects)를 설정한 뒤, 그 프로젝트 내에 [개발자 앱](/ko/resources/fundamentals/developer-apps)을 만들어야 합니다. 해당 개발자 앱의 키와 토큰은 이 타임라인 엔드포인트에서 동작합니다.

**요청 한도**

매일 수많은 개발자가 X API에 요청을 보냅니다. 트래픽 관리를 위해 각 엔드포인트에는 [요청 한도](/ko/x-api/fundamentals/rate-limits)가 설정되어 있으며, 이는 모든 개발자가 앱 또는 인증된 사용자를 대신해 보낼 수 있는 요청 수를 제한합니다. 

인증 방식에 따라 이들 엔드포인트에 적용되는 요청 한도가 다릅니다. 앱 수준 한도는 OAuth 2.0 App-Only로 요청하는 App에 적용되며, 사용자 수준 한도는 OAuth 1.0a User Context로 특정 승인 사용자를 대신해 수행되는 요청에 적용됩니다. 두 한도 모두 15분 윈도우 내 요청 빈도를 기준으로 합니다.

예를 들어, 이 두 타임라인 엔드포인트 모두에 대해 OAuth 2.0 App-Only 인증을 사용하는 앱은 15분 내에 사용자 게시 타임라인에 1,500건(페이지네이션 요청 포함), 사용자 멘션 타임라인에 450건(페이지네이션 요청 포함)의 요청을 보낼 수 있습니다. 동일한 앱이 동일한 15분 내에 서로 다른 두 명의 인가된 사용자(OAuth 1.0a User Context 사용)를 통해서는 각 인증 사용자별로 사용자 게시 타임라인에 900건(페이지네이션 요청 포함), 사용자 멘션 타임라인에 180건(페이지네이션 요청 포함)의 요청을 보낼 수 있습니다.

역순 시간 홈 타임라인은 사용자당 15분 창에서 요청 한도가 180건입니다. 이 엔드포인트를 사용하면 지난 7일 동안 해당 타임라인에 생성된 모든 게시물과, 생성일과 관계없이 최신 800개 게시물을 반환할 수 있습니다.

**필드와 확장**

X API v2는 [fields](/ko/x-api/fundamentals/fields)와 [expansions](/ko/x-api/fundamentals/expansions)를 사용해 API에서 반환받을 데이터를 정확히 선택할 수 있게 합니다. expansions 파라미터를 사용하면 페이로드에서 참조되는 객체를 확장할 수 있습니다. 예를 들어, 이 엔드포인트에서는 expansions 파라미터로 설문, 장소, 미디어 등 다양한 객체를 요청할 수 있습니다.

fields 파라미터를 사용하면 서로 다른 데이터 객체 내에서 수신할 필드를 정확히 지정할 수 있습니다. 기본적으로 이 엔드포인트들이 반환하는 기본 게시물 객체에는 id와 text 필드가 포함됩니다. author\_id나 public\_metrics 같은 추가 필드를 받으려면 fields 파라미터로 명시적으로 요청해야 합니다. 통합 시 고려할 만한 중요한 필드로는 설문 데이터, [metrics](/ko/x-api/fundamentals/metrics), [Post annotations](/ko/x-api/fundamentals/post-annotations), [conversation ID](/ko/x-api/fundamentals/conversation-id) 등이 있습니다.

[X API v2 데이터 사전](/ko/x-api/fundamentals/data-dictionary)에 [필드와 확장을 함께 사용하는 방법](/ko/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) 가이드를 추가했습니다.

**게시 지표**

X API v2 엔드포인트는 요청에 적절한 필드를 포함하는 경우 반환된 게시물 객체에서 직접 게시 지표를 요청할 수 있습니다.

게시 지표에는 사용자 프라이버시와 다음 응답 필드와 관련된 몇 가지 제한 사항이 있습니다:

- `tweet.fields.non_public_metrics`
- `tweet.fields.promoted_metrics`
- `tweet.fields.organic_metrics`
- `media.fields.non_public_metrics`
- `media.fields.promoted_metrics`
- `media.fields.organic_metrics`

위 필드에는 비공개 지표 데이터가 포함되므로, 사용자 게시 타임라인 엔드포인트를 사용할 때 게시물 게시자를 대신해 이 데이터를 조회하려면 해당 게시자의 인가가 필요하며, [OAuth 1.0a User Context](/ko/resources/fundamentals/authentication) 또는 [OAuth 2.0 Authorization Code Flow with PKCE](/ko/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3)를 사용해야 합니다.

예를 들어, 사용자 ID 1234의 사용자 게시 타임라인에서 `non_public_metrics`를 받으려면 요청에 그 사용자와 연관된 액세스 토큰을 포함해야 합니다. [3-legged OAuth flow](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)를 사용하면 사용자가 앱을 인가하고 해당 사용자와 연관된 액세스 토큰 세트를 발급받을 수 있습니다.

사용자 멘션 타임라인을 사용하는 경우, 언급된 작성자가 앱에 자신의 비공개 지표 데이터 접근을 인가하고 OAuth 1.0a User Context로 요청 시 그 사용자의 액세스 토큰을 사용하는 것이 아니라면, 위 필드를 사용할 수 없습니다.

모든 `non_public_metrics`, organic\_metrics, promoted\_metrics는 최근 30일 내에 생성된 게시물에만 제공됩니다. 즉, 위 필드를 요청하면 결과는 자동으로 최근 30일의 게시물만 포함하도록 제한됩니다.

위 필드를 요청한 경우, 인증된 사용자가 작성한 게시물만 반환되며 다른 모든 게시물은 오류가 반환됩니다.

**페이지네이션**

이 엔드포인트들은 응답을 빠르게 제공하기 위해 페이지네이션을 사용합니다. 단일 응답으로 보낼 수 있는 양을 초과하는 결과가 있는 경우(타임라인 엔드포인트는 최대 100개 게시물) 페이지네이션이 필요합니다. 페이지당 반환할 결과 수를 지정하려면 max\_results 파라미터를, 다음 페이지 결과를 받으려면 pagination\_token 파라미터를 사용하세요. 자세한 내용은 [페이지네이션 가이드](/ko/x-api/fundamentals/pagination)를 참고하세요.

**결과 필터링**

이 엔드포인트에는 결과를 필터링하는 데 사용할 수 있는 여러 매개변수가 포함되어 있습니다. start\_date와 end\_date를 사용하면 결과를 특정 기간으로 좁힐 수 있습니다. 특정 게시물 집합을 선택하려면 게시물 ID 대신 since\_id와 until\_id를 사용할 수 있습니다. 사용자 게시물 타임라인에는 결과에서 리트윗과 답글을 제외할 수 있는 exclude 매개변수도 있습니다. 

**트윗 한도(게시 한도)와 반환되는 게시물 수**

사용자 게시물 타임라인과 사용자 멘션 타임라인 엔드포인트는 월별로 반환할 수 있는 게시물 수에 제한이 있습니다. 역순 연대기 홈 타임라인 엔드포인트는 이 제한의 적용을 받지 않습니다.

어떤 타임라인 엔드포인트를 사용하든 반환된 게시물은 프로젝트 수준의 [트윗 한도(게시 한도)](/ko/x-api/fundamentals/post-cap)에 포함됩니다. 사용량은 개발자 포털에 표시되며, ‘월’은 [개발자 포털 대시보드](https://developer.x.com/en/portal/dashboard)에 표시된 구독 갱신일에 시작됩니다. 

사용자 게시물 타임라인 엔드포인트는 사용자의 타임라인에 게시된 가장 최근 3,200개의 게시물만 반환합니다. start\_time과 end\_time을 가장 최근 3,200개를 넘어서는 게시물이 포함된 기간으로 설정하면 성공적으로 응답은 받지만 게시물은 반환되지 않습니다.

또한 사용자 게시물 타임라인 요청에 excludes=replies를 전달하는 경우, 가장 최근 800개의 게시물만 반환된다는 점에 유의하세요.

사용자 멘션 타임라인 엔드포인트는 가장 최근 800개의 게시물 멘션만 반환합니다.

역순 연대기 홈 타임라인 엔드포인트는 가장 최근 3,200개의 게시물을 반환합니다.

**게시물 편집**

편집 가능한 게시물은 원본 게시물이 게시된 후 30분 동안 최대 5회까지 편집할 수 있습니다. 검색 엔드포인트는 항상 해당 게시물의 최신 버전을 제공합니다. 게시된 지 30분 이상 지난 게시물만 요청하면 항상 최종 버전을 받게 됩니다. 그러나 거의 실시간 사용 사례에서 지난 30분 내에 게시된 게시물을 조회하는 경우, 해당 게시물이 수신 이후에 편집되었을 수 있습니다. 이러한 게시물은 검색 또는 Post 조회 엔드포인트로 리하이드레이션하여 최종 상태를 확인할 수 있습니다. 게시물 편집 동작에 관한 자세한 내용은 [게시물 편집 기본 사항](/ko/x-api/fundamentals/edit-posts) 페이지를 참조하세요.  

**엣지 케이스**

- 사용자 게시물 타임라인 엔드포인트에서 30일이 지난 게시물에 대해 비공개 지표를 요청하는 경우, 결과 수가 0인 next\_token이 응답에 표시될 수 있습니다. 이 문제를 피하려면 non\_public\_metrics 매개변수로 요청한 기간이 최근 30일 이내인지 확인하세요. 또한 max\_results의 최소값은 10이어야 합니다. 이는 이 상황을 피하는 데 도움이 될 수 있지만, 여전히 발생할 수 있습니다.

- 프로모션되지 않은 게시물에 대해 프로모션 지표를 요청하면 게시물 데이터 대신 빈 응답이 반환됩니다. 당사 팀은 현재 이 문제를 수정 중입니다.

- 본문 길이가 140자를 초과하는 게시물을 포함한 리트윗의 경우, 전체 게시물 텍스트를 반환하는 대신 text 필드가 잘립니다. 단기적인 해결책은 참조된 게시물을 확장하고 그 확장에서 전체 텍스트를 가져오는 것입니다. 이는 향후 수정할 버그입니다.
