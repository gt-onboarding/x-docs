<div id="retweets-lookup-standard-v11-compared-to-x-api-v2">
  ### 리트윗 조회: Standard v1.1과 X API v2 비교
</div>

표준 v1.1 [v1.1 GET statuses/retweets/:id](https://developer.x.com/en/docs/twitter-api/v1/tweets/post-and-engage/api-reference/get-statuses-retweets-id), [v1.1 GET statuses/retweets/:ids](https://developer.x.com/en/docs/twitter-api/v1/tweets/post-and-engage/api-reference/get-statuses-retweeters-ids)을(를) 사용해 오셨다면, 이 가이드는 표준 v1.1과 X API v2 리트윗 조회 엔드포인트 간의 공통점과 차이점을 이해하는 데 도움을 드리기 위한 것입니다.

- **유사점**
  - 인증
  - 요청당 사용자 수 제한
- **차이점**
  - 엔드포인트 URL
  - 요청 제한
  - 앱 및 프로젝트 요구 사항
  - 응답 데이터 형식
  - 요청 파라미터

<div id="similarities">
  #### 유사점
</div>

**인증**

표준 v1.1과 X API v2의 Retweets 조회 엔드포인트([v1.1 GET statuses/retweets/:id](https://developer.x.com/en/docs/twitter-api/v1/tweets/post-and-engage/api-reference/get-statuses-retweets-id) 및 [v1.1 GET statuses/retweeters/:ids](https://developer.x.com/en/docs/twitter-api/v1/tweets/post-and-engage/api-reference/get-statuses-retweeters-ids))는 모두 [OAuth 1.0a 사용자 컨텍스트](https://developer.x.com/content/developer-twitter/resources/fundamentals/authentication) 또는 OAuth 2.0 베어러 토큰을 사용합니다.

**요청당 사용자 제한**

v1.1과 v2의 GET 엔드포인트 모두에서 Retweets 조회 엔드포인트가 반환하는 최대 사용자 수는 100명입니다. v2 Retweets 조회 엔드포인트의 경우 페이지네이션 토큰이 전달되지 않으며, 기본적으로 100명의 사용자만 제공되며 이 값이 반환 가능한 최대치입니다.

<div id="differences">
  #### 차이점
</div>

**엔드포인트 URL**

- 표준 v1.1 엔드포인트:
  - https://api.x.com/1.1/statuses/retweets/:id.json
    (`id` 매개변수로 지정한 게시물의 가장 최근 리트윗 100개 모음을 반환)
  - `https://api.x.com/1.1/statuses/retweeters/ids.json`
    (`id` 매개변수로 지정한 게시물을 리트윗한 사용자의 사용자 ID 최대 100개 모음을 반환)
- X API v2 엔드포인트:
  - https://api.x.com/2/tweets/:id/retweeted\_by
    (특정 게시물을 리트윗한 계정 목록을 반환)

**앱 및 프로젝트 요구 사항**

X API v2 엔드포인트는 요청을 인증할 때 [developer App](/ko/resources/fundamentals/developer-apps)의 자격 증명을 사용해야 하며, 해당 앱은 [Project](/ko/resources/fundamentals/developer-apps)와 연결되어 있어야 합니다. 모든 X API v1.1 엔드포인트는 독립 실행형 앱 또는 프로젝트와 연결된 앱의 자격 증명을 사용할 수 있습니다.

**응답 데이터 형식**

표준 v1.1과 X API v2 엔드포인트 버전 간의 가장 큰 차이점 중 하나는 페이로드에 어떤 필드를 반환할지 선택하는 방식입니다.

표준 엔드포인트의 경우 기본적으로 많은 응답 필드를 받으며, 매개변수를 사용해 페이로드에 포함할 필드 또는 필드 집합을 지정할 수 있습니다.

X API v2 버전은 기본적으로 사용자 id, name, username 필드만 제공합니다. 추가 필드나 객체를 요청하려면 [fields](/ko/x-api/fundamentals/fields) 및 [expansions](/ko/x-api/fundamentals/expansions) 매개변수를 사용해야 합니다. 이 엔드포인트에서 요청한 사용자 필드는 기본 사용자 객체로 반환됩니다. 확장된 게시물 객체와 필드는 응답의 includes 객체 내에 반환됩니다. 그런 다음 사용자 객체와 확장된 게시물 객체에 있는 ID를 매칭하여 확장된 객체를 사용자 객체와 연결할 수 있습니다.

이 새로운 매개변수에 대해서는 각 가이드 또는 [필드와 확장을 사용하는 방법](/ko/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) 가이드에서 자세히 확인하시기 바랍니다.

또한 표준 v1.1 필드를 최신 v2 필드에 매핑하는 데 도움이 되는 [데이터 형식 마이그레이션 가이드](/ko/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2)도 제공합니다. 이 가이드는 특정 필드를 반환하기 위해 v2 요청에 함께 전달해야 하는 구체적인 확장 및 필드 매개변수도 제공합니다.

특정 필드 요청 방식의 변경 외에도 X API v2는 [Post](/ko/x-api/fundamentals/data-dictionary#tweet) 및 [user](/ko/x-api/fundamentals/data-dictionary#user) 객체 등 API가 반환하는 객체에 대해 새로운 JSON 설계를 도입합니다.

- JSON 루트 레벨에서 표준 엔드포인트는 게시물 객체를 statuses 배열로 반환하는 반면, X API v2는 data 배열을 반환합니다.
- 리트윗 및 인용을 “statuses”로 지칭하는 대신, X API v2 JSON은 리트윗 및 인용 트윗을 기준으로 합니다. contributors 및 user.translator\_type 같은 많은 레거시 및 사용 중단 필드는 제거됩니다.
- 게시물 객체의 favorites와 사용자 객체의 favourites를 모두 사용하는 대신, X API v2는 like라는 용어를 사용합니다.
- X는 값이 없는 JSON 값(예: null)은 페이로드에 기록하지 않는 규칙을 채택하고 있습니다. 게시물 및 사용자 속성은 null이 아닌 값이 있을 때만 포함됩니다.

또한 [Post 객체](/ko/x-api/fundamentals/data-dictionary#tweet)에 다음을 포함한 새로운 필드 세트를 도입했습니다:

- [conversation\_id](/ko/x-api/fundamentals/conversation-id) 필드
- context 및 entities를 포함하는 두 개의 새로운 [annotations](/ko/x-api/fundamentals/post-annotations) 필드
- 여러 새로운 [metrics](/ko/x-api/fundamentals/metrics) 필드
- 특정 게시물에 누가 답글을 달 수 있는지 보여주는 새로운 reply\_setting 필드

**요청 매개변수**

다음 표준 v1.1 요청 매개변수는 두 가지 쿼리 매개변수(user\_id 또는 screen\_name)를 허용했습니다. X API v2는 숫자 사용자 ID만 허용하며, 엔드포인트 경로의 일부로 전달해야 합니다.
