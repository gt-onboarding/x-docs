<div id="standard-v11-compared-to-x-api-v2">
  ### 표준 v1.1과 X API v2 비교
</div>

표준 v1.1의 GET users/show 및 GET users/lookup을 사용해 오셨다면, 이 가이드는 표준과 X API v2의 사용자 조회 엔드포인트 간 유사점과 차이점을 이해하는 데 도움을 드리기 위한 것입니다.

- **유사점**
  - OAuth 1.0a 사용자 컨텍스트
  - 요청당 사용자 수 제한
- **차이점**
  - 엔드포인트 URL
  - 앱 및 프로젝트 요구 사항
  - 응답 데이터 포맷
  - 요청 매개변수

<div id="similarities">
  #### 유사점
</div>

**OAuth 1.0a 사용자 컨텍스트 인증 방식**

표준 엔드포인트는 [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication#oauth-1-0a-2)를 지원하고, 새 X API v2 사용자 조회 엔드포인트는 OAuth 1.0a 사용자 컨텍스트와 [앱 전용](/ko/resources/fundamentals/authentication#oauth-2-0) 인증을 모두 지원합니다. 따라서 이전에 표준 v1.1 사용자 조회 엔드포인트를 사용했다면 X API v2 버전으로 마이그레이션하더라도 동일한 인증 방식을 계속 사용할 수 있습니다. 

사용 중인 인증 라이브러리/패키지에 따라 앱 전용 인증이 가장 손쉽게 시작할 수 있는 방법일 수 있으며, 간단한 요청 헤더 설정만으로 사용할 수 있습니다. 앱 전용 액세스 토큰 생성 방법은 [이 앱 전용 가이드](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)를 참조하세요. 

**요청당 사용자 수 제한**

표준 v1.1 GET users/lookup 엔드포인트에서는 요청당 최대 100명의 사용자를 지정할 수 있습니다. 이는 GET /users 및 GET /users/by 엔드포인트에도 동일하게 적용됩니다. 100명을 모두 지정하려면 ids(GET /users) 매개변수 또는 username(GET /users/by) 매개변수를 쿼리 매개변수로 전달하고, 사용자 ID/사용자 이름을 쉼표로 구분한 목록으로 포함하면 됩니다. 
 

<div id="differences">
  #### 차이점
</div>

**엔드포인트 URL**

- 표준 v1.1 엔드포인트:
  - https://api.x.com/1.1/users/show (단일 ID 또는 사용자 이름 조회)
  - https://api.x.com/1.1/users/lookup (다중 ID 또는 사용자 이름 조회)
- X API v2 엔드포인트:
  - https://api.x.com/2/users (다중 ID 조회)
  - https://api.x.com/2/users/:id (단일 ID 조회)
  - https://api.x.com/2/users/by (다중 사용자 이름 조회)
  - https://api.x.com/2/users/by/username/:username (단일 사용자 이름 조회)

**앱 및 프로젝트 요구 사항**

X API v2 엔드포인트를 사용하려면 요청을 인증할 때 [developer App](/ko/resources/fundamentals/developer-apps)이 [Project](/ko/resources/fundamentals/projects)와 연결되어 있어야 하며, 해당 앱의 자격 증명을 사용해야 합니다. 모든 X API v1.1 엔드포인트는 독립형 앱 또는 프로젝트에 연결된 앱의 자격 증명을 사용할 수 있습니다.

**응답 데이터 형식**

표준 v1.1과 X API v2 엔드포인트 버전 간 가장 큰 차이 중 하나는 페이로드에 어떤 필드를 반환할지 선택하는 방식입니다.

표준 엔드포인트의 경우 기본적으로 많은 응답 필드가 포함되며, 이후 매개변수를 사용해 페이로드에 반환할 필드 또는 필드 집합을 지정할 수 있습니다.

X API v2는 기본적으로 user id, name, username 필드만 반환합니다. 추가 필드나 객체를 요청하려면 [fields](/ko/x-api/fundamentals/fields) 및 [expansions](/ko/x-api/fundamentals/expansions) 매개변수를 사용해야 합니다. 이 엔드포인트에서 요청한 사용자 필드는 기본 사용자 객체에 포함되어 반환됩니다. 확장된 Post 객체와 필드는 응답의 includes 객체에 포함되어 반환됩니다. 그런 다음 사용자 및 확장된 Post 객체에 있는 ID를 매칭하여 확장된 객체를 사용자 객체와 연결할 수 있습니다.

이러한 새 매개변수에 대해서는 각 가이드 또는 [fields 및 expansions 사용하는 방법](/ko/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) 가이드를 참고하시기 바랍니다.

또한 표준 v1.1 필드를 최신 v2 필드에 매핑하는 데 도움이 되는 [데이터 형식 마이그레이션 가이드](/ko/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2)도 제공합니다. 이 가이드는 특정 필드를 반환하기 위해 v2 요청에 포함해야 하는 구체적인 expansion 및 field 매개변수도 안내합니다.

특정 필드 요청 방식의 변경 외에도, X API v2는 API가 반환하는 객체(예: [Post](/ko/x-api/fundamentals/data-dictionary#tweet) 및 [user](/ko/x-api/fundamentals/data-dictionary#user) 객체)에 대해 새로운 JSON 설계를 도입합니다.

- JSON 루트 수준에서 표준 엔드포인트는 statuses 배열에 Post 객체를 반환하는 반면, X API v2는 data 배열을 반환합니다.
- 리트윗 및 인용 “statuses” 대신, X API v2 JSON은 리트윗 및 인용 트윗(Tweets)으로 표기합니다. contributors 및 user.translator\_type 같은 많은 레거시·더 이상 사용되지 않는 필드는 제거됩니다.
- Post 객체의 favorites와 user 객체의 favourites를 모두 사용하는 대신, X API v2는 like라는 용어를 사용합니다.
- 값이 없는 JSON 값(예: null)은 페이로드에 기록하지 않는 규칙을 채택했습니다. Post 및 user 속성은 null이 아닌 값을 가질 때에만 포함됩니다.

또한 [Post 객체](/ko/x-api/fundamentals/data-dictionary#tweet)에 다음을 포함한 새로운 필드 모음을 도입했습니다:

- [conversation\_id](/ko/x-api/fundamentals/conversation-id) 필드
- context와 entities를 포함하는 두 개의 새로운 [annotations](/ko/x-api/fundamentals/post-annotations) 필드
- 여러 새로운 [metrics](/ko/x-api/fundamentals/metrics) 필드
- 특정 Post에 누가 답글을 달 수 있는지를 나타내는 새로운 reply\_setting 필드

**요청 매개변수**

다음 표준 v1.1 요청 매개변수에는 X API v2에서의 동등 항목이 있습니다:

|     |     |
| :--- | :--- |
| **Standard** | **X API v2** |
| user\_id | ids |
| screen\_name | username |

X API v2에서 지원되지 않는 표준 users lookup 요청 매개변수 집합도 있습니다:

| 표준 | 설명 |
| :--- | :--- |
| include\_entities | 이 매개변수는 게시물 페이로드에서 entities 노드를 제거하는 데 사용됩니다. 이제 추가 가능한 필드 및 확장 기능으로 대체되었습니다. |
