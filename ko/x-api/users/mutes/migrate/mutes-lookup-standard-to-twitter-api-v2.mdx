<div id="mutes-lookup-standard-v11-compared-to-x-api-v2">
  ### 음소거 조회: Standard v1.1과 X API v2 비교
</div>

Standard v1.1 [GET mutes/users/ids](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/mute-block-report-users/api-reference/get-mutes-users-ids) 및 [GET mutes/users/list](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/mute-block-report-users/api-reference/get-mutes-users-list) 엔드포인트로 작업해 오셨다면, 이 가이드는 Standard v1.1과 X API v2의 음소거 조회 엔드포인트 간 유사점과 차이점을 이해하는 데 도움이 되도록 작성되었습니다.

- **유사점**
  - 인증
- **차이점**
  - 엔드포인트 URL
  - 요청당 사용자 수 제한
  - 앱 및 프로젝트 요구사항
  - 응답 데이터 형식
  - 요청 매개변수

<div id="similarities">
  #### 유사점
</div>

**인증**

표준 v1.1과 X API v2의 음소거 조회 엔드포인트는 모두 [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication#oauth-1-0a-2)를 사용합니다. 따라서 이전에 표준 v1.1의 음소거 조회 엔드포인트를 사용했다면, X API v2 버전으로 마이그레이션해도 동일한 인증 방식을 계속 사용할 수 있습니다. 

<div id="differences">
  #### 차이점
</div>

**엔드포인트 URL**

- 표준 v1.1 엔드포인트:
  - GET https://api.x.com/1.1/mutes/users/ids.json
    (지정된 사용자가 뮤트한 사용자 ID 목록)
  - GET https://api.x.com/1.1/mutes/users/lists.json
    (지정된 사용자가 뮤트한 사용자 목록)
- X API v2 엔드포인트:
  - GET https://api.x.com/2/users/:id/muting
    (지정된 사용자 ID가 뮤트한 사용자 목록)
     

**요청당 사용자 수 제한**

표준 v1.1 엔드포인트는 요청당 최대 5,000명의 사용자를 반환합니다. 새로운 v2 엔드포인트는 요청당 최대 1,000명의 사용자를 반환합니다. 1,000명을 모두 반환받으려면 쿼리 매개변수로 max\_results=1000을 전달해야 합니다. 이후 응답 페이로드에 포함된 next\_token 값을 다음 요청의 pagination\_token 쿼리 매개변수로 전달하면 됩니다.
 

**앱 및 프로젝트 요구 사항**

X API v2 엔드포인트는 요청 인증 시 [developer App](/ko/resources/fundamentals/developer-apps)의 자격 증명과 [Project](/ko/resources/fundamentals/developer-apps) 연계를 사용해야 합니다. 모든 X API v1.1 엔드포인트는 독립 실행형 앱 또는 프로젝트와 연계된 앱의 자격 증명을 사용할 수 있습니다.

**응답 데이터 형식**

표준 v1.1과 X API v2 엔드포인트 간 가장 큰 차이 중 하나는 페이로드에 어떤 필드를 반환할지 선택하는 방식입니다.

표준 엔드포인트는 기본적으로 많은 응답 필드를 제공하고, 이후 매개변수를 사용해 페이로드에 포함할 필드 또는 필드 집합을 지정할 수 있습니다.

X API v2는 기본적으로 user id, name, username 필드만 반환합니다. 추가 필드나 객체를 요청하려면 [fields](/ko/x-api/fundamentals/fields) 및 [expansions](/ko/x-api/fundamentals/expansions) 매개변수를 사용해야 합니다. 이 엔드포인트에서 요청한 모든 user 필드는 기본 user 객체에 포함됩니다. 확장된 Post 객체와 필드는 응답의 includes 객체에 포함됩니다. 그런 다음 user와 확장된 Post 객체에 있는 ID를 매칭하여 확장 객체를 해당 user 객체와 연결할 수 있습니다. 

이 새로운 매개변수에 대해서는 각 가이드 또는 [fields 및 expansions 사용 방법](/ko/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) 가이드를 참고하시길 권장합니다. 

또한 표준 v1.1 필드를 최신 v2 필드에 매핑하는 데 도움이 되는 [데이터 형식 마이그레이션 가이드](/ko/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2)를 준비했습니다. 이 가이드는 특정 필드를 반환하기 위해 v2 요청에 포함해야 하는 구체적인 expansion 및 field 매개변수도 제공합니다. 
 

특정 필드를 요청하는 방식의 변경 외에도, X API v2는 [Post](/ko/x-api/fundamentals/data-dictionary#tweet) 및 [user](/ko/x-api/fundamentals/data-dictionary#user) 객체를 포함하여 API가 반환하는 객체에 대해 새로운 JSON 설계를 도입했습니다.

- JSON 루트 레벨에서 표준 엔드포인트는 statuses 배열에 Post 객체를 반환하는 반면, X API v2는 data 배열을 반환합니다. 
- 리트윗 및 인용을 “statuses”로 지칭하는 대신, X API v2 JSON은 Retweeted 및 Quoted Tweets로 지칭합니다. contributors 및 user.translator\_type과 같은 많은 레거시 및 사용 중단 필드는 제거됩니다. 
- Post 객체의 favorites와 user 객체의 favourites를 모두 사용하는 대신, X API v2는 like라는 용어를 사용합니다. 
- X는 값이 없는 JSON 항목(예: null)은 페이로드에 기록하지 않는 관례를 채택하고 있습니다. Post 및 user 속성은 null이 아닌 값이 있을 때만 포함됩니다. 
   

또한 [Post 객체](/ko/x-api/fundamentals/data-dictionary#tweet)에 다음과 같은 새로운 필드 세트를 도입했습니다:

- [conversation\_id](/ko/x-api/fundamentals/conversation-id) 필드
- context 및 entities를 포함하는 두 개의 새로운 [annotations](/ko/x-api/fundamentals/post-annotations) 필드
- 여러 새로운 [metrics](/ko/x-api/fundamentals/metrics) 필드 
- 특정 Post에 누가 답글을 달 수 있는지 보여주는 새로운 reply\_setting 필드

**요청 매개변수**

다음 표준 v1.1 요청 매개변수는 X API v2에 상응하는 항목이 있습니다:

| **표준** | **X API v2** |
| :--- | :--- |
| stringify\_ids | 해당 없음 |
| cursor | pagination\_token |
| skip\_status | 해당 없음 |

또한 X API v2에서는 표준 v1.1 Mutes 조회 요청 매개변수 중 지원되지 않는 항목들이 있습니다:

| 표준 | 설명 |
| :--- | :--- |
| include\_entities | 이 매개변수는 Post 페이로드에서 entities 노드를 제거하는 데 사용됩니다. 해당 기능은 추가 필드 및 확장 기능으로 대체되었습니다. |
