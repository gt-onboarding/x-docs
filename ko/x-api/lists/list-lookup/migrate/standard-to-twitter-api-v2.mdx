<div id="list-lookup-standard-v11-compared-to-x-api-v2">
  ### 목록 조회: Standard v1.1과 X API v2 비교
</div>

Standard v1.1의 [GET lists/show](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-show) 및 [GET lists/ownerships](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-ownerships) 엔드포인트를 사용해 오셨다면, 이 가이드는 Standard v1.1과 X API v2의 목록 조회 엔드포인트 간 유사점과 차이점을 이해하는 데 도움을 드리기 위한 것입니다.

- **유사점**
  - 인증 방식
  - 요청 한도
- **차이점**
  - 엔드포인트 URL
  - 앱 및 프로젝트 요건
  - 요청당 데이터 객체 한도
  - 응답 데이터 형식
  - 요청 매개변수

<div id="similarities">
  #### 유사점
</div>

**인증**

두 엔드포인트 버전은 [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication#oauth-1-0a-2)와 [앱 전용](/ko/resources/fundamentals/authentication#app-only-authentication-and-oauth-2-0-bearer-token) 방식을 모두 지원합니다. 따라서 이전에 표준 v1.1 List 조회 엔드포인트를 사용했다면, X API v2로 마이그레이션해도 동일한 인증 방식을 계속 사용할 수 있습니다.

사용하는 인증 라이브러리/패키지에 따라 앱 전용 인증이 가장 쉽게 시작할 수 있는 방법일 가능성이 높으며, 간단한 요청 헤더로 설정할 수 있습니다. 앱 전용 액세스 토큰 생성 방법은 [앱 전용 가이드](/ko/resources/fundamentals/authentication#using-and-generating-an-app-only-bearer-token)를 참고하세요.

**요청 한도**

|     |     |
| :--- | :--- |
| **Standard v1.1** | **X API v2** |
| /1.1/lists/show.json<br /><br />OAuth 1.0a 사용자 컨텍스트: 15분당 75회<br /><br />앱 전용: 15분당 75회 | /2/lists/:id<br /><br />OAuth 1.0a 사용자 컨텍스트: 15분당 75회<br /><br />OAuth 2.0 Authorization Code with PKCE: 15분당 75회 |
| /1.1/lists/ownerships.json<br /><br />OAuth 1.0a 사용자 컨텍스트: 15분당 15회<br /><br />앱 전용: 15분당 15회 | /2/users/:id/owned\_lists<br /><br />OAuth 1.0a 사용자 컨텍스트: 15분당 15회<br /><br />OAuth 2.0 Authorization Code with PKCE: 15분당 15회<br /><br />앱 전용: 15분당 15회 |

<div id="differences">
  #### 차이점
</div>

**엔드포인트 URL**

- 표준 v1.1 엔드포인트:
  - GET https://api.x.com/1.1/lists/show.json
    (지정한 리스트 조회)
  - GET https://api.x.com/1.1/lists/ownerships.json
    (지정한 사용자가 소유한 리스트 조회)
- X API v2 엔드포인트:
  - GET https://api.x.com/2/lists/:id
    (지정한 리스트 조회)

  - GET https://api.x.com/2/users/:id/owned\_lists
    (지정한 사용자가 소유한 리스트 조회)

**앱 및 프로젝트 요구 사항**

X API v2 엔드포인트는 요청을 인증할 때 [developer App](/ko/resources/fundamentals/developer-apps)의 자격 증명과 해당 [Project](/ko/resources/fundamentals/projects) 연계를 사용해야 합니다. X API v1.1의 모든 엔드포인트는 독립 실행형 앱 또는 프로젝트와 연계된 앱의 자격 증명을 사용할 수 있습니다.

**요청당 데이터 객체 한도**

표준 v1.1 /lists/ownerships 엔드포인트는 요청당 최대 1000개의 리스트를 반환합니다. 새로운 v2 엔드포인트는 요청당 최대 100개의 리스트를 반환합니다. 기본적으로 100개의 사용자 객체가 반환되며, 결과 수를 변경하려면 1~100 사이의 숫자와 함께 쿼리 매개변수 max\_results=를 전달해야 합니다. 그런 다음 응답 페이로드에 반환된 next\_token을 다음 요청의 pagination\_token 쿼리 매개변수로 전달할 수 있습니다.

**응답 데이터 형식**

표준 v1.1과 X API v2 엔드포인트 버전 간의 가장 큰 차이점 중 하나는 페이로드에 어떤 필드를 반환할지 선택하는 방식입니다.

표준 엔드포인트에서는 많은 응답 필드가 기본으로 제공되며, 추가로 어떤 필드 또는 필드 집합을 페이로드에 반환할지 매개변수로 지정할 수 있습니다.

X API v2 버전은 기본적으로 리스트 id와 name 필드만 제공합니다. 추가 필드 또는 객체를 요청하려면 [fields](/ko/x-api/fundamentals/fields) 및 [expansions](/ko/x-api/fundamentals/expansions) 매개변수를 사용해야 합니다. 이 엔드포인트에서 요청한 리스트 필드는 기본 리스트 객체로 반환됩니다. 확장된 포스트 또는 사용자 객체와 필드는 응답의 includes 객체로 반환됩니다. 이후 사용자와 확장된 포스트 객체에 있는 ID를 매칭하여 확장된 객체를 리스트 객체와 연결할 수 있습니다. 

가능한 리스트 필드와 확장의 예시는 다음과 같습니다:

- created\_at

- follower\_count

- member\_count

- owner\_id

- description

- private

|     |     |
| :--- | :--- |
| **엔드포인트** | **확장** |
| /2/lists/:id | owner\_id |
| /2/users/:id/owned\_lists | owner\_id |

해당 가이드에서 이러한 새 매개변수에 대해 더 자세히 읽어보시거나, [fields와 expansions 사용 방법](/ko/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) 가이드를 참고하시기를 권장합니다. 

또한 표준 v1.1 필드를 새로운 v2 필드에 매핑하는 데 도움이 되는 [데이터 형식 마이그레이션 가이드](/ko/x-api/migrate/data-format-migration)도 준비했습니다. 이 가이드는 특정 필드를 반환하기 위해 v2 요청에 포함해야 하는 구체적인 expansion 및 field 매개변수를 제공합니다. 

특정 필드 요청 방식의 변경 외에도, X API v2는 [Post](/ko/x-api/fundamentals/data-dictionary#tweet) 및 [user](/ko/x-api/fundamentals/data-dictionary#user) 객체를 포함하여 API가 반환하는 객체에 대해 새로운 JSON 설계를 도입합니다.

- JSON 루트 수준에서 표준 엔드포인트는 statuses 배열에 Post 객체를 반환하는 반면, X API v2는 data 배열을 반환합니다. 

- 리트윗 및 인용을 “statuses”로 지칭하는 대신, X API v2 JSON은 리트윗 및 인용 트윗으로 지칭합니다. contributors 및 user.translator\_type과 같은 많은 레거시 및 사용 중단 필드는 제거됩니다. 

- Post 객체의 favorites와 user 객체의 favourites를 모두 사용하는 대신, X API v2는 like라는 용어를 사용합니다. 

- X는 값이 없는 JSON 값(예: null)은 페이로드에 기록하지 않는 규칙을 채택하고 있습니다. Post 및 user 속성은 null이 아닌 값이 있을 때만 포함됩니다.

**요청 매개변수**

다음 표준 v1.1 요청 매개변수는 X API v2에서 상응 항목이 있습니다:

**ID로 리스트 조회**

|     |     |
| :--- | :--- |
| **표준 v1.1** | **X API v2** |
| list\_id | id  |
| slug | 해당 없음 |
| owner\_screen\_name | 해당 없음 |
| owner\_id | expansions/fields 매개변수로 요청 |

**사용자 소유 리스트 조회**

|     |     |
| :--- | :--- |
| **Standard v1.1** | **X API v2** |
| user\_id | id  |
| screen\_name | 해당 없음 |
| count | max\_results |
| cursor | pagination\_token |
