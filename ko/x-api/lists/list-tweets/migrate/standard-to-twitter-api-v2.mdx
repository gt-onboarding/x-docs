  ### 목록 게시물 조회: Standard v1.1과 X API v2 비교
</div>

표준 v1.1 [GET lists/statuses](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-statuses) 엔드포인트를 사용해 오셨다면, 이 가이드는 표준 v1.1과 X API v2 엔드포인트 간의 공통점과 차이점을 이해하는 데 도움을 드리기 위한 것입니다.

- **유사점**
  - 인증 방식
  - 속도 제한
- **차이점**
  - 엔드포인트 URL
  - 앱 및 프로젝트 요구 사항
  - 요청당 데이터 오브젝트 제한
  - 응답 데이터 포맷
  - 요청 파라미터

<div id="similarities">
  #### 유사점
</div>

**인증**

두 엔드포인트 버전 모두 [OAuth 1.0a User Context](/ko/resources/fundamentals/authentication#oauth-2-0)를 지원합니다. 따라서 이전에 표준 v1.1 목록 게시물 조회 엔드포인트를 사용했다면, X API v2 버전으로 마이그레이션하더라도 동일한 인증 방식을 계속 사용할 수 있습니다.

사용하는 인증 라이브러리/패키지에 따라 App only 인증이 가장 간단한 시작 방법일 수 있으며, 간단한 요청 헤더로 설정할 수 있습니다. App only 액세스 토큰을 생성하는 방법은 [이 App only 가이드](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)를 참조하세요.

**요청 한도**

|     |     |
| :--- | :--- |
| **Standard v1.1** | **X API v2** |
| /1.1/lists/statuses.json<br /><br />OAuth 1.0a User Context로 15분당 900회 요청<br /><br />App only로 15분당 900회 요청 | /2/lists/:id/tweets<br /><br />OAuth 1.0a User Context로 15분당 900회 요청<br /><br />OAuth 2.0 Authorization Code with PKCE로 15분당 900회 요청<br /><br />App only로 15분당 900회 요청 |

<div id="differences">
  #### 차이점
</div>

**엔드포인트 URL**

- 표준 v1.1 엔드포인트:
  - GET https://api.x.com/1.1/lists/statuses.json
    (지정된 목록에서 트윗 조회)
- X API v2 엔드포인트:
  - GET https://api.x.com/2/lists/:id/tweets
    (지정된 목록에서 트윗 조회)

**앱 및 프로젝트 요구 사항**

X API v2 엔드포인트는 요청을 인증할 때 [developer App](/ko/resources/fundamentals/developer-apps)의 자격 증명을 사용해야 하며, 해당 App은 [Project](/ko/resources/fundamentals/projects)와 연결되어 있어야 합니다. 모든 X API v1.1 엔드포인트는 독립 실행형 App 또는 프로젝트와 연결된 App의 자격 증명을 사용할 수 있습니다.

**요청당 데이터 객체 제한**

표준 v1.1 /lists/statuses 엔드포인트는 요청당 최대 5000개의 게시물을 반환할 수 있습니다. 새 v2 엔드포인트는 요청당 최대 100개의 게시물을 반환할 수 있습니다. 기본적으로 100개의 사용자 객체가 반환되며, 결과 수를 변경하려면 1~100 사이의 숫자와 함께 쿼리 매개변수 max\_results=를 전달해야 합니다. 그런 다음 응답 페이로드에 반환된 next\_token을 다음 요청의 pagination\_token 쿼리 매개변수로 전달할 수 있습니다.

**응답 데이터 형식**

표준 v1.1과 X API v2 엔드포인트 버전 간 가장 큰 차이점 중 하나는 페이로드에 어떤 필드를 반환할지 선택하는 방식입니다.

표준 엔드포인트의 경우 기본적으로 많은 응답 필드를 수신하며, 매개변수를 사용하여 페이로드에 반환할 추가 필드 또는 필드 집합을 지정할 수 있습니다.

X API v2 버전은 기본적으로 게시물 id 및 text 필드만 제공합니다. 추가 필드나 객체를 요청하려면 [fields](/ko/x-api/fundamentals/fields) 및 [expansions](/ko/x-api/fundamentals/expansions) 매개변수를 사용해야 합니다. 이 엔드포인트에서 요청한 모든 게시물 필드는 기본 게시물 객체에 반환됩니다. 확장된 객체 필드는 응답 내 includes 객체에 반환됩니다. 그런 다음 기본 객체와 확장된 객체의 ID를 매칭하여 확장된 객체를 기본 게시물 객체에 다시 연결할 수 있습니다. 

가능한 게시물 필드 및 확장의 예시는 다음과 같습니다:

- attachments

- author\_id

- context\_annotations

- created\_at

- geo

- lang

|     |     |
| :--- | :--- |
| **엔드포인트** | **확장** |
| /2/lists/:id/tweets | author\_id |

이러한 새로운 매개변수에 대해서는 각각의 가이드에서 더 자세히 확인하시거나, [fields 및 expansions 사용 방법](/ko/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) 가이드를 참고하시기 바랍니다. 

또한 표준 v1.1 필드를 최신 v2 필드로 매핑하는 데 도움이 되는 [데이터 형식 마이그레이션 가이드](/ko/x-api/migrate/data-format-migration)도 준비했습니다. 이 가이드는 특정 필드를 반환하기 위해 v2 요청에 전달해야 하는 확장 및 필드 매개변수도 제공합니다. 

특정 필드 요청 방식의 변경 외에도, X API v2는 [Post](/ko/x-api/fundamentals/data-dictionary#tweet) 및 [user](/ko/x-api/fundamentals/data-dictionary#user) 객체를 포함하여 API가 반환하는 객체에 대해 새로운 JSON 설계를 도입하고 있습니다.

- JSON 루트 레벨에서 표준 엔드포인트는 게시물 객체를 **statuses** 배열로 반환하는 반면, X API v2는 **data** 배열로 반환합니다. 

- 리트윗과 인용을 “statuses”로 지칭하는 대신, X API v2 JSON은 리트윗과 인용 트윗으로 지칭합니다. **contributors** 및 **user.translator\_type**과 같은 많은 레거시 및 사용 중단 필드는 제거되고 있습니다. 

- **favorites**(게시물 객체)와 **favourites**(사용자 객체)를 모두 사용하는 대신, X API v2는 **like**라는 용어를 사용합니다.

- 값이 없는 JSON 값(예: **null**)은 페이로드에 기록하지 않는 규칙을 채택하고 있습니다. 게시물 및 사용자 속성은 null이 아닌 값이 있는 경우에만 포함됩니다.
   

**요청 매개변수**

다음 표준 v1.1 요청 매개변수는 X API v2에서의 대응 항목이 있습니다:

|     |     |
| :--- | :--- |
| 표준 v1.1 | X API v2 |
| list\_id | id  |
| slug | 해당 없음 |
| owner\_screen\_name | 해당 없음 |
| owner\_id | expansions 매개변수에 author\_id 값을 설정하여 요청 |
| since\_id | 해당 없음 |
| max\_id | 해당 없음 |
| include\_entities | tweet.fields 매개변수에 entities 값을 설정하여 요청 |
| include\_rts | 해당 없음 |
| count | max\_results |
