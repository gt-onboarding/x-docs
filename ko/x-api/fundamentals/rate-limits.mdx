---
title: 요청 한도
---

매일 수천 명이 넘는 개발자가 X API에 요청을 보냅니다. 이러한 대규모 트래픽을 관리하기 위해 요청 횟수에 한도가 적용됩니다. 이 한도는 우리 개발자 커뮤니티가 신뢰하는 안정적이고 확장 가능한 API를 제공하는 데 도움이 됩니다.

허용되는 최대 요청 수는 시간 간격을 기준으로 하며, 일반적으로 지정된 기간 또는 시간 창을 사용합니다. 가장 일반적인 간격은 15분입니다. 예를 들어, 15분당 900건의 한도가 설정된 엔드포인트는 임의의 15분 간격마다 최대 900건의 요청을 허용합니다.

요청 한도는 인증 방식에 따라 달라집니다. 예를 들어 [OAuth 1.0a User Context](/ko/resources/fundamentals/authentication)를 사용하는 경우 각 사용자의 [Access Tokens](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)에 기간별로 고유한 요청 한도가 부여됩니다. 반대로 [OAuth 2.0 Bearer Token](/ko/resources/fundamentals/authentication#oauth-2-0)을 사용하는 경우 앱 자체에 시간 기간별로 별도의 한도가 적용됩니다. 이러한 한도를 초과하면 오류가 반환됩니다.

<div id="rate-limits-and-authentication-method">
  ## 레이트 한도와 인증 방식
</div>

레이트 한도는 개발자 앱과 사용자 액세스 토큰 두 수준에서 설정됩니다:

- **[OAuth 2.0 Bearer Token](/ko/resources/fundamentals/authentication/oauth-2-0/application-only): 앱 레이트 한도**
  이 방식은 개발자 앱을 대신해 일정 수의 요청을 보낼 수 있게 합니다. 이 인증 방식을 사용할 때 한도는 Bearer Token으로 수행된 요청을 기준으로 결정됩니다.
  - 예시: 15분 간격당 450회 한도인 경우, 해당 간격 동안 앱을 대신하여 450개의 요청을 보낼 수 있습니다.

- **[OAuth 1.0a User Context](/ko/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens): 사용자 레이트 한도**
  이 방식은 사용자 액세스 토큰으로 식별되는 X 사용자를 대신해 요청을 보낼 수 있게 합니다. 예를 들어 게시물의 비공개 지표를 조회하는 경우, [3-legged OAuth flow](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)를 사용해 생성한 해당 사용자의 사용자 액세스 토큰으로 인증하십시오.
  - 예시: 사용자당 15분에 900회 한도인 경우, 해당 시간 동안 사용자당 최대 900개의 요청을 보낼 수 있습니다.

<div id="http-headers-and-response-codes">
  ## HTTP 헤더와 응답 코드
</div>

가장 최근 요청을 기준으로, 주어진 레이트 리밋 내에서 애플리케이션의 상태를 파악하려면 HTTP 헤더를 사용하세요.

- `x-rate-limit-limit`: 해당 엔드포인트의 레이트 리밋 상한
- `x-rate-limit-remaining`: 15분 윈도우에서 남은 요청 수
- `x-rate-limit-reset`: 레이트 리밋이 재설정되기까지 남은 시간(UTC 에포크 초)

<div id="error-responses">
  ### 오류 응답
</div>

애플리케이션이 특정 엔드포인트의 요청 한도를 초과하면 API는 응답 본문에 다음 오류 메시지를 포함한 [HTTP 429 “Too Many Requests”](http://tools.ietf.org/html/rfc6585) 응답을 반환합니다:

```json
{ "errors": [ { "code": 88, "message": "Rate limit exceeded" } ] }
```

<div id="recovering-from-a-rate-limit">
  ## 속도 제한에서 복구하기
</div>

이러한 속도 제한을 초과하면 엔드포인트에서 429 'Too many requests' 오류가 반환됩니다. 아래에서 설명하듯, 속도 제한 오류가 발생하면 제한이 언제 재설정되는지 나타내는 HTTP 헤더를 확인하고 그때까지 요청을 중단하는 것이 모범 사례입니다.<br />"too many requests" 또는 속도 제한 오류가 발생하면 요청 빈도를 낮춰야 합니다. 속도 제한 오류가 발생했을 때는 `x-rate-limit-reset:` HTTP 헤더를 확인하여 속도 제한이 언제 재설정되는지 알 수 있습니다.<br /><br /> 또 다른 일반적인 방식은 지수 백오프입니다. 요청 간 간격을 처음에는 짧게(예: 수초) 시작한 뒤, 재시도할 때마다 두 배로 늘립니다. 이는 요청이 성공하거나 요청 간의 합리적인 최대 간격(예: 수분)에 도달할 때까지 계속됩니다.<br /><br /> 이상적으로는 클라이언트가 기존 속도 제한을 인지하고, 현재 초과한 윈도우가 만료될 때까지 요청을 일시 중지할 수 있어야 합니다. 15분 제한을 초과했다면, 다시 시도하기 전에 1~2분 정도 기다리는 것이 합리적입니다.<br /><br /> 요청 수에 대한 이러한 제한 외에도, Standard Basic 액세스 수준은 최근 검색 및 필터드 스트림 엔드포인트에서 월 최대 500,000개의 게시물을 제공합니다. 게시물 수에 대한 월간 한도를 초과한 경우, 앱이 알림을 띄우고 월별 등록일을 파악해 그날까지 요청을 보류하는 편이 더 합리적입니다.

<div id="tips-to-avoid-being-rate-limited">
  ### 레이트 리밋을 피하기 위한 팁
</div>

아래 팁은 방어적으로 코드를 작성하고 레이트 리밋에 걸릴 가능성을 줄이는 데 도움이 됩니다. 특히 결과의 최신성과 관련해서는, 레이트 리밋을 감안할 때 제공하려는 일부 애플리케이션 기능이 사실상 불가능할 수 있습니다. 애플리케이션의 목표가 실시간 정보라면 필터드 스트림 및 샘플드 스트림 엔드포인트를 검토하세요.

<div id="caching">
  #### 캐싱
</div>

자주 사용할 것으로 예상되는 API 응답은 저장하세요. 모든 페이지 로드마다 API를 호출하는 대신 응답을 로컬에 캐시하세요.

<div id="prioritize-active-users">
  #### 활성 사용자 우선순위 지정
</div>

사이트에서 다수의 X 사용자를 추적한다면(예: 현재 상태를 가져오거나 X 사용 통계를 수집하는 경우), 최근에 해당 사이트에 로그인한 사용자에 대해서만 데이터를 요청하는 방안을 고려하십시오.

<div id="adapt-to-the-search-results">
  #### 검색 결과에 맞춰 조정
</div>

애플리케이션이 많은 검색어를 모니터링한다면, 결과가 있는 검색보다 결과가 없는 검색에 대해 쿼리 빈도를 더 낮추십시오. 백오프를 사용하면 인기 있는 쿼리는 최신 상태로 유지하면서, 거의 변화가 없는 쿼리에 대한 불필요한 요청으로 리소스를 낭비하지 않을 수 있습니다. 또는 필터드 스트림 엔드포인트를 사용하고 검색 쿼리로 필터링하는 방식을 고려하십시오.

<div id="denylist">
  #### 차단 목록
</div>

애플리케이션이 요청 한도(rate limit)를 남용하면 차단됩니다. 차단된 앱은 X API로부터 응답을 받을 수 없습니다. 본인 또는 애플리케이션이 차단되었고 실수라고 생각하는 경우 [Platform Support forms](https://support.x.com/forms/platform)를 통해 지원을 요청할 수 있습니다. 다음 정보를 포함해 주세요:

1. 애플리케이션이 차단된 이유에 대한 본인의 판단을 설명해 주세요.
2. 더 이상 요청 한도에 걸리지 않는 경우, 문제를 어떻게 해결했는지 상세히 설명해 주세요.
