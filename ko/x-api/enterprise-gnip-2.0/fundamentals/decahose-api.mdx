<div id="decahose-stream">
  ### Decahose 스트림
</div>

[Enterprise](https://developer.x.com/en/products/x-api/enterprise)

_이 API는 당사 관리형 액세스 수준에서만 제공되는 엔터프라이즈 API입니다. 이 API를 사용하려면 먼저 엔터프라이즈 영업팀을 통해 계정을 설정해야 합니다. [자세히 알아보기](https://developer.x.com/en/products/x-api/enterprise)_

Decahose는 스트리밍 연결을 통해 실시간 X Firehose의 10%를 무작위로 샘플링해 제공합니다. 이는 실시간 샘플링 알고리즘으로 구현되며, 무작위로 데이터를 선택하면서도 X가 Firehose로 전송하는 데이터가 기대되는 저지연으로 전달되도록 보장합니다.

아래는 Decahose에서 제공되는 일부 기능입니다.

- **확장 및 강화된 URL** - 단축 URL을 완전히 해제(unwind)하고 추가 메타데이터(페이지 제목 및 설명)를 제공합니다.
- **스트림 파티셔닝** - 2개의 파티션으로 구성되며, 각 파티션은 Decahose 스트림 볼륨의 50%를 포함합니다.
- **향상된 안정성** - 백엔드 시스템의 지리적 다양성

참고: 이 데이터는 일괄 제공되며, 추가 필터링(예: 키워드)은 지원하지 않습니다.

`ENTERPRISE`

<div id="streaming-likes">
  ### 좋아요 스트리밍
</div>

_이는 관리형 액세스 레벨에서만 제공되는 엔터프라이즈 API입니다. 이 API를 사용하려면 먼저 엔터프라이즈 영업팀과 계정을 설정해야 합니다. [자세히 알아보기](https://developer.x.com/en/products/x-api/enterprise)_

좋아요는 누가 게시물을 좋아하는지에 대한 인사이트와 정확한 좋아요 수를 제공합니다. Gnip의 Firehose와 Decahose는 Gnip을 통해 제공된 게시물과 관련된 공개 좋아요를 전송할 수 있습니다. 이는 게시물과 연관된 실시간 공개 참여 및 잠재고객 지표를 제공합니다.
 

**좋아요 시작하기**

좋아요 데이터를 수집하기 전에 알아두어야 할 사항:

- 좋아요는 별도의 독립 스트림으로 제공됩니다
- 좋아요는 과거에 “Favorites”로 불렸습니다. 확장된 네이티브 형식 페이로드는 이 명명법을 유지합니다
- 스트림에는 공개 좋아요만 포함됩니다
  - 공개란 좋아요를 누른 사용자, 게시물 작성자, 게시물이 모두 플랫폼에서 공개 상태임을 의미합니다
- 좋아요는 리트윗과 매우 유사하며 참여에 대한 공개 신호를 나타냅니다
- 페이로드 요소에는 다음이 포함됩니다:
  - 원본 게시물 객체
  - 원본 게시물을 생성한 액터 객체
  - 좋아요 동작을 수행한 액터 객체
- 원본 콘텐츠에만 좋아요를 누를 수 있습니다
  - 리트윗에는 좋아요를 누를 수 없습니다. 리트윗에 대한 좋아요는 원본 게시물에 적용됩니다
  - 인용 트윗은 좋아요를 누를 수 있습니다
- 좋아요 활동에는 해당되는 경우 Gnip Enrichments가 포함됩니다(구매/적용된 경우)
- 지원 제품/기능
  - 좋아요 스트림은 백필을 지원합니다(구매/적용된 경우)
  - 좋아요 스트림에는 Replay 지원이 없습니다
  - 좋아요에는 검색 또는 과거 데이터 지원이 없습니다
  - PowerTrack에 좋아요 지원을 추가할 즉각적인 계획은 없습니다

**Decahose**

- Decahose에서 제공되는 10% 샘플 게시물에 대해서는 스트림에 해당 공개 좋아요의 100%가 포함됩니다
- **파티션:** 2
- **URL 구조**
  - https://gnip-stream.x.com/stream/sample10-likes/accounts/\<accountName>/publishers/twitter/\<streamLabel>.json?partition=1

<div id="recovery-and-redundency">
  ### 복구 및 이중화
</div>

**소개** 

실시간 게시물을 대량으로 스트리밍할 때에는 데이터 신뢰성과 데이터 완전성을 모두 보장하기 위한 모범 사례가 있습니다. 실시간 데이터를 수집할 때는 연결 시간을 최대화하는 것이 핵심 목표입니다. 연결이 끊어지면 이를 자동으로 감지해 재연결하는 것이 중요합니다. 재연결 후에는 보강이 필요한 기간이 있는지 점검해야 합니다. 이러한 세부 사항을 관리하며 실시간 게시물을 수집하는 구성 요소는 네트워크, 데이터 저장소, 서버, 스토리지 등 다양한 요소를 아우르는 시스템의 일부에 불과합니다. 시스템이 복잡한 만큼, 별도의 스트리밍 환경을 두는 것도 모범 사례입니다. 최소한 개발/테스트용과 프로덕션용 스트림을 구분해 운영하세요.

Decahose에는 이러한 작업을 지원하는 기능이 포함되어 있습니다.

1. 여러 환경을 지원하기 위해 계정에 [추가 스트림](#AdditionalStreams)을 배포할 수 있습니다. 각 스트림은 서로 독립적이며 구분을 위해 서로 다른 stream\_label을 가집니다.
2. 연결 유지를 돕기 위해 각 Decahose 스트림은 [이중 연결](/ko/x-api/enterprise-gnip-2.0/fundamentals/decahose-api#additional-streams)을 지원합니다. 가장 일반적인 아키텍처는 하나의 스트림에 두 개의 연결을 두고, 클라이언트 측에는 이상적으로 서로 다른 네트워크에 위치한 두 개의 독립 소비자를 두는 것입니다. 이 설계로 클라이언트 측 네트워크, 서버, 데이터 저장소 경로 전반에 걸쳐 이중화를 확보할 수 있습니다. 각 연결에서 데이터의 전체 사본이 제공되므로, 클라이언트는 중복 데이터를 허용하고 관리할 수 있어야 합니다.
3. 10초마다 ‘하트비트’가 전송됩니다. 다만 Decahose 스트림은 데이터 볼륨이 충분히 크기 때문에 몇 초 정도의 짧은 무게시 구간만으로도 연결 문제를 시사할 수 있습니다. 따라서 ‘데이터 무음’과 하트비트 부재 모두로 연결 끊김을 감지할 수 있습니다.

연결 끊김은 불가피하므로, Decahose 스트림은 연결 끊김 및 기타 운영 이슈로 누락된 데이터를 복구하는 데 도움이 되는 전용 [복구](/ko/x-api/enterprise-gnip-2.0/fundamentals/decahose-api#overview) 및 [백필](/ko/x-api/enterprise-gnip-2.0/fundamentals/decahose-api#backfill) 기능을 제공합니다.

<div id="additional-streams">
  #### 추가 스트림
</div>

추가 Decahose 스트림을 보유하면 솔루션의 신뢰성을 높일 수 있습니다. 추가 스트림은 각기 고유한 엔드포인트를 갖는 완전히 독립적인 스트림입니다. 각 스트림에는 고유한 stream\_label이 할당되며, 이 레이블은 계정 이름과 함께 해당 스트림의 URL 일부가 됩니다. 아래 예시를 참조하세요:

https://gnip-stream.x.com/stream/sample10/accounts/:account\_name/publishers/twitter/:stream\_label.json

가장 일반적인 관례는 운영 시스템 전용 실시간 스트림 하나와, 개발 및 테스트용 추가 스트림 하나를 갖는 것입니다. 테스트/개발 스트림이 있으면 Decahose 고객은 클라이언트 컨슈머 업데이트를 안전하게 검증할 수 있습니다. 스트림에는 어떤(고유한) 레이블이든 할당할 수 있지만, 운영 스트림에는 ‘prod’, 개발용 추가 스트림에는 ‘dev’ 또는 ‘sandbox’를 사용하는 관례가 있습니다.

스트림 수와 고유 레이블은 계정 담당자를 통해 구성할 수 있습니다.

**이중화 연결**

이중화 연결은 데이터 스트림에 두 개 이상의 동시 연결을 설정할 수 있게 해 줍니다. 즉, 두 개의 별도 컨슈머로 동일한 스트림에 연결하여 두 연결 모두를 통해 동일한 데이터를 수신함으로써 이중화를 제공합니다. 따라서 한 스트림이 끊기거나 앱의 기본 서버가 장애가 발생하는 등의 상황에서 앱에 즉각적인 페일오버가 가능합니다.

특정 스트림에 허용되는 연결 수는 계정 담당자를 통해 구성할 수 있습니다. 이중화 스트림을 사용하려면 기본 연결에 사용한 것과 동일한 URL에 그대로 연결하면 됩니다. 스트림 데이터는 두 연결 모두를 통해 전송되며, 두 연결 모두 스트림 대시보드에 표시됩니다.

청구 목적상, 여러 연결을 통해 수신한 활동 수는 중복 제거되어 각 고유한 활동에 대해서만 한 번 청구됩니다. Decahose에는 두 개의 파티션이 있으므로, 연결 수가 계산되는 방식은 아래 예시와 같습니다:

Connect to decahose partition=1
Connect to decahose partition=1
Connect to decahose partition=2

위 상황은 총 세 개의 연결을 의미합니다. 즉, partition=1에 두 개, partition=2에 한 개의 연결입니다. 일반적으로는 각 파티션에 동일한 수의 연결을 두는 것이 바람직하므로, 이 예시는 partition=2에 대한 이중화 연결이 끊겨 추가 조사가 필요한 상황을 보여줍니다.

**복구**

#### 개요 

Recovery는 최근 X 이력 데이터의 롤링 5일 구간에 스트리밍으로 접근할 수 있는 데이터 복구 도구입니다(기본 데이터 수집용으로 사용하지 마십시오). 이 도구는 소비 애플리케이션이 실시간 스트림에서 데이터를 놓친 경우 데이터를 복구하는 데 사용해야 합니다. 예를 들어 짧은 기간 동안 연결이 끊겼거나, 그 밖의 이유로 일정 시간 실시간 데이터를 수집하지 못한 경우에 활용할 수 있습니다.

#### 복구 사용 

Recovery 스트림을 사용하면 앱이 실시간 스트림과 동일한 방식으로 요청을 보낼 수 있습니다. 다만, 요청하려는 시간 범위를 나타내는 매개변수를 URL에 지정해야 합니다. 즉, Recovery 요청은 API에 “시간 A부터 시간 B까지의 게시물”을 요청하는 것입니다. 이러한 게시물은 실시간 스트림과 유사한 방식으로 스트리밍 연결을 통해 전달되지만, 실시간보다 약간 느린 속도로 전달됩니다. 아래 예시를 참고하세요:

"https://stream-data-api.x.com/stream/powertrack/accounts/someAccountName/publishers/twitter/powertrack.json?startTime=2023-07-05T17:09:12.070Z"

게시물은 지정된 기간의 첫 번째(가장 오래된) 분부터 전달되기 시작하여 마지막 분이 전달될 때까지 시간순으로 이어집니다. 그 시점에 “Recovery Request Completed” 메시지가 연결을 통해 전송되고, 서버가 연결을 종료합니다. 요청이 일치하는 결과가 거의 없거나 전혀 없는 시간대에서 시작된 경우 첫 결과가 전달되기까지 시간이 걸릴 수 있습니다. 해당 시점에 처리 중인 아카이브 구간에서 일치 항목을 Recovery가 발견하면 데이터가 전달됩니다. 전달할 결과가 없을 때에는 연결 시간 초과를 방지하기 위해 스트림이 캐리지 리턴, 즉 “하트비트”를 계속 전송합니다.

Recovery는 짧은 연결 해제로 인해 누락된 데이터를 손쉽게 복구하기 위한 도구이며, 하루 전체와 같은 매우 긴 기간에는 적합하지 않습니다. 장기간의 데이터를 복구해야 하는 경우 인터넷 불안정 또는 기타 이유로 요청 중간에 연결이 끊길 가능성을 줄이고, 긴 요청의 진행 상황에 대한 가시성을 높이기 위해 더 긴 요청을 더 짧은 시간 창(예: 2시간)으로 분할할 것을 권장합니다.

<div id="data-availability">
  #### 데이터 가용성
</div>

5분 백필 윈도우 내에 재연결하지 못한 경우, 복구 기능을 사용해 지난 24시간 내에 누락된 데이터를 복원할 수 있습니다.

스트리밍 복구 기능은 백필 윈도우를 24시간으로 확장합니다. 복구는 누락된 데이터의 시간 구간을 ‘복구’할 수 있게 해줍니다. 'start\_time' 및 'end\_time' 요청 매개변수를 사용해 연결을 요청하면 복구 스트림이 시작됩니다. 연결되면, 복구는 지정된 시간 구간을 다시 스트리밍한 뒤 연결을 종료합니다.  

동시에 최대 2개의 복구 요청(즉, “두 개의 복구 작업”)을 실행할 수 있습니다. 복구는 시작 시간과 종료 시간을 지정한다는 점을 제외하면 기술적으로 백필과 동일한 방식으로 동작합니다. 복구 기간은 단일 시간 범위를 대상으로 합니다.

<div id="backfill">
  #### 백필
</div>

백필을 요청하려면 연결 요청에 backfillMinutes=N 매개변수를 추가해야 합니다. 여기서 N은 연결 시 백필할 분(1–5, 정수만 해당)입니다. 예를 들어 90초 동안 연결이 끊겼다면 연결 요청에 backfillMinutes=2를 추가해야 합니다. 이 요청은 연결이 끊기기 전 30초 구간을 포함해 2분간의 백필을 제공하므로, 귀하의 _컨슈머 앱은 중복 데이터에 견고해야 합니다_.

5분 백필을 파티션 1에 요청하는 Decahose 연결 요청 URL 예시는 다음과 같습니다:

https://gnip-stream.x.com/stream/sample10/accounts/:account\_name/publishers/twitter/:stream\_label.json?partition=1\&backfillMinutes=5

**참고:**

- 연결 시 항상 ‘backfillMinutes=5’를 사용할 수도 있으며, 이 경우 제공되는 중복 데이터를 애플리케이션에서 중복 제거하면 됩니다.

- 5분을 초과해 연결이 끊긴 경우 Recovery를 사용하여 데이터를 복구할 수 있습니다.

**연결 끊김 복구**

연결이 끊긴 후 재시작 및 복구는 다음 단계로 진행됩니다:

- 연결 끊김 기간 파악
  - 5분 이하인가?
    - 스트림에 백필이 활성화되어 있다면 적절한 ‘backfillMinutes’ 매개변수로 연결 요청을 준비합니다.
  - 5분 초과인가?
    - Recovery 스트림이 있다면 끊겨 있었던 기간에 대해 Recovery 요청을 합니다(필요 시 Rules API를 사용해 현재 실시간 규칙 집합으로 요청하는 것이 이상적).
- 새 연결을 요청합니다.

연결 끊김이나 다운타임이 발생했을 때, 다음 전략으로 완화 및 복구할 수 있습니다:

1. **백필 구현**
   백필은 스트림 연결이 끊기기 이전 시점으로 되돌려 재연결할 수 있게 하며, 최대 5분의 끊김을 보완합니다. 연결 요청에 매개변수를 포함해 구현합니다.

2. **다른 위치에서 중복 스트림 소비**
   중복 스트림을 동일한 실시간 환경으로 유입하고 중복 제거가 가능하다면, 일반 스트림과 중복 스트림이 동시에 다운되거나 끊기지 않는 한 복구가 필요하지 않습니다. 중복 스트림을 프로덕션 환경으로 실시간 유입할 수 없다면 별도의 “비상” 데이터 저장소에 기록할 수 있습니다. 그러면 기본 스트림 연결에 끊김이나 다운타임이 발생할 경우, 시스템은 누락된 기간을 보완할 데이터를 보유하게 됩니다.

3. **Recovery 구현**
   끊김이나 다운타임이 기본 스트림과 중복 스트림 모두에 영향을 미치는 경우, 누락된 데이터를 복구하기 위해 Decahose Recovery를 사용하세요. API는 아카이브의 5일 롤링 윈도우를 제공하며, 한 번에 최대 1시간 단위로 요청해 스트리밍하는 방식이 가장 좋습니다. 이는 실시간 스트림과 병렬로 수행됩니다. Recovery가 제공하는 5일 윈도우를 넘어서는 Decahose 데이터 복구 솔루션은 없으므로, 귀하 측에 중대한 다운타임이 발생하더라도 완전한 데이터 사본을 확보할 수 있도록 중복 스트림을 활용하는 것이 중요합니다.

비정상적인 저장 데이터 볼륨을 감지할 때 —\
연결 끊김이나 다운타임이 없었는데도 누락된 데이터를 감지하는 방법:

1. 수신된 게시물(Post) 수 집계
   수집 앱의 가장 앞단에서 수신한 원시 게시물 수를 집계하고, 최종 데이터 저장소에 도달한 게시물 수와 비교할 수 있어야 합니다. 차이를 모니터링하여, 수신 후 데이터가 누락되게 하는 문제를 팀에 경고할 수 있습니다.

2. 비정상적인 저장 볼륨 분석
   최종 데이터베이스의 저장 데이터 볼륨을 분석해 급격한 하락이 있는지 확인할 수 있습니다. 이는 문제를 시사할 수 있으나, 경우에 따라 볼륨 하락이 정상일 수 있는 상황도 있습니다(예: X 플랫폼 장애로 일정 기간 동안 사용자가 게시물을 생성할 수 없는 경우).

<div id="replay-api">
  #### Replay API
</div>

Replay API는 실시간 볼륨 스트림을 보완하는 중요한 구성 요소입니다. Replay는 최근 X의 히스토리컬 데이터에 대해 롤링 윈도우로 스트리밍 액세스를 제공하는 데이터 복구 도구입니다.
